# B-182 実施計画: 占い・診断・おみくじの定義整理と既存コンテンツの分類

## 目的

ターゲットユーザー（手軽で面白い占い・診断を楽しみたい人）が、コンテンツの種類を正しく期待し、混乱なく楽しめるようにする。具体的には、(1) コンテンツカテゴリの定義をドキュメント化し、(2) 既存コンテンツの名称・ラベルの不一致を修正し、(3) UI上でカテゴリを適切に表示する。

## 計画の全体構造

本計画は3つのステップで構成する。ステップ1はドキュメント作業、ステップ2-3はbuilder作業となる。

---

## ステップ1: 定義ドキュメントの作成

### 作業内容
docs/content-categories.md（新規）を作成し、以下の4カテゴリの定義・品質基準を記載する。

### 4カテゴリの定義（調査メモ 19cd03c0f5f に基づく）

1. **診断（Personality Quiz）**: ユーザーの回答に基づいて決定論的に結果が決まる。同じ回答なら常に同じ結果。
2. **占い（Fortune-telling）**: ユーザーの属性（生年月日等）または日時情報で結果が決まる。質問への回答ではなく固有属性が入力。
3. **おみくじ（Omikuji）**: ランダム性が主体。毎回または毎日異なる結果。「引く」体験が目的。
4. **知識テスト（Knowledge Quiz）**: 正誤がある問題に回答し、正答率でランク判定。学習価値がある。

### 品質基準（カテゴリ別）
調査メモの「セクション5: 各カテゴリに適切な品質基準の提案」をそのまま採用する。

### ドキュメントの位置づけ
- site-concept.md には追記しない（既に長大であり、品質基準の詳細まで入れると肥大化する）
- 独立した docs/content-categories.md として作成し、site-concept.md から参照リンクを張る
- site-concept.md のセクション8「品質基準」に、content-categories.md への参照を1行追加する

### 担当
PMが直接作成する（docsの直接更新はPMの責務）。

## ステップ2: character-fortune のタイトル変更

### 判断根拠
- character-fortune の実態は「診断」（8問のスコアリングで6タイプに分類する決定論的構造）
- 現在のタイトル「あなたの守護キャラ占い」は「占い」を名乗っており、来訪者の期待とギャップがある
- ターゲットユーザーの dislikes に「こじつけで納得できない答えや判定」がある。「占い」を期待して診断が出ると違和感を生むリスクがある

### URL変更について
- URLスラッグ（character-fortune）は変更しない。理由: (1) 既にインデックスされたURLの変更はSEOリスクが高い、(2) スラッグは来訪者が直接目にする情報ではないため混乱を生まない、(3) 301リダイレクト設定のコストが発生する
- タイトル（title）とUI上の表示のみ変更する

### 変更内容
- `src/quiz/data/character-fortune.ts` の `meta.title` を「あなたの守護キャラ占い」から「あなたの守護キャラ診断」に変更
- 同ファイルの `meta.description` と `meta.shortDescription` 内の「占い」を「診断」に変更（もし含まれていれば）
- OGP関連のタイトル・descriptionも連動して変わるか確認（metaから自動生成されているなら追加変更不要）
- ブログ記事でcharacter-fortuneに言及している箇所があれば、タイトル表記を更新する

### contrarian-fortune について
- 現在のタイトル「逆張り運勢診断」は「診断」と名乗っており、実態と一致している
- 「運勢」という単語は含まれるが、ユーモアコンテンツとしての演出であり、来訪者が「おみくじ的なランダム性」を期待する程度の誤解は低い
- 変更不要と判断する

### 担当
builderエージェントに依頼する。

## ステップ3: クイズ一覧ページのカテゴリ表示改善

### 現状の問題
- 現在の `/quiz` 一覧ページでは、各コンテンツに「知識テスト」「性格診断」の2種類のバッジしか表示されない（`quiz.type === "knowledge" ? "知識テスト" : "性格診断"` のハードコード）
- 日替わり占い（fortune/daily）のカードは「占い」バッジで表示されている（別途ハードコード）
- 来訪者から見ると、診断・知識テスト・おみくじの区別がUIレベルで分かりにくい

### 変更方針
QuizType を拡張するのではなく、**表示ラベル（displayCategory）をQuizMetaに追加する**方針を採る。

理由:
- QuizType（knowledge / personality）はスコアリングロジックの分岐に使われている型であり、UIのラベルのためにこの型を拡張するのは関心の分離に反する
- 「診断」と「おみくじ」はスコアリングロジック上は同じpersonality型で処理可能だが、来訪者への表示は異なるべき

### 具体的な変更

#### 3-a. QuizMeta に displayCategory フィールドを追加
- `src/quiz/types.ts` の `QuizMeta` に `displayCategory?: string` を追加（optional）
- 未設定の場合は現在のロジック（knowledge → "知識テスト"、personality → "性格診断"）をフォールバックとして維持

#### 3-b. 各クイズデータに displayCategory を設定
| コンテンツ | displayCategory |
|---|---|
| kanji-level | "知識テスト" |
| kotowaza-level | "知識テスト" |
| yoji-level | "知識テスト" |
| traditional-color | "性格診断" |
| yoji-personality | "性格診断" |
| impossible-advice | "性格診断" |
| contrarian-fortune | "性格診断" |
| unexpected-compatibility | "性格診断" |
| music-personality | "性格診断" |
| character-fortune | "性格診断" |

注: 現時点では全てが現状のフォールバックと同じ値になるが、将来の拡充（B-181）で「占い」カテゴリのコンテンツが追加された際にdisplayCategoryが活用される。

#### 3-c. クイズ一覧ページのバッジ表示を修正
- `src/app/quiz/page.tsx` のバッジ表示部分で、`displayCategory` が設定されていればそれを使い、なければ従来のロジックでフォールバックする

#### 3-d. ページタイトルとメタデータの更新検討
- 現在のページタイトルは「クイズ・診断」だが、おみくじ（日替わり占い）も含んでいるため、「占い・診断・クイズ」への変更を検討する
- ただし、site-concept.md では「占い・診断パーク」がサイトコンセプトであり、将来的にはより包括的なタイトルが適切
- 変更案: 「占い・診断」（サイトコンセプトに合致。知識テストは「診断」の広義に含まれる）

## B-181（コンテンツ拡充）への示唆

定義整理の結果から、B-181で拡充する際の方向性を以下に示す。

1. **現在サイトに「占い」カテゴリの正規コンテンツがない**: character-fortuneを「診断」に修正した場合、サイト内で「占い」と正確に呼べるのは fortune/daily（おみくじ）のみであり、これは「占い」ではなく「おみくじ」。site-concept.md のQ25（キャラ占いプラットフォーム）は「キャラクターが結果を告げる」とあるが、入力が属性ベースか質問ベースかで占い/診断の分類が変わる。
2. **品質基準を満たさない既存コンテンツの拡充より、新コンテンツ制作が優先**: 既存の診断コンテンツ（6-8タイプ、8問程度）は品質基準の「最低基準」付近にある。タイプ数や質問数の増加は有効だが、site-concept.md のPhase 2コンテンツ（Q08動物診断、Q14理系診断、Q21日本文化診断）の制作が先行すべき。
3. **相性診断の拡充は効果的**: 調査によると相性診断はリピートとシェアを促進する。music-personality（36組）とcharacter-fortune（21組）は既に相性を持っている。他の診断にも相性を追加する拡充は、B-181として有効。
