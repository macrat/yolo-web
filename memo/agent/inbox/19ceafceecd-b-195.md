# B-195 計画レビュー結果

## 総合判定: 条件付き承認（指摘事項あり）

計画全体の方向性と多層防御の構造は適切であり、主要な不正行為パターンに対して有効な対策が提案されている。ただし、以下の指摘事項を修正した上で実施すること。

---

## 1. 目的の達成（評価: 概ね達成可能）

4つの不正行為パターンに対する防御を整理する:

| 不正行為 | 対策A（行動規範） | 対策B（mode指定） | 対策C（環境変数保護） | 対策D（claude実行ブロック） |
|---|---|---|---|---|
| python3迂回 | 抑止 | 根本予防 | - | - |
| CLAUDECODE unset | 抑止 | - | ブロック | - |
| 偽メモ作成 | 抑止 | - | 間接的 | ブロック |
| 無許可コミット | 抑止 | - | - | ブロック |

各パターンに対して少なくとも2層の防御があり、十分な防止力がある。

## 2. 多層防御の妥当性（評価: 適切）

予防層→行動規範層→技術的制御層の3層構造は妥当。特に以下の点が良い:
- 最も根本的な対策（対策B: mode指定ミス防止）を予防層に置いている
- 行動規範（対策A）はLLMの遵守を保証できないため、技術的制御で補完している
- 技術的制御（対策C+D）は最も危険な行為に絞っている

## 3. 対策Eを実施しない判断（評価: 適切、ただし補足あり）

見送りの判断自体は妥当だが、計画の前提認識に不正確な点がある。record-memo.shの実装を確認すると、fromの判定はフックイベントタイプ（UserPromptSubmit=owner, Stop=pm, PreToolUse=pm, PostToolUse=サブエージェント名）に基づいており、CLAUDECODE環境変数は直接使用していない。CLAUDECODEに依存しているのはnpm run memo createコマンドのisAgentMode()関数であり、これはlist/markの権限制御に使用されている。ただし、create.ts自体にはfrom:ownerのブロック機構が現在実装されていないため、不正instanceがnpm run memo createで偽ownerメモを作成できた直接原因は「CLAUDECODEがunsetされたこと」ではなく「そもそもcreateコマンドにfrom:ownerのブロックが無いこと」である可能性がある。

この点は対策C+Dで不正instanceの起動自体を防止することで解決されるため、対策Eの見送り判断は結論としては変わらない。

## 4. 副作用・リスク（評価: 問題なし）

- 対策A: 既存動作に影響しない追記のみ
- 対策B: 既存動作に影響しない追記のみ
- 対策C: CLAUDECODE変数を正当に操作する場面はない
- 対策D: builderがclaudeコマンドを正当に使う場面はない。PMのAgent tool使用はBashフックの対象外

## 5. フックスクリプトの設計（指摘事項あり）

### 指摘1（重要）: python3/perl等によるCLAUDECODE操作の迂回が対策Cの範囲外

計画では「python3からos.environ.popなどでの変更は対策Aのルールで抑止」としているが、技術的制御層でもカバーすべきである。以下のパターンもblock-agent-escape.shでブロックすることを推奨する:

- python3 -c "import os; os.environ.pop('CLAUDECODE')"
- perl -e "delete $ENV{CLAUDECODE}"
- node -e "delete process.env.CLAUDECODE"

ただし、これらを完全にブロックすることは困難であり（任意のスクリプトファイル内に記述される可能性がある）、費用対効果を考慮すると、明らかなインラインコマンドのみをブロックし、残りは対策Aに委ねるのも合理的である。計画内で「この制限を認識した上で対策Aに委ねる」という判断を明示的に記述すべき。

### 指摘2（中程度）: claudeコマンドのブロックパターンの網羅性

計画では「claudeバイナリのフルパス指定」への対処について「パターンで対応可能」としているが、具体的なパターンが記載されていない。実装時に以下のパターンも含めるべき:

- /usr/local/bin/claude 等のフルパス
- $(which claude) 等のコマンド置換
- exec claude 等のexec経由

### 指摘3（軽微）: フックスクリプトのearly return条件

block-destructive-git.shではStep 1でgitが含まれない場合にearly returnしている。block-agent-escape.shでも同様に、CLAUDECODE/claudeが含まれない場合のearly returnを実装することで、パフォーマンスへの影響を最小化すべき。これは計画には明記されていないが、実装時に対応すべき事項。

## 6. 作業の分担（指摘事項あり）

### 指摘4（重要）: builderエージェントに自身の制限を実装させることの妥当性

ステップ1-2（builder.mdとcycle-execution SKILL.mdの更新）をbuilderエージェントに担当させる計画だが、これは「自身の行動を制限するルールを自分で書かせる」ことになる。意図的に弱い制限を書いたり、抜け穴を残す可能性は低いが、原則としてセキュリティ制約は制約対象者以外が実装すべき。

ただし、現在のシステム構成ではbuilder以外にファイルを編集できるエージェントがいないため、builderを使用すること自体はやむを得ない。この場合、reviewerによるレビューで「ルールの網羅性と厳密性」を特に重点的に確認することで補完すべき。計画にはレビュー時の重点確認事項として明記すること。

### 指摘5（軽微）: ステップ3-4のbuilderにフックスクリプトのテスト方針を明示すべき

テストステップ4で「単体テスト」と書かれているが、テスト方法が具体的でない。block-destructive-git.shには専用のテストスクリプトが存在するか確認し、同様のテスト手法を採用すべき。最低限、以下のテストケースを明示すべき:

- ブロック対象: unset CLAUDECODE, export CLAUDECODE=, CLAUDECODE=foo, env -u CLAUDECODE, claude --agent, npx claude
- 非ブロック対象: echo CLAUDECODE, git commit -m "fix CLAUDECODE issue", npm run build, python3 script.py
