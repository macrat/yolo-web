## 記事設計の要件

### ターゲット読者
「AIエージェントやオーケストレーションに興味があるエンジニア」。この人たちが「自分のプロジェクトでAIエージェントを使うときにどうすればいいか」の参考になる記事にすること。

### ownerの指示（重要）
- 非常に分量の多いコンテンツになるので、複数部構成にしてもよい
- あらゆるポイントを網羅すること
- とてもノウハウが詰まった記事にすること
- 以下のすべてのトピックをカバーすること:
  1. アイデア出しやマーケティング分析の品質があまりにも低くて通常よりも遥かに長い時間がかかったこと
  2. ownerが介入し強制発想法を使ったこと
  3. ひねりの加え方まで指示しなければならなかったこと
  4. サイクルが長くなりすぎたせいでルール逸脱が多発したこと
  5. 4スキル構成が機能しなかったこと

### 記事品質の要件（blog-writing.mdから）
- 「やったことの報告」ではなく「読者が持ち帰れる知識」を提供する
- 内部の作業プロセスをそのまま記事の骨格にしない
- 冒頭の約束を必ず回収する
- 読者が興味を持つ情報を上の方に配置する

### 設計すべき内容
1. 記事の構成を設計する（複数部構成にする場合は各部のテーマと分割の論理を明示）
2. 各部の見出し構成（H2/H3レベル）
3. 各セクションで使うべき具体的なエピソード・データ・数字
4. 各セクションの想定文字数
5. 読者が持ち帰れる学びを各部ごとに明示
6. シリーズ名（ai-agent-opsシリーズの続きとする）
7. 各部のslug、title、description、tags、category

### 注意事項
- 既存の不正確な記事（ai-agent-site-strategy-formulation）は削除予定。その内容の不正確な点（調査1参照）を繰り返さないこと
- 時系列で書くのではなく、テーマ別に構成すること（ただし時系列の補助として参照用タイムラインを記事内に含めてもよい）
- 具体的な事例と一般化可能な教訓を必ずセットで書く
- ownerの判断や指示を正確に記述する（推測で書かない）
- メモの内容に基づいて書く（調査メモのエピソードを活用）
