# 調査依頼: ローカルEmbeddingモデルによる漢字の意味的分類

## 背景
漢字カナル（漢字推測ゲーム）のカテゴリ分類で、214部首を手動で20グループに割り当てるアプローチが計画されていたが、Ownerから「80字の漢字分類に失敗したのに214部首の分類が上手くいくとは思えない」という指摘を受けた。

代替案として、Node.jsで使えるローカルEmbeddingモデルを使って漢字の意味的な類似度を計算するアプローチが提案された（メモ 19cd6c4e27f）。

## 調査してほしいこと

### 1. Node.jsで利用可能なローカルEmbeddingモデルの調査
参考: https://zenn.dev/sirasagi62/articles/a75d0ba39f0125

以下を調査してください:
- Node.jsで動作するローカルEmbeddingモデル（Transformers.js、ONNX Runtime等）
- 日本語に対応しているモデル（漢字の意味をembeddingできること）
- モデルサイズ、推論速度、精度の比較
- インストール・利用方法

### 2. 漢字カナルへの適用方法の検討
現在のゲーム構造:
- 2,136字の常用漢字
- 各漢字に `category` フィールド（1〜20のグループ番号）がある
- `engine.ts` の `evaluateCategory()` が一致/近い/遠いを判定
- 「近い」判定は `categorySuperGroups`（グループのペア）で行っている

Embeddingを使う場合の検討事項:
- **ビルド時にEmbeddingを計算**し、結果をJSONに格納するアプローチ
- カテゴリの「離散グループ」の代わりに「連続的な距離」を使えるか
- フィードバックの「一致/近い/遠い」をコサイン類似度の閾値で判定する方法
- バンドルサイズへの影響（Embedding自体は不要、類似度計算のみクライアントで行う）

### 3. 具体的な実装パターンの検討
以下のようなパターンを検討してください:

#### パターンA: Embeddingベクトルをデータに格納
- 各漢字のEmbeddingベクトル（次元数による）をkanji-data.jsonに格納
- クライアントでコサイン類似度を計算してフィードバック
- 懸念: バンドルサイズ（2,136字 × N次元のfloat配列）

#### パターンB: ビルド時にクラスタリングしてグループ化
- Embeddingベクトルをk-means等でクラスタリングし、離散グループを割り当て
- 現在のcategory構造をそのまま使える
- 「近い」判定はクラスタ間距離で

#### パターンC: ビルド時に類似度マトリクスを生成
- 全漢字ペアの類似度を事前計算してコンパクトに格納
- 懸念: 2136^2 = 456万ペアの格納サイズ

### 4. 何をembeddingするか
漢字の「何を」embeddingするかが重要:
- 漢字1文字だけ（例: "海"）
- 漢字+意味の説明（例: "海 - 広い水域、海洋"）
- 漢字+部首+読み+意味の複合テキスト
- 各漢字のKANJIDIC2のmeaningsフィールド

### 5. 技術制約の確認
- docs/coding-rules.md を読んで技術制約を確認すること
- ビルド時のモデル利用は禁止されていない（ランタイムの外部API呼び出しが禁止）
- ローカルモデルはビルド時に実行し、結果のみをデータとして格納する
- クライアントでの推論は不要（距離計算のみ）
