## 指摘事項

### 指摘1（重要）: 前のブランチに戻る操作がテストケースに含まれていない

`-` を引数とするcheckoutは「直前にいたブランチに切り替える」正当な操作で、ファイル復元ではない。パス区切りもファイル拡張子も含まないため方式Aではブロックされない見込みだが、テストケースに明示的に含めて確認すべき。

対応: テストケース（ブロックされてはいけないコマンド）に追加。

### 指摘2（重要）: ディレクトリパス（拡張子なしだがスラッシュを含む）の扱いが未定義

方式Aの検知条件は「引数にパス区切りを含む」をOR条件としている。`src/` のようなディレクトリ全体の復元は非常に破壊的だが、これがブロックされるかどうかテストケースで明示されていない。ブロックされるべきであり、テストケースに追加すべき。

対応: テストケース（ブロックされるべきコマンド）にディレクトリパスのcheckoutを追加。

### 指摘3（重要）: サニタイズ処理の実装難易度とテスト不足

計画はサニタイズの「方針」を示しているが、bashでの具体的な実装方法（特にヒアドキュメントの除去）は非自明である。以下の点を確認すべき:

- ヒアドキュメントの区切り文字はEOF以外にも任意の文字列が使える（MEMO_BODY, HEREDOC_END等）。これらを正しく除去できるか。
- タブインデントを許容するヒアドキュメントへの対応。
- ダブルクォート内のエスケープへの対応。

テストケース28と29は具体的なコマンド文字列が示されていない。builderが正しいテストを書けるよう、具体例を計画に記載すべき。

対応: テストケース28, 29の具体的なコマンド文字列を計画に追記する。

### 指摘4（中）: パターンにgitプレフィックスが含まれていない点の曖昧さ

現在のフックは各パターンに git プレフィックスを付けて検査している。計画の検知パターン網羅表ではプレフィックスがない。サニタイズ後にgitコマンドの判定をどう行うか（現行のgitチェックを維持するのか、サニタイズ後の文字列に対して行うのか）が明確でない。

ここが曖昧だと、builderが実装時に判断を誤るリスクがある。

対応: 計画の擬似コードまたは設計方針セクションに、gitコマンドの判定ロジック（サニタイズ前にgitを含むか確認、サニタイズ、パターン検査の順序）を明記する。

### 指摘5（中）: force-with-leaseの扱い

計画のパターン10は push の後に -f や --force を含むコマンドにマッチする。しかし --force-with-lease もマッチしてしまう。--force-with-lease は --force より安全な操作（リモートが更新されていないことを確認してからpushする）であり、ブロックすべきかは判断が分かれる。

現在のプロジェクトではmainブランチのみの運用であるため実害は少ないが、設計上の意図を計画に明記すべき。

対応: force-with-leaseをブロック対象に含めるか除外するかを明記する。

### 指摘6（低）: コミット参照付きファイル復元のパターン

特定コミットからファイルを復元する操作（コミット参照、--, ファイルパスの順で引数を指定）は破壊的。パターン3の正規表現はcheckout直後に--がない場合にはマッチしない可能性がある。

ただし、パターン5（ファイルパス検知）で拡張子を検知するため、結果的にはブロックされる。実害はないが、テストケースに入れておくと設計の堅牢性が確認できる。

対応: テストケース（ブロックされるべきコマンド）への追加を検討。
