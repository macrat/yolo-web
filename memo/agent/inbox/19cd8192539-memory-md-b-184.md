## Workflow Lessons

- **キャリーオーバーを最小限に**: タスクを後回しにせず即座に対応すると周辺の問題も発見できる。
- **レビューは価値重視**: 数字やURLの正確性だけでなく、全体としての価値を最優先でレビューする。
- **上流意思決定でのバイアス防止**: サイトコンセプト等の上流決定時、既存のターゲットユーザー定義をsub-agentに見せない。既存定義に引きずられて判断が歪む。ターゲットはコンセプト決定後に定義する。（owner指摘 19cb6756629）
- **技術制約は一次ソースを参照させる**: sub-agentへの依頼で技術制約を自分で要約すると誤りが混入する。coding-rules.mdを直接読ませること。（事故報告 19cb68e8af2）
- **「あえて言わない」バイアス防止**: 「XXXを禁止する」と「XXXは禁止されていない」はどちらもXXXに注意を向けるバイアスになる。バイアスを排除するには、その話題自体に言及しないのが正解。異なるバイアスをかけた複数版を比較して公平性を担保する手法も有効。（owner指摘 19cb6fee624）
- **ゼロベース検討は最後まで一貫**: Phase A〜Eでバイアス排除しても、Phase Fで「既存との整合性」を入れるとバイアス再混入。構想フェーズでは既存文書（site-concept.md, targets/等）をsub-agentに見せない。比較は構想完了後に別フェーズで行う。（owner停止指示、記録 19cb7a91599）
- **過修正バイアスの禁止**: Ownerから「Aの方向に偏っている」と指摘された場合、対応は「Aへの偏りを取り除きニュートラルに戻す」こと。「Bの方向に積極的に振る」のは別のバイアスの注入。「不当な除外をやめよ」→「除外条件を修正する」が正解、「積極的に推進する」は誤り。（事故報告 19cbc29d1ac）
- **依頼メモ作成前のバイアスチェック**: (1)特定方向を「積極的に検討せよ」と指示していないか (2)特定方向を不当に除外していないか (3)「あえて言わない」原則に反する記述がないか (4)Ownerの指摘を「推進」ではなく「公平化」として正しく解釈しているか (5)過去のバイアス教訓を参照したか
- **「Owner指示」ラベルの厳格な使用**: メモに「Owner指示」と書く場合は、Ownerの実際の発言（メモID・会話の引用）に限定する。レビュアーやresearcherの分析を「Owner指示」とラベリングしてはならない。他agentの分析は「レビュアー評価」「researcher分析」、自分の判断は「PM判断」と正確に出所を記載する。（事故報告 19cbc3d0220）
- **根拠の正確な出所記載**: 技術的な判断（例: 削除時に410を返す）の根拠は、「Ownerがそう言ったから」ではなく、技術的根拠（HTTP規約、Google推奨等）とそのリサーチメモIDを記載する。Ownerの発言がきっかけであっても、根拠自体が技術的事実なら技術的根拠を示す。権威への依存ではなく事実に基づく判断を行う。（参考: リサーチメモ 19cb66139f3）
- **タスク粒度の厳守**: 大量のコンテンツ調査を単一エージェントに集約しない。LLMはコンテキスト圧迫で急激に性能劣化する。1メモ1タスク1エージェントのルールを厳守し、カテゴリ・件数で細かく分割する。（Owner指摘）
- **不要な否定セクションは過修正**: Ownerの指摘で「Xは原因ではない」と分かった場合、記事内でわざわざ「Xは本質ではない」と否定するセクションを設けるのは過修正。本質を説明すれば読者は他の原因を推定しない。不要な話題を持ち込むと唐突感や繋がりの無さの原因になる。「あえて言わない」原則の応用。（Owner指摘 cycle-69）
- **外部リソース依存タスクの事前確認**: メールアドレス、外部サービスURL、APIキーなど、コードベース外のリソースに依存するタスクは、builder起動前にOwnerへの確認メモを作成し返答を得てから作業開始。「Owner確認済み」と記載する場合はOwnerの実際の発言メモIDを必ず併記。（事故報告 19cc883b9e8）
- **Owner発言は示唆であり命令ではない**: Ownerの指摘をそのままコピーするのは責務の放棄。PMの責務は、Researcherに調査させ、Plannerに検討させ、総合的に判断すること。Owner発言は示唆やアドバイスであり、検証した上で最終判断する。（Owner指摘 cycle-75）
- **git checkoutを編集取り消しに使わない**: 変更の一部を取り消したい場合はEditツールで該当箇所のみ修正する。git checkoutはファイル全体を最終コミット状態に戻すため、未コミットの他の変更も消失する。（事故報告 19cd01524c7、cycle-51/71/75で3回発生）
- **中間コミットの実施**: builder作業完了後やドキュメント更新後は `wip:` プレフィックスで中間コミットを行い、未コミット作業の蓄積を防ぐ。（事故報告 19cd01524c7）
