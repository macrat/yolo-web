# B-183 実施計画（修正版）: block-destructive-git.sh フック強化

## 設計方針

### アーキテクチャ変更: コマンドのサニタイズを先に行う

現在の問題: COMMAND 全体に grep をかけるため、ヒアドキュメントやecho の引数に破壊的パターンの文字列が含まれると誤発火する（計画メモ・レビューメモの作成時にも実際に発生した）。

改修方針:
1. COMMAND 文字列からヒアドキュメント、クォート文字列を除去（サニタイズ）
2. サニタイズ後の文字列に対してのみパターン検査を適用

具体的なサニタイズ手順:
1. ヒアドキュメント（<<EOF...EOF, <<'EOF'...EOF, <<-EOF...EOF 等、任意の区切り文字）の内容を除去
2. シングルクォートで囲まれた文字列を除去
3. ダブルクォートで囲まれた文字列を除去
4. 残った文字列に対して既存のパターン検査を適用

### 【修正: 指摘4対応】gitプレフィックスの処理順序

サニタイズとgitコマンド判定の処理順序を以下のとおり明確化する。

処理フロー（擬似コード）:

    INPUT=$(cat)
    COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command')

    # Step 1: gitコマンドを含むか判定（サニタイズ前に行う）
    # 理由: サニタイズ前の生コマンドにgitが含まれていなければ、
    # そもそもgit操作ではないので早期リターンできる
    if COMMAND does not contain "git "; then
      exit 0
    fi

    # Step 2: サニタイズ（ヒアドキュメント・クォート内容を除去）
    SANITIZED = remove_heredocs_and_quotes(COMMAND)

    # Step 3: サニタイズ済み文字列に対してパターン検査
    # 注意: パターンには git プレフィックスを含める
    #   例: SANITIZED matches "git\s+reset\s+--hard" ?
    # サニタイズ後もgitプレフィックスで検査することで、
    # echo の後ろにあったgitコマンド文字列がサニタイズで除去されていれば
    # マッチしなくなる

重要ポイント:
- Step 1は「gitコマンドを含まないものを早期スキップ」するだけの最適化
- Step 3のパターン検査は全て git プレフィックス付きで行う（現行と同じ）
- サニタイズ前の生文字列に対するパターン検査は一切行わない

### 【修正: 指摘5対応】パターン10の--force-with-leaseの扱い

判断: --force-with-leaseもブロック対象に含める。

理由:
- mainブランチのみの運用では、force pushの安全なバリアントにも正当な用途がない
- --force-with-leaseは--forceより安全とはいえ、リモートの履歴を書き換える点は同じ
- 現行の正規表現 git\s+push\s+.*--force はそのまま--force-with-leaseにもマッチする。意図的にこのマッチを維持する
- 将来ブランチ運用に移行して--force-with-leaseが必要になった場合は、その時点でパターンを --force(\s or $) に変更して--force-with-leaseを除外すればよい

正規表現（修正版）: git\s+push\s+.*(-f(\s or $) or --force)
- -f は単独フラグとして出現する場合のみマッチ（-fd等の誤マッチ防止）
- --force は --force-with-lease を含めて全てマッチ

### #5 の検知ロジック（checkout ファイルパス --なし）

方式A（パス特徴ベース）を採用する。

検知条件（OR）:
- 引数にパス区切り / を含む（例: checkout docs/cycles/cycle-75.md）
- 引数にファイル拡張子を含む（.md, .ts, .tsx, .js, .jsx, .json, .yaml, .yml, .sh, .css, .html, .svg, .png）
- 引数が . である（既存 #4 と統合）

除外条件（これらはブロックしない）:
- -b または -B フラグが存在する（ブランチ作成）
- -- が引数の前にある（既に #3 で検知済み）

## リスクと注意事項

- feature/xxx 形式のブランチ名: / を含むため検知条件に合致するが、-b フラグ付きのブランチ作成はブロックされない。既存の feature/xxx ブランチへの切り替え（checkout feature/xxx）はブロックされる可能性がある。現在は main ブランチのみで運用しているため実害なし。将来ブランチ運用に移行する場合は方式B（ls-files確認）への切り替えを検討する。
- サニタイズの限界: ネストした $() 内のコマンドやバッククォート内のコマンドは完全にはサニタイズできない。実用上はヒアドキュメントとクォートの除去で十分。
- 新しい破壊的コマンドの追加漏れ: worktree remove 等の破壊的コマンドは現時点では非対象。必要に応じて追加する。
