# B-183 実施計画（修正版v2）: block-destructive-git.sh フック強化

修正版v1からの主要変更: サニタイズ設計を根本的に見直し、クォート付きファイルパスのすり抜け問題を解決。

---

## 設計方針

### サニタイズ設計の根本的見直し

**v1の問題**: クォート内文字列を「除去」するため、git checkout "src/" の "src/" も消えてパス検知をすり抜ける。

**v2の解決策: 2段階サニタイズ**

サニタイズの目的を明確に2つに分離する:

1. **非gitコマンドの除外**（目的: 誤発火防止）
   - echo "git reset --hard" や ヒアドキュメント内の git コマンド文字列による誤発火を防ぐ
   - 方法: コマンドの「実行部分」だけを抽出し、文字列リテラル部分を無視する

2. **gitコマンド引数のクォート剥がし**（目的: パターン検査の正常動作）
   - git checkout "src/" のクォートを剥がして git checkout src/ としてからパターン検査する
   - 内容は保持し、クォート文字だけを除去する

### 処理フロー（擬似コード）

    INPUT=$(cat)
    COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command')

    # Step 1: 早期リターン - コマンドに "git " が含まれなければスキップ
    if COMMAND does not contain "git "; then
      exit 0
    fi

    # Step 2: ヒアドキュメントの内容を除去
    #   <<EOF ... EOF, <<'EOF' ... EOF, <<-EOF ... EOF 等を除去
    #   目的: ヒアドキュメント内の git コマンド文字列を無視する
    CLEANED = remove_heredoc_bodies(COMMAND)

    # Step 3: 各コマンド（; や && や || で区切られた単位）を分離して処理
    #   パイプ（|）で繋がれた部分も個別コマンドとして扱う
    for each SUBCMD in split_by_separators(CLEANED):
      
      # Step 3a: サブコマンドの先頭が git かどうかを判定
      #   先頭の空白を除去し、変数代入(VAR=val)があればスキップして
      #   最初のワードが "git" かを確認
      if first_command_word(SUBCMD) != "git"; then
        continue  # git以外のサブコマンドはスキップ（echoなど）
      fi
      
      # Step 3b: gitサブコマンドからクォート文字を除去（内容は保持）
      #   "src/" -> src/  、 'page.tsx' -> page.tsx
      UNQUOTED = strip_quote_chars(SUBCMD)
      
      # Step 3c: UNQUOTEDに対してパターン検査を適用
      check_destructive_patterns(UNQUOTED)

    done

### v1との根本的な違い

| 観点 | v1 | v2 |
|------|----|----|
| クォート処理 | 内容ごと除去 | クォート文字のみ除去（内容保持） |
| 適用範囲 | COMMAND全体に一律適用 | サブコマンド単位で判定 |
| echo対策 | クォート内容除去で対応 | コマンド先頭がgitかで判定 |
| ヒアドキュメント対策 | 内容除去（v1と同じ） | 内容除去（v1と同じ） |
| git checkout "src/" | すり抜ける | ブロックされる |

### なぜこの方式が安全か

- echoやcatの引数にgitコマンド文字列が含まれていても、先頭コマンドがgitでないためスキップされる（誤発火なし）
- git checkout "src/" は先頭がgitなのでクォートを剥がして git checkout src/ として検査（正しくブロック）
- git commit -m "checkout -- src/" は先頭がgitだがcommitサブコマンド（パターンにマッチしない、ブロックされない）
- ヒアドキュメントは事前に除去されるため、中に何が書かれていても影響なし

### #5 の検知ロジック（checkout ファイルパス --なし）

方式A（パス特徴ベース）を採用（v1と同じ）。

検知条件（OR）:
- 引数にパス区切り / を含む（例: checkout docs/cycles/cycle-75.md）
- 引数にファイル拡張子を含む（.md, .ts, .tsx, .js, .jsx, .json, .yaml, .yml, .sh, .css, .html, .svg, .png）
- 引数が . である（既存 #4 と統合）

除外条件（これらはブロックしない）:
- -b または -B フラグが存在する（ブランチ作成）
- -- が引数の前にある（既に #3 で検知済み）

## リスクと注意事項

- **feature/xxx ブランチ**: / を含むため検知条件に合致するが、-b フラグ付きのブランチ作成はブロックされない。既存の feature/xxx への切り替えはブロックされる可能性がある。現在は main ブランチのみの運用のため実害なし。
- **コマンド分割の限界**: 複雑にネストした $() やバッククォートは完全には分割できない。実用上はヒアドキュメント除去 + セパレータ分割 + 先頭コマンドワード判定で十分。
- **git commit -m の扱い**: コミットメッセージ内に破壊的パターン文字列が含まれていても、サブコマンドの先頭は git commit なのでパターン検査にはマッチしない（テストケース42で確認）。
- **git log -- path の扱い**: logは非破壊操作なのでブロック対象パターンにマッチしない（テストケース43で確認）。
