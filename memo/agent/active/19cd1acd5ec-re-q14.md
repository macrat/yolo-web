# Q14 理系思考タイプ診断 実施計画

## 1. 概要

### 誰のために
「手軽で面白い占い・診断を楽しみたい人」。理系の専門知識は不要で、日常的な行動や好みの質問から「自分の理系脳の形」を可視化する体験を提供する。

### 提供する価値
- **自己発見**: 「自分の理系脳は実はこういう形だったのか」という気づき
- **可視化体験**: レーダーチャートによる5軸の「思考プロフィール」表示。既存の二元論的「理系/文系」診断にはない独自の体験
- **シェア動機**: タイプ名（科学者名）と5軸スコアの組み合わせが「友達に言いたくなる」内容

### 差別化ポイント
既存の「理系/文系」診断はすべて一軸判定。本コンテンツは5軸レーダーチャートによる多次元可視化で、「理系脳の形」という新しい切り口を提供する。

## 4. 結果タイプ一覧

### タイプ名の判断: 科学者名を採用

科学者名をタイプ名に使う。理由:
- 「アインシュタイン型だった!」はSNSシェアの動機として「理論分析型だった!」より圧倒的に強い
- 著名科学者は日本の一般層にも知名度がある人物を厳選
- 診断結果は「あくまでエンターテインメントです」とtrustNoteで明示するため、実在人物とのギャップ問題は管理可能

### 結果タイプの説明文（各300〜500字）

各タイプの説明文は以下の構成で書く:
1. 冒頭: 「あなたは〇〇型思考者」で始める
2. 特徴説明: そのタイプの思考スタイルと日常での表れ方（共感を誘う具体例）
3. 科学者エピソード: 名前の由来となった科学者の面白いエピソード（「この人もこういう思考だった」と親近感を持たせる）
4. 自虐ポイント: タイプの弱点を軽くユーモラスに触れる（共感と笑い）
5. アドバイス: 一言メッセージ

具体的な説明文の全文は、builderが上記構成に従って作成する。

### タイプ決定ロジック

1. 全20問の回答から5軸それぞれのスコアを合算
2. 最高スコアの軸を特定
3. 最高スコアの軸を除いた中で第2スコアの軸を特定
4. (最高軸, 第2軸)の組み合わせで10タイプの中からマッチするものを選択
5. 10タイプに該当しない組み合わせの場合は、最高軸のみで最も近いタイプにフォールバック

フォールバック表:
- theory最高 → 該当なし時 einstein
- empirical最高 → 該当なし時 curie
- quantitative最高 → 該当なし時 turing
- observational最高 → 該当なし時 darwin
- creative最高 → 該当なし時 davinci

## 5. 質問20問（5軸 x 4問）

### 設計原則
- 理系の知識を問わない。日常の行動・好み・考え方で判定
- 各質問は4択。各選択肢は「主に加点する軸」(+3) と「副次的に加点する軸」(+1) を持つ
- 1問の4択で必ず4つ以上の軸をカバーし、偏りを防ぐ
- 1つの軸は20問中16回以上の選択肢に登場する（4問x4選択肢=16が最低ライン）

### ポイント配分の方針
- 各選択肢: メイン軸 +3, サブ軸 +1
- 全20問で各軸の最大獲得可能ポイント: 理論上は4問x3 + α = 12〜20程度
- スコア範囲は0〜20程度になる想定

- **theory**: Q2b, Q3d, Q4a, Q5b, Q6d, Q8b, Q10a, Q11a, Q12b, Q13c, Q15a, Q16a, Q17c, Q20a → ただし1問あたり1選択肢のみカウント。正確には:
  - Q1a, Q2b, Q3d, Q4a, Q5b, Q6d, Q8b, Q9d, Q10a, Q11a, Q12b, Q13c, Q15a, Q16a, Q17c, Q20a
  - 各問で theory+3 の選択肢は1つなので、20問中16問に登場... これは多すぎる。

修正: 上記の表を確認すると、各問で各軸がメイン(+3)として登場するのは原則1つの選択肢のみ。20問 x 4選択肢 = 80選択肢、5軸でメイン配分するので各軸は80/5 = 16選択肢がメイン...ではなくメインは20問x4=80のうち各選択肢に1つのメインなので80メイン枠を5軸で分けると16回ずつ。これは正しい。各軸は20問中16回メイン(+3)として出現する。

しかし1問4択で4軸しかメインにできないため、各問で1軸はメインに登場しない。20問で各軸がメインから外れるのは4回。5軸x4問=20で、各軸は4問でメインに選ばれない。これにより各軸のメイン出現は16回(+3) + サブでの出現。

各軸の最大理論値:
- メイン(+3)が4問で確定: 4 x 3 = 12ポイント
- サブ(+1)での出現: 残り16問のうちサブとして出現する回数は設計次第だが、上記設計では各選択肢にサブが1つあるため、サブ出現も十分ある

結論: この配分で各軸のスコア範囲は十分な差異が出る。問題なし。

## 6. 技術実装計画

### 6.1 スコアリング方式

**既存の `calculatePersonalityPoints()` をそのまま利用する。**

理由: この関数は `choice.points` の全キーに対してスコアを合算する汎用的な実装。pointsのキーを軸名（theory, empirical, etc.）にすれば、各軸のスコアが `Record<string, number>` として得られる。

**ただし `determineResult()` はそのままでは使えない。** 既存のdetermineResultは「pointsのキーがresult IDと一致する前提」で最高スコアのresult IDを返す。Q14では:
1. pointsのキーは軸名（theory等）
2. result IDはタイプ名（einstein等）
3. 最高軸 + 第2軸の組み合わせでタイプを決定する

**対応方針**: 新しい関数 `determineScienceThinkingResult()` を `src/quiz/data/science-thinking.ts` 内に定義する。scoring.tsの汎用関数は変更しない。

```
関数の役割:
1. calculatePersonalityPoints() で5軸スコアを取得
2. 最高スコア軸と第2スコア軸を特定
3. (最高軸, 第2軸) → result ID のマッピングテーブルで結果を決定
4. マッピングに該当しない場合はフォールバックテーブルで決定
```

### 6.2 QuizContainer の拡張

**QuizContainerに `customDetermineResult` propを追加する方式は避ける。** 代わりに、以下のアプローチを取る:

QuizDefinitionの `results` 配列に10タイプを格納し、`points` のキーをresult IDではなく軸名にする。これは既存の型定義には矛盾しないが、`determineResult()` の動作とは合わない。

**具体的な解決策**: QuizContainerを拡張して、slug単位でカスタムのresult決定ロジックを呼べるようにする。

1. `QuizDefinition` に **オプショナルな** `determineResultCustom?: (questions: QuizQuestion[], answers: QuizAnswer[], results: QuizResult[]) => QuizResult` フィールドを追加する
2. QuizContainerの結果判定で、`quiz.determineResultCustom` があればそれを使い、なければ従来の `determineResult()` を使う
3. 既存の診断は影響を受けない（フィールドがundefined）

あるいは、よりシンプルな方法として:

**QuizContainer内で slug === "science-thinking" の場合に分岐する方式**。ResultExtraLoaderと同じパターン。ただしこれはQuizContainerに特定slugの知識を入れることになり、スケーラビリティに問題がある。

**最終判断: `determineResultCustom` フィールド方式を採用する。** 型安全で、既存コードへの影響がなく、今後の拡張にも対応できる。

### 6.3 レーダーチャートコンポーネント (SVG)

**ファイル**: `src/quiz/_components/RadarChart.tsx`

**SVG実装方針**:

1. **座標計算**: 5角形の各頂点は中心点から等間隔（72度ずつ）に配置。中心を (cx, cy)、半径を r とすると、各頂点は:
   - (cx + r * sin(i * 72deg), cy - r * cos(i * 72deg)) (i = 0..4)
   - 12時方向を0度として時計回り

2. **背景グリッド**: 5段階の同心五角形（20%, 40%, 60%, 80%, 100%）を薄い線で描画。各段階のラベル（20, 40, 60, 80, 100）は不要（見づらくなるため）

3. **データ多角形**: ユーザーのスコアを正規化（各軸の最大値を100%とする）し、5頂点の座標を計算。`<polygon>` でエリアを塗りつぶし（半透明）、`<polyline>` で外枠

4. **軸ラベル**: 各頂点の外側に軸名を日本語で表示。`<text>` 要素でテキストアンカーを調整:
   - 上: text-anchor="middle"
   - 右: text-anchor="start"
   - 左: text-anchor="end"
   - 下: text-anchor="middle"

5. **スコア表示**: 各頂点付近にスコア値（パーセント）を小さく表示

6. **レスポンシブ**: viewBoxベースで実装し、containerの幅に合わせてスケール

7. **Props設計**:
```typescript
interface RadarChartProps {
  /** 軸名とスコアのペア配列（表示順） */
  axes: Array<{ label: string; value: number; max: number }>;
  /** テーマカラー */
  color: string;
  /** コンポーネントの幅 (px, デフォルト 300) */
  size?: number;
}
```

8. **アニメーション**: 結果表示時に多角形がゼロから広がるアニメーション。CSS `@keyframes` で `<polygon>` のポイントは直接アニメーションできないため、opacity + scale のシンプルなアニメーションにする

### 6.4 結果表示のカスタマイズ

**ファイル**: `src/quiz/_components/ScienceThinkingResultExtra.tsx`

ResultExtraLoaderに追加するパターンで、science-thinking固有の結果表示を実装:

1. **レーダーチャート表示**: 5軸スコアを計算し、RadarChartコンポーネントで可視化
2. **各軸のスコアバー**: 5軸それぞれの数値をプログレスバーで表示（「理論志向: 75%」のように）
3. **相性機能は入れない**: 10タイプ x 10タイプ = 55組の相性を書くのは工数対効果が低い。代わりに「友達を誘うボタン」のみ設置し、友達のタイプと自分のタイプの比較（レーダーチャート重ね合わせ）を将来的な拡張として残す

### 6.5 OGP画像

**ファイル**: 既存の `src/app/quiz/[slug]/result/[resultId]/opengraph-image.tsx` がすべてのクイズの結果ページOGPを生成している。

Q14の場合もこの汎用OGP画像がそのまま使える（アイコン + タイプ名 + クイズタイトル）。レーダーチャート入りのOGP画像は next/og の ImageResponse で SVG を直接レンダリングするのが難しいため、初回リリースでは見送り、テキストベースのOGPで十分と判断する。
