## 1. コンテンツ設計

### 基本仕様
- **スラグ**: `animal-personality`
- **タイプ**: `personality`（診断カテゴリ）
- **タイプ数**: 12タイプ
- **質問数**: 10問（体験時間2〜3分）
- **結果文字数**: 300〜500文字

### スコアリング方式

**4軸モデル（各2値）で16の理論組み合わせから12タイプを採用する方式**:
- **S軸 (Social)**: 群れ型 (G) vs 単独型 (S) -- 社交性・集団志向
- **A軸 (Action)**: 行動型 (A) vs 慎重型 (C) -- 判断の速さ・リスク許容度
- **E軸 (Energy)**: 活動型 (H) vs 穏和型 (L) -- エネルギーの方向性
- **T軸 (Think)**: 直感型 (I) vs 分析型 (N) -- 物事の捉え方

各質問の選択肢は4つ。各選択肢がprimary（2pt）を1タイプに、secondary（1pt）を1タイプに付与するパターン（music-personalityと同じ構造）。

**配点設計（計算修正済み）**:
- 10問 x 4選択肢 = 40のprimaryスロット
- 40スロットを12タイプに配分 → 各タイプにprimary 3回または4回（8タイプが3回、4タイプが4回、合計8x3+4x4=40）
- secondaryも同様に各タイプ3〜4回を配分
- music-personalityの参考: 8タイプ x 5 primary = 40で均等。本診断は12タイプのため均等にならないが、差は最大1回（3回 vs 4回）に留める

**未使用4パターンへの対処方針**:
16の軸組み合わせから12を採用するため、4つの組み合わせが未使用になる。回答者のスコアがこの未使用パターンに近い場合でも、最高得点のタイプに割り当てられるためタイプ自体は必ず決まる。ただし納得感を担保するため:
- builderは配点設計時に、未使用4パターンそれぞれが「どの採用タイプに近いか」を意識し、近接するタイプとの性格描写に共通要素を持たせること
- 結果文において軸の組み合わせを前面に出さず、「この動物に似たあなたの性格」という自然な描写にすることで、軸の完全一致を前提としない表現にする
- 実装上は単純に合計スコア最高のタイプを採用する（軸ベースのバイナリ判定ではなく、スコアベースの判定）

### 結果文の構成（各タイプ300〜500文字）
1. **動物名とキャッチコピー**（1行）: 例「ニホンモモンガ -- 空飛ぶハンカチの大冒険家」
2. **生態事実に基づく性格描写**（3〜4文）: 動物の実際の行動から性格を類推する面白さ
3. **文化イメージとのギャップ**（1〜2文）: 文化的に知られるイメージと実態の落差をユーモアにする
4. **自己発見の一言**（1〜2文）: 「あなたも〜なところがありませんか?」的な問いかけ
5. **ユーモラスなアドバイス**（1文）: 動物の生態にちなんだ斜め上のアドバイス
6. **保護情報（該当種のみ）**: 特別天然記念物・絶滅危惧種の場合は1文で触れる（constitution.md Rule 1・2との整合）。説教臭くならず「あなたと同じくらい貴重な存在です」的なユーモアで触れる

**結果文での事実正確性に関するルール（レビュー指摘対応）**:
- キャッチコピーにはエンタメ的誇張を許容する（「走れないウサギ」「温泉を発明した革命児」等）
- ただし結果文本文では事実に基づいた記述にする。例:
  - ニホンザル: 「温泉を発明した」→ 結果文では「餌付けされていた子猿が偶然温泉に入ったのが始まり」等、事実を伝える
  - アマミノクロウサギ: 「走れない」→ 「後ろ足が短いため素早く走ることが苦手」と正確に記述

### SNSシェアテキスト
- Xシェア: 「あなたの日本の動物性格は「{動物名}」タイプ! {キャッチコピー} #日本の動物性格診断」
- LINEシェア: 「日本の動物性格診断で「{動物名}」タイプだった! あなたは何の動物?」

## 3. 質問設計の方針

### 質問の方向性
- **日常のシチュエーション問題**を中心とする
- 4軸のうち各質問で2〜3軸を測定する（1問1軸だと質問数が足りない）
- 質問文は「あなたならどうする?」形式

### 質問テーマ案（10問）
builderが具体的な質問文と配点を設計する。
1. **集団行動vs単独行動**: 「休日の過ごし方は?」（S軸）
2. **判断スピード**: 「急に予定が変わったら?」（A軸）
3. **リスク許容度**: 「知らない道と慣れた道、どちらを選ぶ?」（A軸+T軸）
4. **エネルギー方向**: 「朝型か夜型か」（E軸）
5. **直感vs分析**: 「買い物の決め手は?」（T軸）
6. **ストレス対処**: 「困った時にまず何をする?」（S軸+A軸）
7. **コミュニケーション**: 「友達との連絡頻度は?」（S軸+E軸）
8. **目標設定**: 「計画を立てる派vs流れに任せる派」（A軸+T軸）
9. **変化への態度**: 「新しい環境に飛び込むか、慣れた環境を好むか」（A軸+E軸）
10. **自己表現**: 「自分の気持ちをどう伝える?」（S軸+T軸）

### 配点設計の原則（修正済み）
- 各選択肢にprimary（2pt）1タイプ + secondary（1pt）1タイプを設定
- 10問 x 4選択肢 = 40スロットのprimaryを12タイプに配分: **各タイプ3回または4回**（8タイプが3回、4タイプが4回で合計40）
- secondaryも同様に各タイプ3〜4回を配分
- ファイルヘッダコメントにmusic-personalityと同形式で配分表を記載すること（例: nihon-zaru: Q1a, Q5c, Q8b (3)）
- 特定タイプに偏らないように、primary 4回のタイプとsecondary 4回のタイプが重複しないよう調整する

## 4. 相性機能

### 相性マトリクスの設計
- 12タイプの全組み合わせ: 12*(12+1)/2 = 78組（同タイプ12 + 異タイプ66）
- キー形式: `"amami-kuro-usagi--ezo-shika"` （アルファベット順ソートで結合）

### 相性エントリの構造（CompatibilityEntry型を再利用）
```
{
  label: "相性ラベル"（例: "温泉仲間の絆"）,
  description: "相性説明文"（80〜150文字。動物の生態に基づくユーモラスな描写）
}
```

### 相性文の方針
- 動物同士の生態的な関係性（生息域の重複、食物連鎖上の関係等）をベースにユーモアを加える
- 同タイプ同士: 「同じ種の2匹が出会ったら」的な想像
- 異タイプ同士: 生態の違いから生まれる面白い関係性を描写

### 実装パターン
music-personalityと同じパターンを踏襲:
- `getCompatibility(typeA, typeB)` 関数
- `isValidAnimalTypeId(id)` バリデーション関数
- `ANIMAL_TYPE_IDS` 定数配列
