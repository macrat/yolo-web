# Q14 理系思考タイプ診断 実施計画レビュー結果

## 総合評価: B

全体として差別化ポイント（5軸レーダーチャート x 科学者タイプ名）は明確で、ユーザー価値の設計は優れている。ただし、ポイント配分に重大な偏りがあり、一部の科学者の知名度、フォールバック設計の不自然さに問題がある。以下の修正を行えばA評価に到達可能。

---

## 1. ユーザー価値（良好）

### 良い点
- 「理系/文系」の二元論から5軸多次元可視化への転換は、既存市場に対する明確な差別化。「理系脳の形」というコンセプトは秀逸。
- 科学者名によるタイプ名はSNSシェア動機として非常に強い。「アインシュタイン型だった!」は投稿したくなる。
- レーダーチャートの可視化体験は、テキストだけの結果よりも満足感が高い。

### 懸念点
- **20問は上限ギリギリ**: ターゲットユーザーの期待は「1回2〜5分」。20問x15〜20秒=5〜7分で上限超過の可能性がある。researcherが引用したUXデータでも16問以上は完了率25〜45%と低い。計画書では「やりごたえ感」との妥協点としているが、離脱リスクは明示的に認識しておくべき。15問（5軸x3問）も選択肢として検討した痕跡がないのが気になる。ただし、1問1画面+進捗バーの設計で体感時間を短くする方針は適切なので、20問でも許容範囲内と判断する。

## 2. コンテンツの質

### 質問の質（良好）
- 20問すべてが日常的なシチュエーションに基づいており、理系知識を前提としていない。「旅行計画」「料理」「道に迷ったとき」等、誰でも想像できる場面設定。
- 4選択肢はいずれも「これ自分だ」と思える自然な回答になっている。
- Q9（植物を育てる）とQ17（健康管理）はやや「几帳面な人」にしか刺さらない可能性があるが、許容範囲。

### 科学者名の知名度問題（要修正）

| 科学者 | 日本での一般知名度 | 判定 |
|--------|-------------------|------|
| アインシュタイン | 非常に高い | OK |
| キュリー | 高い | OK |
| チューリング | 中程度（映画「イミテーション・ゲーム」で上昇） | OK |
| ダ・ヴィンチ | 非常に高い | OK |

| エジソン | 非常に高い | OK |

| ファラデー | 中程度（中学理科で学ぶが一般層には薄い） | 要注意 |

**フンボルト型は要差し替え**。フンボルトは「忘れられた科学の巨人」と評されるほど一般知名度が低い。診断結果で「フンボルト型です」と出ても「誰?」となり、シェア動機が大幅に低下する。同じ「観察+実験」の特性なら、例えば「ファーブル型」（昆虫記で日本の一般層にも知名度が高い）が適切。

**ナイチンゲール型**は「データ可視化の先駆者」という史実は正確だが、日本の一般ユーザーの認識は「看護師」「白衣の天使」。結果表示で「統計学の母」としてのエピソードを丁寧に紹介すれば逆に「へえ、そうだったんだ」という発見があり、むしろプラスに転じ得る。ただし説明文でこの認識ギャップを積極的にフォローすることをbuilder指示に明記すべき。

## 3. ポイント配分の健全性（要修正 -- 重大）

### 軸ごとのメイン(+3)出現回数

計画書内のポイント配分を全20問について精査した結果:

| 軸 | メイン(+3)出現回数 | 理想値(80/5) | 偏差 |
|----|-------------------|-------------|------|
| theory | 17 | 16 | +1 |
| **quantitative** | **19** | 16 | **+3** |
| **empirical** | **13** | 16 | **-3** |
| observational | 16 | 16 | 0 |
| creative | 15 | 16 | -1 |

**quantitativeが19回、empiricalが13回で、6回の差がある。これは重大な偏り。**

具体的に、empiricalがメイン選択肢に登場しない問が7問もある（Q3, Q10, Q12, Q13, Q15, Q17, Q20）。一方quantitativeが不在なのはQ16の1問のみ。

**影響**:
- quantitativeが最高軸になりやすい → turing型、nightingale型に偏る
- empiricalが最高軸になりにくい → curie型、faraday型に到達困難
- 「実験・検証好き」のユーザーが正しくempiricalに分類されない可能性がある

**修正指示**: Q10, Q12, Q13, Q15, Q17, Q20のうち少なくとも3問で、quantitativeの選択肢をempiricalに差し替え、各軸の出現が15〜17回の範囲に収まるように再調整すること。

### 計画書内の自己検証が不正確

計画書セクション5の「ポイント配分の検証」は、カウントの途中で混乱しており、最終的に「各軸は16回メイン出現、問題なし」と結論しているが、実際には上記の通り13〜19回の幅がある。この検証は信頼できないため、builder向けに正確なカウント表を提示すべき。

## 4. タイプ決定ロジックのフォールバック（要改善）

### フォールバックの不自然さ

10タイプでカバーされていない組み合わせが10通りある。フォールバックは「最高軸のみで決定」だが、これにより不自然なマッピングが生じる:

- **creative + theory → davinci（creative + observational）にフォールバック**: 理論志向が第2位なのに、観察志向のdavinciになるのは不自然。
- **observational + creative → darwin（observational + theory）にフォールバック**: 創造志向が第2位なのに、理論志向のdarwinになるのは不自然。
- **observational + quantitative → darwin にフォールバック**: quantitativeが第2位なのにtheoryのdarwinになる。

**改善提案**: フォールバック時は「最高軸をメインに持つタイプの中で、第2軸のスコアが最も近いタイプ」を選ぶロジックにすべき。あるいは、カバー率を上げて12〜15タイプに増やすことも検討に値する（ただしcontent-categories.mdの推奨は6〜8、10は既に多い）。最低限、フォールバック表に「この組み合わせが来た場合にどのタイプになり、ユーザーにとって違和感がないか」の検証を加えること。

## 5. 技術設計（良好、一部要確認）

### determineResultCustom の設計（適切）
- QuizDefinitionにオプショナルな `determineResultCustom` を追加する方針は、既存コードへの影響がなく、型安全で拡張性がある。slug分岐よりも明らかに優れた設計。
- 既存の `determineResult()` は `points[result.id]` でスコアを引くため、Q14のように軸名とresult IDが異なる場合に使えないという分析は正確。
- **ただし注意点**: `determineResultCustom` の型定義で `QuizAnswer[]` を受け取っているが、関数内で `calculatePersonalityPoints()` を呼ぶことになる。この場合 `questions` も必要で、型定義に含まれているので問題はない。

### SVGレーダーチャート（実現可能）
- viewBoxベースのSVG実装は、外部ライブラリ不要でバンドルサイズを抑えられる。coding-rules.mdの「静的最優先、クライアント優先」方針に合致。
- 5角形の座標計算（72度ずつ）の説明は正確。
- アニメーションをCSS opacity+scaleに限定する判断も妥当（SVG polygon pointsのアニメーションはブラウザ互換性に問題がある）。

### OGP画像
- 初回リリースでレーダーチャート入りOGPを見送る判断は妥当。next/og の ImageResponseでSVGを直接扱うのは制約が多い。

### coding-rules.md準拠
- 外部API呼び出しなし: OK
- クライアントサイド実装: OK（QuizContainerは"use client"）
- 型安全: OK（TypeScript型定義が明示されている）
- マジックナンバー回避: ポイント値(+3, +1)に名前付き定数を使うことをbuilder指示に含めることを推奨

## 修正指示まとめ

### 必須修正（B→A に必要）

1. **ポイント配分の再調整**: quantitativeのメイン出現を16〜17回に減らし、empiricalを15〜16回に増やす。具体的にはQ10, Q12, Q13, Q15, Q17, Q20のうち3問以上でquantitative選択肢をempiricalに差し替える。修正後の正確なカウント表をbuilder指示に含めること。

2. **フンボルト型の差し替え**: 日本の一般層に知名度のある科学者・博物学者に変更する（例: ファーブル型）。

3. **計画書のポイント検証セクションの修正**: 不正確な自己検証を削除し、正確な軸別カウントに差し替える。

### 推奨修正

4. **フォールバックロジックの改善**: 「最高軸のみで決定」ではなく、「最高軸を持つタイプの中で第2軸のスコアに最も近いタイプを選択」するロジックに変更。少なくとも、フォールバック時のユーザー体験（不自然なタイプに振り分けられないか）の検証を追記すること。

5. **ナイチンゲール型の説明文フォロー**: builder向け指示に「看護師イメージとの認識ギャップを積極的にフォローし、データ可視化の先駆者としてのエピソードを重点的に記述すること」を明記。

6. **ポイント値の定数化**: `+3`（MAIN_AXIS_POINTS）と `+1`（SUB_AXIS_POINTS）を名前付き定数として定義するようbuilder指示に含める。
