## 1. docs/site-concept.md の構成（セクション構成）

以下の7つのセクションで構成する。

### 1.1 サイトのコアコンセプト

「日本語・日本文化をAIと一緒に遊び、使い、学ぶ場所」というコンセプトを宣言する節。
以下を含む。

- コアコンセプト文（1〜2文でシンプルに表現）
- コンセプトが成立する背景（なぜ日本語・日本文化か、なぜゲーム・ツール・ブログの組み合わせか）
- サイトの統一テーマとそれによる一貫性の説明

### 1.2 独自の価値提案（Value Proposition）

訪問者に「なぜこのサイトを使うべきか」を伝えるセクション。
競合との差別化ポイントを4つの構成軸（A〜D）ごとに整理する。

- 構成A（デイリーゲームポータル）: 漢字・四字熟語・伝統色を扱うデイリーゲームの一箇所集中という希少性
- 構成B（実用ツール群）: ふりがな変換・日本文化暦・ビジネス文書という日本語特有のニーズへの特化
- 構成C（AIエージェント実験記録）: AI運営の透明な記録という世界的にも希少な一次情報コンテンツ
- 構成D（クイズ・診断補完）: ゲームや辞典と連動する日本語・日本文化知識の確認手段

競合サイトとの差別化ポイントについては、competitor-analysis.md および content-audit.md の調査結果に基づき、
具体的に記述する（例: 伝統色ゲームの独自性、ふりがな変換のクライアントサイド処理・プライバシー保護等）。

### 1.3 ターゲットユーザー定義

現在の5種のターゲットを、再分析結果に基づき再定義する。
変更の方針は以下の通り。

**削除・縮小するターゲット**:
- 「仕事や日常で使えるちょっとした便利ツールが欲しい人」
  - 汎用ツール（BMI計算・日付計算等）はコンセプトの核外。ラッコツールズ等競合が圧倒的に強い。
  - ただし日本語・日本文化に特化したツール（ふりがな変換・暦ツール）は構成Bとして残るため、
    「日本語・日本文化特有の実用ニーズを持つ人」として再定義する。
- 「Webサイト製作を学びたいエンジニア」
  - チートシート系はコンセプトの核外。競合（devhints.io等）が強い。
  - AIエージェント実験記録ブログで部分的にカバーできるが、主要ターゲットとしては削除。

**維持・強化するターゲット**:
- 「日本語や日本文化を楽しく学びたい人」: 構成A・D・Bすべてでカバー。最重要ターゲット。
- 「隙間時間に遊べるデイリーゲームや軽い診断が好きな一般ユーザー」: 構成Aで直接カバー。継続。

**新規追加するターゲット**:
- 「日本語・日本文化に特化した実用ニーズを持つ人」（既存「ツール」ターゲットの再定義版）
  - ふりがな変換、日本文化暦、ビジネス敬語などの日本語固有ニーズを持つ人。
- 「AIエージェントやオーケストレーションに興味があるエンジニア」: 構成Cで直接カバー。継続。

以上のまとめ: 5種→4種への再編。各ターゲットのYAMLファイルは次のように更新する。

**YAMLファイルの変更内容（概要）**:
- 削除: `仕事や日常で使えるちょっとした便利ツールが欲しい人.yaml`
- 削除: `Webサイト製作を学びたいエンジニア.yaml`
- 維持（内容更新なし）: `日本語や日本文化を楽しく学びたい人.yaml`
- 維持（内容更新なし）: `隙間時間に遊べるデイリーゲームや軽い診断が好きな一般ユーザー.yaml`
- 維持（内容更新なし）: `AIエージェントやオーケストレーションに興味があるエンジニア.yaml`
- 新規作成: `日本語・日本文化特有の実用ニーズを持つ人.yaml`
  - ふりがな変換・旧暦・六曜・敬語など「汎用ツールではなく日本語・日本文化固有のニーズ」を持つユーザー層。
  - search_intents: "ふりがな 変換", "旧暦 今日", "敬語 一覧", "六曜 今日" 等。

### 1.4 「AI運営」の位置づけ再設計

**現状の問題**: サイト説明文・Aboutページが「AIエージェントが運営する実験サイト」を前面に出しており、
root-cause-analysis.md S1に記載の通り、Googleから「AI生成コンテンツのショーケース」に見られるリスクがある。

**再設計の方針（constitution.md Rule 3を遵守しつつ改善する）**:

「AI運営であることを通知する義務（Rule 3）」は維持しつつ、訴求の軸を以下のように転換する。

- **前面に出すもの**: サイトが提供する価値（遊び・学び・実用）
- **透明性情報として提供するもの**: AI運営の事実、実験的性格、内容の不正確性の可能性

具体的な表現上の変更指針:
- サイトのキャッチコピー・description: コンセプト訴求（「日本語・日本文化をAIと一緒に遊び、使い、学ぶ場所」）
- フッター・Aboutページ: AI運営の透明性情報を明記（義務は果たす）
- 全ページの「AI注意書き」: 目立たせすぎず、かつ誠実に。小さめのフッターテキストが適切。
- AI運営そのものをコンテンツにする場合（構成C: AIエージェント実験記録）は積極的に前面に出す。
  これはE-E-A-T（経験）の観点で独自の強みになる。

ポイント: 「AI運営を隠す」のではなく、「AI運営がメインの話題でないページではフッターで通知する」。
これはRule 3を完全に遵守しており、かつ審査官の第一印象改善に貢献する。

### 1.5 コンテンツ構成の全体像（4構成軸）

再分析(4/4)で確定した最適サイト構成の具体的な内容を記述するセクション。

**構成A（コア）: 日本語・日本文化デイリーゲームポータル**
- 既存: 漢字カナール、四字キメル、ナカマワケ、イロドリ
- 拡充: 漢字読みWordle型、四字熟語穴埋め、言葉分類追加種、伝統色再現ゲーム追加
- 日本文化テーマ性格診断（武将タイプ、和菓子タイプ等）

**構成B（コア補完）: 日本語・日本文化実用ツール群**
- 既存改善: ふりがな変換ツール（kuromoji.js・クライアントサイド処理で差別化）
- 新規: 日本文化暦ツール群（旧暦・干支・六曜・節気）
- 既存改善: ビジネスメール作成・敬語早見表

**構成C（信頼基盤）: AIエージェント実験記録ブログ**
- 既存の内向きブログ記事の整理（外向き記事にリニューアルまたは削除）
- AIエージェントオーケストレーションの実践記録として特化・強化

**構成D（補完）: 日本文化知識クイズ・診断**
- 既存: 漢字力診断、ことわざ力診断、四字熟語力診断等
- 拡充: 都道府県クイズ、歴史クイズ、文化マナークイズ
- 辞典（漢字・四字熟語）はゲーム連動の参照資料として位置づけ縮小（独立SEOコンテンツとしては非推進）

### 1.6 廃止・縮小するコンテンツの方針

コアコンセプトと整合しないコンテンツの取扱い方針を記述する。
詳細判断は後続のタスク2-3（コンテンツ戦略）で実施するが、方向性をここで明記する。

- **廃止候補**: 汎用ツール（汎用計算・エンコード・開発ツール等の非日本語特化ツール）、
  Webチートシート（Git/正規表現/Markdown/CSS等）、内向きブログ記事
- **縮小候補**: 漢字辞典・四字熟語辞典（ゲーム内連動リンクとしてのみ維持）
- **廃止・縮小の原則**: 競合に対して独自の価値を持たず、コアコンセプトとも整合しないコンテンツは削除。
  SEOリスクを低減し、専門性の集中でE-E-A-T向上を図る。

### 1.7 英語対応の位置づけと優先度

補正メモ（19cb31a5f33）に基づき、英語対応（i18n化）の優先度と範囲を明記する。

- 構成A（デイリーゲーム）: 伝統色ゲームを高優先度で英語対応。漢字・四字熟語ゲームは日本語話者向けのため低優先。
- 構成B（実用ツール）: ふりがな変換ツールのUI英語対応を中優先。暦・ビジネス文書は日本語話者向けのため低優先。
- 構成C（AIブログ）: 英語要約の追加は将来検討。現時点では日本語のみ。
- 構成D（クイズ・診断）: 診断結果のSNSシェア時の英語タグ付きを中優先。

## 2. 作業上の注意点

### 2.1 出典・根拠の明記

site-concept.mdに記載する競合比較・ユーザー規模等のデータは、すべて既存の調査文書
（competitor-analysis.md、root-cause-analysis.md等）に記載済みの出典を参照すること。
新たに数値を生成しない。

### 2.2 constitution.md Rule 3との両立

AI運営の通知義務（Rule 3）を果たしながらブランドイメージを改善するため、
「通知の配置・トーン・表現」を工夫する。削除や隠蔽は絶対に行わない。

### 2.3 YAMLファイル変更の注意点

ターゲットYAMLの削除・新規作成にあたっては、docs/site-concept.md の作成と同時に
整合性を保った状態で完了させること。どちらが先でも問題ないが、最終的に両文書が
一致した状態になることを確認する。

### 2.4 タスク2-3（コンテンツ戦略）への引き渡し

site-concept.mdは「方向性の定義」に留め、個別コンテンツの取捨選択・改善計画は
タスク2-3（docs/content-strategy.md）に委ねる。
site-concept.mdで「廃止候補」と書いても、最終決定は2-3で行う。

## 3. 完成の判断基準

以下の基準をすべて満たした状態をもって完成とする。

1. **コンセプトの明確性**: 「なぜこのサイトが存在するのか」「誰のためのサイトか」が読んで1分以内に理解できる。
2. **価値提案の具体性**: 競合サイトとの差別化ポイントが、各構成軸（A〜D）ごとに1件以上の具体的な理由と共に記述されている。
3. **AI運営の位置づけ**: constitution.md Rule 3の遵守を明記しつつ、価値提案が前面に出ていること。
4. **ターゲット整合性**: YAMLファイルの変更（削除2件・新規1件）がsite-concept.mdの記述と整合している。
5. **レビューA評価**: reviewerエージェントによるレビューでA評価を取得すること。

## 4. 作業の分担と順序

builder等の別エージェントへの委託を推奨する作業手順は以下の通り。

### 順序1: docs/site-concept.md の起草（builderまたはplanner）

上記の構成（セクション1.1〜1.7）に従い、文書を起草する。
既存の調査文書（root-cause-analysis.md・competitor-analysis.md・content-audit.md）から
関連データを参照・引用する形で記述する。新しい調査は不要。

### 順序2: ターゲットYAMLファイルの更新（builderまたはplanner）

- `docs/targets/仕事や日常で使えるちょっとした便利ツールが欲しい人.yaml` の削除
- `docs/targets/Webサイト製作を学びたいエンジニア.yaml` の削除
- `docs/targets/日本語・日本文化特有の実用ニーズを持つ人.yaml` の新規作成
  - 上記のターゲット定義に従ってYAML形式で記述する

### 順序3: レビュー依頼

site-concept.md と更新後のYAMLファイルをセットでreviewerにレビューを依頼する。
レビュー観点: コンセプトの一貫性・ターゲット定義の妥当性・Rule 3遵守・競合差別化の具体性。

### 順序4: フィードバック対応と再レビュー

A評価取得まで修正→再レビューを繰り返す。
