## 計画: 全4タスク（順序依存あり）

### タスク1: Embeddingベクトル生成スクリプトの作成と実行

**目的**: 2,136字すべてのEmbeddingベクトルを事前計算し、JSONファイルとして格納する。

**作業内容**:
1. scripts/generate-kanji-embeddings.ts を新規作成
2. Xenova/paraphrase-multilingual-MiniLM-L12-v2 を @huggingface/transformers 経由で利用
3. kanji-data.json から全2,136字を読み込み、パターンC（"漢字 meanings"、例: "海 sea ocean"）で入力テキストを生成
4. 384次元のfloat32ベクトルを取得
5. **128次元と384次元の両方**を生成して出力する

**128次元への次元削減方法**: 384次元ベクトルの先頭128次元を取得する（Matryoshka Representation Learningにより、先頭の次元が最も重要な情報を保持する。paraphrase-multilingual-MiniLM-L12-v2はこの特性を持つモデルである）。ただし、モデルがMatryoshka対応でない場合はPCA等の次元削減が必要になる。タスク1のbuilderは、まず先頭128次元の切り出しで試し、タスク2の品質検証結果が不十分であればPCAを検討すること。

**int8量子化方法**:
- 各次元について、ベクトル全体の最大絶対値で割って[-1,1]に正規化し、127を掛けてint8に丸める
- **クランプ処理**: 正規化後に値が[-1, 1]の範囲外に出る場合（浮動小数点精度の問題等）は、Math.max(-127, Math.min(127, value)) でクランプする
- モデル出力はL2正規化済みのため各次元は概ね[-1, 1]に収まるが、安全のためクランプは必須

**出力ファイル**: 2つ生成する
- public/data/kanji-embeddings-128.json（128次元版、約356KB raw）
- public/data/kanji-embeddings-384.json（384次元版、約1.1MB raw）

**ファイル形式**:
```json
{
  "meta": { "model": "paraphrase-multilingual-MiniLM-L12-v2", "dims": 128, "quantization": "int8" },
  "embeddings": {
    "一": "<base64文字列>",
    "七": "<base64文字列>",
    ...
  }
}
```

**格納先**: public/data/ に置く。src/data/ に置くとバンドルに含まれるため不可。

**注意**: このスクリプトは一度だけ手動実行し、結果をコミットする。ビルドパイプラインには組み込まない。

### タスク2: 閾値実験スクリプトの作成と実行

**目的**: (A) 128次元 vs 384次元の品質比較、(B) correct/close/wrongの閾値決定、(C) フィードバック情報量の検証。

**作業内容**:
1. scripts/analyze-embedding-thresholds.ts を新規作成
2. タスク1で生成した128次元版と384次元版の両方を読み込む

**ステップA: 次元数の品質比較**
- 128次元と384次元のコサイン類似度のランク相関（Spearman's rho）を計算
- rho > 0.95 であれば128次元で十分と判断。128次元を採用する
- rho < 0.95 の場合、384次元を採用。ただしVercelサイズ制限（メモ19cd6a4aa69: Body exceeded 75000kb limit）のリスクがあるため、PMに判断を仰ぐ

**ステップB: 閾値設計**（修正: correct閾値の再設計）

3段階フィードバックの閾値設計方針:
- **correct**: コサイン類似度が高い上位N%のペア（目安: 0.85以上）。「海」と「湖」、「火」と「炎」のように**非常に近い意味**を持つペアがcorrectになること
  - 重要: 同一文字のみをcorrectとする設計は不採用。理由: 同一文字であればゲーム自体が勝利で終了するため、category列のcorrectがプレイ中に出現しないフィードバックに退化する。3段階が実質2段階になり、プレイヤーへの情報提供量が減少する
- **close**: コサイン類似度が中程度のペア（目安: correct閾値未満 かつ wrong閾値以上）
- **wrong**: コサイン類似度が低いペア

**ステップC: close率の情報量検証**（新規追加）

以下を計算してclose率の妥当性を定量的に検証する:
1. 各閾値候補に対して、ある漢字1文字をguessした場合にcorrect/close/wrongと判定される候補数を計算
2. 1回のカテゴリフィードバックで候補が平均何%に絞り込まれるかを計算する
   - 例: 2,136字中correctが50字、closeが300字なら、「correct」が出た場合候補は50/2136 = 2.3%に絞られる
3. カテゴリフィードバック以外の列（部首: 2段階、画数: 3段階、学年: 3段階+方向、音読み: 2段階、訓読み数: 3段階）と合わせて、6回の試行で候補1字に収束可能かを情報理論的に検証する
   - 各列の情報エントロピー（ビット数）を計算
   - 6回の試行で得られる総情報量が log2(2136) ≒ 11.06 ビットを超えるか確認

**ステップD: 具体的な漢字ペアによる検証**
閾値候補ごとに以下を確認:
- correct判定のペア例10組以上を列挙し、意味的に「非常に近い」と直感的に分かるか
- close判定のペア例10組以上を列挙し、意味的に「関連がある」と直感的に分かるか
- wrong判定のペアの中に、一般常識で「近い」と感じるペアがないか

**出力**: 閾値実験の結果レポート（correct閾値、close閾値、情報量検証結果、具体的な漢字ペア例）。この結果をメモとしてPMに報告し、タスク3に進む前にPMが閾値を承認する。

### タスク3: ゲームエンジンの改修

**目的**: evaluateCategory()をコサイン類似度ベースに変更し、関連コンポーネントを修正する。

**設計方針: プリロード方式を採用**（代替案ではなくメイン設計）

プリロード方式の設計:
- embeddings.ts にモジュールレベルのPromiseとgetEmbeddings()関数を用意する
- モジュールロード時にfetchを開始し、バックグラウンドでデータを取得
- GameContainerのuseEffectでawait loadPromise → setReady(true)
- evaluateGuess内ではgetEmbeddings()を同期呼び出し
- **engine.tsのevaluateGuessシグネチャは同期のまま維持**

この方式のメリット:
1. engine.tsのevaluateGuessシグネチャ変更が不要
2. buildGameStateの復元ロジックが現状のまま動作（useStateの同期初期化制約と衝突しない）
3. 既存テストへの影響が最小限
4. 400KB程度のgzip後データは数百ミリ秒でロード可能、UXへの影響は軽微

**変更ファイル一覧**:

#### 3a. 新規: src/games/kanji-kanaru/_lib/embeddings.ts
- モジュールレベルでfetchを開始するPromise
- let cache: Map<string, Int8Array> | null = null;
- export async function preloadEmbeddings(): Promise<void> — fetchしてbase64デコード、cacheに格納
- export function getEmbeddings(): Map<string, Int8Array> — キャッシュを同期的に返す。ロード未完了時はエラーをスローする。ただし、GameContainerでロード完了をReact状態で管理するため、ロード前に呼ばれることは設計上ありえない

## リスクと対策

### リスク1: Vercelデプロイサイズ制限
- 128次元を第一選択とすることでリスク軽減（356KB raw / 214KB gzip）
- 384次元（1.1MB raw）は品質が不十分な場合のフォールバック。採用する場合はVercelサイズを事前確認

### リスク2: int8量子化による品質劣化
- タスク2でfloat32とint8の類似度ランク相関を確認（r > 0.95で問題なし）
- クランプ処理を明記し、値域外の値による異常を防止

### リスク3: プリロード方式の実装
- engine.tsのシグネチャ変更が不要なため、影響範囲は限定的
- GameContainerでembeddingsReady状態を管理し、ロード完了前にevaluateGuessが呼ばれないことをReactの状態管理で保証する
- ロード未完了時のgetEmbeddings()はエラーをスローするが、設計上呼ばれることはない（防御的プログラミング）

### リスク4: 閾値の妥当性
- タスク2のステップC（情報量検証）とステップD（具体的ペア検証）で定量・定性の両面から検証
- タスク2完了後にPM承認を挟むことで、不適切な閾値でのエンジン改修を防止

### リスク5: ブラウザキャッシュの一貫性
- fetch URLにバージョンパラメータまたはコンテンツハッシュを含める
- ファイル更新時にハッシュを更新する手順を文書化

### リスク6: セーブデータの後方互換性
- 既存プレイヤーのlocalStorageにあるゲーム進行状態のcategory値が新方式で再計算すると異なる可能性
- ゲームのstatus判定は文字一致で行うため、進行には影響しない
- category列の表示色が変わる可能性はあるが、ゲーム機能としては問題なし

### リスク7: 128次元切り出しの品質
- paraphrase-multilingual-MiniLM-L12-v2がMatryoshka Representation Learning対応でない場合、先頭128次元の切り出しでは品質が低下する可能性
- タスク2のステップAで128次元と384次元のランク相関を検証し、品質差を定量評価
- 品質が不十分ならPCA等の次元削減またはフル384次元を検討
