### 1.4 ポイント配分の均衡チェック

各タイプが主ポイント(2pt)を獲得できる質問数: 各タイプ5問中2〜3回。1タイプに偏らないようバランスされている。最大理論値は各タイプ20pt（2pt x 10問）だが、実際にはタイプごとに主ポイント供給が分散されており、10〜14ptが現実的な最高到達点となる。

### 1.5 各タイプの結果テキスト

**フェス一番乗り族（festival-pioneer）**

あなたの音楽ライフは常に「次」を向いている。新譜が出れば最速で聴き、フェスの先行チケットは発売0.3秒で確保済み。Spotifyの「今年のまとめ」では、誰よりも多くの新曲を聴いた証が刻まれる。問題は1つだけ。来月にはもう今月のお気に入りを忘れていること。でも大丈夫、あなたにとって音楽は「出会い続ける」ことそのものだから。次のフェスでまた新しい「人生の1曲」を見つけるでしょう。

---

**プレイリスト伝道師（playlist-evangelist）**

あなたは「この曲を聴いてほしい」という使命感で生きている。友達のLINEに突然URLが飛んできたら、十中八九あなたの仕業。プレイリストの説明文まで丁寧に書くその姿は、もはや音楽キュレーター。「聴いた?」のフォローアップも忘れない。問題は、相手が本当に聴いたかどうかを確認するまで眠れないこと。でも安心してください。あなたの布教は確実に誰かの人生のBGMを変えています。

---

**ひとり開拓民（solo-explorer）**

あなたは音楽の未踏の地を1人で歩く探検家。Spotifyの「あなたへのおすすめ」の先にある「おすすめのおすすめ」まで掘り進む。月間リスナー3桁のアーティストを見つけた時の喜びは、トレジャーハンターが秘宝を発見した瞬間に等しい。問題は、好きなバンドがメジャーデビューすると微妙に複雑な気持ちになること。「売れる前から知ってたんだけど」と言いたいのを我慢する日々です。

---

**リピート戦士（repeat-warrior）**

あなたの再生回数は統計的外れ値。同じアルバムを100周してもまだ新しい発見がある（と本気で思っている）。Spotifyの「今年のまとめ」では、トップ5が去年と完全に一致している。問題は、友達に「まだそれ聴いてるの?」と言われること。でもいいじゃないですか。名作映画を何度も観る人は「通」と呼ばれるのに、音楽だと「しつこい」と言われるのは不公平です。あなたは正しい。

---

**作業用BGM職人（bgm-craftsman）**

あなたにとって音楽は「空気」である。最高の褒め言葉は「あ、BGM変わってたんだ。気づかなかった」。シーン別プレイリストの作り込みは異常で、「雨の日の午後3時・カフェ・やや眠い」というプレイリストが存在する。問題は、音楽の趣味を聞かれると答えに困ること。好きなのは「場面と音楽の組み合わせ」であって、特定の曲やアーティストではないから。あなたは音楽の建築家です。

---

**カラオケ癒し枠（karaoke-healer）**

あなたがいるだけでカラオケの空気が和む。選曲は「みんなが口ずさめる曲」が鉄則。マイクを独占しないし、人の歌にはちゃんとタンバリンを入れる。音楽は「一緒に楽しむもの」というあなたの哲学は、実は一番贅沢な音楽の味わい方。問題は、1人カラオケの存在意義が理解できないこと。音楽は分かち合ってこそ完成すると、あなたの全身が叫んでいます。

---

**深夜シャッフル系（midnight-shuffle）**

午前2時、暗い部屋、シャッフル再生。これがあなたの音楽儀式。自分では曲を選ばない。アルゴリズムと偶然に身を委ね、流れてきた曲に人生の意味を見出す。たまに全く聴いたことのない曲が流れてきて「これは運命だ」と確信する。問題は、翌朝その曲のタイトルを思い出せないこと。でも大丈夫。また今夜、シャッフルの神様が届けてくれます。

---

**歌詞の一節で生きてる人（lyrics-dweller）**

あなたにとって音楽は「音」ではなく「言葉」。メロディより先に歌詞が耳に入る。心に刺さった一節はスクショしてカメラロールに保存してある。通勤中にイヤホンで泣いたことが3回以上ある。問題は、人に好きな曲を説明しようとすると歌詞を朗読し始めてしまうこと。でもそれでいい。あなたが音楽から受け取っているものは、他の7タイプの誰よりも深い。たぶん。

### 1.6 相性マトリクス（8x8 = 36通り）

相性は「関係性ラベル + シナリオテキスト」の形式で表現する。同タイプ同士を含む全36通り（上三角行列 + 対角線）。

**凡例:** `[TypeA] x [TypeB] = 関係性ラベル: シナリオテキスト`

#### 同タイプ同士（8通り）

**festival-pioneer x festival-pioneer = 共鳴する疾風:**
二人とも新しいもの好きすぎて、「先週ハマった曲」が毎週変わる。お互いの推しを交換しても翌週にはもう別の曲の話をしている。二人の会話の速度についてこれる人間は地球上にほとんどいない。

**playlist-evangelist x playlist-evangelist = 終わらない布教合戦:**
「これ聴いて」「いやまずこっち聴いて」が永遠に続く。お互いのプレイリストを聴くだけで1日が終わる。布教先が布教してくるという想定外の事態に、二人とも密かに喜んでいる。

**solo-explorer x solo-explorer = 秘境の同志:**
お互いが発掘してきたアーティストを見せ合う至福の時間。ただし「自分が先に見つけた」主張は譲れない。月間リスナー数が3桁のバンドを同時に発見した日には、奇跡に感謝する。

**repeat-warrior x repeat-warrior = 無限ループの絆:**
同じ曲を延々と聴き続ける二人。話題は「あの曲の2番のサビの後のギターソロ」のような極めて限定的な領域に集中する。周囲には理解されないが、二人の間では全く問題ない。

**bgm-craftsman x bgm-craftsman = 究極の空間デザイナー:**
二人でカフェに入ると、店のBGMの評論会が始まる。「この時間帯にボサノバは安易すぎる」といった会話を本気でやる。共同でプレイリストを作ると最高傑作が生まれるが、完成までに3ヶ月かかる。

**karaoke-healer x karaoke-healer = 最高のハーモニー:**
二人がいるカラオケは全員が楽しい。お互いに歌う順番を譲り合い、全員の選曲にリアクションする。唯一の問題は、二人とも「最後に歌う人」になりたがること。

**midnight-shuffle x midnight-shuffle = 偶然の共鳴:**
深夜に同時刻にシャッフル再生をして、偶然同じ曲が流れた日には運命を感じる。お互いのイヤホンを片耳ずつ交換して、違う曲が同時に流れる体験を「アート」と呼ぶ。

**lyrics-dweller x lyrics-dweller = 言葉の海を泳ぐ二人:**
歌詞の解釈について3時間語れる唯一の組み合わせ。「あの一節の『空』は比喩なのか文字通りなのか」で議論が白熱する。二人の間では、歌詞カードが聖典に見えてくる。

#### 異タイプ同士（28通り）

**festival-pioneer x playlist-evangelist = 最強の拡散装置:**
一人が新曲を見つけ、もう一人が全力で広める。音楽業界が最も恐れる口コミコンビ。二人が組めば、インディーズバンドを3日でバズらせることも不可能ではない。

**festival-pioneer x solo-explorer = 発見競争のライバル:**
「知ってた?」「いや俺の方が先に」という不毛だが楽しい争いが日常。フェス派と配信派の対立は根深いが、新しい音楽への敬意では一致している。

**festival-pioneer x repeat-warrior = スピードと持久力:**
一人は次々と新曲に飛びつき、もう一人は1曲を永遠に聴く。お互いの音楽の接し方が真逆すぎて、一周回って尊敬し合っている。「量の美学」と「深さの美学」の共存。

**festival-pioneer x bgm-craftsman = 素材の供給者と建築家:**
フェス族が大量に仕入れてくる新曲を、BGM職人がシーン別に整理整頓する。最高のプレイリストが生まれるが、「この曲はここじゃない」という職人のダメ出しにフェス族がたじろぐ。

**festival-pioneer x karaoke-healer = お祭りムードメーカー:**
二人がいれば、どんな飲み会も音楽フェスに変わる。フェス族が最新曲を投下し、癒し枠が「みんな知ってるバージョン」に翻訳する。場の空気を制する最強タッグ。

**festival-pioneer x midnight-shuffle = 正反対の音楽冒険家:**
一人は能動的に新曲を探し、もう一人は受動的に出会いを待つ。方法は違うが「まだ知らない曲に出会いたい」という欲望では完全に一致。深夜のフェス配信を一緒に観たら最高の体験になる。

**festival-pioneer x lyrics-dweller = 速度と深度の交差点:**
フェス族が「この曲いいよ」と投げた曲を、歌詞族が3日かけて解析して返してくる。テンポは合わないが、互いの視点が曲の新しい魅力を引き出す。

**playlist-evangelist x solo-explorer = 布教者と隠者の対話:**
伝道師は「みんなに広めたい」、開拓民は「自分だけの秘密にしたい」。正反対のようだが、音楽への愛の深さは同レベル。開拓民が秘蔵の1曲を伝道師に渡した日、世界が少し変わる。

**playlist-evangelist x repeat-warrior = 情報過多 vs 情報固定:**
伝道師が毎日5曲送ってくるが、戦士は1曲目すら聴き終わっていない。「もう聴いた?」「まだ前の聴いてる」のやり取りが永遠に続く。でも年に1回、戦士のヘビロテに伝道師の推薦曲が入る。それが二人の記念日。

**playlist-evangelist x bgm-craftsman = キュレーター対決:**
「みんなに聴かせたいプレイリスト」vs「完璧な空間演出のプレイリスト」。目的が違うだけでスキルセットは同じ。共作プレイリストは「聴かせたい」と「馴染ませたい」の緊張感で名作になる。

**playlist-evangelist x karaoke-healer = ソーシャル音楽の両翼:**
どちらも音楽は「人と共有するもの」と信じている。伝道師がURLを送り、癒し枠が「一緒に聴こう」と誘う。方法論は違うが、目的は同じ。二人がいるグループのLINEは音楽の話題が絶えない。

**playlist-evangelist x midnight-shuffle = 計画と偶然:**
伝道師は「次に聴くべき曲」を完璧に設計し、シャッフル系は全てを偶然に委ねる。お互いの方法を理解できないが、「いい曲に出会えた」という結果は同じ。過程にこだわるか、結果にこだわるか。

**playlist-evangelist x lyrics-dweller = 広さと深さの共存:**
伝道師が100曲を浅く広く聴かせ、歌詞族が1曲を深く深く語る。会話が噛み合わないようで、実は最高の音楽談義になる。伝道師のおかげで歌詞族の世界が広がり、歌詞族のおかげで伝道師が立ち止まる。

**solo-explorer x repeat-warrior = 発見者と定住者:**
開拓民が新大陸を発見し、戦士がそこに永住する。「新しい曲見つけたよ」「まだ前の聴いてる」は日常会話。でも戦士が珍しく「新しい曲聴きたい」と言った時、開拓民は最高の1曲を差し出す。

**solo-explorer x bgm-craftsman = 仕入れと加工の名コンビ:**
開拓民が掘り出してきたレア曲を、職人が適切なシーンに配置する。お互いの才能を認め合う関係。ただし「この曲はBGMにするには存在感がありすぎる」という職人の判断に、開拓民が少し傷つく。

**solo-explorer x karaoke-healer = 対極の音楽観:**
「誰も知らない曲」を愛する人と「みんなが知ってる曲」を愛する人。カラオケでの選曲は永遠に交わらないが、だからこそ互いの選曲が新鮮。癒し枠に「これ歌えるよ?」と開拓民の推し曲を歌われた日、開拓民は複雑な涙を流す。

**solo-explorer x midnight-shuffle = 静かなる探求者同盟:**
どちらも「まだ見ぬ曲」に惹かれる。能動と受動の違いだけ。二人でイヤホンを分け合って無言で聴く時間は、言葉が要らない最高の音楽体験。

**solo-explorer x lyrics-dweller = 発掘と解読の二人組:**
開拓民が見つけてきた曲を、歌詞族が歌詞の意味を解読する。「この曲の歌詞、実はこういう意味だったんだ」と教えられた瞬間、開拓民はその曲をもっと好きになる。最高のバディ。

**repeat-warrior x bgm-craftsman = こだわりの方向性が違う二人:**
戦士は「この1曲」にこだわり、職人は「この場面」にこだわる。どちらも妥協しない完璧主義者。二人の妥協点を見つけるのは難しいが、見つかった時のプレイリストは芸術品。

**repeat-warrior x karaoke-healer = 安定と調和:**
戦士の十八番と癒し枠の定番曲。カラオケでは最も安定感のあるペア。新しい曲は一切入ってこないが、その安心感が良い。10年後も同じ曲を歌っている二人を想像すると微笑ましい。

**repeat-warrior x midnight-shuffle = 固執と放流:**
同じ曲を100回聴く人と、同じ曲を2回聴かない人。音楽の接し方は180度違うが、音楽なしでは生きられない点は完全一致。戦士のヘビロテ曲がシャッフルで流れた日、シャッフル系は「運命かも」と思う。

**repeat-warrior x lyrics-dweller = 反復の達人と解釈の達人:**
戦士が100周目で気づいた音の変化を、歌詞族が歌詞の文脈から説明する。「だからこの部分でギターの音が変わるのか!」という発見は、二人でなければ辿り着けない境地。

**bgm-craftsman x karaoke-healer = 空間演出の名匠ペア:**
職人がBGMで雰囲気を作り、癒し枠が人を巻き込む。パーティーの音楽担当はこの二人に任せれば間違いない。ただし主導権争いがたまに起きる。

**bgm-craftsman x midnight-shuffle = 設計と偶然の音楽論:**
職人は「この場面にはこの曲」と設計し、シャッフル系は「何が来ても受け入れる」。音楽との向き合い方は正反対だが、どちらも音楽を「体験」として捉えている点で深く共感し合う。

**bgm-craftsman x lyrics-dweller = 音と言葉の二面性:**
職人は音楽を「音」として聴き、歌詞族は「言葉」として聴く。同じ曲を聴いているのに全く違うものを受け取っている。お互いの視点を知ると、音楽の奥行きが倍になる。

**karaoke-healer x midnight-shuffle = 陽と陰の音楽体験:**
癒し枠は大勢で歌い、シャッフル系は深夜に1人で聴く。音楽の楽しみ方が完全に反対だが、互いの世界を少しだけ体験すると人生が豊かになる。癒し枠に深夜のシャッフルを教えたら、新しい扉が開く。

**karaoke-healer x lyrics-dweller = みんなの歌と私の歌:**
癒し枠は「みんなで盛り上がれる曲」、歌詞族は「自分の心に響く曲」。違うようで、どちらも音楽に「感情」を求めている。カラオケで歌詞族が選んだ曲を癒し枠が全力でハモった時、最高の瞬間が生まれる。

**midnight-shuffle x lyrics-dweller = 深夜の邂逅:**
シャッフルで偶然流れてきた曲の歌詞に心を撃ち抜かれる。これが二人の共通体験。「あの曲、偶然聴いたんだけど歌詞がすごくて...」という会話が成立する唯一の組み合わせ。深夜の偶然が言葉の発見につながる、ロマンチックな関係。

## 2. 相性判定機能の設計

### 2.1 UXフロー（URLパラメータ型）

1. ユーザーAが診断完了 -> 結果画面に「友達との相性を調べる」ボタン表示
2. ボタンクリック -> `https://yolos.net/quiz/q01-music-personality?ref={typeId}` をクリップボードにコピー or LINEでシェア
3. ユーザーBがそのURLで診断ページにアクセス -> 通常通り診断を受ける
4. ユーザーB診断完了 -> 結果の下に「相性結果」セクションを自動表示
5. 相性結果にもシェアボタンを設置

### 2.2 相性結果の表示構成

```
[あなたのタイプ] x [友達のタイプ]
[関係性ラベル]（例: 「最強の拡散装置」）
[シナリオテキスト]（例: 「一人が新曲を見つけ、もう一人が...」）
[シェアボタン群]
```

### 2.3 相性シェアテキストのテンプレート

```
私は「{自分のタイプ名}」、友達は「{友達のタイプ名}」。
音楽相性は「{関係性ラベル}」でした!
#音楽性格診断 #yolosnet
```

相性シェアURLは: `/quiz/q01-music-personality/result/{自分のtypeId}?with={相手のtypeId}`

### 2.4 相性結果ページのOGP

相性用のOGPも必要。表示内容:
- 上部: 「音楽性格診断 相性結果」
- 中央: 両タイプのアイコン + 関係性ラベル
- 下部: yolos.net

## 3. 技術実装計画

### 3.1 ファイル構成

```
src/quiz/data/q01-music-personality.ts       # QuizDefinition + 相性データ
src/quiz/_components/CompatibilitySection.tsx  # 相性結果表示コンポーネント

src/quiz/_components/InviteFriendButton.module.css
```

### 3.2 データ構造

q01-music-personality.ts には、QuizDefinitionに加えて以下をエクスポートする:

```typescript
// 相性データの型
interface CompatibilityEntry {
  label: string;       // 関係性ラベル
  description: string; // シナリオテキスト
}

// 相性マトリクス（キー: "typeA--typeB" ソート済みペア）
export const compatibilityMatrix: Record<string, CompatibilityEntry> = {
  "bgm-craftsman--bgm-craftsman": { label: "究極の空間デザイナー", description: "..." },
  "bgm-craftsman--festival-pioneer": { label: "素材の供給者と建築家", description: "..." },
  // ... 全36エントリ
};

// ヘルパー関数
export function getCompatibility(typeA: string, typeB: string): CompatibilityEntry | undefined {
  const key = [typeA, typeB].sort().join("--");
  return compatibilityMatrix[key];
}
```

### 3.3 QuizContainerの拡張

既存のQuizContainerを直接変更せず、Q01専用のラッパーコンポーネントを作る方法を推奨する。ただし、既存のQuizContainerがResultCardを表示する際に相性セクションを差し込む必要があるため、以下の2案がある:

**案A（推奨）: ResultCardにchildren propを追加**
- ResultCardに `children?: React.ReactNode` を追加
- QuizContainerで `ref` パラメータが存在する場合に CompatibilitySection を children として渡す
- 既存の他クイズには影響しない（childrenがなければ何も表示されない）

**案B: Q01専用ページコンポーネント**
- `/quiz/q01-music-personality/page.tsx` をカスタム実装
- QuizContainerを使わず、Q01専用のロジックを組む
- 既存コードへの変更ゼロだが、コード重複が多い

案Aが既存フレームワークとの一貫性を保ちつつ拡張性が高いため推奨。

### 3.4 URLパラメータの処理

- クイズページ `/quiz/[slug]/page.tsx` でsearchParamsから `ref` を取得
- `ref` が存在する場合、QuizContainerにpropとして渡す
- QuizContainerが結果フェーズになった時、`ref` の値（相手のtypeId）と自分のtypeIdから相性を計算
- 相性セクションをResultCardの下に表示

### 3.5 結果ページのルーティング

既存の `/quiz/[slug]/result/[resultId]` ページに、クエリパラメータ `?with={typeId}` がある場合に相性情報を表示する。

- 相性表示は既存の結果ページ内にクライアントサイドで処理
- OGP画像は相性用の分岐を追加（`with` パラメータの有無で切り替え）

### 3.6 OGP画像

既存の `/quiz/[slug]/result/[resultId]/opengraph-image.tsx` パターンに準拠。

個人結果のOGP: 既存パターンをそのまま使用（テーマ色 + アイコン + タイプ名）。

相性OGPについては、Route Handlerベースで動的生成するか、静的生成の組み合わせ(8x8=64パターン)にするか、builderの判断に委ねる。静的生成の方がパフォーマンスは良いが、64枚のOGP画像はビルドサイズに影響する可能性がある。

### 3.7 実績システムへの統合

badges.ts の QUIZ_IDS 配列に `"quiz-q01-music-personality"` を追加する。これにより既存のバッジ条件（quiz-first, quiz-all, all-once等）にQ01が自動的に組み込まれる。

### 3.8 registryへの登録

`src/quiz/registry.ts` にQ01のデータファイルをimportし、`quizEntries` 配列に追加する。

## 4. テスト計画

### 4.1 ユニットテスト

- **ポイント集計ロジック**: 全10問に対して特定パターンの回答を入力し、期待するタイプが結果になることを検証
  - 各タイプに偏った回答パターン（8パターン）で正しいタイプが返ることを確認
  - 同点ケースで先頭タイプが返ることを確認

- **相性マトリクス**: 
  - 全36エントリが存在することを確認
  - getCompatibility関数が順序を問わず同じ結果を返すことを確認（getCompatibility("a","b") === getCompatibility("b","a")）
  - 全エントリのlabelとdescriptionが空文字でないことを確認

### 4.2 タイプ偏りチェック

- 全選択肢を均等に選んだ場合（ランダム回答）に特定タイプに偏らないことを確認
- 各タイプの最大到達可能ポイントに極端な差がないことを確認

### 4.3 相性機能のテスト

- `?ref=typeId` パラメータがない場合、相性セクションが表示されないことを確認
- `?ref=invalid-id` の場合、相性セクションが表示されないことを確認
- 正しいrefパラメータがある場合、診断完了後に相性セクションが表示されることを確認
