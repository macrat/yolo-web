# 調査レポート: 漢字カナル サーバーサイド評価API方式の妥当性

## 1. 現行アーキテクチャの全体像

### データフロー

1. **ページ初期化**: `GameContainer.tsx` (Client Component) がマウント時に以下をすべてクライアントで実行する
   - `kanji-data.json` (671 KB) をバンドル内にインポート（静的インポート）
   - `puzzle-schedule-{difficulty}.json` (beginner: 40KB, intermediate: 41KB, advanced: 41KB) を静的インポート
   - `getTodaysPuzzle()` でスケジュールから当日の `KanjiEntry` を決定
   - `localStorage` から過去の推測履歴を復元し、`evaluateGuess()` を再実行して `GuessFeedback[]` を再構築

2. **推測評価**: `handleGuess()` が `evaluateGuess(guessEntry, gameState.targetKanji)` を同期実行（<1ms）

3. **状態保存**: `localStorage` に推測した文字列（`guessChars: string[]`）のみ保存。フィードバック（`GuessFeedback`）は保存せず、ページロード時に毎回再計算する設計。

### ターゲット漢字の保持場所

ターゲット漢字（`targetKanji: KanjiEntry`）は `gameState` として React の状態に保持されており、DevTools から参照可能。`localStorage` には文字列のみ保存（`guessChars: string[]`）で、フルデータは保存しない。

### evaluateGuess の評価項目（すべてクライアントで完結）

- **文字一致**: 文字列比較
- **部首**: 文字列比較
- **画数**: 数値差による3段階評価
- **学年**: 数値差による3段階評価 + 方向性
- **音読み**: 配列の積集合による2段階評価
- **カテゴリ**: `radicalGroup` 整数値の比較（現在の問題箇所）
- **訓読み数**: 件数差による3段階評価

### 既存の Route Handler

現在プロジェクトに存在する Route Handler は4つすべてが `export const dynamic = "force-static"` であり、実質的に静的ファイル生成として動作する。実行時に動的処理を行うAPIエンドポイントは現在ゼロ。

## 2. 他のWordle系ゲームのアーキテクチャ実態

### オリジナルWordle（NYT買収前、2021-2022）
- 完全クライアントサイド
- 全解答リストをJavaScriptバンドル内にハードコード
- ターゲット語はクライアント側で日付から計算
- 結果: ソースを見れば将来の答えも全部わかる状態（広く報道された）

### NYT買収後のWordle（2022年〜）
- サーバーサイド計算に移行
- 答えがクライアントに露出しない設計
- NYTアカウントログイン・ストリーク同期などの機能追加

### 重要な文脈
NYTがサーバーサイドに移行した理由は「ソースコードから翌日以降の答えが一覧できる」というスポイラー問題と、有料サービス（NYT Games）との統合のためである。両方ともyolos.netには当てはまらない（デイリー公開パズル、課金なし、ランキングなし）。

## 3. サーバーサイドAPI方式の分析

### メリット

**チート対策**
- ターゲット漢字がクライアントメモリに存在しない
- DevToolsでのReact状態参照、`localStorage`からの抽出が不可能になる

**Embeddingデータのダウンロード不要**
- もし「推測時にリアルタイムEmbedding計算でカテゴリ判定」する設計なら、384次元×2,136件≒3.1MB（gzip後約1.25MB）のベクトルデータをダウンロードせずに済む

**クライアントコードの一部簡素化**
- 評価ロジックがサーバーに集約される

### デメリット（深刻）

**1. ネットワーク遅延によるゲーム体感の劣化（最大の問題）**
- クライアントサイド: <1ms（同期処理）
- サーバーサイド（Vercel東京リージョン）: p50で50-150ms、p95で300ms以上
- モバイル・不安定接続: p50で200-500ms
- Wordleスタイルのゲームはフィードバックアニメーションが体感品質に直結する。推測確定後に100-300msの「ラグ」は明確に知覚される。

**2. localStorage復元フローの破綻**
- 現在: `guessChars: string[]` を保存し、ページロード時に `evaluateGuess()` を再実行して状態復元
- サーバーサイドAPI化後: ページロード時に過去の推測をAPIで再評価する必要がある（N回のAPI呼び出し）か、`GuessFeedback[]` のフルデータをlocalStorageに保存する設計変更が必要
- N回のAPIコール on page load は体感品質として許容できない

**3. オフライン・不安定環境でのゲーム不能**
- クライアントサイドは完全オフラインでプレイ可能
- サーバーサイドAPIはネットワーク障害時にゲームが一切できなくなる

**4. エラーハンドリングの複雑化**
- 現在: 評価は決してエラーにならない（純粋関数）
- API化後: タイムアウト、5xx、ネットワーク障害への対応が必要
- 「推測中」ローディング状態、リトライロジック、エラー表示UIが必要になる

**5. Vercel Hobbyプランの制約（軽微）**
- 動的Serverless Function数: Hobbyプランは12個/デプロイが上限（現在0個）。1個追加は問題ない
- Function Invocations: 100万回/月。DAU=1,000で月10.5万回（制限の10.5%）、DAU=5,000で52.5%。制限内だが、DAU=10,000で超過
- Active CPU: 4 CPU-hrs/月。評価処理は軽量（<10ms/回）なので問題ない
- **結論**: 現在のトラフィック規模では制限は問題にならないが、スケールすれば上限に近づく

**6. チート対策の実効性が低い**
- 漢字カナルは1日1問のデイリーパズルで競争ランキングが存在しない
- チートによって他者が不利益を受けない
- 答えは毎日SNS等で共有・拡散される（これ自体がエンゲージメントの一形態）
- NYT Wordleでさえ、答えを当てるのを楽しみにしている人はチートしない

## 4. クライアントサイド方式の分析

### メリット

- **即時フィードバック**: <1ms で評価、アニメーション体験が最高
- **オフライン動作**: インターネット接続不要でプレイ可能
- **シンプルな実装**: エラーハンドリング不要、純粋関数として実装済み
- **復元の容易さ**: ページリロード時に `evaluateGuess()` を再実行するだけで状態復元
- **coding-rules.md との一致**: Rule 1「静的最優先、クライアント優先」に完全適合

### デメリット

- ターゲット漢字がクライアントメモリに露出（DevToolsで閲覧可能）
- ただし前述の通り、デイリーパズルへの影響は無視できる

## 5. B-190の本質的な問題に関する重要な発見

**B-190はデータ品質の問題であり、ランタイムアーキテクチャの問題ではない。**

メモ依頼では「Embeddingベクトルをfetchしてコサイン類似度を計算する設計」が言及されていた。しかし `KanjiEntry.category` フィールドはすでに整数値（1-20）として `kanji-data.json` に格納されており、ランタイムでの類似度計算は不要である。

問題は「カテゴリ整数の割り当てが意味的に正確でない（部首番号の機械的グループ化）」ことであり、解決策は「**ビルド時のデータ生成スクリプトで、Embeddingベースのクラスタリングを行い、意味的に正しいカテゴリ整数を割り当てる**」ことである。

このアプローチであれば:
- ランタイムのEmbeddingデータダウンロード（3.1MB）が不要
- サーバーサイドAPIが不要
- 現行アーキテクチャを変更せずにカテゴリ品質のみを改善できる

もし「ランタイムに推測漢字とターゲット漢字のEmbedding類似度を計算したい」場合でも、**ビルド時に各漢字のEmbeddingベクトルを計算してJSONに格納**し、クライアントがそのベクトルデータを取得してコサイン類似度を計算するクライアントサイドアプローチが技術的に実現可能である（ただし約1.25MBのダウンロードが発生するため、ビルド時クラスタリングの方が優れる）。

## 6. 総合推奨: クライアントサイド方式を維持する

### 推奨: クライアントサイド方式の継続

**主たる理由（ユーザー体験）**:
- 100-300msのレイテンシがゲーム体感を明確に劣化させる
- Wordleスタイルのゲームは即時フィードバックが体験品質の核心
- 不安定接続・オフライン環境でのプレイ不能は重大な機会損失

**副次的理由（設計合理性）**:
- coding-rules.md Rule 1「静的最優先、クライアント優先」に完全適合
- チート対策のベネフィットがコストに見合わない（デイリーパズル、競争なし）
- B-190の本質的解決策（ビルド時カテゴリ割り当て改善）はサーバーサイドAPIを必要としない
- 既存のlocalStorageアーキテクチャを維持でき、ページロード復元が高速

**B-190への具体的な含意**:
サーバーサイドAPIへの移行は不要。B-190の解決策は「ビルド時データ生成スクリプトの改善」に集中すべきである。具体的には、Embeddingモデル（例: sentence-transformers）を使ってビルド時に各漢字の意味的カテゴリを計算し、`category` 整数フィールドに格納する。ランタイムアーキテクチャ（evaluateGuess, categories.ts, engine.ts）は変更不要。

**Vercel Hobbyプランとの整合性**:
現在のトラフィック規模では動的APIは制限内に収まるが、不要なコストを負う理由はない。既存の4つのRoute Handlerがすべてforce-staticである設計思想とも一致する。
