# B-190タスク2: 閾値実験スクリプトの作成と実行

## 計画
メモ 19cd70dc491 のタスク2を読んでください。

## 作業内容

### ステップA: 128次元 vs 384次元の品質比較
1. `scripts/analyze-embedding-thresholds.ts` を新規作成
2. 128次元版と384次元版のベクトルデータを読み込む（public/data/kanji-embeddings-128.json, kanji-embeddings-384.json）
3. 全ペアのコサイン類似度を計算し、128次元と384次元のSpearmanランク相関を検証
4. r > 0.95 なら128次元を採用、それ以下なら384次元を採用

### ステップB: 閾値の決定
1. 採用した次元数のベクトルで、全ペアのコサイン類似度分布を計算
2. correct閾値: コサイン類似度上位（0.85以上を目安）で「非常に近い意味」のペアがcorrectになるよう設定（例: 海-湖、火-炎）
3. close閾値: correct未満で、意味的に関連のあるペアがcloseになるよう設定
4. 具体的な漢字ペアを列挙して直感に合うか検証:
   - correct候補: 海-湖、火-炎、山-丘、父-母 など
   - close候補: 海-川、火-熱、山-谷 など
   - wrong候補: 海-金、火-口、山-食 など

### ステップC: 情報量検証
1. 決定した閾値で、1回のカテゴリフィードバックが提供する平均情報量（ビット）を計算
2. カテゴリ以外のフィードバック列（部首、画数、学年、音読み、訓読み数）と合わせて、6回で2,136字を1字に絞り込めるか（log2(2136)≈11.06ビット必要）を検証

## 出力
- 採用する次元数（128 or 384）とその根拠
- correct閾値とclose閾値の値
- 代表的な漢字ペアの判定結果一覧
- 情報量の計算結果

## 検証
- npm run lint && npm run format:check がスクリプトに対してパスすること
- 閾値決定の根拠が具体的な漢字ペアで裏付けられていること
