# cycle-71: 3タスク計画レビュー結果

## 全体所見

3つの計画はいずれも調査メモに基づいた堅実な内容であり、既存コードベースのパターンを正しく踏襲している。ただし、いくつかの重要な指摘事項がある。

---

## 1. プライバシーポリシーページ（19cc85f414f）

### 評価: A（承認）

計画の品質は高い。AdSense要件（A〜F）、個人情報保護法の要件、既存コードパターンとの整合性のいずれも十分にカバーされている。

### 指摘事項

**(1) aboutページにOGP画像が存在しない件**

計画ではprivacyページにopengraph-image.tsxとtwitter-image.tsxを新規作成するとしている。方針自体は正しいが、aboutページにはOGP画像が存在しないことに注意。「aboutページのパターンに準拠」と記載しているが、OGP画像についてはaboutページではなくtoolsやgamesのパターンを参考にすべきである。大きな問題ではなく、計画自体にOGP画像の具体的な実装方針（createOgpImageResponseの使用）が明記されているため、builderが迷うことはないと判断する。

**(2) sitemapへの追加: privacyの日付がhomepageDate計算に含まれていない**

計画ではsitemap.tsにprivacyエントリを追加するとしているが、既存のhomepageDateの計算ロジック（全コンテンツの最新日付の最大値を取る部分）にprivacyの日付を含めるかどうかの言及がない。aboutは現在homepageDate計算に含まれているため、privacyも同様に含めるべきか、あるいは法的ページなので除外するかを明示すべき。ただし、これは軽微な実装判断でありbuilderが既存パターンに倣って対応できる範囲。

**(3) AdSense表現「利用することがあります」は適切**

セクション5(b)のAdSenseの表現について、「利用することがあります」という将来形を使う方針は現状に合致しており適切。この判断は正しい。

**(4) テストカバレッジは十分**

h1見出し、主要セクション、外部リンク、TrustLevelBadgeの存在確認をテストする方針は、aboutページのテストパターンと整合的であり十分。

### 改善提案（任意）

- セクション7「個人情報の開示・訂正・削除」でGitHub Issuesを窓口としているが、contactページが同時に作成される場合、contactページへの誘導も併記すると一貫性が高まる。この点はbuilder作業時に調整可能。

## 2. お問い合わせページ（19cc85f127d）

### 評価: B（軽微な修正で承認）

全体の方針は適切だが、Googleフォーム関連の計画に実装上の懸念がある。

### 指摘事項

**(1) [重要] Googleフォームの事前準備が未解決**

計画のセクション8で「Googleフォームの公開URLが必要」「builderの作業範囲外」と正しく認識しているが、セクション11の代替計画（Phase 1/Phase 2の段階的アプローチ）があるため対処自体は可能。ただし、PMがOwnerにGoogleフォームの準備状況を確認する前にbuilderに依頼する場合、依頼メモにどちらのPhaseで実装するかを必ず明記すること。この点は計画に書かれているが、強調しておく。

**推奨**: 現時点ではPhase 1（mailtoリンク + GitHub Issuesのみ）で実装し、Googleフォームは後から追加する方針をデフォルトとすべき。理由は以下の通り:
- Googleフォームの準備にはOwnerの手動操作が必要であり、サイクル内で確実に完了するかが不透明
- mailtoリンクとGitHub Issuesだけで AdSenseの最低要件は満たせる
- Googleフォームのiframeはデザイン不整合のリスクがあり、サイト全体の品質に影響しうる

**(2) [重要] メールアドレスの確認が必要**

計画では仮に contact@yolos.net を想定しているが、このアドレスが実際に存在するかの確認が取れていない。builder作業開始前にPMがOwnerに確認すべきである。メールアドレスが用意できない場合、mailtoリンクは省略してGitHub Issuesのみを連絡手段とする必要がある。

**(3) ContactForm.tsxをClient Componentにする設計は正しい**

iframeの埋め込みであってもClient Componentとして分離する判断は適切。ただし、Googleフォームが未準備の場合（Phase 1）、このファイル自体が作成されないため、Phase 1実装時にpage.tsxがServer Componentのみで完結できることを確認する必要がある。

**(4) EmailLink.tsxのスパム対策は実用的**

JavaScriptで動的にメールアドレスを組み立てる手法は、基本的なクローラー対策として十分。noscriptフォールバックも適切。

**(5) aboutページの更新方針は適切**

現在のaboutページのお問い合わせセクションをcontactページへのリンクに変更する方針は正しい。ただし、GitHub Issuesへの直接リンクも残すかどうか（contactページにも掲載されるので冗長になる可能性）はbuilder判断に委ねてよい。

**(6) sitemapへの追加が計画に含まれていない**

計画の実装順序（セクション9）にsitemap.tsの更新が含まれていない。privacyページの計画には含まれているため、contactページでも同様にsitemap.tsへのエントリ追加を含めるべき。

**(7) テストの記載がない**

計画にテストの記載が欠けている。privacyページの計画にはテストが含まれているため、contactページにも同等のテスト（ページレンダリング、主要セクションの表示、リンクの確認、フッターテスト更新）を含めるべき。

### 改善提案

- sitemapへの/contactエントリ追加をセクション9の実装順序に追加すること
- テスト方針のセクションを追加すること（privacyの計画と同等の内容）
- Phase 1をデフォルトの実装方針として明確にすること

## 3. 実績システム（19cc8603cb7）

### 評価: B（軽微な修正で承認）

設計の品質は高く、site-concept.mdの仕様に忠実である。データ構造、コンポーネント設計、テスト方針のいずれも堅実。ただし、いくつかの設計判断に関する指摘がある。

### 指摘事項

**(1) [重要] Header.tsxがServer Componentである件**

計画のセクション3で「src/components/common/Header.tsx ← ストリークバッジ表示追加（任意）」としているが、現在のHeader.tsxはServer Componentである（"use client"なし）。StreakBadge（Client Component）をHeader内に配置するには、Header.tsxをClient Componentに変更するか、StreakBadgeのみをClient Componentとして分離してServer Component内に埋め込むかの2つの方法がある。

後者（Server Component内にClient Componentを埋め込む）はNext.js App Routerで可能なパターンであり、Header全体をClient Componentにする必要はない。ただし、計画にはこの技術的考慮が記載されていない。builderへの依頼時にこの点を明示すべき。

**(2) 日付ヘルパーの重複リスク**

計画では src/lib/achievements/date.ts を新規作成するとしているが、既存の crossGameProgress.ts に getTodayJst() 関数がある。また、MEMORY.mdには「UTCの日付取得をtoolsライブラリにスコープ化済み」との記録がある。実績システムの date.ts は crossGameProgress.ts の getTodayJst() をインポートして使う（再エクスポートする）か、共通のヘルパーに統合すべき。計画にはこの選択肢が記載されている（「getTodayJst再エクスポートまたは独自実装」）が、重複を避けるためにインポートを推奨とすべき。

**(3) recordPlayの冪等性の定義に注意**

計画では「同日内の同一コンテンツに対する重複呼び出しでdailyProgressは変化しない。totalStats.perContent.countは加算する」としている。これは意図的な設計判断として理解できるが、totalStats.perContent.countが同日内に何度もインクリメントされると、「累計50回利用」等のバッジ条件が意図より早く達成される可能性がある。ゲームは1日1回だが、クイズは何度でも受けられるため、クイズを繰り返し受けることでcountが急速に増加する。

これが意図的な仕様（クイズの再受験も1回としてカウントする）であれば問題ないが、もし「ユニークな日数」をベースにしたい場合は設計を見直す必要がある。site-concept.mdのバッジ条件は「累計50回利用」となっており、「回」と記載されているため、re-playのカウントを含む現在の設計で問題ないと判断する。ただし、builder向けにこの仕様を明示的にコメントとして残すべき。

**(4) 7タスク分割は妥当だが依存関係に注意**

タスク1→2→3, 4, 5→6→7 の依存関係は明確。ただし、タスク3（トースト）、タスク4（ストリーク表示）、タスク5（コンテンツ統合）はすべてタスク2に依存しつつ互いに独立なので、並列で実行できる。並列実行可能であることを計画に明記するとスケジュール最適化に役立つ。

**(5) AchievementProvider をlayout.tsxに追加する際のパフォーマンス考慮**

layout.tsxにAchievementProviderを追加すると、全ページでProviderがマウントされ、LocalStorage読み込みが発生する。ツールページや辞典ページなど実績と無関係なページでも初期化コストが発生する。実用上はuseEffectの初期化のみなので軽微だが、将来的にProviderの責務が増えた場合の注意点として認識しておくべき。現時点では許容範囲。

**(6) ダッシュボードページのURLについて**

計画では /achievements としている。site-concept.md のサイト構造図でも「実績（横断機能）」として記載されているため整合性がある。ただし、フッターやヘッダーからのリンク追加について計画に言及がない。ダッシュボードへのアクセス導線を明確にすべき。StreakBadgeからのリンクが主要導線になると思われるが、フッターにも追加するかどうかは明示すべき。

**(7) テスト方針は十分**

ストレージ、エンジン、日付のユニットテスト、コンポーネントテスト、SSR互換性テストをカバーしており十分。テストしないものの列挙も適切。

**(8) contentStatのbestTimeフィールド**

計画の型定義ではContentStatに bestTime フィールドがないが、調査メモ（19cc85b731e）の推奨型定義には bestTime?: number がある。site-concept.mdにも bestTime の記載がある。計画の型定義にbestTimeが含まれていないのは意図的な省略か確認が必要。ゲームのbestTimeを実績システムで追跡する場合は型に含めるべき。ただし、Phase Aのスコープでは不要かもしれないので、任意フィールドとして型に含めておくが実装は後回しにする方針でもよい。

### 改善提案

- Header.tsxへのStreakBadge統合方法（Server Component内でのClient Component利用パターン）を計画に明記すること
- date.tsは既存のgetTodayJstをインポートする方針を推奨とすること
- ダッシュボードページへのナビゲーション導線（フッターリンク等）を計画に含めること
- タスク3, 4, 5が並列実行可能であることを明記すること
