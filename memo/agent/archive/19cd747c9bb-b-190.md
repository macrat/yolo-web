# B-190アーキテクチャ判断のための定量調査依頼

## 背景

漢字カナル（Wordleライクな漢字当てゲーム）のカテゴリ判定にEmbeddingベクトルを導入する（B-190）にあたり、アーキテクチャを2案で検討している。

- **案A: クライアントサイド方式** - Embeddingデータ（384次元、gzip後約600KB）をクライアントにfetchし、評価をクライアントで完結させる。ゲームデータ（kanji-data.json 687KB, schedules 122KB）は現状通りバンドルに含む。
- **案B: フルサーバーサイドAPI方式** - ゲームデータもEmbeddingデータもすべてサーバー側に保持。クライアントはAPIで推測を送り、フィードバックだけ受け取る。パズルスケジュールやターゲット漢字もクライアントに送らない。

前回のリサーチ（メモ 19cd716cc1d）にはOwnerから論理的な欠陥が4点指摘された。特に「1KB未満のAPIレスポンスの遅延を問題視しながら、600KBの初回ロードの遅延を無視している」という非対称性が致命的だった。今回はOwnerが指摘した具体的な調査項目について、定量的なデータに基づいて分析してほしい。

## 調査項目

### 1. 入力バリデーションの即時性はどの程度重要か

現在のevaluateGuessは<1msで同期実行される。サーバーサイドAPIの場合、50-200ms程度の遅延が発生する。

調査してほしいこと:
- ゲームUI（特にフリップアニメーション）の現在の動作タイミングを確認（FeedbackCell.tsx等）。ユーザーが「送信」ボタンを押してからフィードバックが表示されるまでの全体フロー
- フリップアニメーション自体に時間がかかるなら、APIレスポンスをその間に取得できる可能性がある
- Webゲーム/Wordleクローンにおいて、50-200msのフィードバック遅延がUXにどの程度影響するかの一般的知見

### 2. 初回ロードのデータサイズが離脱率に与える影響

現在のクライアントバンドル: kanji-data.json (687KB raw / 134KB gzip) + schedules (122KB raw / 12KB gzip) で計146KB gzip。案Aではこれに加えてEmbeddings (約600KB gzip)が追加され、合計746KB gzip程度になる。案Bでは初回ロードのデータはゲームUI+バリデーション用漢字リスト程度に削減される。

調査してほしいこと:
- Webでの初回ロードサイズと直帰率/離脱率の関係についての定量データや研究
- 特にモバイルユーザー（3G/4G接続）における影響
- ゲームページの場合の許容範囲（一般的なWebページとは異なる可能性）
- 案Aの746KB gzip（5-10MB解凍後）が具体的にどの程度のロード時間になるか推定

### 3. 実装複雑性の比較

案Aと案Bのそれぞれの実装で必要な変更・追加を具体的に比較してほしい。

案Aの追加複雑性:
- Embeddingデータのプリロード機構（embeddings.ts, useEffect, 状態管理）
- ロード完了までの入力無効化UI
- ロードエラー時のフォールバック
- 既存のbuildGameStateの復元フローとの統合

案Bの追加複雑性:
- Route Handler新規作成（evaluate, init/reveal等）
- handleGuessの非同期化（async/await）
- loading/error状態のUI
- Vercel Hobbyプランの制約確認

具体的にどちらが変更量が多いか、どちらがリスクが高いかを評価してほしい。

### 4. ユーザーファネルの観点

このゲームの目標は「毎日遊んでもらう」こと。そのためには:
1. まず初めてサイトに来た人に一回遊んでもらう
2. 一回遊んだ人に楽しいと思ってもらう
3. 楽しいと思った人に翌日も来てもらう

各段階で、初回ロード速度とゲームプレイ中のレスポンス速度がどう影響するかを分析してほしい。特に段階1（初回訪問→初回プレイ）では初回ロードの影響が最も大きいはず。

### 5. 現行のゲームデータサイズの調査

案Bを採用した場合、kanji-data.json (687KB) とschedules (122KB) もサーバーサイドに移動できる。これにより初回バンドルサイズがどの程度削減されるか調査してほしい。
- 現在のNext.jsビルド出力で、kanji-kanaru関連のチャンクサイズを確認
- kanji-data.jsonとschedulesを除いたゲームUIのみのバンドルサイズを推定

## 注意事項

- 結論を出す必要はない（それはPMの責任）。定量データと分析を提供してほしい
- 「どちらでもよい」というぼかしは不要。各調査項目について、案Aと案Bのどちらに有利かを明確に述べること
- 前回のレポートで指摘された論理的欠陥を繰り返さないこと。特に、初回ロードの遅延を無視してAPIレスポンスの遅延だけを問題視する非対称な分析は禁止
- docs/coding-rules.md を確認して、技術制約への適合も確認すること
