## 調査対象

- ゲーム4種: irodori / kanji-kanaru / nakamawake / yoji-kimeru
- クイズ・診断5種: traditional-color / yoji-personality / yoji-level / kanji-level / kotowaza-level

## 1. ゲーム4種のコード構造

### 共通アーキテクチャ

全4ゲームは完全に統一されたアーキテクチャを持つ:

```
src/games/<slug>/
  _components/
    GameContainer.tsx   ← "use client" トップレベルコンポーネント（状態管理の中心）
    ResultModal.tsx     ← ゲーム終了時のモーダル
    StatsModal.tsx      ← 統計表示モーダル
    ...その他UIコンポーネント
  _lib/
    types.ts            ← 型定義（GameState, GameStats, GameHistory）
    storage.ts          ← LocalStorageアクセス（loadStats/saveStats/loadTodayGame/saveTodayGame）
    engine.ts           ← ゲームロジック
    daily.ts            ← 今日のパズル選択
    share.ts            ← SNSシェア
  data/
    *-schedule.json     ← パズルスケジュール
```

### LocalStorageキー一覧

| ゲーム | statsキー | historyキー | statsKeyフィールド |
|--------|-----------|-------------|-------------------|
| irodori | `irodori-stats` | `irodori-history` | `irodori-stats` |
| kanji-kanaru | `kanji-kanaru-stats` | `kanji-kanaru-history` | `kanji-kanaru-stats` |
| nakamawake | `nakamawake-stats` | `nakamawake-history` | `nakamawake-stats` |
| yoji-kimeru | `yoji-kimeru-stats` | `yoji-kimeru-history` | `yoji-kimeru-stats` |

### GameStats型の共通フィールド

全ゲームの`GameStats`型に共通して存在するフィールド:
- `gamesPlayed: number`
- `currentStreak: number`
- `maxStreak: number`
- `lastPlayedDate: string | null`  ← 実績システムが`playedToday`判定に使用

irodoriのみ`gamesWon`がなく、代わりに`averageScore`/`bestScore`/`scoreDistribution`を持つ。

### ゲーム完了の検出方法

各ゲームの`GameContainer.tsx`内でゲーム完了を検出するパターンは統一されている:

```typescript
// useRefで前回statusを記憶し、playing→完了への遷移を検出
const prevStatusRef = useRef(gameState.status);
useEffect(() => {
  if (prevStatusRef.current === "playing" && gameState.status !== "playing") {
    // ゲーム完了！ここが統合ポイント
    const timer = setTimeout(() => setShowResult(true), 600);
    ...
  }
}, [gameState.status]);
```

各ゲームの完了状態:
- **irodori**: `status === "completed"` (勝敗なし、完了のみ)
- **kanji-kanaru**: `status === "won"` または `status === "lost"`
- **nakamawake**: `status === "won"` または `status === "lost"`
- **yoji-kimeru**: `status === "won"` または `status === "lost"`

また、`saveStats()`が呼ばれるタイミング（ゲーム終了時）が統合の最適ポイント。`saveStats`は各ゲーム固有のstorageモジュールに定義されており、ゲーム終了時のみ呼ばれる。

### statsKey登録

`src/games/registry.ts`の`GameMeta`に`statsKey`フィールドがあり、全ゲームのstatsキーが一元管理されている:

```typescript
{ slug: "irodori", statsKey: "irodori-stats", ... }
{ slug: "kanji-kanaru", statsKey: "kanji-kanaru-stats", ... }
{ slug: "nakamawake", statsKey: "nakamawake-stats", ... }
{ slug: "yoji-kimeru", statsKey: "yoji-kimeru-stats", ... }
```

## 2. クイズ・診断5種のコード構造

### 共通アーキテクチャ

クイズは全5種が**単一の共通`QuizContainer`**で動作する（ゲームと異なりクイズ固有のContainerがない）:

```
src/quiz/
  types.ts              ← QuizDefinition / QuizAnswer / QuizPhase の型定義
  registry.ts           ← 全5クイズのレジストリ
  scoring.ts            ← スコア計算（knowledge: 正解数 / personality: ポイント合計）
  _components/
    QuizContainer.tsx   ← "use client" 共通クイズコンテナ（intro→playing→result）
    ResultCard.tsx      ← 結果表示
    QuestionCard.tsx    ← 質問表示
  data/
    kanji-level.ts
    kotowaza-level.ts
    traditional-color.ts
    yoji-level.ts
    yoji-personality.ts
```

### クイズの完了検出方法

`QuizContainer.tsx`内の`phase`stateで管理:
```typescript
const [phase, setPhase] = useState<QuizPhase>("intro"); // "intro" | "playing" | "result"
```

完了は`handleAnswer`（personalityタイプ）または`handleNext`（knowledgeタイプ）で`setPhase("result")`が呼ばれた瞬間が完了ポイント。

**重要**: クイズは現状**LocalStorageを一切使用していない**。各受診の記録が残らず、統計もない。実績システム統合には、LocalStorageへの書き込み処理を新規追加する必要がある。

| 四字熟語性格診断 | `yoji-personality` | personality |
| 四字熟語力診断 | `yoji-level` | knowledge |

// ゲームレジストリからALL_GAMESリストを構築
export const ALL_GAMES: GameInfo[] = allGameMetas.map(...)

各ゲームのstorage.tsに`isStorageAvailable()`関数が重複定義されている。実績システム用共通ライブラリでも同パターンで実装すればよい。

## 4. 実績システムとの統合ポイント分析

### ゲーム4種の統合ポイント

各ゲームの`GameContainer.tsx`内、`useEffect`でstatus遷移を検出している箇所の直後が最適な挿入ポイント:

```typescript
useEffect(() => {
  if (prevStatusRef.current === "playing" && gameState.status !== "playing") {
    // ★ここで実績記録フックを呼ぶ
    recordAchievement(slug, todayStr); // 追加
    const timer = setTimeout(() => setShowResult(true), 600);
    ...
  }
}, [gameState.status]);
```

ただし、`saveStats()`が呼ばれるタイミング（ゲームロジック内）で記録する選択肢もある。`saveStats`は必ずゲーム終了時に呼ばれるため、そこでフックする方が状態管理的に安全。

**irodoriの特殊性**: irodoriのみ`GameContainer.tsx`内の`handleSubmit`で最終ラウンド判定後に`saveStats`を呼ぶ。他3ゲームと同様のuseEffectパターンも使用しているため、統合方法は統一できる。

### クイズ5種の統合ポイント

`src/quiz/_components/QuizContainer.tsx`の`handleAnswer`と`handleNext`内:

```typescript
// personalityタイプ
if (currentIndex + 1 >= quiz.questions.length) {
  setPhase("result");
  recordQuizCompletion(quiz.meta.slug, todayStr); // 追加
}

// knowledgeタイプ
const handleNext = useCallback(() => {
  if (currentIndex + 1 >= quiz.questions.length) {
    setPhase("result");
    recordQuizCompletion(quiz.meta.slug, todayStr); // 追加
  }
}, [...]);
```

### 実績システム用LocalStorageキー

site-concept.md (セクション10)の設計では`achievements`という統合キーを1つ使う方針だが、実装上は既存の各ゲームstatsキーとは分離した新規キーを使う設計が望ましい（既存データとの混在を避けるため）。

推奨キー構成:
- `yolos-achievements` ← バッジ解除記録
- `yolos-streak` ← ストリーク（現在・最長）
- `yolos-daily-progress` ← 日別プレイ記録（コンテンツ別）

### クイズ用statsKey

クイズはゲームと異なり`statsKey`が定義されていない（`QuizMeta`型に`statsKey`フィールドなし）。実績システムでクイズの完了を追跡するには、クイズslugをそのままキーとして使う方式（例: `quiz-{slug}-stats`）か、中央集権的な`yolos-daily-progress`に統合する方式を選ぶ必要がある。

## 5. 実装上の重要知見

### 既存crossGameProgressの活用

実績システムの「今日のゲームプレイ状況」はすでに`crossGameProgress.ts`で`stats.lastPlayedDate`ベースで実装されている。実績システムはこれを拡張する形で作ると既存のNextGameBannerとの整合性が保てる。

ただし、現状の`crossGameProgress.ts`はゲームのみ対象でクイズは含まれていない。実績システムではクイズも含めた全コンテンツの進捗を管理する必要があるため、拡張が必要。

### ゲームとクイズの非対称性

ゲームは「1日1プレイ」（デイリーパズル方式）であり、`lastPlayedDate`で自然に当日プレイ済みを判定できる。一方クイズは何度でも受診できるため、「今日受診した」という記録を別途残す設計が必要。

### Reactの状態管理パターン

全ゲームがuseState + useRefでゲーム状態を管理し、LocalStorageとの同期は関数内で直接行うパターン（Custom Hookなし）。実績記録も同じパターンで、各Containerに直接追加するのが最も整合性が高い。

## 6. まとめ: 推奨統合設計

1. **新規ファイル**: `src/lib/achievements.ts` (実績ロジック) と `src/lib/achievements-storage.ts` (LocalStorageラッパー)
2. **ゲーム統合**: 各`GameContainer.tsx`のuseEffect（status遷移検出）内に`recordContentCompletion(slug, date)`を追加（4箇所）
3. **クイズ統合**: `QuizContainer.tsx`の`setPhase("result")`呼び出し箇所に同関数を追加（1箇所、全クイズ共通）
4. **QuizMeta拡張**: クイズのstatsKeyが必要な場合は`QuizMeta`に`statsKey`フィールドを追加、または実績システム側でslugから生成
5. **crossGameProgress拡張**: クイズを含めた全コンテンツ対応に拡張（または新規の統合プログレスモジュールを作成）

最小実装（ストリーク記録のみ）であれば、変更が必要なファイルは5ファイル（GameContainer×4 + QuizContainer×1）と新規の実績ライブラリのみで実現可能。
