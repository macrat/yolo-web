# 調査結果: ゲームフィードバック遅延の許容範囲

## A. 現在のUIタイミングの正確な把握

### フローの全体像

ユーザーが送信ボタンを押してからフィードバックが表示されるまでの流れ:

1. `GuessInput` コンポーネントがフォーム送信を受け取る
2. `handleGuess` (GameContainer.tsx) が同期的に呼ばれる (< 1ms)
   - 入力バリデーション（1文字か、常用漢字か、重複か）
   - `evaluateGuess()` でフィードバック計算 (< 1ms)
   - `setGameState()` で React state 更新
3. React が再レンダリングし、新しい `GuessRow` と `FeedbackCell` が DOM に追加される
4. 各 `FeedbackCell` に `.cell` CSSクラスが適用され、`flipIn` アニメーションが即座に開始される

### アニメーション時間

`KanjiKanaru.module.css` の定義:

```css
.cell {
  animation: flipIn 0.4s ease-out;
}

@keyframes flipIn {
  0%   { transform: rotateX(90deg); opacity: 0; }
  100% { transform: rotateX(0);     opacity: 1; }
}
```

- **フリップアニメーション: 400ms**
- 遅延なしで即座に開始（`animation-delay` の指定なし）
- 1行に6つの FeedbackCell がすべて同時に flipIn を開始する（個別ずらしなし）

### ゲーム終了時の ResultModal 表示

`GameContainer.tsx` のコード:

```typescript
useEffect(() => {
  if (prevStatusRef.current === "playing" && gameState.status !== "playing") {
    // Delay a bit so the last feedback animation plays first
    const timer = setTimeout(() => setShowResult(true), 600);
    ...
  }
}, [gameState.status]);
```

- ゲーム終了（won/lost）時は **600ms の意図的な遅延** を入れてから ResultModal を表示する
- コメントに "Delay a bit so the last feedback animation plays first" と明記されている
- つまり、400ms のフリップアニメーションが終わる前にモーダルが被るのを防ぐ設計になっている

### 現在のタイムライン（クライアントサイド）

```
t=0ms   : ユーザーが送信ボタンを押す
t=~1ms  : evaluateGuess() 完了、setGameState() 呼び出し
t=~5ms  : React 再レンダリング完了、DOM 更新
t=~5ms  : flipIn アニメーション開始
t=405ms : flipIn アニメーション完了
t=605ms : ResultModal 表示（ゲーム終了時のみ）
```

### サーバーサイドAPI方式にした場合のタイムライン（参考）

```
t=0ms       : ユーザーが送信ボタンを押す
t=0ms       : APIリクエスト送信
t=50~200ms  : APIレスポンス到着
t=~205ms    : React 再レンダリング完了、DOM 更新
t=~205ms    : flipIn アニメーション開始
t=605ms     : flipIn アニメーション完了
t=805ms     : ResultModal 表示（ゲーム終了時のみ）
```

### アニメーション中にAPIレスポンスを待つことで遅延を隠せる可能性

**可能性は限定的。**

現在のアニメーション（400ms）は送信後即座に開始される。サーバーサイドに変えると、フィードバックの色が確定するまでアニメーションが開始できない。

もし「先にアニメーション（グレーでフリップ）→レスポンス到着後に色変更」という2段階アニメーションを実装すれば、最大 400ms の遅延を視覚的に隠すことができる。しかしこれは実装の複雑さが増す上、現在の設計（flipIn で最初から色付きで表示）とは別のアニメーション設計が必要になる。

## B. Webゲームにおけるフィードバック遅延の一般的知見

### Jakob Nielsen の基本原則（NN/G）

人間の認知特性に基づく3段階の閾値 ([出典](https://www.nngroup.com/articles/response-times-3-important-limits/)):

| 遅延 | ユーザー体験 |
|------|-------------|
| 0.1秒以下 | 即座と感じる。追加フィードバック不要 |
| 0.1〜1.0秒 | 遅延に気づくが思考の流れは維持される |
| 1秒以上 | 思考の流れが途切れ始める |
| 10秒以上 | 他のタスクに移りたくなる。進捗表示が必要 |

### ゲームジャンル別の遅延許容範囲

([Latency Thresholds for Usability in Games 論文](https://www.academia.edu/48115148/Latency_Thresholds_for_Usability_in_Games_A_Survey) より)

- **リアルタイムゲーム (FPS等)**: 50ms 以下が理想、100ms を超えると顕著に影響
- **ターン制ゲーム**: 明確な研究は少ないが、一般的な研究では 300ms 程度が低注意タスクの許容範囲、170ms がドラッグ等の視覚フィードバックが重要なタスクの許容範囲

### ターン制ワードゲームにおける特性

漢字カナルのようなターン制ゲームでは、以下の理由でリアルタイムゲームより遅延許容度が高い:

1. **思考時間が長い**: ユーザーは次の漢字を考えるのに数秒かかる。フィードバックの 100-200ms の遅延は相対的に小さい
2. **競争性がない**: 対戦相手がいないため、タイミングの有利不利がない
3. **アニメーションが期待値を形成**: Wordle 系ゲームではフリップアニメーションがあること自体が「間」として受容されている

### NYT Wordle のサーバーサイド実装

NYT Wordle は2022年の NYT 買収後にサーバーサイドに答えを移行しているが、**ゲームプレイのバリデーション自体はクライアントサイドを維持**していることが複数の調査で確認されている ([出典](https://howardlee93.medium.com/breaking-down-wordle-80d2a13f2390))。

- 「今日の答え」はサーバーから取得するが、推測のバリデーションと色付けはクライアントで実行
- この設計によりフィードバック遅延ゼロを維持している

### アニメーションによる遅延隠蔽の実例

Wordle のタイルフリップアニメーションは UX 設計として高く評価されており ([Hacker News 議論](https://news.ycombinator.com/item?id=30436902))、「juicy」なゲーム体験の核心として機能している。アニメーション中に遅延を隠すパターン（先にアニメーション開始、確定後に色変更）は理論上は可能だが、実際の Wordle 系ゲームでの採用例は少ない。

## まとめ: 案A vs 案B の比較

| 観点 | 案A（クライアントサイド: 遅延なし） | 案B（サーバーサイド: 50-200ms遅延） |
|------|-------------------------------------|--------------------------------------|
| フリードバック開始タイミング | 送信後 ~5ms（即座） | 送信後 ~55〜205ms |
| アニメーション完了 | 送信後 ~405ms | 送信後 ~455〜605ms |
| ResultModal 表示（ゲーム終了時） | 送信後 ~605ms | 送信後 ~655〜805ms |
| Nielsen 閾値への適合 | 完全適合（0.1秒以内） | 遅延に気づくが許容範囲内（1秒未満） |
| ターン制ゲームとしての許容度 | 問題なし | 実用上は許容される |
| 遅延隠蔽の可能性 | 不要 | 2段階アニメーションで部分的に隠蔽可能（実装複雑） |

**案Aが有利な点**: 遅延ゼロで最高の応答性。400ms のフリップアニメーション中にAPIを待てる設計的余地は存在するが、現在の実装はフリップ開始に色情報が必要なため追加実装が必要。

**案Bが許容される条件**: 50-200ms の遅延はターン制ゲームとしては Nielsen の「思考の流れを維持できる」範囲（1秒未満）に収まる。ただし、遅延を隠す工夫（ローディングインジケーター、または2段階アニメーション）がなければユーザーは遅延を知覚する可能性が高い。特に日本国内のサーバーを使用した場合、50-100ms のレスポンスは現実的で、200ms は混雑時の上限として想定される。
