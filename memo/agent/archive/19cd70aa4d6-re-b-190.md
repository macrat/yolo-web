# B-190計画レビュー結果

## 総合評価: B+

計画は全体的に良く練られており、技術的に妥当で、緊急修正指示の要件を概ね満たす方向性になっている。ただし、いくつかの重要な懸念点と改善提案がある。

---

## 1. 来訪者にとっての価値 (要改善あり)

### 肯定的な点
- コサイン類似度によるカテゴリ判定は、現行の「部首番号範囲による機械的分類」と比較して、圧倒的にプレイヤーにとって直感的なフィードバックを提供できる可能性が高い。
- 「海」と「川」がclose、「海」と「火」がwrongといった直感的な結果が期待できる。

### 懸念点: 閾値設計の「close率15-25%」の妥当性が根拠不足
計画ではclose率15-25%を「望ましい」としているが、その数値の根拠が「ゲーム性の観点から」という曖昧な記述にとどまっている。

Wordleを参考にすると、5文字26文字種の組み合わせ空間に対するフィードバックの情報量が適切にバランスされている。漢字カナルでは2,136字の候補空間に対して6回の試行で絞り込む必要があり、カテゴリフィードバックが提供する情報量（correctの確率、closeの確率、wrongの確率）が十分かどうかの定量的な検証が必要である。

**提案**: タスク2の閾値実験に以下を追加すること。
- 各閾値候補に対して、1回のカテゴリフィードバックで候補が平均何%に絞り込まれるかを計算する
- これを他のフィードバック列（部首、画数、学年、音読み、訓読み数）と合わせて、6回で候補1字に収束可能かを検証する

### 懸念点: 「correct」が同一文字のみという設計は正しいか
現行ではcategory correctは「同じ部首グループ」を意味していた。新設計ではcorrectが「同一文字のみ（コサイン類似度1.0）」になるが、同一文字であれば既にゲーム自体が勝利で終了するため、**category列のcorrectが事実上プレイ中に出現しないフィードバックになる**。

これはフィードバックの3段階が実質2段階に退化することを意味し、プレイヤーへの情報提供量が減少する。

**提案**: 以下のいずれかを検討すること。
- correct閾値をコサイン類似度0.85以上など、「非常に近い意味」に設定する（例: 海と湖、火と炎がcorrect）
- または、UIやHowToPlayModalで「意味」列は2段階（近い/遠い）であることを明示し、プレイヤーの混乱を防ぐ

## 2. 技術的妥当性

### 肯定的な点
- 実験データに基づくモデル選定（paraphrase-multilingual-MiniLM-L12-v2、分離度0.2536）は説得力がある。
- int8量子化 + base64エンコード + 動的ロードというアーキテクチャは合理的。
- コサイン類似度計算のコスト（128次元で数マイクロ秒）は問題にならない。

### 懸念点: 384次元 vs 128次元の選択に矛盾がある
PM依頼メモ(19cd6fc974a)の「確定事項」には「128次元int8 base64で356KB」と記載されているが、planner計画では「384次元（モデルネイティブ）」に変更し「まず384次元で試す」としている。この変更の判断自体は理解できるが、PMの意図した設計との齟齬があるため、PMが承認するかどうか確認が必要。

また、Vercelデプロイサイズ制限の問題（メモ19cd6a4aa69: Body exceeded 75000kb limit）が既に発生している状況で、384次元（約1.1MB raw）の追加はリスクが高い。

**提案**: 
- 128次元（356KB raw / 約214KB gzip）を第一選択とし、384次元は品質が不十分な場合のフォールバックとする方が安全。
- タスク1で384次元と128次元の両方を生成し、タスク2の閾値実験で品質差を定量評価してから決定する手順にすべき。

### 肯定的な点: 入力パターン選択
実験結果ではパターンC（漢字+meanings）とパターンD（漢字+部首+meanings）の分離度差がごく僅かであり（0.2536 vs 0.2522）、よりシンプルなパターンCを採用した判断は妥当。部首情報はradicalフィードバック列で既に提供されているため、categoryフィードバックに部首の影響を含めると情報が冗長になるリスクもあり、パターンCの選択は理にかなっている。

### coding-rules.md違反チェック
- ランタイム外部API呼び出し: なし（ベクトルは事前生成、fetchは自サーバーのstaticファイル）
- データベース/認証: なし
- 型安全: 新規のembeddings.tsにも型が明示されており問題なし
- 静的優先: ベクトルデータの事前生成はこの原則に合致

**技術制約違反は検出されなかった。**

## 4. リスク評価

### 見落とされているリスク

#### リスク5: ブラウザキャッシュの一貫性
Embeddingデータを更新（閾値変更やモデル変更等）した場合、ブラウザキャッシュに古いデータが残り、プレイヤーごとに異なるフィードバックが返される可能性がある。

**対策提案**: ファイル名にハッシュを含める（例: kanji-embeddings.abc123.json）か、Cache-Controlヘッダーで適切なキャッシュ戦略を設定する。Next.jsのpublicディレクトリの静的ファイルはデフォルトでブラウザキャッシュされるため、意図しないキャッシュが残るリスクがある。

#### リスク6: セーブデータの後方互換性
既存プレイヤーがlocalStorageに保存したゲーム進行状態（GuessFeedback内のcategory値）が、新しいEmbedding方式で再計算した結果と異なる可能性がある。buildGameStateのsaved復元ロジック（GameContainer.tsx L108-128）ではevaluateGuessを再計算するため、保存時と復元時でcategoryフィードバックが変わりうる。

**対策提案**: この不整合はゲーム進行には影響しない（status判定は文字一致で行う）ため、大きな問題ではないが、HowToPlayModalの説明更新と合わせて、プレイヤーが混乱しないよう留意すること。

### 計画に記載済みのリスク評価
- リスク1（Vercelサイズ）: 対策は妥当だが、前述の通り128次元優先の方が安全
- リスク2（int8品質劣化）: r > 0.95の基準は妥当
- リスク3（非同期化複雑性）: 代替設計案（プリロード方式）で十分軽減可能
- リスク4（閾値妥当性）: 対策は記載されているが、前述の通りclose率の定量的根拠が不足

## 5. 代替設計の評価（プリロード方式）

### 強く推奨: プリロード方式を採用すべき

計画末尾に記載された「Embeddingプリロード完了後にのみゲームをマウントする方式」は、本計画で最も優れた設計判断である。以下の理由から、これをメインの設計として採用すべき。

1. **engine.tsのevaluateGuessシグネチャを同期のまま維持できる**: 非同期化は影響範囲が大きく（GameContainer、テスト全般）、変更量に対するリスクが高い
2. **既存テストへの影響が最小**: evaluateGuessが同期のままなら、engine.test.tsの変更はevaluateCategory関連のみ
3. **buildGameStateの復元ロジックが現状のまま動作する**: 非同期化するとuseStateの初期化関数が同期でなければならない制約と衝突し、設計が複雑化する
4. **400-500KBのgzip後データのロード時間は数百ミリ秒程度**: ユーザー体験への影響は軽微

具体的な実装方針（モジュールレベルPromise + getEmbeddings()同期関数）も合理的。ただし、以下の点に注意が必要。

**注意**: ロード未完了時にgetEmbeddings()が例外をスローする設計は、GameContainer以外のコードパスから呼ばれた場合にランタイムエラーの原因になりうる。ロード状態をReactの状態管理で確実にガードし、ロード完了前にevaluateGuessが呼ばれないことを型レベルで保証する設計が望ましい。

## 6. その他の改善提案

### HowToPlayModalの更新が不十分
計画ではHowToPlayModalの「意味カテゴリ」の説明を「漢字の意味的な近さ」に変更するとあるが、具体的な説明文案が示されていない。プレイヤーにとって最も重要な情報（「どういう基準でclose/wrongが決まるのか」）を分かりやすく伝える説明文案をタスク3の仕様に含めるべき。

### 量子化方法の記述の正確性
「各次元について、ベクトル全体の最大絶対値で割って[-1,1]に正規化し、127を掛けてint8に丸める」という記述がある。コサイン類似度は方向のみに依存するため一律正規化で十分という説明は正しいが、正規化前のベクトルが既にL2正規化済み（paraphrase-multilingual-MiniLM-L12-v2のデフォルト出力）であることを前提とすべき。モデル出力がL2正規化済みであれば、各次元の値域は概ね[-1, 1]に収まるため、127倍してint8にする方法は妥当。ただし、ごく稀に値が範囲外に出る場合のクランプ処理を明記すべき。

- Embeddingベクトル直接利用のアプローチ自体は適切

- 入力パターンC（漢字+meanings）の選択は合理的
- プリロード方式の代替設計は優れている

### 修正すべき点（優先度順）
1. **correct閾値の再設計**: 同一文字のみ=correctでは実質2段階に退化する問題を解決すること
2. **128次元を第一選択に**: Vercelサイズ制限リスクを考慮し、384次元は品質検証後のオプションとすること
3. **プリロード方式をメイン設計に**: 代替案ではなくメインの設計として採用すること
4. **close率の定量的根拠**: 情報量とゲーム性の両面から適切な閾値範囲を定量的に検証すること
5. **プレイスルーテストの具体化**: 合格基準を明確にすること
