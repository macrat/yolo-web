# B-186計画: 四字熟語きめるデータ拡充・差別化強化

## 1. 目的

**誰のため**: 四字熟語きめるの日常的なプレイヤー（メインターゲット: 手軽で面白い占い・診断を楽しみたい人のうち、デイリーゲーム習慣を持つ層）

**何のため**: 
- デイリーパズルとしての致命的な品質問題（101件で1年間に同じ四字熟語が複数回出現）を解消する
- 漢字カナルとの差別化を強化し、四字熟語きめる独自の魅力を明確にする
- リテンション（継続訪問）を支えるデイリーゲームとしての品質を漢字カナルと同等水準に引き上げる

## 2. 全体方針

### データ拡充
- 101件 → 400件以上を目標とする（最低365件で1年間リピートなし）
- 難易度分布の是正: 難易度3（上級）を5件→60件以上に増強
- 目標分布: 難易度1=約160件, 難易度2=約160件, 難易度3=約80件

### 差別化強化（漢字カナルとの差別化）
漢字カナルは「1文字の漢字」を多属性フィードバック（7軸）で当てるゲーム。四字熟語きめるは「4文字の単語」を位置フィードバックで当てるゲーム。この根本的な違いを活かし、以下の方向で差別化する:

1. **難易度選択の導入**（3段階: 初級/中級/上級）
2. **スキーマ拡充による情報の充実**（origin/structureフィールド追加）
3. **ヒントシステムの拡充**（四字熟語特有の属性を活用）

### やらないこと
- サーバーサイドAPI化（B-193で別途対応）
- フィードバック軸の多属性化（漢字カナルのアプローチと重複するため避ける。位置フィードバックは四字熟語きめるの個性）

## 3. タスク分割

### タスク1: データ生成スクリプト作成とデータ拡充

**内容**: 四字熟語データを101件→400件以上に拡充する

**アプローチ**:
- `scripts/generate-yoji-data.ts` を新規作成する
- 四字熟語の百科事典等の公開一覧を参考に、よく使われる四字熟語をAIで生成・検証する方式を取る（漢字カナルではKANJIDIC2という外部辞書データを使ったが、四字熟語には同等のオープンデータが存在しないため、AI生成+人間的検証のハイブリッドアプローチを取る）
- 各エントリにはyoji, reading, meaning, difficulty, category に加え、新フィールド origin, structure を含める
- 既存101件は維持し、新規データを追加する形にする
- 生成後にバリデーションを実施: 重複チェック、漢字4文字チェック、読みの妥当性チェック

**スキーマ変更**:
```
追加フィールド:
- origin: string  （出典区分: "漢籍" | "仏典" | "日本語由来" | "故事" | "その他"）
- structure: string （構造パターン: "対義" | "類義" | "因果" | "修飾" | "並列" | "その他"）
```

**辞典への影響**: site-concept.mdで辞典は削除対象だが、現時点では辞典コードがまだ存在する。スキーマ追加はフィールド追加のみなので既存の辞典コードは壊れない（参照していないフィールドが増えるだけ）。辞典側の型定義（`src/dictionary/_lib/types.ts`のYojiEntry）にも同じフィールドを追加する必要がある。

**難易度の判定基準**:
- 難易度1（初級）: 小学校で習う、日常会話でよく使う四字熟語（例: 一石二鳥、四面楚歌）
- 難易度2（中級）: 中学〜高校レベル、ニュースや書籍で見かける四字熟語（例: 朝令暮改、臥薪嘗胆）
- 難易度3（上級）: 専門的・文語的、漢検準1級以上レベル（例: 韋編三絶、削足適履）

**カテゴリバランス**: 10カテゴリそれぞれに最低30件以上（合計400件なら各40件前後）

**成果物**: 
- `scripts/generate-yoji-data.ts`
- 更新された `src/data/yoji-data.json`（400件以上）

### タスク2: 難易度選択機能の実装

**内容**: 漢字カナルと同様の難易度選択UIを導入する

**アプローチ**:
- 漢字カナルの `DifficultySelector.tsx` を参考にする
- 3段階: 初級（難易度1のみ）、中級（難易度1+2）、上級（全難易度）
- 難易度別にパズルスケジュールを分離（3ファイル）
- LocalStorageに選択した難易度を保存
- ゲーム統計・履歴も難易度別に管理
- シェアテキストに難易度を含める

**影響範囲**:
- `GameContainer.tsx`: 難易度state追加、パズル取得ロジック変更
- `_lib/types.ts`: Difficulty型追加、GameState/GameStatsに難易度フィールド追加
- `_lib/daily.ts`: 難易度別パズル取得対応
- `_lib/storage.ts`: 難易度別ストレージキー
- `_lib/share.ts`: シェアテキストに難易度反映
- 新規: `DifficultySelector.tsx` コンポーネント
- `StatsModal.tsx`: 難易度別統計表示
- `HowToPlayModal.tsx`: 難易度説明追加

**成果物**: 難易度選択が機能する四字熟語きめる

### タスク3: パズルスケジュール再生成

**内容**: 拡充データに基づくスケジュールを難易度別に再生成する

**アプローチ**:
- `scripts/generate-puzzle-schedule.ts` を拡張するか、四字熟語きめる用のスクリプトを別途作成する（既存スクリプトは漢字カナル専用）
- 漢字カナルと同様の方式: Mulberry32 PRNG + Fisher-Yatesシャッフル
- 難易度別3ファイル x 730日分（2年分）
- 各難易度の出題プール:
  - 初級: 難易度1のデータのみ（約160件、730日で各約4.5回 → 重複はあるが十分分散）
  - 中級: 難易度1+2のデータ（約320件、730日で各約2.3回）
  - 上級: 全データ（約400件、730日で各約1.8回）

**成果物**: 
- `scripts/generate-yoji-schedule.ts`（新規または既存拡張）
- `src/games/yoji-kimeru/data/yoji-schedule-beginner.json`
- `src/games/yoji-kimeru/data/yoji-schedule-intermediate.json`
- `src/games/yoji-kimeru/data/yoji-schedule-advanced.json`
- 既存の `yoji-schedule.json` は削除

### タスク4: ヒントシステム拡充

**内容**: 新規追加フィールド（origin, structure）をヒントとして活用し、ゲーム体験を向上させる

**アプローチ**:
- 現在のヒント開示タイミングを調整:
  - 常時: 難易度（既存）
  - 2回目の推測後: 構造パターン（新規。例: "対義構造"）
  - 3回目の推測後: 読みの最初の文字（既存、タイミング維持）
  - 4回目の推測後: 出典区分（新規。例: "漢籍"）
  - 5回目の推測後: カテゴリ（既存、タイミング維持）
- HintBar.tsxの更新

**成果物**: 段階的ヒント表示が5段階に拡充されたHintBar

### タスク5: ResultModal拡充と辞典連携

**内容**: 結果画面で四字熟語の出典・構造情報を表示し、学習体験を向上させる

**アプローチ**:
- ResultModalに origin と structure の情報を表示
- 四字熟語の背景知識を1行で提示（既存のmeaning表示の下に追加）
- これにより「毎日ひとつ四字熟語を学べる」という価値が加わり、リテンション動機が強まる

**成果物**: 情報が充実したResultModal

### タスク6: HowToPlayModal更新とテスト

**内容**: 遊び方説明を更新し、全体の動作を確認する

**アプローチ**:
- HowToPlayModalに難易度選択の説明を追加
- ヒントシステムの説明を更新（5段階ヒント）
- 全体の統合テスト: 各難易度でのゲームプレイ動作確認
- 既存テスト（`__tests__/`）の更新

**成果物**: 更新されたHowToPlayModal、全テスト通過

## 4. 実行順序

```
タスク1（データ生成スクリプト・データ拡充）
  ↓
タスク2（難易度選択機能） + タスク3（スケジュール再生成）[並列可能]
  ↓
タスク4（ヒントシステム拡充）
  ↓
タスク5（ResultModal拡充）
  ↓
タスク6（HowToPlayModal更新・テスト）
```

タスク1が完了しないと他のタスクに着手できない（データが基盤）。タスク2と3は相互依存があるが、型定義を先に決めれば並列可能。タスク4-6は独立性が高い。

## 5. 注意事項

### 技術的制約（coding-rules.md準拠）
- クライアント完結を維持（サーバーサイドAPIは使わない）
- 外部API呼び出し禁止（データはビルド時埋め込み）
- TypeScript型安全の徹底（anyの使用禁止、新しいフィールドは型定義に反映）
- LocalStorageのみ使用可

### データ品質
- 生成した四字熟語データは必ずバリデーションを実施する
- 実在しない四字熟語が混入しないよう、生成後にチェックリストを適用する
- 読みの正確性（例: 「臥薪嘗胆」→「がしんしょうたん」）を検証する
- meaningフィールドは簡潔かつ正確に（辞典的記述）

### 辞典との共有データ
- `src/data/yoji-data.json` は辞典機能と共有されている
- スキーマにフィールドを追加するだけなので、辞典側の既存機能は壊れない
- ただし `src/dictionary/_lib/types.ts` のYojiEntry型にも同じフィールドを追加すべき
- 辞典は将来削除予定だが、現時点で壊さないことが重要

### LocalStorageの既存データ互換性
- 難易度選択導入により、ストレージのキー構造が変わる
- 既存プレイヤーの統計データが失われないよう、マイグレーション処理を実装する
- 既存データは「中級」（デフォルト、全難易度混合に最も近い）に帰属させる

### バンドルサイズ
- データが101件→400件に増加するため、JSONのサイズ増を確認する
- 1件あたり約100〜150バイトとして、400件で約40〜60KB（gzip後は大幅に圧縮される見込み）
- 漢字カナルの2,136件データ（131KB gzip）と比較して問題ないレベル
