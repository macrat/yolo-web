### 5. 具体的な修正ポイントのリスト

#### 問題点 1: trust_level フィールドが存在しない【必須修正】

フロントマターに `trust_level: "generated"` が存在しない。cycle-86の修正実績と `docs/blog-writing.md` の品質チェックリストに基づき追加が必要。

配置場所: `series_order: null` の直後（seriesの直後の位置）

#### 問題点 2: updated_at は記事修正後に更新が必要【修正時対応】

今回の修正で本文に実質的な変更を加える場合、`updated_at` を修正コミット直前に `date` コマンドで取得した時刻に更新する必要がある。現在は `published_at` と同一値のまま。

#### 問題点 3: 記事の対象読者との整合性【確認事項】

記事のタグは「チートシート, Web開発, 設計パターン」であり、`blog-writing.md` の品質チェックリストには「docs/targets/ で定義されたターゲットユーザーの関心事と記事テーマが合致している」とある。

この記事のテーマ（HTTPステータスコードの選び方）は、サブターゲット「AIエージェントやオーケストレーションに興味があるエンジニア」の `search_intents` には含まれていない。ただしそのターゲットはエンジニアであり、REST API設計はその周辺知識として合致する。現状の記事内容はエンジニア向けとして適切であり、この点は大きな問題ではない。

#### 問題点 4: まとめセクションの末尾段落が記事テーマから逸脱している【内容の改善】

まとめセクションの最後の段落:

> APIのリクエスト/レスポンスで使うJSONの整形には[JSON整形ツール](/tools/json-formatter)、URLパラメータのエンコードには[URLエンコード・デコードツール](/tools/url-encode)が便利です。すべてブラウザ上で動作し、入力データがサーバーに送信されることはありません。

`blog-writing.md` の「記事本文の全セクションがタイトルの約束するテーマに関連している（タイトルと無関係なセクションがない）」というルールに照らすと、ツールの宣伝文句（「すべてブラウザ上で動作し、入力データがサーバーに送信されることはありません」）は記事テーマと直接関係がなく、不自然なセールストーク的な文章になっている。関連ツールへのリンク自体は有益だが、プライバシーに関する補足説明はこの文脈では唐突である。

#### 問題点 5: 「はじめに」セクションの構成がやや冗長【読みやすさの改善】

「はじめに」セクションは、免責文・問題提起・「この記事で得られるもの」・チートシートへの導線の4つが詰め込まれており、情報密度が高い。特に最後の一文「各コードの一覧や詳細な説明は[HTTPステータスコード チートシート](/cheatsheets/http-status-codes)にまとめています。本記事はその中でもとくに...」という記述は、記事の価値を自ら矮小化している印象がある。本記事が「チートシートの補足資料」ではなく「独立した学習記事」であることが伝わりにくい。

#### 問題点 6: 「よくある間違い」小見出しが401/403セクションにのみ存在し構成が不統一【軽微】

401/403のセクションには「よくある間違い」という小見出しがあるが、他のセクション（400/422、301/302）にはない。構成の統一感という観点では、セクション間の見出し構造が不揃いである。ただしこれは各セクションの内容量の差による自然な結果でもあり、必須の修正ではない。
