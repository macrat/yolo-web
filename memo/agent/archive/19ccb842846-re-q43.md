## 1. 誰のため・何のため

**メインターゲット**: 手軽で面白い占い・診断を楽しみたい人（site-concept.md セクション2）

**提供する価値**:
- 「占いの結果に笑う」という他サイトにない体験（品質軸1: ユーモア・意外性）
- 「自分の悩みや性格について、まさかの角度からの発見」（品質軸2: 自己発見の納得感）

### 2-1. 今日のユーモア運勢（日替わり占い）

#### コンセプト
質問なし。ページを開くだけで「今日のあなたの運勢」が表示される。日付とユーザー固有シードを組み合わせた決定論的乱数で結果を選ぶため、同一ユーザーには同じ日に同じ結果が表示されるが、ユーザーごとに異なる結果が出る。翌日には必ず変わる。

【修正】指摘7対応: 全ユーザー同一結果→ユーザーごとに異なる結果に変更。LocalStorageに初回訪問時にランダムなユーザーシードを保存し、日付シードと組み合わせる。

#### 結果データ構造
各結果は以下の要素で構成される。60パターン用意し、日付+ユーザーシードで1つ選ぶ。

```
{
  id: string,
  title: string,           // 運勢名（斜め上）
  description: string,     // 運勢の解説（ユーモア）
  luckyItem: string,       // ラッキーアイテム（斜め上）

  rating: number,          // 運勢スコア（1-5の星、ただし微妙な数値も許容）
}
```

【修正】指摘8対応: 「斜め上の相性診断」との差別化を明確にする。逆張り運勢診断の核は「一般的な占いが言いそうなことの逆転構造」。具体的には各結果文に「普通の占いなら○○と言うところだが、実は△△」という逆転のフレームを必ず含める。一方、斜め上の相性診断は「人間ではない意外な存在との相性」という発見がコア体験であり、逆転構造は使わない。

### 3-2. 日替わり占いの技術詳細

- `src/lib/achievements/date.ts` の `getTodayJst()` を再利用してJST日付を取得
- `daily-fortune.ts` に hashDate(), mulberry32(), getUserSeed() を実装
- getUserSeed(): LocalStorageに "yolos-fortune-seed" キーでランダムな数値を保存。初回訪問時にMath.random()で生成。以降は同じ値を使用
- getDailyFortune(): hashDate(getTodayJst()) + getUserSeed() を組み合わせてシード生成 → mulberry32 → 60パターンから1つ選択
- DailyFortuneCard はクライアントコンポーネント（"use client"）
- useEffect 内で日付+ユーザーシード計算 → 60パターンから1つ選択 → 表示
- 結果表示時に recordPlay("fortune-daily") を呼ぶ
- SSR/SSG時のhydration mismatchを避けるため、初期表示はローディング状態にし、useEffectで結果を確定する

**既存バッジへの影響分析と対応方針**:

| バッジ | 影響 | 対応方針 |
|--------|------|----------|
| 全制覇の序章（全コンテンツ1回ずつ） | 達成条件が9種→13種に増加。既存で達成済みのユーザーも新コンテンツ4種を利用するまで未達成に「見える」 | 許容する。新コンテンツ追加時は条件が増えるのが自然。バッジの解除（earned）は保持される（engine.tsはバッジをunlockしたら消さない仕組み） |
| 真の探求者（全コンテンツ10回ずつ） | 同上。fortune-dailyは10日利用が必要 | 許容する。fortune-dailyのリテンション装置としてむしろ好ましい |
| 今日の全制覇（1日全種類利用） | fortune-dailyが追加されるが、日替わり占いは1回見るだけなので負担は軽い | 許容する |
| 診断コンプリート | QUIZ_IDSを参照するため、新3クイズの追加でQUIZ_COUNT=8に増加 | 許容する。countQuizzesUsed()がQUIZ_IDSを参照するので自動的に正しく動作 |
| 診断デビュー | 影響なし（1つ以上で達成） | 対応不要 |

**重要**: engine.tsの既存実装ではバッジのearned状態は一度trueになったら戻らない。したがって既存ユーザーの達成済みバッジが剥奪されることはない。ただしバッジ一覧画面で「全制覇の序章」の進捗率が変動して見える可能性がある（例: 9/9 → 9/13）。これはUI上の表示の問題であり、バッジ自体は取得済みのまま保持される。

## 5. 品質基準（完成条件）

### 品質軸1: ユーモア・意外性
- [ ] 全結果文を読んで「くすっと笑える」か確認
- [ ] 結果文が予想外の切り口を含んでいるか
- [ ] 不快・攻撃的な内容がないか（constitution.md Rule 2）
- [ ] 達成困難アドバイスが「やれたらすごい」の笑いで、「やれよ」の圧迫感がないか
- [ ] 達成困難アドバイスの結果文が「導入→核心アドバイス→オチ」の3パート構造になっているか
- [ ] 逆張り運勢診断の全結果文に「一般的な占いなら○○だが…」の逆転構造が含まれているか

### 品質軸2: 自己発見の納得感
- [ ] 質問と結果に論理的なつながりがあるか（こじつけ感がないか）
- [ ] 結果パターンに偏りがないか（特定結果が出やすすぎないか）
- [ ] ポイント配分が適切か（primary=2, secondary=1の配分が全結果に均等か）
- [ ] 達成困難アドバイスの質問が間接的な状況質問になっているか（直接悩みカテゴリを聞いていないか）

### 品質軸3: 体験の完成度
- [ ] 全占い・診断が正常に動作するか
- [ ] 日替わり占いが日付で正しく切り替わるか
- [ ] 日替わり占いがユーザーごとに異なる結果を表示するか
- [ ] シェア機能が動作するか（X/LINE/コピー）
- [ ] OGP画像が正しく生成されるか
- [ ] 実績システムと連携しているか（recordPlayが呼ばれるか）
- [ ] モバイルで快適に操作できるか
- [ ] trustLevel と trustNote が設定されているか

### 品質軸4: 日替わり占い50パターンの品質
- [ ] 10パターンずつのバッチレビューを実施したか
- [ ] 各パターンのdescriptionが3文以内かつ80文字以内の推奨を守っているか
- [ ] NGパターン（ダジャレのみ、意味不明、不快、長すぎ）が含まれていないか
- [ ] luckyItemとluckyActionがtitleのテーマと連動しているか

## 6. 4コンテンツの差別化まとめ

【修正】指摘8対応: リリース後の評価基準も含めて差別化を明確化。

| コンテンツ | コア体験 | 笑いの構造 | 差別化のキー |
|-----------|---------|-----------|------------|
| 日替わり占い | 毎日違う運勢を見る楽しさ | 身近なものの占い的再解釈 | 質問なし・日替わり・リテンション |
| 逆張り運勢診断 | 予想の裏切り | 「普通ならこう言うが実は逆」の逆転構造 | 逆転のフレーム |
| 達成困難アドバイス | 正論だが実行不可能なアドバイスへの共感 | 「確かにそうだけどできない」のギャップ | 真面目な口調×非現実的な提案 |
| 斜め上の相性診断 | 意外な存在との相性の発見 | 「なぜこの存在と?」という意外性 | 人間以外の相性対象 |

**リリース後の評価基準**: 逆張り運勢診断と斜め上の相性診断の利用率・シェア率を比較し、体験が類似していると判断される場合（利用率の差が20%以内かつシェア率が同傾向）、片方を別コンセプトに差し替えることを検討する。

## 7. 注意事項

- 技術制約は `.claude/rules/coding-rules.md` を直接読むこと。本メモの記述を鵜呑みにしない
- ユーモアの質が最重要。結果文は「友達に見せたくなるレベル」を基準とし、基準に満たないものはリリースしない
- 日替わり占いの60パターンは量より質。最初は30パターンでもよいが、全パターンのユーモア品質を確保すること
- 既存QuizDefinitionの型を変更しない。達成困難アドバイスのadviceはdescriptionに統合、注釈はrecommendationに格納する
- Q43の各コンテンツのmeta.relatedLinksで相互リンクを設定し、回遊を促進する
