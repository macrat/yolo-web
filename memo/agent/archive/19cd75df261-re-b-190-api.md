# B-190 フルサーバーサイドAPI計画 レビュー結果

## 総合評価: A-（承認、軽微な修正推奨）

前回レビュー（19cd71f661f）の指摘3点がすべて適切に反映されており、アーキテクチャの一貫性が大幅に改善された。来訪者にとっての価値（初回ロード速度の劇的改善 + 意味的フィードバックの品質向上）は明確であり、技術的にも妥当な計画である。以下、観点別に報告する。

---

## 1. 来訪者にとっての価値（合格）

初回ロードの改善効果は非常に大きい。

- 現状: page-specific 687KB(kanji-data.json) + 122KB(schedules) = 約800KB以上がクライアントにダウンロードされている
- 計画後: 約6KB(joyo-kanji-set) のみ。他ゲーム並の50-67KB gzip圏内に収まる見込み

フィードバック遅延（50-200ms）はターン制ゲームにおいて許容範囲であり、ユーザー体験の劣化は最小限。ローディング状態の表示設計（初期化中、guess送信中）も含まれており、UX上の配慮がある。

---

## 2. 前回レビュー指摘3点の反映（合格）

- **(a) HintBar方式の統一**: /api/kanji-kanaru/hints エンドポイントを新設し、targetKanjiへの直接参照を排除。設計が一貫している。合格。
- **(b) ResultModalのtargetKanji依存解消**: ゲーム終了時のevaluateレスポンスにKanjiEntryの表示用フィールド全体（character, onYomi, kunYomi, meanings, examples）を含める設計。ResultModalの現在の使用フィールド（35-38行目）をすべてカバーしている。合格。
- **(c) lookupKanjiの軽量化**: kanji-data.json全体ではなく常用漢字2,136字のSet（約6KB）に置き換え。バリデーションの即座性を維持しつつバンドルサイズを大幅削減。合格。

---

## 3. コールドスタート対策（要確認）

PM追加メモ（19cd7585a88）で3つの対策案が提示されたが、計画本文では「数百msは許容範囲。ローディングインジケータで対応」（リスク1）としか記載されていない。PM提示の3案のうちどれを採用するか、あるいは「対策3のみ（初回推測時のローディング表示）で十分」という判断なのかが不明確。

**推奨**: 計画の現在の記述は実質的に案3（ローディング表示）に該当しており、ターン制ゲームの特性上これで十分と考える。しかし、PMメモへの回答として「案3を採用し、案1（Edge Runtime）・案2（プリウォーム）は不要と判断した」旨を計画に明記すべき。builderがPMメモを読んで追加実装すべきか迷う可能性がある。

**重大度**: 低。計画の実質的内容に問題はなく、記述の明確化の問題。

## 4. セーブデータ復元（合格、注意点あり）

feedbacksのフル保存方式は適切。後方互換フォールバック（旧形式 → N回APIコール）も用意されている。旧データ復元時の表示色変化リスクも明記されており（リスク4）、許容範囲の判断も妥当。

**注意点**: storage.tsの現在のGameHistory型（57-63行目）にはfeedbacksフィールドが存在しない。計画3jで追加予定だが、型変更時にGameHistory[string]のオプショナルフィールドとして追加する必要がある（feedbacks?: GuessFeedback[]）。これは計画に記載済みだが、builderが見落としやすい箇所なので、型定義の変更を具体的に示しておくとよい。

## 5. 技術的妥当性（合格、指摘2件）

### 指摘A: guessNumber管理の曖昧さ（中）

計画3aの「注: ゲーム終了判定にはguessCountも必要」の部分で、guessNumberをRequestに含める方式が「要検討」として残されている。ゲーム終了判定（targetKanjiを返すかどうか）はセキュリティ上重要な判断であり、builderの裁量に委ねるべきではない。

**推奨**: Request に guessNumber: number（1-indexed）を含める方式を確定させること。計画本文の「最もシンプルな設計は...」と書かれた案をそのまま正式採用すればよい。サーバー側でguessCountを管理する方式は、ステートレスなRoute Handlerの設計思想に合わない。

### 指摘B: evaluateのレスポンスにradicalのヒント値がない（低）

現在のHintBarは strokeCount, onYomi数, kunYomi数 の3つを表示しており、hints APIの設計もこの3つを返す。しかし、将来HintBarを拡張して部首のヒント（例: 部首の画数帯）を追加する可能性がある場合、hints APIの拡張性が制限される。現時点では問題ないが、hints APIのレスポンス型を拡張しやすい形にしておくとよい。これは「やるべき」ではなく「意識しておくべき」レベル。

## 6. coding-rules.md 準拠確認（合格）

- **Rule 1（静的最優先、クライアント優先、サーバーは必要に応じて）**: 687KB + 1,087KB のデータをサーバーサイドに移すのは「複雑な機能やバンドルサイズが大きくなる機能は、サーバーコンポーネントやAPIルートで実装する」に完全に合致。バリデーション（6KB）をクライアントに残すのも「コンパクトな機能であれば、速度のためにクライアントサイドで実装する」に合致。適切な判断。
- **Rule 2（ユーザーを危険にさらさない）**: 自前Route Handler。外部API呼び出しなし、DB不使用、認証不要。準拠。
- **Rule 3（シンプルで一貫した設計）**: 関心の分離が明確（サーバー: 評価ロジック+データ、クライアント: UI+バリデーション+表示）。準拠。
- **Rule 4（可読性）**: 問題なし。
- **Rule 5（型安全の徹底）**: EvaluateRequest, EvaluateResponse, HintsResponse の新規型が定義されており、適切。
