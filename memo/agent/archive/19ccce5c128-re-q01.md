# Q01音楽性格診断 実装計画レビュー結果

## 総合評価: B+

全体として、来訪者にとって十分な価値を提供できる計画になっている。コンテンツの企画力（3軸設計、8タイプの個性的な名前、相性機能の差別化）は高く、結果テキストの品質もサイトコンセプトに合致している。ただし、修正必須の指摘が2件あるため、それらを修正すればA評価に到達する。

## 修正必須（2件）

### M1. スラグから計画番号「q01-」を除去すること

Ownerから「Q43やQ01のような計画番号をスラグに使うな、ユーザーにとって無意味で混乱する」という指示があった。現在の計画では以下の箇所すべてに `q01-music-personality` が使われているが、`music-personality` に変更すべき。

影響範囲:
- ファイル名: `q01-music-personality.ts` -> `music-personality.ts`
- meta.slug: `"q01-music-personality"` -> `"music-personality"`
- badges.tsへの登録ID: `"quiz-q01-music-personality"` -> `"quiz-music-personality"`
- URL: `/quiz/q01-music-personality` -> `/quiz/music-personality`
- 相性シェアURL: 同様に修正
- ブログ記事タイトル案からも「Q01」を削除

なお、既存のQ43系コンテンツ（`q43-contrarian-fortune`等）も同じ問題を抱えているが、それは本計画の範囲外。本計画ではQ01について正しいスラグを使うこと。

### M2. ポイント配分の偏りを修正すること

計画では「各タイプ5問中2〜3回」とあるが、実際に集計すると主ポイント(2pt)供給回数は以下の通りで、4〜6回の範囲にばらつきがある:

| タイプ | 主ポイント(2pt)出現数 | 最大獲得可能pt |
|---|---|---|
| festival-pioneer | 6回 | 12pt |
| karaoke-healer | 6回 | 12pt |
| solo-explorer | 5回 | 10pt |
| bgm-craftsman | 5回 | 10pt |
| midnight-shuffle | 5回 | 10pt |
| lyrics-dweller | 5回 | 10pt |
| playlist-evangelist | 4回 | 8pt |
| repeat-warrior | 4回 | 8pt |

festival-pioneerとkaraoke-healerはplaylist-evangelistとrepeat-warriorに比べて主ポイント機会が1.5倍ある。これは診断結果の出やすさに有意な偏りを生む。ランダム回答時にfestival-pioneerとkaraoke-healerが出やすく、playlist-evangelistとrepeat-warriorが出にくくなる。

対策: 各タイプが主ポイントを獲得できる回数を5回に揃える。10問x4選択肢=40スロットのうち、主ポイントスロットは10問x4=40個。8タイプで均等に割ると各5回。festival-pioneerとkaraoke-healerをそれぞれ1回減らし、playlist-evangelistとrepeat-warriorをそれぞれ1回増やすことで均衡が取れる。

## 推奨（4件）

### R1. QuizContainer拡張の案Aについて、既存クイズへの影響を明確にすること

ResultCardに `children` propを追加する案Aは合理的だが、refパラメータの処理をQuizContainerに組み込む場合、Q01固有のロジック（相性マトリクスの参照、CompatibilitySectionの描画）がQuizContainerに入り込むリスクがある。

提案: QuizContainerには「結果フェーズで追加のReactNodeを表示する」汎用的な仕組みだけを入れ、Q01固有のロジックはページコンポーネント（`/quiz/[slug]/page.tsx`）側で組み立てること。具体的には:
- QuizContainerに `resultExtra?: React.ReactNode` propを追加
- Q01のページではsearchParamsからrefを取得し、CompatibilitySectionをresultExtraとして渡す
- QuizContainerは結果フェーズでresultExtraをResultCardの下に描画するだけ

これにより、QuizContainerは汎用的なままQ01固有機能に対応でき、将来の他クイズにも相性機能等を追加しやすくなる。

ただし現在の `/quiz/[slug]/page.tsx` はServer Componentであり、searchParamsの取得はServer側で可能だが、相性セクションの描画はClient Componentで行う必要がある。この点の設計をbuilder向けに明確化しておくとよい。

### R2. 相性OGPの方針を明確にすること

計画では「builderの判断に委ねる」としているが、64パターンの静的生成はビルド時間の増加とページ数の増加によるSEO面でのリスク（noindexにすべき大量の結果ページ）がある。Route Handlerベースの動的生成を推奨方針として明記したほうが、builderが迷わない。

### R3. 相性テキストの一部に質の差がある

36通りの相性テキストは全体的に高品質だが、一部に他と比べてやや弱いものがある:

- **bgm-craftsman x karaoke-healer**: 「主導権争いがたまに起きる」で終わっており、他の相性テキストと比較してオチが弱い。もう少し具体的なシーン描写で締めるとよい。
- **repeat-warrior x karaoke-healer**: 「10年後も同じ曲を歌っている二人を想像すると微笑ましい」は悪くないが、他と比べるとやや平凡。

個人結果テキスト8種は全体的に「くすっと笑える」水準に達しており、特にsolo-explorer（「売れる前から知ってたんだけど」）、repeat-warrior（「名作映画は通と呼ばれるのに」）、bgm-craftsman（「雨の日の午後3時・カフェ・やや眠い」）は秀逸。

### R4. ブログ記事タイトルから「Q01」を除去すること

ブログ記事構成案のタイトル「音楽の聴き方で性格がわかる? Q01音楽性格診断の設計裏話」にも「Q01」が含まれている。スラグと同様に、ブログ記事タイトルからも除去すべき。例: 「音楽の聴き方で性格がわかる? 音楽性格診断の設計裏話」

## 良い点

1. **差別化が明確**: 日本語の音楽診断で相性判定機能を持つサービスが事実上存在しないという調査結果に基づき、相性機能を最大の差別化要素として設計している点は戦略的に正しい。

2. **タイプ名のクオリティ**: 「フェス一番乗り族」「リピート戦士」「深夜シャッフル系」「歌詞の一節で生きてる人」などのタイプ名は、ジャンル名ではなく「その人の生き方」として描写されており、シェアしたくなる名前になっている。

3. **結果テキストのトーン**: 「あるある + 自虐 + 意外性」のバランスが良く、サイトコンセプト（笑えて、シェアしたくなる）に合致。特に各テキストの最後の一文（「でも大丈夫」「あなたは正しい」等）が心地よい読後感を残す。

4. **相性機能のURLパラメータ型設計**: DB不要・認証不要でcoding-rules.mdの制約を完全に満たしつつ、友達間のシェア体験を実現する設計は技術的に適切。

5. **builderがそのまま実装できる完成度**: 全10問の質問・選択肢・ポイント配分、8タイプの結果テキスト全文、36通りの相性テキスト全文がすべて計画内に含まれており、builderが追加の企画判断をする必要がない。

6. **3軸設計**: エネルギー/社会性/探索の3軸は、Spotifyの研究を参考にしつつ「ジャンルではなく聴き方」で分類するという独自の切り口であり、競合との差別化に寄与する。

7. **constitution.md適合性**: Rule 2（楽しい）、Rule 4（品質優先）、Rule 5（多様な実験）すべてに適合。ユーモアの中に人を傷つける要素はなく、各タイプの描写はポジティブな自虐で統一されている。
