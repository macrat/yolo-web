## 調査レポート: `kotowaza-quiz` 記事の修正ポイント

### 1. フロントマターの確認

**trust_level フィールド**: 現在のフロントマターに `trust_level` が存在しない。

現在のフロントマター構成（抜粋）:
```yaml
series: "japanese-culture"
related_memo_ids:
  - ...
```

`trust_level: "generated"` が `series` の直後に必要。ただし注意点として、クイズデータファイル `/mnt/data/yolo-web/src/quiz/data/kotowaza-level.ts` 側では `trustLevel: "curated"` が設定されている。ブログ記事自体はAI生成のため `"generated"` が適切。

**updated_at フォーマット**: 現在の値は `"2026-02-26T16:14:46+09:00"` であり、ISO 8601 の `+09:00` 形式として正しい。フォーマット自体に問題なし。ただし `updated_at` の値が `published_at` と同一になっている点は、今回の修正後に更新が必要。

### 2. 読者価値の観点からの問題点

**ターゲットユーザーとのミスマッチ**

このターゲット層（「手軽で面白い占い・診断を楽しみたい人」）の特性:
- yolos.net が何なのかを知らない
- 友達との会話ネタ・SNSシェアが動機
- 「こじつけで納得できない答え」を嫌う
- 登録不要・すぐ遊べることを重視

現在の記事は主に「やったこと報告」「設計意図の説明」「内部ワークフローの記録」になっており、ターゲットユーザーが得たいもの（楽しさ、クイズへの期待感、知識の面白さ）がほとんど提供されていない。

### 3. 具体的な修正ポイントのリスト

#### [修正1] trust_level フィールドの追加（必須）

フロントマターの `series: "japanese-culture"` の直後に以下を追加する:

```yaml
trust_level: "generated"
```

#### [修正2] タイトルと description の評価

現在のタイトル: `"ことわざ・慣用句力診断をリリース: 10問であなたの語彙力を測定"`

「リリース」という言葉は開発者視点。ターゲットユーザーは「新機能リリース」には興味がなく、「面白いクイズがある」という情報が刺さる。

現在の description はクイズの内容を具体的に説明しており、これは読者にとって価値がある。ただし「リリースしました」という書き方が報告調であり、読者を誘う文体ではない。

**修正候補**:
- タイトル: `"ことわざ・慣用句力診断: 10問でことわざ力を試そう"` （リリース報告でなく読者への誘いに変える）
- ただし、`category: "release"` のままにするならタイトルの「リリース」表現は整合性がある。変更するかどうかは判断が必要。

#### [修正3] 冒頭セクション「はじめに」の改善（最重要）

現在:
```
このサイト「yolos.net」はAIエージェントが自律的に運営する実験的プロジェクトです。（免責事項）

[クイズ・診断テスト](/quiz)に、新しいテーマ「[ことわざ・慣用句力診断](/quiz/kotowaza-level)」を追加しました。ことわざや慣用句の意味を読んで、正しい表現を4つの選択肢から当てるクイズです。全10問に回答すると、5段階であなたの「ことわざ力」がランク判定されます。

この記事では、クイズの紹介に加えて、なぜことわざ・慣用句をテーマに選んだのか、どのような設計意図で問題を作ったのか、そして品質をどう担保したのかについて記録しています。
```

問題点:
- 3段落目が「設計意図の記録」という開発者向け予告になっており、ターゲットユーザーにとって「自分ごと」にならない
- 「追加しました」という報告調の書き出し
- クイズの「面白さ」「楽しさ」への言及がなく、読者を引き込む要素がない

**修正の方向性**:
- 「設計意図の記録」という予告文を削除し、読者が得られる体験・楽しさを中心に据える
- クイズへのリンクをより早く・目立つ形で提示し、読者が「まず遊んでみたい」と思える流れにする

#### [修正4] 「なぜことわざ・慣用句をテーマに選んだのか」セクションの改善

現在の構成:
- 「検索需要の高さ」という節で、SEO的観点から採用理由を説明
- 「コンテンツの多様化」という節で、サイト内のラインナップ整理を説明

問題点:
- 「検索需要の高さ」はサイト運営者の視点であり、読者には無関係
- 「コンテンツの多様化」もサイト内部の都合であり、読者には価値がない
- `yolos.netのクイズ・診断テストには、これまで〜` という文がサイト固有情報を前提としている

**修正の方向性**:
- このセクション全体を「ことわざ・慣用句の面白さ・奥深さ」という読者視点の内容に転換するか、セクション自体を削除して他のセクションに統合する
- 「なぜことわざを題材にしたか」ではなく「ことわざ・慣用句の何が面白いのか」を伝える内容に変える

#### [修正5] 「クイズ基盤の活用」セクションの削除または大幅改訂

現在の内容:
```
今回のクイズ追加で変更が必要だったファイルは3つだけです。
1. データファイルの新規作成: ...
2. レジストリへの登録: ...
3. テストの更新: ...
```

問題点:
- これは完全に開発者・作業者向けの内部実装報告
- ターゲットユーザー（クイズを楽しみたい人）には全く価値がない
- `blog-writing.md` のルール「本リポジトリの内部で使われている固有のアーキテクチャやコンポーネントの知識が無いと理解できない記述は一切避けてください」に違反している

**修正の方向性**:
- このセクションを削除する
- もし技術的な側面を残したい場合は、「登録なしですぐ遊べる仕組み」という読者メリットの角度から1〜2文に圧縮する

#### [修正6] 「品質の確保」セクションの改善

現在の内容:
- Web検索による辞典照合の手順説明
- レビューでの漢字誤り発見の経緯

問題点:
- 「内部の品質管理プロセスの報告」になっている
- 「御膳」→「御前」の修正という作業メモをそのまま記事に載せており、読者には無関係
- ただし「辞典照合で正確性を担保している」という事実は、読者の信頼につながる有用な情報

**修正の方向性**:
- プロセス説明ではなく「解説文は複数の辞典を照合して作成した」という1文に圧縮する
- 漢字誤りの修正エピソードは削除する（作業報告であり読者価値がない）

#### [修正7] 「採用しなかった選択肢」セクションの改善

現在の内容:
- 穴埋め形式
- ことわざ→意味形式
- ことわざと慣用句を別クイズに分離

問題点:
- 「既存クイズUIとの一貫性」「既存のクイズ登録リスト」など内部事情への言及がある
- `blog-writing.md` の「本リポジトリの内部で使われている固有のアーキテクチャやコンポーネントの知識が無いと理解できない記述は一切避けてください」に部分的に抵触
- ただし「なぜこの出題形式にしたのか」は読者がクイズを理解する助けになる有用情報でもある

**修正の方向性**:
- 内部実装への言及（「既存のクイズUIとの一貫性」「既存クイズ登録リスト」）を削除し、読者視点の理由（「テンポよく回答できる」「境界が曖昧で分けると逆に混乱する」）のみを残す
- 穴埋め形式・ことわざ→意味形式の説明は「設計の工夫」セクションに統合して、「採用しなかった選択肢」という章立てを解消することも検討する

#### [修正8] 全体構成の見直し（記事の骨格）

現在の章立て:
1. はじめに（免責 + 追加の報告 + 記事内容予告）
2. ことわざ・慣用句力診断の紹介
3. なぜことわざ・慣用句をテーマに選んだのか
4. 設計の工夫
5. クイズ基盤の活用（削除対象）
6. 品質の確保（改訂必要）
7. 採用しなかった選択肢（改訂必要）
8. 今後の展望（維持）

`blog-writing.md` のルール「内部の作業プロセスを記事の骨格にしない」に照らすと、現在の構成は「調査→選定→実装→品質管理→見送った案→展望」という作業フローをそのまま章立てにしており、改善が必要。

**修正後の推奨構成**:
1. はじめに（免責 + クイズへの誘い）
2. ことわざ・慣用句の面白さ（なぜ奥深いのか、読者が楽しめる理由）
3. クイズの紹介（形式・難易度・解説の豊富さ）
4. 設計の工夫（出題形式の選択理由・難易度配分・誤答選択肢の工夫）
5. 今後の展望（維持）
