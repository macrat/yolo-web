# B-190 アーキテクチャ判断のための定量調査レポート

## 調査スコープ

漢字カナルのEmbeddingベクトル導入（B-190）における案A（クライアントサイド方式）と案B（フルサーバーサイドAPI方式）のアーキテクチャ比較。コードベースの実測値と外部調査データに基づく定量分析。

---

## 1. 現行ゲームUIのタイミング詳細（調査項目1）

### フリップアニメーションの実装

`FeedbackCell.tsx` が適用する `.cell` CSSクラス:
```
animation: flipIn 0.4s ease-out;
```

`@keyframes flipIn`: `rotateX(90deg) -> rotateX(0)` の0.4秒アニメーション。6つのフィードバックセルが**同時に**アニメーションする（段階的ディレイなし）。

### 送信からフィードバック表示までのフロー

1. ユーザーが「送信」ボタンクリック（または Enter）
2. `GuessInput.handleSubmit()` が `handleGuess(input)` を同期呼び出し
3. `evaluateGuess()` が純粋関数として <1ms で実行
4. Reactの `setGameState()` が呼ばれ、次レンダリングでセルが表示される
5. セルが描画されると同時に `flipIn` アニメーション（0.4s）が開始
6. ゲーム終了時: `setTimeout(() => setShowResult(true), 600)` で結果モーダル表示

**重要な発見**: フリップアニメーション（0.4s）はフィードバック表示後に実行されるアニメーションであり、評価結果が出るまでの「待ち時間」とは別物。現行設計では送信→フィードバック表示まで <10ms（同期処理）で、アニメーションはその後に付加される演出。

### API方式（案B）におけるアニメーションとの関係

案Bでは送信後にAPIレスポンスを待ってから `setGameState()` を呼ぶ必要がある。
- ウォームAPI（50-150ms）: フィードバック表示が50-150ms遅れる。アニメーション自体は開始が遅れるだけで見た目は似ている可能性あり
- コールドスタート（平均859ms）: フリップアニメーション完了後（400ms）もユーザーがさらに待つ必要がある。明確にラグとして知覚される

### フィードバック遅延に関する一般的知見

- **100ms未満**: ユーザーにとって「即時」と感じられる（NN/g Nielsen基準）
- **100-300ms**: 知覚可能。不快感は少ないが存在する
- **300ms以上**: 明確なラグとして認識され、インタラクティブゲームでは UX 劣化
- **400ms（ブリンク閾値）**: 「押せなかった？」と感じ始める境界

**案Bへの含意**: ウォームAPI（50-150ms）はJakob Nielsenのガイドライン上問題ない。ただし yolos.net のような低トラフィックサイトでは、**1日数回の最初のリクエストがコールドスタート（平均859ms）になる可能性が高い**。コールドスタート時は明確なラグが発生する。

## 2. 初回ロードサイズと離脱率（調査項目2）

### 実測バンドルサイズ（npm run build の実測値）

ビルドは途中でSGSGマニフェスト書き込みエラーで失敗するが、チャンクファイルは生成済み。`kanji-kanaru.html` から参照チャンクを特定して実測:

| チャンク | 用途 | Raw | Gzip |
|---------|------|-----|------|
| `73a84ed24992b4aa.js` | kanji-data.json（2136エントリ） | 515 KB | **123 KB** |
| `973b03ef84212cbd.js` | 3難易度スケジュール（2190エントリ） | 86 KB | **11 KB** |
| `d0cfaf4ab9c42373.js` | ゲームロジック | 28 KB | 9 KB |
| その他フレームワーク等 | React, Next.js等 | 613 KB | 187 KB |
| **合計** | | **1,242 KB** | **330 KB** |

### 案別の転送サイズと推定ロード時間

| シナリオ | Gzip合計 | 4G (50Mbps) 転送時間 | 3G (3Mbps) 転送時間 |
|---------|----------|---------------------|---------------------|
| 現行ベースライン | 330 KB | 52ms | 860ms |
| **案A（Embedding 600KB追加・int8量子化前提）** | 930 KB | 145ms | 2,423ms |
| **案A（Embedding 1,400KB追加・float32実測想定）** | 1,730 KB | 270ms | **4,520ms** |
| **案B（kanji-data+schedules削除）** | 197 KB | 31ms | 512ms |

※ 転送時間のみ。実際のTTI（Time-to-Interactive）はDNS解決、TLS、JSパース・実行を含む。

### 重要な訂正: Embeddingのgzipサイズ

メモ依頼では「gzip後約600KB」と記載されているが、実際の計算では:
- **float32 (非量子化)**: 384次元 × 2136漢字 × 4バイト = 3.13MB raw。gzip後 **1.2-1.8 MB**（float配列は圧縮率が低い）
- **int8量子化**: 0.78MB raw。gzip後 **500-700KB** → 600KBはこのケースに相当

600KBの推定は int8 量子化を前提とする。float32のまま格納する場合は1.4-1.8MBとなり、Case Aのインパクトは上表より大きくなる。

### 離脱率への影響（外部データ）

- Googleが公表: モバイルで**3秒超のロードで53%のユーザーが離脱**
- HubSpot研究: 0-5秒の範囲で1秒あたり**コンバージョン率4.42%低下**
- モバイルで5秒以内に表示されるサイトは平均的なサイトより**70%長いセッション時間**

**案Aへの適用**: 3G環境で案Aがeager loadingの場合、転送時間だけで2.4〜4.5秒。HTMLのパース・JS実行を加えると3〜7秒のTTIとなる。**Googleの3秒閾値を大幅に超過する可能性がある**。

**ゲームページの許容範囲について**: WPI大学の2024年研究では、ゲームにおける「良い」ロード時間は約15秒、「悪い」は40秒とされるが、これはリッチゲームの事例。デイリーパズルのような軽量HTMLゲームにはWebページの基準（3秒）が適用される。

**案Bへの適用**: 初回ロード197KB gzip。3G (3Mbps) で512ms転送。パース・実行を含む現実的TTIは1.5〜2.5秒。3秒閾値を余裕でクリア。

## 3. 実装複雑性の比較（調査項目3）

### 案A（クライアントサイドEmbedding）

**新規作成・変更が必要なファイル**:
1. `embeddings.ts`（新規）: fetchとキャッシュ、コサイン類似度計算
2. `GameContainer.tsx`: `embeddingsReady` 状態管理、`useEffect` でプリロード
3. `handleGuess` の修正: embeddingsReadyを確認してからguess受付
4. `GuessInput.tsx`: embeddingsロード中の入力無効化UI
5. `categories.ts`: 整数比較からベクトル類似度計算への書き換え
6. ローディングUI（新規）: embeddings取得中の視覚的フィードバック
7. エラーフォールバック: fetch失敗時の対応（カテゴリ評価スキップ or 再試行）

**アーキテクチャリスク**:
- `evaluateGuess` は現在純粋同期関数。embeddings依存の非同期化が必要
- localStorage復元フロー（`buildGameState`）はすでにローカルの `allKanjiData` を使うため、embeddings準備前の復元には制約が生じる
- 遅延ロード（lazy）にする場合、「embeddingsが揃う前にユーザーが推測を送信した場合の挙動」を設計しなければならない

### 案B（フルサーバーサイドAPI）

**新規作成・変更が必要なファイル**:
1. `src/app/api/kanji-kanaru/evaluate/route.ts`（新規、動的Route Handler）: 入力検証、評価実行、GuessFeedbackの返却
2. `GameContainer.tsx`: `kanjiDataJson` と scheduleのimport削除、`handleGuess` の非同期化
3. ローディング状態のUI: guess送信中のローディングスピナー
4. エラー状態のUI: API失敗時のリトライ、エラーメッセージ
5. `storage.ts` の設計変更: `guessChars[]` のみ保存する現行設計は継続可能だが、ページロード時の復元に**N回のAPI呼び出し**が必要になる。これを避けるためには `GuessFeedback[]` のフルデータをlocalStorageに保存する設計変更が必要

**アーキテクチャリスク**:
- localStorage復元フロー（`buildGameState`）の大幅な変更が必要
  - 現行: `guessChars` を保存、`evaluateGuess()` で再計算（サーバー不要）
  - 案B採用後: `GuessFeedback[]` をlocalStorageに保存する設計に変更、またはページロード時にN回API呼び出し
  - N回API呼び出しはページロード時のUX劣化を招くため、GuessFeedback保存が現実的
- 4つの既存Route Handlerはすべて `force-static`。案Bは初の動的Route Handlerとなる

### 変更量の比較

案Aと案Bはほぼ同等の変更量。ただし**リスクの性質**が異なる:
- **案Aのリスク**: embeddings fetch失敗時のフォールバック設計、遅延ロードとゲーム開始の競合
- **案Bのリスク**: localStorage復元ロジックの再設計が必須、コールドスタートによるUX劣化

## 4. ユーザーファネル分析（調査項目4）

### 段階1: 初回訪問 → 初回プレイ（最影響度が高い段階）

初回ロード速度が最も支配的な要因。カジュアルゲームのD1リテンション（業界平均26-27%）を下回ると回収できない。

| 指標 | 現行ベースライン | 案A（eager） | 案A（lazy） | 案B |
|------|-----------------|--------------|-------------|-----|
| 初回JS gzip | 330 KB | 930〜1,730 KB | 330 KB | 197 KB |
| 4G TTI概算 | 1.5-2s | 2-4s | 1.5-2s | 0.8-1.2s |
| 3G TTI概算 | 3-5s | **5-8s** | 3-5s | **1.5-2.5s** |
| 3秒閾値リスク | 中（3G境界） | 高（3G確実超過） | 中 | 低 |
| 有利な案 | - | B | A(lazy) ≈ baseline | **B** |

案B（初回gzip 197KB）は初回訪問体験で明確に優位。特に3G/低速環境のユーザーへの影響が大きい。

**案A lazy の限界**: 初回ロード自体はbaselineと同じだが、embeddingsロード中（3G環境で2-5秒）はカテゴリ評価が使えない。UIブロックが必要になる場合、ユーザーがゲームを試みる頃にまだローディング中の可能性がある。

### 段階2: 初回プレイ → 楽しい（コアUX）

推測評価のレスポンス速度が支配的。

| 指標 | 現行ベースライン | 案A | 案B（ウォーム） | 案B（コールド） |
|------|-----------------|-----|-----------------|-----------------|
| 評価レイテンシ | <1ms（同期） | <1ms（embeds loaded後） | 50-150ms | 859ms（平均） |
| flipIn 400ms 内に収まるか | N/A | N/A | ほぼ収まる | 収まらない |
| 知覚 | 即時 | 即時 | ほぼ即時 | 明確なラグ |
| 有利な案 | - | **A = baseline** | A≈B | A > B |

yolos.netの現在のトラフィック規模では、1日の最初のプレイセッション（あるいは数時間ぶりのセッション）でコールドスタートに遭遇する確率が高い。毎日プレイするユーザーが毎回「最初の1手」でコールドスタートに遭遇するリスクがある。

### 段階3: 楽しい → 翌日もプレイ（リテンション）

2回目以降の訪問では、JSチャンクはブラウザキャッシュに存在する。

- **案A（返訪時）**: embeddingデータもブラウザキャッシュに存在 → 追加コストゼロ。ゲーム体験はベースラインと同一
- **案B（返訪時）**: APIはリクエスト毎にウォーム/コールドの状態次第。毎日プレイするなら1日1回以上アクセスがあるため、ウォームの確率は上がる

## 5. 案B採用時の削減効果（調査項目5）

### kanji-kanaru ページの実測チャンク内訳

kanji-kanaruページ（`/games/kanji-kanaru`）が実際にロードする12チャンクの測定値:

| チャンク | 内容 | Raw | Gzip |
|---------|------|-----|------|
| `73a84ed24992b4aa.js` | kanji-data.json（2136漢字） | 515 KB | 123 KB |
| `973b03ef84212cbd.js` | 3難易度スケジュール合計 | 86 KB | 11 KB |
| `d0cfaf4ab9c42373.js` | ゲームロジック（GameContainer等） | 28 KB | 9 KB |
| その他10チャンク | React, Next.js, 達成システム等 | 612 KB | 188 KB |
| **ページ合計** | | **1,242 KB** | **330 KB** |

### 案B採用時の削減

案Bで `kanji-data.json` と schedules をサーバーサイドに移動した場合:

- **削減**: 123 KB + 11 KB = **134 KB gzip**（現行ページロードの40.5%）
- **残存**: ゲームUI + フレームワーク = **197 KB gzip**
- Raw換算: 1,242 KB → 641 KB（48%削減）

**ゲームUIのみのバンドルサイズ推定**: ゲームロジックチャンク（9KB gzip）+ フレームワーク（188KB gzip）= **197KB gzip** が案Bの初回ロードになる。

## 6. 補足: Vercel Hobby プランと coding-rules.md の確認

### Vercel Hobby プラン制約

- Serverless Function 呼び出し: 100,000回/日（3M回/月）
- 案BでDAU=10,000、4推測/人 = 40,000回/日（制限の40%）
- 現実的なyolos.netのトラフィック水準（DAU数百〜数千）では問題なし

### coding-rules.md 適合性

**Rule 1「静的最優先、クライアント優先、サーバーは必要に応じて」**:
- 案A: 完全準拠（クライアント完結）
- 案B: 準拠。Rule 1は「バンドルサイズが大きくなる機能はサーバーで実装する」を明示的に許可している。kanji-data.json（123KB gzip）の除去はこれに該当する

**Rule 2「ユーザーを危険にさらさない」**:
- 案B APIは漢字1文字を受信してフィードバックを返すだけ。外部API呼び出しなし、PII処理なし。準拠。
