# B-187 調査結果: 既存ゲーム4種のUI/UX現状調査と改善ポイント

## 調査概要

依頼メモ(19cda1c9145)に基づき、4ゲーム（irodori / kanji-kanaru / nakamawake / yoji-kimeru）のUI/UX現状を調査し、改善ポイントを優先度付きで特定した。

---

## 1. ゲームUI構造の全体把握

### 共通アーキテクチャ

すべてのゲームが以下の共通構造を持つ:

- `GameLayout` (`src/games/_components/GameLayout.tsx`): ページ全体ラッパー。パンくず・TrustLevelBadge・FAQ・シェアボタン・関連コンテンツを統一提供
- `GameDialog` (`src/games/shared/_components/GameDialog.tsx`): 全モーダル（HowToPlay / Result / Stats）の共通基盤。ネイティブ `<dialog>` 要素使用
- `GameShareButtons` (`src/games/shared/_components/GameShareButtons.tsx`): 結果共有ボタン。Web Share API / コピー / X共有 / 画像保存を統合
- `CountdownTimer` + `NextGameBanner`: 結果モーダルに全ゲーム共通で表示される

### 各ゲームのコンポーネント構成

| ゲーム | 独自コンポーネント数 | CSSスタイル方式 |
|--------|------|------|
| irodori | 9個（ColorTarget / HslSliders / RoundResult / FinalResult / ProgressBar等） | 個別CSS Modules |
| kanji-kanaru | 9個（GameBoard / GuessRow / FeedbackCell / HintBar / DifficultySelector等） | 単一CSS Module（KanjiKanaru.module.css） |
| nakamawake | 8個（WordGrid / SolvedGroups / GameControls / SolvedGroups等） | 個別CSS Modules |
| yoji-kimeru | 8個（GameBoard / GuessRow / CharFeedbackCell / HintBar等） | 単一CSS Module（YojiKimeru.module.css） |

## 3. UX上の問題点（操作フロー・レスポンシブ・アクセシビリティ・パフォーマンス）

### 3-1. ゲームヘッダーの非統一（中）

**問題**: `irodori` と `nakamawake` のGameHeaderは同一CSS（円形ボタン・`justify-content: space-between`）を持つが、`kanji-kanaru` と `yoji-kimeru` は別スタイル（`position: absolute; right: 0` のボタン配置）を使用。見た目に微妙な差異がある。

- irodori/nakamawake: ヘッダー中央にタイトル・日付、右端に？・📊ボタン（フレックス行）
- kanji-kanaru/yoji-kimeru: タイトル中央固定・ボタン absoluteで右配置（KanjiKanaru.module.css / YojiKimeru.module.css が完全に同一の構造）

**影響**: 視覚的一貫性は概ね保たれているが、kanji-kanaru と yoji-kimeru はまったく同一のCSSを重複して持っている。

### 3-2. アクセシビリティの不足点（中）

現状確認できた点:

- `WordGrid`（nakamawake）: `role="group"` + `aria-label` + `aria-pressed` が適切に設定されている ✅
- `HslSliders`（irodori）: `aria-label` + `aria-valuenow` が設定されている ✅
- `GameDialog`: `aria-labelledby` が設定されている ✅
- ゲームコントロールボタンに`type="button"` が全て設定されている ✅

**不足点**:
1. `GameHeader` のアイコンボタン（?・📊）に `aria-label` があるが、絵文字アイコンのみでテキストなし。フォーカスインジケータのスタイルが CSS上で明示的に定義されていない。
2. `nakamawake`の`WordGrid`内のボタンは`aria-label={word}`を持つが、ゲーム中の「このボタンが何を意味するか」の文脈説明がない。
3. `irodori`の`RoundResult`や`ColorTarget`でカラーパッチを表示する際、色名が回答後まで表示されない（視覚のみに依存する設計）。これはゲーム性の観点で意図的だが、色覚特性のあるユーザーには困難な可能性がある。

### 3-3. キーボード操作の制約（低）

- `kanji-kanaru` / `yoji-kimeru`: テキスト入力フィールドがあるのでキーボード操作は機能する
- `nakamawake`: グリッドのwordButtonはTabキーでフォーカス可能だが、16個を順番に移動するのは大変
- `irodori`: スライダーへのキーボードアクセスは `input[type=range]` で機能する

### 3-4. ローディング・エラー状態のスタイル（低）

`kanji-kanaru`のGameContainerに loading/error 状態があるが、スタイルがインラインで定義されている（`style={{}}`）。他ゲームと一貫性がない。

```
// kanji-kanaru/GameContainer.tsx:401-413
if (loading) {
  return (
    <div style={{ display: "flex", justifyContent: "center", ... }}>
      読み込み中...
    </div>
  );
}
```

### 3-5. `nakamawake`ミス表示の改善余地（中）

現在のミス表示:
```
●●○○ ミス 2/4
```
（`"●".repeat(gameState.mistakes) + "○".repeat(MAX_MISTAKES - gameState.mistakes)`）

「あと1つ」になったときの視覚的警告（例: 赤色表示）がない。ゲームオーバーが近づいていることをより直感的に伝えることができる。

### 3-6. レスポンシブ対応状況

- 全ゲームが max-width: 600px を基準にモバイルファーストで設計されている ✅
- `kanji-kanaru`のGameBoardは `min-width: 340px` を持ち、極小画面でも横スクロールが発生する（7列のグリッドは幅が必要なため設計上妥当）
- `WordGrid`（nakamawake）は480px以下でボタンサイズを縮小するブレークポイントがある ✅

## 4. B-152との関連確認

B-152で指摘されている問題点と現状:

### #11 ナカマワケ: 進行中誤答の永続化不足
- 現状確認: `nakamawake/GameContainer.tsx:290-306`を確認
- 誤答時（`newStatus === "lost"` のときのみ `saveTodayGame` を呼ぶ）
- 正解時は `saveTodayGame` を呼ぶ (`nakamawake/GameContainer.tsx:223-227`)
- **問題確認**: 誤答して `playing` 状態が続く場合（敗北確定前のミス）は保存されない。リロードするとミス数が0に戻る

**UI/UXとの関連**: ミス数の巻き戻りはユーザーに混乱を与える。B-187の改善スコープに含めるべき。

### #15 ナカマワケ: 履歴型と保存値の乖離リスク
- `NakamawakeGameHistory`の型定義: `status: "won" | "lost"` だが、正解グループを全部解いたとき（won）と敗北時（lost）のみ保存されるため、`playing` 状態は保存されない
- 型は適切だが#11の問題（途中状態が保存されない）が問題の本質

### #26 SSR/CSR不整合（ナカマワケ）
- 現状確認: `nakamawake/GameContainer.tsx:92,107` で `shuffleArray(remainingWords)` を `useState` 初期化内で呼び出している
- `shuffleArray` は `Math.random` ベース（`nakamawake/_lib/engine.ts`）
- **問題確認**: SSR時とCSR時でシャッフル順が異なり、hydrationミスマッチのリスクが残る

**UI/UXとの関連**: hydrationミスマッチが起きると画面のちらつきが発生する可能性がある。

### #29 /games 一覧の日付固定化
- 現状確認: `src/app/games/page.tsx:7-14`
- `getTodayFormatted()` がサーバーコンポーネントのビルド時に実行される
- Next.js ISRの場合、ビルド時の日付が長期間表示され続ける可能性がある
- **現在の影響**: Vercel等の静的ホスティングでリバリデーション設定がなければ問題になる

**UI/UXとの関連**: ユーザーが古い日付を見ると混乱する。クライアント側で動的に表示するか、`revalidate` 設定が必要。

**B-152のスコープ判断**: これらはすべてUI/UX改善（B-187）と直接関連する技術的問題であり、同一サイクル内で対応することが望ましい。ただし、スコープ管理上はB-152として別タスクで管理し、B-187の改善と合わせて計画することを推奨。

## 5. CSSスタイルの重複問題（技術的改善）

`KanjiKanaru.module.css` と `YojiKimeru.module.css` は以下のセクションが完全に重複している:
- `.header` / `.headerTop` / `.title` / `.headerButtons` / `.iconButton` / `.headerSub`（ヘッダー部）
- `.hintBar` / `.hintLabel` / `.hintValue`（ヒントバー部）
- `.inputArea` / `.inputRow` / `.inputField` / `.submitButton` / `.errorMessage`（入力エリア）
- `.statsGrid` / `.statItem` / `.statValue` / `.statLabel` / `.distributionRow` 等（統計モーダル）
- 各種アニメーション（`flipIn` / `shake`）

このCSSの重複は、スタイル変更時に2箇所を更新する必要が生じる保守性の問題につながる。

## 6. 改善ポイント（優先度付き）

### P1（高優先度）: B-152関連のバグ修正

これらはUI/UX品質に直接影響するバグであり、B-187の改善作業と平行して対応すべき。

1. **ナカマワケ: 誤答のリアルタイム永続化（#11）**
   - `handleCheck`の誤答パス（`newStatus !== "lost"` の場合も）でミス数を保存するよう修正
   - `NakamawakeGameHistory`の型に `playing` を含めるか、ミス数を別途保存する構造変更が必要

2. **games一覧の日付表示（#29）**
   - `getTodayFormatted()` をクライアントコンポーネントに移動するか、`export const revalidate = 86400` を設定する

### P2（中優先度）: UX改善

3. **ナカマワケ: ミス残り1つ時の警告表示**
   - `mistakeIndicator`でミス数が3/4になった場合に赤色表示するなどの視覚的警告を追加

4. **kanji-kanaru: ローディング/エラー状態のスタイル統一**
   - インラインスタイルをCSS Modulesに移行し、他ゲームと一貫したスタイルにする

5. **ナカマワケ: SSR/CSRハイドレーション整合（#26）**
   - `shuffleArray` を `useEffect` 内またはクライアント側でのみ実行するよう変更

### P3（低優先度）: CSS重複解消と一貫性向上

6. **kanji-kanaru / yoji-kimeru の共通CSS抽出**
   - ヘッダー・入力・統計モーダルのスタイルを `src/games/shared/` 配下の共通CSSに移動

7. **ゲームヘッダーの完全統一**
   - irodori / nakamawake 系と kanji-kanaru / yoji-kimeru 系のヘッダー実装を統合した共通 `SharedGameHeader` コンポーネントを作成

8. **アクセシビリティ強化**
   - ゲームヘッダーのアイコンボタンにフォーカスインジケータCSS追加
   - irodoriでキーボード操作時の色名補助テキスト表示検討（ゲームバランスとのトレードオフあり）
