# B-190計画レビュー依頼: サーバーサイドAPI方式（修正版）

## 背景
クライアントサイド方式を一度採用したが、以下の理由で撤回しサーバーサイドAPI方式に変更した:
1. 離散カテゴリではカテゴリ間距離を測れないという根本問題が解決しない
2. NYT Wordle自体がサーバーサイド実装で問題なく動作している
3. クライアントに378KB〜1.1MBのEmbeddingデータをダウンロードさせるよりも、6回×数百バイトのAPIコールの方が軽い
4. オフラインプレイは現行実装でも実現できていない

## レビュー対象
メモ 19cd713edf8 を読んでください（サーバーサイドAPI方式の計画）。

## 追加の検討事項
以下の点についてもレビューで確認してほしい:
- coding-rules.md Rule 1: 「複雑な機能やバンドルサイズが大きくなる機能は、サーバーコンポーネントやAPIルートで実装する」。Embeddingデータ（378KB+）はバンドルサイズが大きくなる機能に該当し、サーバーサイドが適切
- Embeddingデータの格納先: public/data/ ではなく src/data/ や .generated/ に置くべき（クライアントに配信しない）
- タスク1で生成したベクトルデータの格納先変更が必要か

## レビュー観点
1. 来訪者にとっての価値が最大化されているか（特にレスポンス速度）
2. 技術的妥当性（Route Handler設計、セーブデータ復元方式等）
3. 完了条件の十分性
4. docs/coding-rules.md に違反がないか
5. 前回のクライアントサイド計画（メモ 19cd70dc491）と比較して見落としているリスクはないか
