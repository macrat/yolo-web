# 調査結果: メモページのユーザー体験の網羅的検討

## 調査対象コード

- `src/app/memos/[id]/page.tsx`
- `src/app/memos/thread/[id]/page.tsx`
- `src/app/memos/page.tsx`
- `src/memos/_components/` 配下全コンポーネント
- `src/blog/_components/RelatedMemos.tsx`
- `src/lib/cross-links.ts`
- `src/memos/_lib/memos.ts`
- `.generated/memo-index.json`
- `npm run build` の実行結果

---

## 1. ユーザーのアクセス導線の全体像

ユーザーがメモページにアクセスする導線は以下の5種類：

### 1-a. ブログ記事の「関連メモ」セクション（最重要）
- `src/blog/_components/RelatedMemos.tsx` が `/memos/[id]` へのリンクを生成
- **実際にリンクされているメモ数は16件のみ**（ブログ記事61本のうち related_memo_ids を持つもの）
- これらが「ブログ経由で来訪する可能性が最も高いメモ」

### 1-b. メモ一覧ページ（`/memos`）からの遷移
- MemoCard コンポーネントが `/memos/[id]` へリンク
- スレッド数が2件以上の場合、MemoCard に「N件のスレッド」表示があるが、クリック先はスレッドではなくメモ個別ページ

### 1-c. メモ個別ページ内からスレッドページへ
- MemoDetail コンポーネント: replyCount > 1 の場合のみ「スレッド全体を見る(N件)」バナーを表示し `/memos/thread/[id]` へリンク
- スレッドページ内 MemoThreadView: 個別メモへのリンクは**ない**（スレッド内のメモは全文表示される）

### 1-d. RSSフィード
- `/memos/feed`（RSS 2.0）と `/memos/feed/atom`（Atom 1.0）
- 最新100件のメモを配信。各アイテムは `/memos/[id]` へリンク
- フィードルートは `force-static` 設定

### 1-e. その他（内部参照）
- MemoDetail の「関連ブログ記事」セクション（RelatedBlogPosts）がブログ記事にリンクバック
- /memos ページの robots.txt 設定: `index: false, follow: true` のため検索エンジン経由のアクセスはゼロ

---

## 2. レスポンス速度の影響

### 現在の動作メカニズム
- `generateStaticParams: return []` + `revalidate = false` = **オンデマンド生成 + 永続キャッシュ**
- 初回アクセス時のみサーバーサイドでレンダリング、以降は Vercel Edge Cache から配信

### コールドスタートの影響
- サーバーレス関数は `.generated/memo-index.json`（22.1MB）を nft.json 経由でバンドルに含む
- 関数の初回起動時にこの 22MB JSON を読み込み＋パースするため、コールドスタートが遅い
- モジュールレベルのキャッシュ（`_cachedIndex`）があるため、同一インスタンスでの2回目以降は高速
- Vercel Hobby プランのコールドスタートは一般に 500ms〜2000ms 程度。22MB JSON のパースが加わるため上振れリスクあり

### revalidate = false の意味
- 最初に生成されたHTMLはずっとキャッシュされる（メモは不変なので適切）
- ただし Vercel Edge Cache がパージされた後（デプロイ時など）は再びコールドスタートが発生

### ブログからリンクされている16件を静的生成した場合
- 追加ビルドサイズの推定: 約7.4MB（ブログ1記事あたり約475KB × 16件）
- 75MB 制限内に収まる
- ただしビルド時間は微増（軽微）

## 3. 部分的静的生成の実現可能性

### ビルド時にブログリンク済みメモIDを取得する方法
`getAllBlogPosts()` はビルド時に利用可能なサーバーサイド関数。以下のコードで実現可能：

```ts
// src/app/memos/[id]/page.tsx
export function generateStaticParams() {
  const blogPosts = getAllBlogPosts();
  const linkedMemoIds = new Set<string>();
  for (const post of blogPosts) {
    for (const id of post.related_memo_ids) {
      linkedMemoIds.add(id);
    }
  }
  return Array.from(linkedMemoIds).map((id) => ({ id }));
}
```

ただし thread ページは対象メモIDと threadRootId が異なる場合があるため、`getPublicMemoById` でスレッドルートIDも取得する必要がある。

### 現在のビルド状況（npm run build 実行確認済み）

```
ビルド出力合計: 702MB（.next/全体）
- dictionary/kanji: 520MB（最大）
- blog: 26MB
- memos: 1.7MB（テンプレートのみ、静的HTML無し）
- その他（chunks等）: 約155MB
```

メモセクションは現在 1.7MB のみ（テンプレートファイルのみで、個別メモのHTMLファイルは生成されていない）。

### 全件静的生成した場合の推定サイズ
- 3710件のメモページ + 1186件のスレッドページ = 4896件
- 1件あたり約475KB（ブログページ比較）× 4896 = 約2273MB
- 75MB 制限を大幅に超過（これが今回 return [] にした根拠）

### 16件部分静的の場合
- 約7.4MB 追加。75MB 制限内に十分収まる
- スレッドページは16件のメモに対応するスレッドを静的化すれば追加数は最大16件（ただし多くは同じスレッドを指す可能性あり）

## 4. ローディング体験

### loading.tsx の有無
- `src/app/memos/` 配下に `loading.tsx` は**存在しない**
- `src/app/` 全体を検索しても `loading.tsx` ファイルはゼロ

### 現在の動作
- 動的生成中にユーザーに表示されるのは**ブラウザのデフォルトロードインジケーター（タブのスピナー）のみ**
- コンテンツは一切表示されない状態で待つことになる
- UXとして不十分：コールドスタート時は無反応に見える可能性がある

### 改善余地
- `src/app/memos/[id]/loading.tsx` を作成しスケルトンUIを表示することで、ローディング中も何かを見せられる
- ただし memo は robots: noindex のため、初回アクセスユーザーは少ない（ブログリンク経由が主）

---

## 5. エラーハンドリング・メタデータ

### 存在しないメモIDへのアクセス
- `getPublicMemoById(id)` が null を返した場合、`notFound()` を呼び出す（実装済み）
- スレッドページも同様: `getMemoThread(id)` が空配列を返した場合に `notFound()` を呼び出す
- `src/app/not-found.tsx` は存在しており、主要コンテンツへのリンクを表示する適切なページ

### OGP・メタデータの生成タイミング
- `generateMetadata` が動的に実装されており、ページレンダリングと同時にメタデータ生成
- ただし `robots: { index: false, follow: true }` が全メモページに適用されており、検索エンジンにインデックスされない
- OGPは実際に機能する（SNSクローラーはnoindexを無視して取得するため）
- スレッドページのメタデータ: 同様に動的生成、こちらも `robots: { index: false, follow: true }`

---

## 6. スレッド閲覧体験

### スレッド内ナビゲーション
- スレッドページ（`/memos/thread/[id]`）では全メモを**1ページに全文表示**する（MemoThreadView）
- 前後メモへの個別ページ遷移は不要な設計（スレッド全体が1画面で読める）
- MemoDetail から「スレッド全体を見る」バナーでスレッドページへ誘導

### 速度への影響
- スレッドページも動的生成。大きなスレッド（最大73件）は HTML が長くなる
- 最大73件 × 平均5834バイト（HTML）= 約427KB のコンテンツ。許容範囲内

---

## 7. その他の発見事項

### 最重要: メモ一覧ページの肥大化
- `/memos` の静的 HTML は **1.2MB**、RSC ペイロードは **1.0MB**
- 3722件のメモサマリー（JSON換算で約1.2MB）が全件クライアントサイドの MemoFilter に渡されている
- ページロード時に1MB超のデータをパース・保持するため、特にモバイルで初期表示が遅い
- これは今回の動的ルート化とは別の問題だが、UX的には動的ルート化よりも影響が大きい

### メモはサイトマップ未掲載・検索エンジン非対象
- `src/app/sitemap.ts` にメモページは含まれていない
- 全メモページに `robots: { index: false, follow: true }` が設定
- したがってメモの主要なアクセス元はブログリンクとRSSフィード経由のみ

### メモデータの不変性とキャッシュ設計の整合性
- `revalidate = false`（永続キャッシュ）は「メモは作成後に変更されない」という性質と一致しており適切
- ただし新規メモが追加されてもすでにキャッシュされたURLのHTMLは更新されない（reply_to リンクや replyCount などの動的情報には影響しない実装のため問題なし）

## 総合評価

| 観点 | 現状 | 課題レベル |
|------|------|------------|
| ブログ → メモ遷移の速度 | 16件は動的生成（初回は遅い） | 中（部分静的化で解決可能） |
| メモ一覧の初期ロード | 1.2MB HTML を全件ロード | 高（独立した問題） |
| ローディング状態 | loading.tsx なし、無反応 | 中 |
| 404ハンドリング | notFound() 実装済み | 問題なし |
| OGP/メタデータ | 動的生成、正常動作 | 問題なし |
| スレッド閲覧 | 全文1ページ表示、適切 | 問題なし |
| Vercel制限 | 75MB制限を超えない設計 | 問題なし |

## 推奨事項

1. **ブログリンク済み16件のメモを部分的静的生成**: UXインパクト中、実装コスト低
2. **loading.tsx の追加**: UXインパクト中、実装コスト低（スケルトン表示）
3. **メモ一覧の肥大化対策**: UXインパクト高、実装コスト高（サーバーサイドページネーション等）
