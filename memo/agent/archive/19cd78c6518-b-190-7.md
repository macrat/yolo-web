# B-190タスク7: カテゴリデータ整理と互換性コード簡素化

## 背景
Ownerからの指摘で、以下の2点が見落とされていたことが判明した。

## 作業1: KanjiEntry.categoryフィールドの整理

### 現状
- `KanjiEntry.category: RadicalGroup` (types.ts:10) は1-20の数値で、旧部首グルーピング時代のデータ
- engine.tsの`evaluateCategory`はもはやこのフィールドを使わず、`evaluateSimilarity(guessChar, targetChar)`でEmbeddingベースの評価を行う
- しかし `src/dictionary/_lib/kanji.ts` が `category` フィールドをカテゴリフィルタに使用中
- `src/lib/search/build-index.ts` も `category` を検索インデックスに使用中
- `share.ts` のシェアテキスト生成でも `g.category` を参照

### 作業
- dictionary側で `category` がどう使われているか調査。表示名やUIがあるか確認
- `category` フィールドが辞書機能にとって意味のあるデータか検討
- もし辞書側でカテゴリが重要なら `category` は残すが、ゲーム側での参照が不要なことをコードコメントで明記
- `RadicalGroup` 型名がEmbeddingベースの現在のアーキテクチャで不適切なら、辞書側のみで使うように整理
- kanji-data.json内の `category` 値(1-20)が実際に辞書UIで表示・使用されている場合はそのまま保持。使われていない場合は不要なフィールドを削除できるか検討

## 作業2: 互換性コード（reEvaluateGuesses）の検討と簡素化

### 現状
- GameContainer.tsxに `reEvaluateGuesses` という関数があり、旧形式（feedbacksなし）のセーブデータを読み込んだ際にAPIを順番にコールして再評価する
- `GameHistory.feedbacks` がoptionalになっているのも旧形式互換のため
- このサイトは小規模な実験サイトであり、利用者数はごく少ない

### 作業
- `reEvaluateGuesses` の互換処理を削除し、feedbacksがない旧形式データは新規ゲームとして扱う（ユーザーは最悪1日分の進行中ゲームを失うだけ）
- `GameHistory.feedbacks` をoptionalから必須に変更
- `loadTodayGame` の「lost + guessCount < MAX_GUESSES → playing」マイグレーションも同様に検討（必要性が低ければ削除）
