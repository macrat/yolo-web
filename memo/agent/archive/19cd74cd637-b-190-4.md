# B-190調査4: 現在のゲームバンドルサイズの実測

## 調査目的
案B（フルサーバーサイドAPI）を採用した場合、kanji-data.jsonやschedulesをサーバーサイドに移動することで初回バンドルサイズがどの程度削減されるかを実測する。

## 調査内容

### A. ビルドして実測
1. `npm run build` を実行
2. `.next/static/chunks/` 配下のファイルサイズを調査
3. kanji-kanaru関連のチャンクを特定し、各チャンクのサイズを報告
4. kanji-data.json (687KB raw)、puzzle-schedule-*.json (各40KB raw) がどのチャンクに含まれているか特定

### B. 案Bでのバンドルサイズ推定
kanji-data.jsonとschedulesをバンドルから除外した場合:
- ゲームUIのみのバンドルサイズを推定
- クライアントに残すべきデータ（入力バリデーション用の漢字リスト等）のサイズ

### C. 案A vs 案Bの初回ロード比較
- 案A: 現在のバンドル + Embeddings fetch（約600KB gzip）
- 案B: UI + バリデーションデータのみ（推定値）
- 差分を報告

## 注意
- ビルドの実行が必要。`npm run build` を実行してよい。
- 結論を出す必要はない。実測データを提供すること。
