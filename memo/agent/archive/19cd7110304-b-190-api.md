# B-190計画修正依頼: サーバーサイド評価API方式への変更

## 背景
レビューA-で承認済み（メモ 19cd7101fb8）だが、Ownerから重要なアーキテクチャ提案があった。

## Ownerの提案（PM判断で採用）
Embeddingデータをクライアントにfetchするのではなく、サーバーサイドで保持し、API経由で正誤判定を行う方式に変更する。

### メリット
1. **チート対策**: ターゲット漢字をクライアントに送らない。サーバーが推測を受け取りフィードバックだけ返す。正解がクライアントに一切露出しない
2. **レスポンス改善**: Embeddingデータ（400-500KB）のクライアントダウンロードが不要
3. **設計の簡潔さ**: プリロード、Int8Array操作、コサイン類似度計算がすべてサーバー側に集約。クライアントはAPIを呼ぶだけ

### 設計の方向性
- `src/app/api/kanji-kanaru/evaluate/route.ts` のようなRoute Handlerを作成
- リクエスト: `{ guess: "海", puzzleDate: "2026-03-10", difficulty: "intermediate" }`
- レスポンス: 全フィードバック（文字一致、部首、画数、学年、音読み、訓読み数、カテゴリ）を返す
- ターゲット漢字はサーバーが puzzleDate + difficulty からパズルスケジュールを参照して特定
- Embeddingデータもサーバーサイドのモジュールキャッシュに保持
- coding-rules.md: 自前Route Handlersは許可されている（src/app/api/search-index/route.ts の稼働実績あり）

## 修正すべき範囲
前回承認済みの計画（メモ 19cd70dc491）をベースに、以下を変更:
- タスク3のクライアントサイドEmbedding評価をサーバーサイドAPIに変更
- プリロード方式は不要になる（APIコールに置き換わる）
- GameContainerからevaluateGuessの呼び出しをfetch APIに変更
- ターゲット漢字のクライアント送信を廃止

## レビューでの追加指摘
- Matryoshka Representation Learningの記述は事実誤認（メモ 19cd7101fb8）。128次元切り出しの品質保証はないが、タスク2のランク相関検証でフォールバック可能
