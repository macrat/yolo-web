# B-190 サーバーサイドAPI計画レビュー結果

## 総合評価: B+（条件付き承認）

サーバーサイドAPI方式への変更は技術的に正しい判断であり、クライアントサイド方式の根本的問題（離散カテゴリではカテゴリ間距離を測れない、378KB以上のデータをクライアントに送る非効率性）を適切に解決している。計画全体の構成はよく練られており、タスク1・2で品質検証を挟んでからタスク3に進むフローも妥当。ただし、いくつかの設計上の矛盾と見落としがあるため、以下を修正・確認した上で進めるべき。

---

## 問題点（要修正）

### 問題1: HintBar方式Cの自己矛盾（重大）

計画は方式C（現状維持）を推奨し、「targetKanjiオブジェクト自体はクライアントに存在するが許容」としている。しかし同時に「3d設計で対応する」と述べ、3dではtargetKanjiをnull化する設計を提案している。

現状のGameContainerの実装を確認すると:
- HintBarはgameState.targetKanji.strokeCount等を直接参照している（GameContainer.tsx 352-355行目）
- ResultModalもtargetKanjiの全フィールド（character, onYomi, kunYomi, meanings, examples）を表示に使用している（ResultModal.tsx 31-38行目）

targetKanjiをnull化すると、HintBarもResultModalも動作しなくなる。計画の3dでは「/api/kanji-kanaru/puzzle エンドポイントを追加」する案が出ているが、これは方式Cの「現状維持」と明確に矛盾する。

**修正案**: 方式を明確に1つ選択すること。以下のいずれかを推奨する:
- (a) /api/kanji-kanaru/puzzle エンドポイントを正式採用し、ヒント情報とゲーム終了時のターゲット情報をAPIから取得する設計に統一する。方式Cの記述を削除する。
- (b) チート対策の範囲を明確にする。ターゲット漢字の文字自体をJSバンドルから隠すことが目的であれば、パズルスケジュールJSONから漢字を特定できる現状（puzzle-schedule-*.jsonにkanjiIndexが含まれており、kanji-data.jsonと組み合わせれば特定可能）も同時に対処が必要。

個人的には (a) を推奨する。puzzleエンドポイントの追加コストは小さく、設計が一貫する。

### 問題2: ResultModalのtargetKanji依存が考慮不足

ResultModalはtargetKanjiのcharacter, onYomi, kunYomi, meanings, examplesをすべて表示に使用している。計画の「ゲーム終了時のターゲット漢字開示」ではtargetCharacter（文字列1文字）を返す設計だが、ResultModalが必要としているのはKanjiEntryオブジェクト全体である。

**修正案**: ゲーム終了時のAPIレスポンスにtargetCharacterではなくtargetKanjiEntry（KanjiEntryの必要フィールド全体）を返すか、クライアント側でkanji-data.jsonからlookupする設計にする。後者であればkanji-data.jsonのimportは維持でき、変更が最小限になる。

### 問題3: lookupKanjiの削除は不適切

3dで「lookupKanjiのimportも削除」とあるが、クライアント側のバリデーション（入力が常用漢字かどうかの確認、重複チェック）にはkanjiDataが必要。サーバーに毎回バリデーションを委ねると、無効な入力でもAPIコールが発生する。

現在のGameContainer（246-249行目）では入力直後にlookupKanjiでローカル検証しており、これはUX上重要な即座のフィードバックである。

**修正案**: lookupKanjiとallKanjiDataのimportは維持し、クライアント側のバリデーション（漢字の有効性チェック、重複チェック）は現状のまま残す。evaluateGuessのimportのみ削除する設計にすべき。

## 懸念点（要検討）

### 懸念1: セーブデータ方式Bの後方互換フォールバック

方式Bでは「旧形式のセーブデータ（文字列のみ）が存在する場合、復元時にAPIで再評価するフォールバック」を用意するとある。これ自体は良いが、APIで再評価した結果のcategoryフィードバックは旧方式（離散カテゴリ）と異なる値になる。つまり、旧セーブデータを復元すると表示色が変わる可能性がある。

これはゲーム進行には影響しないが、ユーザーが「さっきまで黄色だったのに白になった」と感じる可能性がある。この挙動は許容範囲内と考えるが、計画に明記しておくべき。

### 懸念2: /api/kanji-kanaru/evaluate のレート制限

不正リクエストやブルートフォース攻撃（全2,136字を総当たりで試す）の防御が計画に含まれていない。ゲームの性質上、1日6回が上限なのでリスクは低いが、APIエンドポイントが公開される以上、最低限のバリデーション（1リクエストあたりの処理頻度）は検討すべき。

ただし、coding-rules.mdのRule 2「外部APIの呼び出し、データベース、認証・ユーザー管理は実装しない」の制約からレート制限の実装は難しいかもしれない。自前Route Handlerでのインメモリカウンタ程度なら許容範囲か。Vercelのサーバーレス環境ではインスタンス間でメモリが共有されないため、実効性は限定的。

**推奨**: 現時点ではレート制限は不要と判断してよい。ブルートフォースで正解を見つけてもゲーム体験が損なわれるのは攻撃者自身であり、他のユーザーに危害は及ばない。

### 懸念3: チート対策の実効性

パズルスケジュールJSON（puzzle-schedule-*.json）はクライアントにimportされており、kanjiIndexが含まれている。kanji-data.jsonと組み合わせれば、APIを呼ばずともターゲット漢字を特定できる。evaluateをサーバーサイドにしてもターゲット漢字の隠蔽は不完全である。

ただし、B-190の本来の目的は「Embedding類似度に基づくカテゴリフィードバック」の実装であり、チート対策は副次的な利点である。チート対策の完全性を追求してパズルスケジュールまでサーバーサイド化するのは過剰だと考える。計画の「チート対策の本質は意味カテゴリの正解ベクトルをクライアントに露出しないこと」という記述は適切。

## 良い点

1. **方式変更の判断根拠が明確**: 離散カテゴリの根本問題、NYT Wordleの先例、データ転送量の比較など、論理的に筋が通っている。
2. **セーブデータ方式Bの選択**: 復元時にN回のAPIコールを避け、オフライン時の表示も維持できる設計は実用的。
3. **タスク1・2のMatryoshka未確認の注記**: 前回レビュー指摘を正しく反映している。
4. **完了条件の充実**: 16→18項目に増え、チート対策テスト、後方互換性、エラーハンドリングが追加されている。
5. **リスク分析の網羅性**: コールドスタート、非同期化に伴うUX変化、オフライン動作、回帰バグの4つの新規リスクが適切に識別されている。
6. **coding-rules.md Rule 1との整合性**: Embeddingデータ（378KB+）をサーバーサイドで処理する判断はRule 1の「複雑な機能やバンドルサイズが大きくなる機能は、サーバーコンポーネントやAPIルートで実装する」に完全に合致している。

## coding-rules.md / constitution.md 準拠確認

- **Rule 1**: サーバーサイドAPI方式はバンドルサイズの大きいEmbeddingデータに対して適切。準拠。
- **Rule 2**: 自前Route Handler（外部API呼び出しなし、DB不使用、認証不要）なので準拠。Embeddingの計算もすべてサーバーサイドJSで完結。
- **Rule 3-5**: 計画段階なので直接的な判定はできないが、設計方針に違反はない。
- **constitution.md**: ゲーム品質の向上（離散カテゴリ→意味的類似度）はRule 4（品質優先）に合致。

## 結論

問題1（HintBar方式の矛盾）、問題2（ResultModalの依存）、問題3（lookupKanji削除の不適切さ）を修正した上で承認すべき。いずれも設計の一貫性に関わる問題であり、builderが作業中に判断に迷う原因になる。修正自体は計画文書の記述変更で対応可能であり、全体のアーキテクチャ変更は不要。
