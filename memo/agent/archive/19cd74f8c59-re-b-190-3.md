## A. 案A（クライアントサイドEmbedding）の変更一覧

### 変更対象ファイル

#### 1. `engine.ts` — `evaluateCategory` の変更（中規模）
現在の実装:
```ts
function evaluateCategory(guessCategory, targetCategory): FeedbackLevel {
  if (guessCategory === targetCategory) return "correct";
  if (areCategoriesRelated(guessCategory, targetCategory)) return "close";
  return "wrong";
}
```
案A後は `areCategoriesRelated` の代わりにベクトル類似度を使う。  
- `areCategoriesRelated` の呼び出し箇所は `evaluateCategory` の1箇所のみ（grep済）。
- 変更箇所: `evaluateCategory` 内の1行。引数型も変わる（RadicalGroup番号ではなくembeddingベクトルが必要か、またはKanjiEntry全体を渡すか）。

#### 2. `categories.ts` — `areCategoriesRelated` の削除（小規模）
- `areCategoriesRelated` は `engine.ts` の `evaluateCategory` からのみ呼ばれている。
- `categorySuperGroups`, `radicalGroupNames` は他から参照されている可能性があるため確認が必要（後述）。
- 削除対象: `areCategoriesRelated` 関数のみ（12行）。`categorySuperGroups`/`radicalGroupNames` は保持可能。

#### 3. 新規: `embeddings.ts` — fetchとデコードとコサイン類似度（大規模・新規）
新たに作るファイル。主な処理:
- `fetch('/embeddings.bin')` または `/embeddings.json` のリクエスト
- base64デコードまたはバイナリパース（Float32Array）
- コサイン類似度計算関数
- キャラクター→embedding インデックス のマッピング
- モジュールレベルのPromiseキャッシュ（1度だけfetch）

推定行数: 80〜120行

#### 4. `GameContainer.tsx` — プリロード状態管理の追加（中規模）
現在は `useState(() => buildGameState(...))` で初期化が同期的に完了している。案A後:
- `embeddingsReady: boolean` の useStateが必要
- `embeddingsError: boolean` の useStateが必要
- useEffect でプリロードをトリガー
- `GuessInput` の `disabled` 条件に `!embeddingsReady` を追加
- ローディングUIの追加（HintBar上部またはGuessInput付近）
- エラー時のフォールバック表示（「カテゴリ評価なしで続行」または「再試行ボタン」）

推定追加行数: 30〜50行

#### 5. `buildGameState` フローとの統合（リスク箇所）
- `buildGameState` はlocalStorageから保存済みゲームを復元し、`evaluateGuess` を再実行する。
- 案Aでは `evaluateCategory` がembeddingを必要とするため、復元時にembeddingsが未ロードだと問題になる。
- 対策: embeddingsロード完了後に `buildGameState` を呼ぶ（= 初期化を遅延させる）か、embeddingなしのフォールバックで復元する。
- これにより初期化フローが複雑化する。現在の同期的な `useState(() => buildGameState(...))` が使えなくなる。

#### 6. エンジンテストの修正（中規模）
- `engine.test.ts` は `evaluateGuess` を直接テストしており、`category` のテストケースが複数ある（行176〜191）。
- embeddingベースに変更すると、テストフィクスチャにembeddingデータが必要になる。fetchをモックするか、テスト専用のembeddingデータを用意する必要がある。
- 現在のテストは純粋関数のみで完結しており変更の影響が大きい。

## B. 案B（フルサーバーサイドAPI）の変更一覧

### 前提確認
現在、`src/app/api/` 配下にRoute Handlerは存在しない（feedのみ）。`src/app/feed/route.ts` は静的生成（`force-static`）。

### 変更対象ファイル

#### 1. Route Handler 新規作成（大規模・新規）
`/api/kanji-kanaru/evaluate` など:
- `kanji-data.json`（671KB）と3つのスケジュールJSON（各41KB）をサーバーサイドでimport
- ゲストリクエストのバリデーション（文字数チェック、常用漢字チェック）
- `evaluateGuess` のサーバー実行
- 正答漢字の秘匿問題: クライアントに正答を渡さないためには、targetKanjiのcharacterをレスポンスから除外する必要があるが、現状のGameStateはclientがtargetKanjiを保持している。APIモデルへの移行で情報設計も変わる。

推定行数: 80〜150行

#### 2. `GameContainer.tsx` の `handleGuess` を非同期化（大規模）
現在:
```ts
const handleGuess = useCallback((input: string): string | null => { ... }, [...]);
```
案B後:
- `handleGuess` が `Promise<string | null>` を返す非同期関数になる
- `GuessInput` に渡すonSubmitの型が変わる → `GuessInput.tsx` も修正必要
- API通信中のloading状態管理: `isSubmitting: boolean`
- ネットワークエラー時のエラー表示と再試行
- APIエラーレスポンスのハンドリング（400/500）

推定追加行数: 50〜80行

#### 3. `GuessInput.tsx` の修正（小〜中規模）
- onSubmit のコールバック型を変更（同期 → 非同期）
- loading中の無効化・スピナー表示

#### 4. `buildGameState` の復元フロー変更（大規模・複雑）
現在はlocalStorageの保存済みguessキャラクターリストからクライアント側で `evaluateGuess` を再実行して復元している。案Bでは:
- 案B-1: 復元時もAPIを叩く（初回マウント時にAPIリクエストが発生）→ネットワーク依存が増える
- 案B-2: feedbackをlocalStorageに直接保存する形に変更 → storage.tsとGameHistoryの型変更が必要
- 案B-3: 初期化用の別エンドポイント（`/api/kanji-kanaru/init`）を用意する
いずれも非自明な変更が必要。

#### 5. `HintBar`/`ResultModal` のデータソース変更
現在は `gameState.targetKanji`（クライアント保持）から `strokeCount`, `onYomi`, `kunYomi` 等を取得している。APIモデルに移行した場合:
- 正答漢字をクライアントに送らない設計なら、HintBarのデータもAPI経由で取得する必要がある
- ResultModal の `targetKanji` の表示も再設計が必要
- ただし「正答を隠す」要件がなければ初期ロード時に正答メタデータを取得するだけで済む（設計判断に依存）

#### 6. `kanji-data.json` と schedules の import 変更
- クライアントバンドルから671KBのJSONを除外できる（バンドルサイズ削減のメリット）
- サーバーサイドで import するため、ビルド設定は変更不要

#### 7. テストの修正（大規模）
- `engine.test.ts` はサーバー環境でそのまま動作するが、APIエンドポイントのテストが別途必要
- fetch/MSWによるAPIモックが必要

## C. リスク比較

### 案Aのリスク

| リスク種別 | 内容 |
|-----------|------|
| 初回ロード遅延 | embeddingsファイルのfetchが完了するまでカテゴリ評価ができない。ファイルサイズ次第（2136漢字×embedding次元数）では数百KB〜数MBになりうる |
| 復元フロー複雑化 | `buildGameState` の初期化が非同期化するため、localStorageからの復元タイミングとembeddingsロードの競合が発生しうる |
| フォールバック設計 | ネットワーク障害時にカテゴリ評価をどう扱うか（degraded mode）の設計が必要 |
| テストの複雑化 | fetchをモックしないと純粋関数テストが書けなくなる |
| バンドルサイズ | kanji-data.json（671KB）はそのまま残る |

**ユーザー影響が出やすい箇所**: ゲーム開始直後（embeddingsロード中）と、低速回線でのエラー時。

### 案Bのリスク

| リスク種別 | 内容 |
|-----------|------|
| 毎回のAPIレイテンシ | 推測ごとにネットワークラウンドトリップが発生。日本のVercelリージョンなら通常50ms以下だが、失敗時は顕著 |
| 復元フロー再設計 | 最も複雑な変更箇所。feedbackをlocalStorageに保存する方式変更か、復元用API追加かで選択肢が分かれ、どちらも非自明 |
| エラー状態 | ネットワーク障害でゲームが操作不能になりうる（handleGuessがAPIに依存するため） |
| 正答情報の設計 | 現状はtargetKanjiをクライアントが保持しているが、APIモデルではそれが変わる。HintBar/ResultModal/shareのデータフローを全面的に再検討する必要がある |
| テスト範囲の拡大 | APIレイヤーのテストが新規に必要 |

**ユーザー影響が出やすい箇所**: 推測送信ごと（レイテンシ感知）、ネットワーク障害時（ゲーム停止）。

**テストが困難な箇所**: 案Bは復元フローが設計の選択に依存するため、実装前に確定しないとテスト設計もできない。

## D. coding-rules.md との適合性

### Rule 1 の原文
> 複雑な機能やバンドルサイズが大きくなる機能は、サーバーコンポーネントやAPIルートで実装する

### 案A の適合性
- kanji-data.json（671KB）は引き続きクライアントバンドルに含まれる。
- さらにembeddingsファイルが追加されるため、クライアント側の総ダウンロードサイズは増加する。
- Rule 1 の「バンドルサイズが大きくなる機能」の観点では、**Rule 1 に対してやや不適合**（サイズ増）。
- ただし、embeddingsはビルド成果物として配信するため「クライアントサイドJS」のバンドルサイズとは別（JSONとして配信される）。解釈により適合とも言える。

### 案B の適合性
- 複雑なロジック（evaluateGuess + kanji-data.json）をAPIルートに移動する。
- クライアントバンドルから671KBのJSONが除外される。
- Rule 1 の趣旨（「複雑な機能はサーバーで」）には**より適合**。
- ただし、Rule 2 のシンプル性（「コンポーネントは狭く、読みやすく、独立してテスト可能に」）の観点では、非同期処理の追加でGameContainerが複雑化する。

### 適合性の結論
Rule 1 の「バンドルサイズ」の観点では案Bが優位。Rule 3（「シンプルで一貫した設計」）の観点では案Aが優位（既存の純粋関数設計を維持）。
