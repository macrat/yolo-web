# 調査レポート: 漢字カナル カテゴリ分類の第三の道

## 1. 現在のゲーム構造の把握

### データ構造
`src/data/kanji-data.json` に2,136字の常用漢字データが格納されている。各エントリは以下のフィールドを持つ:
- `character`: 漢字本体
- `radical`: 部首の漢字文字（例: 水, 人, 手）
- `radicalGroup`: KANJIDIC2の康熙字典部首番号（1〜214の整数）
- `category`: カテゴリID（現在は `radicalGroup` の番号範囲で機械的に割り当て、1〜20）
- `grade`, `strokeCount`, `onYomi`, `kunYomi`, `meanings`, `examples`

### フィードバックシステム
`src/games/kanji-kanaru/_lib/engine.ts` にて `evaluateGuess()` が各属性のフィードバックを計算する:
- **radical**: 完全一致/不一致（close なし）
- **strokeCount**: 一致/±2以内で近い/遠い
- **grade**: 一致/±1以内で近い/方向付き/遠い
- **onYomi**: 共通読み有無
- **category**: 一致/同スーパーグループで近い/遠い ← 今回の問題箇所
- **kunYomiCount**: 一致/±1以内で近い/遠い

### カテゴリの「近い」判定
`src/games/kanji-kanaru/_lib/categories.ts` にて `categorySuperGroups` を定義。現在は隣接する番号グループをペアにしているだけ（例: `mouthAstro: [4, 5]` = 口と囲い+天文と時間）で、意味的関連性がない。

### 現在の問題（コードで確認）
`generate-kanji-data.ts` の `radicalNumberToGroup()` 関数が部首番号の**範囲**でグループ割り当てをしている:
- 部首番号34〜44 → グループ5「天文と時間」に 女(38)・大(37)・子(39) などが混入
- 部首番号61〜65 → グループ7「動作基本」に 心(61) が混入
- 部首番号75〜86 → グループ9「木と欠」に 木・水・火 が同居

## 2. 利用可能なデータの調査

### KANJIDIC2データのフィールド
- `radical`（部首文字）: データ上は198種類のユニーク部首文字が存在
- `radicalGroup`（部首番号）: 1〜211の整数（214基本部首のうち198が常用漢字に出現）
- `meanings`（英語意味）: 例「water, pond, ocean...」など複数の英語意味

### 部首の頻度分布（上位10）

| 口 | 30 | 71 |

### 意味フィールドによるキーワード分類（方向性C）の問題点
英語意味フィールドでのキーワードマッチングは**偽陽性が多発**することを確認:
- "water" で検索すると `力` がヒット（meanings: "bear up, exert" → "water" との無関係な語が含まれる）
- "body" で検索すると204字がヒット（2136字の10%）するが多くが無関係
- ルールベースでの正確な分類は困難で、個別修正が必要になりAI依存の問題が再現する

## 3. 第三のアプローチの検討と推薦

### 各アプローチの評価

#### 方向性A: 部首番号の意味的グルーピング（**推薦**）

214の康熙字典部首番号を**番号範囲でなく意味で**手動で20グループに割り当てる。

**根拠となる重要な発見**:
- 実装の変更箇所は `generate-kanji-data.ts` の `radicalNumberToGroup()` 関数のみ
- 2,136字ではなく**214の部首番号**に対する割り当てなので、一回限りの作業量
- KANJIDIC2の `radicalGroup`（部首番号）は完全に確定的なデータ
- AI不要・分類ミスリスクなし

エントロピー計算で 4.16 bits（理論最大 4.32 bits の96%）= 20グループをほぼ均等に使えており、識別情報として十分。

**インシデントレポートの問題が解決されることを確認**:
- `女`(部首#38): 旧「天文と時間」→ 新「人・家族」✓
- `大`(部首#37): 旧「天文と時間」→ 新「人・家族」✓
- `子`(部首#39): 旧「天文と時間」→ 新「人・家族」✓

- `自然系`: [7,8,9,10,11,12] = 水, 火, 木・植物, 土・山, 天・気象, 動物
- `文明系`: [13,14,15,16,17,18,19] = 建物, 布・衣, 刀・武器, 金・素材, 食・飲, 乗り物, 社会

例: 海(水・液体=7) vs 山(土・山=10) → 両方が自然系 → 「近い」✓ 直感的

#### 方向性B: 学年+部首系統の組み合わせ
grade はすでに別フィードバック軸として存在。category を部首そのもの（198種）にすると細かすぎる。方向性Aの20グループ化と本質的に同じアプローチになるため、A として統合評価。

#### 方向性C: 意味分類の確定的生成
調査の結果、**採用不可**と判断:
- 英語意味フィールドからのキーワードマッチングで偽陽性が多発
- 例: "water" が `力`(bear up)にヒット, "body" が 204字にヒット
- 個別修正が必要になりAI依存の問題が本質的に解決しない

#### 方向性D: その他の可能性（参考）
調査過程で確認した追加の可能性:
- **部首文字そのものをカテゴリ名に使う**: 198カテゴリは多すぎるが、上位20部首で常用漢字の約46%をカバーできる。ただし残り54%が細分散するため、方向性Aの20グループ化の方が優れる。
- **部首→意味クラスタへのAI一括分類**: 214部首のみにAIを適用する場合、作業量は大幅に削減できる。ただしレビューコストがあり、方向性Aの手動割り当て（1回限り）と比較して優位性がない。

## 4. ゲーム性の検証（方向性A）

### 情報量の観点
- category（20グループ、エントロピー4.16bits）
- grade（7段階、方向付き）
- strokeCount（±2クローズ）
- radical（バイナリ、198種）
- onYomi・kunYomiCount（各1〜2bits相当）

合計で約10〜12bits相当の情報を6回の推測で収集できる。log2(2136) ≈ 11.1 bits であり、理論的に6回で解ける情報量は十分。

### 「近い」フィードバックの直感的解釈
方向性Aのスーパーグループ設計で:
- `海`を推測 → カテゴリ「近い」 → 「水じゃないけど自然系」→ 次は山, 木, 雨, 動物等を試す
- `話`を推測 → カテゴリ「近い」 → 「言語じゃないけど人体系」→ 次は体・感情・移動等を試す

**一般的な漢字知識だけで解釈可能**（康熙字典の専門知識不要）。

### 残る制限（許容範囲）
古典的な部首が意味と一致しない例は一定数存在する:
- `天`(heaven) → 部首は`大` → グループ「人・家族」（やや意外）
- `知`(know) → 部首は`矢`(arrow) → グループ「口・言語」（矢は知識の比喩として許容範囲）
- `無`(nothing) → 部首は`火` → グループ「火・光」（やや意外）

これは漢字の歴史的成立過程に由来する**不可避な制限**であり、旧システムの「女が天文グループ」のような理不尽さとは本質的に異なる。部首は漢字の歴史的構成要素であり、プレイヤーが「部首で分類している」と理解すれば許容できる。

## 6. 結論と推薦

**推薦: 方向性A「部首番号の意味的グルーピング」**

### 理由
1. **確定的・再現可能**: AI不要、部首番号→グループの1対1マッピング
2. **権威ある根拠**: 康熙字典の部首体系（1800年以上の歴史）
3. **作業量が現実的**: 214部首の割り当て（2136字の個別作業は不要）
4. **ゲーム性が担保される**: 水・火・木が別グループ、スーパーグループが意味的に隣接
5. **インシデントの問題を直接解決**: 女/大/子が「天文」でなく「人・家族」へ

### 注意点
- グループ20「その他」に約200字（一・二・三等の数字、夕・少等の抽象形）が含まれる。これはゲーム的には「このカテゴリフィードバックはヒントが弱い」ことを意味するが、該当漢字はパズルの対象から除外する（学年1〜3年の基本漢字は多く含まれるが、フィードバックの解釈が難しくなる）か、グループ20の漢字を答えの候補から外すかは別途検討が必要。
