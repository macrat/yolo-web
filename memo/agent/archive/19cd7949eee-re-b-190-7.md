# 完了報告: B-190タスク7

## 作業1: reEvaluateGuessesの要否判断

### 調査結果
- デプロイ済み（B-190変更前、コミット5405429）のGameHistoryには feedbacks フィールドが存在しなかった
- デプロイ済みコードは evaluateGuess をクライアントサイドで直接呼び出していた
- reEvaluateGuesses は「旧フォーマット（feedbacksなし）からの復元」のために存在していた

### 判断
Ownerから「実験サイトでありユーザーが少ないので互換性を維持しない判断も問題ない」とのコメントがあったため、以下の方針を採用:

- reEvaluateGuesses 関数を GameContainer.tsx から完全削除
- loadTodayGame で feedbacks がないセーブデータは null を返す（＝新規ゲームとして開始）
- 「lost + guessCount < MAX_GUESSES → playing」マイグレーションも削除（古い中間形式のための互換コード）
- storage.ts から MAX_GUESSES のインポートも除去（不要になったため）

### 影響
- 最悪ケース: デプロイ済み版で途中のゲームがあるユーザーは、新版デプロイ後に1日分の進行を失う
- 統計データ（GameStats）には影響なし

## 作業2: KanjiEntry.categoryフィールドの精査

### 調査結果
- KanjiEntry.category（RadicalGroup型、1-20の数値）はゲーム評価には使用されていない
- engine.ts の evaluateCategory は evaluateSimilarity（Embeddingベース）を使用
- 辞書側（src/dictionary/_lib/kanji.ts）と検索インデックス（src/lib/search/build-index.ts）が独自に category を使用中（ただし辞書の KanjiEntry は別の型定義）
- share.ts の g.category は GuessFeedback.category（FeedbackLevel型）であり、KanjiEntry.category とは無関係

### 対応
- KanjiEntry.category フィールドに「ゲーム評価には使用しない」旨のJSDocコメントを追記
- share.ts の g.category は GuessFeedback.category であることを確認済み。変更不要
