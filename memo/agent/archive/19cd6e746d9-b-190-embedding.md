# 調査依頼: Embeddingの入力テキスト設計と品質最大化

## 背景
漢字カナルのカテゴリ分類にローカルEmbeddingモデル（ruri-v3-30m-ONNX）+ k-meansクラスタリングを採用する方針。しかし、**何をEmbeddingするか**が品質を決定的に左右する。

漢字1文字や部首単体をEmbeddingするという発想に固執せず、利用可能なすべてのデータフィールドを活用して最高の分類品質を実現する方法を徹底的に調査してください。

## 調査してほしいこと

### 1. 利用可能なデータフィールドの完全な把握
`src/data/kanji-data.json` を読んで、各漢字エントリのすべてのフィールドを列挙してください。例:
- character（漢字文字）
- radical（部首文字）
- radicalGroup（部首番号）
- meanings（英語の意味リスト）
- onYomi（音読み）
- kunYomi（訓読み）
- examples（例語）
- grade（学年）
- strokeCount（画数）
- category（現在のカテゴリ）
- その他あるフィールドすべて

### 2. 入力テキストパターンの網羅的検討
以下のパターンを含め、考えうるすべての組み合わせを検討してください:

- パターンA: 漢字1文字のみ（"海"）
- パターンB: 漢字 + meanings（"海 ocean sea"）
- パターンC: meanings のみ（"ocean sea"）
- パターンD: 漢字 + 日本語の意味説明を構成（meaningから日本語文を生成）
- パターンE: 漢字 + 部首 + meanings（"海 水 ocean sea"）
- パターンF: 漢字 + 読み + meanings（"海 カイ うみ ocean sea"）
- パターンG: 例語を使う（"海岸 海外 海底"）
- パターンH: 漢字の全情報を自然文にした説明文（"海は水の部首を持ち、oceanやseaを意味する漢字で..."）
- パターンI: meanings + examples の組み合わせ
- パターンJ: その他、あなたが考えうる最良のパターン

### 3. ruri-v3の特性を踏まえた最適化
- ruri-v3は日本語文のセマンティック検索用モデル
- 短い単語よりも文レベルのテキストの方が精度が高い可能性がある
- ruri-v3のドキュメントやベンチマーク情報を参照し、どのような入力が最適か調査

### 4. 実際に試す（可能であれば）
- @huggingface/transformers をインストールして、いくつかの漢字で実際にEmbeddingを計算
- 複数パターンの入力テキストで「海」「川」「池」や「火」「炎」「燃」などの意味的に近い漢字グループのコサイン類似度を比較
- どのパターンが最も意味的な類似性を正しく捉えるか実験的に検証

### 5. k-meansのパラメータ検討
- k=20が最適か（情報エントロピーの観点）
- 初期化方法（k-means++等）
- 収束条件
- シード固定の方法

### 6. スーパーグループの自動化
- クラスタ間距離を使って「近い」判定を自動化できるか
- 手動のスーパーグループ設定を完全に排除できるか
- 排除できる場合、categories.tsのcategorySuperGroupsをどう変更するか

### 7. カテゴリ名の自動命名
- 各クラスタの代表的な漢字から名前を自動生成する方法
- meaningフィールドを使ったキーワード抽出

## 重要な注意事項
- フォールバック（部首グルーピング）は設けない。Embeddingアプローチで最高の品質を実現することに集中する
- 手間を惜しまないこと。最高のものを作るために必要なすべての調査を行う
- docs/coding-rules.md を読んで技術制約を確認すること
