# Q43 ユーモア占い・診断ポータル 調査結果レポート

---

## 1. 競合サイト調査

### ShindanMaker (shindanmaker.com)

**形式の特徴:**
- 主軸は「名前入力→ランダム結果」方式。名前をハッシュ値として結果を決定論的に生成する。
- 質問に答えて自分を知るプロセスは弱い。「名前を入れると何かが分かる」という即時性が強み。
- バリエーション: 名前診断・分岐診断（選択肢あり）・AI診断・チャート型・画像/動画型
- 逆張り系診断（「逆張り診断」「クソ逆張りオタク」等）は実在し、ユーザーが自由に作成できるが、玉石混交。
- SNSシェア: 結果テキストをそのままXにポストするのが主流。スクリーンショットシェアも多い。
- 最新動向（2025年末〜2026年初）: name diagnosis v2 バグ修正、canvas effect型追加など機能拡充が続く。

**yolos.netとの差別化ポイント:**
ShindanMakerは「質問で自分を知る体験」が薄い。yolos.netが提供する「8問程度の選択式→自己発見」形式は直接的な競合ではなく、棲み分けが可能。ユーモア形式の品質はUGCのため不安定。

### 診断ドットコム (4ndan.com)

**形式の特徴:**
- UGCプラットフォーム。誰でも診断を作成・投稿可能。
- 「おもしろ診断」「名前占い」「クイズ」カテゴリを提供。
- SNSシェア: 診断結果ごとにカスタムシェアテキストとハッシュタグを設定できる。
- 約600件以上の診断コンテンツ（2021年時点）。質問数は診断ごとに異なる。

**yolos.netとの差別化ポイント:**
UGCのため「ハズレがある」。yolos.netはキュレーション型として「全コンテンツがハズレなし」を訴求できる。

### ホイミー (hoyme.jp)

**形式の特徴:**
- コンセプト: 「自分、見つかる！モヤモヤ、解消！」—本格診断よりエンタメ寄りだが、本格占いも扱う。
- カテゴリ: 動物・生き物診断（犬・猫・恐竜・深海生物）、性格診断（血液型・星座・干支・エニアグラム）、ユニーク診断（フルーツ・食べ物・家電・色彩）、夢占い（40〜50パターン以上）
- UXは簡潔な設問でスムーズ。トップページにカルーセルで人気診断を表示。
- SNSシェア機能あり。
- ユーモア系診断は「ネタ診断」として提供されているが、逆張り・達成困難アドバイス・斜め上の運勢といった特定コンセプトへの特化はなく、差別化余地が大きい。

**yolos.netとの差別化ポイント:**
ホイミーは「面白い診断も本格診断も」という折衷路線。yolos.netは「ユーモア・意外性に100%振り切る」コンセプトで明確に差別化できる。

### 競合まとめ

| サイト | 強み | 弱み（yolos.netの勝機） |
|--------|------|------------------------|
| ShindanMaker | 量・SNS拡散力 | 質問型「自己発見」体験が弱い。ユーモア品質が不安定 |
| 診断ドットコム | コンテンツ量 | UGCで玉石混交。一貫したトーン・世界観がない |
| ホイミー | 本格性・安心感 | ユーモアへの特化度が低い。逆張り系コンセプトなし |

## 2. Q43のコンテンツ形式調査・提案

### 2.1 逆張り占い: 具体的な形式提案

**コンセプト定義:** 一般的な占いが「あなたはこういう運命・性格です」と肯定的または神秘的な結論を提示するのに対し、逆張り占いは「普通そう言うところだが、実は逆だ」という裏切りで笑いを生む。

**形式A: 「逆説の運勢」**
- 質問例: 「今日のラッキーカラーを占います。あなたが今着ている服の色は？」
- 結果例（赤を選んだ場合）: 「今日のアンラッキーカラーは赤です。でも大丈夫、赤を着ていれば不運は赤に向かいます。アンラッキーカラーを着て身代わりにする作戦、古代の賢者たちも実践していました。」

**形式B: 「ズレたアドバイス型占い」**
- 質問例: 「今悩んでいることは？」（仕事・恋愛・お金・健康・人間関係）
- 結果例（仕事を選んだ場合）: 「仕事で迷ったら、その仕事が存在しない惑星に引っ越すのが一番です。ただし移住コストが高額なため、次善策として『気にしない』があります。」

**形式C: 「うらない結果の訂正版」**
- 一般的な占い文を表示した後、「しかし実際は…」と逆の結論を提示する。
- 例: 「今日のあなたは運気絶好調！——と言いたいところですが、AIが計算した結果は別のことを示しています。本日の真の運気は『普通』です。ただし、普通は最も安定した運気です。」

**推奨アプローチ:** 形式Bの「悩み別・状況別の斜め上アドバイス」が最も汎用性が高く、質問設計とユーモア結果文の組み合わせで無限に展開できる。フォーマットは既存 `type: "personality"` の personality quiz の仕組みをそのまま転用可能。

---

### 2.2 達成困難アドバイス: 具体的な形式提案

**コンセプト定義:** 「実行すると面白そうだが、実際に試みるのが難しい（非現実的・大げさ・滑稽）」なアドバイスを、いかにも真面目に提示するギャップで笑いを生む。ネガティブ・不快ではなく、「やれたらすごい」「やるわけないけど面白い」という笑いの温度感。

**質問例:**
- 「今週うまくいかなかったことは？」（人間関係・仕事効率・ダイエット・睡眠・お金節約）

**結果例（ダイエットを選んだ場合）:**
「本日のアドバイス: 自分の部屋を富士山の5合目相当の気温に設定してください（約10℃）。寒さで基礎代謝が上がり、また寒すぎて食欲が失せます。暖房を切るだけで達成できます。なお電気代は節約できます（体脂肪で補う想定）。」

**結果例（人間関係を選んだ場合）:**
「本日のアドバイス: 全ての会話を俳句5・7・5で行ってください。最初は不評かもしれませんが、相手も俳句で返し始めるころには、人間関係が詩的なものに昇華されています。」

**不快にならないための設計原則:**
1. 自分自身に対するアドバイスであり、他者を傷つけない
2. アドバイスの非現実性が明白（笑いの余地がある）
3. 「実行しなくてよい」前提が暗示されている
4. ネガティブな自己評価を引き出す質問は使わない

---

### 2.3 斜め上の運勢: 具体的な形式提案

**コンセプト定義:** 「ラッキーアイテム」「今日の運勢」という一般的な占い形式を踏襲しながら、その内容を予想外の角度から展開する。フォーマットの親しみやすさ×内容の意外性でうまい。

**サブ形式1: 「ラッキーアイテムが斜め上」**
- 例: 「今日のラッキーアイテム: 使い古しの輪ゴム。財布の中で長年眠っている輪ゴムが、本日あなたの運気を静かに支えています。何かをまとめておく安心感が今日の安定をもたらします。」

**サブ形式2: 「ラッキーな出来事が微妙」**
- 例: 「今日の幸運: コンビニのレジ袋が一枚余ります。些細な幸運ですが、AIによれば宇宙の均衡の観点から大変稀少な事象とのことです。」

**サブ形式3: 「今日の注意事項が謎」**
- 例: 「本日の注意: 第3象限への移動を慎んでください。なお第3象限とは、自分の立ち位置から見て斜め左下の空間全体を指します。」

**サブ形式4: 「今日の相性が意外」**
- 例: 「今日の相性が良い存在: 電柱。古くから街に立ち続ける電柱のように、あなたも今日はどっしりと地に足をつけた行動が吉とされています。」

**推奨アプローチ:** 日替わり占い（後述）はこのサブ形式4つを日付シードで組み合わせ、毎日異なる結果パターンを提示する形が最も実装しやすく、かつ笑いの多様性を確保できる。

### 2.4 日替わり占い要素（日付シードによるクライアントサイド実装）

**実装方針:**

日付をシード値として使う決定論的PRNGをクライアントサイドで実装する。外部ライブラリは不要で、シンプルな実装で十分。

```typescript
// 日付文字列 "YYYY-MM-DD" → 数値シードへのハッシュ
function hashDate(dateStr: string): number {
  let hash = 0;
  for (let i = 0; i < dateStr.length; i++) {
    const char = dateStr.charCodeAt(i);
    hash = ((hash << 5) - hash) + char;
    hash |= 0; // 32bit整数に変換
  }
  return Math.abs(hash);
}

// Mulberry32 PRNG（シード値から決定論的に乱数を生成）
function mulberry32(seed: number) {
  return function () {
    seed |= 0;
    seed = (seed + 0x6d2b79f5) | 0;
    let t = Math.imul(seed ^ (seed >>> 15), 1 | seed);
    t = (t + Math.imul(t ^ (t >>> 7), 61 | t)) ^ t;
    return ((t ^ (t >>> 14)) >>> 0) / 4294967296;
  };
}

// 今日の日付（JST）を取得
function getTodayJst(): string {
  return new Intl.DateTimeFormat("en-CA", { timeZone: "Asia/Tokyo" }).format(new Date());
}

// 使用例
function getDailyFortuneIndex(itemCount: number): number {
  const today = getTodayJst();
  const seed = hashDate(today);
  const rand = mulberry32(seed);
  return Math.floor(rand() * itemCount);
}
```

**既存コードとの整合性:** src/lib/achievements/date.ts に `getTodayJst()` が既に実装されている。日替わり占いはこれを再利用し、上記のシード生成関数を追加するだけで実装できる。

**結果の一意性:** 同じ日付に同じシードから同じ結果が生成される。異なるデバイス・ブラウザでも同じ「今日の占い」が表示される（日付が同じなら）。

**コンテンツ設計例:**
- 結果テンプレートを50〜100パターン用意
- 各パターンは「ラッキーアイテム・今日の運勢・斜め上アドバイス・注意事項」の組み合わせ
- シードから複数の乱数を生成し、各要素を独立に選択することで、組み合わせ爆発を実現

## 3. 技術実装パターン調査

### 3.1 既存クイズシステムの型定義と再利用可能性

**src/quiz/types.ts の主要型:**
- `QuizType = "knowledge" | "personality"` — 占い・診断は `"personality"` として実装可能
- `QuizChoice.points?: Record<string, number>` — 回答に対してresult IDへのポイント割り当て

- `QuizDefinition` — meta + questions + results の完結したデータ構造

**占い・診断はquiz type=personalityの変種か？**

結論: **基本的にそのまま再利用可能。ただし日替わり占いには別仕組みが必要。**

| 占い形式 | 既存の仕組みで対応可能か | 補足 |
|----------|--------------------------|------|
| 逆張り占い（質問→結果） | **可能** | type: "personality" として実装。QuizContainer、ResultCard、ShareButtonsをそのまま使用 |
| 達成困難アドバイス（選択→結果） | **可能** | 同上。結果テキストをユーモラスなアドバイス文にするだけ |
| 斜め上の運勢（質問→占い結果） | **可能** | 同上 |
| 日替わり占い（質問なし→日付で結果決定） | **別途実装が必要** | QuizContainerはquestions必須の設計。日替わりは「intro→(日付シード計算)→result」という新しいフローが必要 |

**再利用できる既存コンポーネント:**
- `src/quiz/registry.ts`: QuizDefinitionを登録する仕組みはそのまま使える
- `src/quiz/_components/ResultCard.tsx`: 結果表示・ShareButtonsはそのまま使える
- `src/quiz/_components/ShareButtons.tsx`: X/LINE/Web Share API対応済み（後述）
- `src/quiz/scoring.ts calculatePersonalityPoints()`: 点数計算ロジックは再利用可能

**日替わり占いのための新コンポーネント設計（推奨）:**
```typescript
// src/fortune/_components/DailyFortuneCard.tsx
// QuizContainerの代替。質問フローなしで日付シードから直接結果を表示する
// useEffectで today=getTodayJst() を取得 → hashDate → mulberry32 → result選択 → ResultCard表示

**QuizDefinitionデータ形式との差異:**
日替わり占いはquestionsが不要なため、以下の独自型を定義することを推奨:

  id: string;

  luckyItem: string;   // ラッキーアイテム
  advice: string;      // 達成困難アドバイス
  color?: string;
  icon?: string;
}
```

## 4. SNSシェア機能調査

### 4.1 既存実装の確認

**src/quiz/_components/ShareButtons.tsx の現在の実装:**
- Web Share API (`navigator.share`): `useCanWebShare()` フックで検出。対応時は「結果をシェア」ボタン1個表示
- 非対応時: 「Xでシェア」「LINEでシェア」「コピー」の3ボタンを表示
- Xシェア: `https://twitter.com/intent/tweet?text=...&url=...`
- LINEシェア: `https://line.me/R/share?text=...`
- クリップボードコピー: `navigator.clipboard.writeText()`

**注意点:** 現在の実装では、Web Share API対応時にX/LINEの個別ボタンが非表示になる。site-concept.md の方針（「各SNS個別シェアボタンを主軸とし、Web Share APIを補助的手段として提供」）と実装が逆になっている。

### 4.2 Web Share API の現在の対応状況（2025〜2026年）

- **iOS Safari**: バージョン12.2以降で完全対応。現行バージョン（26.x）でも安定動作。
- **Android Chrome**: バージョン145以降で対応済み。
- **Firefox（全バージョン）**: 非対応。
- **デスクトップChrome/Edge**: 対応しているが、動作は環境依存（Windowsは対応率高い）。
- **グローバル対応率**: 約92.9%（2025年時点）。

**結論:** Android Chromeも現在は安定して対応しており、site-concept.mdの「Android環境では個別シェアボタンを優先」という方針は現時点では必須ではない。ただし、Web Share APIはユーザーが「キャンセル」する可能性があるため、個別ボタンを常時表示する方が安心感がある。

**Q43での推奨実装:**
```
Web Share API対応環境: 個別ボタン(X/LINE) + Web Shareボタン を並列表示
非対応環境: 個別ボタン(X/LINE) + クリップボードコピーを表示
```
これにより既存実装より「site-concept.mdの意図」に忠実になる。

### 4.3 OGP動的画像生成（既存実装の確認）

**既存実装:** `src/app/quiz/[slug]/opengraph-image.tsx` と `src/app/quiz/[slug]/result/[resultId]/opengraph-image.tsx` が既に実装済み。

## 5. 実績システムとの連携

### 5.1 既存実装の確認

**src/lib/achievements/engine.ts:**
- `recordPlay(contentId: string)` が全コンテンツ共通の記録関数
- dailyProgress（日次のboolean記録）と totalStats.perContent（累計カウント）を更新
- バッジ評価を自動実行

**src/lib/achievements/badges.ts:**
- 現在の `ALL_CONTENT_IDS` に含まれるのは: 4ゲーム + 5クイズ（合計9種）
- Q43を追加する場合は `QUIZ_IDS` 配列に `"quiz-q43-humor-fortune"` 等のIDを追加するだけ

**src/quiz/_components/QuizContainer.tsx:**
- QuizContainer の `handleAnswer`/`handleNext` 内で `recordPlay("quiz-" + quiz.meta.slug)` を呼び出し済み

### 5.2 Q43との連携方法

**通常の占い（質問→結果型）:**
既存の `QuizContainer` をそのまま使うため、`recordPlay()` は自動で呼ばれる。slug を `q43-humor-quiz-NAME` 等にすれば、個別に記録される。

**日替わり占い:**
日替わりは1日1回の訪問記録が目的。`DailyFortuneCard` コンポーネント内で占い結果表示時に `recordPlay("fortune-daily")` を呼ぶ。LocalStorage で「今日は既に記録済み」かチェックし、複数回訪問でも1回分としてカウントする（engineの dailyProgress が boolean なので自動的に冪等になる）。

**バッジ設計の拡張（推奨）:**
Q43専用バッジの追加を検討:
- 「占い師デビュー」: 初めてユーモア占いを受ける（ブロンズ）
- 「毎日占い師」: 7日連続で日替わり占いを利用（シルバー）

## 6. 調査サマリーと推奨アプローチ

### Q43の構成案（推奨）

Q43「ユーモア占い・診断ポータル」は以下の複数コンテンツで構成することを推奨:

| コンテンツ | 実装方式 | 質問数 | 結果数 | 特徴 |
|-----------|----------|--------|--------|------|
| 今日のユーモア運勢（日替わり） | DailyFortuneCard（新規） | 0問 | 50〜100プール | 毎日違う・リテンション装置 |
| 逆張り運勢診断 | QuizContainer（流用） | 8問 | 6〜8パターン | 「逆説の運勢」フォーマット |
| 達成困難アドバイス診断 | QuizContainer（流用） | 5〜6問 | 5〜7パターン | 悩み別のユーモアアドバイス |
| 斜め上の相性診断 | QuizContainer（流用） | 8問 | 8〜10パターン | 謎の相性結果（電柱・植物・天気等との相性） |

### 技術実装の工数見積もり（要素別）

1. **日替わり占い（DailyFortuneCard）**: 新規コンポーネント1つ + 結果データ50〜100件 — 中程度の工数
2. **逆張り/達成困難/斜め上 各診断**: QuizDefinitionデータ作成のみ — 工数小（既存フレームワーク流用）
3. **SNSシェア改善**: ShareButtons.tsx の表示ロジック修正（個別ボタン+Web Shareの並列表示） — 工数小
4. **OGP画像**: 既存パターンの踏襲 — 工数なし（データ追加のみ）
5. **実績連携**: badges.ts の QUIZ_IDS にID追加 — 工数最小

### 依存する一次ソース

- src/quiz/types.ts: QuizDefinition, QuizMeta 型
- src/quiz/_components/QuizContainer.tsx: intro→playing→result フロー
- src/quiz/_components/ShareButtons.tsx: 現在のシェア実装
- src/lib/achievements/badges.ts: ALL_CONTENT_IDS（Q43追加箇所）
