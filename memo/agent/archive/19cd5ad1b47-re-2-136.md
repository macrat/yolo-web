## 1. データソース

### 主推奨: KANJIDIC2

**KANJIDIC2**（https://www.edrdg.org/kanjidic/kanjidic2.xml.gz）が最良の選択肢。

- 収録数: 13,108字（JIS X 0208/0212/0213の全漢字）。常用漢字2,136字をすべて含む
- フォーマット: XML（gzip圧縮）
- ライセンス: Creative Commons Attribution-ShareAlike 4.0（CC BY-SA 4.0）
- 提供フィールド:
  - `<grade>`: 学年（1-6=小学校各学年、8=中学校以降の常用漢字残り1,130字、9-10=人名用漢字）
  - `<stroke_count>`: 画数（複数値がある場合、最初が正式、以降は誤記の多いもの）
  - `<rad_value type="classical">`: 康熙字典の部首番号（1-214）
  - `<reading r_type="ja_on">`: 音読み（カタカナ）
  - `<reading r_type="ja_kun">`: 訓読み（ひらがな、送り仮名はピリオドで区切り）
  - `<meaning>`: 英語の意味（`m_lang`属性なし＝英語デフォルト）

#### ライセンス上の注意点
CC BY-SA 4.0のShareAlike条件により、KANJIDIC2のデータを含む派生物を公開する場合、同じライセンスで配布が必要。ただしAI生成のセマンティックカテゴリやexamplesフィールドは独自追加データなので、KANJIDIC2由来フィールドとAI生成フィールドを明確に分離して管理することが望ましい。帰属表記（edrdg.orgへのリンク）は必須。

### Node.js/TypeScriptでのパース手段

- **kanjidic2stream** (npm): Nodeのストリームとして各エントリをオブジェクトに変換できる（GitHub: forrestjacobs/kanjidic2stream）
- **jmdict-simplified** (scriptin/jmdict-simplified): KANJIDIC2を含む主要辞書をJSONに変換済み。プレビルドJSON＋TypeScript型定義が利用可能。MITライセンスだが元データのCC BY-SA 4.0が適用される
- **自前パース**: Node.js + xml2jsでスクリプト作成も容易。ビルド時スクリプト（`scripts/`以下）として実装するのが最適

### 代替/補完データソース

| ソース | 特徴 | ライセンス |
|--------|------|-----------|
| davidluzgouveia/kanji-data (GitHub) | KANJIDIC由来、JLPT付き、JSON済み | MIT |
| mifunetoshiro/kanjium (GitHub) | 音読み・訓読み網羅、CSV/JSON | CC BY-SA |
| KEINOS gist (joyo_kanjis.json) | 2,136字限定JSON、旧字体付き | 未確認 |
| melissaboiko/joyodb | 文科省公式PDFから自動変換、TSV | 未確認（alpha品質） |

**推奨**: KANJIDIC2を正典（ground truth）として使い、grade/strokeCount/radical/readings/meaningsを取得。セマンティックカテゴリとexamplesのみAI生成で補完するアプローチが最も確実。

## 2. セマンティックカテゴリの分類方針

### 現状

- 現在のkanji-data.json: 17カテゴリを使用（emotionとsocietyとmeasurementは未使用）
- ゲームのSemanticCategory型: 20カテゴリ定義
- スーパーグループ（closeヒント用）: 5グループ

```
elements: water, fire, earth, weather, nature
living: animal, plant, body, person
human: emotion, action, language, society
abstract: number, time, direction, measurement, abstract
objects: building, tool
```

### 2,136字への拡張時の課題

常用漢字を20カテゴリに分類した場合、各カテゴリの平均サイズは約107字。小学1年生の80字では「building: 1字」「fire: 1字」のような極端な偏りが生じているが、2,136字に拡張すれば統計的に均一化される。

**問題のある小カテゴリ（現在）**:
- building: 1字、fire: 1字、weather: 1字 → 拡張後は自然に増える

**2,136字全体での推定分布**（AI分類を想定した場合）:
- 大カテゴリ（100字超）: nature, action, person, society, abstract, body, language, number, time
- 中カテゴリ（50-100字）: animal, plant, building, tool, water, direction, earth
- 小カテゴリ（50字未満）: weather, fire, emotion, measurement

### カテゴリ変更の要否

**推奨: 既存の20カテゴリを維持**。理由：
1. ゲームのclose判定ロジック（categorySuperGroups）がすでにこれらに最適化されている
2. カテゴリ追加・変更はGameContainerやTypeScriptの型変更を伴い影響範囲が大きい
3. 2,136字規模では現在の20カテゴリで十分な分解能がある
4. ゲームのヒントとして機能するためには、スーパーグループ内のカテゴリが適度に分散していれば十分

**追加を検討すべきケース（オプション）**:
- 「food」（食べ物関連、2,136字には食材・料理の漢字が多数ある）
- 「economy/commerce」（経済・商業系）
ただしこれらを追加する場合はcategories.ts（スーパーグループ）とtypes.tsへの変更が必要。

## 3. データ生成の方法

### 推奨フロー

```
フェーズ1: KANJIDIC2からの自動抽出（スクリプト化）
  kanjidic2.xml.gz をダウンロード
  → grade 1-8 の漢字を抽出（2,136字）
  → character, radical, radicalGroup, strokeCount, grade, onYomi, kunYomi, meanings を自動生成
  → 出力: kanji-base.json（2,136エントリ、categoryとexamplesはnull/空）

フェーズ2: AIによるカテゴリ分類（バッチ処理）
  kanji-base.json を読み込み
  → 1バッチ50-100字ずつ Claude に投入
  → 20カテゴリのいずれかに分類
  → 各字に対して examples（熟語3-5語）を生成
  → 出力: category-examples.json

フェーズ3: マージ＆検証
  kanji-base.json + category-examples.json をマージ
  → 全エントリの必須フィールド完備を確認
  → カテゴリ分布の偏りチェック
  → 出力: kanji-data-full.json
```

### 実装方法（技術制約への適合）

- ビルド時スクリプト（`scripts/generate-kanji-data.ts`等）でKANJIDIC2を処理
- ランタイムでの外部API呼び出しは禁止（coding-rules.md § 2）
- 生成後のJSONファイルは静的データとして`src/data/`に配置
- ゲームのGameContainer（クライアントコンポーネント）が`import`する形は変わらない

## 4. データの正確性担保

### リスクと対策

| フィールド | リスク | 対策 |
|----------|--------|------|
| character | 低（文字コードは確定的） | KANJIDIC2から直接取得 |
| strokeCount | 低（KANJIDIC2は校閲済み） | KANJIDIC2から直接取得 |
| radical/radicalGroup | 低 | KANJIDIC2 classical分類から直接取得 |
| onYomi/kunYomi | 中（読みが多い字は複雑） | KANJIDIC2から取得。kun読みのピリオド区切りを正規化 |
| meanings | 低（英語は十分） | KANJIDIC2から取得 |
| category | **高（主観性があり、AI分類は一貫性が課題）** | バッチ処理後にカテゴリ分布確認。明らかな誤分類を手動修正 |
| examples | **高（AI生成は存在しない熟語を作る可能性）** | 熟語の実在確認が必要。既存80字の方法論を踏襲 |

### examplesの正確性が最大リスク

現在の80字データでも熟語の正確性が問題になる可能性がある。2,136字規模では以下のアプローチを検討：
- **選択肢A**: KANJIDIC2の関連辞書（JMdict）から熟語を引く（自動化可能だが複雑）
- **選択肢B**: AIで生成し、AIで自己検証（フォールバック的な精度）
- **選択肢C**: examplesフィールドは3字に絞り、より一般的な熟語のみ採用

**選択肢Aが最も安全**だが、実装コストが高い。選択肢Cは安全策として有効。

## 5. 技術制約への適合とバンドルサイズ

### ファイルサイズ推計

| 規模 | エントリ数 | 推定JSONサイズ | gzip後 |
|------|----------|--------------|--------|
| 現在（grade 1） | 80字 | 16 KB | ~7 KB |
| 全常用漢字 | 2,136字 | **~490 KB** | ~200 KB |

### バンドルへの影響

現在のGameContainer.tsxはクライアントコンポーネントで`import kanjiDataJson from "@/data/kanji-data.json"`している。2,136字に拡張するとクライアントバンドルに490 KBが加わる（gzip後約200 KB）。

**この影響は無視できない。** 対策案：

**案A（推奨）: APIルートに移行**
- `src/app/api/kanji/route.ts`でデータを提供
- サーバーサイドのみでデータを保持（クライアントバンドルに含まれない）
- ゲームロード時にfetchで取得（初回のみ、localStorage等でキャッシュ可）
- ただし毎回APIコールが発生するため、対局開始前のローディングが発生

**案B: 分割import（必要な字のみ取得）**
- パズルスケジュールはサーバーコンポーネントで解決し、今日の漢字のみクライアントに渡す
- 検索機能（GuessInput）用の全データは別途遅延読み込み
- 実装が複雑になるが最もユーザー体験が良い

**案C: そのまま静的import（最もシンプル）**
- 490 KB（gzip後200 KB）はページ固有JSとして一度だけ読み込まれる
- 2026年のモバイル回線では許容範囲（ゲームページのみ）
- 実装コスト最小

**coding-rules.md § 1（静的コンテンツ優先）** と § 2（外部API禁止）の観点からは：
- ランタイムの外部API（KANJIDIC2等）は禁止
- 自前サーバー（Route Handlers）は可能
- 静的importも可能

バンドルサイズが問題になるかはbuilderがbundle analyzerで確認すべき。現時点では案C（静的import）→問題が確認されたら案Bへの段階的移行を推奨。

## 6. 総合推奨アプローチ

### データ生成パイプライン（ビルド時）

1. **KANJIDIC2をダウンロード**してスクリプトで処理（Node.js）
   - grade 1-8 = 常用漢字2,136字を抽出
   - character, strokeCount, radicalGroup, onYomi, kunYomi, meanings をそのまま取得
   - radicalGroup（1-214の番号）から radical（漢字）のマッピングは別途必要（別テーブルがKANJIDIC2に含まれる）

2. **AIバッチ処理でカテゴリ＋例語を生成**
   - 50-100字/バッチで20カテゴリ分類
   - examples 3語（過剰な例は品質低下につながる）

3. **検証スクリプトで品質チェック**
   - 全2,136字のフィールド完備確認
   - カテゴリ分布（extreme skewがないか）
   - examples が実在する熟語か（辞書APIを使った事後検証）

### 精度担保の優先順位

1. character/strokeCount/grade/radicalGroup/onYomi/kunYomi/meanings → **KANJIDIC2から自動取得（高精度）**
2. category → **AI分類、分布確認後に手動修正（中精度）**
3. examples → **AI生成、実在確認が必要（低精度リスク）**

examplesの精度が最大のリスク。JMdict（同じEDRDGプロジェクト、同ライセンス）から実在熟語を自動抽出する方法が安全だが、実装が複雑になる。
