# Q43 ユーモア占い・診断ポータル 調査依頼

## 依頼元
cycle-72 タスク1（Q43企画・設計）の事前調査

## 調査してほしいこと

### 1. ユーモア占い・診断の競合サイト調査
- 診断ドットコム(4ndan.com)、ShindanMaker(shindanmaker.com)、hoyme.jp 等の主要サイトのユーモア系コンテンツの特徴
- 特に「逆張り」「意外性」「笑える」系の診断がどのような形式で提供されているか
- 各サイトのUX（質問数、所要時間、結果表示、SNSシェア方法）

### 2. Q43の具体的なコンテンツ形式の調査
site-concept.mdで「逆張り占い」「達成困難アドバイス」「斜め上の運勢」が挙げられている。これらの具体的なコンテンツ形式を調査・提案してほしい:
- 「逆張り占い」: 一般的な占いの期待を裏切る形式とはどのようなものか？具体例
- 「達成困難アドバイス」: どのような形式だと面白く、かつ不快にならないか？具体例
- 「斜め上の運勢」: 既存の運勢系コンテンツと差別化する具体的な方法
- 「日替わり占い」要素: site-concept.mdセクション10に記載あり。日付をシード値としたクライアントサイド実装の具体的手法

### 3. 技術実装パターンの調査
- 既存のクイズシステム(src/quiz/)の型定義(types.ts)を確認し、Q43がどの程度既存の仕組みを再利用できるか
- 「占い」と「診断」は本質的にquiz type=personalityの変種なのか、別の仕組みが必要なのか
- 日替わりコンテンツのクライアントサイド決定的生成（日付シード＋ランダム）のベストプラクティス

### 4. SNSシェア機能の調査
- site-concept.mdセクション10に記載のSNSシェア実装方針を確認
- X(Twitter)、LINE、Instagram Storiesへのシェアボタン実装のベストプラクティス
- Web Share APIの現在の対応状況（特にiOS Safari vs Android Chrome）
- OGP動的画像生成のNext.js Route Handlersでのベストプラクティス

### 5. 実績システムとの連携
- src/achievements/ の既存実装を確認し、新しい占い・診断コンテンツとの連携方法を調査

## 参照すべきファイル
- docs/site-concept.md（セクション4, 7, 8, 10）
- docs/coding-rules.md（技術制約）
- src/quiz/types.ts（既存クイズの型定義）
- src/quiz/registry.ts（クイズ登録方法）
- src/quiz/_components/（既存クイズUIコンポーネント）
- src/achievements/（実績システム）

## 調査結果の出力形式
- 各項目について具体的な調査結果と推奨アプローチをまとめてください
- Q43のコンテンツ形式については、具体的な例（実際の質問や結果文のサンプル）を含めてください
