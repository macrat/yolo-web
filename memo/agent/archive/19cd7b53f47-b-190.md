# 計画策定依頼: 辞書カテゴリ修正＋データ品質改善

## 背景

cycle-80で常用漢字2,136字への拡張を行った際、以下の問題が発生した:
1. 辞書のカテゴリが意味的な文字列（"nature"等）→部首グループ番号（1-20）に変わり、ユーザーにとって無意味な分類になった
2. kunYomiにKANJIDIC2生形式のドット/ハイフンが残存（56.8%の漢字が影響）
3. onYomiに4字でハイフン付き読みが残存
4. 検索インデックスにドット付きkunYomiが含まれ精度低下
5. 88字に例語がない

これらはすべてcycle-80で発生した問題であり、B-190の修正スコープ内で対処する。

## 調査結果の要約（4つのResearcherレポートより）

### データの出自（メモ 19cd7a8387a）
- kanji-data.jsonは2段階で生成: generate-kanji-data.ts → extract-jmdict-examples.ts
- categoryフィールドはKANJIDIC2の元データにはなく、スクリプト内の静的マッピング（RADICAL_TO_GROUP: 214部首→20グループ）で計算した独自分類
- 元データには freq（使用頻度）、jlpt（旧JLPT級）、variants（異体字）、nanori（名前読み）等の未使用フィールドがある
- kunYomi/onYomiのドット・ハイフンはKANJIDIC2の表記をそのまま取り込んだもの

### UXリサーチ（メモ 19cd7a6141e）
- 一般的な漢字辞典の検索方法は15種類あり、基本は「音訓索引」「部首索引」「総画索引」の三大索引
- 本サイトで既存データから即実装可能な分類軸:
  - **学年別フィルタ**（grade 1-7）: 日本人学習者にとって最も直感的
  - **画数フィルタ**（strokeCount）: 「簡単な漢字から」ニーズに対応
  - **部首別表示**（radical/radicalGroup 1-214）: 伝統的な分類
  - **読み索引**（onYomi/kunYomi）: SEO効果も高い
- 分類軸は1つである必要はなく、複数軸を提供すべき（Owner指示）

### categoryフィールド使用箇所（メモ 19cd7a5dd0c）
- 13ファイルが影響: types.ts, kanji.ts, KanjiDetail.tsx, category/[category]/page.tsx, page.tsx, KanjiIndexClient.tsx, build-index.ts, sitemap.ts 等
- URLは /dictionary/kanji/category/[1-20] の形式
- CategoryNavは汎用コンポーネント（kanji固有ロジックなし）
- ゲーム側のKanjiEntry.categoryはゲーム評価に使用されない（embedding類似度で代替済み）

### cycle-80全影響（メモ 19cd7b17191）
- kunYomi: 2,136字中1,214字（56.8%）にドット/ハイフン残存
- onYomi: 4字（王, 音, 応, 縁）にハイフン付き読み
- 88字に例語なし（grade=4の5字 + grade=7の83字）
- 旧カテゴリURL（/dictionary/kanji/category/nature等）→404（リダイレクトなし）
- 四字熟語辞典・色辞典は影響なし

## 計画に含めるべき作業

### 1. カテゴリ分類の再設計
- 現在の無意味な1-20部首グループを、ユーザーにとって価値のある分類に置き換える
- **複数の分類軸を提供する**ことを検討（学年別、画数別、部首別など）
- Ownerの指示: 「表示情報や絞り込みフィールドが1つである必要はないはずです」
- 既存のカテゴリナビゲーション（CategoryNav）の汎用性を活用できる
- URLの設計（/dictionary/kanji/grade/1, /dictionary/kanji/stroke/5 等）

### 2. データ品質の修正
- kunYomi: ドット（送り仮名区切り）とハイフンを除去してクリーンな読みにする
  - 例: 「た.りる」→「たりる」、「ひと-」→「ひと」
  - generate-kanji-data.ts で変換処理を追加し、kanji-data.jsonを再生成
- onYomi: 4字のハイフン付き読み（'-ノウ'等）を除去
  - 同じくgenerate-kanji-data.tsで対応
- 検索インデックス: クリーンなkunYomi/onYomiが使われれば自動的に修正される

### 3. 88字の例語なし問題への対応方針
- JMdictからの自動抽出でカバーできない88字について、対応方針を決定
- 88字は全体の4.1%であり、表示は条件付き非表示で対応済み
- 優先度は低いが、計画に含めて方針を明確にする

### 4. ゲーム側のデータ整合
- ゲーム側のKanjiEntry.categoryフィールドの扱い（ゲーム評価には使用されていない）
- kanji-data.jsonからcategoryフィールドを削除するかどうか
- ゲーム側のradicalGroupフィールドの扱い
