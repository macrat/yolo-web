# Q08 動物性格診断 実装計画

## 1. コンテンツ設計

### 基本仕様
- **スラグ**: `animal-personality`（計画番号プレフィックス不使用の指示に従う）
- **タイプ**: `personality`（診断カテゴリ）
- **タイプ数**: 12タイプ（推奨基準6〜8以上を大幅に上回る。12は十二支との親和性もあり、「日本の動物を12種揃える」という覚えやすい構成）
- **質問数**: 10問（推奨8〜12問の中央値。music-personalityと同じ10問で体験時間2〜3分を想定）
- **結果文字数**: 300〜500文字（推奨基準通り）

### スコアリング方式
music-personalityの3軸モデル（2^3=8タイプ）を参考にしつつ、12タイプへ拡張する。

**4軸モデル（各2値）で16の理論組み合わせから12タイプを採用する方式を推奨**:
- **S軸 (Social)**: 群れ型 (G) vs 単独型 (S) -- 社交性・集団志向
- **A軸 (Action)**: 行動型 (A) vs 慎重型 (C) -- 判断の速さ・リスク許容度
- **E軸 (Energy)**: 活動型 (H) vs 穏和型 (L) -- エネルギーの方向性
- **T軸 (Think)**: 直感型 (I) vs 分析型 (N) -- 物事の捉え方

各質問の選択肢は4つ。各選択肢がprimary（2pt）を1タイプに、secondary（1pt）を1タイプに付与するパターン（music-personalityと同じ構造）。

10問 x 4選択肢 = 40スロット。primary 2pt x 40 = 80ptをタイプに均等分配するため、各タイプにprimary 6〜7回、secondary 6〜7回を割り当てる。全タイプが出現する確率を均等にするためバランス設計が重要。

### 結果文の構成（各タイプ300〜500文字）
1. **動物名とキャッチコピー**（1行）: 例「ニホンモモンガ -- 空飛ぶハンカチの大冒険家」
2. **生態事実に基づく性格描写**（3〜4文）: 動物の実際の行動から性格を類推する面白さ
3. **文化イメージとのギャップ**（1〜2文）: 文化的に知られるイメージと実態の落差をユーモアにする
4. **自己発見の一言**（1〜2文）: 「あなたも〜なところがありませんか?」的な問いかけ
5. **ユーモラスなアドバイス**（1文）: 動物の生態にちなんだ斜め上のアドバイス
6. **保護情報（該当種のみ）**: 特別天然記念物・絶滅危惧種の場合は1文で触れる（constitution.md Rule 1・2との整合）

### SNSシェアテキスト
- Xシェア: 「あなたの日本の動物性格は「{動物名}」タイプ！{キャッチコピー} #日本の動物性格診断」
- LINEシェア: 「日本の動物性格診断で「{動物名}」タイプだった！あなたは何の動物？」

## 3. 質問設計の方針

### 質問の方向性
- **日常のシチュエーション問題**を中心とする（music-personality, character-fortuneと同様のスタイル）
- 4軸のうち各質問で2〜3軸を測定する（1問1軸だと質問数が足りない）
- 質問文は「あなたならどうする?」形式で、回答者が自然に性格を反映できるものにする

### 質問テーマ案（10問）
以下は方向性の案。builderが具体的な質問文と配点を設計する。

1. **集団行動vs単独行動**: 「休日の過ごし方は?」的な社交性測定
2. **判断スピード**: 「急に予定が変わったら?」的なAction軸測定
3. **リスク許容度**: 「知らない道と慣れた道、どちらを選ぶ?」
4. **エネルギー方向**: 「朝型か夜型か」「外出派かインドア派か」
5. **直感vs分析**: 「買い物の決め手は?」「第一印象派か情報収集派か」
6. **ストレス対処**: 「困った時にまず何をする?」
7. **コミュニケーション**: 「友達との連絡頻度は?」
8. **目標設定**: 「計画を立てる派vs流れに任せる派」
9. **変化への態度**: 「新しい環境に飛び込むか、慣れた環境を好むか」
10. **自己表現**: 「自分の気持ちをどう伝える?」

### 配点設計の原則
- 各選択肢にprimary（2pt）1タイプ + secondary（1pt）1タイプを設定
- 10問 x 4選択肢 = 40スロットのprimaryを12タイプに均等配分（各3〜4回）
- secondaryも同様に均等配分
- 特定タイプに偏らないように検証する（全パターン走査してタイプ出現分布を確認）

## 4. 相性機能

### 相性マトリクスの設計
- 12タイプの全組み合わせ: 12*(12+1)/2 = 78組（同タイプ12 + 異タイプ66）
- 既存のmusic-personality（8タイプ、36組）とcharacter-fortune（6タイプ、21組）と同じ構造
- キー形式: `"amami-kuro-usagi--ezo-shika"` （アルファベット順ソートで結合）

### 相性エントリの構造（CompatibilityEntry型を再利用）
```
{
  label: "相性ラベル"（例: "温泉仲間の絆"）,
  description: "相性説明文"（80〜150文字。動物の生態に基づくユーモラスな描写）
}
```

### 相性文の方針
- 動物同士の生態的な関係性（実際の生息域の重複、食物連鎖上の関係等）をベースにユーモアを加える
- 同タイプ同士: 「同じ種の2匹が出会ったら」的な想像
- 異タイプ同士: 生態の違いから生まれる面白い関係性を描写

### 実装パターン
music-personalityと同じパターンを踏襲:
- `getCompatibility(typeA, typeB)` 関数
- `isValidAnimalTypeId(id)` バリデーション関数
- `ANIMAL_TYPE_IDS` 定数配列

## 7. 注意事項・リスク

- **データ量**: 78の相性エントリは手動で書くと品質にばらつきが出やすい。全エントリで「動物の生態事実に基づくユーモア」という基準を一貫させること
- **タイプの偏り**: 配点設計後、全4^10 = 約100万通りの回答パターンでタイプ出現分布をシミュレーションする必要はないが、最低限「各選択肢のprimary配分が均等か」は確認すること
- **絶滅危惧種への配慮**: イリオモテヤマネコ、アマミノクロウサギの結果文に「特別天然記念物」であることを自然に織り込む。説教臭くならず、「あなたと同じくらい貴重な存在です」的なユーモアで触れる
- **coding-rules.md遵守**: 外部API不使用、DB不使用、認証不使用。全データはビルド時埋め込み。型安全を徹底（any禁止、QuizDefinition型に準拠）
