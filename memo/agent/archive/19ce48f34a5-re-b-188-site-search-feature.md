# B-188 実施計画: site-search-feature 記事修正

## 1. 想定読者と提供価値

### 想定読者
ターゲットは「AIエージェントやオーケストレーションに興味があるエンジニア」だが、本記事はbuilding-yolosシリーズの技術記事として、より広くWeb開発者（特にNext.jsでクライアントサイド検索の実装を検討しているエンジニア）にも価値がある。

### 提供価値
- クライアントサイド検索ライブラリ5種の比較と選定根拠（自分のプロジェクトに応用できる知見）
- 日本語コンテンツに対するファジー検索の具体的な設計判断と実装パターン
- 検索インデックスの遅延ロードによるパフォーマンス設計の考え方
- Cmd+Kモーダル方式のUI設計意図

読者が「自分のサイトでも検索を実装しよう」と思ったときに、判断材料として持ち帰れる情報を提供する記事であるべき。

## 2. セクション別の修正方針

Owner指示に従い、公開日時点の記述は原則そのまま残し、追記で変化を補足する。ただし事実の誤りは「公開日時点の記述」ではなく「誤記」なので修正対象とする。

### 2-1. タイトル・description
- **タイトル修正**: 「500件」という数字は公開時点でも正確かどうか不確実であり、かつ現在の2,899件と大幅に乖離している。タイトルから具体的な数字を除くか、表現を変更する。例: 「サイト内検索を実装しました -- Fuse.jsで日本語コンテンツを横断検索」
- **description**: 数字への言及があれば同様に調整する（現状のdescriptionには数字がないため変更不要の可能性が高い）

### 2-2. 「この記事で分かること」（冒頭約束）
- 「500件超の日本語コンテンツに対するファジー検索の実践」 → 数字を除外または更新する
- この約束に対応する内容が本文で薄い（問題E）ため、後述のセクション追記と連動して修正する

### 2-3. 「なぜサイト内検索が必要だったのか」
- 「合計500件以上のコンテンツが存在しています」 → 公開当時の記述として残しつつ、追記で現在のコンテンツ規模（約2,900件）に言及する

### 2-4. 「ライブラリ選定」
- 変更不要。内容は事実に基づいており、読者への価値も高い。

### 2-5. 「UI設計: Cmd+Kモーダル方式」
- 変更不要。

### 2-6. 「技術的な工夫: 遅延ロードでパフォーマンス影響ゼロ」
- **Route Handler記述の修正（問題A, B）**: これは「公開日時点の記述」ではなく事実の誤りである。実装当初からprebuildスクリプトでpublic/search-index.jsonを生成する方式だった（Route Handlerは存在しない）。以下を修正する:
  - 「Route Handler（/api/search-index）でインデックスを提供し」 → 「prebuildスクリプトで静的JSON（/search-index.json）を生成し」のような正確な記述に修正
  - 「ビルド時にforce-staticで静的JSONとして生成」 → 「ビルド前のprebuildスクリプト（scripts/build-search-index.ts）でpublic/ディレクトリにJSONを書き出す方式」に修正
  - 「Route Handler」へのNextJSドキュメントリンクを削除
  - 「約500件」 → 当時の文脈に合わせつつ、追記で現在規模に言及
- 修正は「技術的な工夫」セクションの価値を損なわないよう、遅延ロードのアーキテクチャ説明は維持する

### 2-7. 日本語ファジー検索に関する追記（問題E対応）
- 冒頭で「ファジー検索の実践」を約束しているが、本文での回収が不十分。以下の内容を追記する:
  - Fuse.jsの文字レベルマッチが日本語（漢字・ひらがな・カタカナ混在）にどう作用するか
  - 重み付け設定（title 2.0, keywords 1.5, description 1.0, extra 0.5）の設計意図と日本語検索での効果
  - しきい値設定の考え方（ファジーすぎると無関係な結果が出る、厳しすぎると見つからない）
- 「技術的な工夫」セクション内に小見出しとして追加するか、独立セクションとするかはbuilderの判断に委ねる。ただし記事全体の流れを壊さないこと

### 2-8. 「使い方」
- 変更不要。

### 2-9. 「今後の改善」
- 「マッチ部分のテキストハイライト強化」を削除（HighlightedTextコンポーネントとして実装済み）
- 残り4項目（検索履歴保存、人気検索ワード表示、モバイル戻るボタン対応、インデックスサイズ監視）はbacklog.mdのB-131, B-132, B-153, B-133と整合しているため維持
- 追記として「ハイライト表示は実装済みです」程度の補足を「今後の改善」の前に置くか、セクション外で軽く言及する

### 2-10. フロントマター
- updated_at をコミット直前の現在時刻に更新する（date コマンドで取得）

## 3. 調査で見つかった問題点への対処まとめ

| 問題 | 対処方針 | 根拠 |
|------|----------|------|
| Route Handler vs 静的JSON（問題A,B） | 事実誤りとして修正。正しい実装方式に書き換え | 実装当初からRoute Handlerは存在せず、「公開日時点の記述」ではなく誤記 |
| 500件 vs 2,899件（問題C） | タイトル・冒頭は数字を除外。本文中は当時の記述を残し追記で補足 | Owner指示「公開日時点の記述を尊重」に従う。ただしタイトルは検索結果に表示され読者の信頼に直結するため更新 |
| 実装済み機能が「今後」に記載（問題D） | 「今後の改善」から削除 | 事実と異なる状態を放置するのは読者の信頼を損なう |
| ファジー検索の約束未回収（問題E） | 日本語ファジー検索に関するセクションを追記 | blog-writing.md「冒頭の約束を必ず回収する」 |
| Route Handlerリンク（問題F） | 問題A修正時に連動して削除 | リンク先と記述内容の不整合解消 |

## 4. 完成基準

以下をすべて満たした場合に修正完了とする:

1. タイトルに現在と乖離する具体的数字が含まれていない
2. 「技術的な工夫」セクションの実装方式の記述が実際のコード（public/search-index.json, scripts/build-search-index.ts, useSearch.ts）と一致している
3. 「この記事で分かること」で約束した4点がすべて本文で回収されている（特にファジー検索の実践）
4. 「今後の改善」に実装済みの項目が含まれていない
5. 全内部リンクが有効なルートを指している
6. フロントマターのupdated_atが更新されている
7. blog-writing.mdの品質チェックリスト全項目を満たしている
8. 記事を通読して、読者が「自分のプロジェクトに検索を実装する際の判断材料」を得られる内容になっている
