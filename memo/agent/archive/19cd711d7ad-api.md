# 調査依頼: 漢字カナルのサーバーサイド評価API方式の妥当性検討

## 背景
漢字カナル（Wordleライクな漢字当てゲーム）のカテゴリ評価にEmbeddingベクトルを使う計画（B-190）がある。現在の計画はクライアントサイドでEmbeddingデータをfetchし、コサイン類似度を計算する設計。

これに対し、「Embeddingデータをサーバーサイドで保持し、API経由で正誤判定を返す方式にすべきではないか」という提案があった。evaluateGuess全体（文字一致、部首、画数、学年、音読み、訓読み数、カテゴリ）をサーバーサイドに移し、ターゲット漢字をクライアントに送らない設計。

## 調査してほしいこと

### 1. 現行アーキテクチャの全体像
- src/games/kanji-kanaru/ 配下のコードを読み、現在のデータフロー（パズルスケジュール→ターゲット決定→推測評価→フィードバック表示）を把握すること
- ターゲット漢字が現在どこで決定され、どこに保持されているかを特定すること
- ゲーム状態（localStorage保存）の構造を確認すること

### 2. サーバーサイドAPI方式のメリット・デメリット分析
以下の観点で徹底的に分析すること:

**メリット候補**:
- チート対策（ターゲットがクライアントに露出しない）
- Embeddingデータのダウンロード不要（400-500KB削減）
- クライアントコードの簡素化

**デメリット候補**:
- ネットワーク依存（オフライン不可、レイテンシ）
- サーバー負荷（Vercel Hobbyプランの制約を確認）
- 複雑性の増大（クライアント-サーバー分離）
- 既存の同期的なゲームフローへの影響
- ゲーム状態の復元（リロード時の過去の推測の再評価）
- パズルスケジュールデータのサーバーサイド保持の設計変更量

### 3. クライアントサイド方式のメリット・デメリット分析
同じ観点でクライアント方式も分析すること。

### 4. 他のWordle系ゲームのアーキテクチャ調査
- 本家Wordleや類似ゲームがサーバーサイドAPIを使っているか、クライアント完結かを調査
- それぞれの方式を採用している理由や背景

### 5. coding-rules.mdとの整合性
- docs/coding-rules.md を読んで、サーバーサイドAPI方式が技術制約に適合するか確認
- Route Handlersの既存実績（src/app/api/）も確認

### 6. 総合的な推奨
上記の分析を踏まえ、どちらの方式を採用すべきかの推奨を出すこと。推奨の根拠を明確にすること。

## 注意
- 「どちらでもよい」という結論は不可。明確にどちらかを推奨すること
- メリット・デメリットを列挙するだけでなく、このプロジェクト固有の状況（Vercel Hobbyプラン、個人サイト、ゲームの規模等）を踏まえて判断すること
