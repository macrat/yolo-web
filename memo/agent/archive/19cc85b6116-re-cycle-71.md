# お問い合わせページ 調査レポート

## 調査概要

yolos.netのお問い合わせページ実装に向け、以下の4項目を調査した。

1. 技術制約下で実現可能な選択肢と比較
2. AdSense審査での最低要件
3. 個人サイトの事例
4. 既存サイト構造の確認

---

## 1. 既存サイト構造の確認

### 現状

- `/about` ページ（`src/app/about/page.tsx`）にお問い合わせセクションが存在するが、内容はGitHubリポジトリのIssuesへのリンクのみ。
- フッター（`src/components/common/Footer.tsx`）にお問い合わせへのリンクはなし。
- 専用のお問い合わせページ（`/contact`など）は存在しない。

### 課題

GitHubのIssuesへのリンクは技術者向けのアクセス方法であり、一般ユーザーには敷居が高い。AdSense審査の観点でも「一般ユーザーが連絡できる手段」として機能していない可能性が高い。

---

## 2. AdSense審査での最低要件

### 公式ガイドラインの位置づけ

Google公式のAdSenseヘルプ（support.google.com/adsense/answer/1348695）に「お問い合わせページ」が必須として明示されているわけではない。しかし、複数の情報源（AdPushup、Webtimize Solutions等）およびAdSense合格体験記から、以下が共通認識として確立されている。

**About Us・Privacy Policy・Contact Us の3ページは「審査通過の必須前提」とみなされている。**

### フォーム必須か、メールアドレス記載だけでよいか

明確な公式要件はないが、複数の実証的な情報から以下が結論として得られた。

- **メールアドレスの記載のみでも審査通過は可能**（「メールアドレスなど連絡が取れる連絡先をはっきりと書いておくことも必須と言っていいほど重要」という情報源あり）。
- ただし、フォームの方が**迷惑メール対策**として実務上推奨されており、サイトの**信頼性・誠実さ**をより強くアピールできる。
- 「読者が連絡を取れる手段があるか」が審査上の本質的な要件であり、手段の形式（フォームかメールか）は問わないとみられる。

**結論：メールアドレス記載のみで最低要件は満たせる。フォームは推奨だが必須ではない。**

## 3. 技術制約下での実現可能な選択肢

coding-rules.md の技術制約：

- 禁止：ランタイムの外部API呼び出し / DB / 認証
- 可能：Route Handler（サーバーサイドJS）/ クライアントJS

この制約を踏まえ、選択肢を評価する。

### 選択肢 A：mailtoリンク（メールアドレス掲載）

**仕組み：** `<a href="mailto:xxx@example.com">` でメールクライアントを起動させるだけ。完全に静的なHTML。

| 観点 | 評価 |
|---|---|
| 実装コスト | 最低（1行） |
| 技術制約への適合 | 完全に適合（外部API一切なし） |
| AdSense要件の充足 | 充足（メールアドレス記載で要件を満たす） |
| UX | 低い（メールクライアントが必要、モバイルでの操作感が悪い） |
| 迷惑メール対策 | なし（アドレスがクローラーに収集されやすい） |
| プライバシー | ユーザーデータを外部に送らない |

### 選択肢 B：Googleフォーム iframe 埋め込み

**仕組み：** Googleフォームの公開URLをiframeで埋め込む。フォーム送信はGoogleのサーバーで処理される。

| 観点 | 評価 |
|---|---|
| 実装コスト | 低（iframeタグを貼るだけ） |
| 技術制約への適合 | **要注意**：iframeはGoogleのサーバーへのリクエストを含むが、これはブラウザが行うフレーム読み込みであり、サーバーサイドのAPI呼び出しではない。クライアントサイドの外部コンテンツ表示と同じ扱いで、coding-rules.mdの「ランタイムの外部API呼び出し禁止」には抵触しない。 |
| AdSense要件の充足 | 充足（フォームとして機能） |
| UX | 中（iframeのデザインがサイトと不一致になりやすい、モバイルでの表示が難しい） |
| 迷惑メール対策 | あり（Googleのスパムフィルタが機能） |
| プライバシー | ユーザーデータがGoogleに送られる（Googleフォームの利用規約に従う） |

**補足：** Googleフォームはデザインカスタマイズが非常に限定的で、サイトのデザインと統一できないデメリットがある。

### 選択肢 C：Formspree / FormSubmit 等の外部フォームサービス

**仕組み：** HTMLフォームのaction属性を外部サービス（formspree.io等）のエンドポイントに向ける。ブラウザからPOSTリクエストが送信される。

| 観点 | 評価 |
|---|---|
| 実装コスト | 低〜中（HTMLフォームを自作、送信先をサービスのURLにする） |
| 技術制約への適合 | **要注意**：これはブラウザ（クライアント）からの外部API呼び出しになる。coding-rules.mdは「ランタイムの外部API呼び出し」を禁止しているが、この禁止はユーザーデータ漏洩リスクを防ぐための規定。Formspree/FormSubmitへのフォームデータ送信はまさに「ユーザーデータ（氏名・メールアドレス・メッセージ）を外部サービスに送る」行為であり、**制約の趣旨に明確に抵触する。** |
| AdSense要件の充足 | 充足 |
| UX | 高（サイトのデザインに合わせたフォームが作れる） |
| 迷惑メール対策 | あり（reCAPTCHA等） |
| プライバシー | ユーザーデータが第三者サービスに送られる。制約違反。 |

**結論：Formspree等は制約の趣旨に反するため不採用。**

### 選択肢 D：Next.js Route Handler + メール送信API（Resend/SendGrid等）

**仕組み：** 自前のRoute Handlerでフォームデータを受け取り、外部メール送信APIを呼び出してメールを送信する。

| 観点 | 評価 |
|---|---|
| 実装コスト | 高（Route Handler実装、外部メールAPIのセットアップ、環境変数管理） |
| 技術制約への適合 | **禁止に抵触**：Route HandlerからResend/SendGrid等の外部APIをランタイムで呼び出すことは、coding-rules.mdの「ランタイムの外部API呼び出し禁止」に明確に該当する。 |
| AdSense要件の充足 | 充足 |
| UX | 高 |
| 迷惑メール対策 | あり |
| プライバシー | ユーザーデータが外部メールサービスに送られる。制約違反。 |

**結論：技術制約に明確に違反するため不採用。**

## 5. 推奨案

### 推奨：選択肢AとBの組み合わせ

**段階1（最低限・即実装可能）：メールアドレス掲載**

- `/contact` ページを新規作成し、連絡先メールアドレスを掲載する
- mailtoリンクとして実装（完全静的、技術制約に完全適合）
- AdSense審査の最低要件は満たせる
- 現行のAboutページのお問い合わせセクションを `/contact` へのリンクに変更
- フッターに `/contact` リンクを追加

**段階2（品質向上・Googleフォームの iframe 埋め込み）**

- Googleフォーム（無料・無制限）をiframeとして埋め込む
- サイト内でGoogleのサービスは既にGA（GoogleAnalytics）を使用しており、Googleとの連携に一定の前例がある
- iframeは技術的にはブラウザが外部コンテンツを読み込む仕組みであり、サーバーサイドのAPI呼び出しには該当しない
- デザインの不統一は甘受するか、フォームの下に「または上記のメールアドレスへご連絡ください」と補足する形で対応

**メールアドレスについて**

- 公開メールアドレスは迷惑メールの温床になるリスクがある
- 対応策：Gmailの「+」エイリアス（`username+yoloscontact@gmail.com`）や、ドット区切りの書き方（`name [at] example.com`）でクローラー対策をする手もある
- または、Googleフォームのみを窓口とし、メールアドレスは非公開にする

### 最小実装案（最速でAdSense要件を満たす場合）

`/contact` ページに以下のみを掲載する：

```
お問い合わせ

本サイトに関するご質問・ご意見は、以下のメールアドレスまたはGitHubにてお受けしています。

メール: [メールアドレス]
GitHub Issues: [GitHubリポジトリURL]
```

この最小実装であれば実装コストは数十分程度。AdSense審査の要件（連絡手段の提供）は満たす。
