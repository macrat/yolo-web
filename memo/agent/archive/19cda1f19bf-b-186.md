# B-186調査結果: 四字熟語きめる現状把握と差別化要素

## 1. 四字熟語きめるの現状

### ゲームルール・仕組み
- Wordle型の四字熟語当てゲーム（毎日1問、6回以内に正解）
- 推測ごとに4文字それぞれの位置フィードバックを表示: 緑（正位置）・黄（別位置に存在）・灰（存在しない）
- 入力バリデーション: 漢字4文字（CJK U+4E00〜U+9FFF, U+3400〜U+4DBF）のみ許可
- 重複入力を検出してエラー表示
- ヒントシステム（段階的開示）:
  - 常時: 難易度（★〜★★★）
  - 3回目の推測後: 読みの最初の文字（例: "い..."）
  - 5回目の推測後: カテゴリ（例: "人生・生き方"）
- エポック: 2026-02-14（パズル#1）
- スケジュールが切れた場合はFNV-1a系ハッシュで決定的フォールバック

### 現在のデータ量
- **四字熟語データ: 101件**（`src/data/yoji-data.json`）
- 難易度分布: 難易度1(基本)=55件、難易度2(中級)=41件、難易度3(上級)=5件
  - 上級がわずか5件（全体の5%）と著しく少ない
- カテゴリ分布: 10カテゴリ、各カテゴリ10〜11件と均等
- スケジュール: 365日分（`src/games/yoji-kimeru/data/yoji-schedule.json`）
  - 範囲: 2026-02-14 〜 2027-02-13
  - 使用するインデックス: 0〜99（101件中99件を使用）
  - **1日1問で同じ四字熟語が2〜3回繰り返し出現する**（365日で101種類を使い回し）

### データスキーマ（YojiEntry）
```typescript
interface YojiEntry {
  yoji: string;        // 四字熟語（4文字）
  reading: string;     // 読み（ひらがな）
  meaning: string;     // 意味
  difficulty: number;  // 1-3
  category: YojiCategory; // 10カテゴリ
}
```

カテゴリ一覧: life/effort/nature/emotion/society/knowledge/conflict/change/virtue/negative

### UIの構成
- GameHeader: パズル番号・日付・ヘルプ・統計ボタン
- HintBar: ヒント情報（難易度・読み・カテゴリ）
- GameBoard: 6×4マスのフィードバックグリッド
- GuessInput: テキスト入力（IME対応）+ 送信ボタン
- ResultModal: 結果表示・答え・意味・シェアボタン・カウントダウン
- StatsModal: 統計表示（プレイ数・勝率・ストリーク・分布）
- HowToPlayModal: 遊び方説明

### データ配置
- 四字熟語マスターデータ: `src/data/yoji-data.json`（辞典と共有）
- パズルスケジュール: `src/games/yoji-kimeru/data/yoji-schedule.json`
- GameContainerでimport、クライアントサイドで動作

## 2. 漢字カナルとの比較

### 共通点
- Wordle型デイリーパズル（毎日1問、6回以内、LocalStorage保存）
- 統計・シェア・遊び方の3モーダル
- MAX_GUESSES = 6
- エポック日付方式でのパズル番号管理
- 実績システム（useAchievements）・GA4トラッキング統合

### 漢字カナルが持っていて四字熟語きめるにない機能

| 機能 | 漢字カナル | 四字熟語きめる |

| フィードバック軸数 | 7軸（部首/画数/学年/音読み/意味/訓読み数/学年方向） | 4軸（位置のみ） |
| データ量 | 2,136字（全常用漢字） | 101件 |
| スケジュール | 難易度別3ファイル×730日分 | 1ファイル365日分 |
| サーバーサイドAPI | 評価・ヒント取得（B-190で導入） | なし（クライアント完結） |
| DifficultySelector UI | あり | なし |
| 初期ヒント | 画数・音読み数・訓読み数（常時表示） | 難易度のみ（常時） |

### cycle-80で漢字カナルに行われた拡張の詳細

- タスク4: パズルスケジュール再生成（難易度別3ファイル、各730日分）
- タスク5: バンドルサイズ確認（131KB gzip）・ライセンス帰属表記・最終テスト
- 7属性フィードバック: 部首/画数/学年（方向矢印付き）/音読み/意味（embedding類似度）/訓読み数

## 3. データ構造の分析

### データファイルの場所
- マスターデータ: `/mnt/data/yolo-web/src/data/yoji-data.json`（辞典機能と共有）
- スケジュール: `/mnt/data/yolo-web/src/games/yoji-kimeru/data/yoji-schedule.json`

### データ品質の評価
- 読み・意味フィールドが完備
- カテゴリが10種類と適切に定義済み
- **問題点1**: 上級(難易度3)が5件のみ（偏りが大きい）
- **問題点2**: 101件と少なく、1年365日のスケジュールで繰り返し出現する
- **問題点3**: 出典・由来フィールドが存在しない（B-092で指摘済み）

### 拡充の容易さ（スキーマの柔軟性）
- スキーマは非常にシンプルで、フィールド追加が容易
- 追加候補フィールド（他ゲームの拡充例から示唆）:
  - `origin`: 出典・由来
  - `tags`: タグ（類義語グループ等）
  - その他の補助情報
- マスターデータ（yoji-data.json）は辞典機能と共有しているため、フィールド追加は辞典側にも波及する可能性がある
- スケジュールファイルの再生成スクリプトは存在すると思われるが、現在のスケジュールはJSON形式で手動管理

## 5. 改善方針への示唆

### データ量拡充
- **101件 → 365件以上が最低ライン**（1年間でリピートなしにするため）
- 理想は730件以上（2年分）
- 難易度3（上級）が5件のみ→最低50件以上に増強が必要
- データ追加は `src/data/yoji-data.json` への追記だけでなくスケジュール再生成が必要

### 漢字カナルとの差別化強化
現在の差別化ポイント（四字熟語きめるのみ）:
- 対象が「4文字の四字熟語」という単語レベルの推測
- 各文字の位置情報がフィードバックの主軸

差別化を強化するための方向性（検討材料として提示）:
1. **読み情報の活用**: 四字熟語ならではの「読み（ひらがな）」ヒントを拡充
2. **意味・分類の多角的フィードバック**: カテゴリのみでなく、意味的な距離フィードバックの追加
3. **四字熟語特有の属性活用**: 出典（仏典/漢籍/日本語起源等）、構造（AABB型/対義型等）
4. **難易度選択機能**: 漢字カナルと同様に3難易度化し、より広いユーザー層への対応
5. **辞典連携強化**: 結果モーダルで四字熟語辞典への導線がすでにあるが、より深い解説への誘導

### スキーマ拡充候補
現行スキーマに追加可能なフィールド:
- `origin: string` — 出典（漢籍/仏典/日本語由来等）
- `structure: string` — 構造パターン（例: AABB型、対義構造等）
- `tags: string[]` — タグ・関連キーワード

これらのフィールドは難易度別フィードバックや辞典表示の充実に活用可能。

## 6. まとめ

| 項目 | 現状 | 目標水準（参考：漢字カナル） |
|------|------|--------------------------|
| データ数 | 101件 | 365件以上（繰り返しなし1年分） |
| 難易度選択 | なし | 3段階 |
| フィードバック軸 | 位置のみ（4軸） | 多属性（漢字カナルは7軸） |
| スケジュール | 365日（繰り返しあり） | 730日以上 |
| 実装方式 | クライアント完結 | 漢字カナルはサーバーサイドAPI |

最も優先度の高い課題は**データ量（101件）の不足**であり、1年間で同じ四字熟語が複数回出現する点はデイリーパズルとして致命的な品質問題である。次いで**難易度3（上級）の著しい偏り（5件）**と**漢字カナルとの差別化要素の弱さ**が課題として挙げられる。
