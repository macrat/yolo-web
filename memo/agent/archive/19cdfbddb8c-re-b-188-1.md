## P-1: yolos.net固有のコンポーネント名・パス名が説明なしで使用されている

blog-writing.mdの品質チェック項目「yolos.net固有のアーキテクチャ名、コンポーネント名、ワークフロー用語を説明なしで使用していない」に照らし、以下が問題となる。

### 具体的な箇所

1. **ToolRenderer.tsx / CheatsheetRenderer.tsx**: 記事内で「ToolRenderer.tsx」「CheatsheetRenderer.tsx」というファイル名が繰り返し使用されている。これらはyolos.net固有のファイル名であり、読者にとっては単なる「動的インポートを行うクライアントコンポーネント」でしかない。改良版では「Renderer」という汎用名に変更されており、こちらの方が読者に伝わりやすい。

2. **@/tools/registry, @/cheatsheets/registry**: コードサンプル内でyolos.net固有のインポートパスが使用されている。`toolsBySlug`、`cheatsheetsBySlug`、`getAllToolSlugs`なども同様。読者が自分のプロジェクトに置き換えるには、これらがyolos.net固有であることを理解する必要があるが、その説明がない。

3. **ToolLayout, CheatsheetLayout, ToolErrorBoundary**: テンプレートパターンのコードサンプル内にyolos.net固有のコンポーネント名が使われている。

改良版ではこれらが汎用的な名前（`registry`, `PageLayout`, `ErrorBoundary`）に置き換えられており、読者の応用可能性が高い。

## P-2: 冒頭NOTEの内容が読者にとって不要な内部事情を含む

55行目:
> この記事で解説しているアーキテクチャ（33ツール＋7チートシートの個別ページ構成）は、サイトリメイクに伴い変更予定です。

「サイトリメイク」はyolos.net内部の文脈であり、外部の読者にとっては何のことかわからない。読者の関心は「この技術が自分のプロジェクトで使えるか」であり、yolos.netのサイトリメイク予定には関心がない。

改良版にはこのNOTEが存在しない点も、改良版の方が読者体験として優れていることの傍証となる。

## P-3: 「プロジェクトオーナーの判断」という表現が読者にとって不自然

166行目:
> プロジェクトオーナーの判断により、**UXを最優先する基準**でアプローチAが採用されました。

「プロジェクトオーナーの判断」は内部の意思決定プロセスに関する表現であり、技術記事の読者にとっては不要な情報である。読者が知りたいのは「なぜアプローチAが優れているか」の技術的根拠であって、「誰が決めたか」ではない。

改良版では「UXを最優先し、実装コストを理由にUXを妥協しないことを根本原則としています。この原則に基づき」という表現になっており、判断の主体ではなく判断の原則に焦点が当てられている。

## 技術的正確性の確認結果

- next/dynamicがReact.lazy()とSuspenseの組み合わせであるという記述: Next.js公式ドキュメント（https://nextjs.org/docs/app/guides/lazy-loading）の「next/dynamic is a composite of React.lazy() and Suspense」と一致
- サーバーコンポーネントをdynamic()でインポートした場合の挙動: 同ドキュメントの「If you dynamically import a Server Component, only the Client Components that are children of the Server Component will be lazy-loaded」と一致
- ハイドレーション時のローディングフラッシュの発生メカニズム: React.lazyとSuspenseの動作原理として正確
