# B-190アーキテクチャ決定: フルサーバーサイドAPI方式

## 決定事項
案B（フルサーバーサイドAPI方式）を採用する。

## 根拠（6つの調査結果に基づく）

### 決定的要因: 初回ロードサイズ
- 案A: page-specific 134KB gzip + Embeddings 600KB gzip = 746KB gzip → 3G環境で5-9秒（Google推奨3秒閾値超過、離脱率大幅悪化）
- 案B: page-specific 14-51KB gzip → 3G環境で0.6-1.1秒（快適範囲）
- 現在のkanji-kanaruは他ゲーム(50-67KB)の10倍のバンドルサイズ。案Bで正常化される

### 許容可能な代償: フィードバック遅延
- 50-200msのAPI遅延はNielsenの「思考の流れを維持できる」範囲（1秒未満）
- ターン制ゲームでは300ms程度まで許容される（研究知見）
- フリップアニメーション(400ms)は開始に色情報が必要なため、遅延を隠すには2段階アニメーション化が必要。ただし50-100ms程度なら「ほぼ即座」と知覚される

### 設計原則との整合
- coding-rules Rule 1: 「バンドルサイズが大きくなる機能はサーバーコンポーネントやAPIルートで実装する」→ 687KBのkanji-data.jsonは明確に該当
- NYT Wordleの先例: 答えはサーバーサイドだが、バリデーション+色付けはクライアント維持→同じパターンを採用

## 設計方針

### サーバーサイドに移すもの
- kanji-data.json（687KB）
- puzzle schedules（122KB）
- Embeddingデータ（384次元、1,087KB）
- evaluateGuess（文字一致、部首、画数、学年、音読み、訓読み数、カテゴリ）

### クライアントに残すもの
- バリデーション: 常用漢字チェック（2,136字のリスト、約6KB）、重複チェック
- ゲームUI: GameContainer, FeedbackCell, GuessInput, HintBar, ResultModal等
- localStorage: 推測履歴の保存

### APIエンドポイント設計
1. **POST /api/kanji-kanaru/evaluate** - 推測を受け取りフィードバックを返す
   - Request: { guess: "海", puzzleDate: "2026-03-10", difficulty: "intermediate" }
   - Response: GuessFeedback（文字一致、部首、画数、学年、音読み、カテゴリ、訓読み数）+ ゲーム終了時はtargetKanjiの表示用情報
2. **GET /api/kanji-kanaru/hints** - ヒント情報を返す（HintBar用: strokeCount, readingCount, kunYomiCount）
   - puzzleDate + difficulty でターゲットを特定

### セーブデータ復元方式
- localStorageに保存: guessChars: string[] + feedbacks: GuessFeedback[]（フルデータ保存）
- ページロード時: localStorageからfeedbacksを直接復元（API再評価不要）
- 旧形式（guessCharsのみ）が存在する場合: N回のAPIコールで再評価するフォールバック

### バリデーションフロー
1. クライアント: 文字数チェック → 常用漢字チェック → 重複チェック（すべて同期、0ms）
2. すべてパスしたら: POST /api/kanji-kanaru/evaluate を呼び出し
3. レスポンスを受けてフィードバック表示

## 計画修正の依頼

前回のサーバーサイドAPI計画（メモ 19cd713edf8）をベースに、以下のレビュー指摘（メモ 19cd71f661f）を反映した修正版を作成してください。

### レビュー指摘の反映
1. **HintBar方式**: /api/kanji-kanaru/hints エンドポイントを設けてヒント情報を取得する設計に統一。targetKanjiのクライアント保持は廃止。
2. **ResultModal**: ゲーム終了時のevaluateレスポンスにKanjiEntryの表示用フィールド（character, onYomi, kunYomi, meanings, examples）を含める。
3. **lookupKanji**: クライアントのバリデーションに必要なので保持。ただしkanji-data.json全体ではなく、文字リストのみの軽量データ（約6KB）に置き換える。

### 調査結果からの追加要件
4. **セーブデータ方式**: feedbacksをlocalStorageにフル保存し、ページロード時のAPI再評価を不要にする（調査5の知見）。旧形式のフォールバックも用意。
5. **バリデーションはクライアント**: 有効入力後のみAPIコール（調査5の結論）。
6. **2段階アニメーションは不要**: 50-100msの遅延は「ほぼ即座」と知覚される。単純にAPIレスポンス後にフリップ開始でよい（調査1の知見）。
