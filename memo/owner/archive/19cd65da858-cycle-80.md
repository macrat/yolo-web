# 事故報告: cycle-80 部首グループ分類によるゲーム体験の破壊

## 事故の概要

cycle-80「漢字カナル常用漢字2,136字への拡張」において、カテゴリ分類を「セマンティックカテゴリ（AI分類）」から「部首グループ（康熙字典部首番号の範囲による機械的分類）」に変更した。この変更により、プレイヤーにとって直感的に理解不可能なグルーピングが導入され、カテゴリフィードバックがゲームのヒントとして機能しなくなった。

具体例: 部首「女」（部首番号38）と部首「夕」（部首番号36）が同じグループ5「天文と時間」に分類されている。プレイヤーが「女」の漢字を推測して「近い部首グループ」と表示されても、「夕」を連想することは不可能であり、「天文と時間」というグループ名からも「女」が含まれることは理解できない。

これはconstitution.md Rule 2（ユーザーを悲しませるコンテンツを作らない）、Rule 4（品質優先）に正面から違反する重大事故である。変更前のセマンティックカテゴリ（nature, bodyなど意味ベースの分類）では提供できていた価値を完全に破壊し、プレイヤーを混乱させ苛立たせる理不尽なフィードバックで置き換えてしまった。

## 事故の影響

- **ウェブサイトは現在不正な状態にある**: 本番環境にデプロイ済み（git push完了）
- **ゲームとしてプレイできないレベルで破綻している**: カテゴリフィードバックが手がかりにならないため、2,136字から6回で絞り込むことが不可能に近い
- **変更前に提供できていた価値の完全な破壊**: 80字版のセマンティックカテゴリ（nature, animal等）は意味的に直感的で、プレイヤーにとって有効なヒントだった

## 何が問題だったのか

### 根本原因: 「プレイヤー体験」の完全な欠落

計画・レビュー・実装の全フェーズにおいて、「プレイヤーがこのフィードバックを受け取ったとき、次の推測に活かせるか？」という最も基本的な検証が一度も行われなかった。

全てのレビューは以下の観点にのみ集中していた:
- データの正確性・確定性（AI分類を避けることの安全性）
- ゲームバランス（数学的に6回で解けるか）
- 技術品質（型安全性、テストカバレッジ）

これらはすべて「開発者・技術者の視点」であり、「プレイヤーの視点」が完全に欠落していた。

### 具体的な問題箇所

1. **グループ5「天文と時間」に「女」（部首番号38）が含まれる**: 「女」は「天文」とも「時間」とも関係がない。「大」（部首番号37）、「子」（部首番号39）も同グループ。

2. **グループ7「動作基本」に「心」（部首番号61）が含まれる**: 「心」は「動作」ではない。

3. **グループ9「木と欠」に「水」（部首番号85）と「火」（部首番号86）が含まれる**: 「木」「水」「火」はそれぞれ全く異なる意味カテゴリであり、「木と水と火が近い」というフィードバックはプレイヤーにとって理不尽。

4. **スーパーグループ全般**: 番号が隣接しているだけのグループをペアにしているため、「基本図形」と「人と体上部」が近い、「口と囲い」と「天文と時間」が近い、など意味的に無関係なペアが大量に存在。

### PMの判断の誤り

PMは「AI分類のリスクを回避する」という一点のみを重視し、「機械的で確定的」であることを無条件に肯定した（メモ 19cd5b526af）。cycle-66の「部首グループに変更すれば完全自動化可能」という記述を、「部首番号の範囲で区切れば良い」と安易に解釈し、プレイヤー体験への影響を検証しなかった。

### レビューの形骸化

3回の計画レビュー（B+→B+→A）と1回の実装レビュー（A）を経ているが、いずれのレビューでもプレイヤー体験の検証は行われなかった。reviewerはcategory分類の「一貫性と再現性」を肯定し、「意味的な主観判断を排除している点は良い」と評価した（メモ 19cd5bfc10e, 19cd60bd537）。これは技術的な正しさとユーザー価値を混同した判断であり、レビューの本来の目的（ユーザーに提供する価値の確認）を果たしていない。

## なぜ事故が起きたと考えられるか

### 1. 「機械的 = 安全」というバイアス

AI分類のリスク（曖昧さ、検証困難、再現性欠如）を回避するために「機械的分類」が選ばれたが、「機械的であること」と「プレイヤーにとって有用であること」は全く別の基準である。機械的であれば安全という思い込みが、ユーザー体験の検証を省略する原因となった。

### 2. cycle-66の判定の無批判な踏襲

cycle-66の「部首グループに変更すれば完全自動化可能」は実現可能性の評価であり、ゲーム体験の評価ではなかった。しかしcycle-80では、この実現可能性の評価を「設計の正しさ」として無批判に採用した。「自動化可能」であることと「プレイヤーにとって良い体験」であることは異なる。

### 3. レビュー観点の欠落

reviewerへの指示にもレビュー基準にも、「プレイヤーが実際にプレイした場合の体験」を検証する観点が含まれていなかった。レビューは技術品質、データ品質、ゲームバランス（数学的保証）に集中しており、「このフィードバックはプレイヤーにとって意味があるか」という最も重要な観点が抜け落ちていた。

### 4. 数学的正しさとユーザー体験の混同

「6回で確実に解ける」ことの数学的証明に大量のリソースが投じられたが、「6回で解ける」ことと「プレイヤーが楽しく解ける」ことは別の問題である。数学的にはカテゴリフィードバックが情報を持っていても、プレイヤーがその情報を解釈できなければゲームとして成立しない。

## ドキュメントや手順書への改善提案

### 1. レビュー基準の追加

reviewerへの指示や.claude/skills/cycle-planning/SKILL.md、cycle-execution/SKILL.mdに、以下の観点を必須レビュー項目として追加すべき:

- **エンドユーザー体験テスト**: ゲームやインタラクティブコンテンツの変更時、具体的なプレイシナリオを想定し、「プレイヤーがこのフィードバックを受け取ったとき、次の行動を合理的に判断できるか」を検証する
- **一般常識テスト**: フィードバックやヒントが、専門知識なしに（一般的な教養のみで）理解可能かを検証する
- **前後比較テスト**: 既存機能を変更する場合、変更前に提供できていた価値が変更後も維持されているかを検証する

### 2. constitution.mdの参照強化

constitution.md Rule 2「ユーザーを悲しませるコンテンツを作らない」は、技術的品質だけでなくユーザー体験の品質にも適用されることを、レビュー基準で明示すべき。

### 3. 「技術的に正しい」と「ユーザーにとって正しい」の区別

CLAUDE.mdまたはレビュースキルに、以下の原則を追加すべき:

- 技術的な正しさ（データの確定性、再現性、型安全性）はユーザー価値の必要条件であるが十分条件ではない
- 最終的な判断基準は常に「エンドユーザーにとっての体験」であり、技術的指標はその手段に過ぎない
- 「機械的に決定される」「再現性が高い」等の技術的美点は、ユーザー体験を犠牲にして追求してはならない

### 4. ゲーム変更時のプレイテスト義務化

ゲームの仕様変更を行う場合、レビュー時に最低限以下のプレイテストシナリオを実施する義務を設けるべき:
- 最善ケース（1-2回で正解に辿り着ける場合）のプレイスルー
- 最悪ケース（最大回数近く必要な場合）のプレイスルー
- 各フィードバックが「プレイヤーにとって解釈可能な手がかりを提供しているか」の検証
