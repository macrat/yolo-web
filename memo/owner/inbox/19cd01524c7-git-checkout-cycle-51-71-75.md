# 事故報告: git checkoutによる未コミット作業消失の繰り返し

**報告者**: PM
**発生日時**: 2026-03-09（cycle-75）、過去事例: 2026-03-01（cycle-51）、2026-03-07（cycle-71）
**重大度**: 高（同種事故の3回目の発生）

---

## 1. 今回の事故（cycle-75）

### 何が起きたか

cycle-75の完了直前、Ownerから55組問題の結論の記述について指摘を受けた。PMはcycle-75.mdの当該セクションを修正しようとしたが、まず誤った内容の修正を取り消すために `git checkout docs/cycles/cycle-75.md` を実行した。

このファイルには、サイクル中に蓄積した以下の全記録が未コミットの状態で含まれていた:

- タスク完了チェック4件
- 対応方針の決定結果（全4記事公開停止の判断と根拠テーブル）
- 書き直しに関する分析結果
- 計画レビュー・実装レビューの記録
- キャリーオーバー（B-181）
- 補足事項（55組問題の結論）

`git checkout` により、ファイルは初期コミット時の状態（作業開始時のテンプレート）に戻り、上記すべてが消失した。

### 他ファイルへの影響

4記事のdraft化、blog-writing.mdの更新、backlog.mdの更新は別ファイルであるため無事だった。cycle-75.mdのみが被害を受けた。

### 復旧

メモシステムに全決定の根拠が残っていたため、cycle-75.mdの内容は復元可能だった（メモ 19ccffa34c9, 19cd00249fe, 19ccff66042等）。ただし、復元作業自体に時間とトークンを消費した。

### フックが機能しなかった理由

block-destructive-git.shは `git checkout -- <path>` パターン（`--` 付き）を検知するが、今回のコマンドは `git checkout docs/cycles/cycle-75.md`（`--` なし）であった。gitは `--` なしでもファイル復元として動作するため、フックの検知パターンにギャップがあった。

## 3. 横断分析: 共通パターン

3回の事故に共通する構造的問題を抽出する。

### パターンA: 大量の未コミット作業の蓄積

| サイクル | 未コミット作業の内容 | 蓄積量 |
|---------|-------------------|--------|
| 51 | builderのブログ全面書き直し + レビュー修正 | ~382,000トークン分 |
| 71 | 3 builderの並行作業（3ページ分） | 3builder分 |
| 75 | サイクル全体の進捗記録（決定、レビュー結果等） | サイクル1つ分の全記録 |

全事故で、コミットされていない作業が長時間ワーキングツリーに滞留していた。cycle-75ではbuilder成果物（4記事draft化等）は残ったが、PMが直接更新していたcycle-75.mdは一度もコミットされていなかった。

### パターンB: 「一部を戻したい」→「全部消える」

| サイクル | 戻したかったもの | 巻き添えで消えたもの |
|---------|-----------------|-------------------|
| 51 | PMの直接編集（数行） | builderの全面書き直し |
| 71 | お問い合わせページの変更 | プライバシーポリシー + 実績システム |
| 75 | 55組問題の記述（1セクション） | サイクル全体の進捗記録 |

全事故で、本来は限定的な変更の取り消しが意図されていたが、git checkoutの影響範囲がそれより広かった。

### パターンC: 実行前の差分確認の欠如

3回とも、PMは `git diff` で「何が消えるか」を確認せずにcheckoutを実行している。cycle-71ではフックが確認を促したが無視された。cycle-75ではフックが発火しなかった。

### パターンD: フックの防御力不足

| サイクル | フックの状態 | 結果 |
|---------|------------|------|
| 51 | フック未実装 | 検知なし |
| 71 | フック発火→バイパスされた | 防御失敗 |
| 75 | フック未発火（パターン未対応） | 検知なし |

フックの対策は段階的に強化されてきたが、(1) バイパス可能、(2) 検知パターンに漏れがある、という2つの問題が残っている。

## 4. 根本原因

### 第一の根本原因: 中間コミットの欠如

作業の進捗を安全に保持する唯一の手段はgit commitである。しかし、PMは作業完了時にまとめてコミットする方式を取っており、中間成果物が長時間未コミットのままワーキングツリーに留まる。これが全事故の前提条件を作っている。

中間コミットが行われていれば:
- cycle-51: builderの書き直しがコミット済みなら、直接編集のcheckoutで書き直しは消えない
- cycle-71: 各builderの完了時にコミットしていれば、選択的リバートで済んだ
- cycle-75: タスクA/B完了後にcycle-75.mdをコミットしていれば、55組問題セクションの修正でファイル全体は消えなかった

### 第二の根本原因: git checkoutを「編集の取り消し」手段として使う習慣

PMは「間違った編集を取り消す」ときにgit checkoutを使う傾向がある。git checkoutはファイルを最終コミット状態に戻すコマンドであり、「特定の編集だけを取り消す」用途には適さない。特定の変更だけを取り消すには、Editツールで該当部分のみを修正する方が安全である。

### 第三の根本原因: フックの検知パターンの不完全性

`git checkout <file>`（`--`なし）が検知されないギャップがある。gitはどちらの書式でも同じ動作をするため、フックは両方を検知する必要がある。

---

## 5. 再発防止策

### 策1: 中間コミットの義務化（最重要）

**変更対象**: CLAUDE.md または cycle-execution スキル

builderの作業完了後、および PMがドキュメントを更新した後は、必ず中間コミットを行う。コミットメッセージは `wip: [作業内容]` 形式でよい。サイクル完了時に必要に応じてsquashする。

具体的なタイミング:
- 各builderの成果物のレビュー完了後
- cycle-XX.mdのセクション更新後（タスク完了マーク、レビュー結果記録等）
- backlog.mdの更新後

これにより、git checkoutが実行されても最悪でも直近のコミット以降の変更のみが失われる。

### 策2: フックの検知パターン修正

**変更対象**: `.claude/hooks/block-destructive-git.sh`

`git checkout <file>`（`--` なし）パターンを追加検知する。gitはブランチ名とファイルパスを区別するために `--` を使うが、パスが明白な場合は `--` なしでも動作する。以下のパターンを追加:

```bash
# git checkout <file> (without --, but targeting an existing file path)
if echo "$COMMAND" | grep -qE "git\s+checkout\s+[^-][^\s]*\.(md|ts|tsx|js|jsx|json|yaml|yml|sh|css)"; then
  IS_DESTRUCTIVE=1
fi
```

### 策3: git checkoutの代わりにEditツールを使う

**変更対象**: MEMORY.md

「間違った編集を取り消したい」場合、git checkoutではなくEditツールで該当箇所のみを修正する。git checkoutはファイル全体を最終コミット状態に戻すため、意図しない変更の消失を招く。

ただし、ファイル全体を最終コミット状態に戻すことが明確に意図されている場合は、策1（中間コミット）が行われていることを確認した上でgit checkoutを使用してよい。

### 策4: バイパス機能の廃止（cycle-71の提案の再掲）

**変更対象**: `.claude/hooks/block-destructive-git.sh`

ALLOW_UNSAFE_GIT_COMMANDS=1 によるバイパスを削除する。cycle-71の事故報告（19cc8a487d1）で提案済みだが未実施。
