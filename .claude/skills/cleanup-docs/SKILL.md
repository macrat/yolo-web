# ドキュメントの整理

以下の手順に従ってドキュメントを整理してください。

1. `./docs/` ディレクトリ内のすべてのドキュメントをcode-researcherに確認させ、内容が古くなっているものや誤解を生む可能性があるものを特定する。
   - 内容が将来にわたって有用だが古くなっているものについては、更新としてリストアップする。
   - 内容に歴史的価値はあるが今はもう有用ではないものについては、 `./docs/archive/` へ移動するものとしてリストアップする。
   - それ以外のものについては、削除するものとしてリストアップする。

2. `./docs/research/` ディレクトリ内のすべてのドキュメントをcode-researcherに確認させ、内容が古くなっているものや誤解を生む可能性があるものを特定する。
   - 内容が古くなっているが過去のある時点の記録として有用なものについては、 `./docs/archive/` へ移動するものとしてリストアップする。
   - 記録としての価値があまりないものについては、削除するものとしてリストアップする。
   - 計画や提案などの調査・分析に該当しないドキュメントでまだ有効なものについては、 `./docs/` 直下に移動するものとしてリストアップする。
   - 調査・分析に該当せず、なおかつもはや有効ではないものについては、削除するものとしてリストアップする。

3. `./docs/cycles/` ディレクトリのファイル一覧を確認し、サイクルドキュメント以外のファイルを特定する。
   - 現時点でまだ有効な計画・提案については、 `./docs/` 直下に移動するものとしてリストアップする。
   - 将来にわたって有効な調査・分析のレポートについては、 `./docs/research/` に移動するものとしてリストアップする。このとき、ファイル名の先頭に調査日時を追加する。
   - 過去の記録としての価値があるものについては、 `./docs/archive/` へ移動するものとしてリストアップする。
   - それ以外のものについては、削除するものとしてリストアップする。

4. `./tmp/` ディレクトリ内のすべてのドキュメントをcode-researcherに確認させ、将来にわたって普遍的な知見が記録されているものを特定する。ただし、調査対象はMarkdownファイルのみとする。
   - 調査・分析のレポートについては、 `./docs/research/` に移動するものとしてリストアップする。このとき、ファイル名の先頭に調査日時を追加する。
   - その他で現時点でも有効なドキュメントについては、 `./docs/` 直下に移動するものとしてリストアップする。

5. 上記のリストをもとに、ドキュメントの更新・削除・移動・アーカイブを実行する。
   - 更新する場合はbuilderに作業をさせる。
   - 移動またはアーカイブした場合は、そのファイルに言及している他のドキュメントを探してリンクを更新する。
   - 削除した場合は、そのファイルに言及している他のドキュメントを探して削除された旨を追記する。ただし、削除対象ドキュメントの内容に深く依存している場合は削除を中断し、アーカイブに切り替える。

6. `./docs/anti-patterns/` ディレクトリ内のドキュメントを確認し、重複しているものや書くべき場所が間違っているものを特定する。
   - 各ファイルはサイクルの各フェーズに対応している。そのフェーズで検出できないものやそのフェーズに無関係なものについては、適切なフェーズのドキュメントに移動する。
   - 重複しているものについては、内容を統合して1つの項目にまとめる。このとき抽象化しても構わないが、なるべく具体例も残すようにする。
   - 特定の稀なケースにしか当てはまらないもの（例：特定コンポーネントの特定機能にしか当てはまらないもの）については、今後も多発する可能性が高いのでなければ削除する。
   - 追加・統合・削除をした場合は、通し番号を更新する。

なお、各ディレクトリの説明は `./.claude/rules/doc-directory.md` と `./.claude/rules/tmp-directory.md` を参照してください。どのディレクトリにどのようなドキュメントを保存するべきか、どのようなドキュメントは保存すべきでないか、などのガイドラインが記載されています。
