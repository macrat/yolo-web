# サイクル完了手順

サイクルの全タスクが完了したら、以下の手順を順番に実行してください。

なお、完了手順中に作業漏れや追加作業が見つかった場合は、`/cycle-planning` を再実行して対応計画を立て、`/cycle-execution` を使って実行する手順を踏んでください。
すべての修正が完了したら、もう一度 `/cycle-completion` を実行して完了手続きを行ってください。

## 1. 実装完了確認

サイクルドキュメントに記載されたチェックリストがすべて完了していることを確認してください。
1つでも未完了の項目がある場合は、サイクルを完了させてはいけません。すべての項目を完了させてから完了手続きをやりなおしてください。

問題なければ、 `completed_at` を !`date +"%Y-%m-%dT%H:%M:%S%z"` で更新してください。

## 2. ブログ記事確認

以下の条件に該当する場合、新しいブログ記事が作成されていることを確認してください。

1. /docs/targets/ で定義されているターゲットユーザーのいずれかが興味を持ちそうな変更を行った場合（新しいコンテンツの追加や既存コンテンツの変更など）
2. このサイクル内で調査または実装した内容が、ターゲットユーザーのいずれかにとって有益な情報である場合（SEO・サイト製作・マーケティングに関する調査や複雑な実装を行った場合など）

まだ作成されていなければ、`/cycle-planning` に戻ってブログ記事の計画を立て、執筆し、レビューしてください。
ユーザーは誰で、どんな情報を求めているか、どんな情報には興味がないかをよく検討してください。
完了したら、この手順を最初からやりなおしてください。

ブログはターゲットユーザーにとって有益な情報になっている必要があります。
もし読者にとって分かりにくかったり、価値が薄い内容になってしまっているなら、全面的な書き直しも辞さない覚悟で記事を修正してください。
修正を行った場合はこの手順を最初からやりなおしてください。

また、「今後の展望」を書いた場合は、同じ内容が backlog.md にも記載されていることを確認してください。
もし記載されていないものがあれば、必ず記載してください。

## 3. キャリーオーバー項目の整理

サイクルドキュメントにキャリーオーバー項目がある場合は、内容を整理して /docs/backlog.md に登録してください。

## 4. ワークフロー・アンチパターンチェック

このステップをはじめる前に、PM自身が次の2つの問いに答えを書き出してください。

**問い1: 今回のサイクルで、どんな失敗が起きやすかったか？**
過去の類似サイクルの経験と、今回の作業の性質から、「もし問題が起きていたとしたらどこで・どんな形で起きていたか」を自分の言葉で具体的に書き出してください。アンチパターン集を見る前に書くことが重要です。

このとき、失敗類型の範囲を workflow.md の失敗類型だけに暗黙的に限定しないでください。今回のサイクルで行った作業の種類に応じて、writing.md（記事執筆を行った場合）、planning.md（計画・調査を行った場合）、implementation.md（実装を行った場合）それぞれのカテゴリーに属する失敗類型も含めた全方位の書き出しを行ってください。

**問い2: 今回の作業の各局面で、`docs/anti-patterns/` の各ファイルの各項目はどう当てはまるか？**
問い1で書き出した失敗類型と照らしつつ、サイクルドキュメントの具体的な場面を参照行（L○○）で特定しながら照合してください。以下の形式を守ってください。

- 該当あり: 「今回は○○の場面（cycle-XXX.md L○○）で△△が起きており、AP-WFXX に該当する」
- 該当なし: 「今回は○○の場面（cycle-XXX.md L○○）で△△を確認した結果、AP-WFXX には該当しない」

参照行の明示は省略できません。PM の記憶や推測で記述した内容は、一次資料の行番号を引用できていない時点で照合未完了と見なしてください。

「全項目に該当なし」の一行で済ませることはできません。また「AP-WFXX: 該当なし」の一行記述も同様に禁止です。「該当なし」と判定した項目についても、なぜ該当しないかの具体場面（L○○）を特定した記述が必要です。

---

この2つの回答を書き出したら、reviewerに独立した確認を依頼してください。reviewerへの依頼には、PMが書き出した予想と照合結果を渡し、以下の4観点で確認するよう伝えてください。

1. **範囲の自己縮小**: workflow.md 系の失敗類型だけに絞られていないか。writing.md / planning.md / implementation.md を含む全方位の照合が行われているか
2. **一次資料との整合**: 各具体場面に参照行（L○○）が明示されているか。行番号のない記述は一次資料未照合の可能性がある
3. **「該当なし」判定の具体性**: 「該当なし」と判定した項目に具体場面の特定がなされているか。「該当なし」の一行記述は形式通過の典型パターンである
4. **問い1 と問い2 の紐付け**: 問い1 で書き出した失敗類型が問い2 の照合で漏れていないか。問い1 にある類型が問い2 に対応項目なしで消えていないか

SCAN 実施自体でも AP-WF09（チェックリストの形式通過・自己縮小）が再発しうることに注意してください。cycle-168 では、reviewer 1回目で致命3件、2回目の書き直し後に新規致命1件+重要2件、3回目にも重要1件が新規発覚しました。reviewer の初回指摘で問題がゼロになることは期待しないでください。複数ラウンドの書き直し→再レビューを前提とし、すべての指摘が解消されるまで繰り返してください。

reviewerが問題を指摘した場合は、`/cycle-planning` や `/cycle-execution` スキルを再実行してやり直してください。
すべての問題が解消されたら、もう一度 `/cycle-completion` を実行して完了手続きを行ってください。

## 5. 成果物の確認

コミット前にbuilderが作成・変更した成果物の内容をPMとして確認する。不完全なものをプッシュしてはならない。

- `git diff` で変更内容を確認し、意図通りの変更であることを検証する

## 6. 最後のフォーマット

コミット前に、最終のフォーマットを行ってください。

```bash
npm run format
```

## 7. コミットとプッシュ

すべての変更をmainブランチにコミットしてプッシュしてください。
これにより、サイクルの成果がデプロイされます。

必ずすべてのファイルをコミットしてワークスペースをクリーンな状態にしてからプッシュしてください。
ただし、作業中の一時ファイルや誤って作成されたでファイルがあれば削除してからコミットしてください。
