# コンテンツ信頼レベル分類ルール

## 概要

yolos.net は AI エージェントによって運営されている実験的なウェブサイトであり、constitution Rule 3 に基づき「サイトが AI によって運営されており、コンテンツに誤りが含まれる可能性がある」ことを訪問者に伝える義務がある。

しかし、サイト内のコンテンツにはその性質上、信頼性が大きく異なるものが混在している。確定的なアルゴリズムに基づく計算ツールと、AI が生成したブログ記事では、誤りが含まれるリスクが本質的に異なる。

本ドキュメントは、全コンテンツを信頼性に応じて3段階に分類し、訪問者に対してコンテンツの性質を正確かつ誠実に伝えるためのルールを定義する。これにより、以下を実現する。

- constitution Rule 3 の実質化: 一律の免責表示ではなく、コンテンツの性質に応じた具体的な情報提供
- constitution Rule 2 との両立: 訪問者を不安にさせすぎない、ポジティブなトーンの維持
- constitution Rule 4 の実践: コンテンツの品質の可視化による信頼性の向上

## 信頼レベルの定義

### 一覧

| レベル  | 名称           | 内部キー    | アイコン案     |
| ------- | -------------- | ----------- | -------------- |
| Level 1 | 正確な処理     | `verified`  | チェックマーク |
| Level 2 | AI作成データ   | `curated`   | 本のアイコン   |
| Level 3 | AI生成テキスト | `generated` | AIアイコン     |

アイコンは警告色（赤・黄）を避け、情報を示す中立的なデザインを使用する。

### Level 1: 正確な処理（verified）

- **定義**: 標準仕様やアルゴリズムに基づく確定的な処理。入力が同じなら常に同じ結果を返す。
- **訪問者向け説明文**: 「このコンテンツは標準的なアルゴリズムに基づいて処理しています。実装上のバグがない限り、正確な結果が得られます。」
- **該当条件**:
  - 公開仕様（RFC、Unicode規格等）に基づく処理
  - 数学的に一意な結果が出る計算
  - 業界標準のライブラリ（Web Crypto API等）を使用した処理
- **リスク**: 実装バグのみ

### Level 2: AI作成データ（curated）

- **定義**: 公式資料や辞書を参照して AI が作成したデータ。事実に基づいているが、AI 生成のため誤りが含まれる可能性がある。
- **訪問者向け説明文**: 「このコンテンツのデータは AI が公式資料や辞書を参照して作成しました。正確さを心がけていますが、誤りが含まれる可能性があります。」
- **該当条件**:
  - 明確な出典（辞書、公式ドキュメント、指導要領等）が存在するデータ
  - AI がそれらの出典を参照して生成したデータ
  - データ自体は事実に基づくが、網羅性・正確性の検証は限定的
- **リスク**: AI 生成時の事実誤認、出典の誤読、網羅性の不足

### Level 3: AI生成テキスト（generated）

- **定義**: AI の知識に基づいて生成された解説・意見・創作テキスト。体系的なファクトチェックは行われていない。
- **訪問者向け説明文**: 「このコンテンツは AI が生成した文章です。参考情報としてお読みください。正確でない情報が含まれる場合があります。」
- **該当条件**:
  - AI の学習データに基づく解説・意見
  - 創作的な要素を含むテキスト（診断結果文、解説文等）
  - 体系的なファクトチェックが行われていないテキスト
- **リスク**: 事実誤認、偏り、情報の古さ、ファクトチェック未実施

### 名称選定の理由

- 「正確な処理」「AI作成データ」「AI生成テキスト」は、技術的な前提知識がなくても直感的に信頼度の違いが伝わる表現として選定した。
- 「高い」「中程度」「低い」のような相対的な表現は避けた。Level 3 のコンテンツ（ブログ記事等）も有用であり、「低い」は価値を不当に下げてしまう。constitution Rule 2 への配慮でもある。
- 代替案として「確実」「参考」「創作」も検討したが、「確実」は過信を招く恐れがあり、「創作」はブログ記事の性質と合わないため不採用とした。

## 混在ケースの方針

### 原則: ページ単位で「代表レベル」を1つ設定する

- 各ページには1つの「代表信頼レベル」を設定する
- 代表レベルは、そのページの「主たるコンテンツ」の信頼レベルに基づいて決定する
- 混在がある場合は、ページ内に補足注記（サブレベル表示）を追加する

### パターンA: ツールページ（ロジック + UIテキスト）

- **代表レベル**: verified（ツールの計算結果が主たる価値）
- **補足不要**: UI テキスト（ラベル、説明文）は AI 生成だが、訪問者がツールに求める価値は計算結果であり、UI テキストの正確性は副次的
- **例外**: 敬語早見表、ビジネスメール作成は代表レベル curated（データ自体が主たる価値）

### パターンB: ゲームページ（ロジック + パズルデータ）

- **代表レベル**: curated（パズルデータの正しさがゲーム体験の質を決める）
- **補足注記テンプレート**: 「ゲームの正解判定は正確です。パズルデータは AI が作成しています。」
- **例外**: イロドリは代表レベル verified（パズルデータも数値データであり、ゲームの判定は RGB 距離計算に基づくため）

### パターンC: クイズページ（ロジック + 問題 + 解説）

- **knowledge型の代表レベル**: curated（問題と正解の正しさが主たる価値）
- **personality型の代表レベル**: generated（結果テキストの創作性が主たる価値）
- **補足注記テンプレート（knowledge型）**: 「スコア計算は正確です。問題と正解は AI が辞書を参照して作成しています。解説文は AI の見解であり、誤りを含む可能性があります。」
- **補足注記テンプレート（personality型）**: 「スコア計算は正確です。質問と結果は AI が創作しました。楽しみとしてお楽しみください。」

### パターンD: 辞典ページ

- **代表レベル**: curated（全データが AI 作成）
- **補足注記**: 個別データの出典情報がある場合は、それを記載（例: 敬語早見表の「文化庁『敬語の指針』」参照注記）

### 将来の複合コンテンツへの対応

- Phase 2 以降で混在表示が複雑になりすぎる場合は、「詳細を見る」リンクで信頼レベルの内訳を展開表示する UI を検討する
- 現 Phase 1 では代表レベル + 補足注記のシンプルな構成で十分

## 新規コンテンツ追加時の判定フロー

将来のコンテンツ追加にも適用できる判定フローを以下に示す。

```
Q1: コンテンツの主たる価値は何か？
  |
  +-- 計算結果・変換結果 --> Q2へ
  +-- データの参照・閲覧 --> Q3へ
  +-- テキストの閲覧 --> generated

Q2: 処理は確定的アルゴリズムに基づくか？
  |
  +-- はい（RFC、数学的定義、標準API等に基づく） --> verified
  +-- いいえ（AIの判断を含む処理） --> Q3へ
  |   ※ 現状、Q1で「計算結果・変換結果」に分岐した上で
  |     Q2が「いいえ」となるコンテンツは存在しない。
  |     将来のコンテンツ追加に備えた分岐として残す。

Q3: データに明確な出典（辞書、公式ドキュメント等）があるか？
  |
  +-- はい --> curated
  +-- いいえ（AIの知識のみに基づく） --> generated
```

### 補足ルール

- 1つのコンテンツ内に複数のレベルが混在する場合は、「混在ケースの方針」に従って代表レベルを決定する
- 代表レベルの決定基準は「訪問者がそのページに求める主たる価値を提供する要素のレベル」とする
- 判断に迷う場合は、より慎重なレベル（curated よりも generated）を選択し、補足注記で正確な状況を伝える

## Phase 2 への接続情報

Phase 1（本ドキュメント）でルールを策定した後、Phase 2 では UI に信頼レベルを表示する実装を行う。以下に、データモデルの変更方針を記載する。

### 推奨方針: 各 Meta 型への `trustLevel` 属性追加

既存の各コンテンツの型定義に `trustLevel` フィールドを追加する。TypeScript の型チェックにより、コンテンツ追加時に `trustLevel` の設定漏れを自動検出できるため、保守性が高い。

#### 共通型の定義

`src/lib/trust-levels.ts` に共通の型を定義する。

```typescript
/** コンテンツの信頼レベル */
export type TrustLevel = "verified" | "curated" | "generated";
```

#### 対象インターフェースへの追加

以下の各 Meta 型に `trustLevel: TrustLevel` と、必要に応じて `trustNote?: string`（補足注記テキスト）を追加する。

| ファイルパス               | インターフェース名                 |
| -------------------------- | ---------------------------------- |
| `src/tools/types.ts`       | `ToolMeta`                         |
| `src/games/types.ts`       | `GameMeta`                         |
| `src/quiz/types.ts`        | `QuizMeta`                         |
| `src/cheatsheets/types.ts` | `CheatsheetMeta`                   |
| `src/blog/_lib/blog.ts`    | `BlogFrontmatter` / `BlogPostMeta` |

#### 代替方針: 集中管理マップ

`src/lib/trust-levels.ts` に全コンテンツのマッピングを slug ベースで集中管理する方式も検討した。既存の型定義を変更しないメリットがあるが、コンテンツ追加時のマッピング更新漏れが起きやすく、型安全性が弱いため、推奨方針 A を優先する。

### 静的ページの実装方針

トップページ、About ページ等の静的ページは、既存のレジストリパターンに乗っていないため、Meta 型への属性追加とは別の対応が必要となる。

方針案:

- 静的ページ用の信頼レベル設定を `src/lib/trust-levels.ts` に定数マップとして定義する（例: `{ "/": "generated", "/about": "generated" }` 形式）
- レイアウトコンポーネント側でパスに応じた信頼レベルを参照して表示する
- 静的ページは全て generated であるため、フッター等の共通部分での一括表示も選択肢になる

### メモアーカイブの実装方針

メモアーカイブは AI エージェント間のやりとりの記録であり、全て generated に分類される。

方針案:

- メモアーカイブのページレイアウトに信頼レベル表示を直接埋め込む（個別の Meta 定義は不要）
- メモアーカイブのトップページに「このセクションは AI エージェント間のやりとりの記録です。意思決定の透明性のための公開であり、内容の正確性は保証されません。」の注記を表示する

### UI 表示の方針概要

- 各ページのヘッダー付近に信頼レベルバッジ（アイコン + レベル名）を表示する
- バッジクリックまたはホバーで訪問者向け説明文を表示する
- 混在ケースでは代表レベルバッジ + 補足注記テキストを表示する
- フッターの全体免責表示は維持する

## constitution Rule 3 との整合性

### 現状の課題

- フッターの一律的な免責表示だけでは、コンテンツの種別による信頼性の違いを伝えられていない
- 確定的な計算ツールと AI 生成ブログ記事に同じ免責文が適用されている

### 本施策による改善

- Rule 3 の「content may be broken or incorrect」をコンテンツ種別に応じて具体化する
- verified なコンテンツには「アルゴリズムに基づく正確な処理」であることを明示し、過剰な免責表示から脱却する
- curated / generated なコンテンツには具体的なリスクを伝え、免責表示を実質化する
- フッターの全体免責は維持しつつ、ページ単位の信頼レベル表示で補完する

### 訪問者を不安にさせない配慮

- 各レベルの説明文は「～の可能性があります」という客観的な表現に留め、過度に不安を煽る表現は使わない
- Level 1 でも「実装上のバグがない限り」と正直に限界を伝えつつ、基本的にはポジティブなトーンで記述する
- Level 3 でも「参考情報としてお読みください」とし、「信頼しないでください」のようなネガティブすぎる表現は避ける
