# サイト統一計画（B-545: 根幹の統一）

> **位置づけ**: 本計画は `docs/archive/design-migration-plan.md`（旧デザイン移行計画・cycle-277 でアーカイブ）の後継である。旧計画は「道具箱-as-core」コンセプトを軸（目的#2・Phase 10）として書かれ、cycle-257 の実測でコンセプトが覆った後も注記の重ね塗り（AP-P28）で延命されていた。cycle-277 で決定(a)（下記）を外部検証つきで確定したのを機に、旧計画を破棄・アーカイブし、(1) 決定(a)に基づく根幹統一の構造計画と、(2) 旧計画から引き継ぐ概念非依存の残作業（デザイン体系統一・legacy 撤去）を、一本の整合した計画として立て直した。完了済み Phase 1〜10 の経緯・完了注記は archive 版を歴史記録として参照。

## 土台：決定(a)

**サイトの根幹は「自分を知り、楽しむ。そのための場所」（診断中心）に統一する。**

- 一次記録: `docs/research/2026-07-03-market-research-b545.md`（T1 実測・T2 外部調査4本・T3 統合判断）・`docs/cycles/cycle-277.md`。
- 論拠の骨子: (1) 市場需要——三系統（診断・ツール・辞典）のうち需要が大きく伸びており個人/小規模に参入余地が残るのは診断のみ。(2) 当方実績——診断（play）がサイト全検索クリックの 84% を取る実証済みの主経路（留保付きの1入力として使用）。(3) 差別化レーン——「文化×診断」は診断形式としては飽和だが文化的深みの統合の次元で空いている（確信度: 中・堀は浅い）。(4) SNS シェアは未実現の市場潜在＝統一後の最優先投資先。
- 決定が課す設計制約（要旨・詳細は research doc T3）: **(i)** 薄い量産の禁止（質＞量）。**(ii)** 来訪者を傷つけない診断の設計（見立て遊びの距離・欠損枠づけ禁止・不安導線禁止・娯楽枠の常時明示・科学性僭称禁止）。**(iii)** バイラル単独依存の禁止（常緑成分の併持）。**(iv)** 支え層の保持（辞典=独自文脈・ツール=実用層。核にはしない）。
- 事前トリップワイヤ: TW-1 検索クリック半減監視・TW-2 差別化レーン閉塞監視・TW-3 害の兆候（観測面獲得後の補助）。

## 根幹統一の構造計画

### U1: 正面玄関と自己定義（cycle-277 T6 で出荷）

- **トップ `/`**: 道具箱ダッシュボード → 診断中心の着地面へ。コンテンツとデザイン（U2 の基調視覚言語）を**同時に**出荷する（cycle-276 の「診断コンテンツ＋道具箱デザイン」断裂＝AP-P27 を再演しない）。構成の要点: コンセプト提示＋AI 運営の明示（rule 3）＋主役=診断への発見導線（character-personality 先頭）＋支え層・道具層への二次導線。煽りなし・絵文字なし・a11y（44px/focus-visible/AA コントラスト）。
- **自己定義**: `seo.ts` WebSite JSON-LD・`site-metadata.ts` keywords・`/about` を「無料オンラインツールを集めたサイト」から「自分を知り、楽しむ場所」へ。免責（診断=娯楽・科学的根拠なし）は保持・強化（制約(ii)）。
- **道具箱の降格**: `/toolbox` ルート新設（ToolboxContent 移設・機能無退行）・`/toolbox`→`/` redirect 撤去・sitemap 追加・ナビ/フッターに発見導線を保持（機能した道具箱を埋没させない——cycle-276 の教訓）。
- **ナビ/IA**: 「遊び」先頭（現状は cycle-167 の名残で「ツール」先頭）。ラベルの妥当性（「遊び」が診断の受け皿として最適か）・辞典のナビ掲載は U1 出荷時に判断。

### U2: デザインシステムの基調再定義（cycle-277 T5 で出荷）

`DESIGN.md` を再構成する（詳細は DESIGN.md 自身が正典）:

- **austere の再定義**: 「サイトの基調 identity」から「品質バー」（けばけばしさ・煽り・ダークパターンを避ける規律、可読性・速度の担保）へ。抑制は捨てない——「上品な高揚」の土台として保つ。
- **中心体験の視覚言語の基調化**: 旧 §7「診断の視覚言語（タッチポイント限定の例外）」を、サイトの基調の一部（中心体験=診断の標準表現）へ引き上げ、§1〜6 と統合。道具・辞典本文・共通 UI は引き続き純粋な抑制を保つ（全面への色の氾濫はしない）。
- **勲章（cycle-275）・結果固有色の去就**: 「診断中心の基調として来訪者価値（自己発見のごほうび・見せたくなる完結感）に資するか」で判定（現状追認 AP-P30 禁止）。制約(ii)の害防止原則を視覚言語のレッドラインに組み込む。
- **一体更新**: `.claude/skills/frontend-design/SKILL.md`・design-system README をドリフトなく同時更新。

### U3: 結果体験とシェア（統一後の最優先投資・B-550）

決定(a)の論拠4（SNS シェア=未実現の市場潜在）への投資。結果面を「人に見せたくなる完結した一枚」として磨き、シェア導線・OGP を整備する。B-544（結果ページのフルクライマックス化）・B-553（入口ファーストビュー）と協調。**本計画では方向のみ確定し、実施は統一出荷後のサイクルで**（A/B `quiz_result_visual_v1` の結論 B-526/B-527 とも協調）。

### U4: 支え層の深化（文化×診断の統合体験）

差別化レーン実査（confidence: 中）に基づく投資方向: 診断結果 ↔ 辞典（伝統色の由来・言葉の物語）の統合体験の完成度を上げる。余地が大きいのは「ことわざ」「伝統色」の側。**本格投資の前に日本実機 SERP での再確認を必須とする**（research doc T3 の条件）。辞典の同形式網羅の拡大（新カテゴリの大量追加）はしない。

## 残デザイン移行（旧計画から引き継ぐ概念非依存の作業）

デザイン体系の統一とコード健全化は、コンセプトと無関係に価値がある。旧計画の Phase 8.2 残・Phase 11 を以下に引き継ぐ（原文の完了注記・経緯は archive 版参照）。

### M1: 残ゲーム3ルートの (new) 移行（B-547）

`(legacy)/play/` 残存 3 ルート: `daily`・`irodori`・`yoji-kimeru`。各 1 サイクルで「1 ページ移行の標準手順」（下記）に従い移行。daily は成立性（2PV/9秒）の再評価を含む。

### M2: legacy 撤去（Phase 11 相当・B-337）

M1 完了後、厳密に直列で:

1. **API ルート移行**: `(legacy)/api/` 配下（kanji-kanaru/evaluate・kanji-kanaru/hints・quiz/compatibility・yoji-kimeru/evaluate・yoji-kimeru/puzzle）を `(new)/api/` へ `git mv`。
2. **legacy 撤去**: `src/app/(legacy)/`・`src/app/old-globals.css`・`src/components/common/` を削除。CLAUDE.md Notes の (legacy)/(new) メモも削除。旧 `src/components/search/` 一式・`src/lib/analytics.ts` の search 関連関数・`scripts/build-search-index.ts`・`src/lib/search/{build-index,types}.ts` を撤去し、`package.json` の `generate:static-assets` 呼び出しを外す。
3. **Route Group 解除**: `(new)/` 配下を `src/app/` 直下へ `git mv`・`(new)/` 削除・個別 `metadata` の動作確認。
4. **layout 統合**: GoogleAnalytics・WebSite JSON-LD・sharedMetadata を `src/app/layout.tsx` 単一へ集約。
5. **globals.css 後始末**: legacy 関連コメント・旧トークン互換の撤去・`.scroll-locked` は globals.css のみに残す・DESIGN.md「暫定対応」セクション削除。
6. **本計画の該当節を完了化**: `grep -rE "legacy|old-globals|components/common|\(new\)|\(legacy\)" src/` が空であることが完了基準。

**前提注意**: legacy 撤去は新デザイン方向を不可逆確定させる。U1/U2 の出荷と効果観測（トリップワイヤ）を経てから着手する。

### 1 ページ移行の標準手順（M1 で使用・旧計画から引き継ぎ）

1. 依存コンポーネント確認（`@/components/common/*` の使用を grep）
2. `git mv (legacy)/foo/ (new)/foo/`
3. import パス修正（`@/components/common/*` → `@/components/*`）
4. CSS Module トークン置換（`--color-*` → 新トークン。`:root.dark` → `:global(:root.dark)`・`docs/knowledge/css-modules.md`）
5. DESIGN.md・`/frontend-design` Skill に従いデザイン再設計（トークン置換だけの上塗り禁止=AP-P28。ページ最上位コンテナに `max-width: 1200px; margin: 0 auto` 必須。診断面は DESIGN.md の中心体験の視覚言語に従う）
6. TrustLevelBadge の撤去（B-432 の一括削除方針と整合）
7. テスト調整（`npm test` 当該範囲）
8. 視覚確認（Playwright・w360/w1280×light/dark・w1900 で main 直下 width < 1300px・タップ 44px・AA コントラスト・aria-current）
9. 1 ページ 1 コミット

### 検証方法（共通）

- Playwright 実機: w360/w1280 × light/dark。旧 common 由来のハッシュ class が DOM に出ない。WCAG AA・44px・focus-visible・`aria-current`・構造化データ。
- `npm run lint && npm run format:check && npm run test && npm run build` 全通過。
- production ビルドで GA4 計測動作。

## 完了の定義（B-545 と本計画の関係）

- **B-545（根幹統一）の完了 = cycle-277 で U1 + U2 を出荷**し、サイトの自己定義・正面玄関・デザインシステム・コンセプト文書が単一のコンセプト（診断中心）で整合すること。
- U3/U4（投資）と M1/M2（残移行）は統一後の継続作業であり、B-545 の完了要件ではない（それぞれ B-550/B-544/B-553・B-547・B-337 として backlog 管理）。
