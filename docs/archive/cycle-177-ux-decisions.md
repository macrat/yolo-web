## グループ 1: 初回訪問〜再訪

### 瞬間 1. 初回ロード時の visual feedback

- **想像する場面**: M1a が検索結果から「文字数カウント」と入力してこのサイトに到達。`/toolbox-preview` を開いた瞬間。
- **採用 UI/UX**: SSR で静的 skeleton（タイル枠 + 解決済み displayName、継続シマーなし）+ meta タイトルを即時出力する。CSR hydrate 後にタイル本体（FallbackTile）が **opacity 0 → 1 を 200ms / ease-out の fade-in** で現れる（Layout Shift は SSR でタイル枠サイズを確定させて最小化）。fade-in は「ロード完了の合図」として機能し、瞬間 2 と統合する（初回・再訪ともに同じ fade-in 演出を踏襲）。

## グループ 2: 通常モード（使用モード）

### 瞬間 4. ホバー時のフィードバック（desktop）

- **想像する場面**: M1a がデスクトップ Chrome でツールタイル群を眺め、目的のツールにマウスを乗せる。「これが正しい候補か」を 0.1 秒で確認したい。
- **採用 UI/UX**: `transform: scale(1.01)` + 一時的 box-shadow（淡い） + 背景色変化を 150ms / ease-out の transition で適用する。`@media (hover: hover)` で hover state を分岐し、タッチデバイスでは hover state を持たない（瞬間 44 と整合）。
- **M1a likes**「ページを開いた瞬間に入力欄が見えて、すぐ使い始められること」「結果の根拠や前提が必要最小限だけ添えられており、信頼して使えること」: 0.1 秒で「触れる候補がここにある」と分かるため、目的のタイルを見つけて使い始めるまでの時間が短くなる。

- **M1b dislikes**「動作が重くなること」: scale(1.01) は GPU 合成のみで完結するため、30+ タイル並んでも負荷を生まない。

### 瞬間 5. タップ / クリック時のフィードバック

- **想像する場面**: M1a がモバイルでタイルを指でタップ。指を離した瞬間に「押せた」感がないと、もう一度押すかどうか迷う。

- **M1a likes**「ページを開いた瞬間に入力欄が見えて、すぐ使い始められること」: 「押せた」が即時伝わるため、再タップで迷う時間が無くなる。
- **M1b likes**「同じ手順で済むこと」: 押した手応えが毎回同じで、慣れた手順をなぞれる。
- **M1b dislikes**「動作が重くなること」: 100ms 戻りのみで継続モーションが無い。
- **reduced-motion 時の挙動**: 押下時の scale(0.97) は瞬間変化なので残す（連続モーションではない）。離した時の transition も 0.001ms に縮退して即時戻りに。色変化は維持。
- ~~**§4 追記候補**~~: 削除（v11 訂正）。本瞬間の実装詳細はコードと CSS Module 内に残し、§4 改定は本サイクルで実施しない。

### 瞬間 6. タイル本体クリックで道具詳細ページへ遷移する時の演出

- ~~**§4 追記候補**~~: 削除（v11 訂正）。本瞬間の実装詳細はコードと CSS Module 内に残し、§4 改定は本サイクルで実施しない。

### 瞬間 37. 長押し挙動

## dislikes / likes の正しい意味範囲表（v10 で導入）

本シートで targets/\*.yaml の dislikes / likes を引用する際は、本表に照らして適用可否を必ずチェックする。dislikes は **ツール仕様・データ・URL・結果の破壊的変化** を指すもので、UI/UX の視覚演出やインタラクションフィードバックには適用範囲外。

| 引用文言（targets/\*.yaml フィールドそのまま）                                      | 正しい適用範囲（採用 / 不採用根拠として使える場面）                                                              | 適用範囲外（誤読されやすい場面）                                                                        |
| ----------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------- |
| **M1b dislikes**「慣れた操作手順が、別デザインへの刷新や機能追加で突然変わること」  | ツール仕様（計算ロジック・出力形式・入力要件）の破壊的変更、操作シーケンス（タップ数や順序、入力欄の場所）の変化 | 視覚フィードバック / アニメーションの追加、長押し挙動の追加、編集モード時の見た目変化、モーダル出現演出 |
| **M1b dislikes**「便利そうだから機能を詰め込んだ結果、動作が重くなること」          | パフォーマンス劣化（CPU / GPU / メモリ / FPS / 体感重さ）                                                        | 機能数の制限、UI を豊かにしない根拠                                                                     |
| **M1b dislikes**「以前と同じ入力なのに結果や挙動が前回と変わっていること」          | 計算結果や処理結果（出力値、リンクの動作、表示が壊れて目的が果たせない、保存される状態の意図しない変化）         | 視覚表現の改善、演出の追加                                                                              |
