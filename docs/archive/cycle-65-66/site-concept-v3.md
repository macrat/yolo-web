## 1. サイト名・タグライン

**サイト名**: yolos.net（変更なし）

**タグライン（仮）**: 「毎日あたまの体操 -- パズル・クイズ・診断で遊ぼう」

**補足**: サイト名の由来はAboutページで説明する（ownerフィードバック指摘）。タグラインはデイリーパズルの習慣性と、パズル以外のコンテンツ（クイズ・診断）も包含する表現にしている。最終決定はowner判断。

## 2. ターゲットユーザー

v2（site-concept.md）の5ターゲットから、コンセプトに合致する2ターゲットに絞り込む。

### メインターゲット: 通勤・休憩中に頭を使う遊びをしたい社会人（20〜50代）

- 毎日の隙間時間（5〜10分）にスマホで遊べるコンテンツを求めている
- Wordleのようなデイリーパズルを習慣として楽しむ層
- SNSで結果を共有し、友人と競い合うことに楽しさを感じる
- 登録不要・インストール不要のブラウザ完結型を好む

### サブターゲット: AI技術・エージェント運用に関心のあるエンジニア

- AIエージェントによるサイト運営の実験記録に知的好奇心を持つ層
- ブログ（AI技術記録）の読者。メインターゲットとは独立した流入経路を持つ
- コンセプトとの直接的整合性は不要（別枠）

### v2から削除するターゲット

- 「日本語・日本文化特有の実用ニーズを持つ人」: 実用ツール群（敬語・暦・ふりがな変換等）はコンセプトから外れるため削除。
- 「日本文化に興味のある外国人・海外在住日本人」: 英語圏展開は現時点でスコープ外。多言語対応コストが高く、既存日本語圏ユーザーとの両立が困難（評価レポート: 実現可能性2/5）。
- 「日本語や日本文化を楽しく学びたい人」: 「学び」ではなく「遊び」がコンセプトの軸。学習要素はゲームの副産物として自然に含まれるが、学習目的の来訪者をメインターゲットとはしない。

## 3. 提供価値

### 核となる価値: 「今日もやろう」と思える毎日の習慣

1. **デイリー更新によるリテンション**: 毎日新しい問題が出題されるパズル群で、「今日の分をやらなきゃ」という習慣を形成する。連続日数の記録がモチベーションを維持する。

2. **多ジャンルのパズル集約**: 数学・漢字・色彩・言葉分類など、異なる認知能力を使うパズルを一箇所に集約。1つのサイトで複数のパズルを連続して楽しめる。NYT Gamesの日本語版的ポジション。

3. **SNS共有による社会的楽しさ**: パズル結果やクイズ・診断結果をSNSで共有し、友人と比較する文化を作る。共有が新規ユーザーの流入経路にもなる。

4. **気軽さ**: 登録不要、インストール不要、1回5分。スマホブラウザで完結する。

## 5. コンテンツ間の関係性・回遊設計

### サイト構造

```
トップページ
  |-- 今日のパズル（メイン動線）
  |     |-- irodori（色彩）
  |     |-- kanji-kanaru（漢字）
  |     |-- nakamawake（仲間分け）
  |     |-- yoji-kimeru（四字熟語）
  |     |-- [新規] 数学パズル
  |     |-- [新規] 色彩チャレンジ
  |     +-- [新規] キャラゲーム
  |
  |-- クイズ・診断（サブ動線）
  |     |-- 漢字力診断
  |     |-- 四字熟語力診断
  |     |-- ことわざ力診断
  |     |-- 伝統色性格診断
  |     +-- 四字熟語性格診断
  |
  |-- ブログ（独立動線）
  |     +-- AI技術記録
  |
  +-- About / AI透明性情報
```

### 回遊設計の基本方針

1. **トップページ = 今日のパズル一覧**: 訪問者が最初に目にするのは「今日の○種パズル」。各パズルの完了状態が一目でわかるダッシュボード形式。

2. **パズル間の横移動**: 1つのパズルを終えると「次のパズルへ」ボタンで別ジャンルへ誘導。数学→漢字→色彩のように認知負荷の種類を変えることで飽き防止。

3. **パズル→クイズ・診断の縦移動**: 漢字パズル完了後に「あなたの漢字力は?」と漢字力診断へ誘導。パズルで興味を持った分野の深掘りを促す。

4. **診断→SNS→新規流入**: 診断結果のSNSシェアボタンを目立つ位置に配置。シェアされた結果を見た人がサイトに来訪し、パズルも発見する流れを作る。

5. **連続記録システム**: 全パズル横断の連続日数トラッキング（ローカルストレージ使用）。「今日○種クリア / 連続○日」の表示がリテンション装置として機能する。

6. **ブログは独立**: パズルとブログの間に強い回遊導線は設けない。来訪者の目的が異なるため、無理な誘導は双方のUXを損なう。フッターやAboutページからの自然なリンクのみ。

## 6. 独自性のポイント

**なぜユーザーがこのサイトを選ぶか:**

### 1. 日本語で複数ジャンルのデイリーパズルを一箇所に集約したサイトは存在しない

- 英語圏にはNYT Games（Wordle, Connections, Strands等を集約）があるが、日本語圏には同等のサービスがない
- 個別のWordle派生（ことのはたんご等）は存在するが、いずれも単一ジャンル
- 「日本語版NYT Games」というポジションが空席

### 2. 既存ゲームに独自のひねりを加える路線

- irodori: HSLスライダーによる伝統色再現 = 色見本サイトにはない「当てる」インタラクション
- kanji-kanaru: 5属性ヒントシステム = 単純な正解/不正解ではなく推理の過程が楽しい
- nakamawake: 日本語の語彙・文化知識を活かしたConnections = 英語版とは異なる面白さ
- 新規パズルも同様に「既存ルール + 独自ひねり」で差別化

### 3. デイリー形式 + 連続記録 = 最強のリテンション装置

- 「今日の問題をやらなきゃ」という義務感と楽しさの両立
- 連続記録が途切れたくないという心理的コスト
- 毎日来る理由が明確であり、SEOやSNSに頼らないオーガニック再訪

### 4. AI運営の透明性（ブログによる差別化）

- サイト自体がAIエージェントによる実験であることを包み隠さず公開
- 制作過程のブログ記録がE-E-A-T（Experience）を形成
- 「AIが作ったサイト」は増えているが、制作過程を詳細に公開しているサイトは極めて少ない

## 8. AI技術ブログの位置づけ

### 基本方針

AI技術ブログはパズルコンセプトとは独立した存在として維持する。コンセプトとの整合性は求めない。

### 存在意義

1. **constitution.md Rule 3の具現化**: AIが実験として運営していることの透明性を、最も誠実な形で実現する手段
2. **E-E-A-T（Experience）の源泉**: AIエージェントオーケストレーションの実践記録は、他サイトが持ち得ない一次情報
3. **独立した流入経路**: エンジニア向けの検索・SNS流入はパズルとは別経路で発生し、サイト全体のトラフィックに貢献する

### ブログとパズルの関係

- 強い回遊導線は設けない（ターゲットが異なるため）
- フッター・Aboutページで自然にリンクする程度
- ブログ内で「このサイトのパズルはこうやって作られた」という記事は、両方の読者に価値がある自然な接点

### ブログ記事の個別判断

本ドキュメントのスコープ外。以下の方針で別途実施する:

- 独自の試行錯誤を含む技術記事 → 残す（E-E-A-T Experience）
- AI実験プロセスの記録 → 残す/書き直す
- 汎用技術Tipsで競合にない記事 → 残す
- 価値の低い記事 → 個別に判断

## 9. 削除コンテンツの処理方針

### 使用するHTTPステータス: 410 Gone

ownerフィードバックに従い、削除コンテンツには**410エラー**を返す。301リダイレクトは使用しない。

### 理由

- 301リダイレクト: 「このページは別のページに移動した」という意味。全く別のコンテンツへの301は誤用であり、Googleからの評価を下げるリスクがある。
- 410 Gone: 「このページは意図的に削除された」という意味。Googleに対して明確にインデックスからの除外を伝える。

### 処理手順（実装時）

1. 対象URLに410レスポンスを返すよう設定
2. 410ページには「このコンテンツは終了しました。トップページへ」のリンクを設置（ユーザーへの配慮）
3. Google Search Consoleで削除リクエストを送信（任意: 410で自然に除外されるが、速度を上げたい場合）
4. sitemap.xmlから対象URLを除外

### 対象（owner最終判断後に実施）

- 辞典3種: colors, kanji, yoji
- チートシート7種: cron, git, html-tags, http-status-codes, markdown, regex, sql

## 10. 技術的な実装方針

### coding-rules.mdの制約まとめ

coding-rules.md（`.claude/rules/coding-rules.md`）に従い、以下の制約の中で実装する:

1. **静的最優先、クライアント優先、サーバーは必要に応じて**: デイリーパズルの問題生成はアルゴリズムベースでクライアントサイドまたはビルド時に実行
2. **外部API呼び出し禁止（ランタイム）**: パズルの問題データはビルド時に生成するか、クライアントサイドのアルゴリズムで動的生成
3. **データベース禁止**: ユーザーの連続記録・進捗はローカルストレージに保存
4. **認証禁止**: ユーザー登録なし。全機能を匿名で利用可能

### パズル別の技術アプローチ

| パズル                  | 問題生成方式                         | データ要件                         |
| ----------------------- | ------------------------------------ | ---------------------------------- |
| irodori（色彩）         | ビルド時に伝統色データから出題       | 既存の伝統色データ（約250色）      |
| kanji-kanaru（漢字）    | ビルド時に漢字データから出題         | 既存の漢字データ                   |
| nakamawake（仲間分け）  | ビルド時に問題セットを準備           | 問題データの手動設計が必要（課題） |
| yoji-kimeru（四字熟語） | ビルド時に四字熟語データから出題     | 既存の四字熟語データ               |
| 数学パズル（新規）      | クライアントサイドでアルゴリズム生成 | データ不要（最大の利点）           |
| 色彩チャレンジ（新規）  | 既存色彩データ + アルゴリズム        | 既存の伝統色データ活用             |
| キャラゲーム（新規）    | キャラクターデータ + ゲームロジック  | キャラクター設計が必要             |

### デイリー更新の仕組み

- 日付をシード値としたアルゴリズムで「今日の問題」を決定的に生成する
- サーバーサイドの日次バッチ処理は不要（クライアントサイドで日付に基づいて問題を選択/生成）
- ビルド時に十分なデータを準備し、日付によるインデックス選択で出題

### 連続記録システム

- ローカルストレージにJSON形式で保存: `{ lastPlayDate, streak, totalClears, dailyProgress: { puzzle1: true, puzzle2: false, ... } }`
- デバイス間同期は非対応（DB禁止のため）。これはトレードオフとして受容する
- クリア情報は端末に紐づく。Aboutページでこの制限を明示する

### SNSシェア機能

- パズル結果をテキスト+絵文字で表現（Wordle方式: 色付きマスの並び等）
- Web Share APIまたはクリップボードコピーでシェア
- OGP画像の動的生成はサーバーサイドで対応可能（Route Handlers）
