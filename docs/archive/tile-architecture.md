# タイル・道具箱・詳細ページのアーキテクチャ

> **【コンセプト更新の注記（cycle-259 / B-538）】** 「道具箱（ダッシュボード）をサイトの核に据える」というコンセプトは cycle-257（B-535）で実測の結果**降ろされ**、`docs/site-concept.md` は診断中心へ再センタリングされた（道具箱は削除せず実用層として残すが、サイトの核ではない）。本ファイルの「不変項」は、**タイルを実装する場合に守るべき技術的不変項**（ルートは `<Panel>`・タイルは唯一の実装・詳細ページと道具箱は同一エクスポートを描く）であって、「サイトの全コンテンツをタイル化せよ」という概念的命令ではない。全面タイル化の優先度は下がった（ゲーム全面タイル化 B-493 は P1→P3 降格・cycle-259）。既存タイル（34ツール）と道具箱の技術パターンは引き続き本ファイルが正典。

タイル・道具箱・詳細ページを実コードで成立させるための確立パターン。cycle-226 の縦スライス（url-encode）で実機実証した。**このパターン（＝タイルを作るときの技術契約）から外れると、cycle-175〜225 で約50サイクル繰り返した設計破壊が再発する。**

## 1. 3要素の機能と関係（不変項）

- **タイル**＝唯一の実装。ツールの機能を内包し、**ページ遷移なしにその場で動く**（入力・閲覧・実行がタイル内で完結する）自己完結コンポーネント。ルートは必ず `<Panel>`（`@/components/Panel`）＝DESIGN.md §1「すべてのコンテンツはパネルに収まる」を構造で満たす。
- **ツール詳細ページ**＝そのツールのフルタイルをヒーロー描画する文脈付き SEO 入口（1ツール1 URL・パンくず・howItWorks・FAQ・関連）。本体＝タイルそのもの。
- **道具箱**＝来訪者が好きなタイルを並べて使う個人ダッシュボード。複数タイルが同居し、各タイルは**道具箱を離れずにその場で動く**。
- **守るべき不変項**: タイルは唯一の実装。**詳細ページと道具箱は同じタイル（同一エクスポート）を描画する2つの文脈にすぎない**。プライマリはタイルであって、ページでもカードでもない。同じコンポーネントを両方が描くから、両者の中身は定義上一致する（分裂しようがない）。

```
        ┌─ ツール詳細ページ（タイル＋文脈・1枚・SEO入口）
タイル ──┤  ※同一エクスポートを両方が import
（唯一の実装）└─ 道具箱（タイル×複数・その場で動く・並べ替えは将来）
```

## 2. 1ツール ＝ n タイル（提示バリエーション・分裂禁止）

1つのツールは**唯一の共有エンジン（ロジック）**を持ち、そこから複数のタイルを**別々の見せ方・設定**として出す（例: url-encode の full／encode-only／decode-only、カレンダーの月表示／日めくり）。

- **禁止**: バリエーションごとに別のダンプダウンされたコンポーネント／ロジックを書くこと（＝cycle-220 の kind=widget が起こした「道具箱版と詳細ページ版の分裂＝機能劣後」）。
- **正**: 同一コンポーネントの `variant` prop ＋ 同一の共有ロジック（例: `src/tools/url-encode/logic.ts` の `encodeUrl`/`decodeUrl`）の**設定差**としてバリエーションを出す。url-encode タイルは `variant="full|encode|decode"` で方向を固定／トグル表示を切り替えるだけで、実装は1つ。

## 3. 実装パターン（cycle-226 で実証）

- ツールの単一実装は `src/tools/<slug>/<Name>Tile.tsx`（例 `UrlEncodeTile`）。ルートが `<Panel>`。`"use client"` で自己完結（ToolPageLayout に**機能依存しない**＝道具箱に置いても単独で動く）。**そのツールの UI を描くコンポーネントはこれ1つだけ**にする（別の「ページ本体」実装を残さない＝分裂の構造的排除。grep で別実装ゼロを確認できる状態を保つ）。
- 詳細ページ（`src/app/(new)/tools/<slug>/page.tsx`）は `ToolPageLayout` の中で `<XxxTile variant="full" />` をヒーローに描く。
- 道具箱（実装は `src/app/(new)/toolbox/` のモジュール群。cycle-232 = Phase 10.3 の本公開でトップ `/` のページ本体になり、旧 `/toolbox` は `/` へ permanent redirect）は**同一の `XxxTile` を import** して `variant` 違いの**生きたインスタンス**として並べる。**リンク・カード・詳細ページへの誘導を作らない**（リンク集は cycle-175 の失敗）。
- **複数インスタンス同居**: 道具箱は同一タイルを複数並べるため、タイル内の DOM id は `useId` ベースのインスタンス一意 id にする（ハードコード id は重複・label 誤結合を起こす）。
- **道具箱のタイル寸法**: `src/tools/_constants/tile-grid.ts`（1セル128px・n×m・`calcTilePixels`）を**推奨幅の上限**として使い、`maxWidth`＋親追従（`width:100%`）でレスポンシブにする。固定 `width` にするとモバイル（w360）で横はみ出しする。**機能を枠に合わせて削らない**（cycle-220 の轍）。タイルが規格に収まらない場合は規格側の見直しを起票する。

## 4. 完了の合格条件（書類でなく実物）

「Panel があるか」のような書類審査でなく、**フォアグラウンド Playwright（本番ビルド）で実物を確認する**：

- 道具箱でタイル内のツールが**ページ遷移なしに動く**（`page.url()` 不変・full reload／`framenavigated` 無発生を機械的に確認）。
- 詳細ページのヒーローが**同じタイル**で同じ動作をする。
- n タイル（variant）が各々正しく動く。
- タイルのルートが Panel（computed background がテーマ解決済み `--bg`・border-radius `--r-normal`・box-shadow none。ライト/ダーク両方）。
- 複数インスタンスで DOM id 重複ゼロ・hydration エラーゼロ・ライト/ダーク・w360/w1280 で破綻なし。

## 5. やってはならないこと（約50サイクルの失敗モード）

1. **ナビゲーションを作る**（カード/リンクで詳細ページへ飛ばす）。タイルはその場で動く機能であって目録ではない。
2. **タイルを実装から切り離す**（汎用ラッパーで本体を包むだけ。タイルは実装そのもののルート）。
3. **バリエーション別実装**（widget 等の別コンポーネント＝分裂）。
4. **機能を枠（寸法規格）に合わせて削る**。
5. **詳細ページと道具箱を別実装にする**（同一エクスポートを両方が描くこと）。

詳細な事故の系譜と根本原因は `docs/cycles/cycle-226.md` 末尾「事故報告書（cycle-175 から続く設計破壊）」を参照。

## 6. 固定枠 flex の役割分担（結果膨張型ツール）

タイル・カード・限定領域のような**固定枠 UI** で flex container を使い、子要素に「サイズが動的に膨張する側（結果欄・出力テキストエリア等）」と「ユーザー操作上の最低高さを保つべき側（入力欄等）」が同居するときの CSS 挙動と設計パターン。

- `flex: 1` を複数の子要素に並列適用すると、両者が同じ比率で残余空間を取り合う。結果として膨張側が伸び、操作側が圧迫される。入力欄が縮んで編集不能になるなどの UX 破綻につながる。
- 「結果欄を枠内に収める」要件（`overflow: auto` で対処しがち）と「入力欄を圧迫しない」要件は、同じ flex container 内では両立しない場合がある。overflow は自分の領域を閉じるだけで、兄弟要素から領域を奪われること自体は防げないため。
- 対処は**役割で分担する設計**にする。操作側を `flexShrink: 0`（必要なら表示行数を固定）にして縮まないことを保証し、膨張側を `flex: 1` + `overflowY: auto` にして残余空間内で内部スクロールさせる。これで膨張側のコンテンツ量に関わらず操作側の最低高さが守られる。
- 参考実装が「両方とも固定サイズ（結果固定型）」だと、結果側が膨張しないためこの問題は顕在化しない。**結果固定型の参考実装の前提を、結果が動的に膨張する新対象へそのまま転用しない**こと。膨張側の存在が前提を変える。
- cycle-202 で実証。テキストが長くなると textarea（入力欄）が約 12px まで圧迫され、事実上編集不能になった。
