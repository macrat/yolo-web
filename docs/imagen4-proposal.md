# AI画像生成の仕組み設計 — AIチームが自由に画像を生成・活用するための提案

## 1. エグゼクティブサマリー

### 概要

AIチーム（Claude Code等のAIエージェント）が任意のタイミングで画像を生成し、各種コンテンツに活用できる仕組みを構築する。対象はブログ記事のサムネイル・挿絵、ツール・ゲーム・クイズ等のサムネイル、およびOGP画像。

仕組みの核は、MCPサーバーによる画像生成ツールである。AIエージェントはMCPツール（`generate_image`）を自然言語から直接呼び出すことで、通常のワークフローではAPIキーに触れることなく画像を生成できる。APIキーの保護はMCPプロトコルによる隔離、PreToolUseフックによるbashアクセス遮断、ルールによる禁止の多層防御で実現する。料金制限はサーバーのインメモリカウンター + JSON永続化で確実に制御する。

### 期待効果

- SNSシェア時のCTRが2〜3倍に向上（OGP画像の品質向上による）
- Google Discoverへの掲載可能性が大幅に向上（1200px以上の画像が掲載の前提条件）
- ブログ記事の可読性・理解度向上（挿絵による視覚的補足）
- 全コンテンツの一覧ページにおけるクリック率の向上

### コスト概算

Imagen 4 Fastで月間約$1.56（サムネイル20枚 + 挿絵40枚 + 再生成分、合計78枚想定）。詳細はセクション7参照。

## 2. 画像生成の仕組み設計

### 2.1 MCPサーバー方式（推奨）

#### アーキテクチャ概要

MCPサーバーのstdio方式により、Claude CodeがMCPサーバーをサブプロセスとして起動する。APIキーはMCPサーバープロセスの環境変数に閉じ込められ、Claude Codeのコンテキストには流れない。

```
Claude Code  ---(MCP protocol/stdio)--->  MCPサーバープロセス  ---(HTTPS)--->  Gemini API
                                           [GEMINI_API_KEY はここにのみ存在]
```

#### 既存OSSの活用 vs 自前実装

| 選択肢                        | 概要                                   | メリット                                     | デメリット                                              |
| ----------------------------- | -------------------------------------- | -------------------------------------------- | ------------------------------------------------------- |
| **chug2k/gemini-imagen4**     | TypeScript製、Imagen 4特化             | すぐに利用可能。GEMINI_API_KEY環境変数で認証 | 料金制限機能なし。出力先が固定（`./generated-images/`） |
| **lansespirit/image-gen-mcp** | Python製、マルチプロバイダー           | OpenAI/Google両対応。機能が豊富              | Python依存。Vertex AIサービスアカウント前提             |
| **自前実装**                  | TypeScript + @modelcontextprotocol/sdk | 料金制限・出力先制御を完全にカスタマイズ可能 | 初期実装コストがかかる                                  |

**推奨**: 自前実装を推奨する。理由は以下の通り。

1. 料金制限はownerの必須要件であり、既存OSSにはこの機能がない
2. 出力先ディレクトリの規約（セクション2.1の出力先ディレクトリ規約）に合わせたパス制御が必要
3. TypeScript SDKの最小構成で実装量は少なく、既存OSSのコードを参考にできる

#### ツール定義

MCPサーバーが提供するツール:

| ツール名         | 説明                             | パラメータ                                                          |
| ---------------- | -------------------------------- | ------------------------------------------------------------------- |
| `generate_image` | テキストプロンプトから画像を生成 | `prompt`（必須）, `output`（必須）, `aspectRatio`, `model`, `count` |
| `check_usage`    | 当月の使用状況を確認             | なし                                                                |

```
# ツールパラメータ
generate_image:
  prompt       画像生成プロンプト（必須、最大1000文字）
  output       出力ファイルパス（必須）
  aspectRatio  アスペクト比: 16:9 | 4:3 | 1:1 | 9:16（デフォルト: 16:9）
  model        モデル: fast | standard（デフォルト: fast）
  count        生成枚数: 1〜4（デフォルト: 1）
```

入力はZod スキーマでプロトコルレベルのバリデーションを行い、不正な入力はMCPの層で拒絶される。

#### MCPサーバーの配置場所

```
mcp-image-server/
  src/
    index.ts        # エントリーポイント（MCPサーバー定義）
    imagenClient.ts # Imagen API クライアント
    rateLimiter.ts  # 料金制限ロジック
  dist/             # ビルド出力
  package.json
  tsconfig.json
```

#### `.mcp.json` 設定

```json
{
  "mcpServers": {
    "image-generator": {
      "type": "stdio",
      "command": "node",
      "args": ["mcp-image-server/dist/index.js"],
      "env": {
        "GEMINI_API_KEY": "${GEMINI_API_KEY}"
      }
    }
  }
}
```

`${GEMINI_API_KEY}` はシェル環境変数から展開される。`.mcp.json` はバージョン管理に含めるが、APIキーの実際の値は含まれない。

#### 出力先ディレクトリ規約

| 用途             | パス                                        | 例                                                  |
| ---------------- | ------------------------------------------- | --------------------------------------------------- |
| ブログサムネイル | `public/images/blog/{slug}/thumbnail.webp`  | `public/images/blog/my-article/thumbnail.webp`      |
| ブログ挿絵       | `public/images/blog/{slug}/fig-{n}.webp`    | `public/images/blog/my-article/fig-1.webp`          |
| ゲームサムネイル | `public/images/play/{slug}/thumbnail.webp`  | `public/images/play/kanji-kanaru/thumbnail.webp`    |
| ツールサムネイル | `public/images/tools/{slug}/thumbnail.webp` | `public/images/tools/json-formatter/thumbnail.webp` |
| OGP画像          | 同上（サムネイルをOGPに兼用）               | ---                                                 |

#### MCPサーバーの処理フロー

1. Claude Codeが `generate_image` ツールを呼び出す（自然言語から自動的にツール選択）
2. Zodスキーマによる入力バリデーション（プロトコルレベルで不正入力を拒絶）
3. 料金カウンターの確認（上限超過ならエラーレスポンスを返却）
4. プロセス環境変数から `GEMINI_API_KEY` を取得し、Gemini API経由でImagen 4を呼び出し
5. 生成画像をWebP形式で指定パスに保存
6. 料金カウンターをインクリメント・永続化
7. MCPレスポンスとして生成結果の情報（ファイルパス、サイズ等）を返却

#### 代替案: CLIスクリプト方式

MCPサーバーが何らかの理由で利用できない場合のフォールバックとして、CLIスクリプト方式を残す。

```
npx ts-node scripts/generate-image.ts --prompt "プロンプト" --output public/images/blog/slug/thumbnail.webp
```

CLIスクリプト方式はMCPサーバー方式に比べて以下の制約がある:

- APIキーの隔離が弱い（シェル環境全体に展開される）
- 入力バリデーションが実装依存（プロトコルレベルの保証がない）
- エージェントがbashコマンドを組み立てる必要があり、引数エラーが起きやすい
- 料金制限にファイル永続化 + ファイルロックが必要で複雑

### 2.2 APIキー管理方法（要件1: AIがAPIキーに直接アクセスできない仕組み）

#### 課題

環境変数にAPIキーを置くだけではAIエージェントが`echo $GEMINI_API_KEY`等で読み取れてしまう。

#### 隔離の限界に関する重要な注記

Claude Codeのbashツールはシェルプロファイル（`~/.zshrc`等）を読み込んだ環境で実行される。そのため、シェル環境変数に`GEMINI_API_KEY`を設定した場合、AIエージェントが`echo $GEMINI_API_KEY`や`env | grep GEMINI`等のbashコマンドでAPIキーを読み取ることが技術的に可能である。

MCPプロトコルレベルではAPIキーがクライアント（Claude Code）のコンテキストに流れないという隔離は有効だが、これはMCPの通信経路に限定された隔離であり、bashツールという別経路でのアクセスを防ぐものではない。

したがって、完全な構造的隔離は現在のClaude Codeのアーキテクチャでは実現が困難であり、**MCPプロトコル隔離 + bashアクセス遮断フック + ルールによる禁止の多層防御**として設計する。

#### 設計（推奨）: 多層防御によるAPIキー保護

**第1層: MCPプロトコルによる隔離（通常ワークフローの保護）**

1. **APIキーの保管**: ownerがシェル環境変数（`~/.zshrc` や `~/.bashrc`）に `GEMINI_API_KEY=xxx` を設定する
2. **`.mcp.json` での参照**: `"env": { "GEMINI_API_KEY": "${GEMINI_API_KEY}" }` と記載する。`${GEMINI_API_KEY}` はClaude Codeがサーバープロセス起動時にシェル環境変数から展開してサーバーに注入する
3. **MCPレベルの隔離**: MCPサーバープロセスの `process.env.GEMINI_API_KEY` としてのみMCPプロトコル上に存在し、MCPレスポンスにAPIキーを含めない設計とする。通常のワークフロー（MCPツール呼び出し）ではAIエージェントがAPIキーに触れることはない

**第2層: PreToolUseフックによるbashアクセス遮断（別経路の防御）**

Claude Codeの`.claude/settings.json`にはPreToolUseフックの仕組みが既に存在する（現在はcommitチェック等に使用中）。これを活用し、bashツールの実行前に`GEMINI_API_KEY`を含むコマンドを検出・遮断するフックを追加する。

- `echo $GEMINI_API_KEY`、`env | grep GEMINI`、`cat ~/.zshrc`等のパターンを検出して実行を拒否する
- フックはClaude Codeのプロセス外で実行されるシェルスクリプトであるため、AIエージェントが回避することは困難である

**第3層: ルールによる禁止（意図レベルの防御）**

`.claude/rules/`にルールを追加し、AIエージェントに対してAPIキーの直接参照・環境変数の読み取りを明示的に禁止する。これは構造的な隔離ではなくAIの遵守に依存するが、第1層・第2層と組み合わせることで実効性が高まる。

**第4層: MCPサーバー内の安全設計**

MCPサーバー内でAPIキーをレスポンスに含めない設計とする。エラーメッセージにもAPIキーを含めない。

#### 代替案の検討結果

| 代替案                                                  | 評価   | 不採用理由                                                                                                                                                                                                                                      |
| ------------------------------------------------------- | ------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **案A: `.mcp.json`にAPIキー直書き + `.gitignore`**      | 不採用 | `.mcp.json`は現在playwrightサーバーも定義しておりバージョン管理に含めるべきファイル。APIキーを含むファイルと共有設定ファイルを混在させるとgitignore漏れのリスクがある                                                                           |
| **案B: `.claude/settings.local.json`でMCPサーバー設定** | 不採用 | `.claude/settings.local.json`はClaude Codeの設定ファイル（パーミッション等）であり、MCPサーバーの定義場所ではない。MCPサーバーは`.mcp.json`（projectスコープ）または`~/.claude.json`（local/userスコープ）で管理するのがClaude Codeの仕様である |

#### MCPサーバー方式の優位性（CLIスクリプト方式との比較）

| 観点             | MCPサーバー + 多層防御（推奨）                                                                                | CLIスクリプト + ルールベース制限      |
| ---------------- | ------------------------------------------------------------------------------------------------------------- | ------------------------------------- |
| 隔離の性質       | 多層防御（プロトコル隔離 + フック遮断 + ルール）                                                              | ルールベースのみ（AIの遵守に依存）    |
| 突破の難易度     | MCPプロトコル経由の漏洩は困難。bash経路はフックで遮断するが、フックのパターンをすり抜ける可能性はゼロではない | ルールを逸脱するプロンプトで突破可能  |
| 追加設定の必要性 | `.mcp.json` + PreToolUseフック + ルール                                                                       | `.claude/rules/` へのルール追加が必要 |

#### 代替案: CLIスクリプト + ルールベース制限

MCPサーバーが利用できない場合:

1. リポジトリルートの `.env.local`（`.gitignore` 済み）に `GEMINI_API_KEY=xxx` を記載
2. `.claude/rules/` にルールを追加し、AIエージェントが `.env.local` の読み取り・環境変数の直接参照を禁止
3. `scripts/generate-image.ts` が実行時に `.env.local` を読み込む

この方式はAIエージェントの「ルール遵守」に依存するため、構造的な隔離ではない。

### 2.3 料金制限の仕組み（要件2: 確実な制限）

#### 多層防御の設計

**第1層: MCPサーバーのインメモリカウンター + JSON永続化（ハードリミット）**

MCPサーバーはプロセスが継続動作するため、インメモリでカウンターを管理できる。サーバー再起動時にカウントを引き継ぐため、JSONファイルへの永続化も行う。

- カウンターファイル: `tmp/.image-gen-counter.json`
- 記録内容:
  ```json
  {
    "month": "2026-03",
    "count": 15,
    "cost_usd": 0.3,
    "history": [
      {
        "timestamp": "2026-03-20T10:30:00Z",
        "model": "fast",
        "prompt_hash": "abc123",
        "output": "public/images/blog/my-article/thumbnail.webp"
      }
    ]
  }
  ```
- ツールハンドラー内で生成前にカウンターを確認し、月間上限を超過していれば `isError: true` のレスポンスを返して生成を拒否
- リセットタイミング: 月が変わったら自動リセット
- 上限値: サーバー内の定数として定義（例: `MAX_MONTHLY_IMAGES = 100`, `MAX_MONTHLY_COST_USD = 5.00`）
- ownerが上限値を変更する場合はサーバーの定数を編集して再ビルドする
- `history`フィールドにより、どの画像がいつ生成されたかを追跡可能
- インメモリカウンターのため、CLIスクリプト方式のようなファイルロック（排他制御）は不要。単一プロセスでの逐次処理により競合状態が構造的に排除される

注: MCP調査資料（`2026-03-20-mcp-image-generation-research.md`）のサンプルコードでは日次制限（`DAILY_LIMIT_COUNT`, `DAILY_LIMIT_COST_USD`）で実装されているが、本プロジェクトでは月次制限にカスタマイズする。月単位の予算管理の方がownerにとって分かりやすく、Google AI StudioのMonthly spend capとも粒度が揃うためである。

**第2層: Google AI Studio側の制限**

- Google AI Studioの「Spend」タブで月間支出上限（Monthly spend cap）を設定する（Googleが提供する機能）
- IPM（Images Per Minute）のティアベース制限が自動適用される（Tier 1: 10 IPM）
- これらはMCPサーバー側のカウンターが万一故障した場合のセーフティネットとして機能する
- **注意**: Monthly spend capには約10分の遅延があり、その間の超過分はユーザー負担となる。したがって第1層のMCPサーバー内カウンターが真の防御線であり、第2層はあくまでバックアップである

**第3層: Google Cloud Billing アラート（任意）**

- 予算アラートを設定し、閾値超過時にメール通知を受け取る
- ハードリミットではないが、異常を早期発見できる

### 2.4 AIからの呼び出し方法（要件3: 任意のタイミングで呼び出し）

AIエージェント（Claude Code）はMCPツールとして自然言語から直接画像生成を呼び出す。

#### MCPツール呼び出しの特徴

- Claude Codeが `.mcp.json` に定義されたツールを自動認識する
- 「この記事のサムネイル画像を生成して」のような自然言語の指示から、`generate_image` ツールが自動的に選択・呼び出される
- 引数はJSON Schemaに基づいて構造化されるため、CLIコマンドの組み立てエラーが発生しない
- ツールのレスポンスがそのままClaude Codeのコンテキストに返るため、結果の確認がシームレス

#### 呼び出しタイミング

| タイミング              | 用途              | 例                             |
| ----------------------- | ----------------- | ------------------------------ |
| ブログ記事作成時        | サムネイル + 挿絵 | 新規記事の公開前に生成         |
| 既存コンテンツ改善時    | サムネイル追加    | 画像未設定のコンテンツへの追加 |
| ゲーム/ツール新規作成時 | サムネイル        | コンテンツ一覧での表示用       |
| OGP改善時               | OGP画像更新       | SNSシェア効果の改善            |

#### AIエージェントのワークフロー

1. コンテンツの内容を分析し、画像が必要かどうかを判断する
2. 画像が必要な場合、適切なプロンプトを設計する
3. MCPツール `generate_image` を呼び出して画像を生成する
4. 生成された画像をClaude Codeの画像読み取り機能（Readツール）で確認する
5. 品質・適切性を満たしていれば、コンテンツに組み込む
6. 不十分であれば、プロンプトを調整して再生成する

#### スキルファイルへの記載

`.claude/skills/`にスキルファイルを作成し、画像生成の手順・判断基準・プロンプト設計のガイドラインを記載する。これによりAIエージェントが一貫した品質で画像生成を実行できる。

### 2.5 掲載可否判断フロー（要件4: AIがチェックして判断）

#### 確認方法

Claude Codeは画像ファイルを読み取る機能を持っている。生成された画像ファイルのパスをReadツールで開くことで、画像の内容を視覚的に確認できる。

#### 判断基準

1. **安全性**: 不適切な表現（暴力・性的・差別的な内容）が含まれていないか。Imagen 4の安全フィルタが一次フィルタとして機能するが、AIが二次確認を行う
2. **記事との関連性**: 画像がコンテンツのテーマ・内容を正確に表現しているか
3. **品質**: 不自然なアーティファクト、崩れたテキスト、不適切な構図がないか
4. **ブランド一貫性**: yolos.netの全体的なトーンやスタイルと調和しているか

#### フロー

```
生成 → AI確認 → [合格] → コンテンツに組み込み → コミット
                → [不合格] → プロンプト修正 → 再生成（最大3回）
                           → 3回不合格 → フォールバック（既存Satoriベース画像 or 画像なし）
```

#### 再生成時の注意

- 再生成も料金カウンターに加算される
- 最大3回の再生成で解決しない場合は、無駄なコストを避けるためフォールバックに切り替える

## 3. 画像の用途と適用範囲

### 3.1 ブログ記事サムネイル

- ブログ記事一覧ページのカードに表示する画像
- OGP画像としても兼用（SNSシェア時に表示）
- Google Discoverへの掲載条件である1200px以上の画像を満たす
- アスペクト比: 16:9（OGP標準）
- BlogPostMetaに`thumbnail`フィールドを追加

### 3.2 ブログ記事の挿絵

- 記事本文中にMarkdownの画像記法（`![alt](path)`）で挿入する画像
- 記事の内容を視覚的に補足し、読者の理解を助ける役割
- アスペクト比: コンテンツに応じて4:3または16:9

#### 挿絵の挿入判断基準

| 判断基準       | 説明                                                              |
| -------------- | ----------------------------------------------------------------- |
| 概念の視覚化   | 抽象的な概念を説明する箇所に、視覚的な補足として挿入              |
| 手順の図示     | ステップバイステップの説明で、各段階のイメージを提供              |
| 雰囲気の演出   | 記事の冒頭やセクション区切りで、テーマに合った雰囲気を作る        |
| 過剰挿入の回避 | 1記事あたり1〜3枚を目安とし、画像が内容の邪魔にならないようにする |

#### 挿絵を入れないケース

- 技術的なコード解説記事（コードブロック自体が視覚的要素として機能する）
- 画像による補足が読者の理解に寄与しない場合
- AIエージェントが記事内容を分析し、挿絵の必要性をケースバイケースで判断する

### 3.3 ゲーム・ツール・クイズのサムネイル

- 一覧ページ（`/play/`, `/tools/`等）でのカードに表示する画像
- 各コンテンツの機能や特徴を一目で伝える役割
- OGP画像としても兼用
- アスペクト比: 16:9

#### 現在の対象コンテンツ

- ゲーム: `kanji-kanaru`, `irodori`, `nakamawake`, `yoji-kimeru`, `daily`（5種）
- ツール: `json-formatter`, `password-generator`等（35種）
- 計40種以上のコンテンツにサムネイルを追加可能

### 3.4 OGP画像

- 上記すべてのサムネイル画像がOGP画像として兼用される
- 既存のSatoriベースOGP画像生成（`opengraph-image.tsx`）は、AI生成サムネイルが存在しないコンテンツのフォールバックとして残す
- AI生成サムネイルが存在するコンテンツでは、そちらを優先してOGPに使用する

## 4. コーディング原則との整合性

`.claude/rules/coding-rules.md`のルール2に「外部APIの呼び出し、データベース、認証・ユーザー管理は実装しない」と記載されている。

ただし、このルールの`paths`指定は`src/**/*`であり、MCPサーバー（`mcp-image-server/`配下）および`scripts/`配下は適用範囲外である。画像生成MCPサーバーを`mcp-image-server/`として配置することで、ルール2に抵触しない。

加えて、ルール2の趣旨は「情報漏洩などによりユーザーに危害を加えることを防ぐため」であり、画像生成APIの呼び出し（ユーザーデータを送信しない、ビルド時のみの実行）はこの趣旨に直接的に反するものではない。

**結論**: `mcp-image-server/`配下にMCPサーバーとして実装する方式で、コーディング原則との整合性を確保する。

## 5. ビルド時生成の方針

coding-rules.mdのルール1「静的最優先」に基づき、画像はビルド時（記事作成時）に生成し、リポジトリにコミットして静的配信する。

| 観点          | ビルド時生成（採用） | ランタイム生成             |
| ------------- | -------------------- | -------------------------- |
| ルール1適合性 | 完全に合致           | 抵触                       |
| 表示速度      | 静的配信のため高速   | API応答時間に依存（数秒）  |
| コスト予測性  | 完全に予測可能       | トラフィック依存で予測困難 |
| 障害耐性      | API障害の影響なし    | API障害時に画像なし        |

画像はリポジトリにコミットするため、gitリポジトリのサイズ増加に留意する。WebP形式で圧縮し、1画像あたり50〜200KB程度に抑える。

## 6. AI生成画像の表示

constitution.md ルール3「サイトがAIにより運営されていることを来訪者に通知する」に基づき対応する。

- サイト全体のAI運営告知（既存）の文言を拡充し、「コンテンツ・画像を含むサイト全体がAIにより生成・運営されている」旨を明記する
- Imagen 4はSynthID（不可視電子透かし）とC2PAメタデータを全生成画像に自動付与するため、技術的な透明性は自動的に確保される
- 個別画像へのラベル（例:「AI生成画像」バッジ）付与はUXとのバランスを考慮してowner判断とする
