# デザインガイドライン — 静かに傍にある

## このドキュメントの役割

サイトのすべての UI 判断を、コンセプト「静かに傍にある」に統一するための羅針盤。個別の色値や実装手段を決めるのが目的ではなく、**判断の芯**を決める。具体値（色の hex、間隔、角丸の大きさ）は `src/app/globals.css` に一元化し、本ドキュメントでは**なぜそう判断するのか**だけを書く。

### 誰がいつ参照して、どう使うか

- **PM**: UI タスクを sub-agent に依頼する前と、成果物を受け取ったとき。「静かか／傍にあるか」を問う
- **builder**: UI 関連ファイル（`.module.css`、`src/components/**`、`src/app/**/*.tsx`、`src/**/_components/**`）を編集する前、判断に迷ったとき。後述の「作業前の必読手順」に従う
- **reviewer**: UI 関連成果物のレビュー時。「静かに傍にある」から外れていないかを検証する
- **Owner**: サイクル境界の方向性確認、PM の判断がコンセプトからずれていないかを見る基準として
- **将来のサイクル担当**: 新規 UI タスク（B-309 ダッシュボード、B-310 トップ再設計、B-317〜B-321 新規ツール等）の着手時、ゼロから判断せずに本ドキュメントを出発点にする

### 期待する効果

- サイクル・作業者・コンテンツを越えた UI 姿勢の一貫性
- 「何色にするか」をゼロから考えるコストの削減（「静かか／傍にあるか」で自然に絞れる）
- AI Slop（紫〜青グラデ、`rounded-2xl` + `shadow-lg` などのテンプレ）への抵抗力
- Owner と PM と builder と reviewer の共通言語（「もっと静かに」の意味が揃う）

## このサイトのデザインを一語で

**「静かに傍にある」**

- **静かに**: 装飾で注目を奪わない。動きで邪魔しない。押し付けない。「この色だ」「この機能を見ろ」と主張しない
- **傍にある**: 使う人の用事の脇に居続ける。変わらない位置、変わらない手順、変わらない体感。環境（OS のダークモード、reduced-motion、システムフォント）に黙って寄り添う

このひと言が、yolos.net のあらゆる UI 判断の羅針盤になる。色・形・動き・文面を迷ったら「これは静かか／傍にあるか」と問う。答えがどちらもイエスなら採用、どちらかがノーなら別案を探す。

**この姿勢はサイト全体で一貫する**。ツールでも、ブログ記事でも、辞典でも、チートシートでも、コンテンツの種類で姿勢を切り替えない。違うのはコンテンツ（道具なのか文章なのか）の性質であって、デザインの姿勢はどのページでも同じ。

## コンセプトの階層

cycle-167〜169 で、サイト全体・PM・デザインの 3 層が一語ずつ根を持った：

- サイト全体: 「**日常の**傍にある道具（と、ちょっとした息抜き）」（`docs/site-concept.md`）
- PM キャラクター: 「人々の傍に**在りたいと願う**」（`docs/character.md`）
- デザイン: 「**静かに**傍にある」（本ドキュメント）

「傍」が 3 段そろっている。サイトが「日常の」傍にあり、PM が「在りたい」と願い、デザインが「静かに」そこにいる。どの層も、自分が主役になろうとしない。使う人の用事・時間・読み手のほうが主役。道具は脇役として仕える。

## ガイドラインが決めるのは「姿勢」

このドキュメントは色の hex 値の表ではない。姿勢と判断基準を書く。具体値は `src/app/globals.css` に一元化する（二重管理しない）。姿勢から導かれる制約（例:「Noto Sans JP の 9MB を無条件で読み込まない」）は明文化する。

**文体（書き言葉の温度、warm and eager）は本ガイドラインの領分ではない**。それは `docs/character.md`（書き言葉版）と `.claude/agents/blog-writer.md` が扱う別の話。本ガイドラインはデザインの姿勢だけを決め、ブログ記事の文体には踏み込まない。ただし、文体がどんなに温度を持とうとも、そのまわりのデザインは「静かに傍にある」を守る。

## 出発点: ターゲットユーザー定義

`docs/targets/` に定義された 5 種類のターゲットユーザー像を出発点とする。**名前・年齢・職業などのペルソナ化は意図的に避ける**（`docs/targets/README.md` L50-52 参照: 具体化しすぎると AI がその人物だけに最適化したコンテンツを作ってしまうため、あえて抽象度を保つ方針）。本ガイドラインも同じ抽象度で扱う。

5 種類のタイプと、各タイプが持つ代表的な「願いの形」（ターゲット定義の `likes` / `dislikes` から抽出）：

- **特定の作業に使えるツールをさっと探している人**
  - 開いた瞬間に入力欄が見えてすぐ使い始められること／コピペで結果を受け取ってすぐ元の作業に戻れること／余計な説明や装飾がなく用事だけ静かに片付けられる画面
  - 嫌うもの: 広告やポップアップで入力欄が埋もれること／会員登録やインストール要求／ツール冒頭の長い解説

- **気に入った道具を繰り返し使っている人**
  - 久しぶりに開いても前回と同じ場所に入力欄があり同じ手順で済むこと／ブックマークを開けばすぐ目的のツールが表示されること／仕様変更で使い方が壊されないこと
  - 嫌うもの: 刷新で手順が変わること／後から広告・ログイン要求・通知バナーが増えること／同じ入力で結果や挙動が前回と変わること

- **AI エージェントやオーケストレーションに興味があるエンジニア**
  - AI を使った自動化の具体的な設計判断／反面教師となる具体的失敗例／試行錯誤と判断の考察
  - 嫌うもの: 実践的な話が少ない一般論／再現性のない事例／説明なしの手順

- **AI の日記を読み物として楽しむ人**
  - 作業や出来事の具体的な細部に視点が宿った文章／揺れや迷いをそのまま書いてあること／派手な宣伝文句ではなく淡々とした観察
  - 嫌うもの: 「いかがでしたか」のような定型／起承転結・箇条書きが綺麗すぎること／絵文字や「！」の多用で感情演出すること

- **Web サイト製作を学びたいエンジニア**
  - 手元ですぐ試せるコード例・チートシート・リファレンス／設計判断の「なぜそうしたか」の説明／失敗やエラーの具体事例と解決策
  - 嫌うもの: 一般論や抽象的な話／再現性のない個別事例／コードなしで文章だけで説明される記事

## 5 タイプに共通する「願いの形」

5 タイプは別々のニーズを持つが、願いを見ていくと形が揃っている。**すべて否定形**——「何かを足してほしい」ではなく「邪魔しないでほしい」：

- 検索から来た用事を、装飾で遮らないでほしい（「さっと探している人」「学びたいエンジニア」）
- 慣れた景色を、刷新で壊さないでほしい（「繰り返し使っている人」）
- 書き手の試行錯誤や視点を、周辺の装飾で埋めないでほしい（「AI エージェントに興味のあるエンジニア」「AI の日記読者」）

デザインが積極的にするのは「何かを足すこと」ではなく、「**使う人の時間から退くこと**」。これが「静かに傍にある」の実務面での定義。

## 判断基準: 静かに在るために

### 色

- 背景はニュートラル。オフホワイトとオフな暗色（純白と純黒は眩しい／冷たい、避ける）
- アクセントは 1 色のみ。CTA と重要なフィードバックにだけ使う
- 配色で「おもてなし」しない。カテゴリ別の虹色展開をしない
- グラデーション背景を作らない（特にヒーロー）。Slop シグナル直撃
- バッジ類（カテゴリバッジ、TrustLevel バッジなど）も抑制的な見た目に。主張を強めない
- quiz/games の各コンテンツに個別定義された `accentColor` はゲーム固有のアイデンティティとして `--color-primary` と別系統で並立させる。将来 Slop シグナルに直撃する判断が出た場合に個別見直し（本サイクルは触らない）

### 形

- 角丸は小さめで統一（道具らしさ。派手な `rounded-2xl` はしない）
- シャドウは控えめ、または使わない（浮き立たせない）
- 線は細く、主張しない
- 余白は日本語を読み慣れた人に心地よい間（欧文基準より気持ち広め）
- 装飾を本文や機能の上に被せない（ガラス調背景、派手なサイドバー、記事上のグラデーション等を使わない）

### 動き

- アニメーションは機能的なフィードバックのみ。状態変化を最小限に伝える
- 装飾モーション、スクロール連動エフェクト、パララックスをしない
- トランジションは 0.15 秒以下
- `prefers-reduced-motion: reduce` を尊重

### 割り込み

- 音を出さない（効果音・通知音ともに。将来の機能追加時も同様）
- 使う人の意志で起動するモーダル（SearchModal など）は許容する。歓迎ポップアップ・お知らせバナーのような受動的な割り込みは作らない
- 「新機能！」「登録で特典」的なバナーを作らない
- ナビゲーションは本文・機能の集中を妨げない位置と密度に保つ

### 文字とボタン文言

- 過剰な太字強調、感嘆符、絵文字での感情演出を避ける。文意があるところで最小限に使う
- 本文は読みやすさを優先する（行間・サイズの具体値は W3 で決定。記事でもツール説明でも同じく読み物水準）
- ラベル・補助テキストは小さくしすぎない。判読できる大きさを保つ
- ボタン文言は動詞で短く（「コピー」「変換」「検索」など）。ユーザーが次に起こることを予測できる文言にする

## 判断基準: 傍にあり続けるために

### 位置と手順の一貫性

- 入力欄・ボタン・ナビゲーションの位置を安易に変えない（繰り返し使っている人の景色を壊さない）
- 文言変更も控えめに。「コピー」を突然「クリップボードに保存」に変えない
- ショートカットがあるなら保持

### 環境との整合

- OS のダークモード設定に追従（`prefers-color-scheme` via `next-themes`）
- システムフォントを優先してロード待ちを作らない（ラテン先頭、日本語は OS に任せる）
- `prefers-reduced-motion` を尊重
- スクリーンリーダー・キーボード操作を壊さない

### URL の安定

- ブックマークが生きるように、既存 URL を変えない
- リダイレクトを増やさない
- 正規化・スラッシュ付与などの URL 変更はブックマーク破壊なので慎重に

### 前回値の余地

- localStorage で入力値・設定をそっと覚える実装を許容する
- ただし **同じ入力には同じ結果を返す**（ロジックの安定性が優先）
- 前回値の UI 上の提示も控えめに（勝手に復元する、目立つ通知を出す、は傍を離れる）

## Do / Don't のペア例

静かに傍にある姿勢が、実装のどこで試されるかの代表例。

### 色のハードコードを避ける

```css
/* Don't */
.button {
  background: #2563eb;
  color: #ffffff;
}

/* Do */
.button {
  background: var(--color-primary);
}
```

理由: ハードコードでは、トークンの変更が全体に波及しない。コンセプト変更時に一括移行できない。

### ヒーローに装飾グラデを置かない

```css
/* Don't（旧コンセプトの残り香） */
.hero {
  background: linear-gradient(135deg, #6c3bea 0%, #4466dd 40%, #0a7080 100%);
  padding: 8rem 0;
  text-align: center;
}

/* Do（高さは内容に従う） */
.hero {
  background: var(--color-bg);
}
```

理由: 紫→青→シアン系グラデは AI Slop シグナル。装飾で用事の時間を奪う。ヒーローそのものを作らない選択肢も検討する。

### 動きは機能に閉じる

```css
/* Don't */
.card {
  transition:
    transform 0.6s ease,
    opacity 0.6s ease;
}
.card:hover {
  transform: translateY(-8px) scale(1.02);
}

/* Do */
.card {
  transition: border-color 0.15s ease;
}
.card:hover {
  border-color: var(--color-primary);
}
```

理由: 装飾的な浮き上がりは、静かではない。ボーダーの色変化は「押せる」を伝える機能的フィードバック。

### 割り込みで作業を遮らない

```tsx
// Don't（自動起動の割り込み）
useEffect(() => {
  if (isFirstVisit) showWelcomeModal();
}, []);

// Do（意志ある起動）
<SearchTrigger />; // Cmd+K / Ctrl+K でユーザーが起動（src/components/search/SearchTrigger.tsx）
```

理由: 用事にたどり着く前の受動的な割り込みは、静かに傍にあることの正反対。

### フォントでロードを待たせない

```css
/* Don't（9MB の Noto Sans JP を全ページに適用） */
body {
  font-family: "Inter", "Noto Sans JP", sans-serif;
}

/* Do */
body {
  font-family:
    -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, "Hiragino Sans",
    "Hiragino Kaku Gothic ProN", "Meiryo", sans-serif;
}
```

理由: ロード待ちで文字がちらつくのは、傍にあるべき道具が一瞬消えること。システムフォントは OS の環境に黙って寄り添う。

### 環境設定に追従する

```css
/* Don't（OS 設定を無視した独自カラースキーム） */
.card {
  background: #ffffff; /* 常に白 */
}

/* Do（next-themes の `:root.dark` に追従） */
.card {
  background: var(--color-bg-secondary);
}
```

**注意**: CSS 変数のみで自動対応するため、個別ダーク分岐は通常不要。変えたい場合のみ `:global(:root.dark)` で上書きする。`[data-theme="dark"]` や `@media (prefers-color-scheme: dark)` は使わない（`next-themes` の `attribute="class"` 方式のため、ユーザーのテーマトグルに追従しない）。

## 作業前の必読手順

UI に関わるファイルを編集・レビューする前に、毎回以下を踏む。短いが省略しない。

1. **`src/app/globals.css` を Read する**。使えるトークン（CSS 変数）を確認する
2. **類似コンポーネントを Grep で探す**。複数ヒットした場合は優先順に従う: `src/components/common/` の共通実装 → 近い領域（`src/tools/_components/` 等） → なければ globals.css のトークンに照らして自作（繰り返し利用者の景色を壊さない）
3. **このガイドライン（本ドキュメント）の該当節を読む**。特に「判断基準」の 2 節（静かに在るために／傍にあり続けるために）
4. **「静かに傍にある」と自問する**: 書こうとしている CSS は静かか、傍にあるか

この 4 ステップは `.claude/skills/design-guideline/SKILL.md` に短く再掲する。

## コアトークン（cycle-169 で定義）

<!-- W3 で値を決めて globals.css に追加する。本節はセマンティクスのみを保持する（値の二重管理を避ける） -->

### 色

- `--color-primary`: 主要アクション色
- `--color-primary-hover`: 主要アクションのホバー状態
- `--color-bg`: ページ背景（ニュートラル、オフホワイト／オフな暗色）
- `--color-bg-secondary`: カード背景などの一段下のレイヤー
- `--color-text`: 本文テキスト（純黒を避けたチャコール）
- `--color-text-muted`: 補助テキスト
- `--color-border`: 罫線、区切り
- `--color-focus-ring`: キーボードフォーカスリング（WCAG 2.2 Focus Appearance Minimum 準拠）

### Focus ring

- `outline: none` でフォーカス表示を消さない
- すべての操作可能要素（ボタン、リンク、入力欄など）に一貫したフォーカス表示を置く
- WCAG 2.2 Focus Appearance Minimum 準拠を目標とする
- 具体値（outline の幅・offset・色）は W3 で globals.css に一元化して決定する

### Spacing

- `--space-*`: 基本的な余白。段階数と値は W3 で決定する

### Radius

- `--radius-*`: 角丸の基本セット。値は W3 で決定する

### ブレイクポイント（CSS 変数化しない）

- 640px（小型モバイル）、768px（モバイル主要）の 2 段階
- `@media` 内で直接使う

### font-size（本サイクルではトークン化しない）

既存が 13 段階以上散らばっているため、次サイクル以降で段階移行。本サイクルでは触らない。

## 将来の拡張条件（YAGNI で後回しにしたもの）

以下は「静かに傍にある」の延長として有効だが、本サイクルでは採用しない。条件が満たされたら検討する：

- **Stylelint / PreToolUse hook**: ハードコード事故が 2 件以上観察されたら導入
- **font-size トークン化**: 既存の散らばりが落ち着く後続サイクルで段階的に
- **Taste Skill 風の 3 パラメーター**（DESIGN_VARIANCE / MOTION_INTENSITY / VISUAL_DENSITY）: 複数 builder 並走時に密度のバラつきが 2 件以上蓄積したら導入
- **Noto Sans JP サブセット版の導入**: 「システムフォントでは書き手の温度が伝わらない」事例が具体的に出たら検討
