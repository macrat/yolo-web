# 読み進めるリズムの設計（認知リズム）

密度の高い記事が退屈になるのは、情報が多いからではない。全文が同じ認知モードで書かれているからだ。読者は「観察する」「迷う」「確信する」「確かめ直す」というモードを行き来しながら読む。このモードの切り替えを意図的に設計し、常に「続きを読む理由」を残しておくことで、最後まで読み進める推進力が生まれる。

このドキュメントは、k16shikano さんの [認知リズムを生むための日本語ライティング規範](https://gist.github.com/k16shikano/eb2929f13ed19c97188393d297be8432) を、yolos.net のブログ執筆ルールに合わせて再構成したものだ。原典の判定軸と点検手順を土台に、このサイトの事実確認ルール（経緯の捏造禁止）との接続を加えてある。

## 他のドキュメントとの役割分担

- `docs/character.md` は文体の親方針（SSoT）。「なぜこう書くか」の価値観を持つ。
- `docs/anti-patterns/writing.md` は内容・構成の失敗を扱う。
- `docs/japanese-ai-slop.md` は表層の文体から AI 臭を消す引き算のリスト。
- このドキュメントは、消したあとに読み進めるリズムをどう作るかという足し算を扱う。

japanese-ai-slop.md のリズム観点にも「文の長さと温度にムラを残す」といった基礎的な処方はある。その先、記事全体のどこに緩急を置き、何を材料に緊張を作るかという設計を決めるのが、このドキュメントの仕事だ。

## 基本原理

- 緩急は情報量の増減ではなく、認知モードの切り替えとして設計する。観察→逡巡→断定→再観察の往復が一つの単位になる。
- 記事は常に、少なくとも一つの未回収の緊張（答えの出ていない問い、裏の取れていない確信、あとで返すと約束した答え）を開いておく。緊張がすべて閉じた瞬間、読者は読むのをやめられる。
- 完成した結論を解説する声ではなく、考えながら進む当事者の声で書く。書き手が答えに至る過程の再演が、読者の思考の再演になる。
- 生成側の制約: この規範を使って文を新しく書くとき、リズムの材料は状況側からしか取らない。状況側とは、対象世界の出来事・データ・発言と、書き手の実際の判断状態（思い込み、保留、後悔、譲歩）のことだ。状況に材料が見つからない位置には、何も足さず平坦なまま残す。本文自身を話題にした文でリズムを作るのは、この規範の違反であって適用ではない。新しく書いた文・推敲で整形した文は、書いた時点で「緩みと駄文の見分け方」の判定にかける。
- 装置は実現するものであって、宣言するものではない。「先に答えを半分だけ置く」「最後にもう一度だけ線を引く」のように、これからやる操作を宣言した文は、それ自体が駄文だ。装置が正しく機能していれば、読者は装置の存在に気づかない。この規範の語彙（「緊張」「回収」「答えの半分」等）も本文には書かない。
- 場面を持たない解説記事では、対象そのものの性質（データ、計算、トレードオフ、素朴な期待が事実に破られること）と、読者が抱く推論・反問が「状況」になる。緊張は対象の性質から作る。場面がないからといって、本文の進行を語ることで緊張の代用にしない。
- 拍を作るために文を削らない。導入で必要な文脈共有（範囲、観点、比較軸、未確定事項）を削って短くするのは、緩急ではなく欠落だ。密度を上げる編集は、共有済みの文脈の上でだけ行う。

## このサイトでの適用上の制約（事実の一線）

原典の技法のうち「あとで事実に裏切られる思い込み」を布石として置く手法は、このサイトでは実際にあった判断状態にしか使えない。一人称で自分の経緯を書くときは、一次資料（git log・過去の記事・サイクル記録・チャット履歴）で確認した事実しか書けない（`docs/anti-patterns/writing.md` AP-W16）。リズムのために、実際には抱かなかった思い込みや逡巡を演出として捏造することは、経緯の捏造そのものであり、厳禁だ。

逆に言えば、一次資料には布石の材料が眠っている。当時の判断ミス、あとで外れた見立て、保留にしたまま忘れていた課題。サイクル記録やチャット履歴を掘れば、演出ではない本物の「裏切られた確信」が見つかる。リズムの材料集めは、事実確認と同じ場所で行う。

また、記事冒頭の免責表示（AI 運営の告知）は constitution rule 3 が求める定型の告知であり、この規範の判定対象にしない。

## 文の拍

- 短文で足場を打ち、長めの文で流し、短文で止める。「立てる→流す→止める」を段落の基本拍にする。
- 断定だけで押し切らない。断定（「〜だった」「〜である」）と逡巡（「〜に違いない」「〜とは思う。ただ…」「〜だろうか」）を交互に置く。
- 逡巡は弱さではなく仕掛けだ。あとで事実に裏切られる確信は、読者の予測を誘導してから崩す布石になる（前述のとおり、実際にあった確信に限る）。
- 逡巡と、`docs/japanese-ai-slop.md` が禁じる「判断の先送り」（言い切れるところを推量や問いかけでぼかす）との区別も、回収の有無で決まる。逡巡は、あとで崩されるか答えが返される判断状態の記録であり、緊張台帳に載って回収位置を指せる。回収されない「〜かもしれない」「〜ではないだろうか」は、立場をぼかす hedging であって逡巡ではない。`docs/character.md` の言い切り方針とはこうして両立する。逡巡の例: 「うまくいっているに違いない。」→ 後段に「ところが、記録を見るとそうではなかった」という崩しがある。先送りの例: 「効果があるのではないでしょうか。」→ どこにも答えを返さないまま次の話題へ進む。
- 転回点では「譲歩→転回→短い停止」の拍が使える。転回のあとの短い指示文が、読者の視線を固定する。転回は、実質のある譲歩（状況の更新）を前に置くから効く。中身が要らないのに「Xではなく、Y」の形だけを繰り返すのは、`docs/japanese-ai-slop.md` が禁じる否定並列であって転回ではない。転回の例: 「この方法でも動くだろう。これからも動くだろう。しかし、ここで扱いたいのは、動かなくなった日のほうだ。」（譲歩に実質がある）。否定並列の例: 「単なるツールではない。パラダイムシフトだ。」（譲歩に中身がなく、形だけの反転）。
- 長い断定文を 3 つ以上連続させない。連続していたら、短い足場か停止、または逡巡を挿む。

## 段落の密度波形

- 密な段落が 2〜3 個続いたら、疎の段落を 1 つ置く。疎の段落の機能は、確定事項の一行固定、次の判定対象の提示、視点距離の切り替えのいずれかに限る。
- 視点の距離を固定しない。具体（記録、数値、発言、コード）に寄る段落と、意味づけで一段引く段落を交互に置く。
- 箇条書きは情報の圧縮だけでなく、本文の呼吸を止める「間」としても使える。列挙のあとの一段引いた文は、この間があるから効く。

## 冒頭の設計

冒頭の仕事は、最初の数文で未回収の緊張を一つ作ることだ。型は問わない。使える型の例:

- 読者の実感の言い直しから仮説へ
- 読者への直接の問いかけ。ただし置き去りにせず、すぐ自分の答えを返す
- 確信を帯びた一般命題。あとの本文がその確信を試す
- 書き手の思い込みを肯定的に書き切ってから、事実で崩す場面（実際にあった思い込みに限る）
- 前の記事が残した問いの、当事者の言葉での言い直し

どの型にも共通の条件がある。冒頭で置いた実感・命題・確信は、本文が必ず試すか崩すか回収することだ。`docs/japanese-ai-slop.md` が「決めつけ開幕」として禁じるのは、読者の内心や一般論を根拠なく断定して、そのまま放置する開き方のこと。同じ形に見えても、緊張台帳（後述の点検手順）で回収位置を指せるなら冒頭の型であり、指せないなら決めつけ開幕だ。

- 冒頭の型の例: 「密度の高い記事が退屈になるのは、情報が多いからではない。」→ 本文がこの命題を検証し、答えを返す
- 決めつけ開幕の例: 「文章に自信がある人は少ないはずだ。」→ 断定したきり本文が受けず、別の話へ進む

問いかけも同じ軸で分かれる。直後に自分の答えを返すから型になるのであって、自答せず読者に同意を取りに行く「〜ではないでしょうか」は、`docs/japanese-ai-slop.md` が禁じる媚び・判断の先送りだ。可/不可の例文は同ドキュメントの「全方位肯定と媚び」の項にある。

予告や要約は禁止ではない。態度を帯びた一〜二文（「〜を考えるうえで、これほど適切な切り口もないはずだ」「言い換えると〜という話である」）なら、それ自体が緊張を作る。禁止すべきは、態度のない議題表（「この記事では A、B、C を扱う」）だけだ。`docs/japanese-ai-slop.md` の「削減」が挙げるメタ構造宣言との線引きも、ここで一緒につけておく。迷ったら話題テストにかける。対象への態度・仮説という状況を更新していれば残せる予告、本文の進行だけを伝えていれば削る宣言だ。

読者が抱くであろう抵抗（古い、作為的、実用性がない、自分には関係ない）は、読者の言葉で先に言い、短く処理してから本題に入る。

## 節（セクション）の入り方

- 節の頭で「本節では〜を扱う」と宣言しない。代わりに次のいずれかで入る。
  - 直前の節が残した違和感を、当事者の問いとして言い直す
  - 読者が当然抱く反問をそのまま書く。反問には即答せず、一度「そうしたかった、とは思う」と受けてから崩す
  - 書き手の告白から入る。告白は自己批判のためではなく、直後の論証の足場として使う。中身と同じ文で置くこと（「白状すれば、〜という算段もあった」）。中身から切り離して「白状しよう」とだけ宣言する文は、`docs/japanese-ai-slop.md` が禁じる予告の言い回しであり駄文だ
- 理論・概念・引用は、読者の中に「まだ名前のない違和感」を作ってから導入する。理論は答えではなく命名として入れる。先に理論を出して例で確認する順は、読者の発見を奪う。
- 節と節の橋は、前節の末尾ではなく次節の頭に置く。前節の末尾に「次は〜を見る」型の予告を足すのは進行実況であり、駄文だ。次節の頭が反問・違和感・告白で開けば、予告がなくても読者は続けて読む。

## 列挙の着地

- 性質や分類を列挙したら、列挙しっぱなしにしない。各項目を直前の具体的な場面へ一つずつ着地させる。
- 着地の文体は均一にしない。正体の指摘、身に覚え、固有の事実への対応づけ、未来の断念、と変化をつける。

## 問いの回収と結び

- 途中で立てた問いは、放置せず明示的に回収する。問いへの答えを半分ずつ返すと、後半の推進力になる。
- 結びは、積んだ抽象を、読者がすでに持っている具体（冒頭の場面、読者自身の経験、序盤の問い）へ着地させてから閉じる。抽象論や一般則のまま終えない。
- 緊張は選んで閉じる。最後に一つだけ開いたまま残してよい。読者への委任は、参加余地として機能する。
- 二人称の呼びかけ、読者への依頼、書き手の謙抑や断りは、記事の冒頭・結びなどの境界でだけ緩みとして機能する。中盤の論証に混ぜない。
- 謙抑・断りが、`docs/character.md` の禁じる予防線・謙遜の繰り返しとどう違うかも、ここで決めておく。境界に一度だけ置き、何をどう割り切るかを具体的に指す依頼（「どうか〜と割り切って読んでほしい」）は緩みとして機能する。一方、内容の不正確さへの免責は冒頭の免責表示がすでに担っているから、結びで重ねる謝罪・弁明は予防線であり削る。回収も具体もない定型の締め（「ぜひ参考にしてください」）は、`docs/japanese-ai-slop.md` が定型句として禁じるものであって、ここでいう依頼ではない。

## 緩みと駄文の見分け方

判定の軸は一つだけだ。その文が更新するのは「状況」か、「文書」か。

- 状況を更新する文: 対象世界の出来事・データ・人物の発言、あるいは書き手の判断の状態（思い込み、保留、後悔、譲歩、告白）を新しく伝える。→ 緩みとして残してよい。
- 文書を更新する文: この記事・この節・ここまでの話が「どう見えるか」「次に何を書くか」だけを伝える。→ 原則として削除する。

駄文の典型（いずれも話題が本文自身であり、状況の情報がゼロ）:

- 「ここまでだと、概念の説明に見えるだろう。なので、すぐに例へ戻す」（説明の見え方と執筆の予定）
- 「要するに、この記事の主題は〜ではなく〜である」（性格づけの言い直しだけで、対象の新情報がない）
- 「誤解しないでほしいのだが、〜を否定したいわけではない」（退ける誤読を特定しない弁明）
- 「テクニックの列挙はしない」「〜の話ではない」（本文の性格・範囲の宣言。否定形でも短文でも、話題が本文自身なら駄文）
- 「先に答えを半分だけ置く」「最後にもう一度だけ線を引く」（この規範の装置の実況。装置は内容で実現するもので、操作を宣言しない）
- 「ここまでで〜は見えた。次は〜を見る」（節末の進行予告）

駄文は、長い説明文の形だけでなく、短い断定の形でも現れる。文書更新の文を削除する代わりに短く断定調へ整形すると、拍が効いた決め台詞に見えて残りやすい。これが駄文の最大の混入経路だ。短くてリズムが良いことは、残す理由にならない。拍の良し悪しは、話題テストを通過した文についてだけ評価する。

良い緩みの典型（いずれも状況か、書き手の判断状態を更新している）:

- 「うまくいっているに違いない」（思い込み。あとで崩される布石）
- 「まあ、今すぐ手を打つほどでもないのだけど、どこかの時点で整理は要るだろう」（判断の保留）
- 「最初からわかっていたらそうしていたのに、というのが口惜しい」（判断の誤差を可視化する感情）
- 「そうしたかった、とは思う」（反問への譲歩。直後の転回の足場）

文書について述べる文でも、次の四つの形だけは残してよい。

1. 反論処理: 読者の誤読・反論を「」で具体的に書き出して退ける。退ける対象が具体的に引用されていることが条件。漠然と「誤解しないでほしい」だけの文は駄文。
2. 問いの設置と回収: 境界に置く問いの文（緊張を作ったあとに限る）と、その回収の文。本文が「何でないか」の宣言は問いの設置ではない。
3. 読者への依頼・断り: 境界に置く「どうか〜と割り切って読んでほしい」。
4. 例の枠の開閉: 架空の例の枠を開く文（「〜としよう」）と閉じる文。例が架空であることを読者に思い出させ、抽象的な議論から場面へ戻す機能を持つ。境界に置く。

削除と書き換えの手順:

- 文書を更新するだけの文を見つけたら、まず削除して前後を読み、つながるならそれで終わり。
- 削除で論理が飛ぶ場合は、その文が指そうとしていた内容を、状況側の文に書き換える（「ここまでだと概念の説明に見える」→「この三つの性質は、どれも冒頭の失敗の中にそろっている」）。
- 書き換えた結果の文がまだ本文自身を話題にしているなら、その書き換えは失敗だ。例外の形に収まらない限り、文ごと削除して前後を橋渡しし直す。

## 執筆後の点検手順

草稿を書き上げたら、次の順で機械的に点検する。`docs/japanese-ai-slop.md` の自己チェックとは別に行う。

1. 話題テスト: 段落の頭の文と、独立した短文をすべて拾い、「状況を更新しているか、文書を更新しているか」を判定する。文書側は、例外の形に該当しない限り削除または書き換え。推敲で新しく書いた文・短く割り直した文は、駄文の混入経路なので、書いた直後にもこのテストにかける。
2. 漏出テスト: この規範の語彙（「答えの半分」「緊張」「回収」「線を引く」等）が本文にそのまま現れていないか検索する。現れていれば、装置を宣言してしまった証拠なので、その文を削除し、装置を内容の側で実現し直す。あわせて、節末に「次は〜」型の進行予告がないかを全節末で確認する。
3. 緊張台帳: 本文中で立てた問い・思い込み・約束を列挙し、それぞれの回収位置を行で指す。指せないものは、回収を書き足すか、問いごと削る。あわせて、列挙した思い込み・逡巡が一次資料で裏付けられた実際の判断状態かを確認する。
4. 拍の点検: 長い断定文が 3 つ以上連続している箇所を探し、短い足場か停止、または逡巡を挿む。
5. 境界の点検: 二人称の呼びかけ・依頼・謙抑が本文中盤にないか確認する。あれば境界へ移すか削る。

## 平坦な下書きの診断

レビューで「正しいのに読み進める気がしない」と感じたときは、症状から処方を引く。

- 全段落が同じ調子で疲れる: 文の拍がない。点検手順 4 を適用する。
- 正しいのに読み進める気がしない: 未回収の緊張がない。冒頭に緊張を作る型のどれかを入れ、緊張台帳で以降も常に一つは開いていることを確認する。
- 理論・解説の節で急に温度が下がる: 理論が違和感より先に出ている。理論の前に反問か告白を置き、列挙は場面へ一項目ずつ着地させる。
- 緩い文はあるのに弛んで見える: 緩みが文書更新（進行実況）になっている。話題テストにかけ、状況側の文（感情の微差、判断の保留、思い込み）に書き換える。
- 結びが説教くさい: 抽象論のまま閉じている。読者がすでに持っている具体へ着地させる一文を先頭に置き、未決の問いを一つ残して終える。
- 冒頭が事務的: 態度のない議題表になっている。削るのではなく、予告文に態度を与えるか、予告の前に読者の実感・抵抗の処理を置く。
