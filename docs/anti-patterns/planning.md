# 計画段階のアンチパターン

このチェックリストは、計画・調査・戦略策定のレビュー時に参照し、過去の失敗パターンに陥っていないかを確認するためのものです。

- AP-P01: 計画の根幹にある仮定（検索ボリューム、PV推計、改善効果など）を定量的に実測せず、「実装フェーズで検証する」と先送りしていないか？
  → 前提が崩れると計画全体が無意味になる。（cycle-155, 157, 159, 162, 175, 276で実際に発生）

- AP-P02: 自分が選んだ戦略を否定するデータを積極的に探したか？都合の悪いデータを無視または過小評価していないか？
  → 結論を先に決めてから根拠を集めると、戦略の致命的欠陥が見えなくなる。（cycle-157, 162, 187, 188, 276で実際に発生）

- AP-P03: 今あるコンテンツ、今の設計、今ある状態、今のPVなどを所与として計画を立てていないか？将来追加するコンテンツや拡大するPVなど、今後の展開を見据えた計画になっているか？
  → 現状を固定して計画を立てると、今できることの範囲内に閉じた施策しか出てこず、成長の可能性を過小評価する。（cycle-156, 163, 276, 277で実際に発生）

- AP-P04: Ownerの発言をアドバイス（調査のヒント）として受け取らず、自分で調査・検証せずそのまま計画の前提や結論に組み込んでいないか？
  → Ownerの発言は検討の出発点であり、検証なしに採用するとAP-P01と同じ構造の失敗になる。（cycle-159, 161, 169, 175, 243, 255で実際に発生）

- AP-P05: 前サイクルの失敗を受けて正反対の極端な方針を採用し、constitutionに立ち返らずに前回への反射で方針を決めていないか？
  → 二項対立的な振れは前回とは別の新しい失敗を生む。（cycle-146, 158, 188, 255, 259, 274で実際に発生）

- AP-P06: 既存コンテンツ一覧を確認せず新規作成予定との重複を検証しないまま、またはdocs/research/の既存調査レポートを参照せずに計画を立てていないか？同種の意思決定が過去に行われていた場合、その決定の経緯と根拠を調査したか？
  → 既存コンテンツとの重複は独自価値のない成果物を生み、既存調査の無視は過去の知見と矛盾する計画を生む。過去の意思決定履歴を調査せずに同種の決定を行うと、過去に確立した原則を知らないまま矛盾する判断をしてしまう。（cycle-156, 158, 162, 166, 169, 277で実際に発生）

- AP-P07: 計画の出発点が「来訪者が何を求めているか」ではなく、「運営者が何をしたいか」「実装が容易か」になっていないか？対象ページに誰がどの経路で到達するかを調査したか？UI コンポーネントの選択（タブ vs ボタン vs ラジオ等）を「来訪者の認知モデル」ではなく「実装パターンの参照」で決めていないか？
  → 運営者目線の施策は来訪者の行動と乖離しやすく、来訪経路を誤認すると想定と異なるユーザーに向けた設計になる。UI コンポーネントの選択基準を「視覚パターンの参照」だけで決めると、視覚と挙動と機能の不整合（例: タブ風の見た目だがボタン的なトグル挙動で単一選択のみ）を計画段階で作り込んでしまう。（cycle-113, 139, 169, 175, 181, 184, 247, 253, 277で実際に発生）

- AP-P08: 「ゼロベースで設計する」をbacklogに明示された対象だけに限定し、ページ構成・情報アーキテクチャ・コンポーネント構造など設計全体に適用せず前サイクルのパターンを無意識に踏襲していないか？
  → ゼロベース検討の対象を限定すると、検討範囲外の部分に前回の誤りがそのまま引き継がれる。（cycle-147, 152, 153, 188, 277で実際に発生）

- AP-P09: 計画のゴールが「higher page views by providing the best value for visitors」に正確に対応せず、「Googleの評価向上」「SEOスコア改善」「ドメイン権威向上」など別のゴールに読み替えていないか？
  → ゴールを読み替えると、来訪者への価値を犠牲にしてSEO最適化を優先するなどconstitutionと逆行する判断が生まれる。（cycle-159, 187, 247, 255で実際に発生）

- AP-P10: 「適合度5/5」「高い確信度」等の高評価が出たとき、過去の実績データと矛盾していないか確認せず根拠なき高評価を無批判に採用していないか？
  → 過去の実績と矛盾する高評価はバイアスの産物であり、そのまま採用するとAP-P01と同じ構造の失敗になる。（cycle-157, 188で実際に発生）

- AP-P11: 「前サイクルでAIがこう決めたから」という理由で変更を回避していないか？AIが決めた色・テキスト・レイアウト等が変更可能であることを認識しているか？
  → AIの過去の判断を「変更不可の制約」として扱うと、より良い選択肢を採用できなくなる。（cycle-151, 152, 153, 187, 188, 255で実際に発生）

- AP-P12: 過去に同種の施策が失敗していた場合、その失敗を具体的に分析したか？「なぜ失敗したか」「今回は何が違うか」を明確にしないまま同じ方向の施策を計画していないか？
  → 失敗分析なしに同種施策を再実行すると、同じ失敗を繰り返す。（cycle-163で実際に発生）

- AP-P13: 理論的なフレームワーク（成長モデル、ループ理論等）を先に採用し、そこに施策を当てはめて優先順位を決めていないか？需要×競合の実測データに基づく評価より、枠組みへの適合度を優先していないか？
  → フレームワーク先行で計画すると、枠組みに合わない有望な候補が排除され、枠組みに合うが需要が乏しい候補が優先される。（cycle-165, 260で実際に発生）

- AP-P14: 調査対象のカテゴリを恣意的に限定していないか？「まだ調べていない候補は何か」を調査設計の段階で確認したか？前回の調査範囲をそのまま引き継いでいないか？
  → 調査範囲の限定は有望な候補の見落としを引き起こす。特に、採用したフレームワークに合う候補だけを調査し、合わない候補を除外するパターンや、前回の調査で見つけた候補だけを引き継いで他を探さないパターンに注意。（cycle-163, 165, 175, 277で実際に発生）

- AP-P15: 直近の成功体験に引きずられて、同種の施策を他の候補より不当に優遇していないか？同じ手法で評価可能な全ての候補と公平に比較したか？
  → 成功体験への親近感は、客観的なデータより感覚的な確信を優先させる。ある手法で1本成功したからといって、同じ手法の別候補を他のカテゴリの候補より優遇する根拠にはならない。（cycle-165で実際に発生）

- AP-P16: 計画書のタスク列挙で「達成された」「完了している」「着手条件が揃った」と書く前に、依拠する一次情報（他タスクの状態 / ファイルの存在 / 数値 / 着手条件達成状況）を `Read`・`grep` 等で実体確認したか？複合条件の着手条件（例: 「A 完了 + B 完了」）では、両方の項目を独立に確認したか？数値 literal にラベル（実測値 / 仕様値 / 実装値 / 推定値）と生成元を併記したか？
  → 前提条件を確認せずタスクを実施可能と書くと、未達の前提に基づくタスクが計画書に混入し後で整合性破綻が発覚する。生成元なしのラベルは後続サイクルが典拠を辿れず再現性を失う。（cycle-174, 184, 188, 210, 212 で実際に発生）

- AP-P17: 対応方針が複数考えられる課題に対し、計画段階で **3 案以上をゼロベースで列挙し比較表を作ったか**。最初に思いついた案だけを単独で実装計画化していないか？
  → 単案で実装計画を組むと「なぜ他案ではないのか」が言語化されず、レビューが対症的指摘の応酬になりラウンドが伸びる。（cycle-183で実際に発生）

- AP-P18: Owner / reviewer の指摘を計画に反映する際、文言だけを拾わず「**指摘の背後にある問いの構造**」を 1〜2 文で言語化したか？指摘 1 件に対し対症療法案だけで返答していないか？
  → 指摘の文言だけを拾って対症療法を返すと、複数指摘の根底にある共通の問いに到達できない。（cycle-183, 274で実際に発生）

- AP-P19: 過去サイクルで確定済みの判断根拠の中に **外部仕様（SEO 機能・ブラウザ API・Schema.org 仕様・サードパーティプラットフォーム機能の存続など）** への依存が含まれている場合、本サイクルで実装着手する前にその外部仕様の現在状態を **一次資料（公式ドキュメント・公式ブログ・仕様書）で WebFetch / WebSearch 確認** したか？過去サイクルで「条件付き便益」「将来 X が実装されれば」「外部仕様 Y に依拠」と記述された箇所を本サイクルで再利用するとき、自動的に再点検対象に含めたか？
  → 過去サイクルで確定した判断であっても、その根拠が外部当事者が変更可能な仕様に依存している場合は「固定された事実」ではない。外部仕様は廃止・改名・仕様変更される。（cycle-186で実際に発生）

- AP-P20: 過度に具体的な計画を立てていないか？ コードレベルの実装方法や細かい作業手順など、実装段階で検討すべきことまで計画に含めていないか？
  → `/cycle-planning` フェーズは基本計画の策定とされている。計画段階では「なぜやるのか」「何をやるのか」や「どうなったら完了なのか」だけを明確にし、「どうやってやるのか」は実装段階に委ねるべき。過度に詳細な計画を立てると実態とのズレが生じやすく、また机上の空論で来訪者にとっての価値を損ねやすい。（cycle-188で実際に発生）

- AP-P21: 固定枠 UI（タイル / カード / 限定領域）の中に「サイズが動的に膨張する側」と「ユーザー操作上の最低高さを保つべき側」が同居していないかを検討したか？膨張側が伸びる結果、操作側が圧迫されて機能不全になる構造リスクを計画段階で評価したか？参考実装（既存タイル等）が「両方とも固定サイズ」だったために問題化していないだけで、本サイクルの対象は「片方が膨張する性質」を持つかもしれない、という観点で参考実装の前提と本対象の差を言語化したか？
  → 固定枠で膨張側と操作側を同列に扱うと操作側が圧迫され機能不全になりうる。参考実装が結果固定型だった場合、その前提を膨張型の新対象にそのまま転用すると本パターンに陥る。（cycle-202 で実際に発生）

- AP-P22: 計画書冒頭の目的記述は、その**派生元**（backlog アイテム本文／前サイクルでの議論・キャリーオーバー／原典の問題提起）に遡って整合確認したか？ 派生元では「恒久基準（DESIGN.md・constitution.md 等）への準拠の是正」だったのに、計画書では「部品の再実装」「一括削除」「移行作業」のような**派生作業**だけが目的として書かれていないか？
  → 目的が派生作業にすり替わったまま planning が進むと、その後の指示・builder・reviewer のすべてが派生作業の達成を最大化する方向に進み、本来の目的が指示全体から消失する。reviewer が AP として発火させるためには、計画書の目的記述だけを受け取らず、派生元のドキュメントに遡って突合することが要件になる。（cycle-220, 225, 255 で実際に発生）

- AP-P23: 分析データの期間集計値（PV・クリック数等）と「対象の現在の状態」を組み合わせて実害や因果（「404 に落ちている」「壊れたページが見られている」等）を主張するとき、**イベントの発生日時点で対象がどの状態だったか**（公開中か削除後か・修正前か修正後か）を、発生日・page_title 等のディメンションまで掘って確認したか？
  → 期間集計値は「いつ起きたか」を隠す。期間の途中で対象の状態が変わった場合（記事の削除・バグの修正・ページの移転）、集計値を「現在の状態で起きたこと」として読むと存在しない実害を起票したり、解消済みの問題を未解消と誤認したりする。実害の主張には「イベント発生時点の対象の状態」の確認を伴わせる。（cycle-232, 277 で実際に発生）

- AP-P24: 本サイクルで触る領域（同じディレクトリ・同じファイル系・同じデータ・同じユーザー導線）について、過去サイクルでその領域に確立された戦略・意図・設計判断を未確認のまま、本サイクルの新規変更で無意識に打ち消していないか？
  → 同一領域での過去戦略を確認せずに新規変更を計画すると、過去サイクルが意図して確立した独自性・UX・コンテンツ戦略を新変更が無意識に破壊する。過去サイクルが確立した戦略も、外部仕様と同様に確認すべき一次情報である。特に来訪者価値や独自性に関わる戦略は、構造化データ・メタ情報・description のような「Google や来訪者から見える層」の変更で容易に打ち消されうる。（cycle-246, 259 で実際に発生）

- AP-P25: 選択肢を検討する際、仕様適合・スパム回避・過去戦略整合・実装コストなどの分析を始める前に、その選択肢が constitution Goal（「higher page views for the website by providing the best value for visitors」）と整合するかを最初に問うたか？ constitution Goal に正面から矛盾する選択肢を技術的分析の俎上に乗せていないか？
  → 選択肢の検討で構造化データ・メタ情報・コンテンツ表示・SEO 周辺の判断をするとき、PM はつい仕様適合性（schema.org / W3C / RFC 等の継承関係や定義）や Google ポリシー抵触リスクの分析から入りがちだが、これらは技術的に正しい分析であっても constitution Goal を起点にしていない。Goal を起点にしていれば検討の俎上にすら上がらない選択肢を、技術的分析で真剣に積み上げてしまう壮大な無駄を予防する。（cycle-246, 253, 255, 274 で実際に発生）

- AP-P26: 小サンプルの GA / アクセス解析イベントを「人間の行動」や「需要のシグナル」として解釈する前に、それが能動的な人間の閲覧である証拠を確認したか？ 具体的には、当該セッションに `user_engagement` / `engagement_time_msec`（滞在・操作の計測）が存在するか、サイト本来の操作イベント（クイズ開始・クリック等）を伴っているか、流入経路・着地 URL（内部 ID を人間が知り得るか・sitemap 経由で機械が発見可能か）、既知ボット除外を通過しただけで未知ボット／プリフェッチ／プレビュー生成は残る点、を点検したか？
  → engagement 計測や操作イベントを伴わない page_view は、プリフェッチ・プレビュー生成・自動巡回など「能動的な人間ではない」自動アクセスであることが多い。これを「ユーザーがわざわざやった行動＝需要」と読むと、虚偽の前提の上に scope 判断やコンテンツ（ブログ等）を積み上げてしまう。母数が小さいときほど 1 件の解釈が重くなるため、人間/ボットの判別を経てから「需要」と呼ぶこと。（cycle-249 で実際に発生）

- AP-P27: 進行中の作業（複数サイクルにまたがる移行・改善計画など）が未完了のまま来訪者に実害（デザイン断裂・機能の不整合・壊れた導線など）を与えている状態で、その作業を放置して別の新規タスクに着手していないか？
  → 途中まで進めた作業を放置すると、完了部分と未完了部分の不整合が来訪者にそのまま見える。新しいことに挑戦すること自体は正しいが、未完了の作業が実害を生んでいる場合はそちらの完了が先。（cycle-252, 267, 269, 276 で実際に発生）

- AP-P28: 大掛かりな質的変更（デザインシステム移行・コンセプト刷新・UX 再設計など）を計画するとき、実装コスト・規模・リスクを理由に「本質的な作り直し」を回避する代替手段（トークン互換レイヤー／色エイリアス／旧 UI の構造を温存したままの上塗り／"とりあえず動く" 暫定実装など）を採用していないか？ その手段が来訪者に届ける成果物を「本来あるべき品質より劣るもの」にしていないか？ 計画段階で「これは工数を理由に UX を妥協していないか」を明示的に自問したか？
  → デザインシステム移行は、カラーパレット変更やコンポーネント差し替えのような単純作業ではなく、サイトコンセプトの根本とサイト全体に共通するメタファーを丸ごと入れ替える質的変更である。「トークンを新値にエイリアスすれば色だけ新しくなって楽」「共有コンポーネントが多くて危険だから既存構造を温存しよう」という判断は、技術的には合理に見えても、CLAUDE.md 意思決定原則「実装コスト（時間・ファイル数・変更の複雑さ）は UX で劣る方式を選ぶ理由には決してならない／より良い UX が達成可能ならそれを選ばねばならない」と constitution Goal（来訪者価値の最大化）に正面から反する。色のみの上塗りは移行計画の標準手順（トークン置換だけでは新デザインにならない・必要なら構造そのものを変える）にすら反する。PM 自身が計画段階でこの緊張（規模・リスク vs 本質的品質）に気づきながら規模・リスクを理由に握りつぶすのが典型的な発生形。リスクや規模が大きいことは「品質を下げる理由」ではなく「サイクルを適切に分割して本質的にやり切る理由」である。（cycle-253, 255, 259, 267, 269, 276 で実際に発生）

- AP-P29: 計画で「先例（過去サイクルで同種の作業をしたページ・部品）と同型だから、この要素は保全で済む／触らなくてよい／追加は不要」と判断するとき、その先例が実際にどういう構造になっているかを実コードで確認したか？ 先例の構造を確認せず「同型のはず」という想像だけでスコープを確定していないか？
  → 「この作業は先例の踏襲だから安全」という判断は効率的だが、踏襲元の実構造を確認せずに一般化すると、先例には在って当該対象には無い（または逆）要素を見落とし、スコープを誤る。特に「保全（現状維持）で済む」という結論は、先例も当該対象も同じ状態であることを暗黙の前提にしており、その前提自体が未確認なら結論は崩れる。先例が満たしている設計契約・構造（見出し階層・必須要素・共通レイアウトの分担）を当該対象が満たしていない場合、それは「移行と独立な別課題」ではなく「移行が揃えるべき不統一そのもの」であることが多い。「同型だから」と書く前に、同型と主張する先例を Read/grep で実体確認すること。（cycle-266, 267 で実際に発生）

- AP-P30: 既存の仕様・スコープ・実装・運用を判断の前提にするとき、それが過去に**原理（constitution Goal・来訪者価値）に基づいて明示的に決定された**ものか、実装の流れで**偶発的にそうなっただけ**かを区別したか？「今そうなっているから正しい／このままでよい」と現状を追認していないか？
  → 偶発的な現状（例: ある機能が特定領域だけに実装されているのは、その領域の作業中にたまたま必要性に気づいたからで、サイト全体の方針判断ではない）は「正しさ」の根拠にならない。現状が原理から導かれたものか偶発の産物かを確かめず「現状こうなっている」を肯定の理由に使うと、誤った現状を温存・拡大してしまう。正しくない可能性を常に開いておき、唯一の判断基準（来訪者により高い価値を提供するか）で問い直すこと。「現状追認」は判断ではなく判断の放棄である。（cycle-268, 277 で実際に発生）

- AP-P31: 低トラフィック下での A/B（同時期に来た来訪者へ複数パターンを提示し、一人ひとりの反応を質的に観察する手段）を、統計的検出力・サンプル不足・ノイズを理由に却下していないか？ また時間軸上の before/after（リリース前後）比較を A/B の代替として扱っていないか？
  → 小規模サイトでは Google のインデックス／評価変動が時間軸の流入を支配し、前後比較ではデザイン等の効果を分離できない（前後比較は成立しない）。規模が小さいことは「前後比較で代替する理由」ではなく、むしろ「一人ひとりの行動を丁寧に観る必要がある理由」であり、A/B はそのための手段。A/B を統計的有意性の道具と誤認して「サンプルが足りないから無意味」と却下したり、前後比較を A/B の代わりに据えたりしない。トラフィックの小ささは A/B 却下の理由にならない（運用の正典は `docs/ADR/README.md`）。逆に、**検証手段の理想形（例: サイト全面を貫く1本の A/B）が、破棄の決まった系統の温存＝二重保守を前提とする場合、その保守の代価と品質害を無視して理想形を「唯一の正解」と規定しない**——検証は独立変数を閉じられる最小の面で設計する（本項の旧末尾規定「全トラフィックを1本の全面 A/B に集約せよ」は cycle-278 でこの理由により改訂。経緯は cycle-278.md / `docs/rebuild-plan.md` §1）。（cycle-268, 269, 274 で実際に発生）

- AP-P32: 判断・設計・回答の根拠に使う内部の規範・原則・語彙（コンセプト文・デザイン原則・戦略規範・比喩）について、それぞれが「観測された事実」か「検証済みの結論」か「未検証の仮説」かの地位と、出自（いつ・何を根拠に作られたか）を確認したか？ 仮説の上に仮説を重ねるとき、自分がどの層の上に立っているかの地図を持っているか？ 上位の未検証仮説を、下位の規則や設計の検証基準に使っていないか？
  → 内部文書の語彙は数サイクルで「所与の真実」の顔をする。地位と出自を確認せずに引用すると、未検証仮説の堆積が岩盤のように振る舞い、検証の基準そのものが仮説に汚染される。岩盤として扱ってよいのは constitution と生の観測データだけで、その間にある層はすべて改訂可能な戦略の堆積である。（cycle-277で実際に発生）
