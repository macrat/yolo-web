# 計画段階のアンチパターン

このチェックリストは、計画・調査・戦略策定のレビュー時に参照し、過去の失敗パターンに陥っていないかを確認するためのものです。

- AP-P01: 計画の根幹にある仮定（検索ボリューム、PV推計、改善効果など）を定量的に実測せず、「実装フェーズで検証する」と先送りしていないか？
  → 前提が崩れると計画全体が無意味になる。（cycle-155, 157, 159, 162, 175で実際に発生）

- AP-P02: 自分が選んだ戦略を否定するデータを積極的に探したか？都合の悪いデータを無視または過小評価していないか？
  → 結論を先に決めてから根拠を集めると、戦略の致命的欠陥が見えなくなる。（cycle-157, 162, 187, 188で実際に発生）

- AP-P03: 今あるコンテンツ、今の設計、今ある状態、今のPVなどを所与として計画を立てていないか？将来追加するコンテンツや拡大するPVなど、今後の展開を見据えた計画になっているか？
  → 現状を固定して計画を立てると、今できることの範囲内に閉じた施策しか出てこず、成長の可能性を過小評価する。（cycle-156, 163で実際に発生）

- AP-P04: Ownerの発言をアドバイス（調査のヒント）として受け取らず、自分で調査・検証せずそのまま計画の前提や結論に組み込んでいないか？
  → Ownerの発言は検討の出発点であり、検証なしに採用するとAP-P01と同じ構造の失敗になる。（cycle-159, 161, 169, 175, 243で実際に発生）

- AP-P05: 前サイクルの失敗を受けて正反対の極端な方針を採用し、constitutionに立ち返らずに前回への反射で方針を決めていないか？
  → 二項対立的な振れは前回とは別の新しい失敗を生む。（cycle-146, 158, 188で実際に発生）

- AP-P06: 既存コンテンツ一覧を確認せず新規作成予定との重複を検証しないまま、またはdocs/research/の既存調査レポートを参照せずに計画を立てていないか？同種の意思決定が過去に行われていた場合、その決定の経緯と根拠を調査したか？
  → 既存コンテンツとの重複は独自価値のない成果物を生み、既存調査の無視は過去の知見と矛盾する計画を生む。過去の意思決定履歴を調査せずに同種の決定を行うと、過去に確立した原則を知らないまま矛盾する判断をしてしまう。（cycle-156, 158, 162, 166, 169で実際に発生）

- AP-P07: 計画の出発点が「来訪者が何を求めているか」ではなく、「運営者が何をしたいか」「実装が容易か」になっていないか？対象ページに誰がどの経路で到達するかを調査したか？UI コンポーネントの選択（タブ vs ボタン vs ラジオ等）を「来訪者の認知モデル」ではなく「実装パターンの参照」で決めていないか？
  → 運営者目線の施策は来訪者の行動と乖離しやすく、来訪経路を誤認すると想定と異なるユーザーに向けた設計になる。UI コンポーネントの選択基準を「視覚パターンの参照」だけで決めると、視覚と挙動と機能の不整合（例: タブ風の見た目だがボタン的なトグル挙動で単一選択のみ）を計画段階で作り込んでしまう。（cycle-113, 139, 169, 175, 181, 184で実際に発生）

- AP-P08: 「ゼロベースで設計する」をbacklogに明示された対象だけに限定し、ページ構成・情報アーキテクチャ・コンポーネント構造など設計全体に適用せず前サイクルのパターンを無意識に踏襲していないか？
  → ゼロベース検討の対象を限定すると、検討範囲外の部分に前回の誤りがそのまま引き継がれる。（cycle-147, 152, 153, 188で実際に発生）

- AP-P09: 計画のゴールが「higher page views by providing the best value for visitors」に正確に対応せず、「Googleの評価向上」「SEOスコア改善」「ドメイン権威向上」など別のゴールに読み替えていないか？
  → ゴールを読み替えると、来訪者への価値を犠牲にしてSEO最適化を優先するなどconstitutionと逆行する判断が生まれる。（cycle-159, 187で実際に発生）

- AP-P10: 「適合度5/5」「高い確信度」等の高評価が出たとき、過去の実績データと矛盾していないか確認せず根拠なき高評価を無批判に採用していないか？
  → 過去の実績と矛盾する高評価はバイアスの産物であり、そのまま採用するとAP-P01と同じ構造の失敗になる。（cycle-157, 188で実際に発生）

- AP-P11: 「前サイクルでAIがこう決めたから」という理由で変更を回避していないか？AIが決めた色・テキスト・レイアウト等が変更可能であることを認識しているか？
  → AIの過去の判断を「変更不可の制約」として扱うと、より良い選択肢を採用できなくなる。（cycle-151, 152, 153, 187, 188で実際に発生）

- AP-P12: 過去に同種の施策が失敗していた場合、その失敗を具体的に分析したか？「なぜ失敗したか」「今回は何が違うか」を明確にしないまま同じ方向の施策を計画していないか？
  → 失敗分析なしに同種施策を再実行すると、同じ失敗を繰り返す。（cycle-163で実際に発生）

- AP-P13: 理論的なフレームワーク（成長モデル、ループ理論等）を先に採用し、そこに施策を当てはめて優先順位を決めていないか？需要×競合の実測データに基づく評価より、枠組みへの適合度を優先していないか？
  → フレームワーク先行で計画すると、枠組みに合わない有望な候補が排除され、枠組みに合うが需要が乏しい候補が優先される。フレームワークは十分な実績データが蓄積されてから適用すべき。（cycle-165で実際に発生）

- AP-P14: 調査対象のカテゴリを恣意的に限定していないか？「まだ調べていない候補は何か」を調査設計の段階で確認したか？前回の調査範囲をそのまま引き継いでいないか？
  → 調査範囲の限定は有望な候補の見落としを引き起こす。特に、採用したフレームワークに合う候補だけを調査し、合わない候補を除外するパターンや、前回の調査で見つけた候補だけを引き継いで他を探さないパターンに注意。（cycle-163, 165, 175で実際に発生）

- AP-P15: 直近の成功体験に引きずられて、同種の施策を他の候補より不当に優遇していないか？同じ手法で評価可能な全ての候補と公平に比較したか？
  → 成功体験への親近感は、客観的なデータより感覚的な確信を優先させる。ある手法で1本成功したからといって、同じ手法の別候補を他のカテゴリの候補より優遇する根拠にはならない。（cycle-165で実際に発生）

- AP-P16: 計画書のタスク列挙で「達成された」「完了している」「着手条件が揃った」と書く前に、依拠する一次情報（他タスクの状態 / ファイルの存在 / 数値 / 着手条件達成状況）を `Read`・`grep` 等で実体確認したか？複合条件の着手条件（例: 「A 完了 + B 完了」）では、両方の項目を独立に確認したか？数値 literal にラベル（実測値 / 仕様値 / 実装値 / 推定値・経験的暫定値）を付けるとき、ラベル種別と生成元（実測値ならコマンド/JSON キー、実装値ならファイル+行番号）が literal の直近に併記されているか？
  → AP-WF12（workflow.md）が「PM/planner の事実情報の自ら確認」をレビュー時に外部から発火させるチェックなのに対し、本項目は **計画書執筆中に当事者が自分で発火させるチェック**。タスク列挙時に前提条件を確認しないと、未達の前提に基づくタスクを「実施可能」として計画書に組み込んでしまい、cycle-execution での反映後にようやく整合性破綻が発覚する。複合条件の片側だけを確認して満足する誤りに特に注意（A は確認したが B は未確認のまま「両方達成された」と書く）。**ラベル付き数値 literal は「実機計測で取得した実測値」と「コード参照で取得した実装値（CSS ハードコード / TypeScript 定数）」を運用上明確に区別する**。生成元なしの実測値・実装値ラベルは禁止（後続サイクルが引用したとき典拠を辿れず再現性が失われる）。本項目を AP-WF12 と相互補完させることで、計画書執筆時とレビュー時の両方から呼び出される二重化を確保する。（cycle-174, 184, 188, 210, 212 で実際に発生）

- AP-P17: 対応方針が複数考えられる課題に対し、計画段階で **3 案以上をゼロベースで列挙し比較表を作ったか**。最初に思いついた案だけを単独で実装計画化していないか？
  → 単案で実装計画を組むと「なぜ他案ではないのか」が言語化されず、レビューが対症的指摘の応酬になりラウンドが伸びる。（cycle-183で実際に発生）

- AP-P18: Owner / reviewer の指摘を計画に反映する際、文言だけを拾わず「**指摘の背後にある問いの構造**」を 1〜2 文で言語化したか？指摘 1 件に対し対症療法案だけで返答していないか？
  → 指摘の文言だけを拾って対症療法を返すと、複数指摘の根底にある共通の問いに到達できない。（cycle-183で実際に発生）

- AP-P19: 過去サイクルで確定済みの判断根拠の中に **外部仕様（SEO 機能・ブラウザ API・Schema.org 仕様・サードパーティプラットフォーム機能の存続など）** への依存が含まれている場合、本サイクルで実装着手する前にその外部仕様の現在状態を **一次資料（公式ドキュメント・公式ブログ・仕様書）で WebFetch / WebSearch 確認** したか？過去サイクルで「条件付き便益」「将来 X が実装されれば」「外部仕様 Y に依拠」と記述された箇所を本サイクルで再利用するとき、自動的に再点検対象に含めたか？
  → 過去サイクルで確定した判断であっても、その根拠が外部当事者が変更可能な仕様に依存している場合は「固定された事実」ではない。外部仕様は廃止・改名・仕様変更される。本サイクルで参照する計画書に登場する外部仕様は、計画 r1 立案開始前に一次資料を WebFetch で取得して現在状態を確認することを計画立案フローの最初に組み込む。確認した URL と日付は計画書の「参考にした情報」セクションに明記する。
  **事例**: cycle-186 で、cycle-174 の ① 採用根拠 A-13（Google SearchAction sitelinks searchbox SEO 便益）が 2024-11-21 に廃止済みであることを、計画 r1〜r4 まで Owner / PM / planner / reviewer 全員が見落とした。発見契機は cycle-186 r2 reviewer が `https://developers.google.com/search/blog/2024/10/sitelinks-search-box` を WebFetch して Critical 指摘したこと。（cycle-186で実際に発生）

- AP-P20: 過度に具体的な計画を立てていないか？ コードレベルの実装方法や細かい作業手順など、実装段階で検討すべきことまで計画に含めていないか？
  → `/cycle-planning` フェーズは基本計画の策定とされている。計画段階では「なぜやるのか」「何をやるのか」や「どうなったら完了なのか」だけを明確にし、「どうやってやるのか」は実装段階に委ねるべき。過度に詳細な計画を立てると実態とのズレが生じやすく、また机上の空論で来訪者にとっての価値を損ねやすい。（cycle-188で実際に発生）

- AP-P21: 固定枠 UI（タイル / カード / 限定領域）で flex container を使う計画を立てるとき、**枠内の子要素のうち「サイズが動的に膨張する側」と「ユーザー操作上の最低高さを保つべき側」が同居していないか**を検討したか？膨張側が flex で伸びる結果、操作側が圧迫されて機能不全になる構造リスクを計画段階で評価したか？参考実装（既存タイル等）が「両方とも固定サイズ」だったために問題化していないだけで、本サイクルの対象は「片方が膨張する性質」を持つかもしれない、という観点で参考実装の前提と本対象の差を言語化したか？
  → 固定枠 UI で `flex: 1` を複数子要素に並列適用すると、結果欄が動的に伸びる場合に入力欄が縮んで編集不能になる等の UX 破綻が起きる。`overflow: auto` で「結果欄を枠内に収める」要件は「入力欄を圧迫しない」要件と flex container 内では両立しない場合がある。役割で分担する設計（例: 操作側 `flexShrink: 0`、膨張側 `flex: 1 + overflowY: auto`）を計画段階で意識する。参考実装が結果固定型だった場合、その前提を新対象（結果膨張型）にそのまま転用すると本パターンに陥る。（cycle-202 で実際に発生）

- AP-P22: 計画書冒頭の目的記述は、その**派生元**（backlog アイテム本文／前サイクルでの議論・キャリーオーバー／原典の問題提起）に遡って整合確認したか？ 派生元では「恒久基準（DESIGN.md・constitution.md 等）への準拠の是正」だったのに、計画書では「部品の再実装」「一括削除」「移行作業」のような**派生作業**だけが目的として書かれていないか？
  → 目的が派生作業にすり替わったまま planning が進むと、その後の指示・builder・reviewer のすべてが派生作業の達成を最大化する方向に進み、本来の目的が指示全体から消失する。reviewer が AP として発火させるためには、計画書の目的記述だけを受け取らず、派生元のドキュメントに遡って突合することが要件になる。（cycle-220, 225 で実際に発生）

- AP-P23: 分析データの期間集計値（PV・クリック数等）と「対象の現在の状態」を組み合わせて実害や因果（「404 に落ちている」「壊れたページが見られている」等）を主張するとき、**イベントの発生日時点で対象がどの状態だったか**（公開中か削除後か・修正前か修正後か）を、発生日・page_title 等のディメンションまで掘って確認したか？
  → 期間集計値は「いつ起きたか」を隠す。期間の途中で対象の状態が変わった場合（記事の削除・バグの修正・ページの移転）、集計値を「現在の状態で起きたこと」として読むと存在しない実害を起票したり、解消済みの問題を未解消と誤認したりする。実害の主張には「イベント発生時点の対象の状態」の確認を伴わせる。（cycle-232 で実際に発生）

- AP-P24: 本サイクルで触る領域（同じディレクトリ・同じファイル系・同じデータ・同じユーザー導線）について、過去サイクルでその領域に確立された戦略・意図・設計判断を未確認のまま、本サイクルの新規変更で無意識に打ち消していないか？
  → 同一領域での過去戦略を確認せずに新規変更を計画すると、過去サイクルが意図して確立した独自性・UX・コンテンツ戦略を新変更が無意識に破壊する。AP-P19（外部仕様の一次確認）と並列で「過去の自分（前任 PM）の確立した戦略」も一次資料として確認対象に含める。特に来訪者価値や独自性に関わる戦略は、構造化データ・メタ情報・description のような「Google や来訪者から見える層」の変更で容易に打ち消されうる。（cycle-246 で実際に発生）

- AP-P25: 選択肢を検討する際、仕様適合・スパム回避・過去戦略整合・実装コストなどの分析を始める前に、その選択肢が constitution Goal（「higher page views for the website by providing the best value for visitors」）と整合するかを最初に問うたか？ constitution Goal に正面から矛盾する選択肢を技術的分析の俎上に乗せていないか？
  → 選択肢の検討で構造化データ・メタ情報・コンテンツ表示・SEO 周辺の判断をするとき、PM はつい仕様適合性（schema.org / W3C / RFC 等の継承関係や定義）や Google ポリシー抵触リスクの分析から入りがちだが、これらは技術的に正しい分析であっても constitution Goal を起点にしていない。Goal を起点にしていれば検討の俎上にすら上がらない選択肢を、技術的分析で真剣に積み上げてしまう壮大な無駄を予防する。AP-P24（過去サイクル戦略確認）も AP-P19（外部仕様の一次確認）も、本項目で Goal 照合を通過した選択肢に対してのみ意味を持つ補助軸である。（cycle-246 で実際に発生）
