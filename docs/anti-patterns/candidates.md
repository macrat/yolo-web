# アンチパターン候補メモ

正式な AP リスト本体（planning.md / implementation.md / writing.md / workflow.md）とは別扱いのメモ置き場です。N≥3 件発生時に正式 AP として昇格を検討してください。取り下げた候補の記録も、同じ誤診を繰り返さないためにここへ残します。

このファイルは通常のレビューで読み込む必要はありません（候補の追記・昇格判断のときだけ開く）。

## AP-WF17 候補: 計画書ファイル肥大化によるレビュー全文通読困難

- **概要**: 計画書ファイルが Read 制限超過サイズ（例: 1181 行 / 264KB）まで肥大化し、reviewer がレビュー全文通読困難になった事例（cycle-214 / 1 件目・cycle-215 = 553.9KB / 2718 行 / 2 件目）。
- **発生件数**: N=2（cycle-214, 215）/ N≥3 で AP-WF17 として新設検討。
- **対症療法案**: サイクルドキュメントのディレクトリ形式（`docs/cycles/cycle-XXX/index.md` + 補助ファイル分割。正本は `.claude/rules/doc-directory.md`）により、レビュー経過ログ等は index.md から分離できるようになった。index.md は計画・チェックリスト・完了サマリに絞る。

## AP-WF18 候補: サブエージェントが特定ツールの呼び出しでハングし長時間停止していないか

- **概要**: サブエージェント（reviewer / web-researcher 等）が特定ツールの応答待ちでフリーズし、PM が完了通知を待ち続けて長時間（最大で約 1〜2 日）気付かず停止していた事例。背景監視の盲点 = 「完了通知が来ない＝作業中」と解釈してしまい、ハングと進行中を区別できない。
  - **cycle-218 事例**: 調査サブエージェント（web-researcher）が WebFetch で取得コンテンツを prompt 要約させる呼び出しでハング（最大約 1 日）。
  - **cycle-227 事例**: サイクル全体の最終検証 reviewer が Playwright（`browser_evaluate` / `browser_wait_for`）で**無限待機する JS** を実行してフリーズし、約 2 日停止（Owner が停止）。完了通知が出ても過程が破綻していれば結果は信用できない。安全な書き方は `docs/knowledge/playwright-mcp.md` に記録。
- **発生件数**: N=2（cycle-218, 227）/ N≥3 で AP-WF18 として新設検討。
- **対症療法案**: (1) 長時間（目安: 通常完了時間を大きく超過）完了通知が来ないバックグラウンドエージェントは、進行中とハングを区別するために状態（出力ファイル）を確認し、必要なら停止して再実行する。「通知が来ない＝作業中」と解釈し続けない。(2) ハングしうるツール呼び出しを避ける = 外部 URL は WebFetch の prompt 要約に依存せず WebSearch / Playwright（navigate → snapshot）へ。Playwright の `browser_evaluate` には**解決しない Promise／無限ループ**を渡さず、待ちが要るときは有限タイムアウトか「操作→別 evaluate で結果読み」の2ステップにする。(3) サブエージェントへの指示文に「無限待機する JS / タイムアウト無しの待ちを書かない」を明示する。ツール挙動の詳細知見は `docs/knowledge/playwright-mcp.md`。

## AP-WF19 候補: 計測値 SSoT を「実測値」として記録する際、生成元スクリプトの実出力（生成物のキー・値）と照合せず確定していないか

- **概要**: AP-P21 計測等の SSoT を §補足事項に書き戻す際、「実測値 / 生成元 = <計測スクリプト>」と記載しながら、その生成元スクリプトが当該値を実際には出力していなかった事例。生成元を書いても「その生成元が当該値を吐くこと」を確認していないため、AP-WF12（実体確認）/ AP-P16（生成元併記）の射程をすり抜ける。算術的に自己整合した手計算値を「実測値」と誤記録すると、後続サイクルが引用する基準値の信頼性が崩れる。（cycle-218, 219 で実際に発生）
- **発生件数**: N=2（cycle-218, 219）/ N≥3 で AP-WF19 として新設検討。
- **対症療法案**: 計測値を「実測値 / 生成元 = <script>」と記録する前に、生成物（results.json 等）に当該キー・値が実在することを grep で確認し SSoT の数値と桁まで一致させる。スクリプト修正後は必ず再実行して生成物を更新してから SSoT へ書き戻す。

## AP-WF20 候補: 修正後の再レビューを既存レビュアーの流用で行い、白紙からの全体見直しになっていない

- **概要**: 修正後の再レビューを、そのタスクを前にレビューした既存レビュアーに SendMessage で継続依頼して行うと、レビュアーは自分の前回指摘が直ったかの確認に視野が狭まり、全体を白紙から見直すバイアスがかかる。AP-WF01「最後の修正後にゼロからすべてを見直す」の実効には、再レビューのたびに新規レビュアーをゼロから起動し、前回の指摘・経緯を渡さず白紙で全体を見直させる必要がある。
  - **cycle-243 事例**: 流用での再レビューが見逃した記事中の数値の事実誤り（「7ページで6回」＝実際は個別7ページ5回＋一覧1回）を、新規レビュアーが白紙のレビューで検出した。
- **発生件数**: N=1（cycle-243）/ N≥3 で AP-WF20 として新設検討。
- **対症療法案**: 再レビューは既存レビュアーの SendMessage 継続でなく新規エージェントを起動し、対象ファイルと観点のみを渡して（前回指摘は渡さず）白紙で全体を見直させる。
- **補足（技術的強制）**: SendMessage 経由の再レビュー依頼は `.claude/hooks/block-review-reuse.sh` が機械的にブロックするようになった。以後の発生は「フックをすり抜けた形態」として記録する。

## ~~AP-WF21 候補~~: 取り下げ（cycle-246 で実測再現せず・誤診と確定）

- **取り下げ理由（cycle-246, 2026-06-15 実測検証）**: 「PostToolUse の `prettier --write <絶対パス>` が `.prettierignore` を貫通する」という前提は、現環境（prettier 3.8.3、フック `jq -r '.tool_input.file_path' | xargs npx prettier --write`、`.prettierignore` に `docs/backlog.md` 登録）で**再現しない**ことを確認した。較正テストとして通常ファイルは `prettier --debug-check` で処理対象として出力される一方、`docs/backlog.md` は絶対パス指定でも処理対象に出ず、`prettier --write` を実際に流しても差分ゼロ（ignore が尊重されている）。
- **重要な反パターン記録**: 取り下げ前の「対症療法案」が示していた「Edit/Write を使わず `git show` + bash で書き戻す」運用は、**安全策（PostToolUse 整形・PreToolUse の行長チェックを除く各種フック）を回避するハッキングであり、絶対に採ってはならない**。ファイル編集は常に Edit/Write を使うこと。`cycle-245` の事故は、Notes 記述の文字数超過が真因の可能性が高い（事故時の元の長さを後から再現できないため断定はできないが、現環境では「ignore 貫通」の挙動を再現できないため、元記述は誤診と扱う）。AP-WF21 を信じて bash 迂回をすると、フックで防ぐべき問題（整形漏れ・行長超過・将来追加されるチェック）を黙って素通しさせる。
- **正しい運用**: `docs/backlog.md` 等は通常通り Edit ツールで編集する。コミット時に `backlog-line-length-check.sh` でブロックされた場合は、整形が悪いのではなく自分の記述が長すぎる可能性をまず疑い、要約＋参照先（cycle 番号等）の形に削る。フックの挙動に疑いがある場合は、対症療法ではなく実測（`prettier --debug-check` / `--write` 差分確認）で再現性を取ってから判断する。
- 残骸となった事象記録は `docs/cycles/cycle-245.md` / `cycle-246.md` 参照。

## AP-WF28 候補: 守れていないルールに対し、根本原因に向き合わず新しいルール・手順・チェック項目を書き足して対処していないか／意図的に削ぎ落とされた手順書（skill 等）に思い付きで追記して肥大化させていないか

- **概要**: あるルールを守れなかったとき、それを守らせる新項目を指示文書・手順書に書き足す対処は、ルールの総数を増やすだけで同種の逸脱を止めない（AP 集の発生件数が累積し続けている事実そのものが証拠）。とくに cycle-kickoff 等の手順書は意図的に削ぎ落として運用されており、思い付きの追記はその設計を破壊する。守れていないルールに対する根本対処は、ルールの追加ではなく、(1) ルールの根源にある行動原則＝『望み』を先に立てること、(2) 必要なら技術的に強制すること（フック等。簡略化の運用経緯は blog `workflow-simplification-stopping-rule-violations` / `stop-piling-rules-give-ai-its-wish` に記録）。「自分がそのルールを現に守れていない」状態で新ルールを書いても、守る保証はどこにもない。
  → cycle-256 で、Review always を自ら省略した直後に、その再発防止として cycle-kickoff にレビュー必須ゲートを追記（Owner 第2介入で撤回・原状復帰）。（cycle-256 で実際に発生）
- **発生件数**: N=1。N≥3 で AP-WF28 として新設検討。

## AP-WF29 候補: PM 自身のツール呼び出し（Edit/Write 等）の結果を確認せず次へ進んでいないか／並行実行中の別エージェントの結果に注意を奪われ、自分のフォアグラウンド操作の失敗（エラー応答）を見落としていないか

- **概要**: 自分の Edit/Write がエラー（Read-before-Edit 違反・old_string 不一致等）で失敗しても、結果を確認せず進むと「変更したつもりで変更されていない」状態が黙って残る。とくに並行エージェントの完了を待ちながら自分も状態変更系の操作をしていると、注意が並行側に向き自操作の失敗監視が抜ける。サブエージェント成果の検証（AP-WF04 系）・自動チェック PASS の再実行（AP-WF16 系＝現在はフックへ移管済み）が「他者の出力」を対象にするのに対し、本 AP は「**PM 自身のツール結果**」を対象にする補完で、既存 AP では捕捉されない。
  - **正しい姿**: 状態変更系（backlog の状態遷移・SSoT 更新等）のツール呼び出しは、結果（成功/エラー）を確認してから次へ進む。並行エージェントを走らせている間に自分も編集するときは、自操作の結果確認を省略しない。
  - **根本対処の留意（AP-WF28 との整合）**: 本項を「新ルールの書き足し」で終わらせない。根は「注意の規律」であり、技術的強制が可能なら（フック等で Edit エラーを検知し継続を止める／状態変更後に実体を grep で突き合わせる手順を skill 化する等）そちらを優先検討する。本記録は再発トラッキングのための候補であって、ルール追加そのものを対処と見なさない。
- **発生件数**: N=3（cycle-266〔backlog の Queued→Active 編集が Read 前 Edit で失敗→並行エージェントに気を取られ見落とし→start commit に未遷移状態が混入。完了処理で是正・プロダクト実害なし。詳細 cycle-266.md 事故記録〕・cycle-278〔stale-read で失敗した Edit と、その変更を主張するコミットを同一バッチで実行し、コミットメッセージが未適用の編集を主張する不一致が発生。直後に検出し実体を適用・経緯をコミットに明記。プロダクト実害なし。予防の要点=**状態変更の Edit と、それを主張するコミットを同一バッチで走らせない（Edit の成功を確認してからコミットする）**。詳細 cycle-278.md 追記〕, cycle-284）。**N≥3 に到達したため AP-WF29 として新設を検討する。**
  - **関連事例（射程が近いが N 非カウント）**: cycle-267 M-1（build の完了 exit を確認せず tail 出力だけで stale なローカルサーバーを起動し視覚検証で3ルート 500 を誤検知→再起動で解消・プロダクト実害なし。「自分の操作結果（build 完了）の確認漏れ」で根が近いが、対象が Edit/Write でなくサーバー起動タイミングのため N に含めない。詳細 cycle-267.md M-1）。cycle-275（実装 builder 並走中に PM が `git add -A` で一括コミットし、並走側が書き終えた実装成果を接地コミットに巻き込み、コミットのラベルと内容が乖離。開示で回復・プロダクト実害なし。「並走エージェントに注意が向いた状態での自操作が意図外の結果を生む」で根が近いが、Edit/Write のエラー見落としでなく git add -A の過剰ステージのため N に含めない。詳細 cycle-275.md）。

## AP-WF30 候補: リスク/到達性/影響の評価結論を、品質課題の「対応範囲を縮める」根拠に流用していないか

- **概要**: リスク・到達性・影響度の評価は「優先度・緊急度（どれだけ急ぐか）」を教えるものであって「対応範囲（どこまで直すか）」を決めるものではない。両者を取り違え、「影響が小さい／届かない」という評価結論をそのまま「だから最小限の対応でよい」へ接続すると、品質課題（constitution Rule 4＝every aspect の品質）の対応が不当に縮む。品質課題の対応範囲は、影響度ではなく「あるべき品質状態」から決めるべき。AP-WF09「チェック対象の範囲を恣意的に絞る」がレビュー/チェックの網羅性を対象にするのに対し、本候補は「評価の結論を対応スコープの限定に流用する」意思決定の構造を対象にする。
  → cycle-286 で、依存脆弱性（B-505）の来訪者到達性評価「REACHABLE-VISITOR 0件」を根拠に、フラグ対象だけを overrides で固定する狭い修正に留めた（全依存の陳腐化・既知脆弱バージョンを放置）。品質課題の対応範囲を到達性で縮めた。詳細 cycle-286/incident-1.md。
- **発生件数**: N=1（cycle-286）/ N≥3 で AP-WF30 として新設検討。
- **根本対処の留意（AP-WF28 との整合）**: 新ルールの書き足しで終わらせない。根は「評価の目的（優先度）と対応範囲の決定を分ける」という原則であり、本記録は再発トラッキングのための候補である。
