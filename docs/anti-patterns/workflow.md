# ワークフローのアンチパターン

このチェックリストは、作業の進め方に関する過去の失敗パターンを記録したものです。サイクル完了時やレビュー時に参照し、プロセス上の問題がないかを確認してください。

- AP-WF01: 最後の修正のあと、それがどんな修正であったとしてもレビューを実施したか？ すべての指摘事項に対応したか？
  → 「早く完了させたい」バイアスがレビュー省略・手順ショートカットを引き起こし、品質問題を見逃す。（cycle-118, 129, 147, 148, 159, 161, 168, 171, 181, 188, 256, 257で実際に発生）

- AP-WF02: すべての変更を来訪者目線でレビューしたか？ 過去の失敗事例を参照させたか？
  → レビューの品質は回数ではなく観点の設計で決まり、形式的チェックだけでは本質的な問題を検出できない。（cycle-155, 162, 168, 175, 181, 184, 256で実際に発生）

- AP-WF03: builderへの指示に型定義、JSXコード全文、CSSクラスの全定義、テストケースの詳細仕様など「どう実装するか」の具体的コードを含めていないか？
  → 過剰に具体的な指示はbuilderの設計判断やバイアス検出の機能を無効化し、PMの誤りがそのまま成果物に反映される。設計判断が PM 側に閉じている事実情報（トークン置換マップ・API シグネチャ・外部仕様への適合など）を builder に渡すのは違反ではないが、実装表現の単位（inline style 値・JSX 構造・CSS クラス名・テスト assertion 文言など）まで literal で確定するのは違反である。（cycle-76, 122, 130, 143, 144, 145, 146, 175, 181, 188, 254で実際に発生）

- AP-WF04: サブエージェントの正式な完了通知（task-notification）を受け取ってから次ステップに進んでいるか？`git diff` でファイル変更を確認しただけで完了と判断していないか？「移行完了」「集約完了」「重複解消」のような構造的変更の完了報告を、`grep`/`find`/`Read` での実態確認なしに完了扱いしていないか？
  → 出力ファイルの途中読み取りで「完了」と判断すると、未完成の成果物に対してレビュー・コミット・デプロイが進行する。構造的変更の報告は「ファイルが変更されている」だけでは検証として不十分で、旧呼び出しの残存確認まで行わないと完了とは言えない。（cycle-142, 146, 175, 188, 255で実際に発生）

- AP-WF05: UIやデザインに影響がある変更をした場合、すべての画面をモバイル/PCとライトモード/ダークモードの組み合わせで実際のスクリーンショットを撮ってレビューしたか？
  → モバイルのみの確認でデプロイすると、デスクトップやダークモードで重大な表示問題が後から発覚する。撮影自体の網羅だけでは不十分で、撮ったものを来訪者目線の観点リストで通し評価し、画像と記述を 1 枚ずつ照合する段階までやらないと表示問題は検出できない。撮影の網羅性・着手前撮影・取り忘れ時の復旧手順は `.claude/skills/take-screenshot/SKILL.md` 参照。（cycle-129, 147, 148, 152, 171, 175, 180, 181, 185, 187, 188で実際に発生）

- AP-WF06: サブエージェントに渡す事実情報（数値・日付・コンテンツ内容等）をファイルやコマンドで事前に確認したか？推測や記憶で伝えていないか？
  → 未確認の情報がサブエージェントに渡ると、調査レポートや計画が誤った前提で作成され、将来のサイクルにも誤りが伝播する。調査レポート由来の数値であっても、指示文に転記する前に実体確認しなければ同じ伝播が起きる。（cycle-162, 163, 169, 171, 187, 188, 210, 228, 243 で実際に発生）

- AP-WF07: サブエージェントに複数の独立したタスクをまとめて依頼していないか？1エージェント1タスクの原則を守っているか？逆に、同一ファイルへの変更を含む複数タスクを別エージェントに**並行アサイン**していないか？
  → 複数タスクの一括委任は品質低下とトレーサビリティの喪失を引き起こす。個別の問題を切り分けられなくなり、レビューの精度も下がる。逆方向の同型として、同一ファイルへの変更を含むタスクを並行アサインすると競合が発生して品質を損なう。同一ファイルを触る複数タスクは「並行不能」として明示し、同一エージェントに直列で依頼する。（cycle-163, 181, 184で実際に発生）

- AP-WF08: PMがサブエージェントの作業を代行・改変していないか？サブエージェントの成果物を要約・解釈してファイルに保存していないか？PMの「即時編集」を機械的整形（prettier 等・判断を伴わない）以外に広げていないか？
  → PMがサブエージェントの役割を代行すると、レビューの独立性が損なわれる（自分が書いたものを自分でレビューする構造になる）。成果物の改変はresearcherの調査結果の信頼性を破壊する。コメントやテスト文言の訂正は一見軽微でも人間の判断を含むため「機械的整形」に分類してはならず、builderへ差し戻してレビューを通す。（cycle-163, 168, 188, 228, 243で実際に発生）

- AP-WF09: アンチパターンチェックリストを形式的に通過させていないか？「推測」とラベルを付けるだけで未検証の前提を「対応済み」にしていないか？既存コンテンツの確認を「確認した」と書きながら実際には確認していないなど、チェック項目の実質的な検証を省略していないか？用語を変えて実質的に同じことをしていないか（「フレームワーク」を「方法論」と呼び替える、「長期計画」を「方向性」と呼び替える等）？
  → チェックリストの通過が目的化すると、過去の失敗パターンを認識しながら同じ失敗を繰り返す。形式的なラベル付け（「推測」「未検証」等）は誠実さの装いになるが、検証せずに計画の優先順位を決めている構造は変わらない。名称の変更は同一構造を検出不能にする特に危険な形態であり、表面的な用語ではなく実質的な構造が過去の失敗と同じでないかを検証する必要がある。チェック対象の範囲を恣意的に絞る（固有名詞のみ確認して構造前提の記述は見過ごす、既知パターンのみ照合して未知の変形は素通りさせる等）ことも同じ構造の一種である。（cycle-165, 166, 168, 169, 171, 175, 181, 184, 187, 188, 247, 255, 273で実際に発生）

- AP-WF10: 新しいタスクを既存サブエージェントへのSendMessageで継続させていないか？同じエージェントに連続して異なるタスクを投げてコンテキストを蓄積させていないか？SendMessage を同一タスクの軽微な修正・追加質問のみに限定しているか？
  → SendMessageで継続するとコンテキストが蓄積して性能が急速に劣化する。（cycle-168で実際に発生）

- AP-WF11: PMは自分が責任を持つ成果物（外部公開物・SSoT・エージェント定義・設計書など）の承認判断前に、最終成果物を自分自身で通読確認したか？reviewerやビジュアルチェックの通過を自分の判断の代用にしていないか？特に、整合する必要がある複数ファイル（例：`(legacy)/layout.tsx` と `(new)/layout.tsx`、共通コンポーネント新旧版、metadata と sitemap など）は、並べ読みして差分（不一致 / 欠落）を確認・記録したか？
  → レビューや機械チェックで検出されない本質的問題（記事であれば視点立脚・主張の整合、計画書や設計書であれば前提と結論の整合、エージェント定義であれば意図の伝わり方など）は、PMが自分で読まないと気づけない。複数ファイルが整合する必要がある成果物では、並べ読みをしないと気付けない欠落（GA・JSON-LD・provider 群の片方欠落、SSoT ラベル変更時のソースコード内コメント文言の遡及漏れ等）が見落とされる。（cycle-168, 169, 171, 175, 180, 181, 184, 188, 214, 215, 255, 267, 269, 273 で実際に発生）

- AP-WF12: 計画書に登場するファイルパス・ディレクトリ構造・ファイル名・行番号・コンテンツ数・**他タスクの状態（Active / Queued / Deferred / Done、着手条件の達成状況）** などを、ドキュメント・SSoT・過去サイクルの記述を一次情報として扱い、ファイルシステム/コードでの実体確認を省略していないか？タスク列挙で「着手条件が達成された」「既に完了している」と書く前に、依存する backlog 項目の状態を実体確認したか？複合条件の着手条件（例: 「A 完了 + B 完了」）では、両方の項目を独立に確認したか？
  → AP-WF06 が「サブエージェントに渡す事実情報の確認」に焦点を当てているのに対し、このパターンは「PM/planner が計画立案中に自ら参照する事実情報の確認」に焦点がある。計画書に記述した事実（ファイルの存在・ディレクトリ構造・コンテンツ数・**他タスクの完了状況**・**フレームワーク API の挙動** など）は、過去サイクルの記述や Owner 発言ではなく、`ls`・`find`・`grep`・`Read` 等のコマンドで最新の実体を確認してから書く。確認せずに書くと「実在しないファイルを不在と断言」「実態と異なる数値を計画書に記載」「URL アクセス不可のルートを単位として列挙」「**着手条件未達の他タスクを『着手条件達成』と計画書に記述**」「**フレームワーク API の挙動を思い込みで記述**（例: Next.js の `permanentRedirect()` を 301 と書いたが実際は 308）」といった誤りが計画書に混入し、後続レビューで複数ラウンドの指摘を生む。（cycle-172, 174, 181, 183, 184, 186, 187, 188, 210, 214 で実際に発生）

- AP-WF13: 並行アサインした独立タスクの担当 builder が、自タスクのスコープを越えて隣接タスクの作業まで実施していないか？PM の指示で、スコープ外の操作（典型的には他タスクの `git mv` / 大規模リネーム / 共有設定ファイルの編集 など）を抑止しているか？
  → 隣接タスクの作業に手を出すと、commit 構成が当初の rollback 戦略と乖離し、`git revert` で 1 commit を巻き戻すと隣接タスクまで巻き戻ってしまう（または逆に巻き戻したくないタスクが巻き戻らない）といった事故が起きる。（cycle-183で実際に発生）

- AP-WF14: reviewer が **数値・件数・状態** に関する指摘を採用判断するとき、**自ら一次集計を独立に行わずに、複数レポートの相対比較だけで採否を決めていないか？**
  → 「最新優先」「詳細優先」「整合性チェック」だけで採否を決めると、複数ソースが揃って同方向に間違っている時に検出できない。レポート間の矛盾を見つけたときに「最新を採る」だけで吸収してはならず、必ず一次集計で再確認する必要がある。（cycle-183, 214, 215 で実際に発生）

- AP-WF15: サイクル完了処理を進めた後（あるいは push 後）に来訪者影響のある補修課題が見つかった場合、「同サイクル延長」「別サイクル独立」どちらに振るかを思いつきで決めず、明示的な判断軸（来訪者影響の有無 / 当該サイクル目的範囲との整合 / 本格対応の規模 / 暫定対応長期化への歯止め策）で検証したか？
  → サイクル完了後の補修は時間的圧力が掛かりやすく、「すぐ直したい」バイアスで同サイクルに無理に押し込むと、本来別サイクルにすべき大規模変更が中途半端な暫定対応として残置される。逆に「もう完了したから次サイクル」と機械的に分割すると、来訪者影響がある問題を放置する事故になる。（cycle-183, 215 で実際に発生）

- AP-WF16: builder が報告した自動チェック（`npm run lint` / `npm run format:check` / `npm run test` / `npm run build` 等）の PASS を、reviewer / PM は当該コマンドを独立に再実行して出力（件数・最終行・exit code）で確認したか？ サブエージェントの自己申告のみを根拠に PASS 判定していないか？
  → 自動チェック結果は文字列で再現が容易（プロンプト上に「全 PASS」と書くだけで作れる）であり、サブエージェントの自己申告のみに依拠すると、実際には FAIL しているチェックを誤って採用する事故が起きる。一方で reviewer / PM 側の独立再実行コストは安価（同じコマンドを 1 回叩くだけ）。AP-WF04（完了通知 + 実態確認）が「構造的変更の grep 再確認」に焦点を当てるのに対し、本 AP は「自動チェック PASS の独立再実行」に焦点を当てる補完関係。過去サイクル中盤の PASS を根拠に最終状態を再実行せず通過扱いにするのも本 AP に含む。（cycle-203, cycle-242で実際に発生）

- AP-WF22: 長文を Write ツールで新規作成した直後と、ドキュメント変更を commit する前に、ファイル末尾の残骸タグ（`</content>` 等）を点検したか？ `prettier` / `format:check` の通過を「混入なし」の根拠にしていないか？
  → 長文を Write すると末尾に `</content>` のような不要行が混入することがあり、整形チェック（prettier）は整形上妥当として素通りするため commit まで到達しうる。grep でのみ検出可能で、検出後の除去は可能（回復可能）。原因（Write の content シリアライズ起因か生成側か）は未確定だが、短いファイルでは未観測で長文時に起きやすい傾向（推定）。（cycle-246, 251 で実際に発生）

## 将来 AP 候補メモ

このセクションは AP リスト本体（AP-WF01〜16）とは別扱いのメモ置き場です。N≥3 件発生時に正式 AP として昇格を検討してください。

### AP-WF17 候補: 計画書ファイル肥大化によるレビュー全文通読困難

- **概要**: 計画書ファイルが Read 制限超過サイズ（例: 1181 行 / 264KB）まで肥大化し、reviewer がレビュー全文通読困難になった事例（cycle-214 / 1 件目・cycle-215 = 553.9KB / 2718 行 / 2 件目）。
- **発生件数**: N=2（cycle-214, 215）/ N≥3 で AP-WF17 として新設検討。
- **対症療法案**（未採択・仮想パス例示）: 過去ラウンドのレビュー対応マップを `tmp/cycle-XXX/review-history.md`（例）に外出しし、計画書本体は最新状態のみに絞る。

### AP-WF18 候補: サブエージェントが特定ツールの呼び出しでハングし長時間停止していないか

- **概要**: サブエージェント（reviewer / web-researcher 等）が特定ツールの応答待ちでフリーズし、PM が完了通知を待ち続けて長時間（最大で約 1〜2 日）気付かず停止していた事例。背景監視の盲点 = 「完了通知が来ない＝作業中」と解釈してしまい、ハングと進行中を区別できない。
  - **cycle-218 事例**: 調査サブエージェント（web-researcher）が WebFetch で取得コンテンツを prompt 要約させる呼び出しでハング（最大約 1 日）。
  - **cycle-227 事例**: サイクル全体の最終検証 reviewer が Playwright（`browser_evaluate` / `browser_wait_for`）で**無限待機する JS** を実行してフリーズし、約 2 日停止（Owner が停止）。完了通知が出ても過程が破綻していれば結果は信用できない。安全な書き方は `docs/knowledge/playwright-mcp.md` に記録。
- **発生件数**: N=2（cycle-218, 227）/ N≥3 で AP-WF18 として新設検討。
- **対症療法案**: (1) 長時間（目安: 通常完了時間を大きく超過）完了通知が来ないバックグラウンドエージェントは、進行中とハングを区別するために状態（出力ファイル）を確認し、必要なら停止して再実行する。「通知が来ない＝作業中」と解釈し続けない。(2) ハングしうるツール呼び出しを避ける = 外部 URL は WebFetch の prompt 要約に依存せず WebSearch / Playwright（navigate → snapshot）へ。Playwright の `browser_evaluate` には**解決しない Promise／無限ループ**を渡さず、待ちが要るときは有限タイムアウトか「操作→別 evaluate で結果読み」の2ステップにする。(3) サブエージェントへの指示文に「無限待機する JS / タイムアウト無しの待ちを書かない」を明示する。ツール挙動の詳細知見は `docs/knowledge/playwright-mcp.md`。

### AP-WF19 候補: 計測値 SSoT を「実測値」として記録する際、生成元スクリプトの実出力（生成物のキー・値）と照合せず確定していないか

- **概要**: AP-P21 計測等の SSoT を §補足事項に書き戻す際、「実測値 / 生成元 = <計測スクリプト>」と記載しながら、その生成元スクリプトが当該値を実際には出力していなかった事例。生成元を書いても「その生成元が当該値を吐くこと」を確認していないため、AP-WF12（実体確認）/ AP-P16（生成元併記）の射程をすり抜ける。算術的に自己整合した手計算値を「実測値」と誤記録すると、後続サイクルが引用する基準値の信頼性が崩れる。（cycle-218, 219 で実際に発生）
- **発生件数**: N=2（cycle-218, 219）/ N≥3 で AP-WF19 として新設検討。
- **対症療法案**: 計測値を「実測値 / 生成元 = <script>」と記録する前に、生成物（results.json 等）に当該キー・値が実在することを grep で確認し SSoT の数値と桁まで一致させる。スクリプト修正後は必ず再実行して生成物を更新してから SSoT へ書き戻す。

### AP-WF20 候補: 修正後の再レビューを既存レビュアーの流用で行い、白紙からの全体見直しになっていない

- **概要**: 修正後の再レビューを、そのタスクを前にレビューした既存レビュアーに SendMessage で継続依頼して行うと、レビュアーは自分の前回指摘が直ったかの確認に視野が狭まり、全体を白紙から見直すバイアスがかかる。AP-WF01「最後の修正後にゼロからすべてを見直す」の実効には、再レビューのたびに新規レビュアーをゼロから起動し、前回の指摘・経緯を渡さず白紙で全体を見直させる必要がある。
  - **cycle-243 事例**: 流用での再レビューが見逃した記事中の数値の事実誤り（「7ページで6回」＝実際は個別7ページ5回＋一覧1回）を、新規レビュアーが白紙のレビューで検出した。
- **発生件数**: N=1（cycle-243）/ N≥3 で AP-WF20 として新設検討。
- **対症療法案**: 再レビューは既存レビュアーの SendMessage 継続でなく新規エージェントを起動し、対象ファイルと観点のみを渡して（前回指摘は渡さず）白紙で全体を見直させる。
