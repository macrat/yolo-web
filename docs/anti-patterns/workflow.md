# ワークフローのアンチパターン

このチェックリストは、作業の進め方に関する過去の失敗パターンを記録したものです。サイクル完了時やレビュー時に参照し、プロセス上の問題がないかを確認してください。

- AP-WF01: 最後の修正のあと、それがどんな修正であったとしてもレビューを実施したか？ すべての指摘事項に対応したか？
  → 「早く完了させたい」バイアスがレビュー省略・手順ショートカットを引き起こし、品質問題を見逃す。（cycle-118, 129, 147, 148, 159, 161で実際に発生）

- AP-WF02: すべての変更を来訪者目線でレビューしたか？ 過去の失敗事例を参照させたか？
  → レビューの品質は回数ではなく観点の設計で決まり、形式的チェックだけでは本質的な問題を検出できない。（cycle-155, 162で実際に発生）

- AP-WF03: builderへの指示に型定義、JSXコード全文、CSSクラスの全定義、テストケースの詳細仕様など「どう実装するか」の具体的コードを含めていないか？
  → 過剰に具体的な指示はbuilderの設計判断やバイアス検出の機能を無効化し、PMの誤りがそのまま成果物に反映される。（cycle-76, 122, 130, 143, 144, 145, 146で実際に発生）

- AP-WF04: サブエージェントの正式な完了通知（task-notification）を受け取ってから次ステップに進んでいるか？`git diff` でファイル変更を確認しただけで完了と判断していないか？
  → 出力ファイルの途中読み取りで「完了」と判断すると、未完成の成果物に対してレビュー・コミット・デプロイが進行する。（cycle-142, 146で実際に発生）

- AP-WF05: UIやデザインに影響がある変更をした場合、すべての画面をモバイル/PCとライトモード/ダークモードの組み合わせで実際のスクリーンショットを撮ってレビューしたか？
  → モバイルのみの確認でデプロイすると、デスクトップやダークモードで重大な表示問題が後から発覚する。（cycle-129, 147, 148, 152で実際に発生）

- AP-WF06: サブエージェントに渡す事実情報（数値・日付・コンテンツ内容等）をファイルやコマンドで事前に確認したか？推測や記憶で伝えていないか？
  → 未確認の情報がサブエージェントに渡ると、調査レポートや計画が誤った前提で作成され、将来のサイクルにも誤りが伝播する。（cycle-162でドメイン年齢「5ヶ月」の誤伝達、cycle-163でブログ内容を「Claude Code系」と誤伝達が実際に発生）

- AP-WF07: サブエージェントに複数の独立したタスクをまとめて依頼していないか？1エージェント1タスクの原則を守っているか？
  → 複数タスクの一括委任は品質低下とトレーサビリティの喪失を引き起こす。個別の問題を切り分けられなくなり、レビューの精度も下がる。（cycle-163で3トピックを1 researcherに委任して発生）

- AP-WF08: PMがサブエージェントの作業を代行・改変していないか？サブエージェントの成果物を要約・解釈してファイルに保存していないか？
  → PMがサブエージェントの役割を代行すると、レビューの独立性が損なわれる（自分が書いたものを自分でレビューする構造になる）。成果物の改変はresearcherの調査結果の信頼性を破壊する。（cycle-163でresearcherの成果物をPMが手動で書き直して発生）

- AP-WF09: アンチパターンチェックリストを形式的に通過させていないか？「推測」とラベルを付けるだけで未検証の前提を「対応済み」にしていないか？既存コンテンツの確認を「確認した」と書きながら実際には確認していないなど、チェック項目の実質的な検証を省略していないか？
  → チェックリストの通過が目的化すると、過去の失敗パターンを認識しながら同じ失敗を繰り返す。形式的なラベル付け（「推測」「未検証」等）は誠実さの装いになるが、検証せずに計画の優先順位を決めている構造は変わらない。（cycle-165でAP-P01, AP-P06を「対応済み」としながら実質的に未対応だったことで実際に発生）
