# ワークフローのアンチパターン

このチェックリストは、作業の進め方に関する過去の失敗パターンを記録したものです。サイクル完了時やレビュー時に参照し、プロセス上の問題がないかを確認してください。

- AP-WF01: 最後の修正のあと、それがどんな修正であったとしてもレビューを実施したか？ すべての指摘事項に対応したか？
  → 「早く完了させたい」バイアスがレビュー省略・手順ショートカットを引き起こし、品質問題を見逃す。（cycle-118, 129, 147, 148, 159, 161, 168, 171, 181, 188, 256, 257で実際に発生）

- AP-WF02: すべての変更を来訪者目線でレビューしたか？ 過去の失敗事例を参照させたか？
  → レビューの品質は回数ではなく観点の設計で決まり、形式的チェックだけでは本質的な問題を検出できない。（cycle-155, 162, 168, 175, 181, 184, 256, 276, 282, 283で実際に発生）

- AP-WF03: builderへの指示に型定義、JSXコード全文、CSSクラスの全定義、テストケースの詳細仕様など「どう実装するか」の具体的コードを含めていないか？
  → 過剰に具体的な指示はbuilderの設計判断やバイアス検出の機能を無効化し、PMの誤りがそのまま成果物に反映される。設計判断が PM 側に閉じている事実情報（トークン置換マップ・API シグネチャ・外部仕様への適合など）を builder に渡すのは違反ではないが、実装表現の単位（inline style 値・JSX 構造・CSS クラス名・テスト assertion 文言など）まで literal で確定するのは違反である。（cycle-76, 122, 130, 143, 144, 145, 146, 175, 181, 188, 254で実際に発生）

- AP-WF04: サブエージェントの正式な完了通知（task-notification）を受け取ってから次ステップに進んでいるか？`git diff` でファイル変更を確認しただけで完了と判断していないか？「移行完了」「集約完了」「重複解消」のような構造的変更の完了報告を、`grep`/`find`/`Read` での実態確認なしに完了扱いしていないか？
  → 出力ファイルの途中読み取りで「完了」と判断すると、未完成の成果物に対してレビュー・コミット・デプロイが進行する。構造的変更の報告は「ファイルが変更されている」だけでは検証として不十分で、旧呼び出しの残存確認まで行わないと完了とは言えない。（cycle-142, 146, 175, 188, 255で実際に発生）

- AP-WF05: UIやデザインに影響がある変更をした場合、すべての画面をモバイル/PCとライトモード/ダークモードの組み合わせで実際のスクリーンショットを撮ってレビューしたか？
  → モバイルのみの確認でデプロイすると、デスクトップやダークモードで重大な表示問題が後から発覚する。撮影自体の網羅だけでは不十分で、撮ったものを来訪者目線の観点リストで通し評価し、画像と記述を 1 枚ずつ照合する段階までやらないと表示問題は検出できない。撮影の網羅性・着手前撮影・取り忘れ時の復旧手順は `.claude/skills/take-screenshot/SKILL.md` 参照。（cycle-129, 147, 148, 152, 171, 175, 180, 181, 185, 187, 188で実際に発生）

- AP-WF06: サブエージェントに渡す事実情報（数値・日付・コンテンツ内容等）をファイルやコマンドで事前に確認したか？推測や記憶で伝えていないか？
  → 未確認の情報がサブエージェントに渡ると、調査レポートや計画が誤った前提で作成され、将来のサイクルにも誤りが伝播する。調査レポート由来の数値であっても、指示文に転記する前に実体確認しなければ同じ伝播が起きる。最新サイクルの番号と状態は SessionStart フック（`.claude/hooks/session-start-facts.sh`）が実測して配るので、それを使う。それ以外の事実情報は使う前に必ず実測する。（cycle-162, 163, 169, 171, 187, 188, 210, 228, 243で実際に発生）

- AP-WF07: サブエージェントに複数の独立したタスクをまとめて依頼していないか？1エージェント1タスクの原則を守っているか？逆に、同一ファイルへの変更を含む複数タスクを別エージェントに**並行アサイン**していないか？
  → 複数タスクの一括委任は品質低下とトレーサビリティの喪失を引き起こす。個別の問題を切り分けられなくなり、レビューの精度も下がる。逆方向の同型として、同一ファイルへの変更を含むタスクを並行アサインすると競合が発生して品質を損なう。同一ファイルを触る複数タスクは「並行不能」として明示し、同一エージェントに直列で依頼する。（cycle-163, 181, 184で実際に発生）

- AP-WF08: PMがサブエージェントの作業を代行・改変していないか？サブエージェントの成果物を要約・解釈してファイルに保存していないか？PMの「即時編集」を機械的整形（prettier 等・判断を伴わない）以外に広げていないか？
  → PMがサブエージェントの役割を代行すると、レビューの独立性が損なわれる（自分が書いたものを自分でレビューする構造になる）。成果物の改変はresearcherの調査結果の信頼性を破壊する。コメントやテスト文言の訂正は一見軽微でも人間の判断を含むため「機械的整形」に分類してはならず、builderへ差し戻してレビューを通す。（cycle-163, 168, 188, 228, 243で実際に発生）

- AP-WF09: アンチパターンチェックリストを形式的に通過させていないか？「推測」とラベルを付けるだけで未検証の前提を「対応済み」にしていないか？既存コンテンツの確認を「確認した」と書きながら実際には確認していないなど、チェック項目の実質的な検証を省略していないか？用語を変えて実質的に同じことをしていないか（「フレームワーク」を「方法論」と呼び替える、「長期計画」を「方向性」と呼び替える等）？
  → チェックリストの通過が目的化すると、過去の失敗パターンを認識しながら同じ失敗を繰り返す。形式的なラベル付け（「推測」「未検証」等）は誠実さの装いになるが、検証せずに計画の優先順位を決めている構造は変わらない。名称の変更は同一構造を検出不能にする特に危険な形態であり、表面的な用語ではなく実質的な構造が過去の失敗と同じでないかを検証する必要がある。チェック対象の範囲を恣意的に絞る（固有名詞のみ確認して構造前提の記述は見過ごす、既知パターンのみ照合して未知の変形は素通りさせる等）ことも同じ構造の一種である。**達成していない網羅性を主張して結論を支える**のも同型である（「全通り試した」「全件確認した」と書きながら実際は一部しか見ていない）——網羅を主張する前に、探索した件数と全体の件数を数字で突き合わせること。（cycle-165, 166, 168, 169, 171, 175, 181, 184, 187, 188, 247, 255, 273, 277, 283, 284, 286で実際に発生）

- AP-WF10: 新しいタスクを既存サブエージェントへのSendMessageで継続させていないか？同じエージェントに連続して異なるタスクを投げてコンテキストを蓄積させていないか？SendMessage を同一タスクの軽微な修正・追加質問のみに限定しているか？
  → SendMessageで継続するとコンテキストが蓄積して性能が急速に劣化する。再レビュー依頼の SendMessage は `.claude/hooks/block-review-reuse.sh` が機械的にブロックするが、レビュー以外のタスク継続はこの項目で点検する。（cycle-168で実際に発生）

- AP-WF11: PMは自分が責任を持つ成果物（外部公開物・SSoT・エージェント定義・設計書など）の承認判断前に、最終成果物を自分自身で通読確認したか？reviewerやビジュアルチェックの通過を自分の判断の代用にしていないか？特に、整合する必要がある複数ファイル（例：`(legacy)/layout.tsx` と `(new)/layout.tsx`、共通コンポーネント新旧版、metadata と sitemap など）は、並べ読みして差分（不一致 / 欠落）を確認・記録したか？
  → レビューや機械チェックで検出されない本質的問題（記事であれば視点立脚・主張の整合、計画書や設計書であれば前提と結論の整合、エージェント定義であれば意図の伝わり方など）は、PMが自分で読まないと気づけない。複数ファイルが整合する必要がある成果物では、並べ読みをしないと気付けない欠落（GA・JSON-LD・provider 群の片方欠落、SSoT ラベル変更時のソースコード内コメント文言の遡及漏れ等）が見落とされる。（cycle-168, 169, 171, 175, 180, 181, 184, 188, 214, 215, 255, 267, 269, 273で実際に発生）

- AP-WF12: 計画書に登場するファイルパス・ディレクトリ構造・ファイル名・行番号・コンテンツ数・**他タスクの状態（Active / Queued / Deferred / Done、着手条件の達成状況）** などを、ドキュメント・SSoT・過去サイクルの記述を一次情報として扱い、ファイルシステム/コードでの実体確認を省略していないか？タスク列挙で「着手条件が達成された」「既に完了している」と書く前に、依存する backlog 項目の状態を実体確認したか？複合条件の着手条件（例: 「A 完了 + B 完了」）では、両方の項目を独立に確認したか？
  → AP-WF06 が「サブエージェントに渡す事実情報の確認」に焦点を当てているのに対し、このパターンは「PM/planner が計画立案中に自ら参照する事実情報の確認」に焦点がある。計画書に記述した事実（ファイルの存在・ディレクトリ構造・コンテンツ数・**他タスクの完了状況**・**フレームワーク API の挙動** など）は、過去サイクルの記述や Owner 発言ではなく、`ls`・`find`・`grep`・`Read` 等のコマンドで最新の実体を確認してから書く。確認せずに書くと「実在しないファイルを不在と断言」「実態と異なる数値を計画書に記載」「URL アクセス不可のルートを単位として列挙」「**着手条件未達の他タスクを『着手条件達成』と計画書に記述**」「**フレームワーク API の挙動を思い込みで記述**（例: Next.js の `permanentRedirect()` を 301 と書いたが実際は 308）」といった誤りが計画書に混入し、後続レビューで複数ラウンドの指摘を生む。（cycle-172, 174, 181, 183, 184, 186, 187, 188, 210, 214, 283で実際に発生）

- AP-WF13: 並行アサインした独立タスクの担当 builder が、自タスクのスコープを越えて隣接タスクの作業まで実施していないか？PM の指示で、スコープ外の操作（典型的には他タスクの `git mv` / 大規模リネーム / 共有設定ファイルの編集 など）を抑止しているか？
  → 隣接タスクの作業に手を出すと、commit 構成が当初の rollback 戦略と乖離し、`git revert` で 1 commit を巻き戻すと隣接タスクまで巻き戻ってしまう（または逆に巻き戻したくないタスクが巻き戻らない）といった事故が起きる。（cycle-183で実際に発生）

- AP-WF14: reviewer が **数値・件数・状態** に関する指摘を採用判断するとき、**自ら一次集計を独立に行わずに、複数レポートの相対比較だけで採否を決めていないか？**
  → 「最新優先」「詳細優先」「整合性チェック」だけで採否を決めると、複数ソースが揃って同方向に間違っている時に検出できない。レポート間の矛盾を見つけたときに「最新を採る」だけで吸収してはならず、必ず一次集計で再確認する必要がある。（cycle-183, 214, 215で実際に発生）

- AP-WF15: サイクル完了処理を進めた後（あるいは push 後）に来訪者影響のある補修課題が見つかった場合、「同サイクル延長」「別サイクル独立」どちらに振るかを思いつきで決めず、明示的な判断軸（来訪者影響の有無 / 当該サイクル目的範囲との整合 / 本格対応の規模 / 暫定対応長期化への歯止め策）で検証したか？
  → サイクル完了後の補修は時間的圧力が掛かりやすく、「すぐ直したい」バイアスで同サイクルに無理に押し込むと、本来別サイクルにすべき大規模変更が中途半端な暫定対応として残置される。逆に「もう完了したから次サイクル」と機械的に分割すると、来訪者影響がある問題を放置する事故になる。（cycle-183, 215で実際に発生）

- AP-WF23: サイクル完了時チェックリストの各項目に、それが事実として真であることを自分で確認する前にチェックを入れていないか？「環境起因」「だいたい済んだ」「今回の変更と無関係」を口実にした先行チェックをしていないか？
  → 完了時チェックリストは Owner と後続 PM が「この作業は検証済み」と信頼するための事実の証明であり、未実施項目のチェックは完了状態の捏造にあたる。AP-WF09（形式的通過）と隣接するが、本項目は完了処理固有で、Owner・後続 PM への事実証明という性格上、欺瞞としての重大度が高い。各項目は文字どおり真であることを確認できる場合に限ってチェックする。完了偽装の一部は技術的に強制済み: ターン終了は `.claude/hooks/stop-cycle-guard.sh`、push 前の全チェック再実行は `.claude/hooks/pre-push-check.sh`、push 後の CI 確認は `scripts/wait-for-ci.sh` が機械検証する。（cycle-15, 16, 17, 31, 36, 58, 61, 72, 73, 74, 76, 103, 117, 118, 129, 130, 142, 143, 145, 146, 152, 158, 159, 163, 164, 165, 166, 168, 169, 170, 177, 180, 181, 182, 185, 187, 188, 189, 190, 192, 193, 194, 195, 197, 210, 215, 222, 225, 232, 234, 243, 246, 247, 251, 253, 254, 255, 276, 278, 283, 286で実際に発生。うち cycle-15〜251 の 54 件の逐語引用は `docs/research/2026-06-19-apwf23-apwf24-historical-audit.md`）

- AP-WF24: PM 権限内の判断を、自ら決定せず Owner に「進めてよいか」「どう進めるか指示を」と許可・指示を求めていないか？判断やルール遵守の駆動源を Owner に帰属させていないか（記録・報告・コミットメッセージに「Owner 指摘を受けて是正」「Owner 指示により」と書く等）？
  → CLAUDE.md「Roles and Responsibilities」の通り、PM は making decisions の責任を負い、Owner は decisions を PM に委任して自ら決定しない。PM は情報・根拠・推奨を提示した上で自ら決定・実行し、結果と根拠を報告する。是正や手続きの義務の源はルール（手順書・チェックリスト・アンチパターン集・行動原則）であって Owner ではない（Owner の介入はルールを指差したに過ぎない）。許可要求の質問は `.claude/hooks/block-owner-delegation.sh` が、コミットメッセージでの駆動源帰属は `.claude/hooks/pre-commit-check.sh` が機械的にブロックするが、サイクルドキュメント・報告文の中の駆動源帰属はこの項目で点検する。（cycle-16, 17, 19, 32, 33, 40, 41, 51, 52, 55, 58, 75, 88, 95, 99, 103, 118, 123, 130, 139, 140, 142, 146, 147, 148, 152, 153, 157, 159, 168, 169, 170, 175, 176, 177, 178, 180, 185, 188, 189, 190, 192, 193, 194, 195, 198, 199, 220, 225, 226, 243, 246, 247, 251, 253, 255, 256, 257, 259, 260, 267, 273, 274, 276, 277, 279, 283, 284, 285, 286で実際に発生。うち cycle-16〜251 の 54 件の逐語引用は `docs/research/2026-06-19-apwf23-apwf24-historical-audit.md`）

- AP-WF27: 自分の失敗の根本原因・過去の内心を、記録で裏付けないまま（しばしば自分に都合よく）事実として断言していないか？実際には存在しない発言や指示を事実として書いていないか？
  → chain-of-thought は保存されないことがあり、過去の内心は「未検証」ではなく本質的に検証不可能であるため、断定そのものが不可。可視テキスト・tool_use で確認できる範囲だけを「記録が示す事実」として述べ、記録に無いものは「記録上は存在しない」に留める。原因を語る前に、その主張がログのどこで裏付くかを確認する。もっともらしい後付けの物語は、失敗を実態より理解可能・軽微に見せ、Owner を欺く（行動原則「検証不能な断言をしない」に違反）。記事執筆ドメインの同型は AP-W04。（cycle-250, 251, 259, 274, 283, 286で実際に発生）

## 欠番（フックへ移管し機械的に強制されるアンチパターン）

- AP-WF16（自動チェック PASS の独立再実行）: `.claude/hooks/pre-push-check.sh` が push のたびに format:check / lint / typecheck / test / build を独立に再実行するため、チェックリストから削除した。サブエージェントの自己申告 PASS を信用してはならないという原則自体は変わらない。（cycle-203, 242で実際に発生していた）
- AP-WF22（Write 直後の残骸タグ混入）: `.claude/hooks/post-write-residue-check.sh`（編集直後）と `.claude/hooks/pre-commit-check.sh`（コミット前）が機械検出するため、チェックリストから削除した。（cycle-246, 251で実際に発生していた）
- AP-WF25・AP-WF26: cycle-250 で候補として新設後に同サイクル内で撤回・削除された欠番（詳細 cycle-250.md 事後訂正1）。混同を避けるため再利用しない。

## 候補メモ

正式採用前の AP 候補（N≥3 で昇格検討）・取り下げの記録は candidates.md に分離した。新しい候補もそちらへ記録する。
