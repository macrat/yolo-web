# ワークフローのアンチパターン

このチェックリストは、作業の進め方に関する過去の失敗パターンを記録したものです。サイクル完了時やレビュー時に参照し、プロセス上の問題がないかを確認してください。

- AP-WF01: 最後の修正のあと、それがどんな修正であったとしてもレビューを実施したか？ すべての指摘事項に対応したか？
  → 「早く完了させたい」バイアスがレビュー省略・手順ショートカットを引き起こし、品質問題を見逃す。（cycle-118, 129, 147, 148, 159, 161, 168, 171で実際に発生）

- AP-WF02: すべての変更を来訪者目線でレビューしたか？ 過去の失敗事例を参照させたか？
  → レビューの品質は回数ではなく観点の設計で決まり、形式的チェックだけでは本質的な問題を検出できない。（cycle-155, 162, 168で実際に発生）

- AP-WF03: builderへの指示に型定義、JSXコード全文、CSSクラスの全定義、テストケースの詳細仕様など「どう実装するか」の具体的コードを含めていないか？
  → 過剰に具体的な指示はbuilderの設計判断やバイアス検出の機能を無効化し、PMの誤りがそのまま成果物に反映される。（cycle-76, 122, 130, 143, 144, 145, 146で実際に発生）

- AP-WF04: サブエージェントの正式な完了通知（task-notification）を受け取ってから次ステップに進んでいるか？`git diff` でファイル変更を確認しただけで完了と判断していないか？
  → 出力ファイルの途中読み取りで「完了」と判断すると、未完成の成果物に対してレビュー・コミット・デプロイが進行する。（cycle-142, 146で実際に発生）

- AP-WF05: UIやデザインに影響がある変更をした場合、すべての画面をモバイル/PCとライトモード/ダークモードの組み合わせで実際のスクリーンショットを撮ってレビューしたか？
  → モバイルのみの確認でデプロイすると、デスクトップやダークモードで重大な表示問題が後から発覚する。（cycle-129, 147, 148, 152, 171で実際に発生）

- AP-WF06: サブエージェントに渡す事実情報（数値・日付・コンテンツ内容等）をファイルやコマンドで事前に確認したか？推測や記憶で伝えていないか？
  → 未確認の情報がサブエージェントに渡ると、調査レポートや計画が誤った前提で作成され、将来のサイクルにも誤りが伝播する。（cycle-162, 163, 169, 171で実際に発生）

- AP-WF07: サブエージェントに複数の独立したタスクをまとめて依頼していないか？1エージェント1タスクの原則を守っているか？
  → 複数タスクの一括委任は品質低下とトレーサビリティの喪失を引き起こす。個別の問題を切り分けられなくなり、レビューの精度も下がる。（cycle-163で実際に発生）

- AP-WF08: PMがサブエージェントの作業を代行・改変していないか？サブエージェントの成果物を要約・解釈してファイルに保存していないか？
  → PMがサブエージェントの役割を代行すると、レビューの独立性が損なわれる（自分が書いたものを自分でレビューする構造になる）。成果物の改変はresearcherの調査結果の信頼性を破壊する。（cycle-163, 168で実際に発生）

- AP-WF09: アンチパターンチェックリストを形式的に通過させていないか？「推測」とラベルを付けるだけで未検証の前提を「対応済み」にしていないか？既存コンテンツの確認を「確認した」と書きながら実際には確認していないなど、チェック項目の実質的な検証を省略していないか？用語を変えて実質的に同じことをしていないか（「フレームワーク」を「方法論」と呼び替える、「長期計画」を「方向性」と呼び替える等）？
  → チェックリストの通過が目的化すると、過去の失敗パターンを認識しながら同じ失敗を繰り返す。形式的なラベル付け（「推測」「未検証」等）は誠実さの装いになるが、検証せずに計画の優先順位を決めている構造は変わらない。名称の変更は同一構造を検出不能にする特に危険な形態であり、表面的な用語ではなく実質的な構造が過去の失敗と同じでないかを検証する必要がある。チェック対象の範囲を恣意的に絞る（固有名詞のみ確認して構造前提の記述は見過ごす、既知パターンのみ照合して未知の変形は素通りさせる等）ことも同じ構造の一種である。（cycle-165, 166, 168, 169, 171で実際に発生）

- AP-WF10: 新しいタスクを既存サブエージェントへのSendMessageで継続させていないか？同じエージェントに連続して異なるタスクを投げてコンテキストを蓄積させていないか？
  → SendMessageで継続するとコンテキストが蓄積して性能が急速に劣化する。新しいタスクには新しいエージェントを起動し、SendMessageは同一タスクの軽微な修正・追加質問のみに限定する。（cycle-168で実際に発生）

- AP-WF11: PMは自分が責任を持つ成果物（外部公開物・SSoT・エージェント定義・設計書など）の承認判断前に、最終成果物を自分自身で通読確認したか？reviewerやビジュアルチェックの通過を自分の判断の代用にしていないか？特に、整合する必要がある複数ファイル（例：`(legacy)/layout.tsx` と `(new)/layout.tsx`、共通コンポーネント新旧版、metadata と sitemap など）は、並べて差分を読み合わせる手順を取ったか？
  → レビューや機械チェックで検出されない本質的問題（記事であれば視点立脚・主張の整合、計画書や設計書であれば前提と結論の整合、エージェント定義であれば意図の伝わり方など）は、PMが自分で読まないと気づけない。複数ファイルが整合する必要がある成果物では、並べ読みをしないと気付けない欠落（GA・JSON-LD・provider 群の片方欠落等）が見落とされる。（cycle-168, 169, 171で実際に発生）

- AP-WF13: PM が builder の「対応済み」報告を実機検証せずに次ステップへ進んでいないか？特に「ヘルパに移行した」「集約した」「重複を解消した」のような構造的変更の報告は、`grep`/`find`/`Read` で実態を確認するまで完了と扱わない。
  → builder 報告と実態の乖離は、PM が報告を「自分で確認する」ステップを省くと検出できない。具体的には「移行完了と報告されたが旧呼び出しが残っている」「集約と報告されたが旧コードが残って二重管理」「import を変えた程度で構造課題は未対応」など、報告の言葉だけでは検出できない欠落が生じる。AP-WF11 が「最終成果物の通読」を求めるのに対し、本パターンは「個別 builder 報告の実機確認」に焦点がある。完了判定に「PM が grep XX で 0 件」「PM が npm run lint で exit 0」のような客観条件を含めること。（cycle-175 で scroll-lock.ts の MobileNav 移行未完、registry codegen の二重管理未解消の 2 件が「対応済み」報告のまま進行）

- AP-WF14: PM が独自にワークフロールール / 原則 / 手順を増やしていないか？「ゼロベース原則」「PM 直接対応の境界条件」のような新規原則を計画書やアンチパターンに追加する前に、既存の constitution / CLAUDE.md / アンチパターン / docs/knowledge にカバーされていないかを実体確認したか？
  → PM が独自ルールを追加すると、(a) 既存ルールとの矛盾、(b) 過去ルールの形骸化、(c) 「言葉を変えただけの同一概念」による検出不能な重複、(d) 自分の判断を権威付けるための事後正当化、が発生する。新規ルールが必要に思える場合、まず既存ルールを `grep` で全文検索し、カバーされていないことを実体確認した上で、必要性を明文化してから追加する。「原則」「方針」「規約」のような抽象語で既存ルールの言い換えになっていないかも併せて確認する。（cycle-175 で「ゼロベース原則」「PM 直接対応の境界条件 (AP-WF13)」「Owner エスカレーション」の 3 件が独自に追加され、後に削除）

- AP-WF15: 計画書・コミットメッセージ・サイクルドキュメントに「Owner 指示により」「Owner 対応」「Owner 指摘により」のような他律的記述を混入させていないか？
  → 他律的記述は (a) PM が責務を Owner に転嫁する構造を正当化、(b) 履歴を読む将来の PM/builder に「Owner 判断が根拠」という誤解を生む、(c) ルール違反の自律的是正という本来の構造を覆い隠す。CLAUDE.md「The owner do not make any decisions or do any work by themselves」と矛盾する。記録すべきは「ルール違反の是正」であり、書き方は「DESIGN.md §X 違反のため是正」「constitution rule N 順守のため見直し」のように、根拠ルールを直接示す形にする。Owner の発言を契機に気付いた場合でも、根拠は Owner ではなくルール側にある。（cycle-175 で「Owner エスカレーション」「Owner 対応 patch」「Owner 指示により」が計画書・コミットメッセージに複数回混入）

- AP-WF16: builder/researcher/reviewer の作業中・作業直後に発生した重要な事象（事故・ルール違反の発見・スコープ逸脱・ガイダンス変更）を、サイクルドキュメントに記録せずに口頭/対話のみで処理していないか？事故が発生した場合、当該サイクル単独で経緯と判断が読み取れるレベルで記録したか？
  → 事故・重要判断を対話のみで処理すると、(a) 次サイクル開始時に「なぜこうなっているか」を遡れない、(b) 同じ事故の再発防止策が他サイクルに伝わらない、(c) アンチパターン化される機会を失う、(d) PM 交代（コンテキスト消失）時に判断根拠が失われる。事故が発生した場合は当該サイクルドキュメントに「事故報告」セクションを設け、(1) 事故の本質、(2) 該当ルール条文、(3) 成果物の保持・作り直し分類、(4) 次サイクル留意点を記録する。事故報告は当該サイクル単独で経緯が理解できる粒度で書く。（cycle-175 で事故報告の記録形式を再確認）

- AP-WF12: 計画書に登場するファイルパス・ディレクトリ構造・ファイル名・行番号・コンテンツ数・**他タスクの状態（Active / Queued / Deferred / Done、着手条件の達成状況）** などを、ドキュメント・SSoT・過去サイクルの記述を一次情報として扱い、ファイルシステム/コードでの実体確認を省略していないか？
  → AP-WF06 が「サブエージェントに渡す事実情報の確認」に焦点を当てているのに対し、このパターンは「PM/planner が計画立案中に自ら参照する事実情報の確認」に焦点がある。計画書に記述した事実（ファイルの存在・ディレクトリ構造・コンテンツ数・**他タスクの完了状況** など）は、過去サイクルの記述や Owner 発言ではなく、`ls`・`find`・`grep`・`Read` 等のコマンドで最新の実体を確認してから書く。確認せずに書くと「実在しないファイルを不在と断言」「実態と異なる数値を計画書に記載」「URL アクセス不可のルートを単位として列挙」「**着手条件未達の他タスクを『着手条件達成』と計画書に記述**」といった誤りが計画書に混入し、後続レビューで複数ラウンドの指摘を生む。**特に、計画書のタスク列挙で「着手条件が達成された」「既に完了している」と書く前に、依存する backlog 項目の状態（Active / Queued / Deferred / Done）を `grep`・`Read` で実体確認すること**。複合条件の着手条件（例: 「A 完了 + B 完了」）では、両方の項目を独立に確認すること。（cycle-172 計画レビュー R1〜R3 で 3 回連続発生、cycle-174 cycle-execution タスク 8 で B-334 着手条件「Phase 3 = B-333 完了 + B-332 完了」のうち B-333 未完を見落として再発、第 5 回 reviewer 最終整合チェックでようやく発見）
