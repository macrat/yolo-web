# ワークフローのアンチパターン

このチェックリストは、作業の進め方に関する過去の失敗パターンを記録したものです。サイクル完了時やレビュー時に参照し、プロセス上の問題がないかを確認してください。

- AP-WF01: 最後の修正のあと、それがどんな修正であったとしてもレビューを実施したか？ すべての指摘事項に対応したか？
  → 「早く完了させたい」バイアスがレビュー省略・手順ショートカットを引き起こし、品質問題を見逃す。（cycle-118, 129, 147, 148, 159, 161, 168, 171, 181で実際に発生）

- AP-WF02: すべての変更を来訪者目線でレビューしたか？ 過去の失敗事例を参照させたか？
  → レビューの品質は回数ではなく観点の設計で決まり、形式的チェックだけでは本質的な問題を検出できない。（cycle-155, 162, 168, 175, 181, 184で実際に発生）

- AP-WF03: builderへの指示に型定義、JSXコード全文、CSSクラスの全定義、テストケースの詳細仕様など「どう実装するか」の具体的コードを含めていないか？
  → 過剰に具体的な指示はbuilderの設計判断やバイアス検出の機能を無効化し、PMの誤りがそのまま成果物に反映される。（cycle-76, 122, 130, 143, 144, 145, 146, 175, 181で実際に発生）
  **射程の線引き**: 「DESIGN.md トークン置換マップ」「APIシグネチャ」「外部仕様への適合（公式 docs 引用付き）」のような **設計判断の余地が PM 側に閉じている内容** は、builder に渡すべき事実情報であり AP-WF03 違反ではない。一方、`<html lang="ja">` の attribute 値、`style={{ display: "flex" }}` のような inline style 値、JSX の構造、CSS クラス名、テスト assertion 文言までを literal で確定するのは AP-WF03 違反。判断基準: 「同じ要件を別の builder に渡したら別の実装になる余地があるか」を自問し、余地がない literal 単位の指示は減らす。

- AP-WF04: サブエージェントの正式な完了通知（task-notification）を受け取ってから次ステップに進んでいるか？`git diff` でファイル変更を確認しただけで完了と判断していないか？「移行完了」「集約完了」「重複解消」のような構造的変更の報告は、`grep`/`find`/`Read` で実態を確認するまで完了と扱わない。
  → 出力ファイルの途中読み取りで「完了」と判断すると、未完成の成果物に対してレビュー・コミット・デプロイが進行する。構造的変更の報告は「ファイルが変更されている」だけでは検証として不十分で、PM が grep で旧呼び出しの残存を確認するまで承認しない。（cycle-142, 146, 175で実際に発生）

- AP-WF05: UIやデザインに影響がある変更をした場合、すべての画面をモバイル/PCとライトモード/ダークモードの組み合わせで実際のスクリーンショットを撮ってレビューしたか？
  → モバイルのみの確認でデプロイすると、デスクトップやダークモードで重大な表示問題が後から発覚する。撮影自体の網羅だけでは不十分で、撮ったものを来訪者目線の観点リストで通し評価する段階が品質を決める。**画像と記述の照合誤り（cycle-185 で 2 度発生）も同型のパターンで、画像 1 枚ずつ実体観察 → 記述 → 再照合のサイクルを意識的に回さないと検出できない**。（cycle-129, 147, 148, 152, 171, 175, 180, 181, 185で実際に発生）
  **網羅性ルール**: UI 変更がある対象ページ N について **N × {w360, w1280} × {light, dark} = N × 4 枚** を必ず撮る。撮影漏れがあった場合は cycle-completion を進めず B-333-9 等の検証フェーズに戻して撮り直す。スクショは `./tmp/` 配下に保存し、リポジトリルートに残してはならない（残った場合は速やかに削除し、別 commit でクリーンアップする）。
  **着手前撮影ルール（cycle-185 で運用ルール昇格）**: UI 変更を伴うサイクルでは **kickoff 直後（B-XXX-1 着手前）に「移行前」状態の N×4 枚を必ず撮る**。これは移行後との対照を「同等以上」評価するための必須資料。取り忘れた場合のみ例外フローとして `git stash` または `git worktree add <kickoff-commit>` で過去状態を復元して撮影する（`git stash` は commit 後では使えないため、commit 後に取り忘れた場合は worktree 復旧が事実上唯一の手段になる）。例外フローで撮ったか否かをサイクルドキュメントに明記する。

- AP-WF06: サブエージェントに渡す事実情報（数値・日付・コンテンツ内容等）をファイルやコマンドで事前に確認したか？推測や記憶で伝えていないか？
  → 未確認の情報がサブエージェントに渡ると、調査レポートや計画が誤った前提で作成され、将来のサイクルにも誤りが伝播する。（cycle-162, 163, 169, 171で実際に発生）

- AP-WF07: サブエージェントに複数の独立したタスクをまとめて依頼していないか？1エージェント1タスクの原則を守っているか？逆に、同一ファイルへの変更を含む複数タスクを別エージェントに**並行アサイン**していないか？
  → 複数タスクの一括委任は品質低下とトレーサビリティの喪失を引き起こす。個別の問題を切り分けられなくなり、レビューの精度も下がる。逆方向の同型として、同一ファイルへの変更を含むタスクを並行アサインすると競合が発生して品質を損なう。同一ファイルを触る複数タスクは「並行不能」として明示し、同一エージェントに直列で依頼する。（cycle-163, 181, 184で実際に発生）

- AP-WF08: PMがサブエージェントの作業を代行・改変していないか？サブエージェントの成果物を要約・解釈してファイルに保存していないか？
  → PMがサブエージェントの役割を代行すると、レビューの独立性が損なわれる（自分が書いたものを自分でレビューする構造になる）。成果物の改変はresearcherの調査結果の信頼性を破壊する。（cycle-163, 168で実際に発生）

- AP-WF09: アンチパターンチェックリストを形式的に通過させていないか？「推測」とラベルを付けるだけで未検証の前提を「対応済み」にしていないか？既存コンテンツの確認を「確認した」と書きながら実際には確認していないなど、チェック項目の実質的な検証を省略していないか？用語を変えて実質的に同じことをしていないか（「フレームワーク」を「方法論」と呼び替える、「長期計画」を「方向性」と呼び替える等）？
  → チェックリストの通過が目的化すると、過去の失敗パターンを認識しながら同じ失敗を繰り返す。形式的なラベル付け（「推測」「未検証」等）は誠実さの装いになるが、検証せずに計画の優先順位を決めている構造は変わらない。名称の変更は同一構造を検出不能にする特に危険な形態であり、表面的な用語ではなく実質的な構造が過去の失敗と同じでないかを検証する必要がある。チェック対象の範囲を恣意的に絞る（固有名詞のみ確認して構造前提の記述は見過ごす、既知パターンのみ照合して未知の変形は素通りさせる等）ことも同じ構造の一種である。（cycle-165, 166, 168, 169, 171, 175, 181, 184で実際に発生）

- AP-WF10: 新しいタスクを既存サブエージェントへのSendMessageで継続させていないか？同じエージェントに連続して異なるタスクを投げてコンテキストを蓄積させていないか？
  → SendMessageで継続するとコンテキストが蓄積して性能が急速に劣化する。新しいタスクには新しいエージェントを起動し、SendMessageは同一タスクの軽微な修正・追加質問のみに限定する。（cycle-168で実際に発生）

- AP-WF11: PMは自分が責任を持つ成果物（外部公開物・SSoT・エージェント定義・設計書など）の承認判断前に、最終成果物を自分自身で通読確認したか？reviewerやビジュアルチェックの通過を自分の判断の代用にしていないか？特に、整合する必要がある複数ファイル（例：`(legacy)/layout.tsx` と `(new)/layout.tsx`、共通コンポーネント新旧版、metadata と sitemap など）は、並べて差分を読み合わせる手順を取ったか？
  → レビューや機械チェックで検出されない本質的問題（記事であれば視点立脚・主張の整合、計画書や設計書であれば前提と結論の整合、エージェント定義であれば意図の伝わり方など）は、PMが自分で読まないと気づけない。複数ファイルが整合する必要がある成果物では、並べ読みをしないと気付けない欠落（GA・JSON-LD・provider 群の片方欠落等）が見落とされる。（cycle-168, 169, 171, 175, 180, 181, 184で実際に発生）
  **並べ読みの成果物化**: 「PM が並べ読みした」という宣言だけでは検証不能。並べ読みを行う際は、(i) 対比対象のファイル名と行範囲、(ii) 計画書 / 仕様の確定リスト、(iii) 実装に存在する要素、(iv) 差分（不一致 / 欠落）を 4 列のテーブルとして `./tmp/` または cycle ドキュメント内に残す。これによりレビュアーが「PM が本当に並べ読みしたか」を後から検証できる。cycle-180 では計画書 L353 の「採用ブロック確定リスト」（ThemeProvider / GoogleAnalytics 含む）と global-not-found.js の実装を並べ読みすれば R1 まで持ち込まずに済んだ Major-2（ThemeProvider/GoogleAnalytics 欠落）が、並べ読みなしで通過した。

- AP-WF12: 計画書に登場するファイルパス・ディレクトリ構造・ファイル名・行番号・コンテンツ数・**他タスクの状態（Active / Queued / Deferred / Done、着手条件の達成状況）** などを、ドキュメント・SSoT・過去サイクルの記述を一次情報として扱い、ファイルシステム/コードでの実体確認を省略していないか？
  → AP-WF06 が「サブエージェントに渡す事実情報の確認」に焦点を当てているのに対し、このパターンは「PM/planner が計画立案中に自ら参照する事実情報の確認」に焦点がある。計画書に記述した事実（ファイルの存在・ディレクトリ構造・コンテンツ数・**他タスクの完了状況**・**フレームワーク API の挙動** など）は、過去サイクルの記述や Owner 発言ではなく、`ls`・`find`・`grep`・`Read` 等のコマンドで最新の実体を確認してから書く。確認せずに書くと「実在しないファイルを不在と断言」「実態と異なる数値を計画書に記載」「URL アクセス不可のルートを単位として列挙」「**着手条件未達の他タスクを『着手条件達成』と計画書に記述**」「**フレームワーク API の挙動を思い込みで記述**（例: Next.js の `permanentRedirect()` を 301 と書いたが実際は 308）」といった誤りが計画書に混入し、後続レビューで複数ラウンドの指摘を生む。**特に、計画書のタスク列挙で「着手条件が達成された」「既に完了している」と書く前に、依存する backlog 項目の状態（Active / Queued / Deferred / Done）を `grep`・`Read` で実体確認すること**。複合条件の着手条件（例: 「A 完了 + B 完了」）では、両方の項目を独立に確認すること。（cycle-172, 174, 181, 183, 184, 186で実際に発生）

- AP-WF13: 並行アサインした独立タスクの担当 builder が、自タスクのスコープを越えて隣接タスクの作業まで実施していないか？PM の指示で、スコープ外の操作（典型的には他タスクの `git mv` / 大規模リネーム / 共有設定ファイルの編集 など）を抑止しているか？
  → 隣接タスクの作業に手を出すと、commit 構成が当初の rollback 戦略と乖離し、`git revert` で 1 commit を巻き戻すと隣接タスクまで巻き戻ってしまう（または逆に巻き戻したくないタスクが巻き戻らない）といった事故が起きる。（cycle-183で実際に発生）

- AP-WF14: reviewer が **数値・件数・状態** に関する指摘を採用判断するとき、**自ら一次集計を独立に行わずに、複数レポートの相対比較だけで採否を決めていないか？**
  → 「最新優先」「詳細優先」「整合性チェック」だけで採否を決めると、複数ソースが揃って同方向に間違っている時に検出できない。レポート間の矛盾を見つけたときに「最新を採る」だけで吸収してはならず、必ず一次集計で再確認する必要がある。（cycle-183で実際に発生）

- AP-WF15: サイクル完了処理を進めた後（あるいは push 後）に来訪者影響のある補修課題が見つかった場合、「同サイクル延長」「別サイクル独立」どちらに振るかを思いつきで決めず、明示的な判断軸（来訪者影響の有無 / 当該サイクル目的範囲との整合 / 本格対応の規模 / 暫定対応長期化への歯止め策）で検証したか？
  → サイクル完了後の補修は時間的圧力が掛かりやすく、「すぐ直したい」バイアスで同サイクルに無理に押し込むと、本来別サイクルにすべき大規模変更が中途半端な暫定対応として残置される。逆に「もう完了したから次サイクル」と機械的に分割すると、来訪者影響がある問題を放置する事故になる。（cycle-183で実際に発生）
