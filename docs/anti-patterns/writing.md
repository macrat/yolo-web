# 記事執筆のアンチパターン

このチェックリストは、ブログなどの記事のレビュー時に参照し、過去の失敗パターンに陥っていないかを確認するためのものです。

- AP-W01: 記事の主題が「自分たちが何をやったか」の報告になっていないか？
  → 読者はプロジェクトの内部事情に興味がなく、「読者が何を持ち帰れるか」が記事の価値を決める。（cycle-51, 55, 75, 253で実際に発生）

- AP-W02: 読者が共有していない前提の上で話を進め、読者が自分の状況に応用できない構成になっていないか？ 固有名詞（プロジェクト名・独自略称）だけでなく、「スプリント」「レビュー回」「運用会議」のように一般語として流通していても特定の組織・プロジェクト内で独自の意味が固定化された語（社内用語化した日常語）も同じ問題であり、固有名詞検出では引っかからない点に注意する。さらに語の問題にとどまらず、「何度も差し戻された」「〇〇フェーズで気づいた」のような、ワークフロー・ロール分担・工程区分・経緯の構造そのものが読者に共有されていない前提として使われている記述も同様に判定範囲に含める。語を一般化しても、その語が置かれた文脈の構造が内部固有であれば前提依存は残る。
  → 読者が共有していない文脈を前提にすると、語を抽象化しても記事の応用可能性は回復しない。（cycle-148, 168, 255で実際に発生）

- AP-W03: コンテンツの想定読者と、実際にそのコンテンツを検索・閲覧する人がずれていないか？
  → 作り手の専門領域に引きずられると、実際の読者層の大半が理解できないコンテンツになる。（cycle-117, 118で実際に発生）

- AP-W04: 記述と実態の乖離が起きていないか？ 具体的には、事実・数値・経緯が記憶や推測ではなく一次ソースで裏付けられているか。根拠や反証として引用する研究・文書が記事の主張と同じ現象を扱っているか（用語が似ているだけで測定軸が異なる研究を根拠にしていないか）。成果物として最終的に実装・反映されなかった設計・意図・計画を、「そうした」「そう決めた」と反映済みのように主張していないか（中間案や検討経緯は参照してよいが、採用されなかったものを採用されたかのように語ることは別問題）。「〇点に絞った」「〇つの柱」「〇項目に整理」「前者と後者」のような複数の要素が並存することを前提にした構造的記述が実態と一致しているか（記述のリズムのために実質的に一つの事柄を複数に分けていないか）。これらに限らず、書いてあることと実態が一致していない箇所はないか。
  → 記述と実態の乖離はどの観点でも、読者の信頼を損なうか記事の主張を無効化する。（cycle-69, 70, 95, 168, 187, 253, 255で実際に発生）

- AP-W05: 執筆前にターゲット読者の定義ファイルを実際に開いて確認せずに執筆していないか？
  → 過去の記憶に依存すると、削除・変更されたターゲット定義に基づいて書いたり現存するターゲットを見落としたりする。（cycle-148, 253で実際に発生）

- AP-W06: 書き手の立場や視点を明示して書く記事で、タイトル・description・書き出し・章立て・結びなど記事の骨格を構成する箇所の視点が、宣言した立場ではなく外部観察者のものになっていないか？ 一人称記事で「わたし」と書くことはその典型例だが、体験談・現場の声・立場を明示した記事でも同じ問題は起きる。
  → 言葉の上で立場を宣言しても、記事の骨格が外側の視点で組み立てられていると、読者にその立場から書かれた効果が届かない。（cycle-168で実際に発生）

- AP-W07: 読者の一日を想像して書いたつもりが、一人だけの曖昧な想像にとどまっていないか？ 「金曜の夜」「締切前」のような場面設定フレーズを、具体的な感情・行動・動機を伴わずに装飾として貼っていないか？
  → 一人だけの曖昧な想像は読者層の偏りを生み、具体的な感情・行動・動機を欠いた場面設定は装飾になり読者の状況に届かない。（cycle-168で実際に発生）

- AP-W08: 記事の核メッセージが最初の一文で読者に届くか？ 本題に至るまで迂遠な導入や予防線を重ね、主張の芯が見えないまま読み進ませていないか？
  → 未検証の仮説であっても、強い主張と慎重な扱いは両立する。予防線で記事全体を埋めると、伝えたいことが薄まる。（cycle-168, 255で実際に発生）

- AP-W09: 記事内で提示する主張が、同じ記事内で提示する観察と整合しているか？ 主張に合わせて観察を歪めたり、観察と直接衝突する主張を並べていないか？
  → 記事の内部で自己矛盾が起きると、読者は主張を信頼できない。（cycle-168, 187で実際に発生）

- AP-W10: タイトルが、このサイトを知らず内容に予備知識のない読者（検索結果やSNSフィードでたまたま見る人）にとって、「何の話か」「自分に関係があり読む価値があるか」が一目で伝わるものになっているか？ 内部都合の語・抽象語（一般読者が意味を取れない比喩や設計用語）が混じっていないか？ 字数・好奇心ギャップ・キーワード前方配置といった機械的なSEOチェックを満たしていても、読者に意味が通らないタイトルは失格である点に注意する。
  → SEOチェックリストの形式的充足と、赤の他人にとっての伝達性は別物。前者だけ満たして後者を欠くと、価値ある内容でも読まれない。レビューが機械的項目の充足だけで通すと見逃す。（cycle-221で実際に発生）

- AP-W11: 記事が本当に伝えたい一つの芯（核メッセージ）に、タイトル・各見出し・本文・結びがすべて奉仕しているか？ 記事の各部が、芯と無関係な機能・手順・小ネタの羅列になっていないか？ 技術的な具体が、芯を読者に納得させる証拠として機能しているか（具体の列挙が目的化していないか）？ 見出しだけを拾い読みして芯のストーリーが追えるか？
  → 芯を一つ宣言しても、タイトル・本文・結びが別々の方向を向いていると、読者には芯のない文章として届く。書き手の「魂」は宣言ではなく、骨格すべてが一つの芯に奉仕することで初めて読者に伝わる。（cycle-221, 253, 255で実際に発生）

- AP-W12: 「記事を書く」と決める前に、(a) 誰に向けた何の一文か（単一の主題と読者）を確定し、(b) その価値が既存記事で既に読者へ提供されていないかを確認したか？ 「学びがある題材だ」「具体例で語れる」は、書く動機にはなっても主題の確定ではない。複数の理由を並べて書き始めると主題が揺れ、既存記事と重複すれば読者に新しい価値を与えず「似た記事が並んで迷わせる」状態を作る。
  → 主題と読者を決めず、既存記事の重複確認もせずに書き始めると、何度書き直しても芯が定まらず、最悪の場合「既に提供済みの内容を再生産しただけ」になる。執筆の可否と主題は、最初の一文を書く前に確定・確認する。（cycle-221, 253, 255で実際に発生）

- AP-W13: 記事の `published_at` / `updated_at` は、`date` コマンドで取得した実時刻に基づいているか？ PM がサブエージェントに「12:30 に設定して」のような具体的な時刻文字列を渡したり、サブエージェントが現在時刻を取得せず予想・丸めた時刻を書いたりしていないか？ とくに、コミット時点より未来の時刻になっていないか（検索エンジンのペナルティに直結する致命傷）。新規記事（その記事を初めて書くとき）で `published_at` と `updated_at` が同一値になっているか（改訂を装って updated_at だけ進めていないか）？
  → 日時は「それらしい値」を作文してよい欄ではなく、実時刻の記録である。指示する側が具体的な時刻を渡すと、サブエージェントはそれを実時刻と照合せず転記し、未来時刻や published≠updated の偽装が生まれる。日付という事実情報を確認せず渡している点で、これは AP-WF06（サブエージェントに渡す事実情報の事前確認）の記事ドメイン版でもある。値そのものを渡さず、設定手順（正本は `.claude/rules/blog-writing.md`「published_at / updated_at の設定ルール」）に委ねる。コミット前に frontmatter の時刻を `date` 実測値・git log と照合すれば必ず検出できる。（cycle-241で実際に発生）

- AP-W14: 既存記事を書き換えるとき、それは「同じ話の改善・更新」か、それとも別の過程・別の話を元記事に上書き／合体させていないか？ 元記事が記録していた一つの過程と、その後に起きた別の過程（評価・撤退・方針転換など）は別の話である。
  → 別の話を元記事へ上書きすると、(a) 元の過程の一次記録が失われ、(b) 同一 URL のトピック総入れ替えという SEO 上の悪手（被リンクのトピック関連性喪失・古い published_at 温存による日付偽装）になり、(c) ブックマークや旧リンクからの再訪者が別物に着地する bait-and-switch になる。（cycle-243で実際に発生）

- AP-W15: 内容・構成の点検（AP-W01〜W14）を通過しても、表層の文体がAI臭（AI slop）に陥っていないか？ 具体的には、(a) 立場を保険で薄めていないか（「ケースバイケース」「〜の可能性がある」の積み重ね、全方位肯定）、(b) モノを主語にして書き手が消えていないか（「データが示している」「本記事では」などの false agency）、(c) 抽象語・偏愛語・比喩テンプレで着地していないか（「本質」「解像度を上げる」「羅針盤」）、(d) 段落の長さ・温度・結論が均一でリズムが平坦になっていないか（二項対比でリズムを作っていないか）、(e) 咳払い・メタ構造宣言・定型句・記号の残骸で字数を稼いでいないか（「結論から言うと」「以下の3つの観点から」「いかがでしたでしょうか」、全角ダッシュ）、(f) 命題型の見出しになっていないか。具体リストは `docs/japanese-ai-slop.md` にある。
  → 内容が正確でも、表層がAI臭いと読者は「機械が書いた薄い文章」と判断して離れる。これは公開前に下書きを直せば必ず回復できる失敗であり、内容点検とは別レイヤーとして独立に点検する必要がある。

- AP-W16: 一人称で自分の経験・経緯・過程・因果を書く記事で、それを一次資料（git log・過去の記事・サイクル記録・チャット履歴）で確認せず、記憶や推測で組み立てていないか？ とくに、(a) 出来事の前後関係・因果の順序、(b)「まずXして、次にYした」という過程、(c) 何がきっかけで何に気づいたか、を実際には確かめていないのに、もっともらしい物語として書いていないか。「これを先にやった」「やってみたら足りなかった」のような順序・因果は、記憶では容易に逆転・捏造される。
  → 一人称で語る経緯・過程は、確認しなければ容易に捏造される。AIは記憶を持たず、空いた場所にもっともらしい筋書きを自分で埋めてしまう。これは記述と実態の乖離（AP-W04）であり、書いてある経緯が事実と違えば記事は読者の信頼を失う。一次確認は最初の一文を書く前に済ませる。

- AP-W17: 引用・参考にする外部ソースの著者や系譜を、別個の独立した出所だと決めつけていないか？ 同一著者の一連の仕事（分析記事とその実装、本家とその派生）を、別人の別ソースのように扱って貢献を分散させていないか。
  → 出典の人間関係を取り違えると、貢献の帰属を誤り、読者がソースに当たったとき前提が崩れる。著者・系譜は、名称・URL・ライセンスと同じく一次確認する対象である。

- AP-W18: 記事が依拠・参照・リンクする基準（文体ガイド・チェックリスト等）を、その記事自身が守れているか？ とくに見出し・書式など、基準が具体的な形式を定めている箇所を本文が破っていないか。
  → 基準にリンクしながらその基準に反する文章は、最も直接的に読者の信頼を損なう自己矛盾である。レビューは、成果物をそれが掲げる基準そのもので一項目ずつ照合し、「叙述型だから許容」のような例外解釈で素通りしない。

- AP-W19: 文体のAI臭（記号・偏愛語・書式）を消したことに満足して、読者が持ち帰れる主張（反証可能な一文）を一つも置いていないか？ 文体を磨くのは安全で、主張を置くのは批判されうるから怖い。怖いほうから逃げて磨くほうに時間を使い、文体がきれいになったことを進捗と錯覚しやすい。きれいなのに何も主張していない文章は「人の顔をした空白」で、あからさまにAIが書いた文章より薄気味悪い。
  → AI臭は表層（文体）と深層（書き手の不在＝主張の不在）の二層からなり、表層を消すだけでは消えない。文体の自然さは最低ラインで、その上に恐れず主張を置いて初めてAI臭が抜ける。文体クリーニングと主張を置くことは別の作業であり、それぞれ確かめる。
