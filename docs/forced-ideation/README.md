# 強制発想法によるサイトコンセプト再構築 (cycle-66)

## 目的

cycle-66にて、yolos.netのサイトコンセプトをゼロベースで再構築するために実施した強制発想法プロセスの記録。既存コンテンツや過去の方針に引きずられることなく、網羅的かつ公平にコンテンツアイデアを探索し、データ駆動でサイトの方向性を決定することを目指した。

## 手法の説明

**強制発想法（組み合わせ法）** を採用した。複数の独立した軸を定義し、その全組み合わせを機械的に生成することで、人間のバイアスや思い込みに左右されないアイデア空間の網羅的な探索を行う。

## 軸の定義

4軸の掛け合わせにより **1,728通り** の組み合わせを生成した。

| 軸           | 要素数 | 要素一覧                                                                                                                                                           |
| ------------ | ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 地域         | 3      | 日本語圏, 英語圏, 多言語                                                                                                                                           |
| テーマ       | 12     | 日本文化, 音楽・サウンド, 科学・数学, 言語・文章, ビジュアル・色彩, 地理・旅行, 歴史・人物, 健康・生活, 創作・ものづくり, 動物・自然, カジノ・確率, サブカルチャー |
| フォーマット | 8      | デイリーゲーム, ミニゲーム, クイズ, ツール, 診断・占い, 辞典・リファレンス, シミュレーション, ジェネレーター                                                       |
| 目的         | 6      | 遊ぶ・楽しむ, 学ぶ・知る, 作る・創る, 調べる・変換する, 共有する・競う, リラックスする                                                                             |

計算: 3 x 12 x 8 x 6 = **1,728通り**

## プロセスの流れ

### Phase A: 軸の設計とバランス調整

4つの独立した軸を設計し、各軸の要素数がバランスを保つよう調整した。特にテーマ軸は各テーマが同数の組み合わせを持つようにすることで、特定テーマへの偏りを防止した。

### Phase B: 全組み合わせの機械的生成

1,728通りの組み合わせをPythonで機械的に生成し、`seed=42` でランダムシャッフルした。人間による取捨選択を排除し、全ての可能性を平等に扱うことを保証した。

### Phase C: Haikuモデルによる並行評価

1,728件を32チャンク（各54件）に分割し、Claude Haikuモデルで並行評価を実施した。各組み合わせについて「Webコンテンツとして成立するか」を判定し、成立する1,525件を抽出した。

### Phase D: テーマ別統合

1,525件の成立候補をテーマ別に集約・統合し、重複排除や類似アイデアの統合を行って117コンセプトに絞り込んだ。

### Phase E: 市場調査による選別

117コンセプトに対して市場調査を実施し、独自性・需要・実装可能性・継続性・品質達成可能性の5軸で評価。31件がAランク（総合スコア3.8以上）として選出された。

### Phase F: ゼロベースでのコンセプト案策定

31件の新規候補と既存コンテンツ9件の計40件を匿名化（P01〜P40にリナンバリング）し、ランダムシャッフルしたうえで、サイトの既存情報を一切参照できない環境でコンセプト案4案を策定した。最終的に案A「今日のひらめき -- 日本語デイリーパズルポータル」を推奨案とした。

## バイアス防止の工夫

### 軸の要素数均等化

各テーマが同数の組み合わせを持つよう設計し、特定テーマが過剰に評価される偏りを排除した。

### ランダムシャッフルによる評価順序バイアスの排除

`seed=42` による決定論的シャッフルで、評価チャンクの順序がアイデアの採否に影響しないようにした。

### Phase Fのホワイトリスト方式によるコンテキストバイアス排除

Phase Fでは、エージェントがサイトの既存コンテンツや過去の分析ドキュメントを参照することによるバイアスを完全に排除するため、ファイルアクセスをホワイトリスト方式で制限した。

この方式に至るまでに3回のやり直しを経ている:

1. **第1回（失敗）**: 通常のエージェント実行。既存の `docs/` ディレクトリを参照し、過去のサイト方針に引きずられた提案となった。
2. **第2回（失敗）**: ブロックリスト方式で特定ファイルの読み取りを禁止したが、ブロックリストに漏れがあり、間接的に既存情報を取得してしまった。
3. **第3回（成功）**: ホワイトリスト方式に切り替え、エージェントが読み取れるファイルを `constitution.md` と匿名化された候補リスト（`/tmp/phase_f_anonymized_items.txt`）のみに限定した。カレントディレクトリの読み取りを完全に禁止することで、コンテキストバイアスを排除した。

## 将来の活用方法

- **コンテンツ拡張時の参照**: 新コンテンツを検討する際、`anonymized-candidates.md` の40候補を見直して、未採用だが有望なアイデアを再発見できる
- **コンセプト見直し時の参照**: サイトの方向性を再検討する際、`phase-f-concept-proposals.md` の4案を改めて比較検討できる
- **プロセスの再利用**: 新たなアイデア探索が必要な場合、同じ強制発想法プロセス（軸の再設計 → 全組み合わせ生成 → 並行評価 → 統合 → 市場調査 → ゼロベース策定）を再実行できる
