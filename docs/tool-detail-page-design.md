## 前提: 旧構造は採用根拠にならない

cycle-190 の核心的失敗は「旧構造（cycle-160 の 3 ゾーン / FaqSection / RelatedTools / RelatedBlogPosts / 現行 ToolLayout）が存在すること」を採用根拠にしたことにある。
本ドキュメントでは、各要素の採用根拠を **新コンセプト「日常の傍にある道具」下での target user の likes に応えるか / dislikes と衝突しないか** のみに求める。
旧構造の存在は採用根拠にも不採用根拠にもならない。

## ターゲット 2 経路の要素別要求（T-A-発散 ステップ 1）

### 経路 1: 特定の作業に使えるツールをさっと探している人（検索流入・初訪中心）

出典: `docs/targets/特定の作業に使えるツールをさっと探している人.yaml`（Read 直後に引用）

#### likes → 要求

- **ページを開いた瞬間に入力欄が見えて、すぐ使い始められること**
  - 要求 1-L1: ファーストビュー（スクロールなし・ビューポート上端〜概ね 800px 以内）に「ツール本体の入力インターフェース」が存在すること
  - 要求 1-L2: ツール本体より上位に解説テキスト・装飾バナー・ナビゲーションリンクブロック等を配置しないこと
  - 要求 1-L3: ページタイトル・パンくず等の最小限の識別要素は許容するが、視覚的な重さを最小化すること

- **コピペで結果を受け取って、すぐ元の作業画面に戻れること**
  - 要求 1-L4: 結果エリアに「コピー」ボタンまたは同等の UX が存在すること
  - 要求 1-L5: コピー完了フィードバックが明確に表示され、「コピーできたか不安で確認する」往復コストが発生しないこと

- **余計な説明や装飾がなく、用事だけ静かに片付けられる画面**
  - 要求 1-L6: 補助情報（使い方説明・FAQ・関連コンテンツ・AI 注記等）はツール本体より下位または折りたたみに配置し、デフォルトで視線に入らないこと
  - 要求 1-L7: デザインの装飾（バナー・アイキャッチ画像・図解など）を最小化し、機能要素が視覚的に主役になること

- **入力した内容が外部に送られないなど、不安なく使えること**
  - 要求 1-L8: ローカル処理 / プライバシー保護の旨を、最小限の文字数で視認できる位置に表示すること（長文は不要）

- **結果の根拠や前提が必要最小限だけ添えられており、信頼して使えること**
  - 要求 1-L9: 出典・更新日・処理方式の概要（例: 「ローカル処理」「XX規格準拠」等）を結果表示エリア付近に一行単位で表示できること

#### dislikes → 要求

- **開いた瞬間に広告やポップアップで目的の入力欄が埋もれていること**
  - 要求 1-D1: ファーストビューに広告枠・モーダル・オーバーレイを置かないこと
  - 要求 1-D2: ページロード後にツール本体が視覚的に最優先の要素であること

- **使い始めるのに会員登録やアプリのインストールを求められること**
  - 要求 1-D3: ツール利用に事前操作（登録・ログイン・インストール・有料化壁等）が不要であること

- **仕様や前提が曖昧で、出てきた結果を信用していいのか判断できないこと**
  - 要求 1-D4: 処理の前提・出典・精度の限界を隠さず、ただし過剰にならず表示すること
  - 要求 1-D5: AI 生成コンテンツが含まれる場合は Constitution Rule 3 に基づく AI 注記を表示すること（不信感の原因となる「AI 製かどうか分からない」状態を防ぐ）

- **似たようなツールが並んでいて、どれを使えばよいか迷わせること**
  - 要求 1-D6: 詳細ページ内の関連コンテンツ表示は「役割が重複するツール群の羅列」にならないこと。推薦根拠が明確な場合のみ表示すること

- **ツール冒頭に長い解説記事が挟まっていて、すぐ使えないこと**
  - 要求 1-D7: 使い方説明・機能説明・FAQ はツール本体より後に配置し、スクロールなしには目に入らないこと
  - 要求 1-D8: ページの第一画面で「このツールの説明を読む」を強制しないこと

### 経路 2: 気に入った道具を繰り返し使っている人（再訪・道具箱経由中心）

出典: `docs/targets/気に入った道具を繰り返し使っている人.yaml`（Read 直後に引用）

#### likes → 要求

- **サイト内のすべてのツールやゲームの操作性やトーン&マナーが一貫していること**
  - 要求 2-L1: ツール詳細ページのレイアウト・操作性・視覚的トーンが全コンテンツ（ツール / 遊び / 辞書）で統一された共通構造に従うこと
  - 要求 2-L2: 過渡期（(new) 化済み 1 件と (legacy) 残り 52 件が混在する状態）においても、visitor が操作の一貫性を体験できるよう、共通要素（ナビゲーション・フッター等）は変化させないこと

- **ブックマークしたURLを開けばすぐ目的のツールが表示されること**
  - 要求 2-L3: URL 構造が安定していること（移行後の URL が移行前と同一か、リダイレクト設定済みであること）
  - 要求 2-L4: ツール本体がページロード直後に最優先で表示されること（再訪者は既に使い方を知っているため、説明は不要）

- **前回入力した値や設定が残っていて、作業がさらに短縮されること**
  - 要求 2-L5: 入力値・設定値のローカルストレージ永続化を提供すること
  - 要求 2-L6: 永続化した値が次回ページロード時に自動復元されること

- **同じ入力に対して前回と同じ結果が返ってくること**
  - 要求 2-L7: ツールのロジック変更・更新時は更新内容と変更日を明記し、「結果が変わった理由」が visitor に分かる状態にすること

- **仕様変更があっても、これまでの使い方が壊されないこと**
  - 要求 2-L8: (legacy)→(new) 移行時に操作フロー・入出力仕様・URL を維持すること
  - 要求 2-L9: 破壊的変更が必要な場合は明示的な案内を表示すること

#### dislikes → 要求

- **動作が遅いツール**
  - 要求 2-D1: ツール本体のインタラクション（入力 → 結果表示）が体感的に即座（100ms 以内推奨）であること
  - 要求 2-D2: ページロードに不要なリソース（外部フォント・重い画像・analytics スクリプト以外の外部 JS）を配置しないこと

- **URLが変更されたコンテンツにリダイレクトが設定されておらずたどり着けなくなること**
  - 要求 2-D3: (legacy)→(new) 移行時に必ず旧 URL から新 URL へのリダイレクトを設定すること

- **以前と同じ入力なのに結果や挙動が前回と変わっていること**
  - 要求 2-D4: ツールのロジック・UI を更新した場合は「更新日」と「主な変更点」を詳細ページに表示できる構造を持つこと
  - 要求 2-D5: 意図しない挙動変更（デザイン変更で入力欄の位置や操作手順が変わる等）を QA で防ぐこと

## 要素発散リスト（T-A-発散 ステップ 2 + 3）

最低 12 件の要素を発散列挙する。各要素に以下を記録する。

- 出所タグ（「旧構造由来」「新コンセプト直接導出」「target likes/dislikes 由来」から 1 つ以上）
- 経路 1 likes 応答度: 応える / 応えない / 中立
- 経路 2 likes 応答度: 応える / 応えない / 中立
- 経路 1 dislikes 衝突: 衝突する / 衝突しない / 中立
- 経路 2 dislikes 衝突: 衝突する / 衝突しない / 中立
- 理由

**注意**: 各要素への評価は「旧構造にあったから」ではなく、上記ステップ 1 で導出した要素別要求（1-L1〜1-D8, 2-L1〜2-D5）に照らして行う。

---

### 要素 1: ツール本体（入力インターフェース）のファーストビュー優先配置

ツール固有の入力 UI（テキストエリア・フォーム・ボタン等）がページロード直後にビューポート内に収まるよう、ページ上部での優先配置を設計上の不変原則とする。

- 出所タグ: **新コンセプト直接導出**（「道具」＝まず使えることが価値の本体）、**target likes/dislikes 由来**（要求 1-L1, 1-L2, 1-D7, 1-D8, 2-L4）
- 経路 1 likes 応答度: **応える**（「ページを開いた瞬間に入力欄が見えて、すぐ使い始められること」に直接対応）
- 経路 2 likes 応答度: **応える**（「ブックマークしたURLを開けばすぐ目的のツールが表示されること」に対応）
- 経路 1 dislikes 衝突: **衝突しない**（解説・広告等をツール下に追いやることで dislike「冒頭の長い解説」「埋もれた入力欄」を同時解決）
- 経路 2 dislikes 衝突: **衝突しない**（再訪者は使い方を知っているため、ツールが即座に出ることで動作速度への不満も軽減）
- 理由: 「日常の傍にある道具」の「道具」性の核心は「取り出してすぐ使える」こと。ツール本体が最初に来ることは、サイトコンセプト（docs/site-concept.md）の「毎日使えるようによく考えられている」設計思想とも直接対応する。

---

### 要素 2: 最小限の使い方説明（初回ユーザー向け・ファーストビュー以下に配置）

「このツールで何ができるか」を 1〜3 行で示す説明。FAQ 形式ではなく「最初のアクションを起こすためのヒント」。ツール本体より下位に配置し、スクロールしなければ見えない設計とする。

- 出所タグ: **旧構造由来**（旧 FaqSection の使い方セクションに対応）、**target likes/dislikes 由来**（要求 1-L9, 1-D4）
- 経路 1 likes 応答度: **応える**（「結果の根拠や前提が必要最小限だけ添えられており、信頼して使えること」に対応。ただし長文化すると逆効果）
- 経路 2 likes 応答度: **中立**（再訪者は既に使い方を知っているため直接の価値は低いが、邪魔にもならない）
- 経路 1 dislikes 衝突: **衝突しない**（ツール以下に配置するため「冒頭の長い解説」にならない）
- 経路 2 dislikes 衝突: **衝突しない**（ページ上部でなければ再訪者の邪魔にならない）
- 理由: 旧構造由来タグが付くが、採用根拠は「旧版にあったから」ではなく「経路 1 の初訪ユーザーがツールの用途を確認して信頼して使い始めるために必要」という target 要求から。ツール本体の下に必ず置くことが条件。

---

### 要素 3: コピー / シェア UX（コピーボタン・コピー完了フィードバック）

処理結果をワンクリックでクリップボードにコピーできる UI。コピー完了を視覚的にフィードバックする（例: アイコン変化・トースト通知）。シェアボタンは必須ではないが、検討対象とする。

- 出所タグ: **target likes/dislikes 由来**（要求 1-L4, 1-L5）、**新コンセプト直接導出**（「道具」の完了体験 = 結果を持ち出して元の作業に戻る）
- 経路 1 likes 応答度: **応える**（「コピペで結果を受け取って、すぐ元の作業画面に戻れること」に直接対応）
- 経路 2 likes 応答度: **応える**（繰り返し使う中でのコピー操作が最短になることで「もう考えずに使えるようになった道具」体験に寄与）
- 経路 1 dislikes 衝突: **衝突しない**
- 経路 2 dislikes 衝突: **衝突しない**
- 理由: 「道具」の本来の価値は「使ったあと持ち出せること」。コピーボタンがない場合、ユーザーは手動選択 → コピーという余計な操作を強いられる。この要素はツールの機能性の一部として設計段階から組み込む必要がある。

---

### 要素 4: 入力値・設定値のローカルストレージ永続化

ユーザーが入力した値・選択した設定をローカルストレージに保存し、次回ページ訪問時に自動復元する機能。サーバー送信なし・ブラウザ完結。

- 出所タグ: **target likes/dislikes 由来**（要求 2-L5, 2-L6）、**新コンセプト直接導出**（「日常の傍にある」＝昨日と同じ状態でまた使える）
- 経路 1 likes 応答度: **中立**（初訪者には関係ない機能だが邪魔にもならない）
- 経路 2 likes 応答度: **応える**（「前回入力した値や設定が残っていて、作業がさらに短縮されること」に直接対応）
- 経路 1 dislikes 衝突: **衝突しない**
- 経路 2 dislikes 衝突: **衝突しない**（プライバシー保護の観点でもローカル保存はサーバー送信より安全）
- 理由: 新コンセプト「日常の傍にある道具」の「日常」性の体現として最重要な機能の一つ。他のオンラインツールサイトとの差別化要素でもある。ただし実装はツールごとに保存キーを設計する必要があるため、共通 I/F として設計段階で定義しておく必要がある。

---

### 要素 5: 出典・更新日・信頼性表示

ツールが依拠するデータ・規格・計算式の出典、最終更新日、処理の限界（精度・適用範囲等）を最小限の文字数で表示する要素。ツール本体の近傍（下部または注記エリア）に配置。

- 出所タグ: **旧構造由来**（旧版に存在した出典表示要素に対応）、**target likes/dislikes 由来**（要求 1-L9, 1-D4, 2-L7, 2-D4）
- 経路 1 likes 応答度: **応える**（「結果の根拠や前提が必要最小限だけ添えられており、信頼して使えること」に対応）
- 経路 2 likes 応答度: **応える**（「同じ入力に対して前回と同じ結果が返ってくること」の文脈で、更新日が変わっていれば変更があったことが分かる）
- 経路 1 dislikes 衝突: **衝突しない**（最小限の表示であれば「余計な説明」にならない）
- 経路 2 dislikes 衝突: **衝突しない**（更新日の明示はむしろ dislikes「以前と挙動が違う」への誠実な対応）
- 理由: 採用根拠は旧版の存在ではなく、「信頼して使える道具」としての必要性。特に経路 1 の「仕様や前提が曖昧で結果を信用できない」dislike への対処として不可欠。文字数は最小限に留め、詳細は折りたたみや別ページへ。

---

### 要素 6: AI 注記（Constitution Rule 3 準拠）

AI が生成・管理するコンテンツである旨の表示。Constitution Rule 3「Notify visitors that the website is run by AI as an experiment and that its content may be broken or incorrect」に従い必須。

- 出所タグ: **新コンセプト直接導出**（Constitution Rule 3 という設計上の不変制約）
- 経路 1 likes 応答度: **中立**（target yaml doesnt_know に「AI が自律運営している実験サイトであること」が明記されており、経路 1 ユーザーは AI 注記を見て使い始めをためらう可能性がある。コンパクト表示なら邪魔にならず誠実さの印象に寄与するが、目立つ警告表示だと使用ためらいを誘発するため、表示方法次第で応える / 応えない両方向に転じる）
- 経路 2 likes 応答度: **応える**（サイト一貫性の観点で全ページに統一された注記があることが期待される）
- 経路 1 dislikes 衝突: **衝突しない**（コンパクトに表示すれば「余計な装飾」にならない）
- 経路 2 dislikes 衝突: **衝突しない**
- 理由: Constitution Rule 3 は他のすべての設計判断より優先する不変ルール。採否の余地がなく必須要素。ただし target yaml 経路 1 doesnt_know に「AI が自律運営している実験サイトであること」が明記されており、表示方法が悪いと使い始めをためらわせるリスクがある。コンパクトな表示（フッター集約・1 行注記等）を前提に、使用ためらいを誘発しないデザインを収束段階で判断する。

---

### 要素 7: 関連コンテンツ（ツール / 遊び / ブログを横断した推薦）

現在閲覧中のツールと意味的に関連する別コンテンツへの導線。旧版 RelatedTools（ツール同士のみ）ではなく、新コンセプト下ではツール / 遊び / ブログを区別しない横断推薦。

- 出所タグ: **新コンセプト直接導出**（「道具の合間には息抜きコンテンツも置く」という site-concept.md の方針）、**target likes/dislikes 由来**（要求 1-D6: 「推薦根拠が明確な場合のみ」の制約、要求 2-L1: サイト内回遊による一貫体験）
- 経路 1 likes 応答度: **中立**（初訪ユーザーは目的のツールを使い終えて離脱するため、関連コンテンツへの興味は低い。ただし「他に便利なツールがある」発見体験に寄与する可能性）
- 経路 2 likes 応答度: **応える**（「前に使って気に入った道具に、また迷わず戻ってこられること」と「このサイトにある他のツールの全体像を知る」という interests に対応）
- 経路 1 dislikes 衝突: **衝突しない**（ツール本体より下位に置くことで「似たようなツールが並んでいて迷わせる」dislike を回避できる。ただし推薦根拠が不明確だと衝突に転じる）
- 経路 2 dislikes 衝突: **衝突しない**
- 理由: 採用根拠は旧版の存在ではなく「サイト内の道具 / 息抜きコンテンツの回遊を促し、滞在価値を高める」こと。ただし「推薦根拠が明確な場合のみ」という制約（要求 1-D6）を設ける。機械的な全件羅列ではなく、意味的関連性を持つ推薦が条件。

---

### 要素 8: タイルバリアントのプレビュー表示

このツールが道具箱（ダッシュボード）でどのように表示されるか（サイズ × 機能の各バリアント）を visitor に見せる UI。道具箱への追加トリガーとなる。

- 出所タグ: **新コンセプト直接導出**（「気に入ったツールを 1 つの画面にまとめられるダッシュボード」= site-concept.md の中核機能）
- 経路 1 likes 応答度: **中立**（初訪の目的は目前のタスク完了であり、ダッシュボード登録への動機は薄い。ただし「こんな道具があるのか」という発見体験に寄与する可能性）
- 経路 2 likes 応答度: **応える**（「自分の興味に合わせて最適な道具を選ぶこと・整えること」という interests に直接対応。バリアント選択 UI は「自分の使い方に合ったタイルを置く」体験を実現する）
- 経路 1 dislikes 衝突: **衝突しない**（ツール本体より下位に置けば邪魔にならない）
- 経路 2 dislikes 衝突: **衝突しない**
- 理由: 新コンセプト「日常の傍にある道具」の「毎日の作業の流れに馴染んで使えるようになった道具」を実現する道具箱機能の入口として、詳細ページにバリアントプレビューを置くことは自然。ただし Phase 9（道具箱配線）完了後に完全な価値を発揮するため、本サイクルでは設計のみ確定し、バリアント選択の操作 UI は Phase 9 に後送り可能。

### 要素 9: パンくず（Breadcrumb）ナビゲーション

ホーム > ツール > [ツール名] のような階層表示。SEO（Google の構造化データ = BreadcrumbList schema）への対応。

- 出所タグ: **新コンセプト直接導出**（SEO/アクセシビリティ要件。新コンセプトでは検索流入が主要な流入経路の一つ）、**target likes/dislikes 由来**（要求 1-D6: 「どこにいるか分かる」ことで迷わせない）
- 経路 1 likes 応答度: **中立**（直接の機能価値はないが、「このサイトに他のツールがある」ことを示すことで回遊の起点になる可能性）
- 経路 2 likes 応答度: **中立**（再訪者は URL を直接入力 / ブックマーク利用のため、パンくずの必要性は低い）
- 経路 1 dislikes 衝突: **衝突しない**（コンパクトに 1 行に収まるため「余計な装飾」にならない）
- 経路 2 dislikes 衝突: **衝突しない**
- 理由: 検索流入（経路 1）が主要流入経路である本サイトにおいて、Google 構造化データ対応は SEO 上の必須要件。表示は 1 行・最小限にすることでツール本体のファーストビュー優先（要素 1）と衝突しない。

### 要素 10: ページタイトル（h1）と短い説明文（shortDescription）

ツール名を示す h1 見出しと、検索結果・SNS シェア時にも使われる 1〜2 行の説明文。

- 出所タグ: **新コンセプト直接導出**（SEO・アクセシビリティ・OGP の必須要素）、**target likes/dislikes 由来**（要求 1-L1: ページを開いた瞬間にどのツールか分かることで迷わず使い始められる。要求 1-D8: ツールを識別する最小限の情報として必要）
- 経路 1 likes 応答度: **応える**（「開いた瞬間にどのツールかが分かること」= 迷わず使い始められることに寄与）
- 経路 2 likes 応答度: **中立**（再訪者はすでに知っているが邪魔にはならない）
- 経路 1 dislikes 衝突: **衝突しない**（最小限の文字数で収め、ツール本体への到達を妨げなければ問題ない）
- 経路 2 dislikes 衝突: **衝突しない**
- 理由: 採用根拠は旧版の存在ではなく SEO・アクセシビリティの必要性と「ツールを識別するための最小限の情報」として必要な点。ただし長い説明文は dislike「冒頭の長い解説」に転じるため、shortDescription は文字通り短く（2 行以内）に設計すること。

### 要素 11: 道具箱への追加トリガー / バリアント選択 UI

「この道具を道具箱に追加する」ボタンと、追加するバリアントを選択する UI。道具箱（Phase 9 配線）の入口として詳細ページに配置する。

- 出所タグ: **新コンセプト直接導出**（「気に入ったツールを1つの画面にまとめられるダッシュボード」= site-concept.md の中核機能への入口）
- 経路 1 likes 応答度: **中立**（初訪目的はタスク完了であり、道具箱登録への動機は初訪時点では薄い。ただし「使ってみたら気に入った」瞬間のアクション取得に寄与する可能性）
- 経路 2 likes 応答度: **応える**（「自分の作業の流れに馴染んで使えるようになった道具を道具箱に整理する」行為そのものを支援する）
- 経路 1 dislikes 衝突: **衝突しない**（ツール本体より後か、目立たない位置に置けば邪魔にならない）
- 経路 2 dislikes 衝突: **衝突しない**

### 要素 12: 処理のローカル完結 / プライバシー保護の明示

「入力した内容はブラウザ内でのみ処理され、外部サーバーに送信されません」等の表示。最小限の文字数で、ツール本体の近傍（上部または下部）に配置。

- 出所タグ: **target likes/dislikes 由来**（要求 1-L8: 「入力した内容が外部に送られないなど、不安なく使えること」に直接対応）
- 経路 1 likes 応答度: **応える**（「不安なく使えること」に直接対応。初訪ユーザーが使い始めるかどうかの判断材料になる）
- 経路 2 likes 応答度: **応える**（繰り返し使うツールへの信頼継続に寄与）
- 経路 1 dislikes 衝突: **衝突しない**（1 行以内の表示であれば「余計な説明」にならない）
- 経路 2 dislikes 衝突: **衝突しない**
- 理由: 個人情報・機密情報を扱う可能性があるツール（文字数カウント・SQL フォーマッター等）で、使い始める前の不安を除去する最小コストの要素。文字数カウントのようなシンプルなツールでも、入力した文章が外部に送られないと明示することで離脱を防げる。

### 要素 13: 更新日・変更履歴の表示

ツールが最後に更新された日付と、主要な変更内容の概要（例: 「2026-04-01: 計算方式を〇〇規格に準拠」）。ツール本体の下部か詳細展開エリアに配置。

- 出所タグ: **target likes/dislikes 由来**（要求 2-L7, 2-D4: 「仕様変更があっても壊されない」「以前と挙動が違う」への対処）、**新コンセプト直接導出**（「毎日使えるようによく考えられている道具」は更新に透明性が必要）
- 経路 1 likes 応答度: **中立**（初訪者に直接的な価値はないが、信頼性表示（要素 5）と組み合わせて「信頼できる道具」の印象に寄与する）
- 経路 2 likes 応答度: **応える**（再訪者にとって「今日の結果が前回と違う」状況での原因特定に使える）
- 経路 1 dislikes 衝突: **衝突しない**
- 経路 2 dislikes 衝突: **衝突しない**
- 理由: 要素 5（出典・信頼性表示）と統合できる可能性があるが、「いつ更新されたか」と「どのデータを使っているか」は独立した情報であるため分離して検討する。特に経路 2 ユーザーが「以前と違う結果になった」状況で問題を診断できる透明性として重要。

### 要素 14: タイルの説明（道具箱での提供価値の説明）

タイルが道具箱で何を提供するかを説明するテキスト。バリアント別に「この medium タイルは〇〇を提供します」という説明。道具箱への追加前に visitor が判断できる情報。

- 出所タグ: **新コンセプト直接導出**（道具箱機能の中核機能。タイルの価値を説明しないと道具箱に追加する動機が生まれない）
- 経路 1 likes 応答度: **中立**（初訪者は道具箱への登録動機が低いため直接価値は薄い）
- 経路 2 likes 応答度: **応える**（「自分の興味に合わせて最適な道具を選ぶこと・整えること」に対応。バリアントを選択する判断材料になる）
- 経路 1 dislikes 衝突: **衝突しない**（ツール本体より下位に配置）
- 経路 2 dislikes 衝突: **衝突しない**
- 理由: 要素 8（タイルプレビュー）+ 要素 11（道具箱追加トリガー）とセットで機能する。「どんなタイルが道具箱に追加されるかを見て、判断して、追加する」体験の説明フェーズに相当する。

### 要素 15: 構造化データ（JSON-LD: SoftwareApplication / BreadcrumbList / HowTo 等）

Google 検索向けの structured data マークアップ。BreadcrumbList は必須、SoftwareApplication（無料ウェブアプリとして）は検討、HowTo（使い方手順）はコンテンツ次第で検討。

- 出所タグ: **新コンセプト直接導出**（検索流入が主要経路の一つ。SEO 構造化データは検索流入増加の手段）
- 経路 1 likes 応答度: **応える**（Google 検索での露出増加 → 「ページを開いた瞬間に入力欄が見えて使い始められる」体験の流入源を増やす）
- 経路 2 likes 応答度: **中立**（再訪者は検索経由でなくブックマーク/道具箱経由のため直接価値は低い）
- 経路 1 dislikes 衝突: **衝突しない**（JSON-LD は head タグ内に収まり視覚的影響ゼロ）
- 経路 2 dislikes 衝突: **衝突しない**
- 理由: ページの視覚に影響せず SEO を向上させる要素。実装コストに対してリターンが大きい。特に BreadcrumbList は要素 9（パンくず）と対応する。SoftwareApplication スキーマはツールページの特性（無料・ブラウザ動作）と合致する。

### 要素 16: 操作性・トーン統一のための共通ナビゲーション / フッター

全コンテンツ（ツール / 遊び / ブログ）で統一された共通ナビゲーションとフッター。詳細ページ固有の設計ではなく、サイト全体共通の要素として詳細ページにも適用。

- 出所タグ: **target likes/dislikes 由来**（要求 2-L1: 「操作性やトーン&マナーが一貫していること」に対応）、**新コンセプト直接導出**（過渡期の一貫性問題の構造的解決）
- 経路 1 likes 応答度: **中立**（初訪者は他ページへの回遊動機が低いが、サイト全体の印象に寄与する）
- 経路 2 likes 応答度: **応える**（「サイト内のすべてのツールやゲームの操作性やトーン&マナーが一貫していること」に直接対応）
- 経路 1 dislikes 衝突: **衝突しない**（通常のナビゲーション / フッターはコンテンツエリアと分離されているため）
- 経路 2 dislikes 衝突: **衝突しない**
- 理由: (new) 化済みコンテンツが 1 件でも (legacy) 残り 52 件と並ぶ「過渡期の一貫性違反」への対処として、共通ナビゲーション / フッターが変わらないことが前提。詳細ページ設計での判断よりも、サイト全体の共通コンポーネント設計（Phase B）で確定すべき事項だが、詳細ページ要素として明示的に列挙する。

### 要素 17: 「今日のひとこと」タイプの日替わりコンテンツ（ツール種別依存）

敬語リファレンス・慣用句辞典などの辞書・リファレンス系ツールで有効な「今日のランダムピックアップ」。毎日同じツールを開く理由を作る（日次再訪インセンティブ）。ツール種別によって有無が分かれる。

- 出所タグ: **新コンセプト直接導出**（「日常の傍にある」の「日常」= 毎日開く理由を作る設計）
- 経路 1 likes 応答度: **中立**（初訪者の目的はタスク完了であり、日替わりコンテンツへの期待は薄い）
- 経路 2 likes 応答度: **応える**（「毎日繰り返し遊べるゲーム」に類似した「毎日開く理由」を提供する。「前に使って気に入った道具に、また迷わず戻ってこられること」の継続的な動機づけ）
- 経路 1 dislikes 衝突: **衝突しない**（ツール本体より下位に配置すれば邪魔にならない）
- 経路 2 dislikes 衝突: **衝突しない**
- 理由: site-concept.md 「毎日使えるようによく考えられている」という設計思想を体現する要素。ツールの種類によって実装可能かどうかが分かれるため、共通要素としてではなくオプショナル要素として発散段階に列挙する。收束段階で「実装できるツール種別の条件」を定義する必要がある。

### 要素 18: アクセシビリティ対応（見出し階層・aria ラベル・キーボード操作）

適切な見出し階層（h1 → h2 → h3）、フォーム要素への aria-label、キーボードのみでの操作完結性。視覚的な表現には依存しないが、設計段階でセマンティック構造を定義しておく必要がある要素群。

- 出所タグ: **新コンセプト直接導出**（Constitution Rule 1「日本の法律・倫理基準への準拠」含む基本的品質要件）
- 経路 1 likes 応答度: **応える**（キーボード操作を効率的に行うユーザー、スクリーンリーダー利用者にも「すぐ使えること」を保証する）
- 経路 2 likes 応答度: **応える**（「操作性の一貫性」はアクセシビリティ水準の一貫性も含む）
- 経路 1 dislikes 衝突: **衝突しない**（セマンティック構造はビジュアルに影響しない）
- 経路 2 dislikes 衝突: **衝突しない**
- 理由: 設計段階でセマンティック構造を定義しないと、実装時に後付けで入れることになり整合性が崩れやすい。特に h1〜h3 の見出し階層は詳細ページのゾーン設計（収束段階）と直結するため、発散段階で明示的に要素として挙げておく。

### 要素 19: エラー状態 / 空状態の設計（Empty State / Error State）

ツールの入力が空・無効の場合や、処理エラーが発生した場合の表示。「何も起きていない」「どうすれば使えるか分からない」という混乱を防ぐ設計。

- 出所タグ: **target likes/dislikes 由来**（要求 1-L1, 1-D4: 「すぐ使い始められる」状態を保証する逆説的必要要素）、**新コンセプト直接導出**（「道具」として信頼できるためにはエラー時の誠実な対応が必要）
- 経路 1 likes 応答度: **応える**（初訪者が操作に迷わない状態を作ることで「すぐ使い始められる」を実現）
- 経路 2 likes 応答度: **応える**（「同じ入力に対して前回と同じ結果が返ってくること」の逆として、エラー時に「何が起きているか」が分かる）
- 経路 1 dislikes 衝突: **衝突しない**
- 経路 2 dislikes 衝突: **衝突しない**
- 理由: 「道具」は壊れたときに壊れていると分かることも重要。エラー状態の設計なしにツール詳細ページを設計すると、実装時に各コンテンツで個別実装され一貫性がなくなる。共通エラー / 空状態コンポーネントの設計を Phase B で行うための前提として、発散段階で要素として明示する。

### 要素 20: 動作速度 / パフォーマンス要件

ツール本体のインタラクション（入力 → 結果表示）が体感的に即座であること。ページロード時に不要なリソース（重い外部 JS / 大きな画像 / 不要なフォント等）を配置しないこと。

- 出所タグ: **target likes/dislikes 由来**（要求 2-D1, 2-D2: 「動作が遅いツール」dislike への対処）
- 経路 1 likes 応答度: **応える**（「ページを開いた瞬間に入力欄が見えて、すぐ使い始められること」の前提として、ページロード自体が高速であることが必要）
- 経路 2 likes 応答度: **応える**（再訪者は「毎日使う道具」として速度を強く期待する。「動作が遅いツール」は再訪を妨げる最大要因の一つ）
- 経路 1 dislikes 衝突: **衝突しない**
- 経路 2 dislikes 衝突: **衝突しない**（高速動作は dislike 「動作が遅いツール」を直接解決する）
- 理由: ツール本体がファーストビューに来るだけでなく、インタラクションが体感的に即座でなければ「すぐ使い始められる」は実現しない。設計段階でパフォーマンス要件（100ms 以内推奨・不要リソース排除）を明示することで、実装時のリグレッションを防ぐ。要求 2-D1, 2-D2 から直接導出。

### 要素 21: URL 連続性 / リダイレクト方針

(legacy)→(new) 移行時に旧 URL から新 URL へのリダイレクトを必ず設定する設計原則。URL 構造の安定性を保証し、既存ブックマーク・外部リンクを無効にしない。

- 出所タグ: **target likes/dislikes 由来**（要求 2-L3, 2-D3: 「ブックマークしたURLを開けばすぐ目的のツールが表示されること」「URLが変更されたコンテンツにリダイレクトが設定されておらずたどり着けなくなること」への対処）
- 経路 1 likes 応答度: **中立**（初訪者は既存 URL を持たないため直接の価値は低いが、外部サイトからのリンク切れを防ぐことで検索流入の損失を防ぐ）
- 経路 2 likes 応答度: **応える**（「ブックマークしたURLを開けばすぐ目的のツールが表示されること」に直接対応。リダイレクトが設定されていればブックマークが死なない）
- 経路 1 dislikes 衝突: **衝突しない**
- 経路 2 dislikes 衝突: **衝突しない**（dislike「URLが変更されたコンテンツにリダイレクトが設定されておらずたどり着けなくなること」を直接解決する）
- 理由: (legacy)→(new) 移行が 52 件残っている状況で、URL 連続性を設計原則として明文化しないと移行のたびにリダイレクト設定が漏れるリスクがある。ページ単体の設計要素というよりも移行方針だが、詳細ページ設計の一部として発散段階に明示する。要求 2-L3, 2-D3 から直接導出。

---

### 要素 22: 結果の決定性 / 後方互換性保証

同じ入力に対して同じ結果が返ることを設計原則として担保する。ロジック更新時に変更点を明示するだけでなく、**変わらないことを保証する**設計（テスト・QA・バージョニング）も含む。要素 13（更新日表示）は「変わったことを伝える」要素、本要素は「変わらないことを保証する」要素として独立。

- 出所タグ: **target likes/dislikes 由来**（要求 2-L7, 2-D4: 「同じ入力に対して前回と同じ結果が返ってくること」「以前と同じ入力なのに結果や挙動が前回と変わっていること」への対処）
- 経路 1 likes 応答度: **中立**（初訪者は以前の結果との比較がないため直接の価値は低いが、信頼性の基盤として機能する）
- 経路 2 likes 応答度: **応える**（「同じ入力に対して前回と同じ結果が返ってくること」に直接対応。繰り返し使う中での信頼性の核心）
- 経路 1 dislikes 衝突: **衝突しない**
- 経路 2 dislikes 衝突: **衝突しない**（dislike「以前と同じ入力なのに結果や挙動が前回と変わっていること」を設計レベルで防ぐ）

### 要素 23: モバイル幅でのファーストビュー保証

モバイル幅 375px / タブレット幅 768px / デスクトップ幅 1280px の各幅で、ツール本体がファーストビュー（スクロールなし）に収まる設計原則。要素 1（ファーストビュー優先配置）の「800px 以内」記述はデスクトップ前提であり、モバイル幅でのファーストビュー保証は独立論点。

- 経路 2 likes 応答度: **応える**（「ブックマークしたURLを開けばすぐ目的のツールが表示されること」はモバイルでも同様に期待される）

- 経路 2 dislikes 衝突: **衝突しない**

### 要素 24: 入力支援（プレースホルダ・入力例・リセット・Undo）

### ステップ 2: 独自階層分類

**階層数: 4 階層**

#### 階層数の根拠

3 階層を先に検討した。cycle-160 の 3 ゾーン（ゾーン 1 = 識別情報、ゾーン 2 = ツール本体、ゾーン 3 = 補助情報）を「ヘッダー・本体・フッター」「識別・使用・支援」等の別名に置き換えると、採用要素の配置先が変わらず差別化が責務名の置換にとどまる。

具体的に検証する。採用要素 10 件 + オプショナル 5 件を 3 バケットに割り振ると:

- バケット A（ゾーン 1 相当）: 要素 9 (パンくず) + 要素 10 (h1 + shortDescription)
- バケット B（ゾーン 2 相当）: 要素 1 (ツール本体) + 要素 3 (コピー) + 要素 4 (永続化) + 要素 12 (プライバシー表示)
- バケット C（ゾーン 3 相当）: 要素 2 (使い方説明) + 要素 5 (出典) + 要素 6 (AI 注記) + 要素 13 (更新日) + 要素 7 (関連) + 要素 8 (タイルプレビュー) + 要素 11 (道具箱追加) + 要素 14 (タイル説明) + 要素 17 (日替わり)

この 3 バケット分割には 2 つの問題がある:

1. **「信頼・透明性」要素（5, 6, 12, 13）の責務が 2 バケットに散在する**: 要素 12（プライバシー表示）はツール本体の近傍に置くが責務は「信頼」、要素 5 / 6 / 13 はバケット C に入るが責務は同じく「信頼」。「信頼・透明性」という独立した責務があるのに、それが複数のバケットに散らばる構造になってしまう。

2. **「生活への組み込み」要素（4, 8, 11, 14, 17）がバケット B と C に散在する**: ローカルストレージ永続化（要素 4）はツール本体と一体だが責務は「繰り返し使う体験」。タイルプレビュー（要素 8）/ 道具箱追加（要素 11）/ タイル説明（要素 14）は全員が「生活への組み込み」という新コンセプト固有の責務を持つが、バケット C に混在する。

これは「責務での分類」ではなく「位置での分類」になっており、cycle-160 の 3 ゾーンとの差別化が責務名の置換にとどまる。したがって **4 階層を選択する**。

4 階層にすることで、以下の責務が明確に独立する:

- 階層 1（識別）: パンくず + h1 + shortDescription → 「何のツールか瞬時に分かる」
- 階層 2（使用）: ツール本体 + コピー + プライバシー表示 → 「今すぐ使える」
- 階層 3（信頼・透明性）: 使い方説明 + 出典 + AI 注記 + 更新日 → 「信頼して使い続けられる」
- 階層 4（生活への組み込み）: 永続化 + 日替わり + タイルプレビュー + 道具箱追加 + タイル説明 + 関連コンテンツ → 「この道具を生活に組み込む」

階層 3「信頼・透明性」と階層 4「生活への組み込み」は cycle-160 の 3 ゾーンに対応する概念が存在しない。これにより 4 階層は cycle-160 の 3 ゾーンと責務レベルで差別化できる。

---

#### 階層 1: 識別（何のツールか瞬時に分かる）

- **責務**: visitor がページにたどり着いた瞬間に「これは自分が探していたツールだ」と確認し、使い始めの判断を下せる情報を提供する。迷わず使い始めを促すための最小限の識別情報に限定する。詳しい説明は提供しない（詳しい説明は階層 3 の責務）。
- **配置する要素**: 要素 9（パンくず）、要素 10（h1 + shortDescription）
- **配置根拠**:
  - 要求 1-L1「ページを開いた瞬間にどのツールか分かること」に応える
  - 要求 1-D8「ツールを識別する最小限の情報として必要」
  - shortDescription は 2 行以内に制限し、この階層が「説明の場」に肥大しないよう設計上の上限を設ける
  - パンくずは 1 行・最小限で SEO（BreadcrumbList）要件を満たす
- **この階層で明示的に含めないもの**: 詳細な機能説明・使い方・FAQ・信頼性情報（それらは階層 3 の責務）

---

#### 階層 2: 使用（今すぐ使える）

- **責務**: ツールを「今すぐ使える状態」にする。入力・処理・結果取得・元の作業への復帰というサイクルを最小の摩擦で完結させる。visitor がスクロールや追加操作なしに道具として使える状態を提供する。
- **配置する要素**: 要素 1（ツール本体のファーストビュー優先配置）、要素 3（コピー / シェア UX）、要素 12（処理のローカル完結 / プライバシー保護の明示）
- **配置根拠**:
  - 要求 1-L1「ページを開いた瞬間に入力欄が見えて、すぐ使い始められること」→ 要素 1 が直接対応
  - 要求 1-L4 / 1-L5「コピペで結果を受け取ってすぐ元の作業画面に戻れること」→ 要素 3 が対応
  - 要求 1-L8「入力した内容が外部に送られないなど、不安なく使えること」→ 要素 12 が対応。使い始めの直前・直後に見える位置が前提
  - 要求 1-D1 / 1-D2「ツール本体が視覚的に最優先」→ この階層が視覚的に最初かつ主役の領域
- **この階層で明示的に含めないもの**: 使い方説明・出典・AI 注記（それらは階層 3 の責務）。道具箱追加ボタン・タイルプレビュー（それらは階層 4 の責務）。
- **階層 1 → 階層 2 の順序**: visitor はページに来て「このツールだ」と確認（階層 1）してから「使う」（階層 2）。ただし階層 1 の識別情報は視覚的重さを最小化し、ファーストビューで階層 2 のツール本体が支配的に見える状態を設計上の制約とする（要求 1-L3）。

#### 階層 3: 信頼・透明性（信頼して使い続けられる）

- **責務**: 初訪ユーザーが「このツールは信頼できるか」を判断するための情報、および再訪ユーザーが「何かが変わったか」を確認するための情報を提供する。階層 2 で使い終えた後にスクロールすれば見える位置が前提。ファーストビューには入らない。
- **配置する要素**: 要素 2（最小限の使い方説明）、要素 5（出典・更新日・信頼性表示）、要素 6（AI 注記）、要素 13（更新日・変更履歴の表示）
- **配置根拠**:
  - 要求 1-L9「結果の根拠や前提が必要最小限だけ添えられており、信頼して使えること」→ 要素 5 / 13 が対応
  - 要求 1-D4「処理の前提・出典・精度の限界を隠さず、過剰にならず表示すること」→ 要素 2 / 5 / 6 が対応
  - 要求 1-D5「AI 生成コンテンツが含まれる場合は AI 注記を表示」→ 要素 6 が Constitution Rule 3 として必須
  - 要求 2-L7 / 2-D4「同じ入力に対して前回と同じ結果が返ること / 更新があれば理由が分かる状態」→ 要素 13 が対応
- **この階層の「信頼」要素と階層 2 の「プライバシー表示」（要素 12）の関係**: 要素 12 は「使い始める前の不安解消」という使用体験の一部であるため階層 2 に置く。階層 3 の信頼情報は「使い終えた後に深く確認する」性質を持つ。この二分は dislike「ツール冒頭に長い解説」への抜本対処の構造的根拠でもある（詳細はステップ 3）。
- **要素 5 と要素 13 の統合可能性**: 両者は「出典（何を根拠にしているか）」と「更新日（いつ何が変わったか）」で責務が異なる。ただし実装上は同一の UI エリア（注記エリア）に並べることが自然であり、T-B のコンポーネント設計で統合/分離を判断する。

#### 階層 4: 生活への組み込み（この道具を生活の一部にする）

- **責務**: 一度使ったユーザーが「この道具を日常に組み込む」行動を支援する。繰り返し使う体験の蓄積（永続化）、毎日開く理由（日替わり）、道具箱への登録、関連コンテンツへの発見。cycle-160 の 3 ゾーンには存在しない、新コンセプト「日常の傍にある道具」固有の責務階層。
- **配置する要素**:

  - 要素 17（日替わりコンテンツ）— 採用オプショナル（ツール種別依存）
  - 要素 8（タイルバリアントのプレビュー表示）— 採用オプショナル（道具箱機能存在時）
  - 要素 11（道具箱への追加トリガー / バリアント選択 UI）— 採用オプショナル（道具箱機能存在時）

### ステップ 3: target dislike への抜本対処

#### dislike 1「ツール冒頭に長い解説記事が挟まっていて、すぐ使えないこと」への抜本対処

**対処方針: 「説明は階層 3 に完全移動 + 階層 1 の識別情報を最小化」の二重構造で解決**

cycle-160 の 3 ゾーンは「ツール本体（ゾーン 2）をヘッダー直後に置く」という配置規則で対処していた。これは「説明をツール本体の後に追いやる」という「順序による対症療法」であり、ページ構造の中に「説明の場所」が存在する限り、説明文が長くなるリスクが残る（旧版でも説明文が肥大した例がある）。

新設計では、説明・信頼情報という概念を **「階層 3」という独立した責務ゾーン** に分離し、「階層 1 の識別情報（h1 + shortDescription）は 2 行以内」という上限制約を設計ルールとして明示する。これにより:

1. **階層 2（使用）が常にファーストビューを支配する**: 階層 1 は最小限の識別情報（パンくず + h1 + shortDescription）に制限されており、説明が入り込む余地がない。階層 1 → 階層 2 が画面上端から始まる設計構造において、「使い始められるまでに読まなければならないテキスト量」の上限が設計ルールによって保証される。
2. **説明は「読むか読まないかを選べる領域」に完全移動する**: 最小限の使い方説明（要素 2）・出典（要素 5）・AI 注記（要素 6）・更新日（要素 13）はすべて階層 3 に集約する。初訪ユーザーはツール本体（階層 2）を使った後にスクロールすれば読める。再訪ユーザーは読まなくていい。

これは cycle-160 の「順序による対症療法」から「階層の責務分離による構造的解決」への移行であり、抜本対処といえる。

**旧構造との具体的な差異**: 旧版 ToolLayout では shortDescription の文字数制限がなく、一部のツールで 4〜5 行になっていた可能性がある。新設計では shortDescription = 2 行以内を設計ルールとし、超過する説明は階層 3 の「最小限の使い方説明（要素 2）」に移す。

#### dislike 2「以前と同じ入力なのに結果や挙動が前回と変わっていること」（過渡期一貫性違反含む）への抜本対処

**対処方針: 「階層 3（信頼・透明性）の要素 13 で変化を可視化」+ 「共通インフラ移管した要素 22 で変化を防ぐ」の二重構造**

- **変化を可視化する（要素 13 + 要素 5）**: 階層 3 に配置する更新日・変更履歴（要素 13）と出典（要素 5）の組み合わせで、「何が変わったか」と「何を根拠にしているか」を再訪ユーザーが確認できる。「今日の結果が先週と違う」状況での原因診断が可能になる。
- **変化を防ぐ（要素 22 = 共通インフラ移管）**: 結果の決定性・後方互換性保証（要素 22）は詳細ページのコンテンツ階層ではないが、ツールのロジック実装に対する QA・テスト・バージョニングの設計原則として、共通インフラに移管して明文化する。「意図しない変化」を防ぐ設計基準をインフラ側で保証する。
- **過渡期一貫性違反への対処**: (new) 化済みコンテンツと (legacy) 残り 52 件が混在する過渡期において、共通ナビゲーション・フッター（要素 16 = 共通インフラ移管）を変えないことで「操作のトーン一貫性」を保つ。詳細ページ側のコンテンツ変化（(new) 化によるレイアウト変更等）は要素 13 の更新日表示で visitor に明示する。

### 要素 16 layout 配置先判断（ステップ 3）

**判断: (new) 専用 layout（`src/app/(new)/layout.tsx`）から提供する。root layout は用意しない。**

根拠:

1. **実体確認の結果**（Read 済み）: `src/app/(new)/layout.tsx`（L1〜55）はすでに `src/components/Header`（新版）と `src/components/Footer`（新版）を配置しており、(new) ルート配下のすべてのページ（トップ / ブログ / about / play 等）が共通ナビゲーション / フッターを受け取っている。新版ツール詳細ページ（`src/app/(new)/tools/[slug]/`）も同じ (new) ルート配下に作成されるため、**追加の layout 設定なしに自動的に新版 Header / Footer を引き継ぐ**。

2. **(legacy) との分離**: `src/app/(legacy)/layout.tsx`（L1〜42）は `src/components/common/Header`（旧版）と `src/components/common/Footer`（旧版）を参照しており、(new) の layout とは完全に独立している。root layout が存在しないことで「旧版 layout が (new) ページに影響する」経路は物理的に存在しない。

3. **過渡期一貫性の維持**: 要求 2-L2「(new) 化済み 1 件と (legacy) 残り 52 件が混在する状態においても、共通要素（ナビゲーション・フッター等）は変化させない」への対処として、(new) ルートで使用している Header / Footer（新版）を変更しなければ visitor が体験する共通 UI は変化しない。新版ツール詳細ページを追加しても Header / Footer には変更を加えないため、(new) 既存ページ（トップ / ブログ等）との一貫性が保たれる。

4. **cycle-191 原則との整合**: 旧版 `src/components/common/` 配下を触らないという原則のもとで、(new) 専用 layout の Header / Footer は `src/components/Header`（新版）を使用しており、旧版への依存がない。

**結論**: 新版ツール詳細ページが利用する共通ナビゲーション / フッターは `src/app/(new)/layout.tsx` が自動提供する。T-B 実装でページファイルを `src/app/(new)/tools/[slug]/page.tsx` として作成すれば、layout.tsx の変更なしに Header / Footer が引き継がれる。

### 要素 24 共通入力コンポーネント I/F（ステップ 5）

既存の `src/components/Input/index.tsx`（L1〜77、Read 実体確認済み）は標準 `<input>` 要素の薄いラッパーであり、`type` / `error` の 2 prop のみを独自に持つ。プレースホルダ・入力例・リセット・Undo の入力支援機能は一切持っていない。

#### 共通入力コンポーネントの責務

既存 Input コンポーネントは「スタイル一貫性の確保 + エラー状態の視覚表現」に特化した最薄のラッパーである。新版ツール詳細ページが求める「すぐ使い始められる」（要求 1-L1）入力体験のためには、以下の入力支援機能が追加で必要になる。ただし、これら入力支援機能を既存の Input コンポーネントに直接追加するのではなく、**入力支援のコンテキストを持つ上位コンポーネント（仮称: ToolInputArea）として新設する**設計とする。理由: 既存 Input は汎用プリミティブとして薄さを維持することが価値であり、ツール詳細ページ固有の入力支援 UX を注入すると過剰になる。

#### I/F 分類: 標準提供 / オプショナル / ツール側個別実装

**標準提供（すべてのツールで一律提供）**:

- **プレースホルダ表示**: `placeholder` 属性（既存 Input の `...rest` で HTML 標準として透過済み）—— 標準提供として機能しているが、新版では入力例テキストとの役割分担を明示する（placeholder = 「何を入力するか」の最短ヒント、入力例 = 具体的サンプル）
- **入力欄のリセット**: ワンクリックで入力値を空にするリセットボタンを入力エリア内に常時表示。要求 1-L1「すぐ使い始められる」の実現として、入力欄が埋まっているときに再利用する摩擦をゼロにする

**オプショナル（ツールの性質に応じて選択可能）**:

- **入力例の挿入**: 「試しに入力」ボタンを押すとサンプルテキストを自動入力する UX。初訪ユーザーが「どんな入力が有効か」を試せる。ツール依存の入力例テキストを呼び出し元から渡す形で提供
- **Undo**: Ctrl+Z（標準）はブラウザネイティブとして動作するため、追加実装は基本不要。ただし入力例の自動挿入後に「元に戻す」ボタンを表示する場合はオプショナルで提供

**ツール側個別実装（共通コンポーネントに含めない）**:

- 入力バリデーション（ツールごとに有効な入力形式が異なる）
- 入力モード切替（日本語 / 英語、半角 / 全角 等）
- 複数入力欄の連動（ツール固有のロジック）

#### target 要求 1-L1「ページを開いた瞬間に入力欄が見えて、すぐ使い始められること」への対処

入力支援 I/F の 2 点が要求 1-L1 に直接寄与する:

1. **リセットボタン（標準提供）**: 入力欄に前回の値が残っている場合、リセットボタンがあれば「既存の入力をどうするか判断する手間」なしに即座に新しい用途に転用できる。「ページを開いた瞬間にすぐ使い始められる」ためのクリアランスコスト解消。
2. **入力例の挿入（オプショナル）**: 初訪ユーザーは「何を入力すればいいか分からない」まま使い始めを躊躇することがある。入力例ボタンがあれば「まず試してみる」摩擦が下がり、要求 1-L1 の「すぐ使い始められる」に寄与する。

LocalPersistenceAdapter（コンポーネント 8）との連携: 前回入力値の自動復元（要求 2-L6）は「再訪ユーザーがすぐに前回の作業を続けられる」効果をもたらす。これも「すぐ使い始められる」体験の一部であり、ToolInputArea（仮称）の入力支援とローカルストレージ連携は密接に関わる。ToolInputArea の設計段階で LocalPersistenceAdapter との I/F を明示することを T-B-実装のスコープとする。

**ToolInputArea（仮称）の位置付け**（実装スコープの明確化）:

- 既存 Input コンポーネントを内包しつつ、リセットボタン / 入力例挿入ボタン / Undo 等の入力支援 UX を組み合わせた「ツール詳細ページの入力エリア専用コンポーネント」
- 配置場所: `src/tools/_new-components/ToolInputArea/`（ツール固有の入力支援コンテキストを持つため）
- 対応する T-A 要素: 要素 24（入力支援: プレースホルダ・入力例・リセット・Undo）
- このコンポーネントは T-B-設計の追加コンポーネントとして上記コンポーネント 1〜8 に加える（**コンポーネント 9 として位置付ける**）

### ステップ 4: 共通要素 / 共通しない要素の抽出

3 件のマトリクスを横断して比較する。出発点として cycle-190.md L506-L513 の 6 要素候補を参照しつつ、3 件の事例で意味が成立するかを検証する。

#### 共通要素（Tileable 型に入れる候補）

各要素が 3 ツールすべてで意味を持つかを判定する。

**候補 A: バリアント識別子**

- keigo-reference: `large-full` / `medium-search` / `medium-category-business` / `small-daily-pick` 等
- sql-formatter: `large-full` / `medium-format` 等
- char-count: `large-full` / `medium-text-volume` / `small-char-only` 等
- 判定: **3 件すべてで意味が成立する** → 共通要素に採用

**候補 B: 推奨サイズ（グリッドスパン情報）**

- keigo-reference: large=2×2 / medium=2×1 / small=1×1
- sql-formatter: large=2×2 / medium=2×1（small は成立しないため省略）
- char-count: large=2×2 / medium=2×1 / small=1×1
- 判定: **3 件すべてで意味が成立する**（sql-formatter は small バリアントを持たないことが値として埋まる = 「推奨サイズのあるバリアントのみが型に存在する」設計） → 共通要素に採用

**候補 C: 提供機能の説明テキスト**（visitor が「このタイルで何ができるか」を理解できる文）

- keigo-reference: 例: 「検索と候補一覧のみ。詳細例文は詳細ページで確認できます」
- sql-formatter: 例: 「SQL を整形してコピーするだけの最小機能」
- char-count: 例: 「文字数・バイト数のみをリアルタイム表示」
- 判定: **3 件すべてで値が埋まる** → 共通要素に採用

**候補 D: 単独レンダリング可能なコンポーネント参照**（各バリアントのタイル用コンポーネント）

- keigo-reference: `KeigoReferenceTile.large` / `KeigoReferenceTile.mediumSearch` 等
- sql-formatter: `SqlFormatterTile.large` / `SqlFormatterTile.mediumFormat` 等
- char-count: `CharCountTile.large` / `CharCountTile.mediumVolume` / `CharCountTile.small` 等
- 判定: **3 件すべてで意味が成立する**（各ツールにバリアントごとの Tile コンポーネントが存在する設計） → 共通要素に採用

**候補 E: デフォルトバリアントの指定**（`INITIAL_DEFAULT_LAYOUT` 投入時にどのバリアントを選ぶかの判断材料）

- keigo-reference: `medium-search`（検索特化が最も汎用的）or `large-full`（全機能）
- sql-formatter: `medium-format`（整形特化が最も需要が高い）
- char-count: `small-char-only`（最もシンプルで daily use 向き）or `large-full`
- 判定: **3 件すべてで値が埋まる** → 共通要素に採用

**候補 F: 道具箱側バリアント選択機構**（visitor が同じツールから複数バリアントを道具箱に置ける仕組み）

- keigo-reference: 複数バリアント（medium-search / medium-category-business 等）を別々に道具箱に追加可能
- sql-formatter: バリアント数が少ない（large-full / medium-format 程度）が複数選択は成立する
- char-count: バリアント数が最も少ない（large-full / medium-text-volume / small-char-only）
- 判定: **3 件すべてで意味は成立するが、これは「Tileable 型のフィールド」ではなく「道具箱 UI の振る舞い」の問題**であり、型契約ではなく Phase 9 配線実装の責務。Tileable 型には入れず、設計として明示するのみ → **型契約からは除外**

#### 共通しない要素（Tileable 型に入れない）

- **カテゴリフィルター**: keigo-reference は「基本動詞 / ビジネス / 接客サービス」の 3 カテゴリを持つが、sql-formatter にも char-count にも相当する概念がない。keigo-reference 固有のドメイン構造であり、型契約に含めると他ツールで「空」になる不完全フィールドになる。
- **誤用パターン（ミスタイプ / よくある間違い）**: keigo-reference の「よくある間違い」タブは敬語という特定ドメインの誤用知識を提供する機能であり、sql-formatter や char-count には相当しない。
- **変換操作（ボタン押下による変換）**: sql-formatter の「整形」「圧縮」ボタンは「操作 → 変換」モデルに基づくが、char-count は「リアルタイム計測（ボタン不要）」、keigo-reference は「検索フィルタリング（ボタン不要）」と、インタラクションモデルが異なる。3 件で意味を統一した表現にできないため型に入れない。
- **オプション系（インデント種別・キーワード大文字化）**: sql-formatter の変換オプションは固有であり他 2 件に相当する概念がない。
- **統計種別の選択**: char-count の 6 種の計測値（文字数 / バイト数 / 行数 等）は char-count 固有の設計であり、他 2 件には相当しない。
