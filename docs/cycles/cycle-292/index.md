# サイクル-292

未結線のまま放置されている検索機能（B-598）の処遇を判断し、実行する。検索一式（`SearchModal` / `SearchTrigger` / `SearchInput` / `useSearch` とテスト・CSS）は完成しているが、`SearchTrigger` はアプリのどこからも描画されていない（本ターンの grep で確認：`src/components/search/` 外からの import はゼロ、`analytics.ts` のコメント言及のみ）。にもかかわらず `generate:static-assets` が毎ビルド `public/search-index.json`（実測 673KB）を生成し続けている。これは a11y ではなく製品判断——**結線して来訪者に価値を返す**か、**撤去して健全化する**か。

このサイクルの来訪者は「サイト内で目的のコンテンツ（辞典・診断・ツール・ブログ）を探している人」。もし横断検索が彼らの探索を実際に助けるなら結線が最高の価値。逆に、探索導線が既に十分で検索需要が薄いなら、死蔵コードと毎ビルド 673KB の無駄を撤去するのが誠実。どちらが正しいかはデータと履歴で決める。

## 作業計画

### 目的

cycle-181 以降デッドコードのまま毎ビルド 673KB を生成し続けている検索機能に、来訪者価値の観点から決着をつける。「結線して探索を助ける」か「撤去して健全化する」かを、履歴とデータに基づいて判断し、実行する。

### 作業内容

上記「実施する作業」の順で進める。核心は **判断を実行より先に確定し、実行前にレビューを通す** こと。事故サイクル291の教訓（既決着結論の未調査での再掘削・専用手順の未使用）を踏まえ、履歴再構成を最初の関門に置く。作業はタスク単位でサブエージェントに委譲する（履歴調査・データ調査・実装・レビューはそれぞれ別エージェント）。

### 検討した他の選択肢と判断理由

- **判断を飛ばして即・撤去する**: 不可。撤去が正解の公算は高い（cycle-185 で結線が全撤回された履歴・検索計測CTRの低さの傾向）が、データと履歴を確認せず結論すれば事故サイクルの「未調査での断定」を別方向に反復する。判断はデータで裏取りする。
- **判断を飛ばして即・結線する**: 不可。cycle-185 で結線が「スコパ越境＋ファクトチェック不実施」として全撤回された経緯があり、その否定を調べずに再結線するのは既決着結論の再掘削そのもの。まず履歴で撤回の射程を確定する。
- **B-598 を今サイクルで扱わず別の P1（B-573 a11y全面展開 / B-562 新クラスタ）を選ぶ**: 不可。B-598 は事故サイクル291が正規手順での着手を明示的にキャリーオーバーした精査済み Queued 筆頭。ここで再び飛ばして別案に移れば、事故の構図（精査済み Queued を飛ばす）を反復する。
