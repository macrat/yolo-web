# B-598 処遇判断の記録 — 検索機能は「撤去」

cycle-292 の核心タスク。未結線の検索機能（`src/components/search/` 一式＋索引生成）を **結線するか撤去するか** を、履歴とデータに基づいて判断した記録。

## 結論

**撤去（remove）する。** グローバル検索モーダル一式・索引生成スクリプト・`generate:static-assets` 配線・`public/search-index.json`・関連テスト・検索専用のランタイム依存 `fuse.js`（package.json / package-lock.json）を除去する。これは cycle-186 で確定済みの「作らない」判断を実装面で完遂する作業であり、新たな方針転換ではない。

## 判断の芯

### 1. 「結線しない」は既に確定済みの製品判断（cycle-186）

履歴再構成（サブエージェント調査・原文引用付き）で確定した事実:

- **cycle-174** で「① 新デザインに横断検索を作る」と一旦確定。
- **cycle-185** で暫定結線を全撤回。ただしこの撤回は【やり方（スコープ越境・ファクトチェック不実施）の否定】であって、結線という結論の否定ではなかった（当時は「Phase 5 で結線」を維持）。
- **cycle-186** で「② 作らない（Phase 5 スキップ）」に**再判断・確定**。引き金は「Google sitelinks search box が 2024年11月に廃止」＝結線の主根拠だった SEO 便益の消滅（cycle-186 L61。※cycle-186 自身が L285 で「固有日付は一次資料で直接確認できず月精度に留める」と注記しているため、日付は月精度で記す）。実装コスト・資産流用可否・現状維持しやすさを一切判断材料に含めない、純粋な来訪者価値ベースの判断と明記（L75）。backlog の B-331 は Done 化。
- **cycle-279** の legacy 一掃で呼び出し元（legacy Header）は消えたが `src/components/search/` 本体と `scripts/build-search-index.ts` が残置され、完全な dead code 化。cycle-186 が想定した「Phase 10.2 で legacy と一体で自然消滅」が部分的にしか実行されなかった。
- **cycle-287** 監査 C7 で検出 → **B-598** として管理下に。

→ **「結線」を今選ぶことは、cycle-186 の確定判断の再掘削（re-litigation）になる。** それを正当化するには cycle-186 の根拠を覆す新証拠が要る。

### 2. 現在のデータは cycle-186 の判断を覆さない（むしろ補強する）

GA4/BigQuery（2026-03-28〜07-19・113日・1,631セッション）＋ Search Console:

- **単一PV離脱が支配的**: 全セッションの 79.4%（10人中約8人）が1ページで離脱。グローバル検索が助ける「多ページ探索型」ではない。探索起点になりうるトップ（回遊48%）・辞典一覧（回遊36%）でも「探せず困る」異常な離脱集中は見られない。
- **ブランド/ナビゲーショナルクエリが皆無**: `yolos.net` 2clicks・`site:yolos.net` 0clicks・「yolos＋探索語」型 0件。「サイト内で探し直したい」需要の外部証拠なし。Google 検索（6,560種のクエリ）が事実上のサイト内検索として機能し、来訪者を目的ページへ直接着地させている。
- **検索モーダルの実利用ほぼゼロ**: `search_modal_open` 1回/1ユーザー、`view_search_results` 1回のみ（113日）。実質1〜2ユーザーで動作確認の公算。

→ 新証拠は「結線支持」ではなく「結線不要」の方向。cycle-186 の判断は現在も妥当。

### 3. 位置づけ（撤去は機能の全否定ではない）

撤去は「検索という機能の価値の全否定」ではなく「現状 dead code のまま毎ビルド 673KB を生む状態の解消」。将来フェーズC（クラスタ再設計）で改めて探索導線が必要になれば、その時の要件で作り直す判断を別途行う。

なお、データ調査の副次所見として、**一覧ページのローカルフィルタ**（辞典の `DictionarySearch`・`/blog` 一覧フィルタ・`/tools/keigo-reference` の絞り込み。いずれも撤去対象のグローバル検索モーダルとは別機能で自己完結しており、撤去スコープに含めない）は極小ながら実利用の痕跡がある。投資するなら「グローバル検索」ではなく「一覧の見やすい導線・ローカル絞り込み」——これは本タスクのスコープ外だが、気づきとしてキャリーオーバーに記す。

## 撤去に伴う公開記事の是正（レビュー指摘・必須）

白紙再レビューで、撤去だけでは来訪者に虚偽が残ると判明した。公開済みブログ記事 `2026-02-21-site-search-feature.md`（「サイト内検索を実装しました。Cmd+K で開けます」）が検索機能を来訪者に告知しており、コードを撤去すればこの記事は恒久的に誤りになる。撤去を「誠実な健全化」と位置づける以上、記事側の是正を撤去スコープに必ず含める。

対処方針は blog-writing.md の確立ルールに従う——**本文は当時の記録としてそのまま保ち、書き換えない**（撤去済み機能を現在形で紹介していても「当時こう書いた」という記録であり、本文改変は記録の改竄＝読者を欺く）。代わりに **冒頭に日付つき注記** を加え、現時点では機能が撤去済みであることを伝える。前例: `nextjs-route-handler-...md` のメモフィード廃止追記、`series-navigation-ui.md` 冒頭注記。

- `2026-02-21-site-search-feature.md`: 冒頭に日付つき注記を追加（本文不変）。→ blog-writer に委譲。
- `2026-02-21-dark-mode-toggle.md` L29 の内部リンク `/blog/site-search-feature`: 記事は削除せず注記のみのため **リンクは依然有効。ルール上いじらない**（本文書き換え・リンク除去はしない）。
- 撤去の経緯を語る後続記事（回顧記事）の要否は、読者価値の観点で blog-writer に判断させる（薄い作業報告なら書かない＝CLAUDE.md／cycle-290 の学び）。
