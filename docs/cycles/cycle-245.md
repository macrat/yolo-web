# サイクル-245

このサイクルでは、前サイクル（cycle-244）で復元した早見表記事6本と、それぞれの対になる既存ガイドとの相互リンクを完成させる（backlog B-513）。

## 背景

cycle-244 で、削除されていた早見表コンテンツ6トピック（regex / cron / git / sql / markdown / html-tags）を独立した早見表記事として `/blog` に復元した。このとき、各早見表記事からは対になるガイド記事へのリンク（「なぜそうなるかを理解したい場合はガイドへ」）を設置した。

しかし逆方向 ——ガイド記事から対の早見表記事へのリンク—— は設置していない。複数の reviewer が「回遊性向上のため逆リンクを推奨」と挙げており、cycle-244 では非ブロッキングの別タスクとして B-513 に切り出した。

これは来訪者価値の取りこぼしである。ガイド記事を読んでいる読者の一部は「もう仕組みは分かったから、あとは構文を素早く引きたい」という状態に移る。あるいは検索でガイドに着地したものの本当に欲しかったのは早見表だった、という読者もいる。早見表→ガイドの片方向リンクだけだと、こうした読者はガイド側から早見表に渡る道を持たない。せっかく早見表を取り戻したのに、ガイドにいる読者からは見えていない。相互リンクにすることで、読者は自分の意図（理解したい／引きたい）に合う方へ自由に移動できる。

## 作業計画

### 目的

早見表↔ガイドの相互リンクを完成させ、ガイドにいる読者が自分の意図（構文を素早く引きたい）に合う早見表へ移動できるようにする。回遊性を高め、cycle-244 で復元した早見表コンテンツを実際に読者の目に触れさせる。

### 作業内容

各ガイド記事の本文に、対になる早見表記事への逆リンクを1箇所追加する。配置と文言の方針:

- 読者意図に沿った導線にする。「仕組み・理由はこのガイドで解説しているが、構文を素早く引きたいなら早見表へ」という、早見表側に置いた文言と対をなす案内にする。
- アンカーテキストは具体的に（「こちら」「詳しくはこちら」は使わない。blog-writing.md SEO ガイドライン準拠）。リンク先記事の内容を示す語にする。
- 配置は各記事の構成に合わせて blog-writer が自然な位置を判断する。「素早く引きたい読者」を取りこぼさない観点では冒頭付近の導線が効きやすいが、記事の流れを乱さないことを優先する。
- 本文の主題・既存の記述は変更しない。早見表への案内を1箇所足すだけにとどめる（cycle-244 の早見表記事と対称な最小追加）。
- frontmatter の updated_at を、本文を実質変更するため、コミット直前の実時刻（`date +"%Y-%m-%dT%H:%M:%S%z"`）に更新する。それ以外の frontmatter は不変。

1記事につき blog-writer 1人を割り当てる（CLAUDE.md「Keep task smaller」）。

### 検討した他の選択肢と判断理由

- **逆リンクを追加しない（現状維持）**: 不採用。早見表→ガイドの片方向では、ガイドにいる読者が早見表に渡れず、復元した早見表コンテンツが届かない読者層が残る。cycle-244 の reviewer 複数が推奨した回遊性改善であり、来訪者価値を優先する憲法・CLAUDE.md の意思決定原則に照らして実施する価値がある。
- **逆リンクの配置を全記事で機械的に統一（例: 必ず末尾）**: 不採用。記事ごとに構成（導入の長さ・CTA の位置）が異なり、自然な位置は一律でない。読者の流れを乱さない位置を記事ごとに判断する方が UX が良い。
- **早見表とガイドを1記事に統合し直す**: 不採用。cycle-244 で「早見表を引く読者とガイドで理解する読者は別人であり、1記事に詰めると双方を損なう」と判断し独立記事として復元したばかり。相互リンクこそがその設計の意図（読者が意図に合う方へ移動できる）を完成させる。
