# サイクル-191

cycle-190 は同じ B-314（Phase 7 第 1 弾）に着手したが、旧コンセプト時代の構造（cycle-160 の 3 ゾーン構成 / FaqSection / RelatedTools / RelatedBlogPosts / 現行 Component.tsx レイアウト）を盲目的に踏襲し、機械的なトークン置換と Panel ラップに留まったため、B-314 の本来の目的（後続 53 サイクルが乗る基盤＝新コンセプト下でゼロベース再設計された詳細ページ / 共通コンポーネント / サイズ × 機能のタイルシステム）のいずれも達成できず失敗としてクローズした（全変更を着手前に revert）。

本サイクルでは、cycle-190.md 末尾の振り返り「本当ならやるべきだったこと」(A〜E) に立ち返り、第 1 弾サイクルの本来の責務を果たす。**主目的は「後続 53 サイクルが乗る基盤を整え、その上で第 1 弾コンテンツを 1 件目として移行する」こと**。

## 実施する作業

### サイクル全体の運用ルール

本サイクル中、すべてのタスク（計画書改訂・設計・実装・レビュー対応を含む）に対して以下を **能動性に依存しない構造的ルール** として適用する。違反があれば該当タスクを Done にしない。

1. **実体確認後に書く**: 計画書 / 設計ドキュメント / 実装中に登場する API 名 / prop 名 / フィールド名 / ファイルパス / 他タスクの状態は、Read / grep / ls で実体を確認した **直後** に書く（AP-WF12 / AP-P16 本文準拠）。確認結果を表として `./tmp/` に切り出すのは **しない**（cycle-190 違反 13「能動性依存仕組みのプレッシャー下崩壊」の構造的回避: 表更新義務を作らず、書く瞬間に必ず実体を見る運用にする）。
2. **ファイルを書く前に必ず Read**: 既存ファイルに追記・編集する際は、対象ファイルを **その編集の直前に** Read する。記憶や前回の Read 結果に依拠しない。
3. **reviewer 指摘の最終判断は PM が行う**: reviewer の指摘は false positive 候補も含めて PM が一件ずつ判断する。「reviewer が指摘したから直す」を機械的に適用しない（reviewer 暴走で本サイクルの目的を見失わない構造的歯止め）。判断結果と理由をサイクルドキュメントに残す。false positive 判定の理由は **指摘単位で 1 行以上記述** し、サイクルドキュメントの「レビュー結果」セクションに残す。理由が「スコープ外」のみは不可。
4. **PM Read 観察の縮小は本サイクル中不可**: 計画書で具体的に列挙した PM Read 観察対象は context 圧迫を理由にしても縮小しない。縮小せざるを得ない場合は運用ルール 5 を適用し本サイクルを失敗クローズする（計画書改訂による縮小も本サイクル中は不可）。
5. **境界拡張の禁止**: 計画書で「サイクル中に絶対に拡張しない」と明示した境界（Phase D の実装範囲など、後述）を拡張せざるを得ないと判断した場合は、**本サイクルを失敗としてクローズし次サイクルで再着手** する。サイクル中の合意なき境界拡張は行わない。失敗クローズ判断時は、その理由（拡張が必要になった具体的経緯）をサイクルドキュメントの「キャリーオーバー」セクションに 3 行以上で記録する。
6. **計画書からの逸脱時は計画書改訂を先に行う**: 設計タスクの実装中に B / C 等への手戻りが必要になったら、実装側で吸収（OptionalProperty 追加 / 個別ハードコード等）するのではなく、計画書 / 設計ドキュメントを先に改訂してから実装に戻る（AP-I02 / cycle-190「計画書からの執行中逸脱」構造的回避）。

**スコープ採用方針**: cycle-190.md「本当ならやるべきだったこと」A〜E のうち、本サイクルでは **A（ゼロベース設計）+ B（共通コンポーネント整備）+ C（タイルシステム整備）+ C の検証用に限定した D ミニ適用** までを実施する。第 1 弾コンテンツの詳細ページ全体の (legacy)→(new) 移行（D の本体）は次サイクル（cycle-192）に送り、cycle-191 はあくまで「後続 53 サイクルが乗る基盤を確実に整える」ことを最優先する。理由・他案との比較は「### 検討した他の選択肢と判断理由」参照。

- 直接の受益者は **次サイクル以降のすべての visitor**。本サイクル単体ではどの (new) ルートも公開せず、visitor が目に見える変化を体験する場面は実質的にない。しかし、次サイクル（cycle-192）で第 1 弾コンテンツの (new) 移行が完了した時点から後続 53 サイクル全体にわたって、新コンセプトに沿ったツール詳細体験が visitor に届く。基盤が不適切なまま 53 ツール / 遊びを移行すれば、cycle-190 と同じ構造で 53 件分の機械置換が積み重なり後戻りが困難になる
- 副次的に **次サイクル以降の PM / builder / reviewer**。設計ドキュメント・コンポーネント・タイルシステムが「型に値を埋めるだけで完了する状態」（cycle-190.md L454）に近づくことで、各サイクルの判断負荷が下がり、品質のばらつきが減る

**第 1 弾サイクルの本来の責務との整合**:

- A（ゼロベース詳細ページ設計）: 完遂する
- B（共通コンポーネントを新コンセプト用に整える）: 完遂する
- C（タイルシステム = サイズ × 機能のバリアント設計と Tileable 型完成形）: 完遂する
- D（第 1 弾コンテンツ 1 件を新設計で実装）: 設計検証用の部分実装のみ（詳細ページ全体の (legacy)→(new) 移行は次サイクル）
- E（品質保証）: 基盤化の観点で完遂する

D を分割した理由は「### 検討した他の選択肢と判断理由」セクションの S1〜S4 比較を参照。

### 作業内容

#### Phase A — ゼロベース詳細ページ設計

cycle-190 では「設計判断 X1（現行 3 ゾーン維持）」を計画段階で即座に確定し、新コンセプト下でツール詳細ページに何が必要かをゼロから考える検討を完全に飛ばした。本サイクルではこれを根本から立て直す。

設計の出発点は **新コンセプト「日常の傍にある道具」の target user がツール詳細ページに何を求めるか**。旧構造（cycle-160 の 3 ゾーン / FaqSection / RelatedTools / RelatedBlogPosts / Component.tsx レイアウト）は **出発点にしない**。

**Phase A 出発点（再掲・厳守）**: target user dislike「ツール冒頭に長い解説」「以前と挙動が違う」は cycle-190 同型失敗および (new) 移行進行中の過渡期一貫性違反の核心。T-A 全タスク完了時にこの 2 dislike への対処方針を設計ドキュメントに明示する。

T-A は **発散 → 収束 → ドキュメント化** のステップで進める。発散段階で「12 件以上の要素列挙 + likes/dislikes 判定」、収束段階で「3〜5 階層への分類（cycle-160 の 3 ゾーンを踏襲しない独自階層）」、ドキュメント化段階で「`docs/design-migration-plan.md` への大量追記禁止」を構造的に守る。

##### 設計ドキュメントの保存場所

本サイクルで作成する設計ドキュメントは **`docs/` 直下または `docs/knowledge/`** に置く（`./tmp/` ではない。次サイクル以降にわたって参照する基盤資料のため）。具体的なファイル名・配置（例: `docs/tool-detail-page-design.md` / `docs/knowledge/new-concept-detail-page.md` 等）は実装段階の判断に委ねる。`docs/design-migration-plan.md` への大量追記は **避ける**（cycle-190 違反 9「テンプレ先取り」の構造的回避: T-A-ドキュメント化 の Done 条件で `git diff --stat` 空を機械チェック）。

#### Phase B — 共通コンポーネント整備

T-A で確定した要素を提供するためのコンポーネント群を **新規実装** する。旧版（FaqSection / RelatedTools / RelatedBlogPosts / ToolLayout 等）を出発点にしないため、新版が旧版と 1:1 対応するとは限らない。

##### コンポーネント名前空間方針

cycle-190 違反 2 の構造的再発防止のため、以下の方針を取る:

- 旧版 `src/components/common/` 配下のファイルは **本サイクルで触らない**（既存 (legacy) 33 ツール + GameLayout が依存している共有ファイルへの破壊的変更を物理的に不能にする）
- 新版コンポーネントは `src/components/` 直下（既存の Panel / Button / Input 等と同階層）に新規ディレクトリとして配置するか、`src/tools/_new-components/` 等のスコープを切るか、を T-B-設計 で判断する。判断軸は (i) 後続 53 サイクルで共通利用される範囲、(ii) Phase 8（辞典系移行）でも共有されるか、(iii) Phase 10.2（`src/components/common/` 削除）との整合性
- 名前は新版用の名前を新規に与える。旧版と同名にすると混乱の元（FaqSection という名前自体が新コンセプトに合うかから問い直す）
- `old-globals.css` の値（特に `--r-normal` / `--bg` 等）は **触らない**。本サイクルで定義する新トークン値は `globals.css` または新規 CSS Module 内に閉じる（cycle-190 違反 3 構造的回避）
- 構造的歯止め: T-B-設計 / T-B-実装 / T-D-実装 の Done 条件で `git diff --stat src/components/common/ src/app/(legacy)/ src/app/old-globals.css` が空であることを機械チェック

#### Phase C — タイルシステム整備

cycle-190 の最大の失敗の一つは「タイルバリアントをサイズのみで捉え、機能切り出しの可能性を見落とし、単一の軽量版を実装しただけ」（cycle-190 違反 4）。本サイクルでは **サイズ × 機能のマトリクス** で複数バリアントを設計する。

T-C-事例発散 では **3 件（keigo-reference / sql-formatter / char-count）** を計画段階で確定（後述「### 第 1 弾コンテンツ確定」）し、各事例で「全機能列挙 → 意味的グループ分け → サイズ × 機能の切り出し候補」を試す。3 件の試行から **共通する要素 / 共通しない要素** を独立に列挙し、共通しない要素は型契約に入れない（cycle-190 違反 8 構造的回避）。

3 件で帰納が成立しなかった場合の対応も計画段階で確定: **次サイクルに 4-5 件目を追加検証**（サイクル中の事例拡張は禁止、運用ルール 5 適用対象）。

T-C-型契約 で Tileable 型に反映する。toolbox-registry の codegen / `toTileable()` adapter 関係も運用ルール 1 に従い実体確認直後に記述する。

T-C-検証場所 でタイル単独レンダリング検証手順を整備する。design-migration-plan.md L308「`/storybook` は共通コンポーネント専用」を尊重し、タイル検証は別場所（`/storybook/tiles` 等）に分離する判断を含めて整備する。`INITIAL_DEFAULT_LAYOUT` への投入は本サイクル不実施（Phase 9 配線実装時の前提として T-E-申し送り で独立起票）。

#### Phase D — 第 1 弾コンテンツによる設計検証ミニ適用

**D の責務分割の明確化**（cycle-190 失敗の核心と直結する重要論点）:

- 本サイクルの D = B / C の **設計検証用ミニ適用** のみ。第 1 弾コンテンツの (legacy)→(new) 詳細ページ移行は **次サイクル**
- 本サイクルで作る Tile.tsx は **small 1 件 + medium 1 件のみ**、両者は **異なる機能切り出し**（同じ機能のサイズ縮退ではない）
- `INITIAL_DEFAULT_LAYOUT` への投入禁止、(legacy) ルートのページファイル不触、Phase D 中の Tile.tsx 追加（large 等）禁止
- 境界拡張が必要と判断されたら運用ルール 5 に従い **本サイクル失敗クローズ**

T-D-バリアント設計 / T-D-実装 の核心は「B / C で整えた設計が実コンテンツで成立するかを検証する」こと。検証中に設計の不足が見つかれば運用ルール 6 に従い B / C に戻る（実装側で吸収しない）。対象コンテンツの logic / meta はゼロベース再評価し、結果を設計ドキュメントに記録する（論点 6 項目 5）。

#### Phase E — 品質保証・基盤化の確認

T-E-視覚回帰 では (legacy) 33 ツール + 20 遊びへの副作用が無いことを実機で確認する。本サイクルは共有ファイルを構造的に触らない設計にしているが、それでも `globals.css` のトークン追加 / 新規コンポーネント追加が予期せぬ波及をしないかを確認する。観察対象は後述「### 視覚回帰観察対象」で計画段階に確定。Playwright で複数幅 × ライト/ダーク撮影し、PM が Read で観察（reviewer 代行禁止、運用ルール 4 適用）。

T-E-設計ドキュメント検証 では設計ドキュメントの自己完結性を 7 項目チェックリストで確認する（タスク Done 条件に列挙済み）。

T-E-申し送り で後送り項目を独立 backlog 起票する。各起票項目に必ず (i) 着手条件 (ii) 完了条件 (iii) 参照ドキュメント (iv) 他項目との依存関係 の 4 項目を記述する。具体的な起票項目はタスクの Done 条件に列挙済み。

### 第 1 弾コンテンツ確定

cycle-190 で keigo-reference を選んでいたが、本サイクルでは「設計検証用としての適性」と「target user 2 経路の likes に対する到達度」を判断軸に再評価し、**keigo-reference を第 1 弾コンテンツとして計画段階で確定** する。T-C-事例発散 の 3 件は **keigo-reference / sql-formatter / char-count** で確定。

#### 評価軸

軸 0（最優先）から軸 4 の順で評価:

0. **target user 2 経路の likes に対する到達度** — 検索流入経路の likes（特定の作業に使えるツールをさっと探している人）と再訪経路の likes（気に入った道具を繰り返し使っている人）の両方に高い到達度を持つか
1. **タイルバリアント設計の検証になるか** — サイズ × 機能の複数バリアントが自然に成立するコンテンツか。機能が単純すぎると small バリアントしか作れず C の検証として不十分。複雑すぎると設計の本質と無関係な実装労力が膨らむ
2. **後続 33 ツール / 20 遊びへの設計波及性** — このコンテンツで成立する設計が他コンテンツでも成立するか
3. **GA4 PV** — visitor 価値の大きさ。ただし PV 単独で決めない
4. **logic / meta のゼロベース検証コスト** — A〜C の設計と並行して logic / meta も新コンセプトで再評価する必要があるため、ある程度シンプルな方が本サイクル内で完遂しやすい

#### 第 1 弾確定の根拠

**第 1 弾 = keigo-reference**。確定根拠:

- 軸 1（バリアント設計適性）で 4 機能の自然な切り出しがあり small/medium 異なる機能切り出しの実例が作りやすい
- GA4 PV 1 位（46）で本サイクルの D ミニ適用が次サイクル本移行時に visitor 価値発現に直結

#### T-C-事例発散 の 3 件確定根拠

**3 件 = keigo-reference / sql-formatter / char-count**。選定根拠:

- 後続 33 ツールの設計波及性検証として最小十分

### 検討した他の選択肢と判断理由

#### 論点 1: スコープ分割

cycle-190 で詰め込み失敗が起きた直接の原因は、(1) 詳細ページ移行 + (2) ToolLayout 新デザイン化 + (3) docs 化 を 1 サイクルに詰め込んだこと。cycle-191 はさらに重い責務（ゼロベース設計 + 共通コンポーネント新規実装 + タイルシステム + 1 件適用 + QA）を負うため、スコープ分割を慎重に判断する必要がある。

| 案                | 内容                                                                                         | visitor 価値発現                                   | 失敗再生産リスク                                                           | 基盤整備の確実性                                                                                 |
| ----------------- | -------------------------------------------------------------------------------------------- | -------------------------------------------------- | -------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| S1                | A〜E すべて（基盤 + 共通 + タイル + 1 件適用 + QA）を本サイクルで                            | 本サイクル完了時に第 1 弾コンテンツが (new) で公開 | 高（cycle-190 同型詰め込み、設計 → 実装の途中で判断が劣化）                | △（実装労力に押されて設計品質が下がる）                                                          |
| S2                | A + B + C のみ。D（1 件適用）+ E は次サイクル                                                | 次サイクル完了時に第 1 弾コンテンツが (new) 公開   | 中（B 実装で C の前提が必要になり順序ジレンマ）                            | ◯（B / C を独立して最後まで詰められる）                                                          |

| **S2 改（採用）** | A + B + C + 「C 検証用の D ミニ適用」（小規模実装）。詳細ページ全体の (new) 移行は次サイクル | 次サイクル完了時に第 1 弾コンテンツが (new) 公開   | 低（D を「C の検証」に限定することで実装労力が膨らまない、設計品質を維持） | ◎（C の検証を実コンテンツで行えるため設計の現実性が確保される、かつ B / C の手戻りが許容される） |

**確定 = S2 改**。判断理由（visitor 価値 / 失敗再生産回避 / 基盤整備の確実性 の 3 軸）:

- **visitor 価値**: S1 と比べて第 1 弾コンテンツの (new) 公開が 1 サイクル遅れるが、後続 53 サイクル全体の visitor 価値が「不適切な基盤の上に積み重なる cycle-190 同型の失敗」によって損なわれるリスクと比較すれば、基盤の品質を確実にする方が **総和で** 大きい
- **失敗再生産回避**: cycle-190 詰め込み失敗の核心は「設計が固まる前に実装が走り、実装の制約が設計を歪める」構造。S2 改は D を「C 検証用ミニ適用」に限定することでこの構造を物理的に避ける。D 実装中に B / C への手戻りが許容される設計（T-D-実装 / T-B-実装 後段、運用ルール 6）も同じ目的
- **基盤整備の確実性**: S3 は設計が机上空論化するリスクがあり、S4 は C が次サイクルに送られると B の I/F が後で変わる。S2 改は C の型契約を実コンテンツで検証してから確定するため、後続 53 サイクルが乗る基盤として最も安定する
