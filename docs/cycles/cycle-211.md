# サイクル-211

このサイクルでは、B-314（ツール・遊び詳細ページの新デザイン移行 + タイル化 / 移行計画 Phase 8.1）の第 12 弾として、`image-base64` ツールを `(legacy)/tools/image-base64/` から `(new)/tools/image-base64/` へ移行し、トップページ用タイルを新規実装する。これは Phase 8.1 における**画像入力型タイル初回**（FileReader 非同期パターン応用 / file input × Canvas / blob → base64 文字列出力）であり、cycle-210 で確立した「複合入力型タイル」SSoT に続く第 2 の新パラダイム導入。

### 目的

`image-base64` ツールを新デザイン規約（`(new)/`）へ移行し、トップページに置けるタイル UI を初めて画像入力型として実装する。これにより:

1. **来訪者価値**: 既存ツールの新デザイン化により、ダークモード / a11y / モバイルタップターゲット等の改善を `image-base64` 利用者にも届ける。タイル化でトップページから 1 タップ起動が可能になり、画像 → base64 文字列化のフローが短縮される。
2. **Phase 8.1 標準パターン拡張**: 画像入力型タイル（file input + 非同期 FileReader 読込 + プレビュー + 出力文字列）の SSoT を確立し、後続の `image-resizer`（cycle-212 前後）および B-318 / B-371 / B-372 等の画像系ワークフローツール群（複数サイクル相当）に再利用可能な基盤を作る。

### 作業内容

`/cycle-planning` で詳細化する。主要論点（planner 着手前のラフ）：

- タイルサイズ判定（cols × rows）= 画像入力型は textarea より縦に詰まる可能性が高い。複合入力型 cols=3 rows=3（400×400px）を参照しつつ、画像プレビュー領域の最小可視性を加味して決定。
- AP-P21 判定基準の画像入力型タイル特有適用範囲（cycle-210 補足事項 4 項目 (i)(ii)(v)(vi) との対比）。
- FileReader 非同期パターン（onload / onerror / progress）の標準化。
- 既存 `image-base64` のテスト基盤確認 + 必要に応じて Component.test.tsx 整備。

### 検討した他の選択肢と判断理由

cycle-210 完了ノートで示された 3 候補から (a) image-base64 を選択：

- **(a) image-base64 / image-resizer（採用）**: 画像入力型タイル初回。来訪者ボリュームが大きく（base64 化はエンジニア向けでも image-resizer は一般来訪者層も対象）、新パラダイム確立で後続の image-resizer + B-318 系画像ワークフロー群を一括で前に進める。image-base64 を先にすることで「FileReader → 文字列出力」の単純構造から始めて、image-resizer の「FileReader → Canvas 操作 → blob 出力」へ段階的に複雑度を上げられる。
- **(b) regex-tester / text-diff**: 複合入力型タイル 2 件目（B-452 N≥3 蓄積への寄与）。SSoT 引用の最初の適用先として価値は高いが、(a) と比べて新パラダイム確立の波及効果が小さい。次サイクル以降に回す。
- **(c) dummy-text / password-generator 等の単純構造ツール**: 既存パターン踏襲で実装コストは低いが、新パラダイム確立の機会を逸する。複合入力型・画像入力型を先に潰してから戻る方が SSoT の網羅性が高まる。
