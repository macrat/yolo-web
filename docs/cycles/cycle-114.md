- **DALL-E 3**: $0.04~$0.08/画像でImagen 4 Fastの2~4倍高い。新規依存先追加

- **ストック画像**: 手動選定が必要でAIワークフローと相性が悪い
- **Satori/CSS改善のみ**: cycle-113で「来訪者価値よりも安易な実装を優先」として中止された方針
- **Gemini 2.5/3.1 Flash Image**: 推奨候補として提案書に含めた（Imagen 4との比較テストを推奨）

## レビュー結果

計画に対して3回のレビューを実施した。

### レビュー1回目（6件の指摘）

1. CTR数値根拠が不正確（「5%~300%向上」）→ 信頼できるソースに修正
2. コーディング原則ルール2への抵触検討不足 → A/B/C 3案をowner確認事項に追加
3. ビルド時生成 vs ランタイム生成の選択肢未明示 → 6観点の比較表を追加
4. AI生成画像の表示義務未検討 → constitution.md ルール3に基づく方針を追加
5. Phase 3が時期尚早 → 「将来的な拡張可能性」として言及に格下げ
6. Gemini Flash選択肢なし → Gemini 2.5/3.1 Flash Imageを追加

### レビュー2回目（4件の指摘）

1. CTR数値根拠の引用が誤解を招く → YouTube研究を主要根拠、Google Discoverは参考情報として区別
2. Gemini 3.1 Flash Image価格が不正確 → 1K解像度で$0.067/画像に修正
3. 案(B)の根拠不十分 → coding-rules.mdの`paths: src/**/*`指定への言及を追加
4. Gemini 2.5 Flash ImageのBatch API割引未記載 → $0.0195/画像を併記

### レビュー3回目

指摘事項なし。計画承認。

### ownerフィードバックによる提案書再設計

ownerメモ（ID: 19c900e1b1d）を確認した結果、初版提案書がブログサムネイルに焦点を絞りすぎており、ownerの元の要望（各種コンテンツのサムネイル・挿絵・OGP画像をAIチームが自律的に生成できる仕組み設計）を見落としていたことが判明。提案書を全面再設計。

### レビュー4回目（再設計後、6件の指摘）

1. Imagen 4のBatch API単価が存在しない（事実誤認）→ N/Aに修正、Gemini系のみBatch利用可能と注記
2. RPDとIPMの用語混乱 → IPM（Images Per Minute）に統一
3. Spend capの遅延リスク未記載 → 約10分の遅延を明記、第1層が真の防御線と明確化
4. Gemini 2.5 Flash Imageのinput token料金未注記 → 備考欄に追記
5. カウンターファイルの排他制御未言及 → ファイルロック実装の必要性を追記
6. エグゼクティブサマリーのコスト不整合 → セクション7と整合（$1.56/78枚）

ownerからの追加指摘: 挿絵を入れない基準の「500文字以下」が恣意的 → 文字数閾値を削除、ケースバイケース判断に変更

### レビュー5回目（1件の指摘）

1. Imagen 4 Standardの最大解像度が不正確（2816x1536 → 2048x2048）→ 提案書・調査資料の両方を修正

### レビュー6回目

指摘事項なし。提案書承認。

### MCP調査追加とMCPサーバー方式への提案書改訂

レビュー中にMCPサーバー方式がCLIスクリプト方式より安全（構造的なAPIキー隔離）であることが指摘され、MCP調査を追加実施。提案書をMCPサーバー推奨に全面改訂。

### レビュー7回目（MCP改訂後、4件の指摘）

1. APIキー隔離の主張が不正確（シェル環境変数経由でアクセス可能）→ 多層防御方式に変更、限界を正直に記載
2. ディレクトリ名不一致 → `mcp-image-server/`に統一
3. 日次/月次制限の不一致 → 注記追加
4. 「原理的に困難」の過大表現 → 修正

### レビュー8回目

指摘事項なし。提案書最終承認。
