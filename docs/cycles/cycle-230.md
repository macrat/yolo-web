## 作業計画

### 目的

**派生元**: B-504（cycle-229 起票。Phase 10.1 に対応する backlog タスクの欠落を補うもの）および `docs/design-migration-plan.md` Phase 10.1「公開に向けた中核機構を追加実装する」。その原典は `docs/site-concept.md` L7「ツール間の連携や、気に入ったツールを1つの画面にまとめられるダッシュボード的な機能を通じて、『日常の傍にある』存在を目指す」。

現状の `/toolbox`（noindex プレビュー）は39枚のタイルが**全員に同じ構成で固定表示**されており、「生きたタイルが並ぶ」ことは cycle-226〜228 で達成済みだが、「**気に入ったツールを**まとめる」——つまり来訪者が自分で選ぶ・それが残る——はまだ何も実装されていない。これが Phase 10.1 の中核機構であり、P1 の B-336（道具箱本公開）の着手条件「Phase 10.1 + 10.2 完了」の前半を満たすための作業である。

GA 実測（2026-05-29〜06-11、GA4 property 524708437）: ツール詳細ページは各1〜6PV・/toolbox は1PV。守るべき既存利用はほぼなく、新コンセプトはまだ来訪者に届いていない。道具箱の公開準備を進めることが来訪者価値への最短経路で、かつ noindex のままなので本サイクルの変更が既存来訪者の体験を壊すリスクは極小。

### 作業内容

中核機構を「タイルの追加・削除（選択）＋構成の localStorage 永続化」と定義し、以下を実装する。

1. **カタログのデータ化**（T-3）: 道具箱に並べられるタイルの一覧（39エントリ）を単一ソース化する。現状は ToolboxContent.tsx に JSX としてハードコードされており、「来訪者が選ぶ」ためには選択肢の一覧がデータとして必要になる。
2. **構成の状態管理と永続化**（T-4）: 道具箱の構成（順序付きのタイルリスト）を localStorage に保持する。バージョン付きスキーマ・破損時フォールバックは既存のゲーム統計 storage（src/play/games/\*/\_lib/storage.ts）の先行パターンを参考にする。デフォルト構成は現行39枚と同等とし、Phase 10.2/10.3 の default content（プリセット）が後からこれを置換する。
3. **追加・削除 UI**（T-5）: 編集モード・モーダルなしのシンプルな形（タイルを外す操作＋カタログからの追加操作）。具体的な UI 形状は builder が DESIGN.md と frontend-design スキルに従って設計する。
4. **B-434 の確定**（T-2）: w360 の実態を観察してから方針を決める（cycle-200 の報告時とはタイル実装が全面的に変わっているため、現状を実測せずに対策を決めない）。

**DESIGN.md 恒久要件の本サイクルへの翻訳**（AP-P22 対応・指示書へ必ず含める）:

- タイルのルートは `<Panel>` のまま（既存タイルには手を入れない。本サイクルは道具箱ページ側の機構のみ）
- 追加・削除 UI も「すべてのコンテンツはパネルに収まる」原則に従う。UI 部品は src/components/ の共通部品を使う
- タイル選択が複数選択の性質を持つ場合のフォーム要素はチェックボックス可（DESIGN.md §5）
- アイコン原則なし（必要なら Lucide 線画 1.5px / 16・20・24px ＋ aria-label）、絵文字禁止、装飾色禁止
- ブレークポイントで形を変えるのではなく、どの画面幅でも自然なレイアウト（DESIGN.md §6）
- フォーカス表示は `outline: 2px solid var(--accent); outline-offset: 2px;`
- 編集モード・DnD の視覚表現ルール（DESIGN.md §4）は本サイクルでは**適用対象外**（編集モード・DnD 自体を作らないため）。誤って部分的な編集モード風 UI を作らない

**スコープ外**（移行計画 Phase 10.1 の「公開後に順次追加する機能」の整理に従う）: DnD 並べ替え・編集モード・Undo・モーダル・多タイル管理機構の高度化、タイル間連携（10.4）、シェア（10.5）、プリセット（10.2）、本公開・URL 決定（10.3）、タイル入力値の保持（B-433）、型契約／レジストリの全面再設計（B-502）。

**完了条件**: 来訪者（プレビュー閲覧者）が `/toolbox` でタイルを外す・追加することができ、リロード後・再訪問後もその構成が残る。tile-architecture.md §4 の合格条件（ページ遷移なしに動く等）が回帰していない。4ゲート全通過。B-434 の方針が確定・記録されている。

### 検討した他の選択肢と判断理由

| 案                                                          | 内容                                                               | 判断                                                                                                                                                                                                                                                    |
| ----------------------------------------------------------- | ------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **A: 選択＋永続化（採用）**                                 | タイルの追加・削除と構成の localStorage 永続化のみ。並び順は追加順 | 原設計の核「気に入ったツールをまとめる・残る」を最小の実装で成立させる。Phase 10.1 の「中核機構」の定義に合致し、1サイクルで実機検証まで完了できる規模                                                                                                  |
| B: フル編集モード（DnD 並べ替え・Undo・モーダル込み）       | ダッシュボード編集機能を一括実装                                   | 移行計画が「公開後に順次追加する機能」と明確に整理済み（cycle-229 で原設計へ訂正したばかり）。一括実装は規模超過で、約50サイクルの失敗の主因だった「複雑機能の全揃え優先」への逆戻り                                                                    |
| C: 永続化なしの一時選択のみ（または現状維持で 10.2 へ直行） | 実装最小。選択はリロードで消える                                   | 「気に入ったツールをまとめる」がリロードで消えるなら来訪者にとって意味がなく、サイトコンセプトの「毎日使える」と矛盾。Phase 10 完了基準にも「配置が localStorage に保持される」が含まれており、どこかで必ず実装する。中核機構から永続化を外す理由がない |

なお移行計画 L234 の「localStorage **配置**永続化は公開後に順次追加」との関係: あの列挙は DnD 等とセットの「配置（並べ替え結果）の永続化」を指すと読む。本サイクルが実装するのは「構成（どのタイルを置くか）の永続化」であり、これ無しでは中核機構（まとめられる）が成立しないため Phase 10.1 に含めると PM が判断した。具体仕様の判断は Phase 10.1 着手サイクル PM に委ねられている（移行計画 L236）。この読みの妥当性は T-1 計画レビューの確認対象に含める。
