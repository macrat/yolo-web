# サイクル-135

GAデータ分析の結果、ホームページに111PV（直近2週間）の訪問があるにもかかわらず、play系コンテンツへの遷移が33PV（/play一覧16PV + 個別ページ17PV）にとどまっている。エンゲージメント率も55%と低く、多くの訪問者がホームページで離脱している。

このサイクルでは、ホームページのUXを改善し、訪問者がplay系コンテンツを見つけやすく、クリックしたくなるデザインに改善する。play系コンテンツは20種あり量は十分なので、「コンテンツの追加」ではなく「既存コンテンツの見せ方の改善」にフォーカスする。

## 実施する作業

- [x] 1. 現状分析: ホームページの現在のデザインと導線を確認し、改善ポイントを特定する
- [x] 2. 競合調査: 人気のある診断・占いサイトのホームページデザインを調査する
- [x] 3. 改善計画の策定: 具体的なデザイン変更の計画を立てる
- [ ] 4. 実装: ホームページのUI/UX改善を実装する
- [ ] 5. ビジュアル確認・レビュー

## 作業計画

### 目的

ホームページからplay系コンテンツへのクリック率を向上させ、エンゲージメント率（現在55%）を改善する。具体的には以下の3つの問題を解決する。

1. **未掲載コンテンツの発見可能性**: 全20コンテンツ中8件がホームページに未掲載（word-sense-personality, contrarian-fortune, unexpected-compatibility, traditional-color, impossible-advice, character-fortune, science-thinking, japanese-culture）
2. **ハードコードによるメンテナンス性の低さ**: FEATURED_SLUGS, DIAGNOSIS_SLUGS が手動管理であり、新コンテンツ追加時に更新漏れが発生しやすい
3. **ページの縦長構造**: 6セクションが縦に並んでおり、下部コンテンツ（デイリーパズル、ブログ）へのスクロール到達率が低い

### データモデル: カテゴリ別コンテンツ一覧

タブUI設計の前提となる、実際のデータモデル（`src/play/types.ts` の `category` フィールド）に基づくカテゴリ分類は以下の通り。

| category (データモデル) | タブ表示名 | 件数 | 該当コンテンツ                                                                                                                                                                                                                                    |
| ----------------------- | ---------- | ---- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| personality             | 診断       | 12件 | animal-personality, character-personality, music-personality, yoji-personality, word-sense-personality, impossible-advice, contrarian-fortune, unexpected-compatibility, traditional-color, character-fortune, science-thinking, japanese-culture |
| knowledge               | 知識       | 3件  | kanji-level, kotowaza-level, yoji-level                                                                                                                                                                                                           |
| game                    | ゲーム     | 4件  | kanji-kanaru, yoji-kimeru, nakamawake, irodori                                                                                                                                                                                                    |
| fortune                 | (タブなし) | 1件  | daily                                                                                                                                                                                                                                             |

**タブ名称の選定根拠**:

- 「診断」: personality カテゴリには性格診断・占い風診断・相性診断など多様なコンテンツが含まれる。ユーザーにとって「診断」は最も包括的で直感的な名称
- 「知識」: knowledge カテゴリはレベル判定型のクイズ。「クイズ」より「知識」の方が「自分のレベルを測定する」というコンテンツの本質を伝える。また、personalityカテゴリにもクイズ形式のコンテンツがあるため、「クイズ」だと混乱を招く
- 「ゲーム」: game カテゴリはデイリーパズル系。そのまま直感的
- fortune（1件のみ）はタブとして独立させず、FortunePreviewセクションで専用UIとして表示する（後述）

### 作業内容

#### 作業1: レジストリの自動化（registry.ts の改修）

**何をするか**: FEATURED_SLUGS と DIAGNOSIS_SLUGS のハードコードを廃止し、カテゴリベースで自動的にコンテンツを取得する仕組みに変更する。

**なぜやるか**: 現在、新コンテンツを追加してもトップページに自動露出しないため、cycle-134で追加した word-sense-personality が未掲載のまま放置されている。今後もコンテンツが増えるたびに手動更新が必要な構造は持続不可能。

**どのようにやるか**:

- `getFeaturedContents()` を改修し、ヒーローピックアップ用のコンテンツを自動選出する方式にする（詳細は後述「ヒーローピックアップの選出ロジック」参照）
- `getDiagnosisContents()` は不要となるため削除する（タブUIに統合されるため）
- FEATURED_SLUGS, DIAGNOSIS_SLUGS 定数は削除し、ロジックベースの選出に置き換える
- カテゴリ別のコンテンツ取得は既存の `getPlayContentsByCategory()` をそのまま活用する

**対象ファイル**: `src/play/registry.ts`

#### 作業2: セクション構成の再設計（page.tsx の改修）

**何をするか**: 現在の6セクション構成を再設計し、全20コンテンツが発見可能かつページがコンパクトになる構成にする。

**なぜやるか**: 現在の構成では「まずはここから」3件 + 「もっと診断してみよう」4件 + デイリーパズル4件 + FortunePreview 1件 = 12件しか表示されず、残り8件は /play 一覧に遷移しないと見つからない。ページも非常に長く、下部セクションの視認率が低い。

**どのようにやるか**: 以下の新しいセクション構成にする。

1. **ヒーローセクション**（維持・微調整）: CTAボタンとピックアップ3件は維持。ピックアップ3件を自動選出に変更（後述「ヒーローピックアップの選出ロジック」参照）
2. **「おすすめ」セクション**（「まずはここから」を改名・拡張）: カテゴリ別タブ切り替えUI（「すべて」「診断」「知識」「ゲーム」の4タブ）を導入し、タブUIで全19コンテンツ（fortune の daily を除く）を閲覧可能にする。デフォルト表示は「すべて」タブで6件表示 + 「もっと見る」ボタン。タブ切り替えはClient Componentとして実装する
3. **「今日のユーモア運勢」**（維持）: FortunePreview はリピーター獲得に重要なため維持。fortune カテゴリ（1件: daily）はこのセクションが専用UIとして担当するため、タブUIには含めない
4. **「開発の舞台裏」**（維持・位置変更なし）: ブログセクションは維持

**削除するセクション**:

- 「もっと診断してみよう」: タブ切り替えUIに統合されるため不要
- 「デイリーパズル」: タブ切り替えUIの「ゲーム」タブに統合されるため不要

この変更により、6セクション -> 4セクションに削減され、ページがコンパクトになりながらも全20コンテンツへのアクセスが可能になる（タブUI内19件 + FortunePreview 1件）。

**モバイルナビゲーションについて**: 現状、モバイルではハンバーガーメニューを開かないとplay系コンテンツへの導線がない問題がある。今回のタブUI導入により、ファーストビュー直下の「おすすめ」セクションにplay系コンテンツが一覧表示されるため、モバイルでもスクロールだけでコンテンツを発見できるようになり、この問題は大幅に軽減される。ハンバーガーメニュー自体の改修はスコープ外とする。

**対象ファイル**: `src/app/page.tsx`, `src/app/page.module.css`, 新規タブ切り替えコンポーネント

#### ヒーローピックアップの選出ロジック

ヒーローセクションのピックアップ3件は以下のルールで自動選出する。

**選出ルール**:

1. fortune カテゴリ（daily）は除外する（FortunePreviewセクションと重複するため）
2. personality, knowledge, game の3カテゴリから各1件ずつ、計3件を選出する
3. 各カテゴリ内での選出基準は publishedAt の新しい順（最新のコンテンツが自動的にピックアップされる）
4. タブUIとの重複は許容する（ヒーローピックアップは「注目の3件」として機能し、タブUIは「全コンテンツ一覧」として機能するため、役割が異なる）

**根拠**: 3カテゴリから1件ずつ選出することで、サイトのコンテンツの多様性を初見のユーザーに伝えられる。publishedAt 基準にすることで、新コンテンツ追加時に自動的にピックアップが更新される。

#### 「すべて」タブのデフォルト6件の選出ロジック

「すべて」タブのデフォルト表示6件は以下のルールで選出する。

**選出ルール**:

1. fortune カテゴリ（daily）は除外する（FortunePreviewセクションで表示済みのため）
2. 各カテゴリから最低1件を確保し、カテゴリの多様性を担保する:
   - personality から4件（publishedAt の新しい順）
   - knowledge から1件（publishedAt の新しい順）
   - game から1件（publishedAt の新しい順）
3. 合計6件を表示する。カテゴリ内の順序は publishedAt の新しい順
4. 「もっと見る」ボタンで残り13件を展開表示する（展開後は全19件を publishedAt の新しい順で一覧表示）

**根拠**: 単純な publishedAt 順だと personality カテゴリが直近12件中6件以上を占めるため、knowledge（3件）や game（4件）がデフォルト表示から排除されてしまう。各カテゴリから最低1件を確保することで、初見のユーザーに「診断だけでなく、知識テストやゲームもある」というサイトのコンテンツの多様性を伝えられる。6件はモバイルで2列x3行、デスクトップで3列x2行に収まる数で、ファーストスクロールで全体が視認できる。

#### 作業3: カテゴリ別タブ切り替えコンポーネントの新規作成

**何をするか**: ホームページ用のカテゴリ別コンテンツ表示コンポーネントを作成する。

**なぜやるか**: 全19コンテンツ（fortune除く）をコンパクトに閲覧可能にするには、タブ切り替えが最もスペース効率が良い。競合サイト（Kuizy、MIRRORZ）でも採用されている実績のあるUIパターン。

**どのようにやるか**:

- Client Componentとして `src/app/_components/PlayContentTabs.tsx` を新規作成
- タブ構成（4タブ）:
  - 「すべて」: personality + knowledge + game の全19件（デフォルト6件 + 「もっと見る」で展開）
  - 「診断」(personality): 12件を全件表示
  - 「知識」(knowledge): 3件を全件表示
  - 「ゲーム」(game): 4件を全件表示
- fortune カテゴリ（daily）はタブUIに含めない（FortunePreviewセクションで専用表示するため）
- 各カテゴリタブではそのカテゴリの全コンテンツを publishedAt の新しい順で表示
- 既存の featuredCard スタイルを再利用し、デザインの一貫性を保つ
- モバイルではタブを横スクロール可能にし、カードは2列グリッドを維持
- 「毎日更新」バッジ: DAILY_UPDATE_SLUGS に含まれるコンテンツ（kanji-kanaru, yoji-kimeru, nakamawake, irodori）のカードには「毎日更新」バッジを表示する。旧「デイリーパズル」セクション削除後も、ゲームタブ内のカードでバッジを表示することでリピート訪問の動機を維持する
- 問数表示など既存のカード機能もすべて維持

**対象ファイル**: `src/app/_components/PlayContentTabs.tsx`, `src/app/_components/PlayContentTabs.module.css`

#### 作業4: テストの更新

**なぜやるか**: セクション構成の変更に伴い、既存テストが失敗する可能性が高い。新しいタブ切り替えUIの動作テストも必要。

**どのようにやるか**:

**対象ファイル**: `src/app/__tests__/page.test.tsx`

### 検討した他の選択肢と判断理由

#### 選択肢A: カルーセル（横スクロール）方式

- 競合サイト（Vonvon等）で採用されているパターン
- **不採用理由**: カルーセルは見えていないコンテンツの存在に気づきにくく、クリック率が低い傾向がある（Nielsen Norman Group の調査）。20件すべてを発見可能にするという目的には不適。

#### 選択肢B: 「すべて表示」方式（セクション統合のみ、タブなし）

- 全コンテンツを1セクションに並べ、6件表示 + 「もっと見る」で展開
- **不採用理由**: カテゴリ別のフィルタリングがないと、「占いを探したい」「ゲームを探したい」というユーザーの意図に応えられない。タブがあることで目的指向のブラウジングが可能になる。

#### 選択肢C: ページネーション方式

- 全コンテンツをページ分割して表示
- **不採用理由**: ページ遷移が発生するとユーザー離脱が増える。SPA内のタブ切り替えの方がUXが良い。

#### 選択肢D: 現行構造を維持し、未掲載8件を「もっと診断してみよう」に追加

- 最も実装コストが低い方式
- **不採用理由**: 8件を追加するとセクションが非常に長くなり、ページの縦長問題が悪化する。メンテナンス性の問題も解決しない。

#### 選択肢E: fortuneをタブに含める5タブ構成

- 「すべて」「診断」「知識」「占い」「ゲーム」の5タブ
- **不採用理由**: fortuneカテゴリは現状1件（daily）のみで、タブとして独立させるには件数が少なすぎる。FortunePreviewセクションで日替わり結果のプレビューという専用UIが既にあり、タブ内のカード1枚よりもはるかにリッチな体験を提供できている。

## 補足事項

### 自動選出ロジックの将来的な拡張性

現在の自動選出ロジックは publishedAt（新しい順）を基準としているが、将来的には以下の拡張が考えられる。

- **GAデータ連携**: 人気度（PV数、完了率）に基づくスコアリングで、よりクリックされやすいコンテンツを優先表示
- **A/Bテスト**: ピックアップの組み合わせを変えてクリック率を比較する仕組み
- **季節・時間帯連動**: 時期に応じたコンテンツの優先度変更（例: 新年は占い系を優先）

ただし、これらは現時点では過剰設計となるため、まずは publishedAt ベースのシンプルなロジックで実装し、GAデータで効果測定した上で必要に応じて拡張する方針とする。

### fortune カテゴリの今後

fortune カテゴリが将来的に2件以上に増えた場合は、タブとして独立させることを再検討する。その判断基準は「タブ内に3件以上のコンテンツがあること」とする。
