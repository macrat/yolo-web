## 作業計画

### 目的

B-165 Phase 3-C の新規コンテンツとして、Q20「ユーモア辞書」を制作する。日常用語にユーモラスな再定義を与える辞書コンテンツで、以下の3つの役割を担う。

1. **SNSシェアの起点**: 1エントリが「一言ネタ」として完結し、X/LINEで共有されやすい形式を提供する
2. **サイトの世界観補強**: 「AIならではのズレた面白さ」というブランドトーンを辞書形式で凝縮して体現する
3. **回遊性の向上**: 関連語リンクによるエントリ間回遊と、診断・占いコンテンツへの導線を提供する

### 品質基準（数値目標）

#### エントリ数: 初回リリース30エントリ、最終目標50エントリ

- **30エントリ（初回リリース最低ライン）**: 五十音の主要行をカバーし、「辞書」としての体裁が成立する最小単位。関連語リンクによる回遊網も30エントリあれば意味のあるネットワークになる
- **50エントリ（最終目標）**: 少数精鋭で品質を維持しつつ、訪問者が「もう1つ読みたい」と思える十分な量。調査結果が示す50-100エントリの推奨範囲の下限に合わせる
- **100エントリ以上は目指さない**: 競合「笑える国語辞典」は3,000語超だが、量で勝負する戦略は取らない。各エントリの創作的価値を高く保つことを優先する
- 初回リリースは30エントリとし、品質を確認したうえで段階的に50エントリまで拡充する

#### 1エントリあたりの分量: 合計400-800文字（日本語）

以下の構成要素すべてを含み、合計400-800文字とする。

| 構成要素       | 文字数目安  | 説明                                                                                                             |
| -------------- | ----------- | ---------------------------------------------------------------------------------------------------------------- |
| 見出し語       | -           | 日常でよく使われる日本語の単語                                                                                   |
| よみがな       | -           | ひらがな表記                                                                                                     |
| ユーモア定義文 | 50-150文字  | その単語に対するユーモラスな再定義。エントリの核。SNSシェア時にそのまま使える長さ                                |
| 解説・背景     | 150-350文字 | 定義の笑いがなぜ面白いかを掘り下げる。共感の根拠、日常の観察、状況の描写を含む。これが「薄いコンテンツ」回避の要 |
| 用例           | 50-150文字  | ユーモア定義を使った会話例や状況描写。1-2文                                                                      |
| 関連語         | -           | 他エントリへのリンク2-3語。回遊性を確保                                                                          |

#### 品質チェック基準（エントリ公開の合否条件）

各エントリは以下の5条件すべてを満たさなければ公開しない。

1. **固有の観察がある**: そのエントリにしかない視点・切り口が存在する。テンプレートの穴埋めで生成できるような定義は不可
2. **笑いの根拠が明確**: なぜ面白いのかを解説で説明できる。「共感」「意外性」「論理のズレ」のいずれかの笑いのメカニズムが特定できる
3. **読後に何かが残る**: 読み終えた後に「確かにそうだ」「人に言いたい」と思える。単に変なことを言っているだけでは不可
4. **人・属性を笑っていない**: 笑いの対象は「状況」「概念」「行為」に限定。特定の人、性別、職業、年齢、国籍等の属性を笑いの対象にしない
5. **定義文が50-150文字に収まる**: SNSシェア時の可読性を確保するため、定義文単体で完結し、長すぎない

### 作業内容

#### タスク1: データ設計・コンテンツ制作

**目的**: 辞書データの型定義、見出し語の選定、30エントリ分のコンテンツ制作

**作業内容**:

ステップ1（データ構造設計、コンテンツ制作に先行して完了させる）:

- TypeScriptの型定義を作成（HumorDictionaryEntry インターフェース）
- 辞書データファイルの構造を設計し、空のデータファイルを作成する（TSファイル、ビルド時に静的生成）
- getAllEntries() / getEntryBySlug() などのアクセス関数を実装する

ステップ2（コンテンツ制作）:

- 見出し語30語を選定する。選定基準: (a) 日常的に使われる語、(b) ユーモラスな再定義が自然に成立する語、(c) 五十音の偏りがない語、(d) 他エントリと関連語リンクで結べる語
- 30エントリ分のコンテンツを制作する。1エントリずつ品質チェック基準に照らして検証する
- コンテンツ制作は品質を担保するため、10エントリずつ3バッチに分けて制作・レビューする

**builderへの指示**: `.claude/rules/coding-rules.md` を直接読むこと

#### タスク2: ページ実装（SSG、一覧・個別エントリ）

**目的**: 一覧ページと個別エントリページをSSGで実装する

- DefinedTerm structured data (JSON-LD) を各エントリに設置
- 定義文をファーストビューに配置（SEO・UX双方の要件）
- パンくずリスト: ホーム > 辞書 > ユーモア辞書 > [見出し語]
- レスポンシブ対応（モバイルファースト）

**builderへの指示**: `.claude/rules/coding-rules.md` を直接読むこと

### URL設計

**採用案: `/dictionary/humor/[slug]`**

- 一覧: `/dictionary/humor`
- 個別: `/dictionary/humor/[slug]`（slugは英語、例: `/dictionary/humor/monday`）

**判断理由**:

- サイトの既存構造は `/{大分類}/{コンテンツ}` であり、ユーモア辞書は「辞書」カテゴリに属するコンテンツとして `/dictionary/humor` に配置するのが自然
- 既存辞典（colors, kanji, yoji）は将来削除予定だが、`/dictionary/` カテゴリ自体はユーモア辞書のために存続する
- クローラにも人間にも意味が通る構造

### 検討した他の選択肢と判断理由

#### URL設計について

| 選択肢                          | 判断   | 理由                                                                                           |
| ------------------------------- | ------ | ---------------------------------------------------------------------------------------------- |
| `/quiz/humor-dictionary/[slug]` | 不採用 | 辞書コンテンツをクイズ配下に置くのは意味的に不自然。既存構造 `/{大分類}/{コンテンツ}` と不整合 |
| `/humor-dictionary/[slug]`      | 不採用 | 既存構造を崩してトップレベルに新カテゴリを作る特段の理由がない                                 |
| `/dictionary/humor/[slug]`      | 採用   | 既存構造に従い、辞書カテゴリの一コンテンツとして配置。最も自然な位置                           |

#### エントリ数について

| 選択肢                 | 判断   | 理由                                                                        |
| ---------------------- | ------ | --------------------------------------------------------------------------- |
| 10-20エントリ          | 不採用 | 辞書としての体裁が成立しない。関連語リンクのネットワークも薄くなる          |
| 30エントリ（初回）     | 採用   | 辞書としての最小成立単位。品質を確認してから拡充する段階的アプローチが可能  |
| 50エントリ（最終目標） | 採用   | 調査結果の推奨範囲内で品質を維持できる上限。少数精鋭の方針と整合            |
| 100エントリ以上        | 不採用 | 品質維持が困難。競合（笑える国語辞典: 3,000語超）と量で勝負する戦略は不適切 |

#### エントリの構成要素について

| 選択肢                                               | 判断   | 理由                                                                                                                    |
| ---------------------------------------------------- | ------ | ----------------------------------------------------------------------------------------------------------------------- |
| 定義文のみ（悪魔の辞典スタイル）                     | 不採用 | 1エントリの情報量が少なすぎ、Googleに「薄いコンテンツ」と判定されるリスクが高い。Scaled Content Abuseの要件も満たさない |
| 定義文 + 解説 + 用例 + 関連語（採用案）              | 採用   | 各エントリに固有の創作的価値を担保できる。400-800文字の分量で「薄いコンテンツ」を回避しつつ、読者に価値を提供する       |
| 定義文 + 解説 + 用例 + 関連語 + 語源 + 類語 + 反義語 | 不採用 | 情報過多でユーモアの切れ味が鈍る。辞書らしさは出るが、本コンテンツの核は「笑い」であって「辞書としての網羅性」ではない  |

#### 実績システムの扱いについて

| 選択肢                                    | 判断   | 理由                                                                                         |
| ----------------------------------------- | ------ | -------------------------------------------------------------------------------------------- |
| 実績システムに統合しない                  | 不採用 | 回遊促進の機会を逃す。他コンテンツとの横断的な体験が生まれない                               |
| 1エントリ閲覧ごとに記録                   | 不採用 | 閲覧だけでカウントが大量に蓄積し、実績のインフレを招く                                       |
| 辞書全体で1コンテンツとして記録（採用案） | 採用   | いずれかのエントリを閲覧したら「humor-dictionary」として1日1回記録。他コンテンツと同等の粒度 |
