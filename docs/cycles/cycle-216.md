#### 何のためにやるのか / どんな価値を提供するか

1. **新デザイン詳細ページ移行（来訪者価値）**: keigo-reference 詳細ページは現状 `(legacy)` 配下にあり（**実測値** = `src/app/(legacy)/tools/keigo-reference/` の存在確認 / `ls -d`）、新トークン体系・ダークモード・新フォントの恩恵が来訪者にまだ届いていない。`Component.module.css` の旧トークン残存 = **42 箇所 / 8 種**（**実測値** = `grep -oE "var\(--color-[a-z-]+" src/tools/keigo-reference/Component.module.css | sort | uniq -c` = bg 5 / bg-secondary 5 / border 9 / danger 1 / primary 4 / success 1 / text 9 / text-muted 8）+ hex 4 箇所（**実測値** = `grep -noE "#[0-9a-fA-F]{3,6}"` = `#fff` ×2（L35/L90）+ フォールバック形式 `var(--color-danger, #dc3545)`（L206）/ `var(--color-success, #28a745)`（L212））。本サイクルで新トークン体系へ移行する。
2. **タイル動線による最短 UX = ターゲット需要への直接応答**: 「敬語に迷った瞬間に検索 → 尊敬語・謙譲語を確認 → コピーして元の作業に戻る」という参照・検索型の最短往復を、yolos.net トップから 1 タップで起動できるタイルとして提供する。UX 競合調査（`docs/research/2026-05-30-keigo-reference-tile-ux-design-research.md` 付記）の結論 = 日本語敬語参照に特化した検索機能付き小型ウィジェットは国内にほぼ存在せず（静的コンテンツか AI 変換ツールの 2 択）、yolos.net のインクリメンタルサーチ + カテゴリフィルタ + 並列表示タイルはこのカテゴリでほぼ唯一のアプローチであり差別化優位を持てる。
3. **参照・検索型タイルの設計指針確立（B-314 への波及効果）**: keigo-reference は GA4 直近 30 日 PV が移行未完了 18 ツール中最高（22 PV / 2 位 traditional-color-palette 10 PV の倍超 / **実測値** = §補足事項 / 2026-05-30 取得）であり、移行順序原則「PV 高い順」に合致する。既存移行済 16 件はすべて変換・生成系（入力→出力）であり、keigo-reference は**参照・検索型**という新アーキタイプの初回。ここで設計指針と AP-P21 計測系統を確立できれば、残る辞典系ツール群（traditional-color-palette ほか）の移行にも効く。
4. **信頼性配慮（憲法 rule 2「人を悲しませない」/ rule 4「品質」）**: 敬語は誤った形を提示すると来訪者に恥をかかせるため、出典明示（文化庁「敬語の指針」2007）と謙譲語注記（自分側の行為に使う旨）を最小限添える。固定編集済みデータ（AI 生成ではない）である旨も差別化として検討する。あわせて meta.ts の「40件以上」というハードコード（**実装値** = `src/tools/keigo-reference/meta.ts:25,30`）を実データ実測値「60件」に訂正し、来訪者に正しい前提を示す（B-409 乖離）。

### 検討した他の選択肢と判断理由

#### 論点 1: 結果表示 = 並列表示（普通語+尊敬語+謙譲語）か、タブ切替で 1 形ずつか

（NIT 対応 = research §3.5 の「案 A/B」記号とラベルが衝突するため、本表は内容で参照する。）

| 案                                       | 来訪者価値                                                                   | 400px 収まり                                                          | 採否           |
| ---------------------------------------- | ---------------------------------------------------------------------------- | --------------------------------------------------------------------- | -------------- |
| **3 列並列表示（普通語+尊敬語+謙譲語）** | 「尊敬語と謙譲語どっちだっけ」という最大需要に直接応答。1 画面で比較できる。 | 400px 幅で 3 列が窮屈になるリスク（各列 80〜100px / research §3.5）。 | **第一案**     |
| 2 列 + 種別タブ切替                      | 各セルの可読性は上がる。                                                     | 収まる。                                                              | フォールバック |
| 4 列テーブル横スクロール                 | 詳細ページ移植が最低コスト。                                                 | 横スクロールはタッチで煩雑。                                          | 不採用         |

**判断**: UX 競合調査（`docs/research/2026-05-30-keigo-reference-tile-ux-design-research.md` §3.5 / 同調査内では「2 列+タブ切替」を案 A・「カード形式」を案 B・「4 列横スクロール」を案 C と呼んでいる点に注意 = 本表の記号とは独立）は「2 列 + タブ切替」を推奨しているが、これは**タイルの中心需要を「1 種別だけ知りたい」と想定した場合の推奨**である。本サイクルでは PM 判断として中心需要を「**尊敬語と謙譲語のどちらを使うべきか比較したい（どっちだっけ問題）**」と置く。タブで 1 形ずつ切替にすると比較ができず価値が落ちる（AP-P07 = UI 選択を「実装パターンの参照」で決めない / 来訪者の認知モデルで決める）。よって**3 列並列表示を第一案**とし、400px 幅で読みにくい場合に限り「2 列 + 種別トグル」へフォールバックする。フォールバック判定は T-1 プロトタイプ + T-3 実機計測で行い、計画段階では確定しない（AP-P20 = 過度に具体的な計画にしない）。

#### 論点 2: コピー対象の粒度

行全体一括コピーか、各敬語形の個別コピーか。UX 調査付記は「利用者テストなしには確定できないため、まず個別ボタンを提供し GA イベントで計測」を推奨（ただしこれは尊敬語/謙譲語/丁寧語を全て出す単独ページ前提の文言）。本サイクルは**各敬語形のワンタップ個別コピーを第一案**とする（コピー対象の曖昧さを排除 / ターゲット likes「コピペで結果を受け取ってすぐ戻れる」直撃）。**コピー対象はタイルに表示している形に限る = 尊敬語 / 謙譲語（必要なら普通語）。丁寧語はタイルに表示しないため対象外とし、詳細ページのみでコピー可能とする**（機能リスト = 普通語 + 尊敬語 + 謙譲語の 3 列並列表示と整合 / タイルに出さない形をコピーできる齟齬を排除）。具体的なボタン配置・文言は実装段階に委ねる。

#### 論点 3: recommendedSize

cols=3 rows=3（400×400px）を第一推奨とする。結果リストのスクロール表示に縦余白が必要なため（UX 調査 §3.1 = 160px 可用高で 40px/行なら 4 行）。サイズ系数値は先行 SSoT が薄いため**推定値**として hedge し、実機での収まりを T-3/T-4 で検証する。

#### 論点 4: AP-P21 系統分類（最重要 / cycle-210 T-4 同型事故 + cycle-215 「空証文」事故の二重予防）

参照・検索型は変換・生成系とは別系統である。変換・生成系の SSoT テンプレートを無批判に流用すると、AP-P21 適用範囲のミスマッチが T-4 で初めて発覚する（cycle-210 T-4 R1〜R3 同型事故）。本計画では系統分類・基準値の適用可否を**計画段階で精査済み**とし、以下 2 軸で分離した。

**軸 1: 変換・生成系 vs 参照・検索型**

- (1) 引用適用する SSoT: cycle-210 (i) 40px 下限（`docs/cycles/cycle-210.md:292`）/ cycle-211 (x) コピーボタン文言変化 AP-P21 適用外（`docs/cycles/cycle-211.md:586`）。
- (2) 適用対象外の SSoT: 複合入力型 (v) ±15% / 単一 textarea (v) ±10% / B-452 / B-456 / localStorage 前回値保持（B-433）。
- (3) 新規確立する SSoT（参照・検索型 N=1）: 結果行 40px 下限 (A) / 空状態高さの通常状態との乖離 (B) / 操作側 40px 下限 (C)。

**軸 2: 同軸 vs 同軸ではない（flex 固定起因ゼロ変化率 / CRIT 対応）**

cycle-214/215 は「操作側 flexShrink:0 / 膨張側 flex:1 + overflowY:auto」の flex 固定設計のため操作側 (v) 変化率が 0.00% になり、「同軸ではない参考値」として真の N に積めず、cycle-215 完了時に B-452 が「真の N=1 のまま確定」となった（`docs/cycles/cycle-215.md:388,390` / `docs/backlog.md:95`）。本サイクルの参照・検索型は**完全に同型の flex 固定設計**であり、操作側 (v) 変化率は構造上ほぼ確実に 0.00% になる = この量を真の N に積もうとすれば B-452 と同じ「達成不能な空証文」になる。よって本サイクルは「(v) 変化率の N」を追わず、(v) を参照・検索型では **N/A** と判定し、代わりに (A)(B)(C) を SSoT として確立する（詳細は §作業内容 T-4）。

採否は T-1 で AP-WF14 の機能カテゴリ並べ読み表により独立確認する。

#### 論点 5: 注意点として計画に織り込んだ事項

- 敬語の誤提示は来訪者に恥をかかせる = 憲法 rule 2「人を悲しませない」に直結。出典明示・謙譲語注記を最小限添える。固定編集済みデータ（AI 生成ではない）である旨も差別化として検討する。
- 数値 literal は全て AP-P16 4 分類ラベル付き。サイズ系は先行 SSoT が薄いため「推定値」hedge で記し、実 SSoT は T-4 実機計測で確定する。
- 表記統一: 敬語フィールド名は実装値 `kenjogo`（**実装値** = `src/tools/keigo-reference/logic.ts:14`）に統一する（「kenjougo」表記は使わない）。

#### 論点 6: 文化庁 5 分類 →3 分類簡略表示の脚注の要否

文化庁「敬語の指針」（2007）は 5 分類（尊敬語 / 謙譲語 I / 謙譲語 II（丁重語）/ 丁寧語 / 美化語）だが、現データは謙譲語 I/II を区別せず単一 `kenjogo` として扱う（**実装値** = `logic.ts:14`）。research §5.2 は「UI には 3 分類を表示し脚注に『5 分類を簡略表示』と明示」を推奨。**判断**: タイルは面積が限られ過剰注意は dislikes「冒頭に長い解説」を招く。出典フッタ 1 行に「（文化庁指針の 5 分類を 3 分類に簡略）」を含めるか、それとも詳細ページ側にのみ簡略の旨を置きタイルでは触れないか、を T-3 で実機の面積を見て判断する（計画段階では確定しない / どちらでも信頼性は担保される）。

#### 論点 7: localStorage 前回値保持（B-433）の系統適用可否

変換・生成系で検討される前回入力値の localStorage 復元（B-433 / `docs/backlog.md:83`）は、参照・検索型には**適用しない方針**とする。参照・検索型は来訪のたびに調べたい語が変わるため、前回の検索語 / カテゴリを復元するとむしろ新規検索の妨げになる公算が高い（ターゲット likes は「すぐ使い始められる」であり「前回値復元」ではない）。AP-P21 の系統分離と同粒度で localStorage 系統の判断も明示する（MAJOR 対応）。最終確定ではなく方針であり、T-3/T-4 実機確認で覆る余地は残す。
