# サイクル-255

このサイクルでは **B-525「来訪者価値の継続的計測基盤の設計と実装」**を行う。サイトに変更を加えたとき、その変更が画面の向こうの来訪者にとって本当に良かったのか——直帰が減ったのか、滞在が深まったのか、回遊が増えたのか——を**反復可能に測れる**基盤を作る。

## なぜやるか（来訪者の所在と Owner の指摘）

cycle-254 完了後、Owner から「**新旧デザインの優劣を計測していない**」「**計測機能を持たないままサイト運営を続けることになる**」という指摘があった。前 PM（=本プロジェクトの PM）が当初「工数」を理由に事後コホート比較1回への縮小提案をしたが、それは AP-P28（規模を理由に来訪者価値の劣る選択肢を選ぶ）と同型構造だと自覚し、計測基盤の整備を P1 で起票した（コミット bc54825a）。

これは来訪者の所在に直結する。憲法の Goal は「来訪者価値の最大化によるページビュー向上」。だが今のサイト運営は、B-522 のデザイン移行を cycle-225 以降何サイクルも重ねながら、その変更が来訪者の体験を改善したのかを一度も測っていない。「離脱率30%」を「10人中3人がここで手を止めた」と読み替えてその先の人を見に行く——その入口の数字を、そもそも継続的に取り出せる形にしていない。指標が無ければ、すべての「これは来訪者に良いはず」という判断は検証不能な断言のままになる。これを直すのが本サイクル。

## 作業計画

### 目的

サイト変更の前後で来訪者価値指標を**反復可能・継続的**に測れる基盤を最小実装で確立し、今後のあらゆる「来訪者に良いはず」という判断を検証可能にする。具体的には (1) リリース起点で before/after を正確に分割できる識別子、(2) 来訪者価値指標の定義 SSoT と反復可能な BigQuery クエリ、(3) 計測を毎サイクル運用に接続するプロセス、の3点を揃える。

### 作業内容

**フェーズ1：足場固め（現状の事実確認）**

`docs/` の指標・GA4 設定・BigQuery 在庫を、記憶や推測でなく実測で確定する。MCP（GA4・Playwright）を使うサブエージェントは foreground で実行する（CLAUDE.md 規則）。

- 現行 GA4 イベント（`src/lib/analytics.ts`：level_start/end・search・share・content_rating・toolbox 系・tile_first_interaction）と、GA4 自動収集（page_view・session_start・user_engagement・engagement_time_msec）が BigQuery でどう取れるかを棚卸し。
- 主要サーフェスの実トラフィック量を確定（検出力評価の入力）。

**フェーズ2：設計判断（A/B vs 事後コホート比較）**

両案を**メリット根拠**で評価する。判断の芯として現時点で見えている論点（設計サブエージェントが実データで検証・反証する前提）：

- **問いの構造**: Owner の問い「新デザインは旧より良かったか」は、旧デザインが (legacy) 撤去で消滅進行中のため、今から A/B 振り分けできない。過去の全面移行の評価は **before/after 遡及比較でしか答えられない**。これは工数でなく問いの構造から来る制約。
- **統計的検出力**: /play 主要サーフェスは28日152PV 規模（cycle-254 実測）。A/B で2分割すると各アーム ~76 で、直帰率の現実的な差（数 %pt）を検出する検出力が出ない可能性が高い。設計サブエージェントが最小検出可能効果（MDE）と必要サンプルの概算で定量評価する。
- **静的/サーバーレス制約と憲法ルール2/3**: A/B 振り分けは middleware でのバケット割当＋バリアント出し分けを要し、静的配信・「外部API/DB/認証を実装しない」原則と緊張する。事後コホート比較はクライアント計測＋BigQuery で完結し制約と整合。
- ただし設計サブエージェントには「結論を先に決めず実データで両案を評価せよ」と指示し、高トラフィックサーフェスで A/B が成立するなら正直にそう書かせる。**A/B 却下が成立する場合、その理由は必ず統計的検出力・問いの構造であって工数ではないことを明記**（AP-P28 回避の核）。

**フェーズ3：最小実装**

- **リリース識別子**: `scripts/generate-release-info.ts`（仮）で git short SHA ＋ビルド日付を生成し `src/lib/generated/release-info.ts`（仮）へ書き出し、prebuild/predev/pretest に組み込む。`GoogleAnalytics.tsx` の `gtag('config', ...)` か全イベント共通パラメータとして `release` を付与。git 不在時フォールバック（日付のみ等）。
- **反復可能クエリ**: 指標定義に基づく BigQuery SQL をパラメータ化（比較期間 or release）してリポジトリ保存。

**フェーズ4：プロセス接続と検証**

- 運用手順の文書化、Playwright 実機での `release` 付与確認、テスト追加、レビュー。

各実装タスクは1サブエージェント1タスクで委譲し、作業ごとに reviewer のレビューを受ける（CLAUDE.md 規則）。

### 検討した他の選択肢と判断理由

1. **B-523/B-524（静的結果ページ枠の新化＋ルート移動）を先にやる** — cycle-254 が「次サイクル必須」と記したが、それは Owner の計測指摘より**前**の判断。Owner はその後 B-525 を P1 で Queued 先頭に置き、「計測機能を持たないまま運営を続けている」と指摘した。デザイン移行の「第三弾」を、第一弾・第二弾が来訪者の役に立ったか測れないまま進めるのは、Owner が指差したまさにその構造の再生産。計測基盤を先に作れば B-523 を含む今後の移行を実際に評価できる。順序として計測が先。B-523/B-524 は密接な別ユニットとして次サイクルへ（Queued 据え置き）。
2. **A/B 振り分け基盤をフル実装する** — 一見「計測機能」として強力だが、(a) 旧デザインが消滅進行中で過去移行を A/B できない、(b) 実トラフィック規模で検出力が出ない懸念、(c) 静的/サーバーレス・憲法ルール2と緊張、の3点で現実的でない。本サイクルでは設計判断でメリット評価し、成立しないなら却下理由を定量で明記。将来高トラフィック化や個別機能の小さな出し分けで A/B が有効になる条件はキャリーオーバーに残す。
3. **「事後コホート比較を1回だけ実施」に縮小** — まさに Owner が AP-P28 と指摘した縮小案。1回限りでは「継続的計測基盤」にならず、次の変更でまた測れない。**反復可能なクエリ＋リリース識別子＋運用接続**まで作って初めて「基盤」。却下。
4. **計測は GA4 UI 探索で十分とする** — GA4 UI のイベントレベル保持はデフォルト2ヶ月・最大14ヶ月（一次資料確認済）。継続的な歴史比較の土台にならない。BigQuery エクスポート（無期限・全 event_params 取得）を基盤に据える。

#### 外部仕様への依存（一次資料確認済）

本サイクルの設計は外部仕様（GA4 のイベントパラメータ取り込み・データ保持仕様）に依存するため、計画冒頭で一次資料を確認した。

- **GA4 BigQuery エクスポートは未登録のイベントパラメータも `event_params` RECORD に全取得する**（カスタムディメンション登録は GA4 UI 探索で参照する場合にのみ必要）。確認: https://support.google.com/analytics/answer/7029846 （2026-06-20 WebFetch）。→ リリース識別子 `release` を GA4 イベントに付けるだけで、GA4 UI 登録なしに BigQuery でコホート分割できる根拠。
- **GA4 のイベントレベルデータ保持はデフォルト2ヶ月・最大14ヶ月だが、これは GA4 UI 探索の制限であり BigQuery エクスポートは対象外（無期限保持）**。確認: WebSearch「GA4 data retention default 2 months 14 months BigQuery export」（2026-06-20。複数の二次情報が一致。一次原則は support.google.com の data-retention ヘルプ）。→ 継続的・歴史的な before/after 比較の土台を GA4 UI でなく BigQuery に置く根拠。
