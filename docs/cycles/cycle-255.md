# サイクル-255

このサイクルでは **B-525「来訪者価値の継続的計測基盤」を A/B テスト基盤として**設計・最小実装する。サイトのデザインを変えるとき、それが画面の向こうの来訪者にとって本当に良いのかを、**変更を旧/新で振り分けて実際に比べながら**測れる土台を作る。前方（これから変える分）だけでなく、**既にブラインドで移行してしまった高リスク面**も、旧デザインを復活させて測れるようにする。

## なぜやるか（憲法に立脚した判断）

憲法の Goal は「来訪者価値の最大化によるページビュー向上」。CLAUDE.md の意思決定原則は「複数の手段があるとき、たとえ実装コストが著しく大きくても来訪者価値を最大化する手段を選ぶ／実装コストを劣る UX・手段を選ぶ理由にしてはならない」と定める。

この原則に照らすと、デザイン変更の評価手段として **A/B テストがブラインド全面移行より優れる**ことは明白だ。A/B は変更が来訪者の役に立ったかを実地で学べる。ブラインド移行は「良いはず」という検証不能な断言のまま切り替える。憲法 Goal が「来訪者価値の最大化」を命じる以上、測れる手段があるのに測らずに切り替えるのは原則違反である。そして A/B に伴うコスト（振り分け機構の実装・両バリアントの維持・旧デザインの復活）は、上記原則により A/B を避ける正当な理由にならない。これがわたしの判断であり、本サイクルの芯である。

### 既にブラインド移行した分を測るか——わたしの判断

このサイトは cycle-225 以降、カラフルで絵文字を中心に据えた旧デザインから、簡素に削ぎ落とした新デザインへ、何サイクルもかけて移行してきた。とりわけ cycle-253/254 では、**最も見られている旅程の終点**（character-personality 62PV/28日、/play 最大流入面）のインラインクイズ結果から、絵文字（🎭✨😅💡）・type-color 装飾・中央寄せ・太字を剥ぎ落とした。これらはすべて**測らずに**切り替えた。

憲法ルール2は「helpful **or enjoyable**」を求める。クイズ・診断は息抜き＝**enjoyable の柱**である。カラフルで遊び心のある提示は、息抜きコンテンツの「楽しさ」を実際に担っていた可能性がある。それを剥ぎ落とす変更は、enjoyable の柱に対する**現実的なリスク**を伴う。もし新デザインが楽しさを損なっていたなら、ライブで毎日、来訪者は劣る体験を受け取り続けている——「もう出した」ことはリスクを免除せず、未検出の回帰が人を損ね続けるという意味でむしろ切実だ。

したがって、**移行のうちリスクが集中する高流入・高リスク面（絵文字→シンプルに変えた息抜きコンテンツ、特に最大流入のインラインクイズ結果）は retro A/B の対象とする**、というのがわたしの判断である。剥ぎ落とし前の `*Content`（.tsx/.css）は git 履歴（cycle-253/254 の diff）から再構成でき、旧バリアントとして復活させられる。道具系ツール（電卓・変換など）の落ち着いた簡素化は「日常の傍にある道具」という方向と整合しリスクが低いため、retro A/B は来訪者価値レバレッジで個別に劣後判断する。この切り分けは工数ではなく、enjoyable の柱に対する未検証リスクの大きさに基づく。

（記録として：本サイクルの計画初稿で、わたしは「既移行分は旧デザインが消えたから測れない」「次から A/B すればよい」と消極的に流していた。それは旧デザイン復活の工数回避を構造的制約に見せかける誤りで、上記の憲法判断を自分で行えば成り立たない。AP-P28・AP-WF24 として完了処理で記録する。）

## 作業計画

### 目的

デザイン変更を**旧/新で振り分けて来訪者価値を実際に比べながら**進められる A/B テスト基盤を最小実装で確立する。前方 A/B（これからの変更）と retro A/B（既にブラインド移行した高リスク面の旧デザイン復活）の両方を支え、残る B-522 移行を「変えながら比べて調整する」形にし、既移行で生じたかもしれない enjoyable の柱の回帰を検出可能にする。before/after コホート比較とリリース識別子は、A/B が原理的に組めない全体変更のための補完として併設する（A/B の代替として持ち出さない）。

### 作業内容

**フェーズ1：足場固め（実測）** — MCP（GA4・BigQuery 経由確認・Playwright）を使うサブエージェントは foreground。現行イベント体系と算出可能指標、主要サーフェスのセッション規模を実データで確定（A/B 検出力設計と最初の実 A/B 選定の入力）。

**フェーズ2：設計（A/B を本線として誠実に・retro A/B を含む）** — 設計サブエージェントへ、**結論を A/B 採用に置いたうえで「どう成立させるか」を設計せよ**と指示する（初稿のように A/B 却下へ倒さない）。論点：

- **バリアント割当機構**: 静的配信を壊さず旧/新を出し分ける方式の比較・選定（クライアント乱択＋切替／middleware 乱択 cookie＋rewrite／折衷）。憲法ルール2（外部 DB/認証なし＝端末内乱択で充足）と整合。
- **retro A/B の構成**: cycle-253/254 で剥ぎ落とした旧 `*Content`（.tsx/.css）を git 履歴から復活させ、旧バリアントとして現行と並走させる構成。撤去予定コードの一時復活が移行計画（design-migration-plan）と矛盾しない置き方。
- **記録**: 割当 variant を GA4 共通パラメータ（`experiment_id`/`variant`）として送り、BigQuery でアーム別集計（任意パラメータが `event_params` に全取得されることは一次資料確認済）。
- **検出力**: 実トラフィックから MDE と必要期間を概算。低トラフィックは却下理由でなく期間/対象選定の入力。
- **最初の実 A/B 選定**: 来訪者価値リスク×トラフィック。第一候補＝最大流入の既移行インラインクイズ結果（絵文字/カラフル vs シンプル）。

**フェーズ3：最小実装** — A/B 割当＋GA4 記録の再利用可能機構、リリース識別子（補完）、variant/release 比較クエリ。各タスク1サブエージェント、作業ごとに reviewer。

**フェーズ4：運用接続と検証** — B-522 移行と既移行面への A/B 適用手順、最初の実 A/B の段取り、Playwright 実機での記録確認、テスト、レビュー。

### 検討した他の選択肢と判断理由

1. **before/after コホート比較を本線にする（計画初稿・却下）** — 「旧デザインは撤去済みだから A/B できない」を前提にしていたが、これは「旧デザイン復活は工数がかかるからやらない」を構造的制約に見せかけたもの。憲法の意思決定原則（コストは劣る手段の理由にならない）に反する。A/B を本線に立て直し、before/after は A/B が原理的に組めない全体変更の補完に格下げ。
2. **既移行分は A/B 対象外とし前方のみ A/B する（却下）** — enjoyable の柱（クイズ）で最大流入面の絵文字→シンプル化という現実的リスクを未検証のまま放置する。憲法ルール2・Goal に照らし、高流入・高リスクの既移行面は retro A/B の対象とする。旧バリアントは git 履歴から復活可能で、工数は却下理由にならない。
3. **最初の実 A/B を B-523（静的結果ページ枠）にする（却下）** — B-523 は実測7PV/28日と低流入・低リスク。最初に測るべきは来訪者価値リスク×トラフィックが最大の面＝既移行インラインクイズ結果。B-523 は基盤の後続適用先。
4. **「事後コホート比較を1回だけ」へ縮小（B-525 起票時の案・却下）** — 1回限りでは継続基盤にならず A/B でもない。AP-P28。却下。
5. **計測は GA4 UI 探索で足りるとする（却下）** — GA4 UI のイベントレベル保持はデフォルト2ヶ月・最大14ヶ月（一次資料確認済）。継続比較の土台にならない。BigQuery エクスポート（無期限・全 event_params 取得）を土台に据える。
6. **A/B フルフレームワークを一気に作り実テストを有意水準まで回す（部分採用）** — テストは出してから N 日待って読む性質で1サイクルに収まらない。本サイクルは基盤の最小実装＋最初の実 A/B の段取りまで。結果読み取りはキャリーオーバー。これは工数回避でなくテストの時間的性質に基づく分割。

### 横断的パターン

本サイクルで違反が連鎖した主な構造は以下の 4 つ:

1. **AP-WF09 ＋ AP-WF24 の連環**: Owner が AP-WF24 を指摘 → PM が表面の単語を入れ替えて済ます（AP-WF09）→ 判断構造は変わらず別の場所で AP-WF24 を再発 → の循環が介入 2 → 介入 7 で実際に発生。介入 5 の「言い回しを書き直した」をブログで気づきとして書いたのも、この連環の延長線上。
2. **AP-P28 の構造同一・適用領域違いの再発**: 介入 1 で「事後コホート vs A/B」の局面、介入 2 で「既移行分の対象範囲」の局面、それぞれ同じ「工数回避を構造的制約に見せかける」構造で発生。同じ AP の異なる発生形を、PM がメタ抽象化できていなかった。
3. **Goal を起点にしない判断（AP-P25）が AP-P09 / AP-P28 / AP-W08 の上流原因として機能**: 介入 1（技術論で A/B vs cohort 比較）、介入 5（測れる仕組みを目的化）、介入 6（記事の継続 / 削除を Goal 起点で判断しない）が、いずれも「Goal から逆算する」ステップを飛ばしたことで発生した。
4. **チェック範囲の恣意的限定（AP-WF09 の核心形）**: 介入 8 → 介入 9 の流れは、reviewer 通過 / recall 結果の「成果物の範囲」を PM が自分で広げず、形式的合意で済まそうとしたことで発生。介入 4 への報告で「3 回のレビューを通したから OK」と暗黙に依存したのも同型。

- MCP ツール（GA4・Playwright・BigQuery 経由データ確認）を使うサブエージェントは foreground で実行する（CLAUDE.md 規則）。
