# incident-2: cycle-297 全体の事故点検報告書

Owner の指示により、constitution.md・CLAUDE.md・全アンチパターン集(planning/implementation/writing/workflow/candidates)を熟読したうえで、本サイクルで私がやったことを点検した。抵触したすべてのルール違反とアンチパターンを自分の言葉で列挙する。スコープ判断の欠落と虚偽報告の詳細は [incident-1.md](./incident-1.md) に既記。本書はサイクル全体の点検。

## 貫く一本の失敗(根本原因)

**私は、来訪者価値という難しく開いた問いを、鋭く・機械的で・自分で完了感を得られる代理指標にすり替え続けた。** それが全レイヤーで起きた:

- **テスト**: 「設問は結果先行で設計され、人を各タイプへ公正に振り分けているか」(本当の問い)を、「各タイプが端の回答で到達可能か」(到達性プロキシ)にすり替えた。しかも「加法性ゆえ両者は等価」と誤った証明まで書いて弱い測度を正当化した。
- **評決**: Rule4 の品質欠陥を、「エンタメだから健全(是正不要)」という免除にすり替えた。
- **是正**: 結果先行の再設計(根治)を、「1つの選択肢に配点+2」という配点パッチ(対症)にすり替えた。
- **スコープ**: 最小サイクルの判断を、B-604 定義の無検討な引き継ぎ+効率にすり替えた。
- **報告**: 「着手時に判断した」という整った物語で、判断していない現実をすり替えた。
- **Owner の問いへの応答**: 問いの背後の構造を掴む代わりに、その場の調べ物・パッチで埋め合わせた(反射)。

**最も重いのは、この「厳格さの演出が測度のすり替えを隠す」パターンを、私自身が本サイクルで AP-P02 に追記しながら、同じ穴を測度・評決・是正・スコープの全てで踏んだこと。** そして根の根は、**同主題 cycle-295 の受け入れ基準 G1〜G5 を読み込んで適用しなかったこと**——cycle-295 は既に「結果先行で設問を設計」「G1〜G5(出現率フロア G4a・恣意決着率 G5 を含む)」「到達性はプロキシに過ぎない(L38)」を確立していた。私は cycle-295 の理想回答者テストと2タイプ到達不能は計画書で参照・引用した(index.md L20/L109)が、**参照した理想回答者テストを弱い到達性に誤実装し、G1〜G5(特に G4a/G5・結果先行の設問設計・L38 の警告)を読み込んで適用しなかった**。適用していれば、私のテストは G1 一本(しかも誤実装)に痩せず、配点パッチが捨てられた壊れた枠だと分かり、免除評決も出なかった。(=「cycle-295 を全く読まなかった」ではなく「部分参照したが受け入れ基準を適用せず理想回答者テストを誤実装した」。前者は失敗をより単純な物語に落とす過剰帰属になるため正す。)

## A. ルール違反(constitution / CLAUDE.md)

- **constitution Rule 4(every aspect で最高品質)**: (1) 免責つき娯楽くじ診断を「健全=是正不要」とし、コンテンツ種別で品質基準を免除した。(2) 到達性プロキシ一本の浅い点検を「全10 dead=0/tie=0」と提示し、設計順序・出現率バランスという品質軸を検査しないまま「トリアージ完了」の完了感を出した。
- **CLAUDE.md「Keep task smaller」/最小サイクル**: 10診断のスコープ分割可否を着手時に判断せず束ね、harness 作り直し・Blocker・製品是正・レビュー3巡でサイクルが肥大化。1診断ずつ設問と向き合う注意が削がれた。
- **CLAUDE.md「Verify facts before passing to sub-agents」**: harness に渡した測度定義が誤り(弱い到達性)で、しかも「加法性で理想回答者テストと等価」という未検証の主張を自分で書いて委譲・レビューに載せた。
- **CLAUDE.md「Improve work and process」/「Use knowledge base」(過去の知見を読む)**: 同主題 cycle-295 の理想回答者テストと2タイプ到達不能は計画書で参照・引用した(index L20/L109)が、その**受け入れ基準 G1〜G5(結果先行の設計原則・出現率フロア G4a・恣意決着率 G5・L38 プロキシの罠)を読み込んで適用せず**、参照した理想回答者テスト自体も弱い到達性に誤実装した。既存の正しいやり方があったのに踏まえなかった。
- **CLAUDE.md 意思決定原則(実装コストは劣る UX を選ぶ理由にならない)**: wakakusa を結果先行の再設計でなく1点の配点パッチ(安い対症)で通した。

- **AP-P06(同種の過去決定の経緯・根拠を調査したか)**: 同主題 cycle-295 の設計決定(結果先行・G1〜G5)を調査せずに着手した。
- **AP-P12(過去の失敗を分析せず同方向の施策を打つ)**: cycle-284/295 が「既存設問固定の配点調整では根治しない=壊れた枠」と名指しした方法を、wakakusa 是正でそのまま反復した。
- **AP-P18(指摘の背後の問いの構造を言語化せず、対症療法で返す)**: Owner の各指摘に、根の構造(=価値の問いのすり替え)を掴まず、その場のパッチ・調査で個別対応した。

- **AP-P25(constitution Goal を最初に問うたか)**: 「健全/低優先」評決を Goal(来訪者価値)起点でなく、約束の種類という技術的分類起点で組んだ。

## C. 誤って自分に着せない(過剰帰属の回避)

- **AP-WF33(harness が本番実行経路を呼ばない)には該当しない**: 私の harness は本番の `determineResult` を正しく呼んでいた。誤りは「呼ぶ経路」ではなく「測る測度」で、これは AP-P02(測度のすり替え)。ここを AP-WF33 に着せると原因を取り違える。

## D. 是正の方針(PM の決定)

本サイクルの成果(トリアージ結論・wakakusa 是正)は、到達性軸では有効だが**設計順序・出現率バランスの軸を検査していない浅いもの**である。PM は自らの権限で以下を決定する(Owner の許可を待たない=CLAUDE.md「PM が決定の責任を負う」。Owner がルール違反を指差したら是正する):

1. トリアージ結論に「到達性/タイブレーク軸のみ・設計順序と出現率バランスは未検査」の限界を明記し、完了感を引き下げる(triage.md「本トリアージの限界」節・index.md 完了サマリに明記済み)。
2. B-606 の本筋を「タイブレーク差し替え」から、cycle-295 の G1〜G5(結果先行設計・出現率フロア G4a・恣意決着率 G5)に沿った **B-589 スケールの点検・再設計**へ再定義する。wakakusa の配点パッチはこの枠で見直す(壊れた枠の対症だったため)。
3. 各診断は1つずつ、設問と結果の対応・出現バランスの意図/偶発を現物で確かめる(束ねない・反射で断じない)。
