## 作業計画

### 目的

ターゲットユーザー（手軽で面白い占い・診断を楽しみたい人）が「遊ぶ」という1つの入口から全インタラクティブコンテンツにアクセスできるようにするため、/play統合の詳細な実施計画書を作成する。

cycle-99の調査で/play統合が推奨案として確定したが、URL移行に加えて型統合・レジストリ統合・SEO関数統合・OGP設計刷新・ディレクトリ再編が必要であり、想定より遥かに大掛かりな作業と判明した。計画を先に固めることで、実装サイクルでの手戻りや品質低下を防ぐ。

### 作業内容

1. **play-migration-plan.md の草案レビューと修正**: 既に作成済みの草案に対してレビューエージェントによる批判的レビュー（既存コードに囚われず最適解を追求する観点）を実施し、指摘事項を修正する。全体アーキテクチャ設計（PlayContentMeta統合型、レジストリ、パス生成、SEO関数、OGP 2層設計）、サイクル分割計画（ゲーム移行→クイズ移行→Fortune移行の3段階）、各サイクルの作業内容と受け入れ基準チェックリスト、移行対象ファイルの完全な一覧、リダイレクトマップ、各サイクル終了時のサイト状態定義の全項目について、漏れや矛盾がないことを確認する。
2. **backlog.md の更新**: B-205（フェーズ1: ゲーム移行）、B-206（フェーズ2: クイズ・診断・Fortune移行）を追加し、B-202〜B-204の依存先をB-206に変更する。
3. **site-value-improvement-plan.md のフェーズ4セクション更新**: /play統合の計画策定完了と、今後の実装スケジュールを反映する。

### 検討した他の選択肢と判断理由

- **選択肢1: 計画と実装を1サイクルで実施する** → 不採用。型統合・レジストリ統合・SEO関数統合・OGP設計刷新・ディレクトリ再編に加えてゲーム4種 + クイズ・診断14種 + 日替わり占い1種 = 計19種のコンテンツのURL移行・内部リンク更新・リダイレクト設定が必要であり、1サイクルに収めるとスコープが大きすぎてリスクが高い。
- **選択肢2: 計画を省略して直接実装に入る** → 不採用。影響範囲がフィーチャーモジュール・ルーティング・SEO・OGP・ナビゲーション・ブログ記事内リンクと多岐にわたり、計画なしでは移行漏れや中間状態の不整合が発生する可能性が高い。

### 計画にあたって参考にした情報

- **cycle-99の調査結果**: 選択肢A（/play統合）が推奨案として確定した経緯と根拠。実データに基づくSEOリスク再評価により、分類境界問題の完全解消とサイトコンセプトとの整合が決定的な優位点となった。
- **既存コードベースの構造調査**: quiz/games/fortune の依存関係分析、移行対象ファイルの洗い出し結果。QuizMetaとGameMetaの構造差異（フラットSEO vs seoサブオブジェクト）の把握。
- **Ownerからのフィードバック**: OGP 2層設計（デフォルト層 + カスタマイズ層）、twitter-image.tsx廃止方針、QuizMeta命名問題（占い系コンテンツのメタタイトルに「クイズ」が付く問題）、デザイン調査の重要性。
- **play-migration-plan.md の既存草案**: cycle-99の調査過程で作成された草案をベースに、レビューと修正を行う。
