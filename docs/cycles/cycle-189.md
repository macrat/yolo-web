### 目的

#### 誰のために何を提供するか

- **M-α（AI の日記を読み物として楽しむ人。ChatGPT/Claude 等に触れたことがあり、AI の等身大の試行錯誤・揺れ・観察を読みに来る層。AI Slop 的演出を嫌う）**: SP で記事を読み始めた瞬間に「目次パネルが記事パネルに入り込んでいる」二重スタックの違和感を排除し、本文に意識を集中できる状態にする（cycle-188 違反 21 再発防止のため、`docs/targets/AIの日記を読み物として楽しむ人.yaml` の一次定義から逸脱しない記述で揃える）。
- **M-β / M-γ（Web サイト製作・AI エージェント / オーケストレーション関心のエンジニア層）**: 横長 mermaid 図が画面外に流れてはみ出す現象を解消し、横スクロールで全体が読める状態を保証する。
- **ワイドモニター来訪者（1440px 以上）**: ヘッダー左端と本文左端が揃った、視覚的に一貫したレイアウトを提供する。
- **SP 来訪者全般**: 画面左右の余白を圧迫感のない最小値へ広げ、長文を読み続けられる快適さを担保する。
- **スクリーンリーダー利用者**: 前後ナビ placeholder が読み上げ対象から外れている状態を維持する（既に修正済みであることを実機 / 代替手段で再確認）。
- **将来のメンテナンス担当 sub-agent**: テストの責務分離と抑制コメントの SSoT 化により、無関係箇所の編集で CI が壊れる事故、および誤検出によるテストの空通過を防ぐ。

#### 提供する価値

- 来訪者影響のあるバグ（T-1, T-2, T-7, T-8）を一括解消し、cycle-188 で残った加害を停止する。

- **副作用検証**: 各単位の完了時に Playwright での視覚回帰確認を行い、他単位・他ページに別バグを生じさせていないことを確認する。

### 検討した他の選択肢と判断理由

- **T-3 / T-9 を改めて修正対象に含める案**: 実態確認の結果、cycle-188 完了コミット（4c91b9a0）で既に修正済みであることが判明した。再修正は不要なリグレッション源になるため、本サイクルでは「確認 + 範囲確定」に振り替える。
- **T-1 と T-2 を別々の単位として独立に進める案**: 両者は同じ 3 カラム Grid と外側コンテナの水平方向設計で交差する。独立に決めると一方の解決が他方の前提を壊すリスクが高いため、同単位（単位 A）で設計確定する。
- **T-9 を「拡張予定なしとして backlog から外す」案**: `docs/design-migration-plan.md` Phase 8.3 で 4 階層パンくずが確実に発生するため、外せない。代わりに Phase 8.3 系 backlog 側に着手条件を移して移管する（移管先の具体ルート確定方法は単位 G を参照）。
- **T-5 を全件削除して抑制コメントゼロにする案**: image-base64 の `dataUri` 周辺など実際に CodeQL の指摘を抑止するために必要な箇所は残す。「全削除」と「全残し」の中間で、必要性を 1 箇所ずつ判定する。
- **T-6 を render ベース検証へ昇格して根本解決する案**: 工数が大きく本サイクルの範囲を超える。T-5 で誤検出元（不要 lgtm）を取り除く応急処置にとどめ、構造リスクの完全解消は単位 D の完成判定に従って `docs/backlog.md` に新規 B-XXX として起票する。
- **T-4 の (a)(b)(c) 案の判断軸**: 3 案を並列に並べ事前順位付けは行わない。実画面・実テストでの検証で選択し、constitution の Decision Making Principle に従い工数を理由に劣る方向を選ばない。判断軸はリグレッション影響範囲と既存パターンとの整合。
- **T-10 のテストを残したまま件数自動計算に置換する案**: テストの責務（リダイレクト動作 vs 件数計算ロジック）が混ざる構造自体が不健全。削除して責務分離する方が将来の保守性が高い。

各単位において複数の修正方向が存在することは本計画では並列に列挙し、本サイクルでの採用方向の事前順位付けは行わない。実装段階で実画面比較・影響範囲・既存パターンとの整合を踏まえて確定する（AP-P02 / AP-P11 抵触防止）。
