# サイクル-168

このサイクルでは以下の2タスクを実施する。

- **B-307**: PMキャラクター像の定義
- **B-325**: ターゲットユーザー定義の見直し（キャラクターはターゲットユーザー像に近い方が理想的な結果が得られるため、本サイクルに合流）

cycle-167で策定したサイトコンセプト「日常の傍にある道具（と、ちょっとした息抜き）」の方針の一環として、PMにキャラクター性を持たせてロールプレイさせる方針が定められた。理念的な判断基準（「来訪者への価値を最優先とする」等）は明文化しても一貫して適用されにくいため、発想を変え、キャラクターとしての言動パターンに落とし込むことで日々の判断の一貫性を確保する。

キャラクターを作る前提として、新コンセプトに合わせたターゲットユーザー像の再定義も本サイクルで完了させる。旧コンセプト由来の「占い・診断を楽しみたい人」が削除済みでdocs/targets/README.mdとの不整合も発生しているため、この機会に正規化する。

サイトの独自性と価値を決める重要なタスクであり、Ownerによる介入を伴いながら進める。

## 作業計画

本計画はOwnerとの対話で合意された進め方を元に策定し、reviewerの指摘事項を反映して修正した。作業は8ステップ構成で、各ステップの成果物が次ステップの入力となる直列の流れを基本とする（Step 7のみ並行）。

### 基本原則: Owner主導 ≠ Owner絶対

本サイクルはOwner主導の重要タスクだが、Ownerが絶対的に正しいという意味ではない。各ステップの調査やレビューで適切に検証し、Owner-PM合意のうえで進める。Ownerの判断であっても、調査結果・事実・過去の失敗パターンと矛盾する場合は、PM/reviewer/researcherが根拠を示して異議を唱え、合意を取り直すこと。これは過去のアンチパターンAP-P04（Ownerの発言を未検証のまま前提化）への対策でもある。

### 目的

1. **B-307（PMキャラクター像の定義）**: constitutionの理念を「理念として守る」のではなく、キャラクターの自然な言動パターンとして体現させることで、日々の判断の一貫性を確保する。理念的な判断基準の明文化が一貫適用されにくい問題への構造的対策。**ただし「キャラクター化すれば判断が一貫する」という仮説自体が未検証であり、本サイクルは仮説検証そのものを目的とする**。本サイクル内で仮説の最終結論を出すことは不可能であり、実運用を通じて継続的に検証する。
2. **B-325（ターゲットユーザー定義の見直し）**: 新コンセプト「日常の傍にある道具」に整合したターゲットユーザー像を再定義し、キャラクター設計の土台とする。
3. **副次目的**: キャラクター自身がブログ（AIの日記）の書き手として読者に親しまれるコンテンツ性を持たせ、サイトの独自価値の一部とする。

### 誰のための作業か

- **直接の利用者はPM（このClaude Codeセッションを運用するメインエージェント）**。PMが自然にconstitution準拠の判断を下せるようになることで、最終的な受益者である **yolos.netの来訪者** に安定した品質の体験が届く。
- **サブエージェント（planner, builder, reviewer等）には意図的に適用しない**。同一の性格を持つエージェント同士が協調動作しても見落しの確率は下がらないが、それぞれが異なる視点を持てば問題に気付く確率が上がる。この「多様性による検出力向上」がOutput Stylesを採用する積極的理由の一つでもある。
- **ブログ成果物の読者は既存サブターゲット2件** および **再定義後の新メインターゲット**（Step 2で確定）。なお、Step 2で既存サブ2件の扱いが「削除」または「統合」に決まった場合、本記述および Step 8 の「サブターゲット視点レビュー」記述を Step 2 完了時に書き換えること（計画メンテナンスルール）。

  2. 価値観（最低3候補）
  3. 口癖・言い回し候補（最低5候補）

  8. **ターゲットとの類似度の方針**（完全に共鳴／同じ価値観だが視点は異なる／対照的だが相互理解がある など、最低3候補）

#### Step 5: キャラクター設定（SSoT）の作成

- **目的**: 話し言葉版・書き言葉版の **唯一の親ソース（SSoT）** を作成する。
- **作業**: builderが採用要素を統合したキャラクター設定ドキュメントを作成
- **PMのbuilderへの渡し方（AP-WF03対策）**:
  - 採用要素リスト（Step 4でOwner-PM合意したもの）
  - SSoTの構造要件（3要件別セクション、共通項目/分岐項目の線引き、rule 3整合方針を含むこと）
  - **具体的な文言・口癖の言い回し・キャラクターの台詞などは指定しない**。builderの解釈と創作に委ねる。ただしbuilderが解釈に迷う場合はPMに質問する経路を明示する
- **SSoTの内部構造**:
  - **共通項目**（話し言葉・書き言葉どちらにも継承される）: 性格、価値観、世界観、行動原理、思考の癖、一人称、名前、背景、constitution体現の具体例
  - **分岐項目**（話し言葉版と書き言葉版で異なる運用ルールを持つ）: 言語表出レイヤー（枕詞の使用、文の長さ、情感表現の強さなど）
  - 分岐項目は「SSoTにおいて分岐することを明示」し、話し言葉版・書き言葉版それぞれでどう異なるかの設計方針をSSoTが示す。具体化はStep 7で行う
- **配置（仮決め）**: `docs/character.md`。Step 5開始時に別案があればPM判断で変更可
- **完了判定**:
  - 3要件（①constitution体現、②具体的な口癖/思考パターン、③コンテンツ性）が **それぞれ別セクションとして明示** されている
  - 要件③のコンテンツ性は、どの要素（背景設定・一人称・趣味・弱点・口癖の愛らしさ等）で具体的に担保しているかが明示されている
  - 分岐項目と共通項目の線引きが明記されている
  - constitution rule 3（AI運営の透明性）とキャラクター演出の両立方針が記述されている（例: キャラクター自身がAIであることを自認した上で語る等）

#### Step 6: キャラクター設定のレビュー

- **作業**: reviewerのレビュー → Owner-PM合意
- **レビュー観点（必須チェック項目）**:
  1. 要件①②③それぞれについて、SSoTのどの記述がそれを満たしているかをreviewerが具体的に指摘できるか
  2. キャラクター化が単なる理念の言い換え（AP-WF09: 用語変更による同一構造の隠蔽）になっていないか。判断行動に実質的な違いが生まれるかを評価
  3. constitution rule 3（AI運営の透明性）が、キャラクター演出に埋没したり、読者を誤誘導する構造になっていないか
  4. site-conceptの「息抜き1割以下」原則に照らして、キャラクターの娯楽的要素が道具としての性質を損なわない程度に抑えられているか
  5. 確定したターゲットユーザー像との親和性
- **完了判定**: reviewer指摘の解消後、Owner-PM合意が取れている

#### Step 7: 2種類の最終成果物の作成（並行）

話し言葉と書き言葉では最適な特性が異なるため2形態に分離する。これは論理的帰結として妥当（反証: 1ファイル内分岐案は話し言葉モードで書き言葉ルールを読み込む混線リスクがあり、メンテと可読性でも分離が優位。暗黙使い分け案はAIが明示なく使い分けない性質から成立せず。反証として不成立）。キャラクターの核（SSoT）は共通で、表現ルールだけを分ける。

- **話し言葉側（builder A）**: Claude Code Custom Output Stylesファイル
  - 「あなたは〜」形式のプロンプト
  - SSoTの共通項目を前提とする
  - 分岐項目: 思慮深さを表現するための枕詞（例:「でもちょっと待って、使う人の気持ちで考えてみると……」）を含む
  - 配置: `.claude/output-styles/` 配下
  - 運用設定: `keep-coding-instructions: true`（コーディング機能を保持）。`.claude/settings.json` に `outputStyle` を記載し常時適用
  - 適用範囲: メインエージェントのみ（サブエージェントには適用しない。これは意図的設計であり、多様性により見落しを減らす狙い）
- **書き言葉側（builder B）**: ブログ用文体ガイド
  - SSoTの共通項目を前提とする
  - 分岐項目: 読みやすさ優先でシンプルに言い切る。話し言葉の枕詞は避ける
  - 良い例・悪い例・NG集
  - 配置（仮決め）: `docs/blog-writing.md` への追記。ただし、既存の `docs/blog-writing.md` は実務的なフロントマター規約・Markdown記法に特化しており、キャラクター由来の文体ガイドと性質が異なる。以下の基準で別ファイル化を判断する: (a) 追記量が100行を超える、(b) 「キャラクター文体」と「Markdown実務規約」を別目次で示さないと可読性が落ちる — いずれかに該当する場合は `docs/character-voice.md`（仮）として別ファイル化
- **整合性確保**:
  - 両builderに同じSSoTを参照させる
  - 両ファイルにSSoTへの相互参照リンクを記載
  - 並行実行中にSSoT改訂が必要と判明した場合、両builderを停止してStep 5に戻る
- **完了判定**: 両ファイルが作成され、SSoTと矛盾しない

#### Step 8: 最終成果物のレビューと実証テスト

仮説検証こそが本タスクの目的である。机上レビューだけでなく実応答・実文章を生成して機能を検証する。

- **実証テストの位置づけ**:
  - 本サイクルの目的は「キャラクター化すれば判断が一貫する」という仮説の検証である。本サイクル内で最終結論を出すことは不可能
  - 長時間の自律実行では同一条件でも結果が異なるため、A/B比較テストは有効でない（Owner判断）。実地に試して観察することが唯一の方法
- **実証テスト1（話し言葉）**: Output Stylesを適用した状態で **Ownerが用意するシナリオ** に対する応答を生成し、キャラクター性・constitution準拠・思考パターンが機能するかを観察する。シナリオはOwner側で多種多様に用意するため、本計画には例示しない
  - **合格判定の観察軸**: 各シナリオの応答について以下を観察する。(a) SSoTで定義した性格・口癖・思考パターンが応答に現れているか（要件①②）、(b) constitutionのルール（特に来訪者価値優先）に沿った判断・言動になっているか（要件①）、(c) キャラクターのコンテンツ性が応答を通じて伝わるか（要件③）、(d) キャラクター性がタスク遂行の妨げになっていないか。合否はOwner-PM合意で判断
- **実証テスト2（書き言葉）**: ブログ用文体ガイドに従って本サイクル内で1本以上のブログ記事を書き上げ、文体が機能するかを観察する。書き言葉成果物の実地検証であると同時に、来訪者への価値到達の第一歩とする
  - **公開可否の判断基準と主体**: 以下の3条件をすべて満たす場合に公開する。(a) reviewerが品質観点（内容の正確性・読みやすさ・価値）で合格判定、(b) サブターゲット視点で違和感がない、(c) 要件①②③がブログ文章に現れている。最終判断はOwner-PM合意
- **評価観点**:
  - 要件①②③が実応答・実文章に現れているか
  - サブターゲット（エンジニア向け）2件の視点で書き言葉成果物に違和感がないか
- **戻り修正の分岐基準**:
  - SSoT（Step 5）に起因する問題 → Step 5に戻り両builderを再実行
  - 表現レベル（話し言葉または書き言葉どちらか）のみの問題 → Step 7の該当成果物のみ修正
- **評価**: reviewerとPMで評価、Ownerに報告、Owner-PM合意で承認
- **完了判定**: 実証テスト結果をもとにOwner-PM合意が取れており、本サイクル後に運用を開始できる状態になっている
- **本サイクル後の継続検証**: キャラクター運用の効果はサイクルを重ねて継続的に検証する。本サイクル完了後、3サイクル後にキャラクター運用のふりかえりを行うタスクをbacklogに登録する
  - **Nサイクルの根拠**: 1サイクルで少なくとも数件のconstitution関連判断（施策選択、スコープ調整、コンテンツ削除／追加判断など）が発生する想定。3サイクルでおよそ10〜20件の判断サンプルが蓄積され、一貫性の傾向を観察するには十分。短すぎれば判断数が不足し、長すぎれば軌道修正が遅れるため、3サイクルを目安とする

### 検討した他の選択肢と判断理由

- **B-325を本サイクルに合流 vs 別サイクル化**: 合流を採用。キャラクターはターゲットユーザー像と親和的な方が理想的な結果を得るため、順序として先行する必要がある（Owner判断）。ただし「キャラクター-ターゲット親和仮説」自体の裏付けはStep 1・Step 3の調査で得る
- **配信方式（Output Styles / CLAUDE.md / Skill / --append-system-prompt）**: Output Stylesを採用（Owner判断）。理由:
  - `keep-coding-instructions: true` でコーディング機能を保持可能
  - `.claude/settings.json` に記載することで常時適用される
  - **サブエージェントに波及しないことが意図的メリット** — 同一性格の協調では見落し確率が下がらないが、異なる視点を持つサブエージェントとの協調により問題検出力が上がる
- **Step 7で話し言葉と書き言葉を分けるか統合するか**: 分離を採用。話し言葉の枕詞（迂遠な表現）はブログでは読みにくさにつながるため、表現ルールを分ける必要がある。配信経路も異なる（Output Stylesはメインエージェント向け、ブログ用文体ガイドは執筆時の書面ガイド）。1ファイル分岐案・暗黙使い分け案は反証として不成立
- **Step 8でA/B比較テストを入れるか**: 入れない。長時間自律実行では同一条件でも結果が異なるため、A/B比較では効果検証にならない（Owner判断）。仮説検証は実地運用を通じて継続的に行う
- **Step 8のテストシナリオ例を計画書に書くか**: 書かない。シナリオはOwner側で多種多様に用意するため、計画書に例を書くと語弊を招く（Owner判断）
