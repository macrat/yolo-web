# サイクル-261

このサイクルでは **B-539（デザインシステムの診断中心コンセプトへの再適合）** を実施する。

cycle-257（B-535）で初めて来訪者価値を実測した結果、価値は診断（`/play`）に頑健に集中し、道具箱トップ `/` は organic 集客ほぼ0・中核機能利用ゼロと判明した。これを受けて site-concept（cycle-259）→ targets（cycle-260）と一次文書を「診断中心」へ再センタリングしてきた。その連鎖の次の一手が、視覚言語の再適合である。

現在の設計3文書（`/DESIGN.md`・`docs/design-system-by-claude-design/`・`.claude/skills/frontend-design/SKILL.md`）は、いずれも旧コンセプト「道具箱-as-core／実務的な道具らしい極限ミニマル」前提で書かれ、冒頭に「本格的な再適合は B-539」という注記が貼られたまま宙吊りになっている。本文と注記が矛盾した状態は、UI を作るサブエージェントを誤導し続ける。

**デザインはコンセプトを伝えるためのもの**（cycle-257 の判断）。極限ミニマルは「実務的な道具」を雄弁に伝えるが、「自分を知る・楽しむ」という診断来訪者の体験を伝えているとは言いにくい。本サイクルは、コンセプトから「何を伝えるべきか」を起点に視覚言語を**実デザインで探索**し（プロセスを文言の書き換えに矮小化しない＝cycle-259 の事故の教訓）、方向を決め、3文書を再適合する。

## このサイクルでやること／やらないこと（スコープ）

- **やる**: (1) 診断来訪者の現状体験の接地（GA＋実画面スクショ）、(2) 診断中心の視覚言語が「何を伝え／何にはならないべきか」のデザインブリーフ、(3) 代表的な診断面に対する**実デザイン探索**（HTML/CSS プロトタイプを Playwright でスクショして比較できる具体物）、(4) 来訪者価値起点での方向決定、(5) 3文書の再適合（注記を本文へ昇格・旧枠組みの是正・概念中立なトークン/コンポーネント規約は維持）、(6) 計測・出荷協調の明記。
- **やらない（下流）**: 本番ページへの出荷＝トップ再設計は **B-542**、静的結果面は **B-523**。クイズ結果面の変更出荷だけは走行中 A/B `quiz_result_visual_v1` の結論（2026-07-21目安）と協調するため、本サイクルでは本番コードを出荷しない。本サイクルの成果物は「探索で根拠づけた方向＋再適合した設計文書」。プロトタイプは検証用の具体物であり本番投入しない。
- **重要な枠組み（cycle-259 事故の再発防止）**: 走行中 A/B は検証の**入力の一つ**であって決定者ではない。「絵文字/色 有無の2択をデータに選ばせて待つ」へ矮小化しない。コンセプト起点の探索・方向決定はデータを待たず着手してよい。同時に、決めた方向は「出荷時に計測で検証する仮説」として書き、未計測のデザインを「不変項」に固定しない（A-1 の逆＝正当な創造行為）。

## 作業計画

### 目的

確定済みコンセプト「自分を知り、楽しむ（診断中心）」を、サイトの視覚言語に正しく反映する。現在「道具箱-as-core／極限ミニマル」前提で書かれ注記で宙吊りになっている設計3文書を、コンセプト起点の実デザイン探索に基づいて再適合し、UI を作るサブエージェントが診断来訪者のための視覚言語を一貫して選べる状態にする。来訪者への最終的な価値（診断面が「自分についての発見を楽しむ・人に見せたくなる一瞬」になること）は、本サイクルで方向を決め、下流（B-542/B-523＋A/B）で出荷・検証する。

### 作業内容

上記「実施する作業」のとおり。要点は3つ。

1. **接地を先に**（作業1）。抽象論で視覚言語を決めない。診断来訪者が今この瞬間に何を見ているかを GA と実スクショで掴み、ギャップを指差せる具体物として記述する。
2. **実デザインで探索**（作業3）。cycle-259 で B-539 を「2択の数字選び待ち」へ矮小化した事故の教訓に従い、プロセスを文言書き換えに縮めない。方向の異なる視覚言語案を実際にプロトタイプ化し、スクショで比較して方向を決める。
3. **文書は仮説として書く**（作業5・6）。決めた方向は「出荷時に計測で検証する仮説」として記述し、未計測のデザインを不変項に固定しない。概念中立で有効なトークン・コンポーネント規約は維持し、変えるのは「何を伝えるか」に関わる視覚表現に絞る。

タスクはサブエージェントへ小さく分割して委譲する（プロトタイプは1案1エージェント・レビューは観点別に白紙 reviewer）。MCP（GA・Playwright）を使うサブエージェントは foreground で起動する（CLAUDE.md）。

### 検討した他の選択肢と判断理由

- **B-350（辞典トップ移行）を先にやる**: 不採用。B-350 は既存の新デザインシステムを適用する移行作業だが、そのデザインシステム自体を診断中心へ再適合する判断が宙吊り（注記状態）のまま下位ページへ展開を進めるのは順序が逆。設計の中心を先に正す B-539 が筋（cycle-260 振り返りも下流タスクの筆頭に B-539 を名指し）。B-350 は B-539 後に実施。
- **B-540（AP集 規約準拠クリーンアップ）を先にやる**: 不採用。プロセス文書整備で重要だが来訪者価値の連鎖から遠い。設計3文書の注記宙吊り（サブエージェント誤導源）の解消の方が、再センタリング連鎖の完結として優先度が高い。B-540 は単独で別サイクル。
- **B-539 を本番出荷まで一気にやる**: 不採用。クイズ結果面の出荷は走行中 A/B `quiz_result_visual_v1` の結論（2026-07-21目安）と協調しないと A/B を汚染する。また出荷は面ごとに独立タスク（トップ=B-542・結果面=B-523）として既に切られている。本サイクルは「設計システムの再適合（方向決定＋文書）」に絞り、出荷は下流で行う。これはタスクを小さく保つ原則とも整合。
- **B-539 を文書の書き換えだけで済ませる**: 不採用。これは cycle-259 で犯した「本質作業を工数の軽い暫定枠へ矮小化する」事故（AP-P28/P24/P05）の再演になる。視覚言語の方向は実デザイン探索（プロトタイプ＋スクショ比較）で根拠づける。
