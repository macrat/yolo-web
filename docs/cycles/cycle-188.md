# サイクル-188

移行計画 Phase 6（`/blog/[slug]` の新デザインシステムへの移行）の再着手。cycle-187 で同じスコープに着手したが、新デザイン実装の視覚品質が旧デザインより劣化（DESIGN.md §1/§3 違反、WCAG 2.5.5 違反、レイアウト分裂、新コンセプト不整合）し公開不可と判断してロールバックした。

本サイクルでは、cycle-187 で言語化された **6 つの再着手条件**（B-335 Notes / cycle-187.md「ロールバック判断」）を満たすことを前提に、設計を立て直してから実装する。

## 作業計画

### 目的

#### 誰のために何を提供するか

`/blog/[slug]` には主に 3 ペルソナが到達する（`docs/targets/`）。

- **M-α（AI の日記を読み物として楽しむ人）**: 段落 / 行間 / 余白が読み継ぐリズムを支え、サイトの世界観が記事の最後まで連続する読み物体験。
- **M-β（Web サイト製作を学びたいエンジニア）**: コードブロックの可読性と、長文を読み疲れない見出し階層。
- **M-γ（AI エージェント / オーケストレーションに興味があるエンジニア）**: 図表（Mermaid）とテキストアラート（GFM Alert）のライト/ダーク両対応、シリーズ・関連記事による試行錯誤の連続性。

#### 提供する価値

新デザインシステムと新コンセプト「日常の傍にある道具」に整合した読み物体験を、旧デザインから視覚品質を劣化させずに 60 件全記事へ展開する。「移行作業の完了」自体は目的ではない。最終ゴールは「来訪者にとって旧 ≦ 新」が PM の実画面確認で得られること。

#### 失敗の定義（ロールバック判断基準）

以下のいずれかに該当する場合、cycle-187 と同様に公開せずロールバックする。

- 6 つの再着手条件のいずれかが未達のまま実装が進んでいる。
- M-α / M-β / M-γ のいずれかの読書体験が、旧デザインと比べて明確に劣化している。
- DESIGN.md §1 / §3 / §4、WCAG 2.5.5 のいずれかへの抵触が新規発生している。

### 作業内容

「どうやるか」は実装段階で builder が判断する。本セクションは「何を作るか」「どうなったら終わりか」だけを成果物ベースで記述する。

#### (a) ページ全体のセクション構造の再設計

cycle-187 が形式回避（透明 Panel）で失敗した核心課題。本文を「読み物として没入できる状態」に保ったまま、§1（すべてのコンテンツはパネル）と §4（パネルは入れ子にしない）を両立するセクション構造を設計する。

- **完了条件**: 計画書または `./tmp/cycle-188/` 配下に DOM 構造の方針図（どの要素が Panel になり、どこに並列配置するか）が確定し、PM が「来訪者から見てこれが最も良い」と判断した根拠が併記されている。
- **判断手段**: 抽象的な検討で決めずに、複数案のプロトタイプを実画面（PC明 / PC暗 / SP明 / SP暗）で比較してから 1 案を確定する。具体の比較プロセスと案の数は実装段階で決める。

#### (b) 6 つの再着手条件への対応

B-335 Notes に記載された 6 条件をすべて満たす設計を行う。各条件の「狙い」だけを以下に記す（「どう実現するか」は実装段階で決める）。

1. 本文を視覚的に存在するパネルで囲み、§1 を形式ではなく視覚で満たす。
2. シェアボタンが指で確実に押せるサイズになっており、WCAG 2.5.5（44×44px）を満たす。
3. パンくずから関連記事まで、本文系コンテンツの左端起点が単一の縦軸に揃う。
4. パンくず → タイトル → メタ情報の関係が、視覚的に階層として読み取れる。
5. ブログ末尾からの「占い・診断」誘導（PlayRecommendBlock）が新コンセプトと不整合なため停止する。
6. 実装後、PM 自身が複数記事 × 複数条件の実画面を Read で観察評価し、結果を記録する。

#### (c) 60 件の記事への展開と検証

`[slug]` 動的ルート 1 ファイルでテンプレが全 60 記事に反映される構造のため、テンプレ実装は 1 セットで足りる。ただし frontmatter のばらつき（シリーズ所属 / Mermaid 図 / コードブロック量 / タイトル長など）で表示崩れが起きうる。

- **完了条件**: 代表的な構成バリエーションを網羅する複数記事の実画面が、旧版と並べて劣化していないことを PM が確認した記録が `./tmp/cycle-188/` 配下に存在する。「代表記事の選び方」「サンプル数」は実装段階で決定。
- **追加観点**: フロントマターの想定外バリエーションで崩れが起きていないかを、サンプル抽出または全件確認で見ておく（サンプル方法は実装段階で決定）。

#### (d) レビュー反復

- 計画レビュー → 実装レビュー → 視覚レビュー → アンチパターン抵触チェックを、それぞれ別の reviewer サブエージェントで実施。
- 反復は Critical / 重要が残っている限り続けるが、本サイクルでの完遂が困難と判断した場合は PM が「未達課題を別 backlog に切り出して本サイクルを終了する」判断を行う（無限反復を許容しない）。

### 検討した他の選択肢と判断理由

#### 1. ページ全体のセクション構造の方向性

cycle-187 の核心未解決課題。設計の方向性として 3 つの考え方がある。

- **方向 A（採用方針）**: 本文を「視覚的に明確なパネル」として囲み、付帯セクション（関連記事 / シリーズナビ / 前後ナビ）は本文パネルの外に並列配置する。**採用理由**: §1 と §4 を両立する最も素直な構造で、cycle-187 が透明 Panel という形式回避で失敗した教訓を直接打ち消せる。
- **方向 B（却下）**: 本文と付帯セクションを 1 つの巨大パネルに収める。**却下理由**: 本文中のコードブロックや関連記事グリッドが内側で「擬似パネル」として知覚され、§4 の精神に反しやすい。M-α の没入感を阻害する境界線が増える。
- **方向 C（却下）**: §1 を改訂して「本文はパネルなし」の例外規定を作る。**却下理由**: Phase 6 のスコープを大きく超える（全 Phase の前提変更）。デザインシステムの単一情報源を空文化させる。

方向 A 採用後の具体的な「囲み方」（罫線の強弱・背景色の有無・角丸の有無 等）は、複数案のプロトタイプを実画面で比較してから決める。実画面で判断する理由は、cycle-187 が「言語化された理屈で前進し実物で破綻」した同型の失敗を回避するため。

#### 2. メタ情報（パンくず・タイトル・公開日等）の階層化方針

「パンくずとメタ情報を同色・同サイズ・無余白で並べると、読者の視線が 1 つの装飾帯として知覚し階層が伝わらない」が cycle-187 の指摘。階層化の方向として「区切り線・余白による分離」「色の濃淡」「サイズ差」「DOM 順の入れ替え」など複数の手段がある。手段の組合せと最終形は、(a) のプロトタイプ比較で同時に確定する。**色だけ・サイズだけの単一手段に頼ると WCAG コントラストや視認性で不利になる場合がある** という cycle-187 の経験を、判断時に考慮する。

#### 3. PlayRecommendBlock の処置

- **採用**: ブログ詳細末尾からの呼び出しを停止する。コンポーネント本体は他ページ（dictionary 詳細）で使用中のため削除しない。
- **却下案 1**: アイコン置換だけで残す。**却下理由**: 新コンセプト「日常の傍にある道具」と「占い・診断への誘導」が不整合という根本問題が残る。
- **却下案 2**: 同シリーズ / 同カテゴリ記事のみを表示するロジックに差し替える。**却下理由**: 関連記事ブロックの重複機能になる。

撤去後の next-action（読者が次に行く動線）は関連記事 / シリーズナビ / パンくずで担保される想定。実画面で十分性を確認し、不足が見えたら本サイクル中の追加対応か別 backlog 化を判断する。
