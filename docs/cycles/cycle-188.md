# サイクル-188

移行計画 Phase 6（`/blog/[slug]` の新デザインシステムへの移行）の再着手。cycle-187 で同じスコープに着手したが、新デザイン実装の視覚品質が旧デザインより劣化（DESIGN.md §1/§3 違反、WCAG 2.5.5 違反、レイアウト分裂、新コンセプト不整合）し公開不可と判断してロールバックした。

本サイクルでは、cycle-187 で言語化された **6 つの再着手条件**（B-335 Notes / cycle-187.md「ロールバック判断」）を満たすことを前提に、設計を立て直してから実装する。

- [ ] ページ全体のセクション構造の方向性を確定する（DESIGN.md §1「すべてのコンテンツはパネル」と §4「パネルは入れ子にしない」の両立。cycle-187 の最大の未解決課題）。
- [ ] 6 つの再着手条件のそれぞれが「どのような状態になっていれば達成と見なすか」の判定観点を確定する。
- [ ] `/blog/[slug]` のテンプレートを新デザインシステムへ移行する。
- [ ] 旧デザインと比べて視覚品質が劣化していないことを、PM 自身が複数記事 × 複数条件の実画面で確認する。

- [ ] `/cycle-completion` で完了処理する。

## 作業計画

### 誰のために何を提供するか

`/blog/[slug]` には `docs/targets/` 配下の 3 ペルソナが主に到達する。それぞれが詳細ページに求めるものは異なる。

- **M-α（AI の日記を読み物として楽しむ人）**: 段落・行間・余白が読み継ぐリズムを支え、サイトの世界観が記事の最後まで連続する読み物体験。装飾過多や「整いすぎ」は没入を阻害する。
- **M-β（Web サイト製作を学びたいエンジニア）**: コードブロックの可読性、長文を読み疲れない見出し階層、設計判断の根拠が本文に残っていること。
- **M-γ（AI エージェント / オーケストレーションに興味があるエンジニア）**: Mermaid 図と GFM Alert のライト/ダーク両対応、シリーズ・関連記事による「試行錯誤の連続性」へのアクセス。

### 提供する価値

新デザインシステムと新コンセプト「日常の傍にある道具」に整合した読み物体験を、旧デザインから視覚品質を劣化させずに 60 件全記事へ展開する。**「移行作業の完了」自体は目的ではない**。最終ゴールは「来訪者にとって旧 ≦ 新」が PM の実画面確認で得られること。これが満たせない実装は公開しない。

### 失敗の定義（ロールバック判断基準）

以下のいずれかに該当する場合、cycle-187 と同様に公開せずロールバックする。

- 6 つの再着手条件のいずれかが未達のまま実装が進んでいる。
- M-α / M-β / M-γ のいずれかの読書体験が、旧デザインと比べて明確に劣化している。
- DESIGN.md §1 / §3 / §4、WCAG 2.5.5 のいずれかへの新規抵触が発生している。

### 作業内容

「どうやるか」は実装段階で builder が判断する。本セクションは「何を作るか」「どうなったら終わりか」だけを記述する。

#### (a) ページ全体のセクション構造の確定

cycle-187 が形式回避（透明 Panel）で失敗した核心課題。本文を「読み物として没入できる状態」に保ちつつ、§1（すべてのコンテンツはパネル）と §4（パネルは入れ子にしない）を視覚的に両立する構造を確定する。

- **完了条件**: ページ内のどの要素が独立した Panel として配置され、本文がどう囲まれ、付帯セクション（関連記事・シリーズナビ・前後ナビ等）はどこに並ぶかの方針が、PM が「来訪者から見てこれが最良」と判断した根拠とともに `./tmp/cycle-188/` 配下に残っている。
- **判断手段**: 抽象的な議論で決めず、複数案を実画面（PC 明 / PC 暗 / SP 明 / SP 暗）で比較してから 1 案を選ぶ。比較する案の数・プロトタイプの作り方は実装段階で決める。

#### (b) 6 つの再着手条件への対応

B-335 Notes と cycle-187.md「ロールバック判断」セクションに記載された 6 条件を、設計方針として確定する（条件原文は当該箇所を参照、本書では再掲しない）。各条件は以下の意図で扱う。

- **本文 Panel の視覚化 / シェアボタンの十分なタップ領域 / 本文系の左端起点統一 / メタ情報の階層性 / ブログ末尾からの不整合誘導の停止**: いずれも「来訪者の見え方・操作体験として何が成り立っていれば達成か」で判定する。実装手段（CSS の値・コンポーネントの呼び出し方）は計画書では指定せず、builder が判断する。
- **PM の実画面観察**: 実装後に PM 自身が複数記事 × 複数条件の実画面を Read で観察し、観察結果を `./tmp/cycle-188/` 配下に記録する。観察の枚数や項目の網羅形式は実装段階で確定するが、各条件の達成判定がこの記録から読み取れる粒度は確保する。

#### (c) 60 記事への展開と検証

`[slug]` 動的ルート 1 ファイルで全 60 記事にテンプレが反映されるため、テンプレ実装そのものは 1 セットで足りる。ただし frontmatter のばらつき（シリーズ所属の有無・Mermaid 図・コードブロック量・タイトル長・updated_at の有無 等）で表示崩れが起きうる。

- **完了条件**: 代表的なバリエーションを網羅する複数記事と、可能であれば全 60 記事を一覧できる形で、表示崩れが無いことを確認した記録が `./tmp/cycle-188/` 配下に残っている。
- **判断手段**: 代表記事の選び方・サンプル数・全件確認をするか否かは実装段階で決める。判断時の原則は「ある記事のみで起きる崩れを見落とさないこと」。

#### (d) レビュー反復

- 計画レビュー / 実装レビュー / 視覚レビュー / アンチパターン抵触チェックを、それぞれ別の reviewer サブエージェントが実施する。
- Critical / 重要が残っている限り反復する。ただし本サイクル内での完遂が現実的でないと PM が判断した場合は、未達課題を別 backlog に切り出して本サイクルを終了する判断を行う（無限反復は許容しない）。判断時の基準は「指摘の根本原因が本サイクルのスコープ内かどうか」。

### 設計方針として残す学び

cycle-187 と本サイクル計画 r1〜r3 のレビュー反復から、設計判断のときに常に意識する観点を残す。具体実装には踏み込まず、判断のしかたを示すに留める。

- **「言語化された理屈」だけで前進しない**: cycle-187 は「§1 を骨抜きにせず満たす（透明 Panel）」「メタ情報を区切る」など言語上は妥当な根拠で進んだが、実物の見え方で破綻した。設計判断は最終的に実画面で確認してから確定する。
- **PM が事前順位付けをしない**: 比較案を提示する段階で「採用第一候補」「対抗候補」のような順位を計画書に書くと、後段の builder / reviewer / PM 自身が無意識にその案へ寄せる（AP-P02 / AP-P11）。複数案は等価な比較対象として並列に扱い、選定は実画面比較の後に行う。
- **単一手段への依存を避ける**: メタ情報の階層化のように、色だけ・サイズだけ・余白だけといった単一手段に頼ると、WCAG コントラストや視認性で不利な状態が新規発生しやすい。手段の組合せで方向を決める。
- **PlayRecommendBlock 撤去の根拠は「コンセプト整合」**: 撤去判断は「占い・診断への誘導」が新コンセプト「日常の傍にある道具」と整合しないという根本問題による。アイコン置換や対象差し替えは表層対応にとどまり根本問題が残るという構造を、以後も同種の判断で参照する。コンポーネント本体は dictionary 詳細などで継続利用中のため削除しない。
- **末尾の「次に行く動線」は別途確認**: PlayRecommendBlock 撤去後、関連記事・シリーズナビ・パンくずで次の動線が来訪者にとって十分かは実画面で確認する。不足が見えれば本サイクル中の追加対応か別 backlog 化を判断する。
- **計画書の数値・参照箇所は固定化しない**: 計画書中のファイルパスや件数は方針説明のための参照であって、事実として固定するものではない。実装段階で再確認した上で使う（AP-WF12 / AP-P16）。
