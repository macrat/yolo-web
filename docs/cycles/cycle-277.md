# サイクル-277

**これが最後の1回である。** cycle-268〜276 の9サイクル、B-545（サイトの根幹の統一）は完了していない。オーナー通告：10連続失敗＝オーナー直接介入。バッファは cycle-276 が消費済み。本サイクルで B-545 を完了させる。

cycle-276 の教訓（自己点検・オーナー指摘）を出発点にする：

1. **土台が無かった**——決定(a)=診断中心は現状アクセスの内訳（週119PV程度）に錨を下ろしただけで、「将来 PV を最大化する勝てる概念か」を外部データ（検索需要・市場性・競合）で検証していない（AP-P03/P01/P02）。
2. **パッチを重ねた**——道具箱移行計画への注記の重ね塗り（AP-P28）。
3. **断裂を出荷した**——診断コンテンツ＋道具箱デザインのツギハギをサイトの顔に出した（AP-P27）。
4. **手続きの厚みを証明と誤認した**——reviewer 承認3回はわたしが与えた枠組みの中の検証にすぎなかった（AP-WF02）。

だから本サイクルは、**(1) まずマーケット調査で土台を埋め、(2) 決定を確定し、(3) 計画は破棄して作り直し、(4) トップはコンテンツとデザインを同時に仕上げ、(5) デザインシステム根幹の作り直しを実装まで land させる**。

## 作業計画

### 目的

B-545 を完了させる：サイトの根幹（コンセプト・アーキテクチャ計画・デザインシステム・正面玄関/自己定義）を、外部データで検証された一つのコンセプトに本質的に統一し、実装まで land させる。完了の定義（cycle-275/276 引き継ぎ）＝「サイトが 3〜4 コンセプトのツギハギでなくなる方向に根幹を実際に統一する」こと。トップ改修単独を完了と呼ばない（矮小化禁止）。

### 作業内容

T1〜T9 を順に実施する。順序が本質である：**調査（T1-T2）→決定（T3）→計画（T4）→デザインシステム（T5）→実装（T6）→検証（T7-T8）**。cycle-276 はこの順序を逆走（実装が先・調査なし）して失敗した。

- T1 は PM 自身が実施（GA 確認は PM の責務・CLAUDE.md）。
- T2 は WebSearch/WebFetch を使うサブエージェント複数名に分担（診断系需要／ツール系需要と競合／辞典系需要と競合／反証探し）。出典と日付を記録。
- T3 は PM が決定し、reviewer が枠組みごと検証。
- T4〜T6 はサブエージェントへ委譲し PM が全差分を実体検証。T5/T6 は frontend-design スキル参照必須。
- 決定(a)の結論が「診断中心」以外になった場合も、本サイクルの完了要件（根幹の統一）は変わらない——統一先が変わるだけである。調査結果に正直に従う。

### 検討した他の選択肢と判断理由

- **マーケット調査を飛ばして cycle-276 ロードマップ（austere 降格・§7 基調化）を直接実行する**：却下。cycle-276 の最大の欠落＝土台（外部検証）を再び飛ばすことになり、同じ失敗の10度目になる。オーナー指摘・引き継ぎ #1 に真っ向から反する。

- **B-545 を分割して一部だけ完了扱いにする**：却下。矮小化（cycle-274 型）。完了の定義を守る。
