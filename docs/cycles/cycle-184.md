## 作業計画

### §作業内容 §0: frontmatter `tags` 残置の根拠（再検証、Owner 指摘への回答）

cycle-183 v4 計画段階では「残置」を採用していたが、Owner から「タグを表示しないのに frontmatter で `tags` を維持管理するコストや、書き手の『表示されている』という誤解リスクを上回るメリットがあるか丁寧に確認すべき」との追加指摘を受け、実体検証を行った（レポート: `docs/research/2026-05-08-frontmatter-tags-retain-or-delete-analysis.md`）。

#### 残置のメリット（実体確認済み）

1. **`getRelatedPosts` の品質維持**: `japanese-culture` カテゴリ全 5 記事は `series=japanese-culture` でシリーズ除外ルールにより互いに推薦対象外。タグスコアリングだけが唯一の異カテゴリ記事推薦源で、frontmatter `tags` を削除すると、これら 5 記事の関連記事欄が **0 件** になる
2. **`getPlayRecommendationsForBlog(post.tags)` 依存（cycle-183 v4 で見落とされていた）**: 記事詳細ページ「この記事を読んだあなたに」欄のテーマ一致レコメンドが frontmatter `tags` に依存。削除でフォールバック固定（テーマ無関係推薦）に劣化
3. **検索 137 件のタグオンリーヒット**: 「設計パターン」19 件 / 「Web 開発」16 件 / 「TypeScript」8 件等の主要タグは、タイトル・description にも含まれない記事を多く含み、`tags` 削除で潜在的な検索品質が劣化

#### Owner 指摘論点への対処

- **維持管理コスト**: 実体は「平均 3.4 タグ / 表記ゆれ 1 組のみ（ワークフロー vs ワークフロー連載）」と小さい
- **「表示されている」誤解リスク**: B-389-9-2（writing.md 追記）で予防。文面冒頭で「UI には一切表示されない内部メタデータ」を強調する

### 目的

#### 何のために、誰のためにやるのか

cycle-183 の補修ホットフィックス（>=3 件タグのみ表示する UI フィルタを `BlogCard.tsx` / `TagList.tsx` に追加）は、404 になるタグ pill を UI から隠すための暫定対応である。frontmatter `tags` の追加は記事執筆者の判断であるべきだが、ホットフィックス下では「3 件未満のタグは UI で見えない」という暗黙ルールが常態化し、「タイトルに YAML とあるのにタグ欄に YAML が出ない」という認知的不整合（cycle-183 §AP-I01 運用記録参照）が継続する。X1 本体実装によりタグ pill UI そのものを廃止し、暫定状態を構造的に終了させる。

frontmatter `tags` フィールドは保持する。これにより `getRelatedPosts` のスコアリング（共有タグ数 × 3 + 同カテゴリ +10）、5 系統キーワード検索の `tags` ヒット動線、`getPlayRecommendationsForBlog` のテーマ一致レコメンドが維持される（§0 で再検証済み）。

#### ターゲット likes / dislikes との対応

- **M1c（特定の作業に使えるツールをさっと探している人）**: dislikes「似たようなツールが並んでいて、どれを使えばよいか迷わせること」「仕様や前提が曖昧で…」に該当。タグとカテゴリの役割重複（cycle-183 v4 で言語化された問い）を解消し、束ね軸を「カテゴリ単一」に単純化する
- **M1b（気に入った道具を繰り返し使っている人）**: dislikes「URL が変更されたコンテンツにリダイレクトが設定されておらずたどり着けなくなること」に該当。29 本の旧タグ URL に 301 リダイレクトを設定して経路を保全する
- **M1d（AIエージェントやオーケストレーションに興味があるエンジニア）/ M1e（Webサイト製作を学びたいエンジニア）**: 5 系統キーワード検索（`?q=`）でタグ語ヒットを維持する。タグ pill による束ね閲覧という導線は GA 実測（90 日 4 PV）から事実上機能していないため廃止し、検索動線に集約する

### 来訪者価値

1. **404 リンク混在の構造的解消**: cycle-183 ホットフィックスは「URL が無いタグを描画しない」という UI フィルタで応急処置していたが、タグ pill UI そのものが消えれば「クリックできるように見えて遷移できない」現象は二度と発生しない
2. **情報設計の単純化**: 記事の束ね軸が「カテゴリ単一」になり、M1c の dislikes「どれを使えばよいか迷わせる」を予防。frontmatter `tags` は内部メタデータとして残し、検索・関連記事の精度向上に内部利用される
3. **執筆規約の自由度回復**: ホットフィックス下では「3 件未満のタグは UI で見えない」という暗黙の閾値が記事執筆判断に影響していた。X1 採用後は frontmatter `tags` を「記事内容を端的に表す名詞 3〜6 個を自由に追加してよい」と再定義でき、新分野の記事を縛らない（U-3）

### 作業内容

#### 1. リダイレクト方針（U-1 確定）

**採用方針**: `/blog/tag/:tag*` → `/blog` の 301 ワイルドカード一括リダイレクト。

**根拠**:

- GA 実測で `/blog/tag/*` 配下の直近 90 日 PV が **4 PV**（タグ 26 種類 × 90 日に対して総計 4）。月平均で 1〜2 PV / 月。個別マッピング（`/blog?q=タグ名`）を整備する工数は PV に対して過剰
- `/blog?q=タグ名` への誘導案は (a) URL エンコードが必要、(b) `/blog` の `?q=...` は `canonical: /blog` で集約されるため SEO 上は等価、(c) 「タグ名のテーマを期待した来訪者」を `/blog` トップ + 検索 UI で受ける設計のほうが、5 系統検索の認知を促せる、という理由で採用しない
- `/blog/tag/:tag/page/2` 等の p2 リダイレクトもワイルドカードで一括カバーされる

**注意**: 既存 `paginationRedirects` 内の `/blog/tag/:tag/page/1` → `/blog/tag/:tag` ルールは、新ワイルドカード追加後は二段リダイレクトを生むため、整理する（または削除して新ルールに統合）。

#### 2. TAG_DESCRIPTIONS の処遇（U-2 確定）

**採用方針**: 削除。

**根拠**:

- タグページが消えるため利用箇所がゼロになる
- 将来の X7（サジェスト）/ Phase 5 横断検索でサジェスト文言として再利用する可能性は理論上あるが、現時点で具体施策が立っていない（YAGNI 原則）
- 必要になった時点で `git log` から復元可能。archive ファイル化は不要

将来再利用する余地を残したい場合は B-389 とは独立の backlog として起票する（cycle-184 のスコープ外）。

#### 3. frontmatter `tags` 運用規約の追記（U-3 確定）

**追記先**: `docs/anti-patterns/writing.md` の末尾に新項目として追加。

**理由**: writing.md は記事執筆時の事後検証チェック項目を集約した場所であり、「frontmatter `tags` の運用」は記事執筆時の判断に該当する。CLAUDE.md は全エージェント共通のメタルールが集約されており、特定領域の規約はサブドキュメントに分離するのが整合的。

**伝達要素（PM 確定、文章表現は builder 判断）**: (a) frontmatter `tags` は UI に一切表示されない内部メタデータである（誤解リスク予防のため冒頭で強調）、(b) 用途は `getRelatedPosts` スコアリング・5 系統キーワード検索・`getPlayRecommendationsForBlog` のテーマ一致レコメンド、(c) 記事数を意識せず自由に追加可能（記事内容を端的に表す名詞 3〜6 個推奨）、(d) タグ命名は記事内検索のヒット視点で決める。

## 検討の到達点（タグを単純に削除する案の再評価）

着手前の調査・計画立案を進める過程で、frontmatter `tags` を残置しつつ公開 UI を全廃する X1 案、および補助案として検討した X9 案（タグをカードからは廃止しつつ関連記事ブロック等の脇に補助表示として残す）を実装した場合の影響を、コードと GA 実測値に当てて点検した。その結果、**タグを単純に削除することは読者への提供価値を構造的に毀損する** ことが判明した。本節はその毀損の中身を具体的に記録する。

### 1. レコメンドのブラックボックス化

`src/blog/_components/RelatedArticles.tsx` の「関連記事」欄、および `src/play/_components/PlayRecommendBlock.tsx` の「この記事を読んだあなたに」欄は、いずれも frontmatter `tags` をスコアリング入力としているが、推薦結果に **なぜその記事が選ばれたかを示すラベルや手掛かり** を一切出していない。タグ pill UI が消えるとカードや記事詳細ヘッダーからもタグが見えなくなるため、読者は「ここに並んでいる記事と本文との関係」を画面上の手掛かりだけでは把握できなくなる。レコメンドのスコアリング精度自体は維持されるが、「結果は出るが理由は見えない」状態は、読者がその結果を信頼してクリックする動機を直接削る。

### 2. 関連記事の品質劣化（カテゴリ単独では救えないケース）

`japanese-culture` カテゴリ（5 記事）は、全記事が `series=japanese-culture` に属するため、`getRelatedPosts` のシリーズ除外ルールにより互いに推薦対象から外れる。タグスコアリング（共有タグ × 3）だけが唯一の異カテゴリ記事推薦源で、frontmatter `tags` まで削るとこの 5 記事の関連記事欄が **0 件** になる。「次に読むもの」を提示する動線がページから完全に消えることを意味し、回遊体験を直接押し下げる。

### 3. テーマ一致レコメンドの全廃

ブログ記事詳細の「この記事を読んだあなたに」欄は `getPlayRecommendationsForBlog(post.tags)` を入力としており、frontmatter `tags` を削るとフォールバックの固定リスト（テーマ無関係推薦）にしか到達しない。記事を読み終えた読者に「内容と関係しないコンテンツ」を提示する状態は、M1d / M1e（エンジニア層）の興味関心ラインに反する。

### 4. 「気配で発見する」動線の喪失

M1c の likes には「書き手の癖や語り方が記事をまたいで一貫しており、個として認識できること」がある。これは、タグ pill が一覧 UI 上に提示されていることで「打鍵せずに気配で発見する」体験として部分的に担保されてきた。タグ UI を消すと、(a) 検索動線（5 系統検索）は維持されるが「自分から検索ボックスを開く能動性」を要求する、(b) `/blog?q=*` 流入比率は GA 実測で 6.6% に留まっており、能動検索を使わない来訪者層は確実に存在する。タグ pill が担っていた「眺めて気付く」価値の代替手段が、X1 案の中には存在しない。

### 5. 補助表示として残す案にも残る軸重複

タグをカードからは廃止しつつ関連記事ブロック等の脇に補助表示として残す案（X9）も検討したが、これも結局のところタグ pill が画面に並ぶ点で、「カテゴリ = ナビ動線、タグ = レコメンド理由ラベル」という挙動の差を読者が学ぶ必要が生じ、タグとカテゴリの軸重複（cycle-183 §補修計画 v4 で言語化された読者の混乱の核心）は名目上残る。すなわち cycle-183 §1〜§3 で列挙した X1〜X8 はいずれも、軸重複の構造解消と本節 §1〜§4 の毀損回避を同時に満たす設計にはなっていない。

### 6. 結論

タグを単純に削除する方針はとらない。当面は cycle-183 ホットフィックス（>=3 件タグのみ表示する UI フィルタ）を維持する。タグページの直近 90 日 PV は 4 PV と実害規模が極小であり、ホットフィックスの長期化が来訪者体験を悪化させる兆候は今のところ観測されていない。タグ機能とカテゴリ機能の関係性、およびレコメンド理由の可視化を含む情報設計は、新デザイン移行（Phase 5-10）の進捗と組み合わせて、改めてゼロベースで設計し直す。

### B-391: ブログ分類の軸重複解消とレコメンド理由の可視化（情報設計の再検討）

#### 何をするのか

ブログ記事の「タグ」と「カテゴリ」が両方とも記事を束ねる軸として並列している現状（読者が「どちらでドリルダウンすべきか」を学ばないと使い分けできない構造）と、関連記事ブロックおよび Play レコメンドが推薦理由を読者に提示せずブラックボックス化している現状を、まとめて解消する情報設計を、ゼロから再検討する。「タグ機能を廃止するか否か」という二択（cycle-183 §補修計画 v4 の X1〜X8 の枠組み）からは離れ、（a）軸の重複構造そのもの、（b）レコメンドの透明性、の 2 点を同時に満たす設計を出発点として組み直す。

#### 検討の出発点（cycle-184 §検討の到達点 §1〜§5 で確認した制約）

- **「タグを単純に削除する」だけでは提供価値が毀損する**: §1 レコメンドのブラックボックス化、§2 `japanese-culture` カテゴリの関連記事 0 件化、§3 テーマ一致レコメンドの全廃、§4 「気配で発見する」動線の喪失、§5 補助表示として残す案にも残る軸重複。これら 5 点を回避する設計が必要
- **frontmatter `tags` の内部用途は実害ある形では削れない**: `getRelatedPosts` スコアリング、5 系統キーワード検索、`getPlayRecommendationsForBlog` のテーマ一致レコメンドの 3 用途を維持しないと回遊体験が直接劣化する
- **タグ pill UI を残すと軸重複は名目上残る**: カードや記事詳細ヘッダーにタグ pill が並び続ける限り、「カテゴリとタグはどう違うのか」という読者の問いは解消しない

#### 検討すべき設計の方向性（あくまで出発点であり、これに限らない）

- **(α) レコメンド理由を読者に開く UI 設計**: 関連記事・Play レコメンドの各項目脇に「共有タグ: 設計パターン」のような小さい補助表示を出す。タグ pill をカードから消すと §4 の動線喪失が起きるため、カードのタグ pill を残すか・別の代替動線を用意するかを並行検討する
- **(β) 軸の統合（タグとカテゴリの一体化）**: 現状のカテゴリ 5 種類を主軸とし、タグはサブカテゴリ的に統合する。例: カテゴリ「dev-notes」配下に「設計パターン / Web 開発 / TypeScript / ……」のサブ分類を入れて URL を `/blog/dev-notes/設計パターン` のような階層に再構築する。記事の frontmatter も再設計
- **(γ) タグの役割再定義 + UI 上の挙動差を明示**: タグは「検索キーワードのショートカット」と位置付け、クリック先を `/blog?q=タグ名` にしつつ、視覚的にもカテゴリリンクと差別化（例: `#` prefix、ホバー時のヒント、フォントウェイト変更）。読者がカテゴリ動線とタグ動線を「異なる挙動」として学習しやすくする
- **(δ) 軸の片方廃止 + 代替動線の整備**: カテゴリかタグのどちらかを廃止する案。ただし廃止する側の SEO 価値（カテゴリ 5 URL × N 記事のインデックス）と読者動線（curated description 等）の損失補填が必要

(α)〜(δ) はそれぞれが独立で、組み合わせの可能性もある。出発点として列挙するが、検討時に新たな方向性が浮上したらゼロベースで取り入れる。
