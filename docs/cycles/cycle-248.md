## 作業計画

### 目的

AP 集は「過去の失敗を繰り返さないための事後検証チェックリスト」であり、後続 PM とサブエージェントがレビュー時に参照するガードレールである。ところがそのガードレール自体に `.claude/rules/anti-patterns-directory.md` の Do/Don't 違反（手順混入・事例混入）が蓄積している。これは二重に有害：

1. **検証不能化**：手順記述や個別事例が混じると「自分は今この失敗をしていないか？」という事後検証の問いとして機能しなくなり、項目が長大化して通読・適用されにくくなる。
2. **違反の再生産**：cycle-247 では AP 集へ手順記述・事例（固有 dump 形式名）を追記してしまい Owner に撤去された。手本となるべき AP 集が違反形式を含んでいると、それを真似た追記が繰り返される（cycle-246 で AP-P24/P25・AP-P19/P21 に同種違反、cycle-247 で再発）。だから P2/P3 から P1 へ二度引き上げられている。

ガードレールを正しい形に直すことは、プロセス事故（cycle-243/244 の憲法違反＝来訪者価値の破壊に至った系列）の予防であり、来訪者へ雑な道具を渡さないための基盤整備である。

### 作業内容

- 対象は `planning.md`（AP-P01〜P25）と `workflow.md`（AP-WF01〜16 本体）の 2 ファイル。
- 各 AP 項目を 1 つずつ言語化して読み、Do/Don't 違反を分類する（形式的に「問題なし」で済ませない）。
- 是正は「削除」ではなく原則「相応しい場所への移設 + 問掛け形への整形」。
  - **手順混入**（例: AP-WF05 の「N×4 枚を必ず撮る」「kickoff 直後に撮る」、AP-WF16 の「各ラウンドで独立再実行する」、AP-P19 の「計画立案フローの最初に組み込む」、AP-I 系で確認した技術手順）→ 既存手順書（`.claude/skills/take-screenshot/SKILL.md`・`.claude/rules/testing.md`・`docs/knowledge/*` 等）に収録済みなら AP からは除去するだけ。未収録で保持価値があれば移設してから除去。
  - **事例混入**（例: AP-P19 の `**事例**` ブロック、AP-WF11/12/14 の `**cycle-XXX 事例**`）→ 引用先 cycle ドキュメントに詳細が実在することを確認した上で、AP からは cycle 番号だけ残す。詳細が cycle ドキュメントに無い場合は喪失するため、その判断を監査で明示（安易に消さない）。
  - **一般化された知見**（例: AP-WF11/12 の「一般化原則 = …」）→ 命令形なら問掛け形に変換し、AP の理由(→)に吸収。
- 「将来 AP 候補メモ」セクション（workflow.md 末尾）は、ヘッダーで「AP 本体とは別扱いの staging 領域」と明示され、N≥3 で昇格を検討する候補置き場として設計されている。対症療法案・事例を一時的に保持するのが本来の役割なので、本サイクルでは Do/Don't を本体と同じ強さでは適用しない。ただし「本体 AP と誤認される表記になっていないか」だけは確認する。

### 検討した他の選択肢と判断理由

- **B-390（全 4 ファイルの全項目を問掛け形へ統一）まで本サイクルで実施**：却下。B-390 は P3 で、対象が implementation.md・writing.md を含む全項目に及ぶ。kickoff の「作業は最小限」原則に反する。本サイクルは P1 が名指しする 2 ファイルに限定し、kickoff 中に確認した implementation.md の違反箇所（AP-I07/I09/I10/I11）は B-390 の Notes に具体物として記録して取りこぼしを防ぐ。
- **手順・事例を単純削除（移設なし）**：却下。AP-WF05 の撮影網羅性や AP-I 系の技術手順には保持価値のある知見が含まれ、単純削除は知見の喪失になる。Do/Don't が指す「正しい置き場」へ移設してから除去するのが規定通り。cycle-247 では Owner が撤去で対応したが、それは「既存手順書に収録済み or 一般化理由に吸収済みで AP に重複していた」ケースだったため。本サイクルでも個別に収録済みか判定する。
- **AP 本体の事例を引用先 cycle ドキュメントへ書き戻してから削除**：原則は cycle 番号だけ残す方針だが、古い cycle ドキュメントを遡って加筆するのは慎重に行う（誤って一次記録を改変しない）。詳細が既に cycle ドキュメントに在る場合がほとんどなので、監査で「在ることの確認」を必須とし、無い稀なケースのみ個別に判断する。
