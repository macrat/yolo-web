## 作業計画

### 目的

**誰のための作業か（直接の受益者）**: 将来サイクルの PM とビルダー、とりわけ立て直し本線 B-488/B-489/B-490/B-491 を担う者。立て直しの根拠となる調査・基準（DESIGN.md 準拠状況の棚卸し、AP-P21 等の SSoT 体系）が git 管理外の `tmp/`（消失保証なし）にあると、本線着手時に根拠が消えて再導出・見落としが起き、結果として 34 ツール再構築（B-490）の品質が落ちる。

**最終的な受益者（来訪者）**: B-490 は「特定の作業に使えるツールをさっと探している人」が毎日使うツール群の再構築であり、その品質は立て直し研究資料の保全に依存する。本タスクは内部衛生に見えるが、**来訪者価値を生む再構築の土台（根拠資料）を消失から守る**ことが本質。

**提供する価値**: (1) 立て直しに必要な知見が `docs/`（git 管理・永続）に保全され、本線着手時に確実に辿れる。(2) `docs/` から `tmp/`（消失する場所）への参照が解消され、将来のリンク切れ・知見消失を防ぐ（`.claude/rules/tmp-directory.md` Don't「他ディレクトリから tmp を参照するな」違反の是正＝cycle-220 ③-5）。(3) どの参照を意図的に残したか（残す理由つき）を本ドキュメントに記録し、将来サイクルが ③-5 を再提起して同じ調査を繰り返す無駄をなくす。

### 判断の枠組み（全 tmp 参照に適用する分類基準）

code-researcher が docs 配下の全 tmp 参照を洗い出した（`grep -rn "tmp/" docs/ --include="*.md"` で再現可能。棚卸し結果は実行 builder にタスク指示で直接渡し、恒久的な判定結果は本ドキュメント「補足事項」に記録する。棚卸しは本タスク完了後に陳腐化する作業成果物のため永続 doc 化しない）。各参照を次の基準で分類して処理する。**「将来も参照価値のある知見か」を軸に判断し、編集コストや件数の多さを判断理由にしない**（CLAUDE.md: 実装コストを理由に劣った結果を選ばない＝保全すべきものを件数を理由に切らない／逆に価値のない使い捨てファイルを「リンク維持のため」に保全して docs を汚さない）。

- **(保全)** 参照先が現存し、**ongoing な立て直し本線（B-488〜491）に役立つ／再利用可能な知見・基準を含む**もの → `docs/`（research / knowledge / archive のうち内容に適した場所）へ保全し、その**ファイルへの参照を保全先パスへ修正（repath）**する。
- **(参照解消)** 参照先が使い捨て（査読ラウンド中間物・スクショ・計測中間ファイル等）で**結論が既に参照元 doc 本文に内包されており保全不要**、または参照先が既に消失している → 参照を、現存する保全済みパスへ修正・削除・中立化（「当時スクショを取得した」等の事実記述に変える）のいずれかで解消する。
- **(意図的に残す＝違反でない)** 次のいずれか。本ドキュメントに「残す理由」を記録する。
  - OS のシステム `/tmp/`（プロジェクト `./tmp/` ではない）への言及。
  - `tmp/` の**使い方を説明するルール記述**（例: 「スクショは `./tmp/` に保存する」）や、命名規約の例示・未採択案の仮想パス例示（実ファイルへのリンクではない）。
  - **完了済みサイクルドキュメント（`docs/cycles/`）内の、使い捨てファイルへの歴史的言及**。結論はそのサイクル文書本文に残っており、tmp パスは「当時こう作業した」という封印済みの作業記録。価値ある研究を指す場合は (保全) 側で個別に repath するが、使い捨てファイルへの歴史的言及まで遡って書き換えるのは記録の改ざんに近く、知見保全にも参照健全性にも寄与しないため残す。

### 作業中の注意点

- **件数・編集コストで判断しない**: 保全すべき価値ある知見を「件数が多い」を理由に切り捨てない。逆に価値のない使い捨てファイルを「リンクを維持するため」に保全して `docs/` を汚さない（doc-directory ルール「古くなったものは残さない／ソースを読めば分かることは書かない」）。
- **完了済みサイクル文書の扱い**: 結論は各サイクル文書本文に内包されている。使い捨てファイルへの歴史的言及まで遡って一括書き換えはしない（記録改ざんに近く価値がない）。ただし**価値ある研究ファイルを保全した場合の、そのファイルへのパス修正は正当な訂正**として行う（ファイルが移動した事実への追従）。
- **B-491 の領分に踏み込まない**: cycle-219 の計測スクリプト/results.json が tmp にある問題（③-8）は、「再現必須成果物を tmp に置かないルール化」＝B-491 の対象。本サイクルは既存の参照解消と知見保全に限定し、再発防止ルールの新設は B-491 に委ねる。
- **事実は着手時に再確認**: 調査レポートの「ファイル現存/消失」「保全推奨」は出発点。各 builder は着手時に対象ファイルを実際に開き、現状（現存・内容・既に docs に同等物がないか）を確認してから処理する。

### 検討した他の選択肢と判断理由

- **全 tmp 参照（cycle docs 677 行含む）を機械的に削除/書き換える案**: ルール Don't を字面どおり適用すれば全件違反だが、完了済みサイクル文書の使い捨てファイルへの歴史的言及は「封印済み作業記録」であり、結論は本文にある。遡って一括書き換えしても知見保全にも参照健全性にも寄与せず、記録改ざんに近い。よって「価値ある知見の保全」と「将来のリンク切れ防止」という**ルールの趣旨**に立ち返り、保全対象と前向きな参照解消に絞る（cycle-221 も自サイクルの研究根拠のみ docs/research へ移動し全履歴は遡及していない＝同じ判断の前例）。
- **substantive tmp 研究ファイル約 30 件をすべて docs/archive へ移す案**: それらの大半は cycle-200〜219（立て直しで作り直す失敗サイクル群）の計画根拠で、立て直しに上書きされ前向き価値が乏しい。全件アーカイブは doc-directory ルール「古くなったものは残さない」に反し `docs/` を汚す。**前向き価値の有無で個別判定**し、本線に役立つもの・再利用可能な基準のみ保全する。
- **保全を copy でなく mv（tmp から削除）する案 / tmp の使い捨てファイル 47 件を一括削除する案**: tmp は git 管理外で放置でいずれ消えるため、削除は本タスクの本質（保全と参照解消）でない。保全は「docs に内容を作り git add する」ことが要で、tmp 原本の削除は任意。使い捨てファイルの一括削除は B-487 のスコープ外（衛生作業）として扱わない。
