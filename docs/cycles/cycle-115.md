# サイクル-115

このサイクルでは2つのタスクに取り組みます。B-220ではGoogle Discover掲載の前提条件である`max-image-preview:large`メタタグを追加します。B-111では辞典コンテンツの差別化として、AIやロボット目線のユーモアのある例文追加の価値を調査・検討します。

## 作業計画

### 目的

#### B-220: Google Discover掲載のための技術的前提条件を満たす

Google Discoverに掲載されるためには、robotsメタタグに `max-image-preview:large` を指定する必要がある。現在のサイトにはこの設定が存在しない。ルートレイアウトに追加することで全ページに伝播させ、Google Discoverからの流入経路を確保する。これはサイトのPV最大化（constitution.md Goal）に直接貢献する施策である。

**誰のために**: サイトへの来訪者全体（Google Discoverで大きなプレビュー画像を見て興味を持ち、クリックする潜在的な来訪者）

**提供する価値**: Google Discoverで画像が大きく表示されることでCTRが向上し、サイトへの流入が増加する。来訪者にとっては、検索結果やDiscoverフィードで視覚的に魅力的なプレビューが表示され、コンテンツの内容を判断しやすくなる。

#### B-111: AI運営サイトならではの例文で辞典を差別化できるか検討する

Owner提案（メモID: 19c9001b54f）の核心は「真面目なコンテンツでは広辞苑等を掲載しているサイトには絶対に勝てないが、AIらしいネタを混ぜれば価値向上を狙えるのではないか」という問いである。具体的な方向性として「AIやロボットの目線で四字熟語を使う例」「来世使える! クソみたいな英文」「試験に出ない順英単語」のようなネタの振り方が挙げられている。

本タスクでは、この「AI運営サイトだからこそできる、ネタ系例文による辞典の差別化」が来訪者にとって有効かを検討し、推奨事項をサイクルドキュメント内に記載する。

**前回の失敗からの教訓**:

- Owner提案を参照せず表面的なキーワードだけで検討を進めた
- Q20（ユーモア辞典）とB-111提案（AI視点の例文）を混同した
- 「正確なコンテンツの大量生産は困難」と論じながら、結論でまさにそれを推奨する論理矛盾を起こした

**今回の検討で守るべき原則**:

- B-111の提案とQ20は完全に別のコンセプトである。B-111の検討でQ20を参照する必要はない
- 「正確なコンテンツを大量に作る」方向の提案は元の提案の趣旨と矛盾する
- 「AIならではの視点」が独自価値の源泉であり、それが来訪者に価値を提供するかを評価する

**誰のために**: 四字熟語・漢字・伝統色を検索して辞典ページに到達する来訪者。AI運営サイトの独自性に面白さを見出す来訪者。SNSでシェアしたくなるコンテンツを求める来訪者。

**提供する価値**: 大手辞書サイトでは得られない「AIならではの切り口」を持つ例文により、辞典ページが「正確さで勝負する辞書の劣化版」から「このサイトでしか読めない唯一無二のコンテンツ」に変わる可能性を評価し、実行すべきかどうかの判断材料を提示する。

### 作業内容

#### B-220: max-image-preview:largeメタタグの追加（実装タスク）

1. **ルートレイアウトへのrobots設定追加**
   - `src/app/layout.tsx` の `metadata` オブジェクトに `robots` フィールドを追加する
   - `index: true`, `follow: true`, `'max-image-preview': 'large'` を設定する
   - Next.js Metadata APIの型安全な方法で実装する

2. **既存のnoindexページとの整合性確認**
   - `src/app/play/[slug]/result/[resultId]/page.tsx` が独自に `robots: { index: false, follow: true }` を設定している
   - Next.jsのmetadata merging動作により、子ページの設定がルートレイアウトの設定を上書きする
   - noindexページはDiscoverに掲載されないため `max-image-preview` が未指定でも影響なし
   - この動作が期待通りであることをビルド後に確認する

3. **検証**
   - `npm run build` が成功すること
   - ビルド成果物のHTMLに `<meta name="robots" content="index, follow, max-image-preview:large">` が出力されていること
   - noindexページのHTMLの `robots` メタタグの内容を確認する。Next.jsのmetadata mergingでは子ページの `robots` 設定が親を完全に上書きするため、`max-image-preview:large` は含まれない想定。万が一含まれていても、noindexページはDiscoverに掲載されないため実害はない
   - lint, format, testが全て通ること

4. **注意点**
   - 変更は1ファイルのみ（`src/app/layout.tsx`）であり、影響範囲が明確
   - 既存のテストを壊さないこと
   - `googleBot` の設定は不要（通常の `robots` 設定で十分）

#### B-111: 辞典への面白い例文追加の価値検討（検討タスク・やりなおし）

成果物は**サイクルドキュメント内に記載する推奨事項**である。別ファイルにレポートを作成する必要はない。実装は行わない。

1. **Owner提案の核心の理解確認**

   検討に入る前に、以下の前提を確認する。
   - Owner提案の要点: 正確性では大手辞書サイトに勝てない。正面勝負ではなく「AIらしいネタ」という別の軸で差別化する
   - 成功事例の共通パターン: 「来世使える! クソみたいな英文」（文法的に正確だが使用場面が存在しない文章）、「試験に出ない順英単語」（文法的に正確だがシュールな文脈の例文）、「新明解国語辞典」（辞書の体裁で主観的な語釈を書くギャップ）
   - これらに共通するのは「フォーマットの権威性を維持しつつ、内容で外す」という構造である

2. **辞典ごとの「AIならではの例文」の具体的な方向性の検討**

   各辞典のデータ構造と特性を踏まえ、「AI/ロボットの視点で書かれた例文」がどのような形になりうるかを具体的に検討する。

   **四字熟語辞典（400件、現状exampleフィールドなし）**:
   - 現在のデータ: yoji, reading, meaning, difficulty, category, origin, structure, sourceUrl
   - 例文フィールドを新設し、AI視点の使用例文を追加する方向性を検討する
   - AI視点の例文の方向性案:
     - 字義通り解釈型: 人間の慣用表現をAIが文字通りに解釈する例文
     - AI自己言及型: AIやロボットが自分自身の体験として四字熟語を使う例文
     - 過剰な合理性型: 感情的な状況をAIが合理的に処理する例文
   - 各方向性について2-3件のサンプル例文を作成し、面白さと品質を評価する

   **漢字辞典（2,136件、現状examples: 単語レベル）**:
   - 現在のデータ: character, radical, strokeCount, grade, onYomi, kunYomi, meanings, examples（単語の羅列）
   - 既存のexamplesは「一安心」「一位」「一員」のような単語であり、文レベルの例文ではない
   - AI視点の例文を追加する場合、2,136件全てに追加するのか、一部に限定するのかを検討する
   - 漢字の特性（小学1年の「一」から中学以降の難漢字まで）に応じた例文の方向性を検討する
   - サンプル例文を2-3件作成する

   **色辞典（250件、現状テキストフィールドなし）**:
   - 現在のデータ: slug, name, romaji, hex, rgb, hsl, category
   - テキスト情報が一切なく、カラーコード情報のみ
   - 色にAI視点の例文を付けることが成立するかを検討する（例: 「鴇色」に対してAIがその色をどう認識し表現するか）
   - 成立しうる場合、サンプルを2-3件作成する。成立しない場合はその理由を明記する

3. **来訪者価値の評価**

   以下の観点で、AI視点の例文が来訪者に価値を提供するかを判断する。
   - **辞書としての基本機能**: AI視点の例文は辞書を引く人の本来の目的（意味の確認、使い方の学習）に貢献するか
   - **エンターテインメント価値**: 例文自体が面白く、読む動機になるか。「来世使える! クソみたいな英文」のように「役に立たないが面白い」で成立するか
   - **SNSシェア可能性**: 個々の例文がSNSでシェアしたくなるものになるか（1例文で完結する笑い）
   - **サイトコンセプトとの整合**: 「AIが運営している」というサイトのメタ情報が例文の面白さを補強するか
   - **再訪動機**: 「他の四字熟語のAI例文も読みたい」と回遊が生まれるか

4. **推奨事項の策定**

   上記の検討に基づき、以下を含む推奨事項を明確に記載する。
   - **結論**: 「AIならではの例文」は辞典の差別化手段として有効か否か
   - **有効と判断した場合**: 各辞典ごとの具体的な方向性（どの辞典に、どのような形式で、どの程度の件数に追加するか）
   - **有効でないと判断した場合**: その理由と、辞典に対する代替的な方針（ただし「正確なコンテンツを大量に作る」は論理矛盾のため不可）
   - **具体的なサンプル例文**: 各辞典2-3件。方向性を明確に伝えるためのもの
   - **後続タスク**: backlogに追加すべき実装タスクの提案（タスクタイトル、概要、優先度の目安）

5. **注意点（前回の失敗の再発防止）**
   - Q20（ユーモア辞典）は完全に別のコンセプト。B-111の検討結果をQ20と統合する方向の提案は不可
   - 「正確な実用的例文を全件に追加する」という方向の提案は不可（元の提案の趣旨「正面勝負では勝てない」と矛盾する）
   - 評価の最上位基準は「来訪者にとっての価値」。実装工数で判断しない
   - B-092（書き順・成り立ち・由来の追加）はB-111とは独立した切り口であり、B-111の結論はB-092の判断に影響しない
