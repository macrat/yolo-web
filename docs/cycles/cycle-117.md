## 作業計画

### 目的

四字熟語辞典の全400件に「AI視点の使用例文」を追加し、他の辞典サイトにはない独自コンテンツを確立する。メインターゲット（手軽で面白い占い・診断を楽しみたい人）にはSNSでシェアしたくなる笑いを、サブターゲット（AIエンジニア）にはAIの動作原理をユーモラスに捉える知的楽しさを提供する。これにより辞典ページの滞在時間向上とSNS経由の流入増加を狙う。

### 作業内容

全体を7つのフェーズに分割し、順序通りに進める。各フェーズ内のタスクは依存関係がなければ並行して実行可能。

#### フェーズ1: 型定義とデータ基盤の準備

   - テストファイルの mockYoji にも example フィールドを追加する

#### フェーズ2: 例文生成（カテゴリ別10バッチ）

400件を10カテゴリ別に分割し、カテゴリごとに1つのサブエージェントを起動して例文を生成する。カテゴリ別に分ける理由は、同カテゴリ内で例文のトーンや題材が被らないよう制御するためである。

各バッチの件数:

- change（変化）: 37件
- conflict（対立・闘い）: 26件
- effort（努力）: 36件

- knowledge（知識）: 39件
- life（人生）: 58件
- nature（自然）: 30件
- negative（否定的）: 47件
- society（社会）: 46件
- virtue（美徳）: 48件

各サブエージェントへの指示に含める内容:

- 対象カテゴリの全四字熟語一覧（yoji, meaning, origin を含む）
- 3タイプ（字義通り解釈型、AI自己言及型、過剰な合理性型）の定義と具体例
- 品質基準（説明なしに笑いが伝わる、1件で完結する笑い、四字熟語の意味と正確に対応、フォーマットの権威性を維持しつつ内容で外す）
- 同カテゴリ内でのタイプ・題材の分散指示（同じパターンばかりにならないこと）
- 出力形式: JSONの配列（yoji と example のペア）を `tmp/examples-{category}.json` に出力

サブエージェントの出力は `tmp/` ディレクトリに一時保存する。

#### フェーズ3: 例文のレビューと修正（カテゴリ別10バッチ）

品質を確保するため、フェーズ2で生成した例文を別のサブエージェントでレビューする。生成と同じくカテゴリ別に10バッチで実行する。

レビュー観点:

- 四字熟語の意味と例文の対応が正確か
- 説明なしに笑いが伝わるか（解説が必要なものは不合格）
- 1件で完結しているか（前提知識が必要なものは不合格）
- 同カテゴリ内でタイプや題材が十分に分散しているか
- 不自然な日本語やAIっぽい定型表現がないか
- 長すぎないか（目安: 100文字以内が望ましいが、面白さのために必要なら超過可）

不合格の例文はレビューエージェント自身が修正案を出力し、修正後の `tmp/examples-{category}-reviewed.json` を生成する。

### 注意点

- **例文の品質が最重要**: 400件を一括生成すると品質にばらつきが出る。カテゴリ別に分割し、生成とレビューを分離することで品質を担保する
- **meaning と origin の正確な把握**: 例文生成時に各四字熟語の meaning を必ず参照し、意味と例文の対応を正確にする。origin（日本/中国/不明）も考慮し、出典に無理のない文脈を作る
- **3タイプの分散**: 400件全体で3タイプが偏らないよう、各カテゴリ内で概ね均等に分散させる
- **辞典の体裁維持**: UIは既存の `.section` スタイルに準拠し、辞典としての権威性を損なわないデザインにする。例文の面白さはテキストで表現し、UIで「ふざけている感」を出さない
- **型安全性の限界とその対策**: `yoji.ts` では `as YojiEntry[]` で型アサーションしているため、JSONデータに `example` フィールドが欠落していてもコンパイル時には検知できない。JSONデータに対する本当の安全策はフェーズ4のスクリプト検証とフェーズ7の定量チェックである。一方、テストの mockYoji はリテラルオブジェクトなので、必須フィールドの欠落はTypeScriptが検知する。UIでは `as` キャストの限界を考慮し、実行時ガード（falsy check）を入れる
- **JSONファイルのサイズ**: 400件に平均80文字の例文を追加しても約32KB増で、パフォーマンスへの影響は無視できる

### 検討した他の選択肢と判断理由

1. **exampleフィールドをオプショナル（`example?: string`）にする案**
   - 却下理由: 全400件に追加する計画であり、段階的追加の必要がない。必須にすることでテストのmockデータなどリテラルオブジェクトでの欠落をコンパイル時に検知できる。ただしJSONデータに対しては `as` キャストの制約で型チェックが効かないため、スクリプト検証で補完する

2. **例文を50件ずつ8バッチで分割する案**
   - 却下理由: カテゴリをまたいだバッチでは、同カテゴリ内での題材・タイプの重複制御が困難になる。カテゴリ別分割のほうが品質制御しやすい

3. **例文を別ファイル（yoji-examples.json）に分離する案**
   - 却下理由: YojiDetailコンポーネントで追加のデータ読み込みが必要になり、コードが複雑化する。32KB程度の増加はパフォーマンスに影響しないため、既存のyoji-data.jsonに統合する方がシンプル

4. **例文の3タイプを明示的にフィールド（exampleType）で管理する案**
   - 却下理由: ユーザーにとってタイプの区別は不要（面白ければよい）。開発側の管理のためだけにフィールドを増やすのは過剰。レビュー時にタイプの分散をチェックすれば十分

5. **UI表示位置をページ最上部（意味の直後）にする案**
   - 却下理由: 辞典としての基本情報（四字熟語、読み、意味、構成漢字）の流れを優先し、補足コンテンツとしての例文は構成漢字の後に配置する方が自然

## 補足事項

### owner指摘: 読者目線の欠如（重大な問題）

本サイクルで作成した400件のAI視点例文は、**ITエンジニアにしか理解できない専門用語（API、トークン、パラメータ、GPU、loss、エポック、リトライ、プロンプト等）に依存した内容** になっており、メインターゲット（手軽で面白い占い・診断を楽しみたい人）には全く理解できないものだった。

これは **constitution.mdへの完全な違反** である。constitution.mdは「来訪者にとっての価値」を最優先すると定めているにもかかわらず、本サイクルでは「AIとして何が書けるか」という作り手の視点だけで例文を設計し、**読者が何を求めているか、読者にとって何が面白いかを一切調査・検証しなかった**。

#### 何がダメだったのか

1. **読者目線での価値判断を完全に欠いていた**。B-111で定義された3タイプ（字義通り解釈型・AI自己言及型・過剰な合理性型）をそのまま採用し、「この例文を読む人は誰で、その人にとって面白いか」を一度も問わなかった。
2. **マーケティング調査を実施しなかった**。「一般受け（幅広い人に浅く刺さる）」vs「ITエンジニア特化（一部の人に深く刺さる）」のどちらがSNS拡散で効果的かという基本的な調査すら行わなかった。
3. **レビューが技術的正確性にしか着目していなかった**。10カテゴリ×レビュー＋最終レビューという手厚いレビュー体制を敷いたにもかかわらず、すべてのレビューが「四字熟語の意味との対応」「パターンの重複」「文字数」といった技術的観点のみで実施され、**「読者がこれを読んで面白いと思うか」「読者がこれを理解できるか」という最も重要な観点が完全に欠落していた**。
4. **読者が求めているものを無視して作られたコンテンツは全くの無価値である**。どれだけ技術的に正確で、パターンが分散していて、ビルドが通っていても、ターゲットユーザーに理解されなければ何の意味もない。

#### 本来何をしなければならなかったのか

1. **読者にとっての価値を徹底して調査すること**。例文を書く前に、ターゲットユーザー（`docs/targets/` で定義）が四字熟語の例文に何を求めているか、どのような例文ならSNSでシェアしたくなるかを調査すべきだった。類似の成功事例（新明解国語辞典、「来世使える英文」等）がどの層にウケているかの分析も必要だった。
2. **「一般受け vs 特定層特化」のマーケティング判断を先に行うこと**。幅広い人に浅く刺さる内容と、一部の人に深く刺さる内容のどちらがPV最大化に効果的かを調査し、その結果に基づいて例文の方向性を決定すべきだった。
3. **すべてのレビューを読者目線で実施すること**。レビュー観点の筆頭は「読者がこれを読んで面白いと思うか」「読者がこれを理解できるか」であるべきだった。技術的正確性やパターン分散は二の次である。読者にとっての価値がないコンテンツは、どれだけ技術的に完璧でも無価値である。
4. **読者にとって価値があるかどうかを徹底してレビューすること**。レビューの最終判断基準は常に「この例文はターゲットユーザーにとって面白いか、シェアしたくなるか」でなければならない。この基準を満たさない例文は、他のすべての基準を満たしていても不合格とすべきだった。

この反省を踏まえ、次サイクル（B-225）では例文の全面書き直しを実施する。書き直しにあたっては、まずマーケティング調査を行い、読者にとっての価値を明確にしてから着手する。
