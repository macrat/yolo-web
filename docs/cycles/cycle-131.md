## 作業計画

### 目的

AIエージェントやオーケストレーションに興味があるエンジニアに対して、検索結果からのクリック率向上と記事内での読者体験の改善を行う。具体的には以下の価値を提供する。

- **検索結果での発見可能性の向上**: ターゲットユーザーが検索しそうなキーワードを先頭に配置し、検索結果でのボールド表示とクリック率を改善する
- **モバイル環境での訴求力向上**: スマートフォンで表示が切れてもタイトル・descriptionの意味が通じるようにする
- **記事内での読者体験の改善**: 見出し構造の明確化、段落の読みやすさ向上、内部リンクの追加により、読者が得られる価値を最大化する

記事の核心的な価値（前提汚染のメカニズムと4つの診断の問い）は維持し、事実を変えない。slugも変更しない。

### 作業内容

#### タスク1: タイトルの改善

**現在の問題点**

- 37文字でスマートフォン表示の境界線上にあり、ダッシュ以降「AIマルチエージェントの前提汚染」が省略されるリスクがある
- 主要キーワード（AIマルチエージェント、AIエージェント等）がタイトル後半にあり、SEO上不利
- 「前提汚染」はこのプロジェクト独自の造語であり、検索キーワードとして機能しにくい

**改善方針**

- ターゲットは「AIエージェントやオーケストレーションに興味があるエンジニア」であり、専門用語の使用は問題ない。ただし造語（前提汚染）はタイトルでの優先度を下げる
- 好奇心ギャップ構造（「指示通りに動いたのに間違っていた」）は現タイトルの強みなので維持する
- 主要キーワードを先頭付近に配置する
- 28〜36文字（全角換算）の範囲を目指す

**改善案**（最終的なタイトルはレビューを経て決定する）

- 案A: 「AIマルチエージェントの落とし穴 — 指示通りに動いたのに全員が間違えた理由」（38文字）
  - 長所: キーワード先頭配置、ネガティブワード（落とし穴）、好奇心ギャップ維持
  - 短所: 「前提汚染」という記事固有の概念が消える。38文字で目標範囲（28-36文字）を超えており、モバイル表示で後半が切れるリスクあり
- 案B: 「【失敗事例】AIマルチエージェントで全員が正しく動いても間違える構造」（34文字）
  - 長所: ブラケット使用（CTR 38%向上の調査あり）、キーワード先頭、構造的問題を示唆、34文字で目標範囲内
  - 短所: 「失敗事例」がやや一般的
- 案C: 「AIエージェントが正しく動いて間違える — マルチエージェントの前提汚染とは」（38文字）
  - 長所: キーワード先頭、好奇心ギャップ維持、「前提汚染」を残しつつ「とは」で説明記事であることを示唆
  - 短所: 「前提汚染」の検索ボリュームが小さいため効果は限定的。38文字で目標範囲を超えており、モバイル表示で後半が切れるリスクあり

**推奨案**: 案B を推奨する。選定基準の優先順位と判断理由は以下の通り:

1. **目標文字数の範囲内であること**（最優先）: 案Bのみが34文字で目標範囲（28-36文字）内。案A・Cは38文字で範囲外
2. **主要キーワードの先頭配置**: 3案とも満たしている
3. **好奇心ギャップ・CTR向上要素**: 案Bはブラケット【失敗事例】によるCTR向上効果がある。「失敗事例」がやや一般的である点はレビューで最終判断する

**注意事項**

- slugは変更しない
- タイトルと記事内容の一致を維持する（クリックベイトにしない）
- 好奇心ギャップの構造は必ず維持する

#### タスク2: descriptionの改善

**現在の問題点**

- 136文字でPC表示には適切だが、スマートフォンでは後半（70文字以降）がカットされる
- カットされた場合、「前提汚染のメカニズムと4つの診断の問いを実例から解説する」という記事の価値提示部分が見えなくなる
- 「マルチエージェントでオーケストレーターの前提が誤ると」の部分は技術用語が連続し、やや取っつきにくい

**改善方針**

- 冒頭50文字に最重要情報を凝縮し、モバイルで切れても意味が通じるようにする
- 反直感的な冒頭（「全員が正しく作業したのに結果が間違っていた」）は強力なフックなので維持する
- 記事の核心的価値（メカニズムの解説 + 4つの診断の問い）を必ず含める
- 120文字前後を目標とする（PC表示で全文見える範囲）

**改善案**

- 案A（短縮版・51文字）: 「全員が正しく動いたのに結果が間違っていた。前提汚染のメカニズムと、4つの診断の問いを実例から解説する。」
  - 長所: モバイルでもほぼ全文表示される、簡潔
  - 短所: 「マルチエージェント」キーワードが入らない。SEO観点では主要キーワード不在により、検索結果でのボールド表示が減り、クリック率が低下するリスクがある。ただし、タイトル側に「マルチエージェント」が含まれていれば、descriptionに無くても致命的ではない
- 案B（バランス版・77文字）: 「全員が正しく動いたのに結果が間違っていた。AIマルチエージェントで起きる前提汚染のメカニズムと、自分のワークフローを診断する4つの問いを実例から解説する。」
  - 長所: 主要キーワード（AIマルチエージェント）を含む、読者へのベネフィット（自分のワークフローを診断できる）が明確
  - 短所: モバイルでは後半が切れるが、冒頭だけでも意味が通じる
- 案C（問題提起強化版・65文字）: 「全員が正しく動いたのに結果が間違っていた — その原因はオーケストレーターの前提にあった。前提汚染の構造と4つの診断の問いを解説。」
  - 長所: 原因への好奇心を喚起、ストーリー性がある
  - 短所: 「実例から」が省略される

**推奨案: 案B**

選定基準の優先順位:

1. 主要キーワード（AIマルチエージェント）の含有
2. 冒頭50文字で意味が完結すること
3. 読者へのベネフィットの明確さ

判断理由: 案Bは主要キーワード「AIマルチエージェント」を含みつつ、冒頭の反直感的なフック（「全員が正しく動いたのに結果が間違っていた」）で50文字以内に意味が完結する。77文字でモバイル表示では後半が切れる可能性があるが、技術記事の主要ターゲットはPC検索が多いエンジニアであり、PC表示では全文が表示される。案Aはキーワード不在のSEOリスクがあり、案Cは「オーケストレーター」という専門用語が冒頭近くに来るため取っつきにくい。

**注意事項**

- 記事内容との一致を保つ（Googleによるスニペット書き換えを防ぐ）
- 他の記事のdescriptionと構成が重複しないようにする

#### タスク3: 本文の改善

**現在の問題点と改善方針**

**(a) 見出し構造の改善**

- 現在の問題: H2見出しだけを読んだとき、「前提汚染のメカニズム」「前提はどう崩壊したか」は内容を想像しにくい。特に「前提はどう崩壊したか — ゼロベースレビューと外部データ」は長すぎる
- 改善方針: H2見出しだけで記事全体のストーリーが把握できるようにする。見出しに「何がわかるか」を含める
- 全H2見出しの改善方向性:
  - 「失敗の発端」 → 問題の具体性を高める方向。例: 「前任AIの定義を疑わなかった」のように、何が起きたかを示唆する
  - 「前提汚染のメカニズム」 → 読者にとっての発見を示唆する方向。例: 「なぜ全員が正しく動いても間違えるのか」のように、疑問形で好奇心を喚起する
  - 「前提はどう崩壊したか — ゼロベースレビューと外部データ」 → 短縮しつつ発見を示唆する方向。例: 「前提を破壊した2つの武器」のように、読者の関心を引く表現にする
  - 「自分のワークフローを診断する4つの問い」 → 現状でも読者へのベネフィットが明確であり、大きな変更は不要。必要に応じて微調整のみ
  - 「まとめ」 → 現状維持、または内容を示唆する形に微調整。例: 「学びと残された問い」など
- 具体的な文言は実装時に決定する

**(b) 段落の読みやすさ**

- 現在の問題: いくつかの段落が長く（5文以上）、特に「前提を受け継いだまま実装した」セクション内の段落が情報密度が高い
- 改善方針: 1段落を3-4文以内に分割する。1段落1アイデアの原則を徹底する
- 注意: 事実は変えない。文の分割・再構成のみ行う

**(c) 太字の適切な使用**

- 現在の問題: 各セクションの冒頭に太字で要約文があるのは良い構造だが、一部の太字が強調としてではなくトーン変更のために使われている箇所がある
- 改善方針: 太字は「スキャンしたときに最重要情報を拾える」目的に限定する。全体の30%以下を維持する

**(d) 内部リンクの追加**

- 現在の問題: 記事内に内部リンクが1箇所（思考バイアスとコンテキストエンジニアリングの記事へのリンク）しかない
- 改善方針: 同シリーズ（ai-agent-ops）の関連記事への内部リンクを自然な文脈で2-3箇所追加する。アンカーテキストは具体的な内容を示すものにする（「こちら」「詳しくはこちら」は使わない）
- 具体的なリンク先候補（既にリンク済みの `ai-agent-concept-rethink-1-bias-and-context-engineering` を除く）:
  - `src/blog/content/2026-02-14-five-failures-and-lessons-from-ai-agents.md`（slug: `five-failures-and-lessons-from-ai-agents`）: 「AIエージェント運用で遭遇した5つの失敗と解決策」。記事内の失敗事例の文脈で「他にもAIエージェント運用で遭遇した失敗事例」としてリンク可能
  - `src/blog/content/2026-03-19-ai-agent-verification-step-skip.md`（slug: `ai-agent-verification-step-skip`）: 「AIエージェントは「最後の確認」を省略する」。「レビューが機能しなかった理由」セクションの文脈で、検証ステップが省略される構造的問題としてリンク可能
  - `src/blog/content/2026-02-23-workflow-skill-based-autonomous-operation.md`（slug: `workflow-skill-based-autonomous-operation`）: 「AIエージェントを4つのスキルで自律運用する」。レビューループの設計に言及しており、「自分のワークフローを診断する4つの問い」セクションでワークフロー設計の参考としてリンク可能
- 上記3件から文脈に最も合う2-3件を選んで追加する。リンク先の記事の内容を確認し、文脈に合ったリンクのみ追加する

**(e) 導入部の改善**

- 現在の問題: 導入部は好奇心ギャップを使った優れた構造だが、「この記事で読者が得られるもの」のリストがやや形式的で、読者の動機付けが弱い
- 改善方針: 読者が「自分のワークフローでも起きうる問題だ」と感じる共感ポイントを強化する。ただし、内容を変えるのではなく表現を改善する

**(f) 導入部の「冒頭の約束」回収チェック**

- 導入部の「この記事で読者が得られるもの」に記載されている以下の3項目が、本文中で明確に回収されているかを確認する:
  1. 「前提汚染のメカニズムの理解（なぜ全員が正しく動いても間違った結果になるのか）」 → 「前提汚染のメカニズム」セクションで回収されているか
  2. 「なぜレビューがあっても前提汚染を検出できないのかという構造的な説明」 → 「レビューが機能しなかった理由」セクションで回収されているか
  3. 「自分のマルチエージェント構成で同じことが起きていないか診断するための4つの問い」 → 「自分のワークフローを診断する4つの問い」セクションで回収されているか
- 回収が不十分な項目がある場合は、本文側の該当箇所を補強する（事実は変えずに、約束した内容との対応を明確にする）

**(g) updated_at の更新**

- blog-writing.mdのルールに従い、コミット直前に `date` コマンドで取得した現在時刻で updated_at を更新する

**(h) 目次について**

- サイトにはTableOfContentsコンポーネントが既に実装されており、H2/H3見出しから目次が自動生成されるため、目次に関する追加対応は不要である

**注意事項**

- 記事の事実や主張は一切変更しない
- 記事の構成（セクション順序）は維持する。大幅な構成変更は行わない
- 内部用語（サイクル番号、タスクID等）を新たに追加しない（現状既にこのルールは守られている）

### 検討した他の選択肢と判断理由

1. **タイトルから好奇心ギャップを除去してキーワード中心にする案**
   - 例: 「AIマルチエージェントの前提汚染 — メカニズムと診断方法」
   - 却下理由: 好奇心ギャップは現タイトルの最大の強み。調査でも逆説・矛盾構造は技術記事で特に有効とされており、除去するとCTR低下のリスクが高い

2. **descriptionを70文字以下に短縮してモバイル完全対応にする案**
   - 却下理由: PC検索が主要流入元であるエンジニア向け技術記事では、PC表示での情報量を犠牲にするデメリットが大きい。冒頭50文字で意味が通じる構成にすることで、モバイルとPCの両方に対応できる

3. **記事構成を大幅に変更する案（セクション順の入れ替え等）**
   - 却下理由: 現在の構成は「問題提起 → メカニズム → 解決策 → 読者への応用」という論理的な流れで、読者の理解を段階的に深める構造になっている。構成変更はリスクが大きく、改善効果は見出し・段落レベルの改善で十分に得られる

4. **「前提汚染」という造語をタイトルから完全に削除する案**
   - 却下理由: 「前提汚染」は記事の核心概念であり、独自性の源泉でもある。タイトルから完全に消すと記事のアイデンティティが弱まる。ただし、タイトル案によっては「前提汚染」を含めない選択も許容範囲とし、最終判断はレビューで行う
