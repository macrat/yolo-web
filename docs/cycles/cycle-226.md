# サイクル-226

このサイクルでは、cycle-220 → cycle-225 と**二度連続で未達に終わった「全ツールの DESIGN.md §1（パネル）準拠」**を、今度こそ達成する。

cycle-225 の事故報告（`docs/cycles/cycle-225.md` 末尾）が確定させた事実は明快だ——全34ツールの最上位レンダー要素は `<div className={styles.container}>` のままで、Panel コンポーネントを import しているツールページは **0/34**。DESIGN.md §1 は「すべてのコンテンツはパネルに収まった形で提供される」と恒久要件として定めているのに、それが2サイクル連続で指示書から滑り落ちた。

二連敗の根因は「規律と良心で守る」アプローチに留まったこと。だから今サイクルは**構造で守らせる**：

1. **B-497** — `<Tile>` コンポーネントを新設し、内部で必ず `<Panel>` をラップする。各ツールが `<Tile mode="page">` を使えば（または `ToolPageLayout` の側で機構的にラップすれば）、Panel 準拠が「使うだけで保証される」状態にする。型契約や lint ではなくコンポーネント設計で「タイル＝パネル」の同一性をコードに具現化する。
2. **B-499** — Playwright で全34ツールに navigate し、Panel 描画・computed background・border-radius・box-shadow を機械的に assert するゲートを新設する。PM の独立検証が lint/format/test/build しか見ていなかった穴を塞ぎ、**書類でなく機械の戻り値（exit 0/1）で本来の目的達成を判定する**。

この2つは分離不能だ。B-497 の完了条件そのものが「B-499 のスクリプトが exit 0 を返すこと」であり、B-499 は「B-497 完了時に同時に追加する」もの。だから1サイクルで一緒に達成する。

最終受益者は来訪者（`docs/targets/`）。全ツールが一貫した「パネルに収まった道具」として揃うことで、探す→使う→また使うの体験の見た目とトンマナが本当に統一される。cycle-225 までで機能・単一実装・a11y・コピーは揃ったが、**見た目の土台（パネル）だけが欠けたまま**だった。それを埋めるのが本サイクル。

## スコープと前提

- **対象は34ツールのページ**（cycle-225 で単一実装に統合済み）。遊び（ゲーム）はスコープ外（別レイアウト・cycle-225 と同じ線引き）。
- **B-497 と B-499 を同一サイクルで実施**（完了条件が相互依存・分離不能）。
- cycle-225 末尾「次サイクルへの設計制約（Tile コンポーネント）」が、文脈なしで読んでも実行可能な形で設計制約・参照点・現状事実・完了判定を固定している。本サイクルはそれを起点にする。
- **B-498（PM 指示文プロセス改善）は本サイクルのスコープ外**（Queued 据え置き）。理由: B-499 の機械ゲートが今サイクル内での Panel 準拠を構造的に保証するため、今サイクルの失敗防止には B-499 で足りる。B-498 は将来サイクルでの要件脱落を防ぐ上流予防策であり独立して着手できる。最小スコープを守る。
- **規律ベースが2連敗した教訓**: サブエージェントの「DESIGN 準拠 OK」自己申告を一次資料照合なしに受理しない。PM が DESIGN.md を一次資料として実機（Playwright・computed style）と独立照合する step を必ず持つ（B-499 がその機械化）。
