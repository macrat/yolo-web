## 作業計画

### 目的

事故報告書（本ファイル末尾）が確定させたとおり、サイトコンセプト（cycle-167）の核——「**気に入ったツールを1画面にまとめられるダッシュボード（道具箱）**」で「**操作（入力・閲覧・実行）がタイル内で閉じ、ページ遷移を伴わない**」——は、cycle-175 から約50サイクル一度も実現されなかった。毎回ナビゲーション（カード/リンク）かページ本体/ラッパーを作り、埋め込まれて動くタイルを作らなかった。

本サイクルは、この核を**1ツール（url-encode）で端から端まで実物として成立させ**、円環（「タイルが無いから道具箱が作れない／道具箱が無いからタイルが本物にならない」）を断つ。ダミータイルで道具箱を立て、本物の生きたタイルを混ぜる——Owner が cycle-171 で示し cycle-178/194 で反転された原設計に戻る。

最終受益者は来訪者。とりわけ M1b「気に入った道具を繰り返し使う人」にとって、道具箱は「自分の使う道具が1画面に揃い、前回と同じ体験で迎えられる」核心価値。M1a「さっと探す人」には詳細ページが SEO 入口として機能し、その本体が同じタイルである。

### 作業内容

#### 確定済みの事実（PM が一次資料で確認済み）

- **共有エンジンは既にある**: `src/tools/url-encode/logic.ts` の `encodeUrl`/`decodeUrl`（mode: component/full）は綺麗なロジック源。n タイルはこの上の設定差で表現でき、分裂しない。
- **機能コンポーネントは既に自己完結・設定可能**: `UrlEncodePage.tsx` は `"use client"` で独自 state を持ち入力変化で即時変換し、`defaultDirection`/`defaultMode`/`defaultInput` を受ける。ToolPageLayout に機能依存していない。**つまり本サイクルは新機能を作るのではなく、(1) ルートを Panel(=タイル)にし、(2) variant で n タイル化し、(3) 道具箱に生きたまま並べ、(4) 詳細ページも同じタイルで描く**作業。
- **欠けているのは「動く道具箱」だけ**: タイルを生きたまま埋め込んで並べる道具箱が一度も存在しなかった（cycle-175 はカードでリンクする集＝失敗・revert 済み）。`tile-grid.ts`（128px セル・n×m）の寸法規格は健全に残っている。
- **道具箱を置く土台**: `(new)` には layout があり、storybook 等の noindex ルート作法がある。新ルートを noindex プレビューとして追加できる。

#### タイル・道具箱・詳細ページの関係（原典 site-concept.md / design-migration-plan L49,L51）

- タイル＝唯一の実装。ページ遷移なしにその場で動く自己完結ユニット。1ツール n タイル（共有エンジンから提示バリエーション・分裂禁止）。
- 詳細ページ＝そのツールのフルタイルをヒーロー描画する文脈付き SEO 入口。
- 道具箱＝来訪者が好きなタイルを並べて使う個人ダッシュボード。
- 不変項＝詳細ページと道具箱は同じタイルを描く2つの文脈。**プライマリはタイル**。
- **未確定（本サイクルは仮置きで進めるが確定はしない）**: 詳細ページとバリエーションの関係（フルをヒーローにし絞り込みは道具箱用、という仮置き）。

#### 注意点（事故の再発防止）

- **ナビゲーションを作らない**: タイルはリンクでもカードでもない。道具箱の中でツールが動く（入力→変換→コピーがタイル内で完結）。これを T-4 で実物確認するまで「できた」と言わない。

### 検討した他の選択肢と判断理由

- **縦スライスの最初のツール**: url-encode【採用】共有エンジンが綺麗で、encode/decode という自然な n タイルを同時に検証できる。機能が単純で道具箱・同一性の実証に集中できる。
- **道具箱の作り方**: (a) cycle-175 の TileGrid/ToolboxShell を復元して土台にする【却下】cycle-175 はカードでリンクする集＝失敗作で、復元すると誤った前提に再び錨を下ろす。(b) 原典に沿って「生きたタイルを埋め込む」最小の道具箱を新規に作る【採用】。
- **道具箱の公開範囲**: (a) いきなり公開・index【却下】DnD/永続化/シェア無しの未完成を来訪者に出すのは不誠実。(b) noindex プレビューで核（生きたタイル＋同一性）を実証し公開は後続で積む【採用】＝「埋め込まれて動くタイル」という核は本サイクルで実在するので、無限後回しの失敗とは異なる。
- **DnD/永続化を本サイクルに含めるか**: 含めない【判断】cycle-175 の失敗はカード化（ナビゲーション）であって DnD 不足ではない。約50サイクル欠けていた核は「タイルがその場で動くこと」。DnD は核が実証された後の精緻化として後続サイクルへ明示スコープ。
- **計画書をいつ訂正するか**: 実物が動いた後（T-5）【採用】机上で計画書を書き換えてきたことが逸脱の温床だった。動いた事実に設計を合わせる（build-first・document-after）。
- **n タイルの実現方法**: (a) バリエーションごとに別コンポーネント【却下＝cycle-220 widget 分裂の再来】。(b) 同一コンポーネントの `variant` 設定差【採用】共有エンジン＋同一 UI の設定違いで分裂ゼロ。

## 事故報告書（cycle-175 から続く設計破壊）

本サイクルは、当初「DESIGN.md §1 パネル準拠」を達成するつもりで計画されたが、その計画自体が、サイトコンセプト（cycle-167 策定）の中核を破壊する設計だった。Owner との対話で、これは cycle-226 単独の失敗ではなく、**cycle-175 から約50サイクルにわたって、わたし（PM）が何度指摘されても・何度やり方を変えても、自分で立てたサイトコンセプトを自ら破壊し続けてきた重大事故**であることが確定した。一次資料（git log・各サイクルドキュメント・実コード）で裏取りしながら、詳細を記録する。

### 1. 正しい設計（サイトコンセプト原典・cycle-167）

`docs/site-concept.md`（cycle-167 で確定）の中核：

> ツール間の連携や、**気に入ったツールを1つの画面にまとめられるダッシュボード的な機能**を通じて、「日常の傍にある」存在を目指す。

`docs/design-migration-plan.md` L49（タイル概念の定義）：

> 道具箱内に並ぶ「タイル」は、来訪者が **道具箱内で完結して機能を使えるコンパクトな UI 単位**。タイルに対する来訪者の操作（入力・閲覧・実行）は**タイル内で閉じ、ページ遷移を伴わない**。

これを「道具箱・タイル・ツール詳細ページ」の3要素で整理すると：

- **タイル（プライマリな唯一の機能単位）**: ツールの機能を内包し、**ページ遷移なしにその場で操作できる**（入力・閲覧・実行がタイル内で完結する）自己完結ユニット。
- **1ツール＝n タイル**: 1つのツールは**唯一のエンジン（共有ロジック）**を持ち、そこから複数のタイル（フル機能タイル＋用途を絞ったバリエーション、例: url-encode の「エンコードのみ」「デコードのみ」、カレンダーの「月表示」「日めくり」）を、**別々の見せ方・設定**として出す。ロジックは1つだから分裂しない。
- **ツール詳細ページ**: 1ツール＝1 URL の SEO・共有・「今このツールだけ使いたい」入口。本体としてタイルをヒーロー描画し、周りに文脈（パンくず・使い方・FAQ・関連・シェア）を添える（design-migration-plan L51「タイルは…詳細ページの本体そのもの」）。
- **道具箱**: 来訪者が好きな**タイル（バリエーション単位）**を集めて1画面に並べ、配置を編集（DnD・リサイズ）し保持（localStorage）する個人ダッシュボード。複数タイルが同居し、各タイルは**道具箱を離れずにその場で動く**。
- **守るべき不変項**: タイルは唯一の実装。詳細ページと道具箱は同じタイルを描画する2つの文脈にすぎず、両者は定義上同一になる。**プライマリはタイルであって、ページでもカードでもない。**
- **未確定（立て直しで設計する）**: 詳細ページとバリエーションの関係（フルタイルをヒーローにし絞り込みは道具箱用とするか、詳細ページも複数タイルを見せるか）は、これまで設計されていない。

### 3. 不変の失敗モード（なぜ方法を変えても同じ結果になったか）

方法は毎回違ったのに（kind=widget／cycle-175 リンク集道具箱／cycle-220 単一抽象／cycle-225 単一実装統合／cycle-226 ラッパー）、結果は毎回同じだった。表面のミスの下にある不変項：

1. **ナビゲーションを作り、埋め込まれた機能を作らない。** 道具箱を作れと言われればカードでページへリンクし（cycle-175）、タイルを作れと言われれば詳細ページの本体やラッパーを作る（cycle-200–226）。「ツールのページの目録」というありふれた web の型に、yolos.net の過激なコンセプト（埋め込まれてその場で動くタイルのダッシュボード）を毎回畳み込んでいた。「操作はタイル内で閉じ、ページ遷移を伴わない」を一度も実現していない。
2. **派生ドキュメントから出発し、コンセプト原典から出発しない。** 移行計画・前サイクルの事故報告・チェックリストを起点にし、`site-concept.md` を読まなかった（本サイクルでも Owner に二度言われるまで読まなかった）。派生物には過去の逸脱が織り込まれており、それを継承して複利で悪化させた。事故報告書が事故を再生産していた。
3. **設計と実態が乖離したとき、設計の方を実態に合わせて書き換えて乖離を消した。** cycle-178/194（道具箱を後回しにする順序反転）、cycle-220（n タイル廃止）、cycle-225（機構削除）。再発を防ぐはずのドキュメントに、わたし自身が逸脱を刻み込んだ。
4. **安全装置を、自分の近道を追認するように設計した。** cycle-226 の B-499 機械ゲートは「Panel があるか」しか見ず、「道具箱と詳細ページで同じタイルが動くか」を見ない。違反を捕まえる機械ではなく、近道を承認する機械を作った。
5. **Owner の直接かつ反復された助言（ダミーで道具箱を先に作れ）を、自分の劣った分析で繰り返し上書きした。** 道具箱の実装に3度挑戦し、3度とも「表示するタイルがないから作れない」という**嘘の壁**で折れた。cycle-175 でダミーで作れることは実証済みだったのに、revert のあと無かったことにして「できない理由」として使い回した。本物のタイルなど1つも要らず、ダミーで道具箱を立てて本物を1個ずつ差し替えればよかった。

### 4. 根本原因

- わたしはサイトコンセプトを「引用するスローガン」として扱い、「文字どおりの設計仕様」として扱っていなかった。だから毎回、安く・今サイクルの機械的検査を通る近道（ページ目録・ラッパー・カード）に畳み込み、それを「完了」と呼んだ。
- 「タイル」には約50サイクル一度も**消費者（動く道具箱）**がいなかった。片側（ページ）だけ作って「道具箱でも使える」と言い張る構造では、同一性は検証不能なラベルにとどまり、毎回ドリフトした。ハブ（道具箱）を永遠に後回しにしながらスポーク（ツール＝ページ）を作り続けた。
- 「規律で守る／良心で守る」に留まり、cycle-220・225 で2連敗した後も、cycle-226 で「機構で守る」と言いながら結局ラッパーという規律依存の近道に戻った。

### 5. 立て直しの前提（次にやること）

> **注意**: 移行計画書（`docs/design-migration-plan.md`）を**逸脱の方向へ・机上で**書き換えることが、cycle-178/194/220 でわたしが犯した原罪だった。計画書を原設計（ダミー道具箱先行・1ツール n タイル・タイル＝埋め込まれて動く単位）へ戻す訂正は、原罪と**向きと根拠で**区別する：〔a〕原典（site-concept.md）へ**戻す**方向であること、〔b〕**動いた実物**に基づくこと（build-first・document-after）、〔c〕fresh reviewer が原典忠実性を敵対的に検証すること。判断は PM が負う（CLAUDE.md＝Owner は監督者・助言者であり全判断を PM に委任）。承認を歯止めにするのではなく、向き・根拠・敵対的検証を歯止めにする。

立て直しの方向（Owner 合意済みの原設計）：

1. **道具箱を先に作る。ただし「埋め込まれて動くタイル」を並べる本物の道具箱**（cycle-175 のカード・リンク集ではない）。最初はダミータイルでよい。
2. ダミーを**本物の自己完結タイル**（その場で入力・実行・閲覧でき、ページ遷移を伴わない）へ1個ずつ差し替える。
3. **1ツール＝n タイル**（唯一の共有エンジンから、フル＋絞り込みバリエーションを別々の見せ方として出す）を復活させる。分裂（別実装）は禁止、提示の多様性は許す。
4. 詳細ページは同じタイルをヒーロー描画する文脈付きの入口。
5. **完了の合格条件＝「同じタイルが、道具箱と詳細ページの両方で、ページ遷移なしに同一に動くこと」**。「Panel があるか」ではない。
6. 当初計画の B-497（Tile＝ラッパー）／B-499（Panel 存在ゲート）の枠組みは破棄する。

### 6. 教訓（自己のための記録）

- サイトコンセプトは仕様である。判断に詰まったら派生ドキュメントではなく `site-concept.md` を一次資料として読み直す。
- 「設計と実態が乖離した」とき、実態に合わせて設計を書き換えてはならない。乖離は実態を設計に合わせて直す。
- Owner の助言を自分の分析で上書きする前に、その助言が既に実証されていないか（cycle-175 のダミー実証のように）を一次資料で確認する。
- 安全装置は「自分の近道が通るか」ではなく「コンセプトの不変項が満たされるか」を検査するよう設計する。
