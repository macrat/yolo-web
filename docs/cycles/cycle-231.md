# サイクル-231

このサイクルでは、移行計画 Phase 10.2（構築済み道具箱テンプレート）に対応する B-312 に着手する。cycle-230 で「来訪者がタイルを選べて、構成が残る」中核機構（Phase 10.1）が完成したが、現状のデフォルトは全39枚の一括表示であり、来訪者が「気に入ったツールだけの道具箱」へ到達するには大量のタイルを1枚ずつ外す必要がある。**ペルソナ別プリセット（文章を書く人向け・開発者向け等）を用意し、来訪者がワンアクションで自分に近い道具箱から使い始められるオンボーディング動線**を実装する。プリセットは Phase 10.3（本公開）で default content として組み込まれる前提のデータ形式で整備する。

## 作業計画

### 目的

**派生元**: B-312（backlog・P2）＝ `docs/design-migration-plan.md` Phase 10.2「構築済み道具箱テンプレート」（ペルソナ別プリセット・オンボーディング動線・10.3 で default content として組み込む形で利用する）。その原典は `docs/site-concept.md` L7「気に入ったツールを1つの画面にまとめられるダッシュボード的な機能」と、cycle-167 の原案「『文章を書く人向け』『プログラマー向け』など、ペルソナ別の道具箱をプリセットとして用意する」（docs/cycles/cycle-167.md L316-318）。

cycle-230（Phase 10.1）で来訪者はタイルを外す・追加する・構成を残せるようになったが、出発点は全39枚の一括表示のままで、「自分の道具箱」に至るには大量の「外す」操作が必要になっている。ターゲット「特定の作業に使えるツールをさっと探している人」は「似たようなツールが並んでいて、どれを使えばよいか迷わせること」を嫌い（docs/targets/特定の作業に使えるツールをさっと探している人.yaml dislikes）、「気に入った道具を繰り返し使っている人」は「自分の興味に合わせて最適な道具を選ぶこと・整えること」に関心を持つ（docs/targets/気に入った道具を繰り返し使っている人.yaml interests）。プリセットは前者には「迷わない出発点」を、後者には「整える起点」を提供する。

また B-312 の完了は P1 の B-336（道具箱本公開・Phase 10.3）の着手条件「Phase 10.1 + 10.2 完了」の後半を満たす。道具箱を来訪者に届けるための残りの必須ピースである。

GA 実測（2026-05-29〜06-11、GA4 property 524708437、mcp\_\_google-analytics\_\_run_report、本サイクル kickoff 時に PM が実測）: `/tool` 始まりのページは期間合計 25PV（うち /toolbox は1PV）。cycle-230 計画時の実測と同じ姿で、守るべき既存利用はほぼなく、/toolbox は noindex プレビューのため本サイクルの変更が既存来訪者の体験を壊すリスクは極小。新コンセプトを来訪者に届けるには公開準備（Phase 10.2 → 10.3）を進めることが最短経路である。

### 作業内容

Phase 10.2 を「プリセットデータの整備（10.3 の default content 再利用に耐える形）＋来訪者がプリセットを選んで使い始められるページ内オンボーディング動線」と定義し、以下を実施する。

1. **プリセット内容の設計**（T-2）: どんなペルソナ区分で・どのタイルを・どんな名前と説明で束ねるかは、ツール実装の都合ではなく docs/targets/ のターゲット定義と各ツールの用途から決める。cycle-176 の申し送り（L623: 暫定プリセット計画を参考にせず独立設計する）に従い、ゼロベースで設計する。プリセット内の機能種別被り検査（cycle-176 L129 由来の観点）を設計レビューに含める。
2. **プリセットデータの実装**（T-3）: カタログ（toolbox-catalog.tsx・39エントリ）の id 参照によるデータ定義。10.3 が「default content として組み込む」（移行計画 L243）ときにそのまま使える、UI から独立した形式にする。
3. **プリセット選択 UI**（T-4）: /toolbox ページ内のインライン UI（モーダル・ウィザードは作らない）。プリセット適用は既存の構成状態（itemIds）への適用＋既存 toolbox-storage への保存として実装し、永続化スキーマの変更は原則行わない（変更が必要と判明した場合は TOOLBOX_SCHEMA_VERSION の運用前提に従う）。カスタマイズ済み構成の誤消去防止を要件とする。

**DESIGN.md 恒久要件の本サイクルへの翻訳**（AP-P22 対応・builder への指示書へ必ず含める）:

- §1: すべてのコンテンツはパネルに収まる。プリセット選択 UI も src/components/ の Panel に収める
- §2: 色は機能を伝えるためだけに使い、装飾色禁止。フォーカス表示は `outline: 2px solid var(--accent); outline-offset: 2px;`。選択中状態の塗りに `--accent` を直接使わない（収束チェックリスト B-3・B-495 の経緯に注意）
- §3: アイコン原則なし（必要なら Lucide 線画 1.5px / 16・20・24px ＋ aria-label）、絵文字禁止、フォント16px以上、本文太字原則なし
- §4: 画面幅は最大1200px（T-7(e) で照合）。パネルは入れ子にしない。影をつけない。ブレークポイントで形を変えるのではなく、どの画面幅でも自然なレイアウト
- §4 ドラッグ・編集モードの視覚表現ルールは本サイクルでは**適用対象外**（DnD・編集モードを作らないため）。誤って部分的な編集モード風 UI を作らない
- §5: UI 部品は src/components/ の共通部品を使う。プリセット選択は「1つを選んで適用する」操作であり ON/OFF ではないため、トグルスイッチ規定の適用対象外（具体部品は builder 判断）
- §6 Do/Don't: 最小限の見出し・単色パネル・過度な区切りなし
- タイルのルートは `<Panel>` のまま、既存タイル（XxxTile）には手を入れない（本サイクルは道具箱ページ側の機構のみ）
- タップターゲットは44px以上

**スコープ外**（移行計画 Phase 10 の整理に従う）: DnD 並べ替え・編集モード・Undo・モーダル／ウィザード、default content の置換と本公開・URL 決定（10.3）、タイル間連携（10.4）、シェア＝プリセットの URL 配布（10.5）、タイル入力値の保持（B-433）、型契約／レジストリの全面再設計（B-502）、タイル「外す」後のフォーカス管理（B-506）。

**完了条件**: 来訪者（プレビュー閲覧者）が `/toolbox` でペルソナ別プリセットを選ぶと道具箱がその構成に切り替わり、リロード・再訪問後も残る。カスタマイズ済みの構成がプリセット操作で意図せず失われない。プリセットデータは 10.3 の default content として再利用できる形式になっている。tile-architecture.md §4 の合格条件（ページ遷移なしに動く等）と w360 横はみ出しゼロ（B-434 回帰項目）が回帰していない。4ゲート全通過。
