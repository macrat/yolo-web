## 作業計画

### 目的

新コンセプト「日常の傍にある道具」のもとで、新デザインに **横断検索**（ブログ・ツール・遊び・辞典が混ざって出る検索）を作るかどうかを二択（① 作る / ② 作らない）で確定させる。判断結果が後続フェーズ（B-331 Phase 5、B-334 Phase 4、B-335 Phase 6、B-340 計測コードの扱い）の着手スコープと、Phase 1.1 の前提として `docs/design-migration-plan.md` に固定される情報を決める。

このサイクルが来訪者に生む価値は、「移行期間中も移行完了後も、新コンセプトのターゲット（M1a 初回・M1b リピーター）の主要動線にとって最適な検索体験のかたちを確定させる」こと。逆に判断を保留すると後続全 Phase が進まず、新デザインが来訪者に届かない。

### 判断対象の正確な再確認

#### 二択の中身

- **①**: 新デザインの Header（または相当の常駐 UI）に、ブログ記事・ツール・遊び・辞典項目を **混ざって** 一括で返す検索を実装する（Phase 5 = B-331 を実施）。
- **②**: 新デザインに上記の混合検索を作らない（Phase 5 = B-331 をスキップ）。

#### 二択と独立に必ず維持される機能（判断対象外）

- ツール一覧 `/tools` 内のツール絞り込み（カテゴリ別検索）
- 遊び一覧 `/play` 内の絞り込み
- ブログ一覧 `/blog` のカテゴリ・タグナビゲーション
- 辞典 `/dictionary/*` 内の各種絞り込み（colors の色名引き、kanji の検索、yoji の検索 等）
- グローバルナビゲーション（Header の主要セクションリンク）からの到達

これらは Phase 4 / Phase 8 の通常スコープで全部実装・移行される。今回判断するのは「これらと別軸で、異種コンテンツを跨いで一括検索する機能を持つかどうか」**だけ**。

#### 判断材料として使う／使わないものの線引き

使う:

- `docs/site-concept.md`（新コンセプトの中核思想）
- `docs/targets/` 全 5 件（M1a / M1b / S1 / S2 / S3 のニーズ・好悪）
- `docs/design-migration-plan.md` Phase 1.1 / 4 / 5 / 7 / 9 / 10.2 の構造
- `docs/cycles/cycle-172.md` / `cycle-173.md` の失敗認定セクション（同じ失敗を再演しないため）
- `docs/constitution.md` / `CLAUDE.md`

使わない:

- 旧サイト「占い・診断パーク」由来の GA4 / BigQuery / Search Console データ（来訪者層が新コンセプトのターゲットと別。cycle-172 / 173 の核因）
- 業界一般の UX ベストプラクティス（コンセプト固有の価値判断であり一般論は答えを持たない）
- 実装コスト・既存コード資産（ロジック流用可否は供給側の都合で来訪者の利益とは無関係）
- B-340 で蓄積中の利用ログ（旧コンセプト来訪者のもの）

各項目は「誰が・何をしているとき・何が起きるか」を必ず含める。末尾に主に効くターゲットを `(...)` で付記。

D-12. **ブログ 100+ 件の記事内検索の質**: Phase 4 でブログ一覧に絞り込みは入るが、その粒度がタグ駆動・タイトル前方一致など限定的だと、S1 / S3 が「過去の似た話題の記事」を全文検索的に探す需要に応えられない。D-1 の補強。ただしブログ一覧の絞り込み実装範囲（タグのみ / 全文 / 部分一致）は Phase 4 着手時に決める論点であり、② を採用してもブログ内全文検索の充実で部分緩和できる (S1 / S3) [中: Phase 4 設計次第で部分緩和可能]
D-13. **AI 独自性の体現面が 1 つ減る**: 「AI が書いたものと AI が作ったものが混ざる検索体験」（A-14）は失われる。代替は C-13（各ページ + 意図的クロスリンクで独自性を表現）だが、検索 UI ほど一望性のあるシグナルにはならない (S1 / S2 / S3) [中（第 4 回再検証で強→中に降格）: A-14 の損失側。A-14 の中格下げに連動]

#### 論理評価のルール

- **強**: 検索窓の存在そのものに固有の便益・損失で、ランキング設計・クロスリンク・Phase 4 通常スコープなどでは原理的に置き換えられない
- **中**: 一定の論理性はあるが、別手段（ランキング設計、クロスリンク、ブログ全文検索充実等）で減衰可能、または効果規模の検証が弱い
- **弱**: 過大評価 / 独立変数の混同（検索結果中身と検索窓存在の混同等） / 射程歪曲 / 検証可能な事実に支えられていない / 供給側都合 / 副次的観察 / 既に他項目で評価済み

判断は **強・中項目のみ** を根拠材料とする。弱項目は捨てる。

### 判断

#### 判断ロジックの履歴

  - 旧根拠 3（道具箱との役割重複 = B-5 依拠）: 役割が違うため並立可能。論理破綻
  - 旧根拠 4（メイン優先）: 1〜3 が崩れた時点で前提崩壊

- **第 4 回 Owner 指摘**: A-9 / A-14 の「AI 自律運営独自性の一望性体現」が検証不能・他サイトで再現可能で過大評価。連鎖して A-4 / A-5 / A-6 も「主動線で起きるか」「あれば便利レベルか」の問いに耐えず過大評価。**A-4 / A-5 / A-6 / A-9 / A-14 を全て「中」に降格、対応する D-2 / D-5 / D-7 / D-13 も連鎖降格**
- **第 4 回再構築（本セクション）**: A〜D 全項目に強評価ゼロ。中項目総和のみで判断する

**結論を維持するために強評価を温存することはしていない**。降格の結果として強根拠が消えたことを正直に記録し、判断ロジック全体を中項目総和に組み替える。

- **第 6 回判断（2026-05-11 cycle-186 中）— ② 転換に再判断**: cycle-186（Phase 5 実装サイクル）中に、PM が WebFetch で一次資料（https://developers.google.com/search/blog/2024/10/sitelinks-search-box ）を確認し、**Google sitelinks search box が 2024-11-21 に正式廃止** されていることが判明。cycle-174 ① 採用の中位メリット A-13（SearchAction による SEO 便益）は本サイクル時点で実在しない便益となった。残る 5 項目（A-7 / A-4 / A-5 / A-1-2 / A-6）はいずれも本ファイル本文自身が「中項目で別手段による代替可能性あり」と認める弱点を持ち、サイトデザイン切替（Phase 6/7/8/9/10.2）が全来訪者・全ページ訪問にもたらす便益と比較して桁違いに薄い。CLAUDE.md L9 Decision Making Principle「If the better UX option is achievable, it must be chosen」に従い、② 転換が来訪者価値最大化に適うと PM が確定（Owner 同意済み）。Phase 5（B-331）の新デザイン実装は実施せず、横断検索は Phase 10.2 で legacy 全廃と一体で自然消滅する。詳細は `docs/cycles/cycle-186.md` 参照。

#### 結論

**判断結果: ① 新デザインに横断検索を作る（Phase 5 = B-331 を実施する）。**

##### 結論の根拠

1. **横断検索を持つことのメリット（実在）**:
   - 名前を覚えていなくても思いついた言葉で目当てのコンテンツに辿り着ける（M1b の名前忘れ救済を含む、A-7 等）
   - ツール / 辞典 / ブログを先に選ばなくてもキーワード一つで候補が出る（未分化キーワード探索、A-4）
   - キーボードで navigation を介さず直接ジャンプできる（Cmd+K、A-5）
   - 記事を読んでいて出てきた言葉から関連ツール / 辞典項目へ飛べる（記事 → 道具回遊、A-1 / A-2）
   - 視覚 navigation を辿るのが負担な来訪者の到達手段が増える（A-6）
   - Google 検索結果上にサイト内検索ボックスが表示される（SearchAction による SEO 便益、A-13）。**条件付き便益**: 現状のモーダル UI 実装では SearchAction の `target` 要件（`/search?q=...` のような独立ページ）を満たせないため、本便益を享受するには Phase 5 で独立ページ実装が必要

2. **横断検索を持つことのデメリット（実在）**: ほぼゼロ。第 1〜4 回で列挙した B 群項目を Owner と検証した結果、いずれも検索の存在そのものに固有のデメリットではないと判明:
   - 「検索結果のランキング次第で迷う」（B-1 / B-2 / B-8）は実装の質の問題で、検索の存在そのものに固有ではない
   - 「Header の 1 枠スペース消費」（B-7）は現状のレイアウト（左ハンバーガー / 中央ロゴ / 右 actions スロット）に余裕があるため発生しない
   - 「クロスリンク動機弱化」（B-11 / B-13）は運営規律の問題で、検索とは独立

3. **判断**: メリットがあり、実在のデメリットが無い。ゆえに来訪者にとって作るほうが良い。これ以上の判断構造は不要。

### 移行期間中の対応

① を採用するため、横断検索は移行完了後も新デザインに存在する。移行期間中の動線変化は次の通り:

- **Phase 4 完了時点**: `(new)/` の Header に新版横断検索が結線される（Phase 5 = B-331 で実装）。Phase 4 と Phase 5 の順序関係上、Phase 5 が Phase 4 と Phase 6 の間に挟まる
- **Phase 5 完了後 〜 Phase 7 進行中**: 新ヘッダーと旧ヘッダーの両方に横断検索が存在する。検索結果から legacy / 新側どちらのページにも遷移可能（design-migration-plan.md Phase 5 完了基準のとおり）
- **Phase 10.2 完了時点**: 旧 `src/components/search/` は撤去され、新 `src/components/Search*/` のみが残る。`/search-index.json` は新旧で共有される単一ファイルのまま継続

#### 新旧来訪者混在の具体化

Phase 7 進行中、検索エンジンから `(legacy)/` 個別ページへ直着地する新コンセプト来訪者は、レガシー Header の横断検索に最初に遭遇する。新版に遷移すると新版横断検索（同じ機能、新デザイン）が利用可能。検索の有無自体は新旧で同じで、UI の見た目だけが移行に応じて切り替わる。これは新コンセプト来訪者にとって混乱の元にならない（機能の連続性が保たれる）。

#### Phase 5 設計時に Phase 4 が用意すべき要素

Phase 5 が Phase 4 完了後に実装されるため、Phase 4（B-334）実装時点で以下を新ヘッダー側に用意しておく必要がある:

- 新ヘッダーの actions スロットに「検索トリガー（アイコン or ボタン）」を配置できる構造
- モバイル Header での検索アイコン配置（B-7 の評価通り 1 枠分）

これらは Phase 4 のスコープ拡張ではなく、Phase 5 着手の前提として Phase 4 のヘッダー設計に含めるべき要素。design-migration-plan.md Phase 4 の補足として明記する。
