### 目的

本サイクルは「実装に着手する前に決めておかないと、後続 Phase のスコープが揺らぎ手戻りする」2 つの判断を確定させることが目的。

- **後続 Phase の手戻りを防ぐ**: Phase 4（一覧設計）、Phase 5（検索実装の要否）、Phase 8（既存コンテンツの削除/移行範囲）の作業範囲は本サイクルの判断結果で決まる。判断を後送りにしたまま実装に進むと、Phase 4 で作った一覧 UI を Phase 5 の検索方針確定後に作り直す、Phase 8 直前で削除/移行の方針を慌てて決めるなどの再作業が発生する。
- **来訪者にとって最終的に良い形に着地させる**: 検索の存在/形態、辞典・cheatsheets・achievements の存続/廃止は、ターゲットユーザー（特に「特定の作業に使えるツールをさっと探している人」＝初回来訪のツール利用者、「気に入った道具を繰り返し使っている人」＝リピーター）の体験に直接影響する。「日常の傍にある道具」というコンセプトに合わない要素を残すことも、価値ある経路を不用意に消すことも、来訪者価値を損なう。
- **判断の根拠を後から再現できる形で残す**: cycle-169・170 が失敗した直接の原因は前提が未検証のまま結論に組み込まれたこと、cycle-171 はそれを是正して立て直したこと（直前 5 サイクルの教訓）を踏まえ、本サイクルでは「どのデータをどの期間で見て、どう評価したから (a)/(b)/(c)/(保留) を選んだ」をドキュメントとして残す。後続 Phase でこの判断を引き継ぐサブエージェントが、根拠を再確認できる状態にする。

注意：本サイクルは判断のみで実装作業ゼロ。来訪者から見える変化はない。だからこそ「決めることに時間と工数を投資する」サイクルとして位置づける。

### 作業内容

判断は逐次。Phase 1.1（検索方針）→ Phase 1.2（既存コンテンツ整理方針）の順で直列に進める。両者は独立した判断のため順序は厳密でなくてもよいが、Phase 1.1 の結論は Phase 4・5（後続 Phase）の作業範囲に直接波及する一方、Phase 1.2 は Phase 8 の範囲に主に波及する。Phase 1.1 を先に終わらせれば Phase 1.2 のリサーチ中に Phase 4 系 backlog の状態遷移も確定できるため、進行管理上の利点で先に置く。

#### 共通の進め方（両判断に適用）

1. **判断のサブエージェント分割**:
   - **Phase 1.1**: 「現状計測の十分性チェック」「データ取得・分析」「候補評価と推奨」の 3 ステップに分け、それぞれ別サブエージェントに委譲する（判断の性質上、データ収集と評価を分離した方が AP-P02 の回避効果が高い）。
   - **Phase 1.2**: ルートのグループ単位で 1 サブエージェントが収集・評価・推奨を一括する分け方にする。サブエージェント分割の単位はグループ（`dictionary トップ` 1 単位、`dictionary/colors 系` 3 単位、`dictionary/humor 系` 2 単位、`dictionary/kanji 系` 2 単位、`dictionary/yoji 系` 2 単位、`cheatsheets` 8 単位、`achievements` 1 単位）だが、**判断はそれぞれのサブルート単位（合計 19 単位）で個別に下す**。`dictionary` トップ単独は cheatsheets グループのサブエージェントに同梱するか、`dictionary/colors 系` のサブエージェントに同梱するなど、サブエージェントを増やしすぎない単位で割り振る。3 ステップに機械的に分けると、ルートごとの個別判断が分散して再統合のコストが大きい。
   - 小タスク単位での委譲は CLAUDE.md「Keep task smaller」原則に従う。

2. **GA4 データ取得の前提条件**:
   - 取得期間は **過去 90 日** を基本（cycle-161 の 30 日窓よりサンプル数を厚めに取り、季節性を平準化するため過去 90 日を採用）。月次換算は「過去 90 日合計 ÷ 3」。サンプル数が著しく少ない場合は過去 180 日も併せて確認し、その際は「過去 180 日合計 ÷ 6」で月次換算する。
   - 本サイクルで使う PV 指標は 2 種類:
     - **月次 OS PV**（Organic Search 由来、自然検索流入の PV を月単位で集計したもの）: cycle-161 の「辞典平均 0.003 PV/ページ」や cycle-167 L51-52 の表記に揃える。SEO 流入価値の評価軸として使う。
     - **月次全体 PV**: 流入元を問わない全 PV の月平均。コンセプト整合・来訪者の体験全般の評価軸として使う。
   - GA4 から取得する際は両方を取得する（GA4 の `sessionDefaultChannelGroup = 'Organic Search'` での絞り込みで OS PV を、絞り込みなしで全体 PV を）。月次は「過去 90 日合計 ÷ 3」（または 180 日 ÷ 6）に揃える。
   - cycle-167 L52 の「0.003 PV/ページ」は **月次 OS PV** の値であり、本サイクルでもこの値はそのまま「月次 OS PV」として参照する。
   - データソースは GA4 MCP（`mcp__google-analytics__run_report`）。BigQuery は 2026-03-28 以降のみで期間が短いため本サイクルでは使わない。
   - 取得は foreground sub-agent で行う（CLAUDE.md「Use foreground sub-agent for MCP tools」）。
   - 取得した生データは `tmp/` 配下に保存（`docs/research/` には判断の最終根拠だけを置く）。

3. **判断保留の選択肢を許容する**: GA4 のサンプル数が判断に必要なラインを下回る、データに重大な違和感がある、複数の候補が同程度の評価で決め切れない場合は **「保留」** という選択肢を残す。保留時のフロー（後続 backlog の状態遷移）は Phase 1.1 セクションと Phase 1.2 セクションのそれぞれで具体化する。判断を無理に下すことの方が、後続 Phase でより大きな手戻りを生む。

4. **アンチパターン回避**:
   - AP-P01: 「Phase 4・5・8 で検証するからここでは仮置き」を許さない。本サイクルで実測ベースに確定する。
   - AP-P02: 自分が選びたい候補を否定するデータも積極的に探す。例えば「検索撤去」を支持したいなら、検索を実際に使っているユーザーがいないか反証データを取得する。
   - AP-P04: 本サイクルでは Owner 発言を判断材料に直接使わない。GA4 実測と既存リサーチ（`docs/research/` 既存ファイル）のみを根拠とする。
   - AP-P06: 過去サイクル（特に cycle-161 のデータ分析、cycle-167 のコンセプト策定）を必ず参照し、今回の判断と矛盾しないか確認する。
   - AP-P07: 判断軸の出発点は「来訪者の行動・体験」に置く。運営コストや実装の容易さは副次的な評価軸。
   - AP-P09: 判断の根拠は「来訪者の体験への影響」で行う。SEO スコア・ドメイン権威・運用コスト効率化を主軸の判断軸にしない（副次評価軸としてのみ参照）。
   - AP-P10: Phase 1.1 の (a)/(b)/(c) や Phase 1.2 の「移行/削除」のいずれかが推奨候補として高い確信度で出てきたとき、過去実績データ（cycle-161 の OS PV、cycle-167 の表、`docs/research/2026-03-22-dictionary-seo-value.md` 等）と矛盾していないかを必ず照合する。「適合度高」「明らかに撤去」のような直感的な高評価が出たら、過去サイクルの実測値と整合するかを検算してから採用する。
   - AP-P11: cycle-167 等で確定した「コンセプト整合」を絶対条件として扱わない。GA4 データがコンセプトと矛盾する形で価値を示しているルートがあれば、コンセプトより実測を優先して移行判断する（ただし覆す場合は根拠を明示）。

#### Phase 1.1: サイト内検索の方針判断（B-332）

**前提（実装の現状）**:

`src/lib/analytics.ts` で実装されている検索関連 GA4 イベントは **`trackSearch`（`search` イベント、`search_term` のみ）の 1 つだけ**。検索モーダル開閉、検索結果クリック、検索→離脱は **イベントとして発火していない**。本サイクルではイベント追加実装は行わない（実装作業ゼロのスコープ）ため、以下は「現状取得できるデータの範囲で判断する」前提で進める。

**ステップ 0: 現状計測での判断可否の事前判定（最初に行う）**:

最初のサブエージェントは以下を判定する。

- 過去 90 日の `search` イベント数を取得し、月次に換算（合計 ÷ 3）。
- セッション全体に対する `search` 発生セッションの比率を取得（GA4 標準の session スコープで `eventCount > 0 where eventName = 'search'`）。
- 上位 `search_term` を取得（最低 50 件、可能なら 100 件）。
- **`page_referrer` が代替指標として機能するかを GA4 実データで確認**: 本サイトの検索はモーダル UI のため、`search` イベント直後の `page_view` の `page_referrer` は検索モーダルのオープン元 URL を返してしまい、検索結果クリック後の遷移経路を正確に追跡できない可能性が高い。`search` イベント発火セッション内の連続イベント列を実データで確認し、`page_referrer` が「結果クリック前のページ」と「結果クリック後の着地ページ」を区別できているかを検証する。**判定基準**: `search` イベント発火セッションのうち、「`search` 直後の `page_view` の `page_referrer` が `search` 発火直前ページ以外（＝検索結果クリックによる遷移を示す URL）になっているセッション」が一定割合以上ある場合に「機能している」と判定する。閾値の具体値（例: 50%、70% など）は判断時に GA4 全体トラフィック量・`search` 発火セッション数を見て設定する（L138 の最低サンプル数ライン同様、絶対値ではなく実データ分布を踏まえて設定する運用）。機能していなければ、`page_referrer` ベースの遷移先推定はステップ 1 以降の判断材料に使わない。
- これらだけで「(a)/(b)/(c) のいずれかを根拠を持って選べるか」を判定する。
- 判定が **「足りる」** の場合: ステップ 1 に進む。
- 判定が **「足りない」** の場合: 判断は **「保留」** を選ぶ（保留時のフローは下記）。

**取得・評価するデータ**:

取得可能（GA4 から直接取得する）:

- `search` イベント数（過去 90 日合計、月次換算）
- `search_term` の上位クエリ一覧（カテゴリ偏りの定性分析用）
- セッション内 `search` 発生有無の比率（検索利用率の代理指標）
- `page_referrer` を使った検索前後のページ遷移パターン（検索後にどのカテゴリのページに着地しているかを推測）
- **検索を使わなかったセッションの動線分布**: landing page → 次ページの遷移パターンを取得し、「検索が無い場合に現状の既存導線（ヘッダーナビ・カテゴリ一覧。なお道具箱は Phase 9.2 / B-336 で将来実装予定のため Phase 1 時点では評価対象外）で動線が成立しているか」を評価する材料にする
- 全体トラフィックに対する検索利用者の比率

取得不能（本サイクルでは判断材料に使わない、または代替で間接評価する）:

- 検索モーダルの開閉数（SearchTrigger クリック / Cmd+K / Ctrl+K 発火数） → **判断には使わない**
- 検索結果クリック数および遷移先のカテゴリ分布 → ステップ 0 で `page_referrer` が代替指標として機能すると確認できた場合のみ間接的に推測する（精度低い前提で扱う）。機能しないと判定された場合は **判断には使わない**
- 検索 → 離脱（結果クリックなし）の割合 → **判断には使わない**

**評価軸**:

1. **来訪者にとっての必要性**: 検索が無い場合に「特定の作業に使えるツールをさっと探している人」「気に入った道具を繰り返し使っている人」の動線が壊れるか。検索を使わなかったセッションの動線分布から、現状のフォールバック動線（ヘッダーナビ・カテゴリ一覧。道具箱は Phase 9.2 で将来実装予定のため Phase 1 判断時点では評価対象に含めない）で代替可能かを評価。
2. **「日常の傍にある道具」コンセプトとの整合**: 新コンセプト下で想定する「道具箱中心の体験」（Phase 9.2 で実装予定）が成立した将来像で、横断検索が中心動線の一部として残るのが自然か、それとも道具箱と一覧で十分か。ただし AP-P11 に従い、コンセプト整合だけで判断しない（実測が反証するなら覆す）。なお Phase 1 時点では道具箱は未実装のため、現状動線の評価とは混同しない。
3. **横断 vs カテゴリ別の必要性**: `search_term` の傾向と `page_referrer` ベースの推測遷移先がカテゴリ横断か、カテゴリ内で完結しているか。
4. **運用コスト**（副次評価軸）: search-index 生成、Fuse.js のメンテナンス、新デザイン用 view の新規実装の重さ vs 利用実績。AP-P09 に従い主軸にはしない。

**候補と意思決定基準（暫定）**:

- (a) 横断検索維持: 全体に対する検索利用比率が一定以上、かつクエリがカテゴリ横断している場合。
- (b) カテゴリ別検索に分割: 検索利用比率は無視できないが、クエリと推測遷移先が明確にカテゴリ内で完結している場合。一覧ページ内に絞り込み（タグ・カテゴリ・テキスト検索）を埋め込む形で代替する。
- (c) 検索撤去: 利用比率が極めて低く、検索を使わなかったセッションの動線が既に成立している場合。Phase 5 と Phase 10 の旧 search 撤去が同時に終わる。
- (保留): ステップ 0 でデータ不足と判定された場合、または (a)/(b)/(c) が同程度で決め切れない場合。

最低サンプル数のラインは判断時に GA4 全体トラフィックを見て設定する（極端に小さいサイトなので絶対値ではなく比率と絶対値の両面で評価する）。

**保留時のフロー（正規ルート）**:

「保留」を選んだ場合、以下を Phase 1.1 出力に明記し、関連 backlog の状態を遷移させる。

- (a) どのデータ（イベント・期間）が揃えば判断できるか
- (b) いつまでに再判断するか（推奨: 必要イベント追加実装後 30〜60 日）
- (c) 後続 backlog の状態遷移:
  - **B-331（Phase 5: 検索実装の要否）**: Blocked に遷移。Notes に「Phase 1.1 保留中。再判断条件 X を満たした時点で再開」と記載。

### 検討した他の選択肢と判断理由

- **Phase 1.1 と 1.2 を別サイクルに分ける**: 別サイクルにすると判断間隔が空き、後続 Phase の着手も遅れる。両判断は独立しており、データ取得対象（GA4）も同じツール、判断者（PM）も同じため、本サイクルで両方完結させる方が手数が少ない。サイクル内でサブエージェント分割するため、品質は確保できる。
- **Phase 1.2 で「全ルート一括で判断」する**: 辞典 10 ・cheatsheets 8 ・achievements 1 の計 19 サブルートをまとめて「全部削除」「全部移行」と決める案は工数最小だが、サブルートごとに PV・流入経路・コンセプト整合が異なる以上、粗い判断は来訪者価値を損なうリスクが高い（AP-P09: ゴールの読み替え、運用コスト先行）。サブルート単位で個別判断する。
- **Phase 1.1 で GA4 を見ずに「コンセプト的に道具箱と検索は両立しないから撤去」と即断する**: cycle-167 のコンセプトに合うかどうかだけで結論を出す案は AP-P02（都合の悪いデータの無視）と AP-P04（検証なしの結論採用）に該当する。実測データで「特定の作業に使えるツールをさっと探している人」「気に入った道具を繰り返し使っている人」が検索を実際に使っているか確認してから判断する。
- **Phase 1.2 を Phase 8 直前まで先送りする**: 「実際に削除するときに判断すれば良い」案は、Phase 4（一覧設計）の段階で「辞典・cheatsheets を一覧に含めるか」を決められないと一覧 UI が作り直しになる。Phase 1 で確定させる方が後続スコープを安定させる。
- **判断を「絶対の確定」にする**: 後で気づいた事実があっても変えられない仕様にする案は、AP-P11（過去判断を変更不可とみなす）と同型の失敗を生む。本サイクルでは「現時点でのベスト判断」と位置づけ、後続 Phase で重大な反証が出た場合は再判断可能とする（ただし変更時はコストを伴うので根拠を明示する）。
- **検索データのサンプル数が足りない場合に強引に判断する**: 直近 5 サイクル（特に 169・170）の失敗パターンと同型のため避ける。「保留」を正規の選択肢として準備し、保留する場合の再判断条件を明示する設計にする。

## 失敗サイクル認定（2026-05-01 Owner 指摘により追記、2 段階で更新）

### 全面失敗

本サイクルは **Phase 1.1 / Phase 1.2 ともに失敗** と認定する。当初は Phase 1.1 のみを失敗とし Phase 1.2 は妥当と扱っていたが、Owner 指摘により Phase 1.2 も前提を調査せず推測で判断していたことが判明し、全面失敗認定に書き直した。

#### (1) 問いの種類を見誤った（データ判断 vs 設計判断）

Phase 1.1 は当初から **データ判断ではなく設計判断** で答えるべき問いだった。本質は「ツール・遊び・ブログのように **異なる分野のコンテンツを横断して検索したい需要が新コンセプト下のターゲットにあるかどうか**」であり、これは旧コンセプトの利用ログでは答えられない（来訪者の動機・コンテンツ分布・動線が新旧で大きく異なるため、旧データで新サイトの需要を予測する論理は成立しない）。

本サイクルは「サンプル数 n=5 で確定不可」として保留判定したが、これは前提誤認。サンプル数を増やしても旧サイトの傾向しか見えず、新コンセプトの判断には届かない。

cycle-173（B-340: 計測強化実装）は本サイクルの保留判定を引き継ぎ、30 日データ蓄積のための計測 infrastructure を整備したが、これも前提が誤っていたため成果物として無効。両サイクルとも本質を見失って「考えるフリ・分析するフリ」をしたに留まった。

#### (2) (a)/(b)/(c) の選択肢設定が誤っていた

本サイクルが受け継いだ「(a) 横断検索維持 / (b) カテゴリ別検索に分割 / (c) 検索撤去」の 3 分割は実態として誤っていた。実態は **二択**:

- **① 新デザインに横断検索を作る**（= Phase 5 で実装）
- **② 新デザインに横断検索を作らない**（= Phase 5 をスキップ）

理由:

- ツール内・遊び内の絞り込みは Phase 4 の通常スコープとして **必須実装** される（横断検索の有無と無関係）。来訪者がツール一覧から目的のツールを探せるようにするための基本機能
- 旧 (b)「カテゴリ別検索に分割」は、絞り込みが必須実装である以上、(c) と本質的に同じ「② 横断検索を作らない」に collapse する
- レガシー検索の物理撤去（Phase 10.2）は ② を選んだ場合の自然な帰結であり、独立した判断項目ではない

#### (3) 移行期間中の動線継続性に懸念がある

② を選んだ場合の動線変化:

- Phase 4 完了時点: `(new)/` のトップ・一覧ページから横断検索動線が消える
- ただし Phase 7 までツール/遊び個別ページは `(legacy)/` のまま残るため、その Header には横断検索がある
- Phase 7 以降、個別ページが順次 `(new)/` に移行するにつれ、横断検索にアクセスできるページが段階的に縮小していく

`(new)/` のトップ・一覧という主要動線から検索トリガーが消える時点で、来訪者の発見動線に影響が出る可能性がある。② を選ぶ場合は、Phase 7 以降の段階的縮小も含めて来訪者にとって許容できる設計か確認する必要がある。本サイクルではこの観点を考慮せず保留判定していた。

### Phase 1.2（既存コンテンツ整理の方針確定）も失敗

当初の判断結果（移行 8 / 削除 11）は **無効** とする。失敗の構造は Phase 1.1 と同じ「前提調査の不足 + 選択肢設定の欠陥」だった。

#### (1) 前提調査が不十分だった（推測で判断していた）

kanji 系 2 単位を「削除」と決めたが、削除判断の根拠とした内部リンクの実態確認が **誤っていた**。リサーチ doc L214 で「ゲーム（kanji-kanaru）との URL 依存は確認できず、削除してもゲームに影響なし」と書いていたが、実態は:

- `src/dictionary/_components/kanji/KanjiDetail.tsx` L114: 漢字辞典 → 漢字カナール
- `src/app/(legacy)/play/kanji-kanaru/page.tsx` L44: 漢字カナール → 漢字辞典
- → **双方向のクロスリンクが存在**しており、削除すると漢字カナール側のリンクが dead link 化する

加えて yoji 系を「移行」と決めたが、yoji 詳細から kanji 詳細への内部リンクの実態を見ていなかった:

- `src/dictionary/_components/yoji/YojiDetail.tsx` L46-66: yoji 詳細の「構成漢字」セクションが各漢字を `/dictionary/kanji/[char]` にリンク
- → yoji 移行 + kanji 削除の組み合わせは、移行後の四字熟語詳細から **大量の dead link** を発生させる

これらは grep 1 回で発見できた事実情報。データ収集と評価に時間を使う前に確認すべき前提を、調査せず推測で「削除しても影響なし」と判断していた。

#### (2) 選択肢設定が「移行 / 削除」の二択に閉じていた

第三の選択肢「**インタラクティブ検索ツール化**」が考慮されていなかった。kanji-data.json（2,136 字）と yoji-data.json（400 語）のデータをそのまま使い、入力 → 検索 → 詳細表示 のツール形式に転換する選択肢は、新コンセプト「日常の傍にある道具」との整合が現行の参照リファレンス形式（移行）よりもむしろ高い。cheatsheets についても cycle-167 L379 で「ツール統合の示唆」が触れられていたが、本サイクルでは「ツール統合は判断対象外」として全 8 単位を削除確定にしていた。

選択肢を「移行 / 削除」に閉じたことで、データ資産を活かす第三の道を最初から見落としていた。

### 当初の判断（移行 8 / 削除 11）の扱い

無効化する。Phase 1.2 の代替方針は次サイクル以降で再策定する（`docs/backlog.md` B-315 の Notes と B-338 の Notes を参照）。

### 教訓（cycle-173 と共通）

- **問いの種類を最初に見極める**: 「データで答えるべき問い」と「設計原則で答えるべき問い」を最初に切り分ける。サンプル数や統計手法の前にこの分類を確認する
- **コンセプト変更後は旧データの参照価値を疑う**: 来訪者の動機・コンテンツ分布・動線が変わるなら、旧データで未来を予測する論理は成立しない
- **選択肢が複合的な問いを束ねていないか確認する**: 「撤去 / 維持」のような単純な二項対立に見えても、内部で複数の独立した判断（実装するか・既存をいつ消すか・移行期間中どうするか）が混じっていることがある
- **選択肢を最初から狭めない**: 「移行 / 削除」のような既存形式の二択ではなく、「データ資産をどう活かすか」のような視点で選択肢を広げる。コンセプトに照らした第三の道（ツール化等）を最初に列挙する
- **前提となる事実情報を grep / Read で確認してから推測で書かない**: 「内部リンクの依存はない」のような断定は実コードを確認してから書く。確認していないなら「未確認」と明記する
- **移行期間中の来訪者体験の連続性を設計する**: 段階的移行では「最終形が良ければよい」ではなく、「移行の各段階で来訪者の動線が壊れない」が必須
- 教訓の一般化は `docs/anti-patterns/planning.md` への追加を次サイクル以降で検討する
