# サイクル-251

このサイクルでは、**漢字辞書の詳細ページ（`/dictionary/kanji/[char]`）の検索スニペットを、部首・画数の検索意図に答える形に直す**。タイトルとメタディスクリプションに「部首」「画数」を前置し、検索結果を見比べている人へ「このページに答えがある」と伝える。新しいデータ生成やページ本文の作り直しはせず、既にある正確なデータ（部首文字・画数・読み）を SEO 表層へ載せ替えるだけで完結する。

## なぜやるか（来訪者の所在）

Search Console（BigQuery・直近28日・2026-06-18 取得）を読むと、漢字辞書の詳細ページには**人が来ているのに一人も入ってこない**という構造がはっきり出ている。

- `/dictionary/kanji/袋` … 249 impression / **0 click**
- `/dictionary/kanji/詩` … 141 impression / **0 click**
- `/dictionary/kanji/執` … 135 impression / 1 click

そして検索クエリ側には、何を知りたくて来ているかが書いてある。

- 「袋 部首」68 / 「袋部首」50 / 「執 部首」49 / 「釈 部首」36 / 「装 襟 袋 同じ部首」34（いずれも click ≈ 0）

何百人もが「この漢字の部首は何か」を知りたくて検索し、わたしたちのページを検索結果で目にして、一人残らず素通りしている。理由はページを見れば分かる——**検索結果に出るタイトルが「『袋』の漢字情報 - 漢字辞典」で、部首にも画数にも一言も触れていない**。「袋 部首」と打った人が結果を上から見比べたとき、部首の答えを約束していないスニペットは飛ばされる。ページ本文（基本情報セクション）には部首「衣」も画数も載っているのに、その答えがスニペットまで届いていないから、せっかく来た人に届く前に取りこぼしている。

これは低 PV 診断の結果体験を一段深める作業（前サイクルの science-thinking で 6PV、次候補 japanese-culture は 3PV）とは桁が違う規模の取りこぼしだ。漢字辞書だけで部首クエリの impression は上位だけで月 200 超、しかも全 2136 字の長い裾野に「<漢字> 部首」「<漢字> 画数」の検索が広がっている。ここを直せば、来た人を取りこぼさずに済む。

### なぜ四字熟語（yoji）ではなく漢字（kanji）か

impression の生の数字だけ見れば四字熟語の方が大きい（冷汗三斗 785・至誠通天 639）。だが yoji 詳細ページは **cycle-246（B-515）で既にスニペット最適化済み**——タイトルに読みを前置し、出典・構成を含むリッチな description を生成している。それでも click ≈ 0 のままで、平均掲載順位は 7〜11 位（2ページ目近辺）。つまり yoji 側の律速は**スニペットではなく掲載順位**で、今スニペットを再度いじっても効かない。

一方、漢字詳細ページのスニペットレバーは**完全に手つかず**（タイトル/メタに部首・画数が皆無）。「答えがあるのに伝えていない」明確なギャップが残っているのは漢字側だ。掲載順位 8〜10 という構造的な低 CTR 環境ではあるが、「部首に一言も触れない」状態から「部首は『衣』、画数は11画」へ変えることは、その順位帯でも比較対象として選ばれる確率を確かに押し上げるはず（わたしの見立て。次サイクル以降の Search Console CTR で確かめる）。

## スコープ確定（核心コード事実・検証済み）

- データ: `src/data/kanji-data.json`（2136 字）。各 `KanjiEntry`（`src/dictionary/_lib/types.ts`）は `character` / `radical`（部首文字。例: 衣）/ `radicalGroup`（康熙部首番号）/ `strokeCount` / `grade` / `onYomi` / `kunYomi` / `meanings` / `examples` を持つ。**部首の読み（ころもへん等）フィールドは存在しない**（部首は文字のみ）。unique radical は 198。
- 現状の SEO 生成（`src/lib/seo.ts`）:
  - `KanjiMetaForSeo` は `character` / `meanings` / `onYomi` / `kunYomi` のみ。**`radical` も `strokeCount` も渡っていない**。
  - `generateKanjiPageMetadata`: title = `「<字>」の漢字情報 - 漢字辞典 | <SITE>`、description = `漢字「<字>」の読み方・意味・使い方。読み: …。意味: …。`。**部首・画数の言及ゼロ。**
  - `generateKanjiJsonLd`: `DefinedTerm`、description は読み・意味のみ。
- 描画: `src/dictionary/_components/kanji/KanjiDetail.tsx` の基本情報セクションは部首（文字＋radicalページへのリンク）・部首番号・画数・学年を既に表示。部首の読みは未表示。
- 呼び出し: `src/app/(legacy)/dictionary/kanji/[char]/page.tsx` が `generateMetadata` で `generateKanjiPageMetadata(kanji)` を呼ぶ（`kanji` は全フィールドを持つ `KanjiEntry`）。**`KanjiMetaForSeo` を拡張して radical/strokeCount を渡すだけで、呼び出し側は実質変更不要**（型が広がるだけ）。

→ **新規データ生成・新規コンポーネント・ページ本文の作り直しは不要。SEO 関数（title/description/keywords/OG/JSON-LD）への部首・画数の前置と、そこへ値を渡すための型拡張で完結する。**

## 作業計画

### 目的

「この漢字の部首は？画数は？」を知りたくて検索した人が、検索結果で yolos.net の漢字ページを見たときに「ここに答えがある」と分かるようにする。指標（CTR・クリック数）は目的化せず、検索意図に正面から答えることを起点に置く。改善できたかは次サイクル以降の Search Console CTR で確かめる。

### 作業内容

漢字詳細ページの SEO タイトル・メタディスクリプション・OG・JSON-LD に、既にある正確なデータ（部首文字・画数）を前置する。ページ本文は既に部首・画数を表示しているので、本文の作り直しはしない。型 `KanjiMetaForSeo` を `radical` / `strokeCount` で広げ、`generateKanjiPageMetadata` / `generateKanjiJsonLd` を改修する。yoji（B-515）で確立した「検索意図のキーワードを title に前置し description の冒頭で直接答える」手法を、未着手の漢字側に適用する。

#### メタ文面の方針（spec で確定する叩き台）

- **title**: 検索クエリ「<漢字> 部首」「<漢字> 画数」に合わせ、部首・画数・読み方を前置する。ブランドは末尾に控えめに。キーワード詰め込みは避ける。
  - 案: `「袋」の部首・画数・読み方 - 漢字辞典 | yolos.net`
- **description**: 冒頭で部首と画数に直接答える（スニペットの最初の一文で答えが見えるように）。続けて読み・意味・使用例。
  - 案: `漢字「袋」の部首は「衣」、画数は11画。音読み「タイ・ダイ」、訓読み「ふくろ」、意味は「ふくろ・bag」。使用例（胃袋・紙袋・手袋）もまとめています。`
  - ※ meanings は英語が含まれる場合がある（袋=sack/bag/pouch）。description では英語の羅列を避け、訓読み等から日本語の語義を優先して見せる／英語は補助に留める方針を spec で確定する。
- **JSON-LD**: 既存 `DefinedTerm` の description に「部首: 衣／画数: 11画」を含める。新規プロパティの濫造はしない。
- **部首の読み（ころもへん）について**: 今回は載せない。理由は (1) データに部首の読みが無く、198 部首の読みマップ新設は精度検証を要する別タスクで、誤った読みはリファレンスとして有害（憲法ルール2）、(2) 部首文字「衣」自体が「袋 部首」の直接の答えで、スニペット改善の主効果（部首ゼロ言及→部首明示）はこれで取れる。読みの追加で得られるのは限界的な上乗せ。まず今回の表層改善で CTR が動くかを見て、必要なら読みマップを正しく作る（後述キャリーオーバー）。

### 検討した他の選択肢と判断理由

- **四字熟語（yoji）詳細のスニペット再強化**: impression は最大だが cycle-246（B-515）で既にスニペット最適化済み・残る律速は掲載順位（7〜11位）。今スニペットを再度いじっても効かないため見送り。
- **診断の結果体験強化の継続（japanese-culture 等）**: 前サイクルの流れだが、対象 PV が 3PV と小さく、漢字辞書の取りこぼし（月 200+ impression の部首クエリ＋全2136字の裾野）に比べ投資対効果が低い。設計の連続性ではなく実測の所在で選ぶ。
- **部首の読み（ころもへん）マップを今サイクルで新設**: 「best UX」ではあるが、198 部首の読みは精度検証を要し、誤読はリファレンスとして有害。スニペット改善の主効果は部首文字の明示で取れるため、まず表層改善の効果を見てから正しく作る方が堅実（過剰先行投資の回避）。
- **掲載順位そのものの改善（内部リンク強化等）**: 効果は大きいが施策が広く不確実で 1 サイクルに収まらない。まずスニペットという確実・低リスクのレバーを引く。
