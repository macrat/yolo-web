# サイクル-278

**オーナー主導サイクル。** cycle-268〜277 の10サイクル、B-545（サイトの根幹の統一）は完了せず、cycle-276 引き継ぎが予告した通りオーナーの直接介入に至った。本サイクルはオーナーが規定した以下の4ステップのプロセスで進める。

1. **市場調査**をして、現在までに PM（歴代含む）が企画した**診断中心のサイトコンセプトの妥当性を検証**する。
2. 1の調査結果を元に**サイトコンセプトを練り直す**。
3. **デザインシステムをゼロから再構成**する（B-559）。
4. **次サイクルでサイトの作り直しをするための計画**を立てる（B-560 の決着を含む）。

実装（サイトの作り直し）は本サイクルではなく次サイクルで行う。本サイクルの成果物は、検証済みのコンセプト・ゼロベースで導出されたデザインシステム・作り直し計画の3点である。

**進め方**: オーナー主導のため、各ステップの成果物はオーナーに報告し、確認を得てから次ステップへ進む。判断そのものは PM が下して報告する（オーナーへの判断委譲はしない＝AP-WF24）。

**cycle-277 自己点検との対応**: このプロセスは cycle-277 の欠陥系列に正面から対応する——ステップ1は欠陥2（候補空間を既存3系統に限定した）と欠陥5（決定(a)の柱「クリックの84%」の質＝検索意図の充足が未検証）、ステップ3は欠陥3/8（デザインシステムの規則群を所与のまま組み替えた・§7を未導出のまま伝播させかけた）、ステップ4は欠陥4（10サイクルの発端の問い＝検証方針が未決着）に対応する。

**心構え（調査結果への正直さ）**: ステップ1の結論が「診断中心の支持」「修正」「棄却」のどれになっても、その結論に正直に従う。cycle-277 の決定(a)を守ることは本サイクルの目的ではない。守るべきは constitution の Goal（来訪者への最良の価値による PV 最大化）だけである。岩盤は constitution と生の観測データのみ（AP-P32）。

## 実施する作業

### ステップ1: 市場調査（診断中心コンセプトの妥当性検証）

- [ ] **T1: 現状データの自己接地＋「84%クリックの質」の実査（PM 自身が BigQuery で実施）**。cycle-277 T1 は量（クリック数・PV）を見た。今回は質を見る——(1) 検索流入の最新状況（クエリ構成・週次トレンド・cycle-277 出荷後の変化の有無）、(2) **診断への検索着地セッションの着地後行動**（エンゲージ時間・診断開始率・完走率・直帰）を、意図充足の代理指標として設計して読む。特に「アニメキャラ診断」系クエリ（欠陥5 が名指しした意図ミスマッチ疑い）の着地後行動を分離して読む。
- [ ] **T2: 外部市場調査（サブエージェント4名に分担・全出典 URL とアクセス日を記録）**。
  - [ ] **T2-1: 検索意図の実査**——主要流入クエリ（「あなたに最も近いアニメキャラ診断」等）で日本の実 SERP を確認し、上位が何を返しているか・検索者が何を求めているか・当サイトの提供物（オリジナル24タイプ）とのギャップを一次情報で確定する。cycle-277 キャリーオーバー「日本実機 SERP での再確認」の実行を兼ねる。
  - [ ] **T2-2: 診断系で成功している個人・小規模サイトの実査**——大手でない診断サイトで実際に伸びている例を探し、何が勝ち筋か（SEO 構造・シェアの通貨・結果体験・更新頻度）を構造として抽出する。
  - [ ] **T2-3: 候補空間の拡張調査（欠陥2 対応）**——既存3系統（診断/ツール/辞典）の外に、AI が運営する個人規模サイトが勝てるコンテンツ類型がないかをゼロベースで探索する。「まだ調べていない候補は何か」を調査設計に明示的に含める。
  - [ ] **T2-4: 反証調査（敵対的）**——「診断中心」コンセプトを壊す方向の証拠（需要の減衰・供給過剰・AI 量産診断への評価・YMYL リスクの実例）を積極的に探す。

## 作業計画

### 目的

10サイクル失敗し続けた B-545（サイトの根幹の統一）を、オーナーが規定したプロセス——検証→コンセプト→デザインシステム→計画——で土台から立て直す。本サイクルの完了=4ステップの成果物（妥当性判定・改訂コンセプト・ゼロベース導出済みデザインシステム・作り直し計画）がすべて reviewer 検証とオーナー確認を経て揃うこと。サイトの実装変更は本サイクルのスコープ外（次サイクル）。

### 作業内容

T1〜T8 を順に実施する。順序が本質：**検証（T1-T3）→コンセプト（T4）→デザインシステム（T5）→計画（T6）**。後段は前段の結論を入力とするため、ステップを跨いだ先行着手はしない。各ステップ末でオーナーへ報告し確認を得る。

- T1 は PM 自身が実施（GA 確認は PM の責務・CLAUDE.md）。
- T2 は 4 タスクをそれぞれ別のサブエージェントに委譲（keep task smaller）。SERP 実査等で MCP ツールが要る場合は foreground で実行する。
- T3/T4/T5/T6 の成果物はそれぞれ独立の reviewer に検証させる（1タスク1レビュアー）。レビューへの問いに「所与としている層」「候補空間の外」を含める（cycle-277 教訓2）。
- cycle-277 の調査資産（research doc・レビュー済みの数値）は再利用するが、その結論（決定(a)）は所与にしない。再利用する数値は出典元を明示する。
- 本サイクルはコード変更を予定しない（成果物はドキュメント）。lint/format/test/build のゲートは完了時に全走する。

### 検討した他の選択肢と判断理由

- **cycle-277 の決定(a)を所与にして、ステップ1を省略または差分調査に縮める**: 却下。欠陥2（候補空間の限定）・欠陥5（84%クリックの質未検証）は決定(a)の土台そのものの穴であり、オーナーのステップ1「妥当性を検証する」はまさにこれを埋めることを求めている。既存調査の**数値・出典は再利用**するが、**結論は再検証の対象**とする。
- **デザインシステム再構成（ステップ3）や作り直し（実装）を前倒しして同サイクルで実装まで進める**: 却下。オーナーのプロセスが実装を次サイクルに明示的に置いている。また cycle-277 欠陥8 の教訓（導出が先・適用が後）とも整合する。
- **B-560 を独立タスクとして別途扱う**: 却下。検証方針（cycle-268 結論の実行 or 棄却）は「作り直しをどう検証するか」の問いであり、作り直し計画（ステップ4）と不可分。計画の中で決着させるのが一貫する。
- **B-553（診断入口の再設計）等の Queued P1 も Active に含める**: 却下。B-553 は「導出済み視覚言語の適用」＝実装であり次サイクル。今サイクルは最小構成（B-545 の再着手範囲 + B-559 + B-560）に留める。
