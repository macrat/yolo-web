## 作業計画

### 目的

site-concept の核心的な約束——「診断の結果は、来訪者が自分の記号として持ち帰り、他者に見せられる」——を、character-personality（最大の成長エンジン）で**実物として成立させる**。フェーズRは札・印・共有の器（UI）を入れたが、実測は概念が現実に起きていないことを示している:

- **共有がほぼ発生していない**: share イベントは全サイトで直近28日わずか2件、character-personality からは **0件**。
- **持ち帰れる実物が存在しない**: 札（Tsutsumi）は画面上のビジュアルどまりで、画像として保存できない。Tsutsumi 自身のコメントが「画像生成は C4 で決める」と先送りを明記。共有される実体は現状サーバOG画像＋テキストURLのみ。
- **測る土台も欠けている**: `save`（札保存）イベントは実装ゼロ・保存ボタンもゼロ。結果面で使う InviteFriendButton（相性招待の共有）は共有計測を一切撃っていない。

この乖離を、character-personality の結果面に閉じて（面単位のインライン増分）埋める。持ち帰れる記号を作れば、共有ループ（貼られた札→新規来訪）が0から立ち上がり、同時に「持ち帰れる記号が手に入る」という約束が入口の着火（下記の30%離脱）も間接的に助ける。

### 来訪者（誰が・なぜ来るか・市場）

rebuild-plan §4 に従い、targets 定義に頼らず今サイクルの実測から推論する。

- **誰が**: 「自分はどんなタイプか」を知りたい人。作品指定なしの自己投影診断を求めて、性格診断・自己分析系の検索から `/play/character-personality` に着地する（直近28日の実測: このクラスタ単独でサイト全クリックの **69.4%**（274/395）、診断系合算で **85.8%**。平均掲載順位 6.1・CTR 7.17%）。
- **なぜ**: 自分を言い当ててほしい／その結果を「自分の記号」として認識し、他者に見せたい（SNSでの自己表現）。診断系の需要はエバーグリーンで大きく、成長の駆動は結果の社会的共有＝バイラルループ。
- **ファネルのどこで手が止まっているか（実測・GA4 28日）**: PV 340 → 診断開始 238（**開始率70%**）→ 完走 203（完走率85%）→ **共有 0**。最大の離脱は「来たのに始めない」入口の30%（102人/月規模）だが、入口（B-553）はフェーズRで作り直したばかりで、効果を見る前に再設計するのは早計。**未着手のまま残っているのは結果面の「共有できる記号の中身」であり、そこが概念の約束と現実の最大の乖離**。C-b はここを埋める。

### 作業内容

1. **設計フェーズ（先に部品と方式を確定）**: rebuild-plan §2 R-2 の「器は少数・中身は多数」に倣い、まず画像化手段・保存/共有UX・計測契約・ADR を確定してから実装する。画像化はサーバOG流用を第一候補（client canvas の font/CLS/プライバシー懸念＝T5申し送りを回避）。

2. **札の画像化＋保存/共有アクション**: character-personality 結果面に「保存」「共有」を実装。`navigator.canShare({files})` で特徴検出し、対応環境（モバイル Safari/Chrome=共有の主戦場）は `navigator.share` でファイル共有、非対応はダウンロード／コピーへフォールバック。生成画像は札（店号・品名・印・記号面・タイプ名）の設計言語に整合させる。

3. **計測の是正・新設（B-551）**: (a) `save` イベント新設、(b) share のキー統一で level_end と横断集計可能に、(c) 面ごとに割れている content_id 接頭辞規約を統一、(d) InviteFriendButton 共有の計測追加。これで 07-27 の観測窓と C-b 固有観測の土台を作る。

4. **ADR 登録と観測**: 「持ち帰れる記号を実在させれば save/share が0から立ち上がる」という賭けを、観測指標（save率・share率・共有経由の新規流入）・観測窓・撤退基準・双方向ガードつきで起票。フェーズR全体の SC 方向読み（ADR001）とは別の、面単位のインライン観測（§1(d)）。旧 `quiz_result_visual_v1` の轍（読まれない実験）を踏まないよう、読む予定と撤退基準を先に固定する。

### 検討した他の選択肢と判断理由

- **C-a 新クラスタ立ち上げ（B-562）を選ばない理由**: 面を広げる拡張は、全ページ同時変更で Google 再評価が進行中の今（rebuild-plan §5 が「成長局面の一斉切替はタイミングとして賭け」と明記）はリスクが高い。既存最大エンジンの未完成部（共有できる記号）を先に完成させる方が、低リスクで概念の約束を果たせる。
- **入口の30%離脱（B-553）を再設計しない理由**: 入口はフェーズRで作り直したばかり。効果（07-27〜）を見る前に再度いじると、R の効果測定を交絡させ、かつ二重手戻りになる。結果面の共有できる記号を磨けば入口の着火も間接的に助かる。
- **辞典の処遇（C-c・B-564）を選ばない理由**: 漢字辞典はインプレッションの51.6%（12,456）を持ちながら CTR 0.08%＝死蔵だが、処遇判断は検定エンジン（C-a）と同時に行う設計（rebuild-plan §3 C-c）。C-a を今回見送るため C-c も次以降。B-521 の材料（ADR001）は蓄積済み。
- **ツール処遇（B-566）を選ばない理由**: ツール群はクリックほぼ0（1件/28日）だが、処遇は「最高価値に磨いてから判断」する専用の大きな取組（cycle-279 で一律削除を撤回）。今サイクルの最小スコープに載せない。
- **計測是正（B-551）だけを単独でやらない理由**: 純粋な計測配管は今サイクルの来訪者価値がほぼ0（constitution rule 4=品質＝来訪者価値優先）。来訪者が体感する「持ち帰れる札」と同梱し、その効果を測る土台として計測を直す。C-b と B-551 は「磨く／測る」の表裏で密結合のため2件同時着手を正当化（kickoff の最小スコープ原則の例外条件に該当）。
- **画像化を client canvas キャプチャにしない（第一候補としない）理由**: T5 申し送りが font/CLS/プライバシーの確認を条件にしている。サーバは既に結果OG画像を決定的に生成しており、これを札デザインに寄せて流用すればクライアント canvas の複雑さを回避できる。最終決定は設計フェーズで詰める。

### 計画にあたって参考にした情報

外部仕様（ブラウザ API）への依存があるため、計画冒頭で一次資料を確認した:

- **Web Share API のサポート状況**（保存/共有アクションの土台）: 2026-07-13 確認。global 90.3%・iOS Safari 12.2+／Android Chrome 150+ で完全対応（共有の主戦場モバイルで使える）。Firefox 非対応・デスクトップは差あり。出典: [caniuse web-share](https://caniuse.com/web-share)。
- **ファイル共有（Web Share API Level 2 / `navigator.canShare({files})`）**: 2026-07-13 確認。File 配列の共有は仕様上サポートされるが実装依存（"Limited availability"）。**必ず `navigator.canShare({files})` で特徴検出**し、非対応時はダウンロード等へフォールバックする実装が必須。出典: [MDN Navigator.canShare()](https://developer.mozilla.org/en-US/docs/Web/API/Navigator/canShare)。
- **実測データ（BigQuery: GA4 `analytics_524708437` / Search Console）**: SC 2026-06-13〜07-10（28日）でクラスタ別クリック地形、GA4 2026-06-15〜07-12（28日）で診断ファネルとイベント別発生数を取得。character-personality=クリック69.4%、PV→開始70%→完走85%→共有0、share 全サイト2件、save 未実装、を確認。出荷当日 2026-07-13 は両データ0件（データ不足）。
- **コード現状調査（Explore）**: character-personality の24タイプ定義（`src/play/quiz/data/character-personality*.ts`）、結果面（ResultCard／Tsutsumi／第三者向け静的ページ）、OG画像生成（`opengraph-image.tsx`）、share/save/level 計測実装、A/B基盤（`AB_EXPERIMENTS={}`＝空・旧実験の機能残骸なし）を裏取り。B-551 のチケット前提（接頭辞不一致）は実コードでは半分外れで、真の欠陥は「share が item_id／level が content_id キーで結合不能」「面ごと接頭辞規約非統一」「save 未実装」「InviteFriend 未計測」と判明。
