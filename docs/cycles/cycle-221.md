### 対象バグと正答化方針（設計判断・確定）

#### B-485: char-count / byte-counter の数え方統一（①-1）

確定済み事実（調査レポート `docs/research/2026-06-03-char-count-byte-counter-counting-discrepancy-investigation.md`）:

- 食い違う関数は `countChars` / `countCharsNoSpaces`（char-count は `text.length`＝UTF-16、byte-counter は `Array.from`＝コードポイント）と `countWords`（char-count は `Intl.Segmenter`、byte-counter は空白split）。`countLines` / `countBytes` は既に一致。
- char-count の meta は「文字数はUnicodeコードポイント単位でカウント」と明記するのに実装は `text.length` で矛盾。
- char-count のテストには絵文字（サロゲートペア）検証が無い（byte-counter には有る）。

**正答化方針（確定）**:

1. **文字数・空白なし文字数はコードポイント基準（`Array.from`）に統一する。** 根拠: (i) char-count の meta の約束（コードポイント単位）と一致する、(ii) 絵文字1文字=1個という直感に合う、(iii) byte-counter は既にこの基準のため「繰り返し使う人」の結果は変わらない。char-count 側を是正する。
2. **単語数は `Intl.Segmenter`（日本語形態素・利用不可時フォールバック）に統一する。** 根拠: 空白splitは日本語文全体を1単語と誤数する。byte-counter 側を是正する。
3. **正答仕様を単一の定義（SSoT）として持ち、両ツールがそれを参照する。** 根本原因は「同じ数え方ロジックの二重定義・SSoT不在」であり、各 logic.ts を手作業で揃えるだけでは再び乖離する。共通の数え方定義を SSoT 化して両ツールが参照することで、再乖離を構造的に防ぐ（この SSoT は B-490 のタイル再構築でもそのまま使われる）。具体的なモジュール構成は execution で決める。
4. char-count の meta の記述と実装を一致させる（コードポイント単位の記述を実装が満たす状態にする）。
5. 両ツールに正答を保証する回帰テストを追加する（特に char-count に絵文字＝サロゲートペアのテスト、両ツールが同一入力で同一の文字数・空白なし文字数・単語数を返すことを担保するテスト）。

**スコープ外（厳守）**: Component / Tile の UI には一切手を入れない。char-count と byte-counter は統合も廃止もせず2ツール維持（補完関係＝目的別の入口。B-490 に継承）。`countParagraphs`（char-count 固有）・`analyzeByteDistribution`（byte-counter 固有）は各ツール固有機能なので SSoT 化の対象外。

#### B-486: cron-parser の範囲・リスト説明の正答化（①-3）

確定済み事実（調査レポート `docs/research/2026-06-03-cron-parser-japanese-description-bug.md`）:

- バグ箇所は `logic.ts` の `buildFullDescription` 内、`parseInt(minute.raw)` / `parseInt(hour.raw)`。`parseInt("9-17")=9`・`parseInt("9,12,15")=9`・`parseInt("0-30/5")=0` のため、範囲・リスト・ステップ・複合の各パターンで「X時Y分」短縮形に誤って流れ込み、`0 9-17 * * *` が「毎日 9時0分」等と誤表示される。
- 「次回実行リスト」は `parseFieldValues` が作る正しい数値配列を参照しており **parseInt を使わず正しい**。説明生成と次回実行は別ロジックで、共通中間表現は無い。
- `describeCronField` 自体は範囲・リスト・ステップの正しい説明文を生成できる（バグは buildFullDescription が短縮形へ誤誘導する点のみ）。

**正答化方針（確定）**:

1. **`buildFullDescription` の「X時Y分」短縮形を、分・時の両フィールドが単一数値（`/^\d+$/`）のときだけ適用する。** それ以外（範囲・リスト・ステップ・複合）は `describeCronField` の正しい説明にフォールバックさせる。これにより、`parseInt` 誤誘導による事実誤答（範囲・リスト・ステップ・複合がすべて「9時0分」等に化ける）が**全パターンで解消**する。
2. **次回実行リストのロジックは触らない**（既に正しい）。説明生成のみを修正することで、説明と次回実行リストが矛盾しない状態にする。
3. **本サイクルのスコープは「事実誤答の解消」まで（複合の冗長表現は『正確』として許容）。** 複合指定（例 `9-17,20`）は `describeCronField` のカンマ分岐が全値展開し「9時と10時と…と17時と20時」という**冗長だが正確**な説明になる（次回実行リストと矛盾しない）。この冗長さは事実誤答ではないため本サイクルでは許容し、簡潔な表現への改善は B-490 の cron-parser タイル再構築に委ねる（緊急サイクルは正確性の回復を目的とし、表現の簡潔化は対象外）。フォールバック後の文が日本語として意味が通り次回実行と矛盾しないことは確認する。
4. 範囲（`9-17`）・リスト（`9,12,15`）・ステップ（`0-30/5`, `*/15`）・複合（`9-17,20`）の各パターンで、説明生成（`buildFullDescription` 結合）が**事実誤答（「9時0分」化）を起こさず次回実行リストと矛盾しない**ことを保証する回帰テストを追加する（現状は `.values` のテストはあるが `.description` の結合テストが無く、バグが全テスト通過していた）。

**スコープ外（厳守）**: Component / Tile の UI には一切手を入れない。複合指定の冗長表現の簡潔化は本サイクル対象外（B-490 へ）。

### 検討した他の選択肢と判断理由

- **数え方の統一先（B-485）**: 文字数を UTF-16（`text.length`）基準に統一する案もあったが、char-count の meta が「コードポイント単位」と明記しており、コードポイント基準にすれば meta と実装の矛盾も同時に解消できる。かつ byte-counter は既にコードポイント基準のため変更が片側で済み、「繰り返し使う人」への影響が最小。よってコードポイント基準を採用。
- **「結果が変わる」を避けるために正答化を見送る案**: 「繰り返し使う人」の dislikes（結果が変わる）を理由に現状維持する選択肢もあるが、2ツールが矛盾値を返し続けることは「信用できない結果」というより根源的な dislikes に直撃し続ける。誤答の固定化は来訪者価値に反するため、一度の正答化で正しい状態に収束させる方を採る。
- **SSoT 化せず各 logic.ts を個別修正する案**: 工数は小さいが、二重定義のまま手作業で揃えるため再乖離する（①-1 の根本原因がそのまま残る）。根本解決（cycle-220 の精神）として SSoT 化を採用。なお UI を含む実装の一本化は B-490 の役割で、本サイクルは logic の数え方 SSoT までに留める。
- **cron 説明の修正範囲**: 範囲・リストのみ直す案もあったが、同じ `parseInt` 誤誘導でステップ・複合も誤るため、根本（単一数値判定への置換）を直して全パターンの事実誤答を一度に解消する。
- **複合指定の冗長表現を本サイクルで簡潔化するか**: 複合（`9-17,20`）はフォールバック後 `describeCronField` が全値展開して冗長になる。(a) 本サイクルで `describeCronField` のカンマ分岐を簡潔表現に改修する案と、(b) 冗長だが正確なので許容し B-490 の再構築に委ねる案を比較した。本サイクルは「公開中の事実誤答を止める」緊急タスクであり、冗長表現は誤答ではなく次回実行とも矛盾しない。表現改善まで広げると logic 修正のスコープが説明文設計に踏み込み緊急の趣旨から外れるため、(b) を採用（簡潔化は B-490 のタイル再構築でまとめて扱う）。
