### 作業内容

#### T-1: 切り口・構成の確定（PM が方針を固め、blog-writer に渡す）

記事の差別化軸＝**「意味で選ぶ HTML（なぜそのタグを選ぶか）」**。フラットなタグ表・全網羅カタログにしない。少なくとも以下の「混同しやすい使い分け」を「なぜ」から扱う方針を blog-writer に渡す（最終的な構成・見出し・分量・取捨選択は blog-writer に委ねる）:

- **`div` / `section` / `article` / `main` / `nav` / `aside`**: 「とりあえず div」をやめる判断基準。section と article の違い（自己完結して再利用可能か）、main は1ページに1つ、見出しを持つかどうかの観点
- **`strong` / `b`、`em` / `i`**: 見た目（太字・斜体）ではなく意味（強調・重要性／別の声・専門用語）で選ぶ。スクリーンリーダーでの読み上げ差
- **`ul` / `ol` / `dl`**: 順序の有無、定義リストの使いどころ
- **`header` / `footer`**: ページレベルとセクションレベルの両方で使える点、何を入れるか
- **なぜ意味が大事か（記事の背骨）**: アクセシビリティ（スクリーンリーダー）・SEO（検索エンジンの構造理解）・保守性（後から読める）の3観点を、各使い分けの「なぜ」として一貫して通す

基本構造（`<!DOCTYPE>`・head・body 等）や全タグ網羅は土台として最小限に触れるか、必要なら割愛する。紙幅の中心はあくまで「使い分けの判断」に置く。

**薄さ克服のための構造要件（必須・R-1 対応）**: 各使い分けは「**症状（ありがちな書き方で具体的に何が壊れるか＝実害）→ なぜそうなるか → どう選ぶか**」の3層を必ず持たせる。とくに「症状＝実害」を1行で終わらせない——「`div` だらけだとスクリーンリーダーで何が読み上げられないか」「`b` を使うと検索エンジン/支援技術にどう伝わらないか」のように、実害を具体的に言語化する。元チートシートの素材（`Component.tsx` L1051〜の使い分けガイド）は「判断に迷ったら：〜」止まりで症状＝実害の描写が希薄なため、ここを blog-writer が補う必要がある（補う事実は MDN/WHATWG/W3C WAI で裏取り）。この3層構造が「1記法=1行で完結する飽和カタログ」への回帰を構造的に防ぐ（cycle-240 markdown 記事が薄さを克服した決め手と同じ）。各使い分けは可能な限り **before（ありがちな書き方）→ after（意味の通った書き方）のコード例**を伴わせる（想定読者の dislikes「コード例がなく文章だけ」を回避）。

**内部リンク（M-1 対応）**: 本記事の「なぜ」の背骨であるアクセシビリティ・SEO の深掘り先として、`2026-05-28-aria-status-implicit-aria-live.md`（アクセシビリティの具体事故）・`2026-03-02-nextjs-seo-metadata-and-json-ld-security.md`（SEO メタ情報）への内部リンクを自然な箇所に1〜2本置けるか blog-writer に検討させる（無理に差し込まない）。

**フロントマター方針**（直近 markdown/sql 記事に準拠）:

- `category: "tool-guides"`
- `tags`: ["チートシート", "Web開発", ...]（既存タグ体系に合わせ blog-writer が選定。アクセシビリティ・SEO の観点を扱うため適宜追加可）
- `related_tool_slugs`: PM が `src/tools/` を実測した結果、html-tags の「タグ使い分け」に正対するツールは存在しない。隣接候補は `html-entity`（HTML 特殊文字のエスケープ／実体参照変換）のみだが、本記事の主題（意味で選ぶタグ）とは別軸（エスケープ）のため、本文で `<`/`>`/`&` のエスケープに触れる箇所があれば自然な導線として `html-entity` を設定してよい。触れないなら空配列（git/sql 記事も空配列の前例あり）。blog-writer が本文の内容に応じて判断する
- `trust_level: "generated"`、`series: null`、`draft: false`
- 冒頭に AI 運営・内容不正確の可能性の注記（constitution ルール3／既存記事の慣例）

## キャリーオーバー

- **B-342/345/347 と B-349 の循環依存**: cron / http-status / regex は「既存ガイドへの統合＋リダイレクト」に再スコープ済みだが、リダイレクト自体は B-349（Phase 9.2.h）の責務とされており、一方 B-349 の着手条件は「B-342〜B-348 全完了」。これら3件には独立した content 成果物が（既存ガイドが内容を網羅していれば）残っておらず、現状のままでは互いを待ち合う循環になる。本サイクル（B-344）完了後、Phase 9.2 で残るのはこの3件のみになるため、次回以降のキックオフで「B-342/345/347 を B-349 に畳み込む（既存ガイドへのリダイレクト先記録＋撤去を一括実施）」方向の再スコープを検討すべき。**ただし畳み込みの判断の前に、3つのチートシート（cron/http-status/regex）の内容が既存ガイド記事で本当に網羅されているかの網羅性検証を先に置くこと**（網羅されていなければ補筆が必要で単純な畳み込みにはならない＝この網羅性は現時点で未検証）。backlog.md にも反映する。
- **【インシデント】記事 frontmatter の時刻偽装（Owner 指摘）**: 本サイクルで PM が blog-writer に `published_at: 12:30`・`updated_at: 13:10` という**実時刻に基づかない具体時刻**を渡し、しかも updated_at がコミット時点より未来になっていた（検索エンジンのペナルティ直結の致命傷）。`date` 実測値（12:41:49）に両者同一で是正。さらに git log と全直近記事を照合し、markdown 記事（cycle-240）も新規記事なのに published_at(23:04:24)≠updated_at(23:12:37) になっていた違反を発見し updated_at を published_at に揃えて是正（他記事は問題なし・未来時刻の偽装はコミット済みには無し）。再発防止としてアンチパターン **AP-W13**（時刻は date 実測に基づくこと・PM は具体時刻を渡さず手順を渡す・未来時刻禁止・新規記事は published_at==updated_at・コミット前に date/git log と照合）を `docs/anti-patterns/writing.md` に追加。
