# サイクル-170

このサイクルでは B-308「デザインガイドラインとUIコンポーネント集の策定」を再挑戦する。cycle-169 は失敗サイクルとしてクローズされたため、その D 節「次サイクルへの引き継ぎ」を出発点に、Owner から与えられた具体スコープに沿って進める。

## このサイクルで作るもの

Owner からの本サイクルへの指示（要約）:

1. `docs/design-system-by-claude-design/` にある Claude Design 製のデザインシステム定義を素地に使う
2. `docs/research/` 配下の Claude Code へデザインシステムを適用させるためのベストプラクティス調査を参照する
3. `.claude/skills/` にデザインシステムのドキュメントを置く（何がどこにあるか分かりやすく）
4. `src/components/` に共通コンポーネント群を置く（基本的な汎用要素のみ：ヘッダー、フッター、パネル、ボタン、入力欄、記事エリアなど）
5. シンプルで分かりやすいドキュメントを目指す（長期に渡って改善していく前提）
6. 最終的にすべての色名・角丸サイズ等の定数は CSS 変数で管理することを目指す
7. 本サイクルではデザインシステムを完成させるところまで。**既存サイトには手を加えない**（次サイクル以降）

### 目的

このサイクルは、yolos.net の道具たちが今日の誰かの傍に静かに在りつづけるための、「揺るがない判断基準」を 1 セット作る作業だ。

> **Named Aesthetic Tone: `functional-quiet`（機能的な静けさ）** — 装飾より機能、対比より余白、訴求より静けさを選ぶ。来訪者の一日の流れを邪魔せず、必要なときだけ手元に届く道具の佇まい。

来訪者は 5 タイプ——金曜 18 時に締切の文字数を確かめる M1a、月曜の朝に同じ手順で戻ってくる M1b、通勤電車で AI の日記を読む S3、運営の作り方を覗きに来るサブE1・E2——それぞれの「一日」が、わたしの想像の中で並行して動いている。彼らに毎回違う色・違う角丸・違う余白の道具を渡してしまったら、それは「傍に在る」ではなく「気を散らせる」になる。

`character.md` の核「画面の向こうの一日を実際には見られないからこそ、絶えず想像し続ける」「数字は入口で、その先にいる人を見に行く」を、UI 判断の羅針盤として埋め込む。具体的には、上記 Named Aesthetic Tone を全コンポーネント・全ドキュメントの判断基準として固定し、ボタンの hover 一つを決めるときも「これを使う人は今日どんな一日を送っているか」に戻れる構造にする。

ペルソナ化はしない（`docs/targets/README.md` L50-52 の方針）。「20 代の編集者の田中さん」のような具体名ではなく、「金曜 18 時、締切前にコピペで貼り付ける手」のような場面を名指す。これが S3 の好む「具体的な細部」と M1a の現実の動きの両方に効く。

このサイクルは設計と素材の整備で終わる。実サイトに反映するのは次サイクル以降。**見えない準備の品質が、来週・来月・来年の道具の質を決める**——という前提で書いていく。

skill 配下のドキュメントは**自己完結**させる。素地ディレクトリ `docs/design-system-by-claude-design/` は本サイクル内で物理削除されるため、それへの相対リンクや import を残さない。参照する外部ファイルは `src/app/globals.css` のような恒久存在に限る。これは「Claude Design 出力の追跡可能性は git log で確保される」「本番デザインシステム進化に伴う素地との乖離リスクを根絶する」（Owner 直接指示）の 2 点から導かれる判断。

### 設計判断 — CSS 変数の体系と移行戦略

**最大の論点。** 既存 `--color-*` 体系（CSS 内 1,885 箇所参照、ライト・ダーク両方定義済み）と素地のデザインシステム変数体系（ライトのみ、別命名）は別物。これをどう同居させるか。

**比較した 3 案:**

| 案                                                       | 概要                                                                       | 利点                                                           | 欠点                                                                                                                                   |
| -------------------------------------------------------- | -------------------------------------------------------------------------- | -------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| A: 既存 `--color-*` を残し、新変数を追加するだけ（併存） | `globals.css` の `:root` に新変数を追記。新コンポーネントだけ新変数を使う  | Owner 指示「既存サイトに手を加えない」を完全に満たす。実装最小 | 一時的に 2 つの体系が併存する。次サイクル以降で段階置換が必要                                                                          |
| B: 新変数で全置換                                        | 既存 1,885 箇所を新変数に書き換え                                          | 体系が 1 つに統一される                                        | Owner 指示違反（既存サイトに手を加える）。本サイクルでは無理                                                                           |
| C: プリミティブ層を別途設けてセマンティックでエイリアス  | プリミティブ変数を導入し `--color-bg: var(--primitive-...)` のように再構築 | 長期的に最もきれい（Synthesis B-1）                            | 既存変数の値を保ったままエイリアス化するだけでも全変数の見直しが必要で、これも「既存サイトに手を加える」に該当。ダーク値の再検証も発生 |

**採用: 案 A（併存）**。Owner 指示の制約と本サイクルのスコープ（設計と素材の整備のみ）に最も合致する。次サイクル以降で「新コンポーネントの利用範囲を広げ → 既存ファイルを 1 つずつ新変数に置換 → 最終的に `--color-*` を新変数のエイリアスに変える」という段階移行を取る。新旧変数の対応表を変数用途ドキュメント末尾に載せて移行ガイドにする。

**新変数の命名衝突リスクと最終命名統一の判断基準**:

短く頻出する変数名（背景・前景・アクセント等）は、将来 Tailwind / shadcn / 他のサードパーティ CSS を導入したときに衝突する可能性がある。本サイクルでは以下の判断を取る:

- **本サイクル**: 採用する命名は builder が決定する。判断材料は (1) 既存 `--color-*` 系との視覚的区別ができること、(2) 素地由来の意味論を踏襲しつつ Owner が見て直感的に理解できること、(3) 将来のサードパーティ衝突リスクが用途ドキュメント内に明記されること
- **次サイクル以降での再考余地**: サードパーティ CSS 導入の議論が起きた時点で prefix 付きへ rename する余地を残す。判断基準は「(1) 衝突が実際に観測された／(2) 将来導入予定のライブラリが同名変数を提供している、のいずれか」。それまでは本サイクル採用の命名を正とする
- **キャリーオーバーに含める項目**: 「次サイクル以降で命名統一の判断（prefix の有無、`--color-*` を新変数のエイリアスへ寄せるか）を行う」を T-10 の残課題リストに追加する

ダークモード変数は本サイクルでは定義しない。理由: 素地はライトのみ提供しており、ダーク値はコントラスト検証（WCAG AA テキスト 4.5:1 / UI 3:1）を別途やる必要がある（Synthesis B-2）。本サイクルで雑に決めてしまうと「整ってはいるが刺さらない」（Synthesis D-2 リスク C）が発生する。次サイクルとして正式に切り出す。

検証手段: 新コンポーネントが新変数を正しく参照していることは、動作確認ページ（T-09）をビルドし Playwright でスクリーンショットを撮って確認する。`npm run build` が通ること、CSS の未定義変数が発生しないこと（DevTools 確認）が合格条件。

### 設計判断 — `src/components/` の共通コンポーネント

**配置方針: `src/components/design-system/` という新ディレクトリ**

ディレクトリ名は Owner 指示「分かりやすくある必要がある」「`yolos-ds` のような意図が分かりづらい略語ディレクトリは排除」「リポジトリはすべて yolos.net のものなので `yolos` という言葉は意味を持たない」を満たす形で planner（PM）が決定する。`design-system/` は短く・略語なし・意図（=デザインシステムの一部）が直接読める。代替案として `ui/`（汎用的すぎ出自が読み取れない）や `design/`（意図が広すぎる）を比較したが採らない。

理由: `src/components/common/Header.tsx` `Footer.tsx` などの既存共通コンポーネントは実サイトで稼働中（git で確認済）。これを改修すれば「既存サイトに手を加える」になり Owner 指示違反。新規部品を別ディレクトリに並行作成し、サイト本体への差し替えは次サイクル以降の仕事として明確に分ける。

**本サイクル完了時点での 2 系統併存について（明示）**: 本サイクル終了時点で、既存 `src/components/common/Header.tsx` `Footer.tsx`（実サイト稼働中）と新版（`src/components/design-system/` 配下、実サイト未接続）の **2 つの Header と 2 つの Footer が同じリポジトリに並走**する。この期間中は `common/` 配下が「正」で、`design-system/` 配下は「次サイクルで実サイトに接続するための候補」という位置づけ。混乱を避けるため、T-05 のチェック項目として新版コンポーネントの冒頭コメントに「現在の本番は `common/` 配下、本ファイルは次サイクルでの差し替え候補」の旨を明記する作業を含める。2 系統併存の解消は次サイクル以降のキャリーオーバー（T-10 で列挙）。

**新規作成する部品**: Owner 例示「ヘッダー、フッター、パネル、ボタン、入力欄、記事エリア」を最小ラインとする。各部品の具体名・variant 名・size 名・状態名・slot 名・props は builder の判断範囲。

API 設計の指針（Synthesis C-2）:

- props は最小に絞る（過剰な props を持たせない）
- variant・size・状態は型リテラル等で閉じ、自由文字列で開放しない（cycle-169 で問題化した型穴を防ぐ）
- 1 箇所でしか使わないコンポーネントは作らない（Rule of Three）

スタイル: 既存パターン踏襲で CSS Modules を採用。CSS-in-JS は導入しない（既存にない & ビルド構成変更が発生する）。

各部品は新変数のみ参照する。既存 `--color-*` は使わない。これによって新変数体系の動作確認が動作確認ページ（T-09）で完結する。

### 設計判断 — サブエージェント配線

Synthesis A-2 が示す 3 層配線のうち、本サイクルで採るのは **層 2（`paths` グロブによる自動ロード）+ 層 3（system prompt 直書き）** の 2 層。

- **層 1（`skills:` フィールドプリロード）は今回見送り**。理由: Claude Code の `skills:` フィールド仕様の確定確認とテストに 1 サイクルを使うほどの価値があるか不明。`paths` グロブで自動ロードされ、かつ system prompt にデザインシステム参照ルールが埋まっていれば、builder/reviewer は UI 編集前に Skill 内容に触れる。次サイクルで効果を見てから判断
- **層 2（PreToolUse hook の additionalContext）も今回見送り**。理由: cycle-169 計画でも YAGNI 判定済み。hook を作る前に「skill + system prompt」の自然な配信で機能不足が出るか観察したい
- **採用: `paths` グロブ + system prompt 直書き**。Owner 指示「シンプルで分かりやすい」「長期に渡って改善」に最も合う

builder / reviewer の system prompt に追記する内容の主旨は T-08 の受け入れ基準で定義する（具体文面・項目数は builder の判断範囲）。趣旨は次のとおり: builder は UI 編集前に SKILL.md を Read することと、色・角丸・余白の新規ハードコードを避けること。reviewer は UI 変更レビューで SKILL.md の NEVER 節違反と Named Tone からの逸脱を確認すること。

### 検討した他の選択肢と判断理由

- **Skill のディレクトリ名**: 汎用名（design-guideline 等）vs プロダクト名（yolos-design 等）vs 機能名（frontend-design） → Owner 直接指示で `frontend-design` を採用。リポジトリはすべて yolos.net のものなので `yolos` という言葉自体は意味を持たない（Owner 指示）。略語（yolos-ds 等）は意図が読み取れず排除
- **コンポーネントディレクトリ名**: `ui/`（汎用的すぎ）vs `design/`（意図が広すぎ）vs `design-system/`（短く意図が直接読める） → 後者。Owner 指示「分かりやすい」「略語の排除」を満たす
- **CSS 変数命名**: 既存 `--color-*` 系の延長で命名 vs 素地由来の命名を踏襲 vs prefix 付きの独自命名 → 採用命名は builder 判断とし、「(1) 既存 `--color-*` との視覚的区別 (2) Owner が見て直感的 (3) サードパーティ衝突リスクが用途ドキュメントに明記」の 3 条件を満たす形で決定
- **既存 Header/Footer の改修**: 改修 vs 並行新設 → 並行新設。Owner 指示「既存サイトに手を加えない」を遵守
- **Storybook 導入**: 導入 vs `_dev/` 確認ページ → 後者。既存に Storybook なし、本サイクルの動作確認は最小ページで足りる
- **ダークモード**: 本サイクルで定義 vs 次サイクル送り → 次サイクル送り。コントラスト検証が別工数
- **素地ディレクトリの扱い**: 残置 vs 物理削除 → Owner 直接指示で物理削除。残せば本番デザインシステムの進化に伴い乖離・混乱の原因になる
- **ロゴアセットの扱い**: 素地内に残す vs リポジトリ慣例に従う恒久配置先へ移動 → 後者。素地が削除される以上、必要なアセットは恒久存在ディレクトリへ退避が必要。ただし実サイト読み込み配線は次サイクル送り
- **Stylelint / hooks**: 導入 vs 導入しない → 導入しない。Synthesis 原則 3 と cycle-169 合意（YAGNI）通り
