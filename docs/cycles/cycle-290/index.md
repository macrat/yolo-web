# サイクル-290

このサイクルでは、ブログ記事の**目次(TOC)アンカーが飛ばない不具合**（B-596 / cycle-287 監査 C4）を是正します。見出しのインラインコード内に `<body>` `</script>` `<a>` のような山括弧を含む記事で、目次の項目をクリックしても該当見出しへスクロールせず、読者が長い技術記事の中で目的の節へ辿り着けない状態になっています。あわせて、目次の**表示テキスト自体がコード部分（`<body>` 等）を丸ごと落として空白になる**副次の不具合も直します。原因は「DOM の見出し id」と「目次リンクの href」を**別々の2パスで別ロジックで生成**しているための食い違いなので、id 生成を単一の生成元（SSoT）へ寄せて食い違いが構造的に起きないようにします。

## 作業計画

### 目的

技術ブログの読者が、目次から目的の節へ確実に飛べるようにする。yolos.net のブログは長い技術記事が中心で、見出しにコード（`<body>` `</script>` `next.config.ts` 等）を含むのはこの領域の常態。目次は「長い記事のどこに何があるか」を示し、読みたい箇所へ一足で運ぶための道具であり、そこが飛ばないのは「地図はあるのに道が繋がっていない」状態にあたる。10人が目次を開いて目当ての節を押しても、山括弧入り見出しの節では手元が動かず、読者はスクロールで探し直す羽目になる。ここを直すことは、来てくれた読者を目的地まで確実に届けることそのもの。

### 作業内容

**不具合の構造（一次確認済み・2026-07-20）**

id は2つの独立したパスで生成され、山括弧を含むインラインコードで食い違う:

- **DOM 側（`markdownToHtml` → `createHeadingExtension`）**: marked の `parseInline` が `` `<body>` `` を `<code>&lt;body&gt;</code>` に描画（コード内の `<` は `&lt;` へエスケープ）→ `inner.replace(/<[^>]*>/g,"")` で**タグは剥がれるがエンティティ `&lt;body&gt;` は残る** → `generateHeadingId` で `&` `;` 等が落ちて **`ltbodygt`** という無意味な id になる。
- **TOC 側（`extractHeadings`）**: 生 markdown 行から**先にバッククォートを剥がして** `<body>` を露出させ、続く `replace(/<[^>]*>/g,"")` が **`<body>` を HTML タグとみなして丸ごと削除** → id からも表示テキストからも「body」が消える。

結果、`罠1: <body>...` の見出しは DOM id=`罠1-ltbodygt-の-...` / TOC href=`罠1-の-...` で不一致＝ジャンプ不能。加えて TOC 表示テキストも「罠1: の style...」と `<body>` が抜ける。プレーンなインラインコード（`useRegexWorker`）は両経路とも同結果になり id 一致（＝この既存 id は温存すべき）。

**是正方針（SSoT 化）**

個別パッチで両パスの結果を合わせにいくのではなく、**id と見出し情報の生成元を1つにする**。描画パス（`markdownToHtml`）が見出しを描画する際に `{ level, text, id }` を収集して HTML と一緒に返し、`blog.ts` はその収集結果を TOC に使う。`extractHeadings` の独立した第2パスは廃止する（または収集結果に委譲する薄いものにする）。これにより「DOM に実在する id」と「TOC の href」は同じ計算の同じ出力になり、将来どんな見出し記法が来ても食い違いが原理的に発生しない。

破損ケースの id/表示テキストは、コード内容を活かす向きに整える: `&lt;body&gt;` のようなエンティティを復号してから id/表示テキストを作り、id は `body`（山括弧は非英数として除去）・TOC 表示は `<body>` を含む形にする。プレーンなインラインコード見出しの既存 id は不変を厳守する（外部からの深いリンクの巻き添え防止。内部リンクは 0 件確認済）。

正確な API 形（`markdownToHtml` の戻り値の形・`extractHeadings` の去就・エンティティ復号の実装手段）は、実装前に `design-decision.md` に選択肢と判断理由を書いてから確定する。

### 検討した他の選択肢と判断理由

- **`extractHeadings` 側だけを描画パスに合わせて id を一致させる（2パス維持）**: 生成元が2つ残る限り、重複見出しの連番カウンタの同期前提など「食い違いの芽」が残り、将来の記法追加で再発しうる。cycle-287 監査が「共通化」を求めた趣旨に沿って、SSoT 化を選ぶ。
- **見出しにコードを書かない運用ルールにする**: 技術記事で `<body>` 等を見出しに出すのは読者にとって自然で有用。表現を削って不具合を回避するのは本末転倒（来訪者価値を毀損）。→ 記法は維持し、生成側を直す。
- **id を `ltbodygt` のまま TOC href だけ合わせる**: ジャンプは直るが URL ハッシュ・TOC 表示に無意味語が残り、共有時に読み手を惑わせる。コード内容を活かす復号を選ぶ。
