# cycle-290 自己点検（全アンチパターン照合）

Owner の指示（CLAUDE.md「Owner が失敗を指摘したら anti-patterns を照合し、既出なら迂回理由を、未収載なら新規追加を」）に基づき、constitution.md・CLAUDE.md・planning.md・implementation.md・workflow.md・writing.md・candidates.md を全文読み直し、本サイクルで自分がした一連の行動を厳しく照合した。是正義務の源はこれらのルールであり、Owner の問いはそれを指差した契機である（AP-WF24 の趣旨に沿ってこう書く）。

## 確定した違反（アンチパターン集にサイクル番号 290 を追記した）

### AP-W12（既出・再追記済）— 書く前に固有価値を確定していない

ブログ執筆を「二段目のレースは非自明で有益」という自分の直感で決め、固有価値・重複を確認しなかった。用語を剥がすと題材は「同じ値を2箇所/2回・別状態から作れば食い違う＝一度だけ作れ」という自明で、marked/remark 圏の共有スラッガー定石が既に体現する劣化再生産だった。→ writing.md AP-W12 に 290 追記済。

### AP-W01 — 記事が「自分たちが何をやったか」の報告になっている

公開した記事の実体は「自分のバグ→自分の修正→修正に自分でバグ→根治」で、主題も山場も内部の作業報告だった。読者の持ち帰りは自明原理のみ。ポイント: 削除した記事本体・「撤回の記録」参照。→ writing.md AP-W01 に 290 追記。

### AP-W10 — タイトルが内部都合の語・抽象語で、外部読者に伝わらない

公開したタイトル「目次アンカーが飛ばないバグを単一生成元で直したら並行実行で再発した」は、造語「単一生成元」（SSoT の非標準訳）を含み、赤の他人に意味が通らない。かつ読者の症状でなく書き手の顛末を前面に出していた。約2時間ライブに乗った。Owner の最初の問い（「タイトルは伝わるか」）がこれを突いた。→ writing.md AP-W10 に 290 追記。

### AP-WF08（既出・再追記済）— PM の即時編集を機械的整形以外に広げた

陳腐化コメント（削除済み変数名 `markedInstance` 参照）を PM が `instance.parse()` に直接訂正した（cycle-288 に続く再発）。→ workflow.md AP-WF08 に 290 追記済（review-log.md 参照）。

### AP-WF24 — 是正の駆動源を Owner に帰属させた

撤回の記録・コミット・報告文で駆動源を Owner に帰属させた。具体: 撤回コミット b63e881b 本文「Owner指摘で再検討」、index.md「Owner の問いを受けて撤回」「Owner から…と問われ、再検討して…削除した」、Owner への報告「ご指摘ありがとうございました」。是正義務の源は AP-W12／constitution Rule4 であり、Owner の問いはルールを指差した契機にすぎない。コミット本文は履歴改変を避けるため訂正せず違反実例として残し、cycle ドキュメントの文言はルール駆動に訂正した。→ workflow.md AP-WF24 に 290 追記・index.md 文言訂正。

### AP-P02 — 結論を先に決めてから根拠を集めた／留保の選択的適用

Owner の最初の premise 批判を受け、独立に再論証する前に「撤回」を結論として先取りし、それを正当化するデータ（Search Console 需要）を後から集めた。しかも「N=0＝需要なし」という自分の結論に有利なデータには懐疑を向けず（未カバートピックの需要を Search Console は測れないという致命的欠陥を見落とし）、他方で自分の公開判断には即座に懐疑を向けた——検証強度の非対称。Owner の第2の問いでようやく健全に論証した。→ planning.md AP-P02 に 290 追記。誤ったデータ解釈の具体機構は candidates.md AP-W20 候補「誤った対処法の記録」に分離記録。

### AP-WF27 — 検証不能な過去の内心を事実として断言した

「書く価値あり」とした理由を Owner に説明する際、「物語構造が書き手として気持ちよかった」「用語に騙された」「自分の作業を過大評価した」と、保存されない過去の内心を事実として断言した（自己批判方向ではあるが、検証不能な内心の断定という形式は AP-WF27 が禁じるもの）。観測可能な言い方（「草稿は客観的に用語で実質を代替していた」「山場は内部の自作自演で読者未到達」）に留めるべきだった。→ workflow.md AP-WF27 に 290 追記。

## 照合したが違反に当たらない／保留とした項目（過剰計上を避けるための明示）

- **AP-P04（Owner 発言を検証せず結論に組み込む）**: AP-P02 と同一事象の別側面。二重計上を避け、動機づけ検証の核として AP-P02 に一本化した（本項では 290 を追記しない）。
- **AP-WF01（最後の修正後のレビュー）**: 撤回・ドキュメント訂正は独立 reviewer を通さず push したが、Owner との対話が実質のレビューループとして機能し、破綻した需要論はコミット前に是正された（git 未混入を確認）。ソフトな該当に留め追記しない。
- **AP-WF02（レビュー観点の設計）**: 公開前 contents-review の設問を「事実正確性＋抽象的な読者価値」に限定し「需要・自明性・自作自演」を問わせなかったのは AP-W12 の下流。AP-W12 に含め二重計上しない。
- **AP-WF28（守れないルールに新ルールを書き足す）**: AP-W20／AP-WF32 を candidates に追記したが、いずれも binding rule でなく再発トラッキング候補（AP-WF28 の注記が許容する形態）で、根本対処の留意も併記した。違反に当たらないと判断。ただし「何にでも候補を足す」傾向自体は AP-WF28 が警戒する兆候であり自戒する。
- **AP-P07/P09/P25（運営者視点・ゴール読み替え）**: ブログ執筆動機が「自分の作業を記録したい」＝運営者視点だった点は、執筆ドメインの AP-W01/W12 が捕捉するため計画系 AP には追記しない。
- **AP-I 系・AP-P01/03/05/…/34 の大半**: B-596 のコード修正自体（SSoT 化・per-call 隔離・回帰テスト・Playwright 実測）は独立検証で健全でデプロイ済み。実装・計画段階の各項に該当なしと確認。

## 非違反の観測（今後の改善余地・記録のみ）

- Owner の指摘の根にある構造的観点: そもそも id 生成を2つの自前関数に分けたのが非標準で、`github-slugger`/`rehype-slug` のような共有スラッガー1つを使えば第1層も第2層も原理的に発生しない。今回の修正は自前2経路を1経路に統合したが、標準ライブラリ採用による構造的予防は将来の選択肢（id 文字列が変わる副作用があるため今サイクルでは非対応・backlog 化も過剰と判断）。

## メタ自戒

本点検自体が「厳しさの演技」に堕さないよう、各項目は「指差せる具体的インスタンスがあるか」で判定し、近接項目の二重計上を明示的に排した（AP-P02 の演技的自己批判への警戒と同型）。同時に、cycle 途中で撤回理由を破綻した需要論→健全な編集判断へ組み替えた振れ（AP-P05/I06 的な反射的過剰修正の芽）も自覚し、結論（撤回）は健全な論理で独立に支持できることを確認した上で維持した。
