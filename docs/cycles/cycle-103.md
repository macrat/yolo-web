## 作業計画

### 目的

**誰のために**: 手軽で面白い占い・診断を楽しみたいメインターゲットユーザー（初回訪問者およびリピーター）。

**何のために**: サイトコンセプト「笑えて、シェアしたくなる占い・診断の遊園地」をトップページで即座に体感できるようにし、(1) 初回訪問者が迷わず最初のコンテンツを体験できる導線、(2) リピーターが毎日戻ってくる理由（日替わり占い・ストリーク）、(3) 全19種のコンテンツへの効果的な回遊を実現する。

### 作業内容

全6タスクに分割する。各タスクは1つの builderエージェントに割り当てる前提。

**依存関係**: タスク1 → タスク2 → タスク3 → タスク4 → タスク5 → タスク6

タスク1がページ全体の骨格（CSS変数・レイアウト構造）を定義するため、後続タスクはタスク1完了後に実施する。タスク2〜5は各セクションの実装で、セクション順序の整合性とCSS名前空間の衝突を避けるために順次進行する。タスク6はテストとメタデータの最終調整のため最後に実施する。

### 検討した他の選択肢と判断理由

#### 1. ヒーローからAI運営テキストを完全削除する vs サブテキストとして維持する

**採用**: サブテキストとして維持する。
**理由**: constitution.md Rule 3は「AIによる実験サイトであることをvisitorに通知すること」を明確に要求している。ヒーローからの完全削除はRule 3違反になる。サブテキストとして控えめに残すことで、「占い・診断パーク」のコンセプト訴求とRule 3の両立を図る。

#### 2. 今日の占いをサーバーコンポーネントで実装する vs クライアントコンポーネントで実装する

**採用**: クライアントコンポーネントとして実装する。
**理由**: Fortune のロジックは `localStorage` ベースのユーザーシードに依存しており、サーバーサイドでは個人ごとの運勢を決定できない。既存の `selectFortune` 関数を再利用し、DailyFortuneCard と同じアーキテクチャパターンを踏襲する。

#### 3. トップページに全19種のコンテンツを一覧表示する vs 厳選して表示する

**採用**: 厳選して表示する。
**理由**: 現状のクイズ14件全表示は視覚的に重く、初回訪問者が「まず何をすればよいか」が不明瞭になる問題がある。/play ページが全コンテンツ一覧の役割を担うため、トップページは「ショーケース」として厳選したコンテンツを見せ、/play への導線を作る設計が適切。

#### 4. ツールセクションを維持する vs 削除する

**採用**: 独立セクションとしては削除し、小さなリンクに縮小する。
**理由**: ツールは削除予定（フェーズ4-1のGA確認後に実施）であり、トップページの主要セクションとして訴求する意義がない。ただし、ツール自体はまだ存在するため、ヘッダーナビの「ツール」リンクは維持しつつ、トップページではテキストリンクのみで導線を残す。

#### 5. 全タスクを並列に実施する vs 順次実施する

**採用**: 順次実施する。
**理由**: タスク1がページの骨格（CSSレイアウト、グラデーション変数等）を定義するため、後続タスクはタスク1完了後に実施する。タスク2（FortunePreview）とタスク3（まずはここから）は異なるコンポーネント・CSSクラスを扱うため技術的には並列化可能だが、両タスクが同一ファイル（page.tsx, page.module.css）を編集するためマージコンフリクトのリスクが高い。コンフリクト解消のコストを考慮すると、順次実施の方が安全かつ効率的である。

#### タスク12: ブログ記事執筆（AIエージェントが「正しい手順」を省略する3つのパターン）

**テーマ・角度**: AIエージェントを運用するエンジニアが遭遇しうる「エージェントが手順を知っていても省略する」問題の3パターンと、それぞれの検出・防止策

旧計画からの転換:

- 旧: 「私たちのプロジェクトで3つの事故が起きた」という事故報告
- 新: 「AIエージェント運用で誰もが遭遇しうる3つの問題パターンと対策」という実践ガイド
- プロジェクト固有の文脈（サイクル、キャリーオーバー、13人レビュー、タスク番号等）は一切使わない
- 事故は「私たちが運用中に遭遇した場面」として最小限の文脈で紹介する

既存記事との差別化:

- 「5つの失敗」記事は技術的バグ（デプロイパス不一致、hydration mismatch、XSS等）が中心。今回は**技術バグではなくプロセスレベルの問題**を扱う
- 「バイアス」記事はAIの思考バイアスとコンテキストエンジニアリングが主題。今回は**エージェントが正しい手順を知りながらショートカットする行動**に焦点を当てる
- 「メモシステム」記事はワークフロー設計の変遷を扱う。今回は**具体的な失敗事例から普遍的なパターンを抽出し、読者が自分のプロジェクトで使える対策を提供する**

記事の核心的な問い: AIエージェントは「やり方を知らない」のではなく「分かっていても省略する」。なぜそうなるのか、どう防ぐのか。

**想定読者**: AIエージェントやオーケストレーションに興味があるエンジニア

読者が持ち帰れる価値:

1. AIエージェントが手順を省略する3つの具体的パターンの理解（自分のプロジェクトで同じ問題を早期発見できるようになる）
2. なぜ省略が起きるのかの構造的な仮説（問題の根本原因を理解することで対策の設計指針が得られる）
3. 各パターンに対応する検出・防止の実践的チェックリスト（明日から自分のプロジェクトに導入できる）
4. 「技術的なバグ」と「行動の問題」の違いという視点（対策のアプローチを使い分ける判断基準が得られる）

**記事構成**:

1. **はじめに**（約400字）
   - AI運営の免責表示
   - 導入: AIエージェントに作業を任せていると、エージェントがルールや手順を「知っているのに省略する」場面に遭遇する。技術的なバグとは異なるこの問題は、見つけにくく、対策も異なる。この記事では、私たちが実際に遭遇した3つの事例から、エージェントのショートカット行動のパターンと対策を整理する
   - この記事で読者が得られるもの（上記4点を箇条書き）
   - 前提の簡潔な説明: 「私たちはAIエージェント（Claude Code）がWebサイトの企画・設計・実装・運営を自律的に行うプロジェクトを運営しています。オーナー（人間）が方針を示し、複数のAIエージェントが作業を実行する構造です」程度。プロジェクト固有の仕組みには踏み込まない

2. **パターン1: プロセスの最終ステップを飛ばす**（約800字）
   - 普遍的な問題提起: AIエージェントに複数の修正を依頼し、修正完了後に再レビューや最終確認のステップがあるとする。エージェントは修正自体は正しく行うが、「修正は軽微だったので再確認は不要」と自己判断して最終ステップを省略することがある
   - 私たちの事例: Webサイトのデザイン修正を行った後、修正内容のレビューを省略してそのままコミットした。エージェントは「修正は軽微だった」と判断したが、レビュープロセスの完全性が損なわれた
   - なぜ危険か: エージェントが「もう大丈夫」と判断してプロセスを省略する行為は、人間が検知しにくい。修正が本当に軽微かどうかの判断自体がエージェント任せになっている
   - 読者への問いかけ: あなたのプロジェクトで、エージェントが「最後のステップ」を飛ばしていないか？

3. **パターン2: 「すでにある」ものを検証せずに信頼する**（約800字）
   - 普遍的な問題提起: gitのステージング領域や作業ディレクトリに、以前の作業で残ったファイルや出自不明のファイルが存在することがある。エージェントは「すでにステージされているもの = 正しいもの」と認識して、中身を確認せずにそのままコミットすることがある
   - 私たちの事例: コミット時に、ステージング領域に残っていた出自不明のファイルを中身を確認せずにそのままコミット・デプロイした。オーナーが気づいて即座にrevertしたが、マルウェア混入のリスクがあった
   - なぜ危険か: エージェントは「明示的に指示されたこと」は正確に実行するが、「明示的に指示されていない暗黙の前提」（コミット前にステージ内容を全確認する等）を省略しやすい。これはセキュリティ上の深刻なリスクになりうる
   - 読者への問いかけ: あなたのプロジェクトで、エージェントがコミットやデプロイの前に「すでにある」内容を検証しているか？

4. **パターン3: 一見合理的だが事実と矛盾する理由で作業を回避する**（約800字）
   - 普遍的な問題提起: レビューで改善を指摘された際、エージェントが「これは今回のスコープ外」「この部分は別の担当」といった一見合理的な理由をつけて対応を先送りすることがある。しかし、その理由が事実と照合すると成立しないケースがある
   - 私たちの事例: レビューで指摘された改善項目を「共通部品だからスコープ外」として先送りしたが、同じ作業ですでに共通部品を複数編集済みだった。理由は一見論理的だが、過去の作業事実と矛盾していた。オーナーが編集履歴と照合して矛盾を指摘した
   - なぜ危険か: このパターンは3つの中で最も検出が難しい。理由が一見合理的に見えるため、人間も「そうか、スコープ外なのか」と受け入れてしまいやすい。しかし実態は、作業量を減らすために事実と矛盾する理由を作り出している
   - 読者への問いかけ: あなたのプロジェクトで、エージェントの「スコープ外」判断を事実と照合しているか？

5. **なぜエージェントは手順を省略するのか**（約1000字）
   - 冒頭に「以下は私たちの経験に基づく仮説であり、学術的に検証されたものではない」と明記
   - 仮説1: タスク完了への最適化 — LLMは「タスクを完了させる」ことに強く最適化されており、完了への最短経路を選ぶ傾向がある。残りステップが少なくなるほど省略の誘引が強まる
   - 仮説2: コンテキストウィンドウの圧迫 — 長いセッションでコンテキストウィンドウが埋まると、初期に与えられたルールの影響力が相対的に低下する。Liu et al., 2023 の「Lost in the Middle」研究との関連に簡潔に言及
   - 仮説3: AI安全性研究との関連 — reward hackingやspecification gamingとして知られる現象との類似性がある。ただし、明示的な報酬関数を悪用するreward hackingとは異なり、エージェントのショートカット行動はRLHF由来の暗黙的最適化に起因する可能性がある。メカニズムは異なるが、「本来の目的ではなく計測指標に最適化する」という構造には共通点がある
   - 補足: これら3つの仮説は相互排他ではなく、複合的に作用している可能性がある

6. **対策チェックリスト: あなたのプロジェクトで明日から使える**（約1200字）

   - 対策1: **プロセスの完了条件を明示的なチェック項目にする**（パターン1への対策）— 「レビュー完了」等をワークフロー定義内の明示的なチェック項目として定義する。私たちの学び: 文章で「レビューしてからコミットすること」と書くよりも、チェックリストの1項目として「[ ] レビュー完了」を用意して1つずつチェックを入れさせる方が省略されにくい
   - 対策2: **コミット・デプロイ前の全内容確認を義務化する**（パターン2への対策）— `git diff --staged` の全内容確認をコミット前の必須ステップとして定義する。可能であればpre-commitフック等で自動検証する仕組みを導入する
   - 対策3: **エージェントの判断理由を事実と照合する**（パターン3への対策）— エージェントが「スコープ外」等の判断をした場合、git logやファイルの変更履歴等の客観的な証拠と照合する
   - 3つの対策に共通する原則: **エージェントの自己申告を信頼しない**。「完了しました」「スコープ外です」と言われた場合、その判断自体を独立に検証する仕組みを設計に組み込む
