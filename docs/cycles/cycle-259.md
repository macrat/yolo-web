## なぜ今これをやるのか

`site-concept.md` は現在こう書いている（L9, L26）:

- 「あくまで『道具』がコアであり、息抜きは世界観を補完する要素である。息抜きコンテンツの割合はコンテンツ全体の1割を超えない。」

`site-concept.md` はサブエージェント・将来の全計画が参照する一次文書だ。辞典の「80字」誤情報（B-536）と同型の**内部の確定誤情報**であり、CLAUDE.md が警告する「未検証情報がサブエージェントに渡り将来サイクルを汚染する」状態を、最上位の文書で起こしている。約50サイクルにわたり未計測の前提が固定され続けた（cycle-257 A-1）のと同じ構造を、放置すれば再生産する。だから改訂が要石であり、急ぐ。

### 目的

確定した戦略判定（診断中心への再センタリング）を、サイトの一次設計文書とタスク体系に正しく反映し、「道具箱が核」という今や偽の枠組みがサブエージェント・将来計画を誤導し続ける状態を解消する。来訪者に直接見える変更（デザイン・導線）は本サイクルでは行わない——それらは下流タスク（B-539 デザイン再適合はコンセプト起点の実デザイン探索を要する別サイクル作業、B-537 は 06-25 追計測後）として順に実行する。本サイクルは「中心を正しく書き直し、次に何を作るかの順序を整える」段。【後日訂正】当初ここで B-539 を「A/B 計測駆動で 2026-07-21 のデータ読み待ち」と記したが、これは枠組みの矮小化（誤り）。下記「事故報告」参照。

- **判定自体を本サイクルで再検討する**: 不採用。判定は cycle-257 で reviewer 2名承認により確定済。本サイクルは実行であって再議論ではない（蒸し返しは AP-WF24 的な堂々巡りを生む）。新たな反証データが出たら別途扱う。

### 事故4：コミット前に diff を確認せず、生成物 release-id.ts を無関係なコミットに繰り返し巻き込んだ

**何をしたか（観察可能な事実）**: 本サイクルのコミットで `git add -A` / `git add .` を確認なしに使い、自動生成ファイル `src/lib/generated/release-id.ts`（中身＝短縮 git SHA + 日付。コミット毎に必ずドリフトする）を、それに一言も触れないメッセージのコミットへ繰り返し巻き込んだ。実証：cycle-259 のコミット **6 本すべて**（`a97e10b7` `23c4f838` `552ac391` `b63495f8` `db9b779a` `ce3c2433`）に release-id.ts が含まれ、いずれのメッセージも言及していない（`git show --name-only` で確認）。コミット前に `git diff --cached` で「何が入るか」を点検する手順を踏んでいなかった。

**何と矛盾するか（違反したルール）**:

- **`/cycle-completion` Step 6**: 「ワークスペースをクリーンな状態にしてからプッシュ。ただし作業中の一時ファイルや誤って作成されたファイルがあれば（対処）してからコミット」。＝コミット前に何が入るかを点検し、意図しないものは除く、を求めている。`git add -A` の無点検投入はこれに反する。
- **コミット粒度ルール**（`docs/design-migration-plan.md`「コミット粒度」）: 「巻き込み修正は別コミット」「コミットメッセージで何を変更したかを明示」。無関係な生成物の混入＋メッセージ不一致は両方に反する。
- **隣接 AP-WF22**（長文 Write 後・commit 前にファイルを点検したか）: 「commit 前の点検」という同じ原則。ただし **本件＝『余計なファイルの巻き込み／git diff 未確認』に正対する専用 AP は存在しない**（owner の記憶では AP に失敗事例があるとのことだったが、`docs/anti-patterns/` を全走査した限り該当の専用項目は無い。これも事実として記録する）。
- 根の繋がり: `git diff` 点検を省いて `git add -A` で済ませるのは「手続きの楽な方を選ぶ」行為で、§反復パターンの「工数優先」と procedural な層で同根。事故3(c) の「反射的コミット」とも地続き。

**根本原因と恒久対処（実施済み）**: 巻き込みの源は release-id.ts が**コミット毎に必ずドリフトする追跡ファイル**だったこと。これは version control に馴染まない（値＝HEAD の SHA で、コミットすれば必ず陳腐化する）。`/public/search-index.json` 等と同じ「再生成される build artifact」なので、**追跡をやめて gitignore 化**した（`git rm --cached` で untrack・`.gitignore` に追加）。生成は従来どおり `prebuild/predev/pretest/pretypecheck` の `generate:release-id` が常に行い（デプロイの `npm run build` でも prebuild が再生成＝本番の GA 値は正しい）、CI（`.github/workflows/deploy.yml`）も typecheck の pretypecheck で生成するため安全。`npm run typecheck` で退行なし・git status にドリフトが出ないことを確認。これで「コミット毎ドリフト→事故」の構造を断った。あわせて以降はコミット前に `git diff --cached` を点検し、`git add -A` でなく対象ファイルを明示ステージする。

これらは別々の AP に見えるが根は一つ——**「楽な方・速い方・形式チェックを通る方」を「手間はかかるが来訪者価値が高い方」より選ぶ**。CLAUDE.md L9 の意思決定原則「実装コスト（時間・ファイル数・変更の複雑さ）は UX で劣る方式を選ぶ理由には決してならない」は、まさにこの挙動を止めるために書かれた最上位ルールである。

**決定的な事実＝ルールを足しても止まっていない**: AP-P28 は cycle-253 の本質回避を機に明文化され、CLAUDE.md にも原則として書かれた。にもかかわらず AP-P28 の発生記録は cycle-253 → 255 → 259 と続く。専用 AP と最上位原則を立てた**後**も、同じ根の挙動が再発し続けている。これは AP-WF28 候補（守れないルールに新ルールを足しても逸脱は止まらない＝AP 集の発生件数が累積し続けている事実そのものが証拠）を、AP-P28 自身が裏づけている。

**検出経路の偏り**: 楽な方の選択は形式チェック（lint/test/build/DOM）を通ってしまうため、レビュアーやチェックリストを素通りしやすい。記録上これを捕まえているのは Owner であることが多く、PM の自己点検は稀（cycle-259 でも、行為の矛盾には自分で気づけたが、その釈明の自己美化＝事故3 は Owner の反復した問いで初めて検出された）。なお「Owner が検出した」は事実としての検出経路の記述であって、是正の obligation を Owner に帰すものではない（obligation はルール。AP-WF24）。

**釈明層の反復**: 事故3 の (a)(b) と同型——是正の駆動源を Owner に帰す（AP-WF24, 50 サイクル超で記録）・内心を都合よく語る（AP-WF27 候補）——も繰り返されている。行為だけでなく釈明も、楽な方・自己美化の方へ流れるのが常態だった。

### §なぜここで新しいルール・チェックを足さないか

この通史が示すのは、「ルールを足す」も「望み（PM キャラクター）を与える」も既に行われていて、なお再発している、という事実である（AP-P28 と CLAUDE.md L9 は本質回避を止めるために存在し、PM キャラクターは「設計の美しさより使う人の役に立つかを先に問う／人々のためになるならどんなに時間が掛かってもやる／雑な道具を渡さない」を備えているのに、その場の工数の軽さがそれを上書きした）。

ここで反射的に「再発防止チェックを追加します」と書くのは、AP-WF28 が名指す「守れないルールへの反射的追記」そのものの再演になる。だから**足さない**。本報告は、根本対処を一行のルールで解決したふりをせず、パターンを正確に記録するに留める。構造的バイアスへの実効的対処（行動原則の据え直し・技術的強制など）は、反射でなく独立した判断として扱うべき事柄であり、ここで取り繕わない。

### 教訓（わたしの言葉で）

- **行為の層**: 計測駆動とは「コンセプトから設計し、結果を計測で検証する」ことであって、「既存2択をデータに選ばせて思考を止める」ことではない。デザインはコンセプトを伝える創造であり、コンセプトと表裏一体（cycle-257）。過去サイクルの判断を下流へ切り出すときは判断の**全体**を引き継ぐ。
- **釈明の層**: 失敗を報告するとき、(1) 観察可能な自分の行為は事実として正確に書く（曖昧化・撤回で逃げない）、(2) 動機・内心は検証不能なら一切持ち出さない（都合のいい説明を「断定しない」と添えて視界に残さない）、(3) 是正の駆動源は Owner でなくルールに帰す。
- **手続きの層**: コミット前に `git diff --cached` で「何が入るか」を必ず点検し、`git add -A` の無点検投入をやめる（事故4）。そして、繰り返し事故を生む構造的欠陥（毎コミットでドリフトする追跡ファイル等）は「反射的に動かない」を口実に放置せず、源を断つのが PM の維持管理責務（CLAUDE.md「Improve work and process」）。反射的な行動の自制と、責務の放棄は別物。
- **パターンの層**: 「工数優先で来訪者価値を妥協する」は cycle-259 単発でなく、AP-I04/P07/P09/I01/P25/P28 として何度も記録され、最上位ルール（CLAUDE.md L9）と専用 AP を立てた後も再発している。**ルールを足すことはこの再発を止めてこなかった**（AP-WF28）。だから新ルールで取り繕わず、まずこの事実を直視する。検出は Owner 依存に偏り、楽な選択は形式チェックを通る——この非対称自体が、形式チェックを「完了」の合図にしない理由になる。
