# サイクル-284

cycle-279 のフェーズ R は全 81 ページ・全 36 ツールを一斉に「店構え」へ移行した。品質の担保は**トークン直書き検査と §8 禁止パターンの静的ゲート**だった。それは「墨の色が正しいか」は見るが、**「来訪者がその画面を実際に操作できるか」は一度も見ない**。

cycle-280 はその死角の実害を2件見つけた（一覧カードのクリック標的がタイトル文字列だけに縮んでいた／診断 meta に別診断のタイプ名が混入していた）。どちらも静的ゲートを素通りしている。cycle-280 はこれを「移行が UX を検証していなかった」という**根本**として記録し、B-573（緊急）を起票した。cycle-281 はそのうち「クリック領域＝視覚単位の全体」という規定のスライスだけを DESIGN.md に足した。**規定は足したが、実際に検証する手段は今も無い。**

本サイクルは、その死角そのものを塞ぐ。抽象的な「ベストプラクティスの全面適用」ではなく、**来訪者が実際にいる場所で、実際に操作して確かめる**。

## 作業計画

### 目的

**来訪者が実際に操作できる状態を、実際に操作して確かめ、できていないものを直す。**

フェーズ R は「見た目が店構えに揃っているか」を静的に検査して出荷した。だが来訪者はデザインを鑑賞しに来ているのではなく、**診断を最後まで進めて結果を受け取りに来ている**（実測: 全 PV の31%が単一の診断面）。設問がキーボードで選べない・focus が見えない・SR で選択肢が読めない・タップ標的が小さすぎるなら、その人はそこで詰まって帰る。それはデザインの瑕疵ではなく、**来訪者を締め出していること**である。

### 作業内容

上記 T1〜T6。順序は T1（測る手段）→ T2（測る）→ T3/T4（直す）→ T5（塞ぐ）→ T6（レビュー）。

**サブエージェントの使い方**（CLAUDE.md「Keep task smaller」）: T2 の監査は面ごとに1エージェント（character-personality／play+top／共通の器／word-sense-personality）。T3 の是正は欠陥1件＝1エージェント。Playwright・GA4 等の MCP を使うものは**すべてフォアグラウンド**で起動する（CLAUDE.md の規定）。

### 検討した他の選択肢と判断理由

- **B-583（店構え統一と印の要否の再検討）を採らなかった理由**: cycle-283 事故の本丸であり P1 だが、事故報告自身が「腰を据えて来訪者価値からやり直す本丸で、**圧の下で即断しない**」と定めた。加えて、B-583 が問うのは「共有・検索の瞬間に目に入る1枚の記号」であり、実測ではシェア流入はほぼゼロ・チャネルは検索が支配的（cycle-283 接地）。一方 B-573 が問うのは**すでに来ている来訪者の 85.8% が今まさに触っている操作面が使えるか**。人数と切実さの両方で B-573 が先。B-583 は次サイクル以降に、圧の無い状態で着手する。
- **B-575（結果体験そのものの価値＝本丸）を採らなかった理由**: 本丸だが、「結果体験の価値を上げる」は**結果に辿り着ける**ことが前提になる。辿り着けるかを検証していない現状で中身を盛るのは、土台を確かめずに上物を積むこと。T2 は character-personality の**結果面まで含めたフルフロー**を実際に通すので、B-575 の着手材料（実際に人が何に詰まるか）も同時に得られる。
- **B-573 を「全面掃討」として 81 ページ全部に広げなかった理由**: 全面を一度に宣言して雑に通すのは cycle-283 が踏んだ過剰宣言そのもの。実測が示すとおり来訪は操作面に極端に集中し、かつ**共通の器（のれん・品書き・Pagination）は全ページで共有される**ため、/play フローの監査は自然に大半の共有面をカバーする。残り（ブログ・辞典・個別ツールの固有 UI）は、T2 の所見が「共通器の構造欠陥」なのか「面ごとの個別欠陥」なのかを判定してから、次スライスとして起票する。**カバーしなかった範囲は正直に書く**（「全面適用済み」と書かない）。
- **axe を最初から CI ブロッキングにしなかった理由**: 全面が未検証の状態でゲートを赤にすると、通すこと自体が目的化して雑な抑制（ルール除外・aria の貼り付け）を招く。まず実態を測り、是正し、その後に維持ラインとして締める。

## 補足事項

**cycle-283 事故を踏まえた本サイクルの規律**（同じ病を繰り返さないための、私自身への拘束）:

1. **チェックを入れる根拠は「実機で再検証して直ったことを見た」ことだけ**とする。コードを書いたこと・レビューが承認したことを根拠にチェックを入れない（AP-WF23＝検証前の完了宣言の再発防止）。
2. **所見は「来訪者が何をできなくなっているか」で書く**。規格番号や内部語彙で書かない（AP-I01/AP-P07 の再発防止）。
3. **カバーしなかった範囲を正直に書く**。「全面適用」と書けるのは全面を実際に見たときだけ（AP-WF09＝形式的通過の再発防止）。
4. **PM 権限内の判断をオーナーに仰がない**（AP-WF24）。
5. **一次資料・実物で確かめてから断定する**。記憶・要約・過去サイクルの記述を事実として引用しない（AP-P04/WF12/WF27）。本計画の WCAG 記述と GA4 数値は、すべて本サイクルで実際に取得した。
