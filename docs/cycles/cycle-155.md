##### a) 初期トラフィック獲得の起爆剤

**結論: メインターゲット向け・サブターゲット向けの2軸で起爆剤を構成する。**

メインターゲット（手軽で面白い占い・診断を楽しみたい人）とサブターゲット（AI/Web技術に興味があるエンジニア）では、リーチ手段と行動動機が質的に異なる。単一の起爆剤に依存せず、各ターゲットに直接リーチする施策を明確に分ける。

**起爆剤A: メインターゲット向け -- バイラルシェア設計強化 [P1]**

メインターゲットに直接リーチする最も効果的な手段は、診断結果のSNSバイラルである。Z世代の31.3%が占い結果をSNSシェアする（リスミィ調査）というデータがあり、診断コンテンツは本質的にバイラル適性が高い。現在のシェアは月6セッション（X経由）に留まっており、シェア設計の改善余地が大きい。

現状の課題:

- シェアテキストが汎用的で好奇心を喚起しない（現在: `${quizTitle}の結果は「${result.title}」でした！あなたは? #${quizTitle} #yolosnet`）
- OGP画像がタイプ別に差別化されていない診断が多い
- catchphraseが未導入の診断が残っている（15種中5種が対応済み）
- X経由のWebトラフィックが構造的に減少している環境下で、リンク回避策が未対応

施策:

1. 全診断のシェアテキストをキャッチフレーズ組み込み型に刷新し、「自分もやりたい」を喚起する設計にする
2. OGP画像をタイプ別に差別化し、タイムライン上での視覚的訴求力を向上させる
3. 未対応診断へのcatchphrase展開を完了させる
4. Xのリンク抑制対策（リンクなしテキスト+リプライにリンク等）を検討・実装する

**起爆剤C: サブターゲット向け -- Qiita/Zenn技術記事投稿 [P1]**

yolos.netの最大の独自資産は「AIが全自律運営するWebサイト」という実験そのものである。エンジニア層への認知獲得と被リンクによるドメインオーソリティ向上を狙う。

ただし、以下の制約とリスクを認識する:

実現可能性の制約:

- **アカウント作成・管理にはOwnerの手動操作が必要**。Qiita/Zennのアカウント登録、メール認証、初期設定等はAI自律運営の範囲外であり、Ownerの協力が前提となる。投稿プロセスもOwnerを介する可能性がある。
- **投稿頻度はOwnerの稼働に依存する**。AI側で記事を作成できても、投稿操作がボトルネックになりうる。

リスクと対策:

- **AI生成コンテンツへのコミュニティの反感リスク**: Qiita/Zennのコミュニティでは、AI生成記事への批判が存在する。対策として、constitution Rule 3に基づき「AIが自律運営するサイトの記録である」ことを記事冒頭で明示し、AIが書いた記事であることを隠さない。これにより「AI自律運営の実験記録」というユニークな文脈が成立し、通常のAI生成コンテンツとは異なる価値を提供できる。
- **宣伝色への警戒**: 記事の価値提供を第一とし、yolos.netへのリンクは文脈に自然に配置する。「診断サイトの宣伝記事」ではなく「AI自律運営の技術記録」として読まれることが重要。

想定する記事テーマ:

- 「AIエージェントが155サイクル自律運営したWebサイトの全記録」（プロジェクト全体の紹介 / サブターゲット: AIエージェント興味層）
- 「Claude Codeで全自動サイト運営: 実践ノウハウと事故事例」（技術的詳細 / サブターゲット: AIエージェント興味層）
- 「AI自律運営サイトのGA分析で分かった"種なき畑への水やり問題"」（戦略の話 / サブターゲット: AIエージェント興味層）
- 「Next.js + AI自動生成で15種の診断コンテンツを作った設計」（実装の話 / サブターゲット: Webサイト製作を学びたいエンジニア）

注: サブターゲットには「AIエージェントやオーケストレーションに興味があるエンジニア」と「Webサイト製作を学びたいエンジニア」の2つが定義されている（docs/targets/参照）。本サイクルでは前者を主軸とするが、上記4番目の記事テーマにより後者にもリーチする。ただし、Webサイト製作ターゲット専用の施策は本サイクルの対象外とし、記事投稿の反応を見て後続サイクルで判断する。

サブターゲットからメインターゲットへの波及について:
エンジニア経由での診断バイラルには構造的弱点がある（エンジニアの来訪動機は技術的知見であり、診断シェア動機とは質的に異なる。フォロワーもエンジニアが多く、占い・診断好き層へのリーチは限定的）。そのため、Qiita/Zenn施策の主目的はあくまで被リンクによるSEO強化とサブターゲットの直接流入であり、メインターゲットへの波及は副次的効果として位置づける。

**起爆剤の優先順位:**

1. バイラルシェア設計強化（起爆剤A）: メインターゲットへの直接リーチ。実装後すぐに効果計測可能。
2. メインキーワードSEO（起爆剤B）: メインターゲットの検索流入。効果発現に3-6ヶ月要するが、最も持続的。
3. Qiita/Zenn技術記事（起爆剤C）: サブターゲット流入+SEO被リンク効果。Ownerの協力が必要なため、並行して準備。

##### b) 成長ループの設計

起爆剤で得たトラフィックを持続的成長につなげるため、2つのループを並行して回す。

**ループ1: SEOコンテンツループ（安定基盤）**

```
[メインキーワードSEO + 診断結果ページのインデックス化]
  → 「性格診断 面白い」等のヘッドキーワード + 「○○タイプ 性格」等のロングテール検索で流入
  → 診断を体験（エンゲージメント率100%・滞在3分46秒の既存実績）
  → 結果ページが新たな検索対象に
  → 内部リンクでサイト内回遊
```

- 根拠: Organic Search流入者の平均滞在7分28秒、エンゲージメント率60%（直近30日）。コンテンツの質が高いことはデータで証明済み。「発見されていない」問題を解決すれば成長する。
- ヘッドキーワード戦略: /play/ ページのメタデータ最適化+カテゴリ別LPで、メインターゲットのsearch_intentsに対応する。
- ロングテール戦略: 各診断結果タイプの解説ページ（例: 「逆張り楽観主義者とはどんな性格か」）でニッチな検索を捕捉する。現在、detailedContentを持つ結果ページは既にindex:trueになっている（コード検証済み）。対応済み診断は約10種、未対応が約5種。

**ループ2: SNSバイラルループ（成長加速）**

```
[ユーザーが診断を体験]
  → 結果が「シェアしたくなる」形式で表示される（キャッチフレーズ、差別化OGP）
  → ワンタップでXにシェア（最適化されたテキスト+OGP画像）
  → シェアを見た友人が「自分もやってみたい」と来訪
  → 新規ユーザーが診断を体験
```

- 根拠: Z世代の31.3%が占い結果をSNSシェアする（リスミィ調査）。NFX研究の「アイデンティティの投影」「検証」動機に診断は強く訴求。
- 16Personalitiesの成功: 4文字のMBTIコードが「共通言語」になった。yolos.netの診断結果にも「語りたくなるラベル」を付与する。
- Wordleの成功: スポイラーなしシェア設計。結果の核心を伝えつつ、全体は隠す設計が好奇心を刺激する。
- X経由トラフィックの構造的減少（2019年比83%減）に対応するため、リンクなし投稿（画像+テキスト）→リプライにリンクという回避策も考慮する。

**2つのループの関係:**

SEOループが安定基盤（継続的な検索流入）を提供し、バイラルループが成長加速（スパイク的な新規流入）を生む。どちらかに依存せず、両輪で回すデュアル戦略とする。

- **広告出稿**: 広告収入がないサイトに広告費をかける合理性がない。オーガニック施策で基盤を作る段階。

#### 成果物2: ウェブサイト要件定義

戦略を実現するためにサイトに必要な変更を要件として定義する。

##### 要件一覧

**REQ-1: 診断結果シェアテキストの最適化 [P1 / 戦略根拠: 起爆剤A・ループ2]**

なぜ必要か: 現在のシェアテキスト（`${quizTitle}の結果は「${result.title}」でした！あなたは? #${quizTitle} #yolosnet`）は、タイムライン上で「自分もやりたい」という好奇心を喚起する要素が不足している。競合（mgram: ハッシュタグ8個、16Personalities: 4文字コード）と比較して、結果の「語りやすさ」が低い。

要件:

- 診断ごとに最適化されたシェアテキストテンプレートを持つこと
- 結果の核心（キャッチフレーズまたはタイプの一言要約）がシェアテキストに含まれること
- ハッシュタグは1-2個に絞り、視認性を高めること
- Xのリンク抑制アルゴリズムに対応するため、リンクなしテキスト+リプライにリンクという選択肢も検討すること

**REQ-2: 診断結果OGP画像のタイプ別差別化 [P1 / 戦略根拠: 起爆剤A・ループ2]**

なぜ必要か: SNSでシェアされた際のXカード表示が診断の「顔」になる。全タイプ同一デザインでは、タイムライン上で結果の違いが視覚的に伝わらず、「自分は何が出るか」という好奇心が喚起されにくい。既にcycle-154でcontrarian-fortuneのOGP画像をタイプ固有色化した実績がある。

要件:

- 全15種の診断の結果ページOGP画像が、タイプごとに異なるビジュアル（色・レイアウト）を持つこと
- OGP画像にキャッチフレーズまたはタイプ名が視認可能なサイズで含まれること
- Xカードサイズ（1200x630px）に最適化されていること
- 注: B-218（AI生成画像によるOGP改善）およびB-219（画像生成MCPサーバー）は現在Deferred。本要件はそれらの前提なしで、テキスト+色ベースのOGP改善として定義する

**REQ-3: 診断結果ページのSEO最適化完了 [P1 / 戦略根拠: 起爆剤B・ループ1]**

なぜ必要か: 各診断結果タイプに対応するSEO可能なページが、ロングテール検索（「逆張り楽観主義者 性格」「天邪鬼 タイプ」等）の受け皿になる。

既存実装の検証結果:

- `/play/[slug]/result/[resultId]/page.tsx` のgenerateMetadataにおいて、`detailedContent`を持つ結果ページは既に`robots: { index: true, follow: true }`が設定されている。`detailedContent`がない結果ページは`noindex`になる。
- `generateStaticParams`により全クイズ x 全結果IDの静的ページが生成されている。
- メタデータ（title, description, OGP, twitter card, canonical URL）は結果ごとに動的生成されている。
- 約10種の診断でdetailedContentが実装済み。残り約5種（science-thinking, japanese-culture, word-sense-personality, kanji-level, kotowaza-level等）が未対応。

残課題:

- detailedContent未対応の診断への展開（これが完了すれば約160ページ全てがインデックス可能になる）
- 構造化データ（Schema.org）の追加が未実施
- 結果ページのtitleに検索キーワードとの親和性が低い可能性（現在は「結果タイトル | クイズ名の結果 | yolos.net」形式）

要件:

- 全診断の結果タイプにdetailedContentを実装し、全結果ページをインデックス可能にすること
- 構造化データ（Schema.org Quiz/FAQPage等）の追加を検討すること
- 結果ページのtitleにメインターゲットが検索しそうなキーワード（「性格」「タイプ」「診断結果」等）との親和性を持たせること
- 診断本体への内部リンク（CTA: 「あなたもこの診断を受けてみよう」）が結果ページに含まれること（実装済みだが、全ページで機能していることを確認する）

**REQ-4: メインターゲット向けSEOランディングページ [P1 / 戦略根拠: 起爆剤B・ループ1]**

なぜ必要か: メインターゲットのsearch_intents（「性格診断 面白い」「占い 面白い 無料」「おもしろ 診断」「MBTI 診断 無料」「暇つぶし 診断」等）に対応するランディングページが不在。現在の /play/ 一覧ページのtitleは「遊ぶ | yolos.net」であり、メインターゲットの検索キーワードとの一致度が低い。

要件:

- /play/ 一覧ページのtitle/descriptionをメインターゲットのsearch_intentsに最適化すること（例: 「無料の性格診断・おもしろ占い | yolos.net」等）
- 診断カテゴリ別（性格診断、占い、心理テスト等）のランディングページを作成し、カテゴリ固有のsearch_intentsに対応すること
- /play/daily のメタデータが「今日の運勢」キーワードに対応していることを確認・最適化すること
- 各ランディングページから個別診断への内部リンク構造が明確であること

**REQ-5: Qiita/Zenn投稿用技術記事の作成ガイドライン [P1 / 戦略根拠: 起爆剤C]**

なぜ必要か: 起爆剤Cの実行にはガイドラインが必要。AI自律運営の制約上、記事の正確性・魅力の品質管理プロセスと、Ownerとの協力体制を明確にする必要がある。

要件:

- Qiita/Zennへの投稿プロセス（記事作成→レビュー→**Ownerによる投稿操作**→効果計測）が定義されていること
- yolos.netへのリンクの適切な挿入方法（宣伝色を出さず、文脈に自然に配置する方法）が定義されていること
- 投稿頻度の目安（月1-2本、Ownerの稼働制約に応じて調整）が定義されていること
- 記事テーマの選定基準（エンジニア読者への価値提供が第一、yolos.netへの送客は第二）が定義されていること
- constitution Rule 3に基づく「AI運営であることの明示」方針が記事テンプレートに含まれていること
- AI生成コンテンツへのコミュニティ反感リスクへの対策（記事冒頭でのAI自律運営実験であることの明記、技術的価値の提供を主軸とする等）が定義されていること

**REQ-6: SNSシェアイベントの計測確認 [P2 / 戦略根拠: KPI計測]**

なぜ必要か: バイラルループの効果を測定するためには、シェアボタンのクリック数を計測する必要がある。

既存実装の検証結果:

- ShareButtons.tsx にて `trackShare()` 関数が既に実装されており、X/LINE/コピー/Web Share APIの各クリック時にGA4カスタムイベントが発火する。
- パラメータとして platform（twitter/line/clipboard/web_share）、contentType、contentIdが送信される。

残課題:

- 全診断の結果ページでcontentType/contentIdが正しく渡されているか確認が必要（propsとして渡されていない場合、trackShareが発火しない）
- GA4上でshareイベントが実際に記録されているかの確認が必要（イベントは実装されているが、発火条件を満たしていない可能性）

要件:

- 全診断の結果ページのShareButtonsコンポーネントにcontentType/contentIdが正しく渡されていることを検証・修正すること
- GA4上でshareイベントが正しく記録されていることをリアルタイムレポートで確認すること

**REQ-7: 診断結果の「会話のネタ」化要素 [P2 / 戦略根拠: ループ2]**

なぜ必要か: 16PersonalitiesのMBTI 4文字コード、mgramのハッシュタグ8個のように、診断結果が友人間の会話で引用されやすい「記号」を持つことが、バイラルの鍵である。

要件:

- 結果ページ上でそのラベルが視覚的に目立つ位置に配置されていること
- 注: cycle-150〜154の結果体験再設計でcatchphraseを導入済みの診断（yoji-personality, character-personality, impossible-advice, unexpected-compatibility, contrarian-fortune）は既に対応済み。未対応の診断への展開が必要

**REQ-8: 「友達と比較」機能の導線 [P3 / 戦略根拠: ループ2]**

なぜ必要か: 診断の最大のバイラル動機は「友達と結果を比較したい」という心理。music-personalityには既に相性判定機能があるが、その存在を結果ページ上で明確に訴求できていない可能性がある。

**REQ-9: デイリーコンテンツの強化（Wordle型共通体験） [P3 / 戦略根拠: ループ2]**

なぜ必要か: Wordleの成功要因の1つは「全員が同じ問題に挑戦する」共通体験と「1日1問」の希少性である。yolos.netには既にデイリー占い（/play/daily）が存在するが、現在は一方向的な結果表示であり、共通体験やシェア設計が不十分な可能性がある。

要件:

- デイリーコンテンツが「今日全員に共通のテーマ/問題」を提供すること
- 結果がスポイラーなしでシェア可能な形式であること（Wordleの絵文字グリッドに相当する設計）
- 「今日のうちにやらなきゃ」という適度な緊急性を持つ設計であること

### 検討した他の選択肢と判断理由

#### 1. UGC診断プラットフォーム化（不採用）

診断メーカーやkuizyのように、ユーザーが診断を作成・投稿できる機能を提供する選択肢。UGCはバイラルの最強エンジンであり、shindanmakerの成功がそれを証明している。

**不採用理由:**

- constitution Rule 4「量より質」に反する。UGCは品質管理が困難であり、低品質コンテンツの量産になるリスクが高い。
- 実装コストが極めて大きい（診断エディタ、モデレーション、ユーザー管理等）。coding-rules.md Rule 2「認証・ユーザー管理は実装しない」に抵触する。
- 15種の厳選された高品質診断という現在の強みを毀損する可能性がある。

#### 2. X公式アカウントの積極運用を起爆剤とする（不採用: 主軸としては）

X公式アカウントで日常的に投稿し、フォロワーを獲得して診断コンテンツへ誘導する選択肢。

**主軸として不採用の理由:**

- Xのアルゴリズムが外部リンク付き投稿を94%抑制する報告がある。外部サイトへのトラフィック誘導が構造的に困難。
- AI自律運営によるXアカウント運用の品質管理が難しい（不適切な投稿リスク）。constitution Rule 1（法令遵守）・Rule 2（有害コンテンツ禁止）への配慮が必要。
- 補助チャネルとしては採用。ユーザーがシェアしたときの受け皿として最適化する方向。

#### 3. 有料広告出稿（不採用）

Google Ads、X Ads等で診断コンテンツを直接宣伝する選択肢。

**不採用理由:**

- 広告収入がないサイトに広告費をかける合理性がない。
- CPAが不明確であり、効果測定の基盤も未整備。
- 先にオーガニック施策で基盤を作り、広告はスケールフェーズで検討すべき。

#### 4. 全診断の結果体験再設計を先に完了させる（不採用: 今すぐには）

B-273〜B-276（character-fortune、science-thinking、japanese-culture、word-sense-personality）の結果体験再設計を先に完了させてからトラフィック獲得に取り組む選択肢。

**不採用理由:**

- Ownerの指摘の核心は「種なき畑への水やり」。結果体験の改善はループ内の最適化であり、初期トラフィック獲得（種蒔き）にはならない。
- ただし、バイラルループの効果を最大化するためには結果体験の質が重要であるため、トラフィック獲得施策と並行して継続する。
