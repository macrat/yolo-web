### ① 来訪者価値の軽視

#### 正確性バグ（「同じ入力なのに結果が信用できない」= ターゲットの最大の嫌悪に直撃）

- **①-1 char-count と byte-counter が同一入力で値が食い違う**: 「あa😀」で文字数 4 vs 3、単語数 2 vs 1。`char-count/logic.ts` の `countChars`=`text.length`（絵文字を2と数える）、`byte-counter/logic.ts:18-21`=`Array.from(text).length`（1と数える）。単語数も `Intl.Segmenter` vs 空白split で不一致。さらに `char-count/meta.ts:23` は「コードポイント単位でカウント」と明記するのに実装は `text.length` で**ドキュメントと実装が矛盾**。byte-counter にはサロゲート検証テストがあるのに char-count にはない。
- **①-2 text-diff のサマリ件数がタイルと詳細ページで食い違う**: タイル `TextDiffTile.tsx:72-91` は行数、詳細 `Component.tsx:32-39` はハンク数を数える。同一入力で数字が変わる。詳細は「+N行」表示なのに実体はハンク数で**ラベルと中身が不一致**。
- **①-3 cron-parser の日本語説明が範囲・リスト指定で誤る**: `logic.ts:269-278` の `parseInt("9-17")` により `0 9-17 * * *` が「毎日 9時0分 に実行」と表示。`0 9,12,15 * * *` も「9時0分」に化ける。最も目立つ主役枠だけ誤り、次回実行リスト（9:00,10:00…）と自己矛盾。

#### 矛盾・誤誘導するUX

- **①-4 email-validator の既定表示が矛盾シグナル**: 既定 `test@gmial.com`（`EmailValidatorTile.tsx:51`）が緑「✓有効」バッジで大きく出て、直下に「もしかして: test@gmail.com」。`valid:true`→緑の構造（タイポを `suggestions` にしか入れない）のため、最大の価値であるタイポ提案を見落とさせる。全来訪者の第一印象がこれ。
- **①-5 image-resizer の GIF 誘導が誤り**: タイル `ImageResizerTile.tsx:110-113` がGIFを「詳細ページをご利用ください」と拒否するが、詳細ページもアニメGIF非対応で1フレーム目を無言出力。誘導先で目的を達成できないのに達成できるかのように案内＝来訪者を裏切る。
- **①-6 password-generator の強度バーが常に固定値**: `PasswordGeneratorTile.tsx:103` が `evaluateStrength(DEFAULT_OPTIONS)` を呼ぶため再生成しても永遠に「強い」固定（生成結果を反映しない）。動的に見えて不変＝狭い縦スペースの浪費。

#### 機能欠落・工数妥協

- **①-7 base64 が URL-safe Base64（`-`/`_`）非対応**: JWT・URLクエリで頻出なのにデコード失敗（`logic.ts:23-39`）。
- **①-8 html-entity の decode 取りこぼし**: セミコロン無し（`&copy`）・大文字（`&AMP;`）エンティティを変換せず素通り（`logic.ts:19-77`）。encode/decode 非対称。
- **①-9 qr-code のエラー処理不備**: 入力過大時にタイルは無言で失敗（`QrCodeTile.tsx:64-69`）、詳細ページは英語の生JSエラーを表示（`Component.tsx:88-92`）。hash-generator の日本語フォールバックと不整合。
- **①-10 image-base64 がサイズ情報・プレビューを削除**: `logic.ts:32-38` が `mimeType/originalSize/base64Size` を返すのにタイルは捨てている。「Base64化で約1.33倍に肥大」という判断材料を来訪者が得られない。
- **①-11 fullwidth-converter タイルが全オプションON固定**: 「英数字だけ半角化しカナは残す」（自らFAQで謳う頻出ニーズ）をタイルで満たせない（`FullwidthConverterTile.tsx:55,63`）。

#### 空状態・仕上げの雑さ

- **①-12 text-diff が入力前から「差分なし」と断定**: 空入力時に「テキストに差分はありません」と出て「比較完了して一致」と誤読させる。

## 作業計画

### 目的

cycle-200〜219 で蓄積された問題を可視化し、来訪者にとっての価値を回復するための立て直し計画を確定する。最優先は「道具箱というサイトコンセプトの達成」と「DESIGN.md への実質的な準拠」である。

### T-2: グルーピングと分析

上記の4分類が一次グルーピング。立て直し計画策定のため、全項目に**来訪者にとっての価値の観点**から重要度ラベル（緊急 / 高 / 中 / 低 / 対応不要）を付与した。判定軸は実装工数ではなく「来訪者の一日にどれだけ響くか」。ツール詳細ページは公開中のため、そこで起きている誤答・実害を最優先に置く。

#### 緊急（公開中の誤答・即時の実害）

- **①-1** char-count と byte-counter が同一入力で文字数・単語数が食い違い、meta と実装も矛盾（公開中ツールが矛盾値を返す）
- **①-3** cron-parser が `9-17` `9,12,15` 等の頻出式で日本語説明を誤り次回実行と自己矛盾（公開中の事実誤答）

#### 高

来訪者への誤誘導・実害があるもの、およびサイトの存在理由（道具箱＝タイルが本体）とデザインの土台に関わるもの。

- **①-2** text-diff の件数がタイル/詳細で食い違い・ラベルと実体不一致
- **①-4** email-validator 既定が緑「有効」とタイポ提案を同時表示＝矛盾シグナル
- **①-5** image-resizer の GIF 誤誘導＋アニメ無言消失

### T-3: 立て直し計画

本サイクルは計画策定のみで、修正作業は次サイクル以降。ここでは基本設計・方針レベルでロードマップを確定し、コードレベルの詳細には踏み込まない。

#### 立て直しの背骨（何を回復するのか）

立て直しの目的は、来訪者にとっての価値の回復であり、その中核は2つ。

1. **公開中の誤答をなくす**（来訪者の最大の嫌悪「同じ入力なのに結果が信用できない」への直撃を止める）。
2. **サイトコンセプト「道具箱＝タイルが本体」を実体化する**。タイルを劣化版の別実装に留めず本体化し、ツールページをタイルが見られ・使える場にする。DESIGN.md の根幹（Panel・共通コンポーネント）に実質準拠する。

この2つは依存関係が異なる。1は個別ツール単位で独立に直せる即効性のある実害是正。2は土台（設計方針の確定→共通コンポーネント化）を先に固めないと、個別タイルを直しても再び土台のないまま作り直す二度手間になる。したがって**「緊急ホットフィックスは土台再設計を待たず先行」「土台確定→個別タイル本体化の順」という二系統並行**で組む。

#### 設計上の最重要前提の再確認（②-1/②-2 の根本治療）

問題の根は「既存 Component を所与とし、タイルをその制約から逆算した劣化版の別実装にした」こと。立て直しでは**既存の Component / kind=widget 設計を所与にしない**。「タイルが本体、Component はそのフル機能版」ではなく、「タイルが本体であり、ツールページはそのタイル本体を据える場（＋必要な説明＋導線）」という関係に作り替える。これは Phase 2.2 が定義した「タイルは道具箱内で完結する UI 単位」という不変項を、ツールページにも適用するということ。kind（single/widget/multi）の再評価はこの再設計の一部であり、20ツールを「フル機能のままタイル化できるか」で独立再評価する（②-2 が指摘する無検証伝播の解除）。

ただしツールページには二面性がある。立て直しはこの二面性の実現方針自体を論点として扱い、決め打ちしない（後述「ツールページの二面性」）。

#### フェーズ構成とロードマップ

全問題を1サイクルに詰め込まない。重要度ラベル（緊急/高/中/低）と依存関係（②土台→①タイル本体化）に基づき、5フェーズに分割する。フェーズは概ね直列だが、Phase R0 は他フェーズと独立に先行できる。各フェーズは複数サイクルに分かれてよい（1ツール=1サイクルの粒度原則を維持）。

**Phase R0: 緊急ホットフィックス（公開中の誤答の停止）**

- **狙い**: 大規模再設計を待たずに、公開中のツール詳細ページが返す事実誤答・矛盾値を止める。来訪者が今日損しているものを最優先で消す。
- **対象**: ①-1（char-count/byte-counter の値矛盾＋meta矛盾）、①-3（cron-parser 範囲・リスト指定の日本語説明誤り）。いずれも「緊急」ラベル。
- **位置づけ**: Phase R2 以降でこれらのツールはタイル本体化の中で作り直されるが、それを待つと数サイクル分の誤答放置になる。よって**詳細ページ側（公開中の Component）に対する最小限の正答化のみ**を先行する。タイル設計には踏み込まない（踏み込むと R2 と二重作業になる）。R2 で作り直す際、このホットフィックスで確定した「正しい数え方・正しい説明ロジック」を仕様の起点として引き継ぐ。
- **①-19（char/byte 機能重複）との関係（手戻り懸念の解消）**: ①-19 の「char-count と byte-counter を統合するか / 片方を廃止するか」というツール統廃合の判断は R2（RB-7/RB-8 系）に委ね、**R0 では現状の2ツール構成を所与として、両ツールが同一入力で同一の文字数・単語数を返すよう値を一致させるだけ**にとどめる。仮に R2 で統合・廃止する結論になっても、R0 で確定した「正しい数え方」（コードポイント単位の数え方・単語分割ロジック）はその統合先タイルに引き継がれるため、R0 の作業（数え方の正答化）が無駄になることはない。R0 で byte-counter 側に行うのは数え方の整合のみで、ツール構成に踏み込む作業（統合・廃止のための再設計）は一切しない。
- **依存**: なし。即着手可。R1〜R4 と並行してよい。
- **完了条件**: char-count と byte-counter が同一入力で同一の文字数・単語数を返し、meta の記述と実装が一致する。cron-parser が範囲（`9-17`）・リスト（`9,12,15`）指定で日本語説明と次回実行リストが矛盾しない。各々に正答を保証する回帰テストがある。

#### ツールページの二面性の実現方針（論点・決め打ちしない）

ツールページには2種類の来訪者が来る（`docs/targets/` 準拠）。

- **「特定の作業に使えるツールをさっと探している人」**: 検索から直接ツールページに着地し、今すぐ使って結果をコピペし元の作業に戻りたい。likes=「開いた瞬間に入力欄が見えてすぐ使い始められる」。dislikes=「冒頭に長い解説、似たツールで迷う」。
- **「気に入った道具を繰り返し使っている人」**: タイルを見て道具箱に入れるか判断したい。likes=「操作性・トンマナの一貫性」。

タイル本体をツールページに据える際、この二面性を両立させる必要がある。R2-設計で以下の案を比較して決定する（決め打ち禁止）。

- **案A: タイル本体をページ主役に据え、その上下に最小限の説明＋道具箱導線を添える**。タイルが本体なのでページとタイルで機能が一致し、二度実装・機能劣後が原理的に起きない。「さっと使う人」は据えられたタイル本体をそのまま操作、「繰り返す人」は同じタイルを道具箱に追加できる導線を見る。
- **案B: ページ上部に「すぐ使う」用のタイル本体、下部に詳細な説明・FAQ・関連ツールを段組みで配置**（案Aの説明量を増やした variant）。
- **案C: タイル本体＋「道具箱に追加」CTA を主役にし、説明は折りたたみ/別タブで二次的に置く**。
- 各案を、二面性の両立度・実装の一貫性（機能劣後の再発防止）・「冒頭に長い解説を挟まない」dislikes 回避・SEO（説明テキスト量）の観点で比較し R2-設計で確定する。いずれの案でも**ツールページに表示されるのはタイル本体（または同一機能）であり、機能を絞った別実装ではない**ことを不変条件とする。

### 検討した他の選択肢と判断理由

立て直し計画の構造（フェーズ分割・順序・二面性の実現）について、最初に思いついた案だけでなく複数案を検討した結果を記録する（AP-P17）。

#### 論点1: 緊急バグの扱い（ホットフィックスを先行するか、土台再設計を待つか）

- **案1-A（採用）: 緊急バグ（①-1/①-3）を詳細ページ側で先行ホットフィックス（R0）し、土台再設計を待たない**。
- 案1-B: 土台（R1/R2）再設計の中で緊急バグもまとめて直す（先行ホットフィックスしない）。
- 案1-C: 緊急バグもタイル本体化の中で直すが、それまで該当ツールを一時的に非公開にする。
- **判断理由**: ①-1/①-3 は公開中ツールの事実誤答で、来訪者の最大の嫌悪（信用できない結果）に直撃している。土台再設計は数サイクルを要するため、案1-B では誤答が長期放置される。案1-C の非公開は「繰り返し使っている人」の dislikes（前回動いたものが動かない）に反する。案1-A は最小限の正答化で即座に実害を止め、R2 でその正答仕様を引き継ぐため二度手間も最小。ただしタイル設計に踏み込まず R2 と二重作業にしない線引きを明示した。

#### 論点2: 土台（共通コンポーネント化）と個別タイル本体化の順序

- **案2-A（採用）: 土台確定（R1 + R2-設計を一体）→ 個別タイル本体化（R2-実装）の順**。
- 案2-B: 個別タイルを直しながら共通部品を抽出していくボトムアップ。
- 案2-C: 土台と個別を完全に分離し、R1 を全て終えてから R2-設計に入る。
- **判断理由**: ②-1/②-2 の根本は「土台（タイルが本体という設計とその共通部品）が無いまま個別実装を量産し、無検証で伝播させた」こと。案2-B は同じ構造（土台不在のまま個別を進める）を再生産しかねない。一方、共通部品の最終形はタイル本体が何を要求するかに依存するため、案2-C のように R1 を完全に先に固めると R2-設計で手戻りする。よって R1 と R2-設計を相互依存として一体で確定し、その後に個別実装へ展開する案2-A を採った。

#### 論点3: ツールページの二面性の実現（前述の案A/B/C）

- ツールページにタイル本体を据える具体形（案A: タイル主役＋最小説明、案B: タイル＋下部に詳細説明段組み、案C: タイル＋道具箱CTA主役で説明は二次）を列挙した。
- **判断理由**: 本計画段階では「タイル本体をページに据える（機能を絞った別実装にしない）」という不変条件のみを確定し、A/B/C の最終選択は R2-設計サブフェーズに委ねる。理由は、説明量・SEO・二面性両立度のトレードオフは実タイルの寸法と GA4 到達経路データを見て判断すべきで、計画段階で決め打ちすると AP-P07（実装パターン参照で UI を決める）や AP-P20（過度に具体的な計画）に陥るため。複数案を残したまま R2-設計に渡すことを意図的な設計とした。

#### 論点4: コピーボタンの変換系の扱い

- 案4-A: 変換系も「コピーが主目的でない」として一律削除。
- 案4-B: 変換系は一律維持。
- **案4-C（採用）: 原則（コピーが主目的のみ設置）だけ確定し、変換系の個別可否は Owner 判断を仰ぐ**。
- **判断理由**: 変換系（url-encode 等）の出力は「他所で使う変換文字列」であり持ち帰り対象の側面と、短く選択コピー可能で知る対象の側面が混在する（②-15）。一律削除（4-A）も一律維持（4-B）も、ツール特性を見極める原則（②-16）に反して別の不統一を生む。線引きが原則だけでは一意に定まらないため、Owner 判断を仰ぐのが妥当（AP-P04 に留意し、Owner 発言を前提化せず判断を仰ぐ論点として明示）。

#### 論点5: B-314（既存タイル化タスク）の扱い

- 案5-A: B-314 を継続し、立て直しを B-314 の延長として進める。
- **案5-B（採用）: B-314 の前提（Component 所与・kind=widget・タイルは別 UI）を覆すため、RB-7 でアーキテクチャを再定義し B-314 を置換/廃止候補とする**。
- **判断理由**: B-314 と移行計画 Phase 8 は「タイル＝詳細ページとは別の簡素な UI（widget）」を許容する設計で、これがまさに ②-1 の根本原因を生んだ枠組み。案5-A は誤った前提を温存する。立て直しは前提自体を作り替えるため、B-314 を所与にせず再定義する案5-B を採った（AP-P03/AP-P11：過去の AI 判断を変更不可の制約として扱わない）。なお移行計画 design-migration-plan.md の Phase 8/Phase 10 記述自体も RB-7 で見直し対象になりうる旨は T-4 で backlog に申し送る。
