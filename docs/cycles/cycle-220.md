### ① 来訪者価値の軽視

#### 正確性バグ（「同じ入力なのに結果が信用できない」= ターゲットの最大の嫌悪に直撃）

- **①-1 char-count と byte-counter が同一入力で値が食い違う**: 「あa😀」で文字数 4 vs 3、単語数 2 vs 1。`char-count/logic.ts` の `countChars`=`text.length`（絵文字を2と数える）、`byte-counter/logic.ts:18-21`=`Array.from(text).length`（1と数える）。単語数も `Intl.Segmenter` vs 空白split で不一致。さらに `char-count/meta.ts:23` は「コードポイント単位でカウント」と明記するのに実装は `text.length` で**ドキュメントと実装が矛盾**。byte-counter にはサロゲート検証テストがあるのに char-count にはない。
- **①-2 text-diff のサマリ件数がタイルと詳細ページで食い違う**: タイル `TextDiffTile.tsx:72-91` は行数、詳細 `Component.tsx:32-39` はハンク数を数える。同一入力で数字が変わる。詳細は「+N行」表示なのに実体はハンク数で**ラベルと中身が不一致**。
- **①-3 cron-parser の日本語説明が範囲・リスト指定で誤る**: `logic.ts:269-278` の `parseInt("9-17")` により `0 9-17 * * *` が「毎日 9時0分 に実行」と表示。`0 9,12,15 * * *` も「9時0分」に化ける。最も目立つ主役枠だけ誤り、次回実行リスト（9:00,10:00…）と自己矛盾。

#### 矛盾・誤誘導するUX

- **①-4 email-validator の既定表示が矛盾シグナル**: 既定 `test@gmial.com`（`EmailValidatorTile.tsx:51`）が緑「✓有効」バッジで大きく出て、直下に「もしかして: test@gmail.com」。`valid:true`→緑の構造（タイポを `suggestions` にしか入れない）のため、最大の価値であるタイポ提案を見落とさせる。全来訪者の第一印象がこれ。
- **①-5 image-resizer の GIF 誘導が誤り**: タイル `ImageResizerTile.tsx:110-113` がGIFを「詳細ページをご利用ください」と拒否するが、詳細ページもアニメGIF非対応で1フレーム目を無言出力。誘導先で目的を達成できないのに達成できるかのように案内＝来訪者を裏切る。
- **①-6 password-generator の強度バーが常に固定値**: `PasswordGeneratorTile.tsx:103` が `evaluateStrength(DEFAULT_OPTIONS)` を呼ぶため再生成しても永遠に「強い」固定（生成結果を反映しない）。動的に見えて不変＝狭い縦スペースの浪費。

#### 機能欠落・工数妥協

- **①-7 base64 が URL-safe Base64（`-`/`_`）非対応**: JWT・URLクエリで頻出なのにデコード失敗（`logic.ts:23-39`）。
- **①-8 html-entity の decode 取りこぼし**: セミコロン無し（`&copy`）・大文字（`&AMP;`）エンティティを変換せず素通り（`logic.ts:19-77`）。encode/decode 非対称。
- **①-9 qr-code のエラー処理不備**: 入力過大時にタイルは無言で失敗（`QrCodeTile.tsx:64-69`）、詳細ページは英語の生JSエラーを表示（`Component.tsx:88-92`）。hash-generator の日本語フォールバックと不整合。

## 作業計画

### 目的

cycle-200〜219 で蓄積された問題を可視化し、来訪者にとっての価値を回復するための立て直し計画を確定する。最優先は「道具箱というサイトコンセプトの達成」と「DESIGN.md への実質的な準拠」である。

### T-2: グルーピングと分析

上記の4分類が一次グルーピング。立て直し計画策定のため、全項目に**来訪者にとっての価値の観点**から重要度ラベル（緊急 / 高 / 中 / 低 / 対応不要）を付与した。判定軸は実装工数ではなく「来訪者の一日にどれだけ響くか」。ツール詳細ページは公開中のため、そこで起きている誤答・実害を最優先に置く。

#### 緊急（公開中の誤答・即時の実害）

- **①-1** char-count と byte-counter が同一入力で文字数・単語数が食い違い、meta と実装も矛盾（公開中ツールが矛盾値を返す）
- **①-3** cron-parser が `9-17` `9,12,15` 等の頻出式で日本語説明を誤り次回実行と自己矛盾（公開中の事実誤答）

#### 高

来訪者への誤誘導・実害があるもの、およびサイトの存在理由（道具箱＝タイルが本体）とデザインの土台に関わるもの。

### T-3: 立て直し計画

本サイクルは計画策定のみで、修正作業は次サイクル以降。決定はすべて「今あるコード（全34ツールの `Component.tsx`）＋ cycle-200〜219 の20の失敗作」からの導出で**今確定**する。「後で決める」「GA4で判断」「作り直しの中で分かる」「案を残す」を一切行わない。実験は cycle-200〜219 で既に終わっており、要件データは目の前に揃っている。

#### この計画の唯一の核（不変前提）

**タイルとは「所定のデザイン（DESIGN.md のパネル）を当てた `<div>` に、そのツールのフル機能 UI を共通部品で組んだもの」であり、ただ一つの抽象である。** `<div>` に収まらない Web UI は存在しないので、「タイルに収まるか」「どう収めるか」という問いは成立しない。これを計画全体の不変前提として固定する。ここから直ちに次が従う。

- **ツールを分類してはならない。** アーキタイプ分類（A〜G 等）は破棄する。漏れ・重複のある分類を設計軸にすると必ず破綻する。これは `②-1` kind=widget の失敗の再来である。タイルは1種類であり、ツールごとに中身（コントロール）は違うが、形は一つ。
- **サイズ／収納の分析は一切不要・禁止。** 「フル機能が収まるか」「サイズ別の収納方式」「最難所はどれか」という問いは、核の前提（`<div>` には何でも収まる）の下で意味を持たない無用の心配である。寸法・タブ/アコーディオン/大型タイルといった「収納方式の確定」を計画に登場させない。
- 計画が扱うのは一点だけ ——「**単一の Tile 形に全ツールのフル機能をそのまま入れ、共通部品で所定のデザインを当て、実装を1つにし、道具箱とツールページの両方に同じタイルを描画する**」。

旧 T-3 はここを外し、(a) ツールをアーキタイプ A〜G に分類し、(b) アーキタイプ別に寸法・収納方式（タブ/上下分割/大型タイル）を「確定」し、(c) ツールページ提示方式を案A/B/C のまま「土台構築（B-489）」へ先送りしていた。(a)(b) は `②-1` と同じ「分類を設計軸に据えて劣化版を作る」構造の再発であり、(c) は先送り禁止に反する。本書はこの3点を破棄して全置換する。

#### 立て直しの根本診断（旧計画はなぜ破綻したか）

旧計画（破綻して廃棄済み）は「先に公開中の Component ページを正答化・デザイン是正（リフォーム）し、その後タイルを本体化して Component を取り壊し、さらに手を入れる」という構造だった。これは**直した家を取り壊してまた直す**計画で合理性を欠く。真因は、`①-1`（char/byte の値矛盾）`①-2`（text-diff の件数矛盾）`①-3`（cron 説明誤り）のような乖離と、タイル/詳細ページのデザイン不整合の**根が「Component と Tile という同じツールの二重実装そのもの」**であるのに、二重実装の解消を目標に据えず Component を温存したまま両側の症状を別々に直そうとした点にある。

事実確認: 全20ツールが `src/tools/<slug>/Component.tsx`（公開中・ツールページが描画）と `src/tools/<slug>/<Name>Tile.tsx`（道具箱用・noindex プレビューのみ）の**二重実装**を持つ（`find src/tools -name "*Tile.tsx"` = 20件、`find src/tools -name "Component.tsx"` = 34件で実機確認済み）。ツールページ `src/app/(new)/tools/<slug>/page.tsx` は一貫して `Component` を import し `Tile` は皆無。同じツールのロジックが2箇所にある以上、両者の乖離は「直し続ける」対象であって構造的に再発する。実装を1つにして初めて起こり得なくなる。

#### ツールページの提示方式の確定結論（案を残さず1つに確定）

サイトコンセプト（道具箱）とターゲット定義（`docs/targets/`）から導出して1つに確定する。

- ターゲット1「特定の作業に使えるツールをさっと探している人」: 検索から直接着地し、今すぐ使って結果をコピペし元の作業に戻る。likes=「開いた瞬間に入力欄が見えてすぐ使い始められる」、dislikes=「冒頭に長い解説、似たツールで迷う」。
- ターゲット2「気に入った道具を繰り返し使う人」: タイルを見て道具箱に入れるか判断する。likes=「操作性・トンマナの一貫性」。

両ターゲットの likes は**タイルそのもの**（即使える・一貫したトンマナ）が満たす。よって提示方式を次の1つに**確定**する（案A/B/C は残さない）。

> **ツールページは、そのツールのタイル（フル機能の単一実装）を主役（ファーストビュー）として描画し、その下に最小限の説明（1〜2文の用途）と道具箱への導線（「道具箱に追加」/ 関連ツール）を添える。** 長い解説・FAQ・SEO テキストはタイルより下に二次的に置き、ファーストビューを占有させない。

根拠: (i) タイル＝ページ本体なので機能劣後・二重実装が原理的に起きない（`②-3/②-4` の根治）、(ii) ファーストビューがタイルなのでターゲット1の「開いた瞬間使える」を最短で満たし「冒頭の長い解説」dislikes を作らない、(iii) 同一タイルがツールページと道具箱で一致するためターゲット2の「一貫性」を満たす、(iv) 説明・FAQ をタイル下に確保することで SEO テキスト量も両立する。ターゲット定義から一意に定まるため確定する。

#### 立て直しの不変原則（これを外したら失敗）

1. **タイルは1種類・分類しない**（核）。所定デザインを当てた `<div>` ＋ツールのフル機能 UI＝単一の抽象。サイズ・収納方式の問いは立てない。
2. **1ツール＝1実装**。タイルが本体であり、ツールページはそのタイルを描画する。重複実装である Component は**廃止する**。`①-1/①-2/①-3` のような乖離は「直す」のではなく実装を1つにして**起こり得なくする**。各ツール作り直しの完了条件に「当該ツールの Component が削除され、ツールページがタイルを描画し、実装が1つになっている」を必ず含める（追跡可能な成果物）。
3. **作って壊す／直して壊すをしない**。廃止予定の Component ページを先にリフォームしない。各ツールは**一度だけ**正しく作り直す。その1回で、フル機能化（`②-4`）・DESIGN.md 準拠（Panel/共通部品/トークン/フォーカス/角丸/トグル＝`②-6〜②-13`）・正しいロジック（`①-1/①-2/①-3` 等）・コピーボタン原則（`②-15/②-16`）・a11y（`②-9/①-18`）・hydration（`①-15`）・空状態（`①-12〜①-14`）・誤誘導解消（`①-4/①-5/①-6/①-9/①-11`）・機能欠落補完（`①-7/①-8/①-10`）を**まとめて**満たし、最後に Component を削除する。同じツールに二度手を入れない。
4. **実装コストを理由に劣った UX を選ばない**（CLAUDE.md）。
5. **先送りしない**。部品語彙・フル機能の源・提示方式は本計画で確定済み。Phase 8 是正の土台（B-488/B-489）以降は「決める」フェーズではなく「確定済み仕様どおり一度だけ作る」フェーズである。

### 検討した他の選択肢と判断理由

立て直し計画の構造について、最初に思いついた案だけでなく複数案を検討した結果を記録する（AP-P17）。アーキタイプ分類・サイズ/収納分析は核（タイルは単一抽象・`<div>` に何でも収まる）の下で意味を持たないため、代替案としても登場させない。

#### 論点1: 二重実装（Component と Tile）をどう扱うか ＝ 立て直しの背骨

- 案1-A: 公開中の Component ページを正答化・デザイン是正（リフォーム）し、その後タイルを本体化して Component を取り壊す（**旧計画。破綻として廃棄**）。
- 案1-B: Component を温存し、Component と Tile の両方を別々に正しく保つ（乖離を継続的に直し続ける）。
- **案1-C（採用）: 二重実装の解消そのものを不変原則かつ追跡可能な成果物に据え、各ツールを一度だけ作り直して Component を削除し、実装を1つにする**。
- **判断理由**: `①-1/①-2/①-3` の乖離とデザイン不整合の真因は「同じツールの二重実装」である。案1-A は廃止予定の家をリフォームしてから取り壊す捨て仕事を含み、かつ Component を温存する局面が残るため乖離の温床が消えない（旧計画が破綻した直接原因）。案1-B は二重実装を恒久化し、乖離を構造的に再生産し続ける（人手で2箇所を同期し続ける前提＝必ず再びズレる）。案1-C のみが、実装を1つにすることで乖離を「直す」のではなく「起こり得なくする」。完了条件に「Component 削除・実装が1つ」を明示することで、二重実装解消を検証可能な成果物にした。

#### 論点2: 緊急バグ（①-1/①-3）の扱い ＝ 捨て仕事を作らずに実害を止める

- 案2-A: Component の UI を作り込んで先行ホットフィックス（**廃止予定ページのリフォームになるため不可**）。
- 案2-B: 緊急バグも作り直し（B-490）まで放置し、その中で直す。
- 案2-C: 緊急バグもタイル化の中で直すが、それまで該当ツールを一時非公開にする。
- **案2-D（採用）: UI 非依存層（logic.ts 等）の正答化のみを緊急タスク（B-485/B-486）で先行し、作り直し後タイルが同じ logic を使うことで捨て仕事にしない**。
- **判断理由**: `①-1/①-3` は公開中の事実誤答で、来訪者の最大の嫌悪（信用できない結果）に直撃。作り直しは数サイクルを要するため案2-B では長期放置。案2-C の非公開は「繰り返し使う人」の dislikes（前回動いたものが動かない）に反する。案2-A は不変原則3（直して壊さない）に反する。案2-D は logic 層という UI と独立な層のみに介入し、その logic を作り直し後タイルが再利用するため捨て仕事にならない。`①-1/①-3` は logic 層で完結することがコードで確定済みのため、案2-D を一意に採れる（退避路の二択を残す必要がない）。

#### 論点3: 土台確定と個別作り直しの順序

- 案3-A: 個別タイルを直しながら共通部品を抽出していくボトムアップ。
- 案3-B: 土台（共通部品語彙の確定・準備＋タイルを描画するツールページの器）を先に作ってから個別作り直しに入るトップダウン（**採用**）。
- **判断理由**: `②-1/②-2` の根本は「土台（タイルが本体という設計とその共通部品）が無いまま個別実装を量産し、無検証で伝播させた」こと。案3-A は同じ構造（土台不在のまま個別を進める）を再生産しかねない。共通部品は"作り直しに使う部品の用意"なので先に固めても捨て仕事にならず（旧計画の R1=既存ページへのリフォームとは本質的に異なる）、むしろ各ツールでバラバラの独自実装が再発するのを防ぐ。よって案3-B を採った。

#### 論点4: ツールページの提示方式

- 案4-A: タイルを主役（ファーストビュー）に据え、その下に最小限の説明＋道具箱導線を添える（説明・FAQ・SEO テキストはタイルより下に二次配置）。
- 案4-B: 上部に「すぐ使う」用のタイル、下部に詳細な説明・FAQ・関連ツールを段組みで配置（説明量を増やした variant）。
- 案4-C: タイル＋「道具箱に追加」CTA を主役にし、説明は折りたたみ/別タブで二次的に置く。
- **採用＝案4-A**。
- **判断理由**: 両ターゲット（`docs/targets/`）の likes はタイルそのもの（即使える・一貫したトンマナ）が満たし、ターゲット1の最大の dislikes（冒頭の長い解説）はタイルを主役に据えれば回避できる。案4-B/4-C は説明やCTAを前に出す分だけファーストビューのタイル占有を弱め、「開いた瞬間使える」を遅らせる。SEO テキスト量は案4-A でもタイル下に確保でき両立する。ターゲット定義から一意に案4-A へ定まるため、後の工程への先送りはせず本計画で確定する（旧計画は案A/B/C を「土台構築（B-489）」まで持ち越しており、これは先送り禁止に反するため破棄した）。

#### 論点5: コピーボタンの変換系の扱い

- 案5-A: 変換系も一律削除 / 案5-B: 変換系は一律維持。
- **案5-C（採用）: 原則（コピーが主目的のみ設置）だけ確定し、変換系の個別可否は Owner 判断を仰ぐ**。
- **判断理由**: 変換系（url-encode 等）の出力は「他所で使う変換文字列」（持ち帰り対象）と「短く選択コピー可能で知る対象」の側面が混在する（`②-15`）。一律削除（5-A）も一律維持（5-B）もツール特性を見極める原則（`②-16`）に反して別の不統一を生む。線引きが原則だけでは一意に定まらず、最終線引きは権限の所在の問題のため Owner 判断を仰ぐ（AP-P04 に留意し、Owner 発言を前提化せず判断を仰ぐ論点として明示）。これは先送りではなく、原則確定と権限委譲の切り分けである。

#### 論点6: B-314（既存タイル化タスク）の扱い

- 案6-A: B-314 を継続し、立て直しを B-314 の延長として進める。
- **案6-B（採用）: B-314 の前提（Component 所与・kind=widget・タイルは別 UI）を覆すため、B-488（計画を単一抽象に更新）で前提を上書きし B-314 を置換/廃止候補とする**。
- **判断理由**: B-314 と移行計画 Phase 8 は「タイル＝詳細ページとは別の簡素な UI（widget）」を許容する設計で、これがまさに `②-1` の根本原因（二重実装と機能劣後）を生んだ枠組み。案6-A は誤った前提を温存する。立て直しは前提自体を作り替えるため B-314 を所与にせず上書きする案6-B を採った（AP-P03/AP-P11: 過去の AI 判断を変更不可の制約として扱わない）。

#### 論点7: B-314 廃止で宙に浮く「未移行14ツール＋遊び」のタイル化スコープの扱い

- 案7-A: 本立て直しの scope を移行済み20ツールに閉じ、未移行14ツール＋遊びは触れない（**＝`③-6` サイレントドロップの再発になるため不可**）。
- 案7-B: 本立て直しの scope 外として design-migration-plan の継続項目に残すだけにする（cycle-220 では追跡しない）。
- **案7-C（採用）: 当該スコープを B-490（未移行14＋遊び＝Phase 8.2 を含む）として本計画に明示的に引き取り、「対応不要ではなく後続で必ず扱う」ことを追跡可能な形で残す**。
- **判断理由**: B-314 は design-migration-plan Phase 8.1（ツール34ルート＝移行済み20＋未移行14）と Phase 8.2（遊び）を担っていた。B-488 で B-314 を置換・廃止する以上、未移行14ツール＋遊びのタイル化が宙に浮く。これは本サイクルが最も戒めた `③-6`（サイレントドロップ）の再発に直結する（案7-A）。案7-B は「別ドキュメントに残す」名目で実質的に追跡を手放すリスクがある。サイトコンセプト（道具箱＝全ツール・全遊びがタイル）に照らせば最終的に全件がタイル化されるべきであり、落とさず後続で必ず扱うことを B-490 に集約して可視化する案7-C を採った。これらは現状 Tile 未実装＝二重実装でないため「作って壊す」問題は生じず、B-489 の単一実装の土台をそのまま新規適用できる。優先度は移行済み20ツール（公開中の二重実装乖離の解消）より低いが、scope からは落とさない。
