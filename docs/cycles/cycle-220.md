### ① 来訪者価値の軽視

#### 正確性バグ（「同じ入力なのに結果が信用できない」= ターゲットの最大の嫌悪に直撃）

- **①-1 char-count と byte-counter が同一入力で値が食い違う**: 「あa😀」で文字数 4 vs 3、単語数 2 vs 1。`char-count/logic.ts` の `countChars`=`text.length`（絵文字を2と数える）、`byte-counter/logic.ts:18-21`=`Array.from(text).length`（1と数える）。単語数も `Intl.Segmenter` vs 空白split で不一致。さらに `char-count/meta.ts:23` は「コードポイント単位でカウント」と明記するのに実装は `text.length` で**ドキュメントと実装が矛盾**。byte-counter にはサロゲート検証テストがあるのに char-count にはない。
- **①-2 text-diff のサマリ件数がタイルと詳細ページで食い違う**: タイル `TextDiffTile.tsx:72-91` は行数、詳細 `Component.tsx:32-39` はハンク数を数える。同一入力で数字が変わる。詳細は「+N行」表示なのに実体はハンク数で**ラベルと中身が不一致**。
- **①-3 cron-parser の日本語説明が範囲・リスト指定で誤る**: `logic.ts:269-278` の `parseInt("9-17")` により `0 9-17 * * *` が「毎日 9時0分 に実行」と表示。`0 9,12,15 * * *` も「9時0分」に化ける。最も目立つ主役枠だけ誤り、次回実行リスト（9:00,10:00…）と自己矛盾。

#### 矛盾・誤誘導するUX

- **①-4 email-validator の既定表示が矛盾シグナル**: 既定 `test@gmial.com`（`EmailValidatorTile.tsx:51`）が緑「✓有効」バッジで大きく出て、直下に「もしかして: test@gmail.com」。`valid:true`→緑の構造（タイポを `suggestions` にしか入れない）のため、最大の価値であるタイポ提案を見落とさせる。全来訪者の第一印象がこれ。
- **①-5 image-resizer の GIF 誘導が誤り**: タイル `ImageResizerTile.tsx:110-113` がGIFを「詳細ページをご利用ください」と拒否するが、詳細ページもアニメGIF非対応で1フレーム目を無言出力。誘導先で目的を達成できないのに達成できるかのように案内＝来訪者を裏切る。
- **①-6 password-generator の強度バーが常に固定値**: `PasswordGeneratorTile.tsx:103` が `evaluateStrength(DEFAULT_OPTIONS)` を呼ぶため再生成しても永遠に「強い」固定（生成結果を反映しない）。動的に見えて不変＝狭い縦スペースの浪費。

#### 機能欠落・工数妥協

- **①-7 base64 が URL-safe Base64（`-`/`_`）非対応**: JWT・URLクエリで頻出なのにデコード失敗（`logic.ts:23-39`）。
- **①-8 html-entity の decode 取りこぼし**: セミコロン無し（`&copy`）・大文字（`&AMP;`）エンティティを変換せず素通り（`logic.ts:19-77`）。encode/decode 非対称。
- **①-9 qr-code のエラー処理不備**: 入力過大時にタイルは無言で失敗（`QrCodeTile.tsx:64-69`）、詳細ページは英語の生JSエラーを表示（`Component.tsx:88-92`）。hash-generator の日本語フォールバックと不整合。
- **①-10 image-base64 がサイズ情報・プレビューを削除**: `logic.ts:32-38` が `mimeType/originalSize/base64Size` を返すのにタイルは捨てている。「Base64化で約1.33倍に肥大」という判断材料を来訪者が得られない。
- **①-11 fullwidth-converter タイルが全オプションON固定**: 「英数字だけ半角化しカナは残す」（自らFAQで謳う頻出ニーズ）をタイルで満たせない（`FullwidthConverterTile.tsx:55,63`）。

## 作業計画

### 目的

cycle-200〜219 で蓄積された問題を可視化し、来訪者にとっての価値を回復するための立て直し計画を確定する。最優先は「道具箱というサイトコンセプトの達成」と「DESIGN.md への実質的な準拠」である。

### T-2: グルーピングと分析

上記の4分類が一次グルーピング。立て直し計画策定のため、全項目に**来訪者にとっての価値の観点**から重要度ラベル（緊急 / 高 / 中 / 低 / 対応不要）を付与した。判定軸は実装工数ではなく「来訪者の一日にどれだけ響くか」。ツール詳細ページは公開中のため、そこで起きている誤答・実害を最優先に置く。

#### 緊急（公開中の誤答・即時の実害）

- **①-1** char-count と byte-counter が同一入力で文字数・単語数が食い違い、meta と実装も矛盾（公開中ツールが矛盾値を返す）
- **①-3** cron-parser が `9-17` `9,12,15` 等の頻出式で日本語説明を誤り次回実行と自己矛盾（公開中の事実誤答）

#### 高

来訪者への誤誘導・実害があるもの、およびサイトの存在理由（道具箱＝タイルが本体）とデザインの土台に関わるもの。

- **①-2** text-diff の件数がタイル/詳細で食い違い・ラベルと実体不一致
- **①-4** email-validator 既定が緑「有効」とタイポ提案を同時表示＝矛盾シグナル
- **①-5** image-resizer の GIF 誤誘導＋アニメ無言消失

### T-3: 立て直し計画

本サイクルは計画策定のみで、修正作業は次サイクル以降。ここでは基本設計・方針レベルでロードマップを確定し、コードレベルの詳細には踏み込まない。

#### 立て直しの根本診断（旧計画はなぜ破綻したか）

旧計画は「先に公開中の Component ベースのページを正答化・デザイン是正（リフォーム）し、その後タイルを本体化して Component を取り壊し、さらに手を入れる」という構造だった。これは**直した家を取り壊してまた直す**計画であり、合理性を欠く。根本の見落としは、`①-1`（char/byte の値矛盾）`①-2`（text-diff の件数矛盾）のような実装の食い違いと、タイルと詳細ページのデザイン不整合の**真因が「Component と Tile という同じツールの二重実装そのもの」**であるのに、二重実装の解消を目標に据えず、Component を温存したまま両側の症状を別々に直そうとした点にある。

事実確認: 全20ツールが `src/tools/<slug>/Component.tsx`（公開中・ツールページが描画）と `src/tools/<slug>/<Name>Tile.tsx`（道具箱用・noindex プレビューのみ）の**二重実装**を持つ。ツールページ `src/app/(new)/tools/<slug>/page.tsx` は一貫して `Component` を import しており、`Tile` は皆無（例: char-count の page.tsx は `CharCountComponent` を描画）。同じツールのロジックが2箇所に書かれている以上、両者の乖離（`①-1/①-2`）は「直し続ける」対象であって、構造的に再発する。

#### 立て直しの不変原則（これを外したら失敗）

1. **1ツール＝1実装**。タイルが本体であり、ツールページはそのタイル本体を描画する。重複実装である Component は**廃止する**。`①-1/①-2/①-3` のような乖離は「直す」のではなく、実装を1つにして**構造的に起こり得なくする**。「Component の廃止・タイル本体への一本化」を不変原則かつ**追跡可能な成果物**とし、各ツール作り直しの完了条件に「当該ツールの Component が削除され、ツールページがタイル本体を描画し、実装が1つになっている」を必ず含める。
2. **作って壊す／直して壊すをしない**。廃止予定の Component ベースのページを、後で取り壊すのに先にリフォーム（正答化・Panel 化・デザイン是正）しない。各ツールは**一度だけ**正しく作り直す。その1回の作り直しの中で、フル機能化（`②-4`）・DESIGN.md 準拠（Panel/共通コンポーネント/トークン/フォーカス/角丸/トグル＝`②-6〜②-13`）・正しいロジック（`①-1/①-2/①-3` 等の正答化）・コピーボタン原則（`②-15/②-16`）・a11y（`②-9/①-18`）・hydration（`①-15`）・空状態（`①-12〜①-14`）・誤誘導解消（`①-4/①-5/①-6/①-9/①-11`）・機能欠落補完（`①-7/①-8/①-10`）を**まとめて**満たし、最後に当該ツールの Component を削除する。同じツールに二度手を入れない。
3. **二面性の維持**。ツールページには「さっと使いたい人」と「タイルを探して道具箱に入れるか判断したい人」の2種類が来る（`docs/targets/` 準拠）。両立は維持するが、その不変前提は「**ツールページに描画されるのはタイル本体（単一実装）であり、機能を絞った別実装ではない**」こと。具体形は決め打ちしない（後述）。
4. **実装コストを理由に劣った UX を選ばない**（CLAUDE.md）。

#### 設計上の最重要前提（②-1/②-2 の根本治療）

`②-1/②-2` が指摘する通り、kind=widget 判断は「Component を所与とし、タイルをその制約から逆算した劣化版の別実装にする」誤った枠組みに立っていた。立て直しでは**既存の Component / kind=widget 設計を所与にしない**。kind（single/widget/multi）は再評価対象であり、その判定基準から `②-1` が藁人形と断じた「128px に収まるか」を捨て、**実タイル寸法・実機計測に基づく基準**に置き換える。20ツールを「フル機能のままタイル本体化できるか」で独立再評価する（無検証伝播の解除）。

#### フェーズ構成とロードマップ

順序の背骨は **「土台＝単一実装アーキテクチャの確定」→「ツールごとに1回で作り直す」**。土台を固めずに個別タイルを量産すれば、`②-1/②-2` と同じ「土台不在のまま個別実装を伝播させる」構造を再生産する。一方、共通コンポーネントを"部品として用意・拡張"する作業は作り直しに必ず使われ無駄にならないが、"既存 Component ページに部品を当てるリフォーム"は廃止予定の家を直す捨て仕事なのでしない。

この原則の下で4フェーズに分割する。Phase F0（緊急バグ）のみ、捨て仕事を作らない範囲で先行を許す（後述）。Phase F1（土台確定）→ Phase F2（ツールごとの作り直し）が直列の本線。Phase F3（プロセス衛生）は全体と並行。各フェーズは複数サイクルに分かれてよい（1ツール=1サイクルの粒度原則を維持）。

**Phase F0: 緊急の公開中バグへの最小介入（捨て仕事を作らない範囲）**

- **狙い**: 公開中ツールが返す事実誤答・矛盾値（`①-1` char/byte 値矛盾＋meta 矛盾、`①-3` cron 範囲・リスト指定の日本語説明誤り）を、土台確定を待たずに止める。来訪者が今日損しているものを最優先で消す。いずれも「緊急」ラベル。
- **「直して壊す」を避けるための線引き（最重要）**: Component の UI を作り込む形の先行ホットフィックスはしない（それは廃止予定の家のリフォームになる）。介入は **UI 非依存層（`logic.ts` 等）の正答化に限定**する。作り直し後のタイル本体も同じ `logic.ts` を使う設計なら、この修正は捨て仕事にならず、正答仕様の起点として F2 に引き継がれる。
- **判断（捨て仕事を作らない観点）**: 各緊急バグについて F2 で「次のいずれが捨て仕事を最小化するか」を着手時に判定する論点として残す。(a) `logic.ts` 等の UI 非依存層の正答化のみを F0 で先行する、(b) 該当ツールの作り直し（F2）を最優先で前倒しし F0 を省く。原則は「`logic.ts` 修正だけで誤答が止まり、その logic を作り直し後タイルが使う」場合は (a)、「ロジック修正が UI と不可分でリフォームになってしまう」場合は (b)。`①-1/①-3` はいずれも数え方・パースのロジック層に起因するため (a) で止められる見込みが高いが、実装着手時に確認する。
- **`①-19`（char/byte 機能重複）との関係**: char-count と byte-counter を統合するか片方を廃止するかのツール統廃合判断は F1/F2 に委ね、F0 では「両ツールが同一入力で同一の文字数・単語数を返す」よう数え方を一致させるロジック修正だけにとどめる。統合・廃止に踏み込むとリフォームになるためしない。F0 で確定した「正しい数え方」は統合先タイルに引き継がれ無駄にならない。
- **依存**: なし。即着手可。F1〜F3 と並行してよい。
- **完了条件**: char-count と byte-counter が同一入力で同一の文字数・単語数を返し、meta の記述と実装が一致する。cron-parser が範囲（`9-17`）・リスト（`9,12,15`）指定で日本語説明と次回実行リストが矛盾しない。各々に正答を保証する回帰テストがある。Component の UI には手を入れていない（リフォームをしていない）。

**Phase F1: 単一実装アーキテクチャと共通部品の確定（土台）**

- **狙い**: 「タイルが本体／ツールページはタイル本体を描画する／Component は廃止する」という単一実装アーキテクチャを確定し、それを支える共通コンポーネントを部品として準備・拡張する。F2 でツールを1回で正しく作り直すための土台。**ここでは既存 Component ページにリフォームを当てない**（F1 の成果物は"設計の確定"と"再利用可能な部品"であって、廃止予定ページへの装飾ではない）。
- **F1 で確定すること（決め打ちしない論点を含む）**:
  - (i) **ツールページの二面性の実現方針**を案から選定（後述「二面性」の案A/B/C を比較）。不変前提＝描画されるのはタイル本体（単一実装）。
  - (ii) **kind 再評価の判定基準**を確定（128px 藁人形を捨て、実タイル寸法・実機計測ベース）。20ツールの再評価そのものは F2 各ツール着手時に実施。`①-19` の char/byte 統合是非もこの基準で結論づける枠組みに含める。
  - (iii) **タイル本体が持つべきフル機能の定義**（`②-4` で削られた機能の復元方針）。
  - (iv) **必要な共通コンポーネント（Panel / Button / Input / ToggleSwitch）の準備・拡張**。事実確認: これらは `src/components/` に実在するがツールページ（`src/tools/*/Component.tsx`）では未使用（`②-6/②-7`）。タイル本体化が要求する形に合わせて拡張し、トークン規定（`--shadow-button` / `--border-strong` / `outline-offset 2px` / border-radius トークン / `--fg-invert`＝`②-8/②-10/②-12/②-13`）と WCAG AA コントラスト（`①-16/B-440`＝`--accent`+前景色の値再調整は共通部品側に集約）を部品レベルで満たす。
  - (v) **`②-14`（準拠チェックの形骸化）への対策**として、DESIGN.md 根幹（Panel・共通コンポーネント使用、`outline:none` 不在など）を機械的に検出できる確認手段を確立する（grep ベースの不使用検出等。具体はフェーズ内で決定）。これにより F2 各ツールの「DESIGN.md 準拠」を実態検出可能にする。

- **対象**: `②-6/②-7`（共通部品準備）、`②-8/②-10/②-12/②-13`（トークン規定を部品に内包）、`①-16/B-440`（コントラスト＝グローバルトークン値是正）、`①-17/B-330/B-394`（ダークモード手法統一・テーマトグル追従・コントラスト精密検証）、`B-431`（共通部品の旧 `--color-*` トークン残存解消）、`②-14`（準拠検出手段の確立）、`②-9`（トークン/部品側で `outline:none` を起こさない方針確定。個別 UI 是正は F2）。
- **依存**: F0 と独立。F2 の前提（F2 は F1 完了後に着手）。
- **完了条件**: 単一実装アーキテクチャ（タイル本体描画・Component 廃止・二面性方針）が文書として確定。kind 再評価基準・フル機能定義が確定。共通部品（Panel/Button/Input/ToggleSwitch）がトークン規定と WCAG AA を満たす形で準備・拡張済み。グローバルトークンの `--accent`+前景色が両モードで AA（4.5:1）達成、ダークモードがテーマトグルに追従、旧 `--color-*` 参照ゼロ。DESIGN.md 根幹の機械的検出手段が動く。**この時点で既存 Component ページへのリフォームは行っていない**（部品の用意とグローバルトークン是正のみ）。

**Phase F2: 移行済み20ツールを1回で作り直す（タイル本体化・Component 廃止）**

- **スコープ境界（指摘A対応・最重要）**: 本フェーズの対象は **Tile を既に持つ「移行済み20ツール」**（base64 / byte-counter / char-count / cron-parser / email-validator / fullwidth-converter / hash-generator / html-entity / image-base64 / image-resizer / kana-converter / keigo-reference / line-break-remover / password-generator / qr-code / regex-tester / text-diff / text-replace / traditional-color-palette / url-encode）に**限定**する。これらは `①/②` の問題リストが実コードで指摘した対象そのものであり、二重実装（Component + Tile）が現存して乖離を起こしているツール群。**未移行14ツール（Tile 未実装）＋遊び（play）のタイル化は本フェーズの対象外で、後続の Phase F4 で必ず扱う**（落とさない。下記 F4 とスコープ表参照）。F2 完了条件の「Component.tsx 削除」は**この20件**を指す。
- **狙い**: `②-1/②-2` の根本治療。各ツールを F1 で確定した単一実装アーキテクチャで**一度だけ正しく作り直し**、最後に当該 Component を削除する。これにより二重実装が解消し、`①-1/①-2/①-3` 系の乖離が構造的に起こり得なくなる。
- **進め方**: 1ツール=1サイクルの粒度で、ツールを1件ずつ作り直す。順序は「緊急/高ラベル → 中ラベル」かつ GA4 PV 実測の高い順を基本（着手時に GA4 を確認）。各ツールの作り直しは相互に独立。
- **各ツール作り直しの完了条件（不変原則2・3を内包＝1回でまとめて満たす）**:
  1. タイル本体が Component と同等以上のフル機能を持つ（機能劣後ゼロ＝`②-4`）。
  2. ツールページがそのタイル本体を描画し（`②-3`）、二面性方針（F1 確定）に沿う。
  3. **当該ツールの Component.tsx（および関連の Component 専用 CSS）が削除され、実装が1つになっている**（不変原則1の追跡可能成果物）。kind 再評価の結論が記録されている。
  4. 当該ツールに該当する①・②の個別問題がすべて解消されている（下表）。
  5. hydration error ゼロ。当該タイルに回帰テストがある。
- **ツール別に拾う個別問題（サイレントドロップ防止のため明示）**:
  - image-base64: `①-10`（サイズ情報・プレビュー削除の復元）＋デコード機能復元（`②-4` 致命）。
  - cron-parser: ビルダー復元（`②-4` 致命）＋`①-3` の最終正答化（F0 で logic 先行した場合はタイル本体に継承）＋コピーボタン削除（`②-15`。F2 の作り直し時に外す）。
  - char-count / byte-counter: フル統計復元＋`①-1` 継承＋`①-19`（統合 or 片方廃止 or 2ツール維持を F1 基準で結論し実装）＋`①-13` 空状態。
  - text-diff: `①-2`（件数・ラベルの一致）＋`①-12`（空入力で「差分なし」誤表示の解消）＋コピーボタン削除（`②-15`）。
  - email-validator: `①-4`（緑「有効」とタイポ提案の矛盾シグナル解消）＋コピーボタン削除（`②-15`）。
  - image-resizer: `①-5`（GIF 誤誘導・アニメ無言消失の解消）。
  - password-generator: `①-6`（強度バー固定値→動的化）＋`①-15/B-469`（hydration）＋`②-11`（チェックボックス→トグル）。
  - base64: `①-7`（URL-safe Base64 対応）。
  - html-entity: `①-8`（decode 取りこぼし・encode/decode 非対称の解消）。

#### ツールページの二面性の実現方針（論点・決め打ちしない）

2種類の来訪者（`docs/targets/` 準拠）。

- **「特定の作業に使えるツールをさっと探している人」**: 検索から直接着地し、今すぐ使って結果をコピペし元の作業に戻りたい。likes=「開いた瞬間に入力欄が見えてすぐ使い始められる」。dislikes=「冒頭に長い解説、似たツールで迷う」。
- **「気に入った道具を繰り返し使っている人」**: タイルを見て道具箱に入れるか判断したい。likes=「操作性・トンマナの一貫性」。

不変前提＝**ツールページに描画されるのはタイル本体（単一実装）であり、機能を絞った別実装ではない**。その上で具体形を F1 で以下の案から選定（決め打ち禁止）。

- **案A: タイル本体をページ主役に据え、その上下に最小限の説明＋道具箱導線を添える**。タイルが本体なのでページとタイルで機能が一致し、二度実装・機能劣後が原理的に起きない。
- **案B: 上部に「すぐ使う」用のタイル本体、下部に詳細な説明・FAQ・関連ツールを段組みで配置**（案A の説明量を増やした variant、SEO テキスト量を確保）。
- **案C: タイル本体＋「道具箱に追加」CTA を主役にし、説明は折りたたみ/別タブで二次的に置く**。
- 各案を、二面性の両立度・実装の一貫性（機能劣後の再発防止）・「冒頭に長い解説を挟まない」dislikes 回避・SEO（説明テキスト量）の観点で比較し F1 で確定する。

### 検討した他の選択肢と判断理由

立て直し計画の構造について、最初に思いついた案だけでなく複数案を検討した結果を記録する（AP-P17）。

#### 論点1: 二重実装（Component と Tile）をどう扱うか ＝ 立て直しの背骨

- 案1-A: 公開中の Component ページを正答化・デザイン是正（リフォーム）し、その後タイルを本体化して Component を取り壊す（**旧計画。破綻として廃棄**）。
- 案1-B: Component を温存し、Component と Tile の両方を別々に正しく保つ（乖離を継続的に直し続ける）。
- **案1-C（採用）: 二重実装の解消そのものを不変原則かつ追跡可能な成果物に据え、各ツールを一度だけ作り直して Component を削除し、実装を1つにする**。
- **判断理由**: `①-1/①-2/①-3` の乖離とデザイン不整合の真因は「同じツールの二重実装」である。案1-A は廃止予定の家をリフォームしてから取り壊す捨て仕事を含み、かつ Component を温存する局面が残るため乖離の温床が消えない（旧計画が破綻した直接原因）。案1-B は二重実装を恒久化し、乖離を構造的に再生産し続ける（人手で2箇所を同期し続ける前提＝必ず再びズレる）。案1-C のみが、実装を1つにすることで乖離を「直す」のではなく「起こり得なくする」。完了条件に「Component 削除・実装が1つ」を明示することで、二重実装解消を検証可能な成果物にした。

#### 論点2: 緊急バグ（①-1/①-3）の扱い ＝ 捨て仕事を作らずに実害を止める

- 案2-A: Component の UI を作り込んで先行ホットフィックス（**廃止予定ページのリフォームになるため不可**）。
- 案2-B: 緊急バグも作り直し（F2）まで放置し、その中で直す。
- 案2-C: 緊急バグもタイル本体化の中で直すが、それまで該当ツールを一時非公開にする。
- **案2-D（採用）: UI 非依存層（logic.ts 等）の正答化のみを F0 で先行し、作り直し後タイルが同じ logic を使うことで捨て仕事にしない。logic 修正が UI と不可分でリフォームになる場合は、その代わり当該ツールの作り直しを最優先前倒しする**。
- **判断理由**: `①-1/①-3` は公開中の事実誤答で、来訪者の最大の嫌悪（信用できない結果）に直撃。作り直しは数サイクルを要するため案2-B では長期放置。案2-C の非公開は「繰り返し使う人」の dislikes（前回動いたものが動かない）に反する。案2-A は不変原則2（直して壊さない）に反する。案2-D は logic 層という UI と独立な層のみに介入し、その logic を作り直し後タイルが再利用するため捨て仕事にならない。logic と UI が不可分なツールは前倒し作り直しに切り替える退避路も用意した。

#### 論点3: 土台確定と個別作り直しの順序

- 案3-A: 個別タイルを直しながら共通部品を抽出していくボトムアップ。
- 案3-B: 土台（アーキテクチャ確定＋共通部品準備）を完全に終えてから個別作り直しに入るトップダウン（**採用**）。
- **判断理由**: `②-1/②-2` の根本は「土台（タイルが本体という設計とその共通部品）が無いまま個別実装を量産し、無検証で伝播させた」こと。案3-A は同じ構造（土台不在のまま個別を進める）を再生産しかねない。共通部品は"作り直しに使う部品の用意"なので先に固めても捨て仕事にならず（旧計画の R1=既存ページへのリフォームとは本質的に異なる）、むしろ各ツールでバラバラの独自実装が再発するのを防ぐ。よって案3-B を採った。

#### 論点4: ツールページの二面性の実現（案A/B/C）

- 案A: タイル主役＋最小説明 / 案B: タイル＋下部に詳細説明段組み（SEO テキスト確保）/ 案C: タイル＋道具箱 CTA 主役で説明は二次。
- **判断理由**: 本計画段階では「タイル本体をページに据える（機能を絞った別実装にしない）」という不変前提のみを確定し、A/B/C の最終選択は F1 に委ねる。説明量・SEO・二面性両立度のトレードオフは実タイルの寸法と GA4 到達経路データを見て判断すべきで、計画段階で決め打ちすると AP-P07（実装パターン参照で UI を決める）や AP-P20（過度に具体的な計画）に陥るため、複数案を残したまま F1 に渡す。

#### 論点5: コピーボタンの変換系の扱い

- 案5-A: 変換系も一律削除 / 案5-B: 変換系は一律維持。
- **案5-C（採用）: 原則（コピーが主目的のみ設置）だけ確定し、変換系の個別可否は Owner 判断を仰ぐ**。
- **判断理由**: 変換系（url-encode 等）の出力は「他所で使う変換文字列」（持ち帰り対象）と「短く選択コピー可能で知る対象」の側面が混在する（`②-15`）。一律削除（5-A）も一律維持（5-B）もツール特性を見極める原則（`②-16`）に反して別の不統一を生む。線引きが原則だけでは一意に定まらないため Owner 判断を仰ぐ（AP-P04 に留意し、Owner 発言を前提化せず判断を仰ぐ論点として明示）。

#### 論点6: B-314（既存タイル化タスク）の扱い

- 案6-A: B-314 を継続し、立て直しを B-314 の延長として進める。
- **案6-B（採用）: B-314 の前提（Component 所与・kind=widget・タイルは別 UI）を覆すため、RB-3 でアーキテクチャを再定義し B-314 を置換/廃止候補とする**。
- **判断理由**: B-314 と移行計画 Phase 8 は「タイル＝詳細ページとは別の簡素な UI（widget）」を許容する設計で、これがまさに `②-1` の根本原因（二重実装と機能劣後）を生んだ枠組み。案6-A は誤った前提を温存する。立て直しは前提自体を作り替えるため B-314 を所与にせず再定義する案6-B を採った（AP-P03/AP-P11: 過去の AI 判断を変更不可の制約として扱わない）。

#### 論点7: B-314 廃止で宙に浮く「未移行14ツール＋遊び」のタイル化スコープの扱い（指摘A）

- 案7-A: 本立て直しの scope を移行済み20ツールに閉じ、未移行14ツール＋遊びは触れない（**＝`③-6` サイレントドロップの再発になるため不可**）。
- 案7-B: 本立て直しの scope 外として design-migration-plan の継続項目に残すだけにする（cycle-220 では追跡しない）。
- **案7-C（採用）: 当該スコープを Phase F4 / RB-12 として本計画に明示的に引き取り、「対応不要ではなく後続で必ず扱う」ことを追跡可能な形で残す**。
- **判断理由**: B-314 は design-migration-plan Phase 8.1（ツール34ルート＝移行済み20＋未移行14）と Phase 8.2（遊び）を担っていた。RB-3 で B-314 を置換・廃止する以上、未移行14ツール＋遊びのタイル化が宙に浮く。これは本サイクルが最も戒めた `③-6`（サイレントドロップ）の再発に直結する（案7-A）。案7-B は「別ドキュメントに残す」名目で実質的に追跡を手放すリスクがある。サイトコンセプト（道具箱＝全ツール・全遊びがタイル本体）に照らせば最終的に全件がタイル本体化されるべきであり、落とさず後続で必ず扱うことを RB として可視化する案7-C を採った。これらは現状 Tile 未実装＝二重実装でないため「作って壊す」問題は生じず、F1 の単一実装アーキテクチャをそのまま新規適用できる。優先度は F2（公開中の二重実装乖離の解消）より低いが、scope からは落とさない。
