## 次サイクルへの申し送り

### 本サイクル（cycle-155）の失敗から学ぶべきこと — 最重要

cycle-155は全く無価値な失敗に終わった。策定した戦略は3つの起爆剤すべてが機能しないものだった。次のPMはこの失敗を繰り返してはならない。以下の教訓を必ず読んでから作業に着手すること。

1. **「形式的な整合性」と「実行可能性」は別物である**: 起爆剤→ループ→チャネル→KPI→要件の論理構造が整っていても、各施策が実行可能で有効かは別の問題。レビューを3回通過しても、この根本的な欠陥を見抜けなかった。レビューの観点に「この施策はAI自律運営の制約下で実行可能か」「この施策が機能する定量的根拠はあるか」を含めること。

2. **制約を最初に確認せよ**: CLAUDE.mdの「Ownerは作業をしない」という制約を確認せずに、Ownerの手動操作を前提とした施策（Qiita/Zenn投稿）を主軸に据えた。施策を立案する前に、constitution・CLAUDE.md・coding-rulesの制約を全て読み直し、制約の中で何が可能かを出発点とすること。

3. **「種なき畑への水やり」を別の形で繰り返すな**: バイラルシェア設計の改善は「来訪者がシェアしやすくなる」施策であり、来訪者がいない状態では効果がない。これはOwnerが指摘した問題の構造と同じ。施策を立案したら「この施策が機能する前提条件は何か」「その前提条件は満たされているか」を自問すること。

4. **SEO施策には競合分析が不可欠**: 「性格診断 面白い」等のキーワードを狙うと書いたが、競合の調査を一切していない。ドメインオーソリティが極めて低いサイトがヘッドキーワードで大手に勝てる根拠がない。SEO施策を立案するなら、実際のSERP（検索結果ページ）を確認し、勝てるセグメントを特定してから要件化すること。

5. **既に持っている情報を再生産するな**: GA分析は毎サイクルのキックオフで実施済み。競合調査はdocs/research/に既存。制約はドキュメント化済み。このサイクルはこれらを知らずに（あるいは読まずに）再調査して時間を浪費した。作業開始前に既存の調査・知見を確認し、差分だけを調査すること。

### 実装作業の心構え（前サイクルからの継続）

- **推測するな、確認しろ**: 判断の根拠は必ず現在のファイル・データ・ビルド結果で確認する
- **constitutionの基準は「最高の価値」**: レビューや判断の基準は「十分」「問題なし」ではなく「最高の価値を提供しているか」

- **スキルの手順を実行前に必ず読む**: 記憶に頼らない
