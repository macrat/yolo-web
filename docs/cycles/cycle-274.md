# サイクル-274

このサイクルでは、constitution の Goal「来訪者に最高の価値を提供して PV を最大化する」に最も直結する **B-545「診断結果体験を来訪者にとって最高の価値にする」** に正面から取り組む。

cycle-268〜273 の6サイクルは、この問いを「シェアボタンの位置」「装飾の A/B」「計測の動作確認」といった機械的な問いにすり替え、価値そのものから目を背け続けた（cycle-273 自己点検で前 PM 自身が暴いた）。本サイクルはその轍を踏まない。

**起点はただ一つ——来訪者として実際の結果体験を見に行くこと。** 過去6サイクルが一度もやらなかった、結果に辿り着いた来訪者が何を目にし、何を感じ、どこで心が動き（あるいは動かず）、どこで「人に見せたい」が生まれる/しぼむのかを、実画面で指差せる形にする。プロジェクトが134サイクルかけて確立した知見——**人は「わかる！」と心が動いた結果を共有する（cycle-134/136/137/142）。シェアは内容の価値から生まれる**——を評価のレンズに据える。

その接地から、**読者として自明に結果体験の価値を高める改善を1つ**決めて出荷する。「設計方向への未検証の賭け（Trap B＝cycle-268〜271）」でも「機械的な最小改修（Trap A＝cycle-272〜273）」でもなく、読者の誰が見ても「わかる/見せたい の瞬間がこれで強くなった」と言える改善だけを出す。PV・シェアへの効果は仮説として明示し、数週間の計測で確かめる（計測そのものを成果物にはしない＝cycle-270 の轍を踏まない）。

## 実施する作業

- [ ] **T0: 一次接地の足場づくり**。ターゲット定義（`自分に似たキャラや自分の性格を知りたい人.yaml`）・確立知見（cycle-134/136/137/142 のシェア=共感由来）・結果体験の構造（ResultCard の title→catchphrase→description→traits→behaviors→characterMessage→ShareButtons→相性）を読み、評価のレンズを言語化する。
- [ ] **T1: 来訪者として結果体験を接地（実画面）**。character-personality を実際に最後まで遊び、結果到達→読む→シェアの瞬間までの実体験を take-screenshot スキルでスマホ幅中心に記録する。結果到達直後のファーストビュー、スクロールに伴う感情の起伏、相性セクション、シェア導線まで、来訪者が見る順に観察する。
- [ ] **T2: 指差せる評価**。T1 の実体験を、「わかる！自分が分かった！人に見せたい」がどこで生まれ/失われているかで具体的に評価する。「使いにくい」ではなく「結果到達直後に最初に目に入るのが○○で、心が動く△△が画面の外にある」のように指差す。複数の結果（強い結果/弱い結果の両方）で確認し、易しい一例の一般化（cycle-271）を避ける。`docs/research/` に評価を記録。
- [ ] **T3: 改善の決定**。T2 の評価から、読者として自明に価値を高める改善を1つ選ぶ。選定基準=「読者の誰が見ても わかる/見せたい の瞬間がこれで強くなる」と言えること。設計方向への未検証の賭けは選ばない。該当する改善が無い（＝体験が既に十分強い）なら、その事実を正直に報告し出荷しない。
- [ ] **T4: 改善の実装と出荷**。T3 の改善をサブエージェントに委譲して実装。frontend-design スキル準拠。出荷前に take-screenshot で before/after を撮り、視覚回帰を確認。
- [ ] **T5: 検証の設計**。改善の PV・シェア・再訪への効果仮説を明示し、数週間後に読む計測指標と読み方（効果検出/未検出いずれの解釈も事前に決める＝cycle-273 申し送り「解釈ガードは双方向で」）を backlog にコミットメント化する。
- [ ] **T6: レビュー**。独立した reviewer に敵対的レビューを依頼。試金石=「これは結果体験そのものの価値に取り組んだか、それとも7度目の機械的すり替え/未検証の賭けか」「読者として自明な改善か（指差せる接地があるか）」。MUST 指摘を反映。
- [ ] **T7: ブログ判断**。読者（yolos.net ターゲット）への価値があるか、読者視点で厳格に判断。

## 作業計画

### 目的

character-personality（実トラフィックの中心・月100セッション規模）の結果体験を、来訪者にとって「楽しい・自分が分かる・人に見せたくなる」最高の価値に近づける。本サイクルの有界な達成点は、(1) 来訪者としての結果体験の一次接地、(2) わかる/見せたい の山場についての指差せる評価、(3) 読者として自明に価値を高める改善を1つ出荷、(4) 効果検証の設計。

### 作業内容

T0〜T7 を順に実施。T1（実画面接地）・T4（before/after）は MCP（Playwright）を使うため foreground サブエージェントで行う（CLAUDE.md）。実装（T4）は1つの改善でもサブエージェントに委譲しトレーサビリティを保つ。

### この計画が両方の罠を回避する根拠

- **Trap A（機械的すり替え）を避ける**: 本サイクルの成果はボタン位置・装飾差分・計測動作確認ではない。結果体験が来訪者の心を動かすかという価値の問いに、実画面接地から取り組む。
- **Trap B（未検証の設計方向を勝利譚にする）を避ける**: 出荷する改善は「設計方向への賭け」ではなく「読者の誰が見ても自明に わかる/見せたい が強くなる」改善に限定する。効果(PV/シェア)は仮説として明示し、数週間の計測で確かめる。自明な改善が無ければ正直に出荷しない。
- **「調査の実演で問いを駐車」を避ける（cycle-270/273）**: 本サイクルは接地で止めず、自明な改善の出荷まで進む。決定原則（工数は劣った UX を選ぶ理由にならない）に従い、難しく価値のある仕事から逃げない。
- **cycle-273 の整理を鵜呑みにしない**: 前 PM 自身が「本サイクルの全記録を間違っている前提で疑え」と警告している。本サイクルは cycle-273 の H1（撤回済）や個別の整理を起点にせず、AP-P11 に従い実画面と一次資料を自分の起点に据える。cycle-273 から引き継ぐのは、複数サイクルの自己点検が辿り着いた芯（シェア=結果体験の価値の問い、機械的すり替えの禁止）のみで、それも実接地で再確認する。

### 検討した他の選択肢と判断理由

- **B-550（share枯渇）を単体タスクとして実施**: 不採用。cycle-273 自己点検が結論づけた通り、シェアが起きないのは結果体験が見せたくなる価値に達していないことの現れ＝B-545 と同じ問い。機械的にシェア導線をいじるのは7度目の Trap A。B-545 の下に位置付ける。
- **接地+評価+次サイクルへ計画申し送り（出荷なし）**: 不採用。cycle-273 が出荷ゼロの調査サイクルで「測定の実演で問いを駐車」と断罪された。決定原則と6度の失敗の警告は、本サイクルでの実改善出荷を強く支持する。
- **結果体験の全面リデザインを一気に出荷**: 不採用。未検証の設計方向を勝利譚にする Trap B そのもの（cycle-268〜271）。本サイクルは読者として自明な改善1つに絞る。
- **B-537/B-540/B-524 等の他 P1/P2**: 不採用。いずれも内部リファクタ・メタ作業・回遊計測で、Goal 直結の B-545 を先送りする理由にならない（cycle-273 と同じ判断）。

## 補足事項

- 本サイクルは B-545 という巨大な問いの「最初の一歩を、逃げずに価値そのものへ踏み出す」サイクルである。B-545 を1サイクルで「解決」はできない。本サイクルで結果体験のすべてを最高にすることは目指さず、実接地→指差せる評価→自明な改善1つ→検証設計までを誠実にやり切る。残る改善余地は次サイクル以降へ申し送る。
- GA4/BigQuery でのデータ確認は、改善対象の選定根拠（どの結果・どのセクションで来訪者が離脱/滞留しているか）や効果検証の指標設計に必要な範囲で行う。データは入口で、その先の来訪者を実画面で見に行く起点にする。
