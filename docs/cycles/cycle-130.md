# サイクル-130

このサイクルでは、cycle-129（B-091）の失敗を受け、play系コンテンツのカスタマージャーニーをゼロから設計し直す（B-231）。現状、play系コンテンツへの流入はほぼゼロ（直近7日間でmusic-personalityの3PVのみ）であり、サイト全体のPVも週96と低迷している。来訪者の現実的な心理状態（消極的・懐疑的・注意力散漫）を前提に、各ページへの全到達導線・ユーザー心理・アクション優先順位をカスタマージャーニーとして整理し、おすすめ・関連コンテンツの導線をゼロから設計し直す。

- [ ] 1. 全到達経路の洗い出しと技術的検証（どのページのどの要素がユーザーに実際に表示されるか）
- [ ] 2. カスタマージャーニーマップの作成（全5シナリオ、各シナリオの心理状態・アクション優先順位を含む）

## 作業計画

### 目的

cycle-129では「カスタマージャーニーを作成せずにいきなり実装した」ことが根本原因となり、メインターゲット（プレイ完了者）に導線が表示されないという致命的な失敗に至った。本タスクでは、実装（B-232）に先立ち、来訪者の現実的な心理状態を前提としたカスタマージャーニーマップと回遊導線の設計ドキュメントを作成する。これにより、B-232の実装時に「誰に」「どの心理状態で」「何を」「どこに」表示するかが明確になり、cycle-129の失敗を繰り返さない。

### 作業内容

#### 作業1: 全到達経路の洗い出しと技術的検証

cycle-129の最大の教訓は「ユーザーに表示されないものは未実装と同じ」である。設計に入る前に、現在の全ページ種別について、ユーザーが実際にどの要素を見るかを技術的に検証する。

具体的な検証項目:

- 各ページ種別（/play一覧、/play/[slug]クイズ・診断、/play/daily占い、/play/games/\*ゲーム、/play/[slug]/result/[resultId]静的結果）のコンポーネント構造を確認
- QuizContainerのphase遷移（intro/playing/result）ごとにユーザーに見える要素の特定
- ゲームのResultModal（モーダル）がページ本体の要素を遮蔽する事実の確認
- スクロール位置の観点からファーストビューに含まれる要素の特定（特にモバイル）
- Playwrightでの実機スクリーンショット取得による視覚的検証

成果物: 検証結果を `docs/research/2026-03-29-play-page-visibility-audit.md` として記録

#### 作業2: カスタマージャーニーマップの作成

全到達経路を以下の5シナリオに分類し、各シナリオのカスタマージャーニーを作成する。

**シナリオ1: Organic Search経由の新規来訪者**

- 検索クエリの推定（「性格診断」「面白い占い」等）
- ランディングページ別の心理状態と期待
- 第一印象からプレイ開始までのハードル分析

**シナリオ2: SNSシェアリンク経由の来訪者（静的結果ページ）**

- 友達の結果を見に来た心理状態（好奇心+懐疑心）
- /play/[slug]/result/[resultId] の現状のUI評価
- 「自分もやってみたい」への転換ポイント

**シナリオ3: プレイ完了者（クイズ・診断）**

- 結果確認後の心理状態（達成感+自己確認）
- QuizContainer内のインライン結果表示からの動線
- シェア行動と次コンテンツへの回遊の優先順位

**シナリオ4: プレイ完了者（ゲーム）**

- ResultModal内での心理状態
- モーダル内で完結すべき導線 vs モーダル外の導線
- NextGameBannerの現状評価

**シナリオ5: /play一覧ページからの探索者**

- ヘッダーナビ/トップページCTAからの到達
- カテゴリ選択と個別コンテンツ選択の心理
- 初回プレイ開始までのファネル

各シナリオで以下を明記する:

- ユーザーの心理状態（現実的・悲観的に設定）
- ユーザーが実際に見る画面要素（作業1の検証結果に基づく）
- アクション優先順位（そのシナリオでユーザーにしてほしいことを順位付け）
- 離脱リスクとその対策
- 現状の導線の問題点

成果物: `docs/research/2026-03-29-play-customer-journey-map.md`

#### 作業3: 回遊導線の再設計案

カスタマージャーニーマップに基づき、全ページ種別の導線を再設計する。cycle-129の教訓として「既存の仕様を前提にしない」ことが重要であり、必要であればページ構造やコンポーネントの分割・統合も検討する。

設計対象:

**A. クイズ・診断の結果表示（QuizContainer内）**

- ResultCard内に回遊導線を組み込む設計（現在はQuizContainer外にRelatedQuizzes/RecommendedContentがあるため、プレイ完了者に表示されない問題の解決）
- 結果の核心説明の直後に配置する「次のコンテンツ」の選定ロジックと文言
- シェアボタンと回遊導線の優先順位と配置順序

**B. ゲームのResultModal内**

- モーダル内に収めるべき導線の設計（NextGameBannerの改善を含む）
- 他ジャンル（診断・占い）への導線のモーダル内配置
- モーダル内の情報量制限（認知負荷の管理）

**C. 静的結果ページ（SNSシェアリンク用）**

- 「自分もやってみる」CTAの改善（文言・配置・視覚的強調）
- 友達の結果を見た後の心理に合わせた導線設計
- コスト感の提示（所要時間・問数の明示）

**D. /play一覧ページ**

- 初回来訪者向けの「最初の一歩」の設計
- イチオシ/ピックアップの選定基準の見直し
- コンテンツカードに表示する情報の見直し（所要時間・問数の追加等）

**E. 占いページ（/play/daily）**

- 結果表示後の回遊導線
- 日替わりコンテンツのリピート促進

各設計には以下を含める:

- 具体的なUI配置（ワイヤーフレーム的な構成図）
- 表示するコンテンツと文言（結果連動型の文脈付き推薦を含む）
- 選択肢の件数と選定ロジック
- 情報のヒエラルキー（何を強調し、何を控えめにするか）
- デスクトップ/モバイルでの違い

成果物: `docs/research/2026-03-29-play-navigation-redesign.md`

### 検討した他の選択肢と判断理由

**選択肢A: カスタマージャーニーと導線設計を1つのドキュメントにまとめる**
不採用。カスタマージャーニー（現状分析+心理分析）と導線再設計（解決策）は性質が異なり、B-232の実装時に別々に参照する必要がある。また、技術的検証結果は独立したドキュメントとして残すことで、将来の設計変更時にも参照できる。

**選択肢B: シナリオを3つ（新規来訪者・プレイ完了者・SNS来訪者）に統合する**
不採用。プレイ完了者のクイズとゲームでは技術的な制約が全く異なる（インライン結果 vs モーダル結果）。統合すると「ユーザーに実際に表示されるか」の検証が曖昧になり、cycle-129の失敗を繰り返すリスクがある。5シナリオに分けることで、各シナリオの具体性と検証可能性を確保する。

**選択肢C: 回遊導線の設計をページ種別ではなくコンポーネント単位で行う**
不採用。コンポーネント単位の設計はエンジニア視点であり、来訪者視点ではない。来訪者は「ページ」を体験するのであり、「コンポーネント」を体験するのではない。ページ種別ごとの設計とすることで、来訪者の実際の体験に即した設計となる。ただし、設計ドキュメント内で実装時の参考としてコンポーネントへのマッピングは記載する。
