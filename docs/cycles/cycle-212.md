#### 何のためにやるのか / どんな価値を提供するか

1. **来訪者価値の核心（CLAUDE.md Decision Making Principle 準拠）**: `image-resizer` は `(legacy)` 配下に残存しており、ダークモード対応・新トークン体系・モバイルタップターゲット改善が来訪者にまだ届いていない。同時にトップページに「画像を D&D してリサイズ → DL」の最短動線が存在しないため、主要競合（iLoveIMG / Squoosh / Bulk Resize Photos）に対する「ブラウザ完結 + サーバー送信なし + シンプル UI」の差別化軸を最も多い初回流入面（トップ）で示せていない。本サイクルで両方を解消する。
2. **新デザイン詳細ページ移行**: `Component.module.css` 内の旧トークン（`--color-*` 系 = T-1 で再実測）を新トークン（`--bg` / `--fg` / `--accent` / `--danger` / `--danger-soft` 等）に一括置換。cycle-203〜211 で確立済の SSoT マッピング（cycle-211 と同 9 種マッピング表）を踏襲し、新規マッピングは原則発生させない（image-resizer に warning 系トークンは使われていない見込みだが T-1 で実測確認）。
3. **画像入力型タイル 2 件目 SSoT 引用適用**: cycle-211（image-base64）で「**File 入力 + 非同期 FileReader + 入力前後で UI 構造が変わる画像入力型タイル**」の SSoT 補足事項 (i)〜(x) を初確立。本サイクルは画像入力型タイル 2 件目として SSoT を**初回引用適用**し、N=2 のデータポイントを蓄積。後続の B-318 / B-371 / B-372 系画像ワークフローツール群に向けて再現性を実証する。
4. **B-456（画像入力型 AP-P21 (v) 基準値 N≥3 件見直し）に向けた N=2 蓄積**: cycle-211 §補足事項 (ii) で「画像入力型 N≥3 件で見直し（B-456）」と明示。本サイクルが N=2 = 中間点であり、(i)(ii)(iii)(iv)(v)(vii) を引用適用するか / (vi)(viii)(x) を再評価するかを T-4 実測で判定し、§補足事項に SSoT 書き戻して B-456 着手時のデータベースを整える。
5. **競合差別化の最大活用**: M1a の主要競合 iLoveIMG（サーバー処理 = プライバシー懸念）/ Squoosh（多機能だが詳細設定で学習コスト高）/ Bulk Resize Photos（一括処理特化）に対し、yolos.net image-resizer は「ブラウザ完結 + 3 アクション（**ファイル選択 → 幅入力 → ボタン押下 = リサイズ + DL 起動**）で完結」のシンプル軸で差別化可能。タイル UI ではこの差別化を最大化する設計（後述 §論点 B〜F）を採択する。

### 論点と判断

以下の論点について「採用案 / 検討した他案 / 判断理由」を明記する。**PM 判断は planner が行う**（owner 承認不要）。仮説の根拠が薄い論点は「T-X で実機確認後に確定」と明示する。すべての数値 literal には 3 分類ラベル（実測値 / 仕様値 / 推定値 + 経験的暫定値）を付ける。AP-P17 に従い、各論点で**来訪者価値 → 範囲整合 → 規模 → 歯止め** の 4 軸評価を実施。

#### 論点 A: タイルの recommendedSize

**第一推奨 = 案 A1（cols=3 rows=3 = 400×400px = 仕様値）/ T-3 Playwright 実機確認で確定**

- **要素数（推定値 + 経験的暫定値 / §論点 B 案 B4 + §論点 I 案 γ 採用反映）**: タイトル + ドロップゾーン or ファイル名行 + 幅 input + プリセットボタン [50%] [25%]（幅 input と横並べ = 行数は 1）+ リサイズ実行ボタン + **結果サイズ表示 2 段（γ 採択 = 1 行目 19 字 + 2 行目 18 字 = **約 80px（推定値 + 経験的暫定値 / §論点 I 採用欄の規模欄引用 = 1 行 40px × 2 段 / cycle-211 (iii) 引用適用）**）** + 詳細リンク = **7〜8 要素 / 行数 6〜7**。cycle-211 image-base64（**6〜7 要素**（**推定値 + 経験的暫定値** / cycle-211 L291 引用 = `タイトル + ファイル選択 UI（ドロップゾーン）+ プレビュー + base64 出力テキスト + Data URI 出力テキスト + コピーボタン 2 個 + 詳細リンク = 6〜7 要素 / 案 E 採択で 1 要素増`）） より +1〜2 要素（γ 2 段化が主因）。
- **案 A1: cols=3 rows=3 (400×400px = 仕様値)** ← **採択**
  - **来訪者価値**: M1a 最短動線
  - **範囲整合**: image-base64 と同枠 + 操作モデル一貫
  - **規模**: 過去 12 タイル中 cols=3 rows=3 は 3 件（qr-code / text-replace / image-base64）= 先例あり
  - **歯止め（退避案 A2 cols=3 rows=4 への切替トリガー / 優先度順）**:
    - **(1) §論点 I 案 γ 採用の結果テキスト 2 段（約 80px）が含まれる (c) リサイズ後系統の総高さが cols=3 rows=3 枠超過した場合**（**最優先トリガー** / T-3 実機確認で判定 / γ 2 段は本サイクル中で確定済の必須要素のため最も発動可能性が高い）
    - **(2) §論点 B 案 B4 採用のプリセットボタン併設で要素過密 or 枠超過した場合**（T-3 builder 判定 / B4 縮退（B1 dimensions のみ）への切替と A2 移行はそれぞれ独立判断）
    - **(3) §論点 H 系統 X / 系統 Y のいずれかが 100ms 超過 → 案 H2（spinner 表示）切替時の自動切替**（T-1 / T-4 ベンチ実測で判定）
- **案 A2: cols=3 rows=4 (400×536px = 仕様値)**: 縦長で要素を余裕を持って配置可能。退避案として保持（T-3 で A1 が枠超過した場合、または §論点 H で H2 spinner 表示に切替が発生した場合に切替）。M1b の操作モデル一貫性（image-base64 と同枠）からは A1 優位

#### 論点 C: タイルに含める出力フォーマットと入力受け入れ範囲

論点 C は (C-1) 出力フォーマット / (C-2) GIF 入力時の挙動 / (C-3) SVG 入力時の挙動の 3 つに分割して planner が確定する。

##### 論点 C-1: 出力フォーマット + `accept` 属性の限定

**採用案 = 案 C2（元ファイル形式維持 / セレクタ省略）+ `accept="image/png,image/jpeg,image/webp"` 明示限定**

- **案 C2: 元ファイル形式維持（JPEG → JPEG / PNG → PNG / WebP → WebP）+ accept 明示限定** ← **採択**
  - **来訪者価値**: M1a dislikes「結果を信用していいか判断できない」を回避 = 入力した画像と同じ形式が出力されるという最も自然な期待値 / JPEG 入力者が PNG 出力で巨大化する違和感を防止 / `accept` 明示限定で GIF / SVG / BMP 等の Canvas で扱いにくい形式をブラウザ標準ファイル選択時点で除外 = 「選んだのに失敗する」UX 破綻を予防
  - **範囲整合**: フォーマット変換ニーズは「画像 リサイズ」検索意図とは別軸 = 詳細ページに残存 = 動線分離成立 / `image/*` のままだと既存 Component.tsx と同じ広い範囲を受けるが、タイルではエラーケース最小化のため限定推奨
  - **規模**: `file.type` から MIME 推定するロジックを追加（推定値 + 経験的暫定値 = 約 5 行程度の追加 / `getMimeType()` 等既存 logic.ts 再利用可能性 T-1 で確認）+ `accept` 属性は 1 文字列の literal 化のみ
  - **歯止め**: 既存詳細ページ Component.tsx は `image/*` 維持（既存ユーザー価値を後退させない / タイルと詳細ページの非対称設計は許容）
- **案 C1: PNG 固定**: JPEG 入力 → PNG 出力でファイルサイズが膨張する違和感 = 「リサイズしたのに大きくなった」UX 破綻 → 不採用
- **案 C3: PNG / JPEG / WebP セレクタ含める**: タイル枠圧迫 + 案 B4 採択（プリセット併設で要素+2 済）と整合性なし → 不採用

##### 論点 C-2: GIF 入力時の挙動

**採用案 = 案 C2a（accept 明示限定で GIF をブラウザ標準ファイル選択時点で除外 / D&D 時は MIME 判定でエラー表示）**

GIF はアニメーション形式で Canvas drawImage では 1 フレーム目のみ静止画化される = 「アニメが消えた」UX 破綻リスク。

- **案 C2a: accept 限定で除外 + D&D 時は「アニメ画像は詳細ページをご利用ください」エラー表示** ← **採択**
  - **来訪者価値**: アニメ GIF をリサイズしたい来訪者には「タイルでは扱えない」+「詳細ページ動線」を明示 = 暗黙変換による驚き回避
  - **規模**: エラー文言 1 種追加（既存 5 種 + 1 = 6 種 / **cycle-212 で新規追加文言**）/ 「アニメ画像は詳細ページをご利用ください」は **19 字（仕様値 = Python `len()` 実測）= 既存最長と同字数**。**最長文言は 2 種が 19 字で並ぶ**（既存「ファイルサイズが20MBを超えています」+ 新規「アニメ画像は詳細ページをご利用ください」）= T-4 エラー枠 1 行収納検証は**両 19 字文言を実機検証対象に追加**（376px 内 1 行収納可否は文字組によって微差が生じる可能性があるため両方計測）
  - **歯止め**: 既存詳細ページ Component.tsx は GIF 受け入れ継続（cycle-212 では touch しない）
- **案 C2b: GIF を受け入れて 1 フレーム静止画化 + 「アニメ情報は失われます」警告表示**: 警告 UI 追加でタイル枠圧迫 + 暗黙変換の驚き残存 → 不採用
- **案 C2c: GIF を受け入れて警告なし（既存挙動継承）**: 「アニメが消えた」UX 破綻 → 不採用

##### 論点 C-3: SVG 入力時の挙動

**採用案 = 案 C3a（accept 明示限定で SVG をブラウザ標準ファイル選択時点で除外 / D&D 時は MIME 判定でエラー表示）**

SVG はベクター形式で Canvas drawImage で扱えるが CORS / セキュリティ制約あり（タイント canvas 化リスク）+ 「リサイズ」概念が px 寸法ではなく viewBox 換算となり来訪者期待と乖離。

- **案 C3a: accept 限定で除外 + D&D 時はエラー表示** ← **採択**
  - **来訪者価値**: SVG リサイズは概念的に「リサイズ」ではなく「ラスタライズ」= 期待と乖離 / セキュリティ制約による失敗を予防
  - **規模**: 論点 C-2 と同じ accept 文字列に SVG 不含で実現 / 追加実装なし
  - **歯止め**: 既存詳細ページ Component.tsx は SVG 受け入れ継続（cycle-212 では touch しない）
- **案 C3b: SVG を受け入れて Canvas ラスタライズ**: タイント canvas / セキュリティ制約の不安定要因 → 不採用

#### 論点 D: 品質スライダー（JPEG/WebP）

**採用案 = 案 D1（詳細ページ専用 / タイルでは quality=80 固定）**

- **案 D1: タイルでは quality=80 固定（仕様値 = Component.tsx L32 デフォルト値）** ← **採択**
  - **来訪者価値**: 競合 iLoveIMG も品質 UI なし / quality=80 はブログ・SNS 用途で十分（**経験的暫定値** = Web 一般慣行）/ 設定軸を増やさず最短動線維持
  - **範囲整合**: 品質詳細設定は Squoosh 的高度ニーズ = 詳細ページに残存
  - **規模**: 案 C2 採択（元形式維持）と整合 = JPEG/WebP 入力時のみ quality 適用
  - **歯止め**: PNG 入力時は quality 適用なし（既存 logic.ts L73-78 同型）
- **案 D2: タイルにも品質スライダー含める**: タイル枠圧迫 + 案 B1 / 案 C2 採択（最小化）と整合性なし → 不採用

#### 論点 E: 元画像プレビュー / 結果確証の付与方法

**採用案 = 案 E3（元画像プレビュー省略 + 寸法ビフォーアフター + 推定容量変化を結果テキストで表示 / 具体的な行構造は §論点 I で確定 = 案 γ 2 段構造採用）**

- **案 E3: 元画像プレビュー省略 + 寸法ビフォーアフター + 推定容量変化を結果テキストで表示**（**例示値 = 推定値** / 行構造は §論点 I 案 γ 採用 = 1 行目 `1920×1080 → 800×450`（**19 字 = 実測値**） + 2 行目 `(推定 145KB / DL 開始)`（**18 字 = 実測値** / `145KB` は**例示値 = 推定値**）） ← **採択**

#### 論点 H: ローディング表示（Canvas 処理時間）

#### 論点 I: 結果 img への `role="status"` 付与

**採用案 = 案 γ（結果サイズ表示テキストを明示的 2 段構造 / 各行 ≤ 19 字 / `role="status" aria-live="polite"` を 2 段全体を囲む 1 要素に付与 / img 自体には付与しない）**

**結果テキストの 1 行収納可能性比較（AP-P17）**: cycle-211 §補足事項 (iii) SSoT エラー枠 = **376px / 19 字 1 行収納**（**実測値**）。1 行に「ビフォーアフター寸法 + 推定容量 + DL 起動通知」をすべて含めると 30〜45 字（**実測値 = Python `len()`**）となり 376px 内に収納不可（≥ 1.6 倍）。来訪者価値「情報的確証」と「枠超過リスク回避」の両立のため、テキスト構造を 3 案で比較:

| 案    | 構造                                                             | 行数 / 各行字数 (実測値 = Python `len()`)                                                                    | 1 行収納可否                    | 来訪者価値                                                        | (c) リサイズ後系統の高さ予測                                                   |
| ----- | ---------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ | ------------------------------- | ----------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| α     | 1 行短縮（容量情報を視覚から除外 / DL 通知は `visually-hidden`） | 例: `1920×1080 → 800×450` = **19 字（実測値）**                                                              | ○（376px 内 1 行）              | M1a「メール添付前にサイズを知りたい」が**満たせない**（容量欠落） | 容易（40〜50px / cycle-211 (iii) 引用適用可）                                  |

- **採用案: 案 γ（明示的 2 段構造）** ← **採択**
  - **来訪者価値**: M1a dislikes「結果を信用していいか判断できない」+「メール添付前に容量を知りたい」+「DL されたか分からない」を**3 つすべて解消**（α は容量情報欠落で 1 つ不充足 / β は枠超過で UX 破綻リスク）/ AT 利用者にも `role="status"` で 2 段テキストが順次読み上げられる
  - **範囲整合**: cycle-211 (iii) SSoT 376px / 19 字を**各行に独立適用**（1 行目 19 字 / 2 行目 18 字 = 両行とも 376px 内 1 行収納）/ 2 段で 1 つの `role="status"` 要素 = カウント不増（count=1）
  - **規模**: テキスト 1 要素（内部に `<span>` 等で 2 段構造）= 高さ約 80px（**推定値 + 経験的暫定値** = 1 行 40px × 2 段 / cycle-211 (iii) 引用適用）

    - **1 行目仕様** = `{元幅}×{元高さ} → {後幅}×{後高さ}`（最大 19 字以内に収まる範囲。寸法桁数が多い場合は `→` 前後の空白除去等で短縮 = builder 判断 / 例: `1920×1080→800×450` = **17 字 = 実測値（Python `len()`）** / 数値はすべて整数 px）

- **却下案 β（複数行展開許容）**: 45 字 = 2〜3 行展開で (c) リサイズ後系統が 60〜90px 占有 → cols=3 rows=3 枠超過リスク顕在化 / A2 縮退発動率上昇 → 不採用

- **却下案: img に `role="status"` 付与 / テキスト省略**: スクリーンリーダーで状態変化が伝わらない → 不採用

#### 論点 J: ダウンロード後の状態変化

**採用案 = 実装する（cycle-211 (x) コピーボタン文言変化 SSoT を「ダウンロード完了表示」として引用適用 / AP-P21 適用外として SSoT 化）**

  - **規模**: ボタン 1 個の文言変化 = useState 1 個追加 / DL 起動後 onClick で setState
  - **歯止め（planner 確定）**: **文言が戻るタイミング = 数秒後（2 秒 = 経験的暫定値 / cycle-211 image-base64 コピーボタンと同型）**。次のファイル選択時ではなく時間経過で戻すことで、同じファイルを再 DL したい来訪者の混乱を防止
- **却下案: 実装しない**: DL 完了の状態変化が来訪者に伝わらず M1a 不安残存 → 不採用

#### 引用適用（N=2 データポイント蓄積）

- **(i) 画像入力型タイル特有の AP-P21 適用範囲 SSoT**: 「未選択時 / 選択後の構造比較は適用外」+「各系統内で AP-P21 (i) 下限 40px / (vi) エラー枠 1 行収納を適用」を引用適用。本サイクルは §論点 G で 4 系統独立評価に拡張する形で SSoT 適用。

### 検討した他の選択肢と判断理由

Phase 8.1 第 13 弾の対象として、次サイクル候補は以下の 3 軸で比較した:

| 候補              | 軸 (a) 画像入力型 2 件目（SSoT 引用適用）                                                                                                          | 軸 (b) 複合入力型 2 件目（SSoT 引用適用）                                                               | 軸 (c) 単純構造ツール                                        |
| ----------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------ |
| **image-resizer** | **○ 採択**: cycle-211 SSoT 引用適用 + N=2 蓄積 + B-456 中間点 / Canvas blob 出力という新規軸 / M1a 主要ユースケース（SNS / メール / ブログ）が広い | 不該当                                                                                                  | 不該当                                                       |

| text-diff         | 不該当                                                                                                                                             | ○: cycle-210 SSoT 引用候補 / 入力 2 系統（旧 + 新）+ diff 出力 / 競合（diffchecker）強力                | 不該当                                                       |
| dummy-text        | 不該当                                                                                                                                             | 不該当                                                                                                  | ○: 単純構造（文字数指定 + 生成）/ S1 主用途 / M1a 用途は狭い |

**採択 = image-resizer**:

1. **画像入力型 SSoT 引用適用の緊急性**: cycle-211 §補足事項 (i)〜(x) は N=1 段階の暫定 SSoT。N=2 で引用適用しないと SSoT としての汎用性が証明されず B-456（N≥3 見直し）への移行も停滞する。複合入力型は cycle-210 で N=1 + cycle-210 §補足事項 4 項目があるが、本サイクルで N=2 を進めると複合入力型 SSoT 引用適用が後回し（次サイクル候補）。**N=1 段階の SSoT は引用適用ペースが速いほど価値が積層する**ため、cycle-212 = image-resizer = 画像入力型 N=2 を優先採択。
2. **来訪者価値の規模**: M1a 主要ユースケース（SNS 投稿前 / メール添付前 / ブログアイキャッチ / フリマ商品画像 / GitHub/Notion 添付）は regex-tester（S1 中心）/ text-diff（diff 用途 = やや限定的）/ dummy-text（S1 中心）より明確に広い。
3. **競合差別化**: yolos.net image-resizer の「ブラウザ完結 + シンプル UI」差別化軸を最も多い初回流入面（トップ）で示せていない状態を解消できる。

**不採用となった理由（簡記）**:

- **regex-tester**: S1 中心の用途 / cycle-211 SSoT 引用適用ペースを優先（B-456 中間点を進める）
- **text-diff**: 複合入力型 SSoT 引用適用は次々サイクル候補として保持 / Phase 8.1 全体スケジュール上順次扱い
- **dummy-text**: 単純構造ツールはタイル化価値が相対的に薄い（M1a の能動的検索意図が乏しい）
