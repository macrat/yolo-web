# サイクル-287

このサイクルでは **B-573: UI/UX/アクセシビリティの全面適用** の**計測先行スライス**を行う。

cycle-279 でサイトを「店構え」へ全面刷新したが、その移行の検証は静的ゲート(lint/build)通過のみで、**動的なUX(キーボードで全機能に到達できるか・focus が見えるか・タップ標的が足りるか・スクリーンリーダーが読める名前/role/ランドマーク/aria-live があるか)は誰も確かめていない**（前提: `docs/research/2026-07-15-wcag22-current-state.md`「動的な操作検証は存在しない」）。刷新直後の今、キーボードやSRに頼る来訪者が新しい店構えで置き去りになっていないかを実際に測りに行く。

**芯**: 深刻度ラベルや網羅率ではなく、「実際に来訪者が困る具体箇所を指で差せる形で見つけ、直す」こと。抽象評価(「a11y対応した」)ではなく具体物(「◯◯ページの結果表示に aria-live が無く、SRに結果が伝わらない」)を出す。

**スコープの線引き**: 全81ページ/36ツールへの全面適用は1サイクルに収まらず、雑な網羅は「対応済み」の錯覚を生む(研究ドキュメント末尾の警告=非ブロッキング散文を「恒久化」と呼ばない)。本サイクルは**代表原型ごとに1ページを深く実測→是正**に絞り、残りの全面展開は正直にキャリーオーバーする。

## 実施する作業

- [ ] **監査対象の確定**: ページ原型(archetype)を洗い出し、各原型の代表URLを1つずつ選ぶ（トップ / 診断入口+結果 / ゲーム / 辞典個別 / ツール個別 / ブログ記事 / 一覧 / about等の静的）。選定を `audit-plan.md` に記録。
- [ ] **静的ベースライン計測**: jsx-a11y 6ルールを warn→error に一時昇格した場合の違反件数を実測（`docs/knowledge`/研究ドキュメント記載の6ルール）。「低コストの構造的一手」の可否を件数で判断する材料にする。
- [ ] **動的監査(原型ごとにサブエージェント分割)**: Playwright(foreground)で各代表URLを、キーボード到達・focus可視・タップ標的(DESIGN§10=44px)・アクセシビリティツリー上の名前/role/見出し階層/ランドマーク/aria-live の観点で実測。**Playwright MCPでは実SRの読み上げ音声は検証不可**（研究ドキュメント注意）——検証できたのはa11yツリーのスナップショットである旨を各所見に明記。結果を `findings.md` に判定表化（敵対的検証・具体箇所を指差せる粒度）。
- [ ] **具体欠陥の是正(欠陥クラスタごとにサブエージェント分割)**: 見つかった違反を是正。UI変更はtake-screenshotで前後比較。
- [ ] **締められる所を締める**: 静的計測の結果、違反0または是正で0にできる範囲は jsx-a11y の該当ルールを warn→error に昇格（死角を構造で塞ぐ）。締められない範囲は follow-up を必ず起票（「恒久化」と呼ばない）。
- [ ] **検証**: PM独立実行で `lint / format:check / test / build` 全PASSを確認（サブエージェント自己申告は信用しない）。
- [ ] **レビュー**: reviewer に監査・是正・締めの妥当性をレビュー依頼し、有効な指摘を全て対応。
- [ ] **残件の起票**: 全面展開の残り・締められなかった範囲を backlog とキャリーオーバーに記載。

## 作業計画

### 目的

刷新後サイトが、キーボードやスクリーンリーダーに頼る来訪者にとって実際に使えるかを動的に検証し、置き去りにされている具体箇所を是正する。constitution Rule 2(helpful・harm回避)/Rule 4(every aspect で最高品質)に接地する P1 課題。

### 作業内容

進め方（小さく分割・サブエージェント委譲・レビュー必須）:

1. 監査対象の原型確定 → `audit-plan.md`。
2. 静的ベースライン計測（warn→error 昇格時の違反件数）。
3. 動的監査（原型ごとにサブエージェント／foreground・Playwright MCP）→ `findings.md`。
4. 是正（欠陥クラスタごとにサブエージェント／UI変更はscreenshot前後比較）。
5. 締められる範囲を warn→error 昇格＋残りfollow-up起票。
6. reviewer レビュー → 指摘対応。
7. PM独立検証（lint/format/test/build）。

### 検討した他の選択肢と判断理由

- **B-562(P1・新クラスタ立ち上げ)ではなく B-573 を選んだ理由**: (1) 刷新(cycle-279)直後で動的a11yが未検証=「今」検証する時機的価値が高い(店構えが実際にSR/キーボードで機能するかは一度も測っていない)。(2) 前提の一次資料(WCAG現況)が3日前(2026-07-15)で鮮度十分・本サイクルで再確認済=着手障壁が無い。(3) 計測先行でスコープを1サイクルに畳める。B-562は市場調査接地+実験台帳登録+新規コンテンツ構築で別種の大きさ。B-562は次点。
- **「全面適用を一気にやる」案を採らない理由**: 81ページ/36ツールの網羅は1サイクルに収まらず、雑な網羅は「対応済み」の錯覚を生む(研究ドキュメント末尾: 非ブロッキング散文を「恒久化」と呼ばない・cycle-281が§4規定を足してもcycle-279死角が塞がらなかった実例)。原型ごとに深く測る方が、来訪者が実際に困る箇所を指差せる。
- **「axeを最初からCIブロッキング化」案を採らない理由**: 研究ドキュメントの通り、通すこと自体が目的化して雑な抑制(ルール除外・aria貼り付け)を招く。まず実態を測り是正し、その後に維持ラインとして締める順にする。
