## D. 使ったクエリ / レポート定義

すべて `npx tsx .claude/skills/analyze-bigquery/scripts/query.ts "<SQL>"` で実行。dataset `analytics_524708437.events_*`。取得日 2026-07-21。

1. **A 全体比**: `_TABLE_SUFFIX BETWEEN '20260328' AND '20260720'`、`device.category` で GROUP BY、セッション = `COUNT(DISTINCT CONCAT(user_pseudo_id, CAST(ga_session_id AS STRING)))`、ユーザー = `COUNT(DISTINCT user_pseudo_id)`。`ga_session_id` は event_params の int_value。
2. **A 月別変動**: 上記を `FORMAT_DATE('%Y-%m', event_date)` で GROUP BY。
3. **A 累積窓比較 / C 窓探索**: 同 base を CROSS JOIN した窓ラベル別に distinct セッション・ユーザーを集計。トレーリング28日は `d BETWEEN DATE_SUB(end, INTERVAL 27 DAY) AND end`。
4. **B 面別比**: `event_name='page_view'` の page_location を path 化（ドメイン・クエリ除去）→ `/dictionary`,`/play`,`/tools`,`/blog`,`/`(完全一致) で CASE 分類 → 面×デバイスで distinct セッションキーを集計。

対象期間はいずれも 2026-03-28〜2026-07-20（一部の C 探索は 06-01〜07-20 のサブ窓）。

## E. データ上の注意点・解釈上の留保

- **GA4 MCP 未取得**: 2026-02-14〜 の広い窓の裏取りができていない。BigQuery は 2026-03-28 開始のため、それ以前は不明。より広い窓での全体比は別途 GA4 MCP が使えるセッションで取得が必要。
- **bot / 自己トラフィック**: GA4 の標準 bot 除外は効くが、PM/開発者自身のアクセス（多くは desktop）が混入している可能性は排除できない。desktop 比が高めに出ている一因の可能性あり。デバイス比を確定させる際は自己トラフィック除外の検討が望ましい。
- **標本の小ささ**: 全期間でも総セッション 1,679 と小規模。月別・面別ではさらに小さく、特に 2026-03（25セッション）・面別「その他」（45）・ツール系（152）は小標本。数%の変動は誤差レベルとして扱う。
- **セッション計数の近似**: 本レポートのセッションは `(user_pseudo_id, ga_session_id)` の distinct 近似で、GA4 本体の sessions 指標と完全一致は保証されない（数%の差が出うる）。
- **窓依存性が大きい**: モバイル比は月で 24〜62% と振れる。DESIGN §10 の是正記述では「単一固定値」ではなく「窓・指標・出所」を明記し、可能なら面差（診断系はモバイル優勢／他面は desktop 優勢）に触れるのが誠実。
- **「7割超」は全体では成立しない**: 全体のモバイル比は 45〜48%（セッション/ユーザー、全期間）。「7割超」は診断系（/play, 74.5%）という**一面の値**であり、サイト全体の実測ではない。
