# サイクル-294

サイトの玄関である性格診断——検索流入Top20がほぼ「アニメキャラ診断/似てるキャラ診断」で埋まり、`/play/character-personality` が突出したPVを集めている(cycle-294計画時のGA実測)——で、来訪者が受け取る「あなたはこのタイプ」の結果が公正に決まっているかを検証する。`scoring.ts:82` は personality 判定で strict `>` を使い、同点時は `quiz.results` の**配列先頭タイプが必ず勝つ**。この配列順バイアスが実害(高頻度の同点で先頭タイプに偏る/到達不能なタイプが生まれる)を起こしていないかを全12診断で実測し、実害があれば是正する。

- [ ] 計測harnessの作成(`tmp/`): 各診断の回答空間を全列挙(組合せが大きい場合は代表サンプリング)し、(a)勝者タイプ分布 (b)同点発生率 (c)同点時に配列順で敗退して**到達不能になるタイプ**の有無 (d)配列順を無視した場合との差分、を出力する
- [ ] 計測実行と実害判定: 各診断の結果を `cycle-294/findings.md` に記録し、重大度(実害あり/軽微/無害)を診断ごとに判定する
- [ ] 是正方針の決定: 実害があれば、来訪者価値の観点でタイブレーク方針(例: 同点は不定を避けつつ公平にする/配点表側の是正)を決める。scoring.ts の一元修正で足りるか、個別診断の配点是正が要るかを切り分ける
- [ ] 是正の実装(必要な場合のみ): scoring.ts またはデータ側を修正し、`scoring.test.ts` に同点・到達可能性のテストを追加する
- [ ] 検証: `npm run lint && npm run format:check && npm run test && npm run build`
- [ ] レビュー: reviewer サブエージェントに検証設計と是正内容をレビュー依頼し、有効な指摘に対応する

## 作業計画

### 目的

来訪者が診断に求めるのは「自分はどのタイプか」という答えそのもの。その答えが配列の並び順という無関係な実装都合で歪んでいるなら、玄関のコンテンツで静かに来訪者を裏切っていることになる(constitution rule 2・4)。実測で歪みの有無と大きさを確かめ、あれば正す。

### 作業内容

1. **棚卸し**: 全12 personality診断(animal/character-fortune/character-personality/contrarian-fortune/impossible-advice/japanese-culture/music/science-thinking/traditional-color/unexpected-compatibility/word-sense/yoji)の配点構造を確認。
2. **計測**: 回答空間の列挙(設問数×選択肢数が現実的なら全列挙、大きければ層化サンプリング。列挙規模は harness が自動判定し、サンプリング時はその旨を findings に明記=silent cap 禁止)。指標=勝者分布・同点率・到達不能タイプ・配列順バイアスの寄与。
3. **判定**: 診断ごとに実害度を記録。「同点はほぼ起きない/起きても到達不能タイプを生まない」なら無害、「特定タイプが配列順だけで恒常的に勝つ/負ける」なら実害。
4. **是正**: 実害があれば方針決定のうえ最小修正。無害なら「検証の結果無害」を根拠付きで記録してクローズ(是正しないことも正当な結論)。

### 検討した他の選択肢と判断理由

- **B-573(a11y全面展開・P1)**: 81ページ/36ツールと規模が大きく最小サイクルに載らない。分割済サブチケット(B-594等)が別途ある。今回は見送り。
- **B-594(/play/daily h1不在・P3)**: 具体的だが影響面が daily 単独。玄関の診断結果の公正さ(全診断横断・#1ページ直撃)の方が来訪者価値が大きいと判断。
- **配点表の目視レビューで済ませる案**: 同点頻度・到達可能性は目視では確実に測れない。回答空間の実測(列挙/サンプリング)の方がこのタスクに正しい道具。AP-P33(境界=失敗軸の両端を必ず含める)の趣旨にも沿う。
