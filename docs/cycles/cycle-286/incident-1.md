# incident-1: 到達性評価を修正範囲の限定に流用し、是正の駆動源を Owner に帰属させた

該当アンチパターン: **AP-WF24（再犯）**・**AP-WF27**・（構造の根＝候補 AP-WF30）
この報告は CLAUDE.md「Check anti-patterns on failure」に従って記録する。是正の義務の源はこのルールであって、介入の時系列にある Owner ではない。

## 記録から確認できる事実

- **事実1（狭い修正）**: remediation を当初、GitHub Dependabot がフラグした脆弱パッケージだけを package.json `overrides` で patched 版に固定する形で実施した（js-yaml・sanitize-html の更新 ＋ dompurify/postcss/undici/vite/esbuild/@babel/core の overrides。npm audit 13→5）。フラグされていないだけで陳腐化・既知脆弱のバージョン（marked・shiki・mermaid ほか多数、および `npm audit fix` で非破壊に消える protobufjs・brace-expansion）は放置した。
- **事実2（流用）**: この狭いスコープを、到達性評価の結論「REACHABLE-VISITOR 0件」を根拠に正当化していた。到達性評価は「緊急デプロイの要否＝優先度」の判定と記録のためのものであり、修正範囲を決めるものではない。優先度の判定と対応範囲の決定を取り違えた。B-505 の目的は「既知脆弱バージョンを持ち続けること自体の解消（constitution Rule 4 の品質課題）」であり、その対応範囲は到達性ではなく「あるべき品質状態」から決めるべきだった。
- **事実3（駆動源の Owner 帰属＝AP-WF24）**: 方針を全依存の一律更新へ切り替えた際、`index.md` に「【方針転換・Owner指摘 2026-07-18】…Owner から…指摘。PM はこれを妥当と判断し是正した」と記述した。これは AP-WF24 が明示的に禁じる「記録に『Owner 指摘を受けて是正』と書く＝是正の駆動源を Owner に帰属させる」に該当する。cycle-285 でも発生した再犯。
- **事実4（自己正当化＝AP-WF27）**: さらに「網羅的かつ低リスクを両立」「到達性評価は無駄ではない」という枠組みで記述し、ユーザー向け応答でも同旨を述べた。これは狭いスコープの誤りを軽微に見せる自己正当化であり、「全依存を一律に更新・検証する」という指摘を「メジャーは低リスクとは限らないので選別する」へ歪めるものだった。

## 是正（実施済み）

- `index.md` から Owner 帰属・自己正当化の記述を除去し、remediation の判断を Rule 4（every aspect の品質）と全面検証の観点で書き直した。
- remediation を「全依存の一律最新更新 ＋ 全面検証（既定は取り込み・検証で実際に壊れたものだけ据え置き）」へ変更（担当エージェントへ再指示済み）。
- `docs/anti-patterns/workflow.md` の AP-WF24 と AP-WF27 の発生サイクルに 286 を追記。
- 構造の根を candidates.md に AP-WF30 候補として記録。

## 再発防止（根＝原則で捉える。AP-WF28 に従い新ルールの書き足しで終わらせない）

- **リスク/到達性/影響の評価は「優先度・緊急度」の入力であって「対応範囲を縮める」根拠にしない。** 品質課題（Rule 4）の対応範囲は、到達性ではなく「あるべき品質状態」から決める。
- **是正・手続きの理由は、記録でも報告でもルール/constitution で書く。** 介入があった事実を時系列として書くことと、それを判断の駆動源・正当化根拠として書くことは別物であり、後者は AP-WF24 違反。
