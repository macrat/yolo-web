本 Phase 7 の原典記述（`docs/design-migration-plan.md` L103）は「Phase 2.1 で確定した『メタ型統合 / 分離 + 1 対多サポート可否』決定に従う」と書かれているが、Phase 2.1 の確定値そのもの（URL 構成 / メタ型統合・分離の選択 / 1 対多サポート可否）は原典本文に明示されていない（current-state-audit §A-1〜A-3 で実体確認済）。Phase 2.1 の決定を発掘する経路として git log・過去サイクル md も調査したが、本計画立案時点で「明示的に Phase 2.1 を結論付けた一次資料」は確認できていない。

### 目的

**誰のために何の価値か**: 本 Phase の直接の受益者は来訪者ではない（原典 L97 「来訪者から見える変化はない」）。しかし Phase 7 は Phase 8 以降で各ツール / 遊び / 辞典 / cheatsheet をタイル化し、最終的に Phase 10 でダッシュボード（道具箱）として来訪者に公開する一連の作業の **唯一の合流点** である。Phase 7 の型契約・サイズ規格・レジストリが Phase 8 着手時点で「3 形態 (a)/(b)/(c) のいずれを採るか各コンテンツが選べる形」で揃っていなければ、Phase 8 各サイクルが個別に基盤を継ぎ足す形になり、最終的なダッシュボードでタイル群の品質・一貫性が崩れる。これは「来訪者が日常的に使う道具箱」の UX 品質を直接損なう。

つまり Phase 7 の来訪者価値への寄与は: 「Phase 8 で全コンテンツがそれぞれの最適形態（1 対 1 / 1 対多 / 複数バリエーション）を採れる土台を、サイクルあたりの認知負荷を増やさない形で用意する」ことにある。

**完了の意味**: Phase 8.1 の最初のコンテンツ（GA4 PV 高い順 + 構造単純な順で選ばれる 1 件）の作業者が、「自分のコンテンツが 3 形態のうちどれを採るか選択し、型契約に従ってタイル定義を書き、レジストリ経由で hidden 検証ルートで単独表示確認できる」状態。逆に言えば、これができる最小限を超えた整備は本 Phase の責務外。

**6 回目失敗時の汚染最小化方針**: 過去 5 連敗が「型契約完成宣言時点で実物検証なし → Phase 境界で破綻 → クリーンアップ」の構造を持っていたため、本サイクルは「成果が Phase 8 で使われない場合に除去が最小コストで済む」形を必須要件にする。具体的には: (1) 新規ファイル追加は判断軸に絞った最小集合とし、独立した削除単位（ディレクトリ単位 or 明示的に列挙可能なファイル集合）として配置する、(2) 既存 4 系統の中央集中ファイル群への侵襲的変更を避ける、(3) hidden 検証ルートは「ルート + レジストリ消費の最小骨格」に絞り UI の作り込みを行わない、(4) サイクル完了時に「6 回目失敗時に除去すべきファイル集合」をキャリーオーバーに記録する。

**判定根拠の言語化（R-MINOR-r2-4 対応）**:

- **fortune の「slug 概念なし」判定**: `src/play/fortune/` 配下は `data.ts` / 単一画面構成で、複数の固有名（slug）を持つコンテンツ集合ではなく「入力に応じた状態を持つ単一機能」。Phase 8 タイル化単位として分割すべき情報構造を持たないため、機能全体で 1 エントリと判定
- **humor-dict の「機能全体で 1」判定**: `src/humor-dict/data.ts` の Record は 1500+ 語彙だが、各語彙は同一辞典内の項目であって独立 URL / 独立画面を持たない。辞典全体で 1 タイル = 1 エントリ
- **件数の実体確認**: tools 34 / cheatsheets 7 / games 4 / quiz 15 / dictionary 3 / humor-dict 1 / fortune 1 は current-state-audit と一致（数値更新は execution 着手直前に再確認する）

**この事実から導かれる制約**:

- 「per-slug meta.ts に optional フィールドを追加」モデルは tools / cheatsheets には自然だが、配列ベースの play(games/quiz) と const ベースの dictionary には機械的には適用できない
- ただし「タイル化単位」は「ファイル単位」とは独立に決められる。tools/cheatsheets は per-slug 単位、play(games/quiz) は配列要素単位、dictionary は const 単位、humor-dict / fortune は機能全体単位、と「タイル化単位の単位粒度」を系統ごとに別途定義する必要がある
- 論点 3 と論点 5 はこの「タイル化単位の粒度の違い」を明示的に扱える設計でなければならない

### 作業内容

本 Phase の作業を 4 つのタスクに分割する。各タスクは独立した sub-agent に委譲できる粒度とし、依存関係と検証手段を明示する。実装の具体的なフィールド名・関数シグネチャ・ファイル構造・テストツール API 名は本計画には書かず、execution フェーズで sub-agent が原典と本計画の判断軸に従って決定する。

**判断軸を execution で発火させるためのチェックポイント設計（R-MAJOR-r2-1 対応）**: cycle-198 が「3 形態想定を型契約に反映する」「robots.txt には hidden URL を載せない」を計画書に書きながら実装段階で機能させられなかった構造を踏まえ、本計画書では各判断軸を「execution の具体工程で発火する検証手順」として T-1〜T-4 の検証項目に組み込む。詳細は各タスクの「判断軸 → execution 検証手順」項参照。

#### T-1: タイル登録の型契約整備（Phase 7.1 の型部分）

**目的**: Phase 8 各サイクルが (a) 1 対 1 / (b) 1 対多 / (c) 複数バリエーション の 3 形態から任意の 1 つを選択でき、型レベルでその選択が表現できる型契約を整備する。

**作業範囲**:

- 4 系統で共有可能なタイル定義の型を新設する（論点 3 採用案に従う。詳細は後述）
- 3 形態を区別する型表現を新設する（論点 2 採用案に従う）
- 「タイル用コンポーネント参照、推奨サイズ、入出力プレースホルダ等」（原典 L104）に相当する共通フィールドを定義
- 推奨サイズの表現は T-2 で定義するサイズ枠定数に依存させる

**判断軸**:

- AP-I02 を踏まない（optional でケースを並立させない）
- 4 系統の「タイル化単位の粒度」差異を吸収できる
- 共通型のフィールドに「タイル化済 / 未着手」の運用ステートが optional として滲み出ない

**検証**:

- 型レベル（型エラーなし）
- 単体テストで 3 形態すべてが型契約を通過することと、不正組み合わせが型エラーで弾かれること（論点 4 採用案に従う）
- 加えて、現存 4 系統それぞれから 1 つ実例を選び、その実例にタイル定義を「ドライランで」付与した場合に型契約を満たせるかを単体テストで確認する（論点 4 で追加採択、後述）

**判断軸 → execution 検証手順（R-MAJOR-r2-1 対応 / cycle-198 が「3 形態想定を反映」を書いて失敗した構造の再生産を防ぐ発火点）**:

- **AP-I02 回避**: builder 完了時点で「タイル定義型および registry エントリ型に optional フィールド (`?:` / `| undefined`) が 1 つも含まれないこと」を grep で確認し、結果を builder 完了通知に貼り付ける（cycle-198 と同型の「optional で問題を回避」が起きていないかの execution 段階での発火点）
- **4 系統横断適用**: 4-β のドライラン単体テストが 4 系統 8 件（後述 4-β の選定基準）すべて緑で通る。失敗が 1 件でもあれば「4 系統横断適用可能性」判断軸が機能していないことの発火点として execution を一時停止し、PM に判断を仰ぐ
- **4-β 実例選定の言語化**: 後述 4-β 選定基準に従って選んだ実例 8 件それぞれについて、「なぜこの実例を選んだか」「どの形態 (a)/(b)/(c) を採るか」を builder が言語化し、完了通知に含める（keigo-reference 固定踏襲を防ぐ発火点、AP-P11）

**依存**: T-2 完了後着手（T-1 が T-2 のサイズ定数型を参照するため）

#### T-2: サイズ枠規格の確定と定数化（Phase 7.2）

**目的**: 1 セル 128px × 128px、セル間マージン 8px、n × m セル拡張可能を「ただ 1 箇所」に定数化し、Phase 8 / Phase 10 で参照する単一情報源を作る。

**作業範囲**:

- 原典 L106-110 に明示された値（1 セル 128px / マージン 8px / n × m 展開式）をサイズ枠定数の単一ファイルに定数として定義
- TypeScript からの参照と CSS Module からの参照を同じ値で共有する仕組み（具体方式は execution で判断。判断軸: 「数値直書きを Phase 8 以降の各タイル実装が行わない」が満たされること）
- n × m タイルの実ピクセル計算ヘルパー（原典 L109 「(128n + 8(n-1))px × (128m + 8(m-1))px」相当）

**検証**:

- 計算ヘルパーの境界値（1×1 / 1×2 / 2×1 / 3×3 等）が原典の式通りに返ることを単体テスト

#### 論点 1: Phase 7 を本サイクルでどこまでやるか

過去 5 サイクル（cycle-191/192/193/195/198）が Phase 7 完結に失敗している事実を踏まえ、「スコープ縮小していれば完結できたか」を各失敗サイクルについて個別評価した:

- cycle-191: 検証用 UI を Phase 7 範囲外まで作り込み失敗。スコープを 7.1 のみに縮小していれば回避の可能性
- cycle-192: cycle-191 成果の評価枠組み不在で失敗。スコープではなく検証定義の問題
- cycle-193: 「基盤 N 個」を実物観察前に固定して失敗。スコープではなく事前固定の問題
- cycle-195: 過去サイクル判定を所与継承して失敗。スコープではなく方法論の問題

**3-β の中核ロジック（AP-I02 回避の根拠）**: 「タイル化済 / 未着手」を **型内 optional ではなく集合への所属の有無** で表現する。これにより registry エントリ型は required フィールドのみで構成可能となり、消費側コードは型レベルで required を前提にできる（optional handling を強いられない）。具体的な集合の表現方式（配列 / Map / Set / 他）および iteration / lookup の API は execution 判断。

**3-α 不採択の根本理由**: optional field の有無で「タイル化済 / 未着手」を表現すると、消費側コードは常に optional 分岐を強いられ、これが AP-I02 が問題視する「型システムでなく運用で守る」構造になる。registry 方式は「タイル化済」を「配列要素として存在する」で表現するため、消費側コードは型レベルで required を前提にできる。

**4 系統への適用**:

- tools / cheatsheets: per-slug メタファイルは touch しない。新設の単一 registry に 1 エントリ = 1 タイル化済 slug として登録
- play (games / quiz): 既存中央集中ファイルは touch しない。同じ registry に登録
- play (fortune) / humor-dict: 単一機能なので 1 エントリ = 機能全体で登録（語彙単位タイル化はしない）
- dictionary: 同上（3 辞典それぞれが 1 エントリ）

本サイクル kickoff で PM は当初、cycle-198 Active の B-426 をいったん Queued に戻し、5 連敗の連鎖を断つために来訪者影響が具体で小スコープな B-330（ShareButtons 系ダークモード手法統一）を本サイクルで実施する判断をしていた。Owner から「2 つ前のサイクルからの重要な申し送りを見落としている」との指摘を受け、cycle-197 補足事項を再確認し、選定を B-426 に訂正した。

見落としていたのは cycle-197 補足事項 §「cycle-197 選定判断と B-428 取り下げの振り返り」および §「次サイクルの方向性」で、要点は以下:

1. `docs/design-migration-plan.md` は Phase 7（B-426 タイル基盤）→ Phase 8（B-314）→ Phase 9〜11 の順序を明文化している。cycle-197 PM はこの計画書を無視し「4 連敗回避 / 成功体験継続」を理由に Phase 7/8 を回避し、共有 Button/Input（B-386）と全ツール独自 button/input 修正（B-428）を選んだ。Owner 指摘で B-428 は取り下げられた。
2. Owner 指摘の本質: 「既存ツールのUIは全面的に刷新される予定です。なぜ破棄されるページの修正に時間を使うのですか？ サイトのデザインがページによってバラバラというUX上の致命的な状態を放置してまでやることですか？」
3. 次サイクル方向性: 「cycle-198 では `docs/design-migration-plan.md` の Phase 7（B-426 タイル基盤）または Phase 8（B-314）に戻る」

cycle-199 PM の判断ミスは構造的に cycle-197 PM と同型で、B-330 の対象 4 ファイル中 3 ファイル（`src/play/quiz/_components/`, `src/play/games/shared/_components/`, `src/humor-dict/_components/`）が `(legacy)` 配下で Phase 8 / 9 で新デザイン側に移行される予定であり、B-428 取り下げと完全に同じ「廃止予定 UI への投資」パターンだった。「5 連敗を回避したい」「小さく確実な成功で連鎖を断ちたい」という選定理由も、cycle-197 PM の「4 連敗回避 / 成功体験継続」と同じ言い訳構造で計画書を無視していた。

歯止め策はサイクル本体の作業ではないため計画フェーズに持ち込まないが、cycle-completion での反省として本セクションに残す。
