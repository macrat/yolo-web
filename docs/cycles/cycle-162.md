# サイクル-162

cycle-161で中期戦略（診断・占い中心、OS PV 294→350PV/月）は策定済みだが、長期目標6万PV/月への道筋が未策定。「AIブログが成長エンジンになる」という仮説が出ているが、調査・検証が未実施。

このサイクルでは、AIブログ・診断バイラル・その他の選択肢を調査・比較検討し、検証済みの前提に基づいて長期ビジョンを策定する。前提未検証のまま計画を積み上げないことがcycle-161の最大の教訓。

## 実施する作業

- [ ] Step 1: 現状分析 — GA4データ・サイトの現状を把握し、長期戦略の出発点を明確にする
- [ ] Step 1.5: 過去サイクルの知見の徹底調査 — 全サイクル・全リサーチレポートから各候補（A〜F）に関連する知見を網羅的に抽出し、Step 2の調査範囲を限定する
- [ ] Step 2: 成長エンジン候補の調査 — AIブログ・診断バイラル・SNS・その他の候補を洗い出し、それぞれの検索ボリューム・市場規模・競合状況を実データで検証する
- [ ] Step 3: 比較検討と評価 — 各候補を実現可能性・成長ポテンシャル・constitutionとの整合性で評価し、最適な成長エンジンを決定する
- [ ] Step 4: 長期ビジョンの策定 — 検証済みの前提のみに基づいて、6万PV/月への道筋を策定する
- [ ] Step 5: レビュー — 戦略の実行可能性・前提の検証度・constitutionとの整合性を確認する

## 作業計画

### 目的

6万PV/月というconstitutionのGoalから逆算した長期ビジョンを策定する。cycle-161で策定した中期計画（OS PV 294→350PV/月）が長期目標と整合するかを検証し、必要であれば中期計画を破棄・再策定する。このサイクルの成果物は「検証済みの前提に基づく長期ビジョン文書」であり、実装作業は含まない。

### 作業内容

#### Step 1: 現状分析（GA4データ・サイト構成の把握）

**目的**: 長期戦略の出発点となる正確な現状を把握する。

**作業**:

1. GA4で直近30日間のデータを取得する（チャネル別PV、コンテンツ別PV、エンゲージメント率）

3. サイトの現在のコンテンツ構成（診断数、ブログ記事数、辞典ページ数等）を確認する
4. 現状の数値を長期ビジョン文書のベースラインとして記録する

**アウトプット**: 現状数値の記録（長期ビジョン文書のセクション1）

**作業**:

   - その候補に関連する過去の調査・検討・実装の履歴

   - 検証済みの数値データ（検索ボリューム、PV実績、エンゲージメント率等）

### 注意事項

- **「検証済み」と「未検証」の厳密な区別**: 全ての前提について、根拠（GA4データ、aramakijake実測値、成功事例の具体的数値）を明記する。「おそらくこうだろう」は「未検証の推測」として扱う
- **200倍成長の現実直視**: 300PV→6万PVは200倍。「現実的に不可能」という結論も許容する。その場合は到達可能な上限目標と、6万PVに必要だが欠けている条件を明記する
- **積み上げ型ではなく逆算型**: 「今できることの延長線上にある目標」ではなく、「6万PVに必要な条件から逆算して、何が足りないかを特定する」アプローチを取る
- **Ownerのアドバイスの扱い**: 「中期計画を破棄してやり直す」は検討対象だが、調査結果に基づいて自分で判断する
- **サイト開始日の正確な把握**: yolos.netのリポジトリ初回コミットは2026-02-13。サブエージェントに渡す情報は必ず事実確認すること。本サイクルではドメイン年齢を「約5ヶ月」と誤ってサブエージェントに伝え、3件のリサーチ文書が汚染されたため全削除した

### 検討した他の選択肢と判断理由

1. **実装作業を含める案**: 長期ビジョン策定と同時にシェアテキスト改善等の実装を行う案。却下理由: cycle-155-159で「調査不十分なまま実装に着手して失敗」を繰り返した教訓。このサイクルは調査・策定に集中し、実装は次サイクル以降に回す

2. **中期計画をそのまま維持して長期ビジョンだけ追加する案**: cycle-161の中期計画（350PV/月）を前提として、その上に長期ビジョンを載せる案。却下理由: 中期計画自体が長期目標と整合するかを検証せずに前提にすることは、未検証の前提の上に計画を積むことと同じ

3. **成長エンジン候補を1つに絞って深掘りする案**: 例えばバイラル成長だけを深掘りする案。却下理由: 複数候補を比較しないと最適な選択ができない。特に「バイラルにはフォロワーが必要」という制約があるため、バイラル単独では成立しない可能性が高い

- <このサイクルで完了できなかった作業や、次のサイクルに持ち越す必要のある作業があれば、ここと /docs/backlog.md の両方に記載する。例えば、「XXXの機能にバグを見つけたが、本サイクルのスコープ外なので次回以降のサイクルで修正する予定。backlog.mdにも記載済み。」など。>

## 次サイクルへの申し送り

<前サイクルの申し送りを確認し、引き続き有効なものを残す。今回のサイクルで学んだ教訓や次以降のPMに役立つ知見を追記する。以下のルールに従うこと:

- 普遍的に役立つこと、注意すべき落とし穴、Ownerからの指摘事項を書く
- このサイクル固有の話や過度に具体的な実装詳細は書かない（コードやbacklogを見ればわかることは不要）
- 次のサイクルのPMが読んで実際に行動を変えられる内容だけを書く>

## 補足事項

<追加で補足しておくべきことがあれば記載する。とくに無い場合は「なし」と記載する。>

## サイクル終了時のチェックリスト

- [ ] 上記「実施する作業」に記載されたすべてのタスクに完了のチェックが入っている。
- [ ] `/docs/backlog.md` のActiveセクションに未完了のタスクがない。
- [ ] すべての変更がレビューされ、残存する指摘事項が無くなっている。
- [ ] `npm run lint && npm run format:check && npm run test && npm run build` がすべて成功する。
- [ ] 本ファイル冒頭のdescriptionがこのサイクルの内容を正確に反映している。
- [ ] 本ファイル冒頭のcompleted_atがサイクル完了日時で更新されている。
- [ ] 作業中に見つけたすべての問題点や改善点が「キャリーオーバー」および `docs/backlog.md` に記載されている。

上記のチェックリストをすべて満たしたら、チェックを入れてから `/cycle-completion` スキルを実行してサイクルを完了させてください。
なお、「環境起因」「今回の変更と無関係」「既知の問題」「次回対応」などの **例外は一切認めません** 。必ずすべての項目を完全に満してください。
