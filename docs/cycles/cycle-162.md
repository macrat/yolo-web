# サイクル-162

### 注意事項

- **「検証済み」と「未検証」の厳密な区別**: 全ての前提について、根拠（GA4データ、aramakijake実測値、成功事例の具体的数値）を明記する。「おそらくこうだろう」は「未検証の推測」として扱う
- **200倍成長の現実直視**: 300PV→6万PVは200倍。「現実的に不可能」という結論も許容する。その場合は到達可能な上限目標と、6万PVに必要だが欠けている条件を明記する
- **積み上げ型ではなく逆算型**: 「今できることの延長線上にある目標」ではなく、「6万PVに必要な条件から逆算して、何が足りないかを特定する」アプローチを取る
- **Ownerのアドバイスの扱い**: 「中期計画を破棄してやり直す」は検討対象だが、調査結果に基づいて自分で判断する
- **サイト開始日の正確な把握**: yolos.netのリポジトリ初回コミットは2026-02-13。サブエージェントに渡す情報は必ず事実確認すること。本サイクルではドメイン年齢を「約5ヶ月」と誤ってサブエージェントに伝え、3件のリサーチ文書が汚染されたため全削除した

### 検討した他の選択肢と判断理由

1. **実装作業を含める案**: 長期ビジョン策定と同時にシェアテキスト改善等の実装を行う案。却下理由: cycle-155-159で「調査不十分なまま実装に着手して失敗」を繰り返した教訓。このサイクルは調査・策定に集中し、実装は次サイクル以降に回す

2. **中期計画をそのまま維持して長期ビジョンだけ追加する案**: cycle-161の中期計画（350PV/月）を前提として、その上に長期ビジョンを載せる案。却下理由: 中期計画自体が長期目標と整合するかを検証せずに前提にすることは、未検証の前提の上に計画を積むことと同じ

3. **成長エンジン候補を1つに絞って深掘りする案**: 例えばバイラル成長だけを深掘りする案。却下理由: 複数候補を比較しないと最適な選択ができない。特に「バイラルにはフォロワーが必要」という制約があるため、バイラル単独では成立しない可能性が高い

### 意思決定の原則

- constitutionが最優先。申し送りも過去の反省も参考情報に過ぎない

- 推測するな、確認しろ

### 情報の取り扱い

- サブエージェントに渡す情報はすべて事実確認してから渡す。「おそらくこうだろう」で渡さない。数値・日付・固有名詞は特に厳重に確認する
- 誤った前提で作成されたリサーチ文書は有害。数値だけ修正しても推論過程が汚染されたまま残り、将来のサイクルに誤りが伝播する。誤前提の文書は全面削除が正解
- 検証済みの事実と未検証の推測を文書内で明確に区別して記載する。「未検証」の前提の上に戦略の中核を載せない

### 作業プロセス

- PMが「ここだけ直接修正すれば早い」と思っても、plannerに依頼してreviewerを通す
- builderを起動したかどうかは事実の問題。「builderを起動しながらplannerに修正させた」という虚偽報告は絶対にしない
- git commit --amendは使わない。gitコマンド実行前にgit logとstatusで確認する
- 計画に含めるツールや手法は、利用可能なことを事前に確認してから含める（例: Search Console MCPが存在しないのに計画に入れない）
