# ADR（意思決定と実験の記録）

大きな意思決定（アーキテクチャ・戦略）と実験（賭け）の記録を、**1件=1ディレクトリ**で残す。旧 `docs/experiments.md`（単一ファイル台帳）は肥大化して LLM が扱えなくなるため、この形式に移行した（cycle-278・オーナー指示）。**ADR 兼実験記録**として運用する——「決定を記録するもの」と「賭けて観測するもの」を同じ書式で扱う。

## 形式

- パス: `docs/ADR/YYYY-MM-DD-ADR###-タイトル/index.md`（例: `docs/ADR/2026-07-12-ADR001-コンセプト刷新/index.md`）
- `ADR###` は通し番号。日付は起票日。
- **index.md と同一ディレクトリに、任意の参考資料**（調査メモ・データ・試作コード等）**を置いてよい**。index.md が正典で、参考資料はその根拠・付録。

## index.md の書式

```
# ADR###: タイトル
- 状態: 計画中 | 走行中 | 観測待ち(期日) | 採用 | 記録済み | 打ち切り | 廃止
- 起票日 / 出荷日:
- 決定・仮説: 何を決めた/何に賭けたか（実験は「来訪者価値の仮説」を明記）
- 根拠: 一次資料・調査・実測へのポインタ
- 観測手段と評価窓: 何で・いつ判定するか
- 撤退基準: 何が起きたら殺す/覆すか（数値は Search Console 側に）
- 結果 / 学び: （事後に記入）
```

## 運用規約

1. **事前登録が先、出荷が後**。賭け（新コンテンツ・実験枠・大きな変更）は、出荷前に ADR を起こして仮説・観測手段・評価窓・撤退基準を書く。起票日と出荷日を分けて記録する（事後の合理化を防ぐ）。
2. **サイクルとの接続**: kickoff 時に「観測期日が来た ADR」を確認し、来ていれば読む作業をサイクルに含める。completion 時に新しい ADR・結果の記録を反映する。
3. **観測手段**（組み合わせて使う）:
   - **Search Console**（表示・順位・CTR）: 数値の撤退基準を置ける唯一の側（外部母集団）。反応は約3ヶ月遅れ——評価窓に織り込む。
   - **GA4 行動データ**: 小 n・計測欠落を常に疑い、単独の kill 根拠にしない。
   - **A/B テスト**（`src/lib/ab/` 基盤・面単位のインライン実験）: 統計的有意の道具ではなく、同時期に来た来訪者への対照提示を個人単位で質的に観察する手段。サイト全面の A/B はしない（破棄済み系統の温存=二重保守が前提になるため——`docs/rebuild-plan.md` §1）。
   - 時間軸の before/after は方向読みのみ（因果を断定しない・双方向ガード）。
4. **記録の誠実さ**: 負けの記録は勝ちと同じ粒度で書く。打ち切りは打ち切りとして残す（ただし「結果がどう出ても使い道がない」記録はオーナー判断で残さないことがある——記録は目的でなく手段）。
