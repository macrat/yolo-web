# 技術記事の品質と事実記述に関する知見

cycle-95（2026-03-16）で得られた知見。

## 技術記事の品質基準

- **推測で書かず一次ソースを徹底調査する**: 記事のすべての記述は、チャット履歴・コミットログ・当時のドキュメント・コードなどの一次ソースに基づくこと。調査が不十分なまま執筆した推測は、後から大量の修正が必要になる。

- **読者にとって不要な内部情報は削除する**: コミットハッシュ・日時・内部ID・メモIDなどは読者にとってほぼ価値がない。これらを含める場合は、読者にとっての意味を明示すること。意味を持たせられない数字・IDは削除する。

- **ターゲットユーザーになりきった批判的レビューを行う**: `docs/targets/` にあるターゲットユーザーの定義を確認し、そのユーザーとして記事全体を批判的に読み直す。「理解できるか」「価値ある学びがあるか」「独自性があるか」を問い続けること。全面書き換えも辞さない姿勢で行う。

- **「XXXが問題だったが本当はYYYだった」構文を避ける**: この構文は、前のバージョンの記事に対する訂正が入り込んだ結果として生まれやすい。読者は最新版しか読まないため、訂正の経緯を知らない読者には意味不明になる。最初から正しい本題をダイレクトに書く。

- **記事は単独で成立させる**: 「（〜としてではなく）」「〜ではありません」という補足が唐突に登場する場合、それは前バージョンへの返答が混入しているサインである。すべての記述が、その記事を初めて読む読者にとって文脈なしに理解できるかを確認する。

- **数値は意味とセットで提示する**: 数字単体では読者に何も伝わらない。「多いのか少ないのか」「期待通りか否か」「何と比較してどうなのか」が分からない数字は削除するか、意味が伝わる説明を必ず付ける。

- **教訓は現実的で読者が実践できる内容にする**: 「ありとあらゆる不可能を想定せよ」のような無理な理想論は読者の役に立たない。読者が具体的に実践できる、現実的な行動指針として書く。

- **Ownerの作業とAIの作業を厳密に分けて記述する**: 「私たちは〜と考えました」など、Ownerの判断とAIの作業を曖昧に混同する表現は避ける。誰が何をしたのかを明確に分けて記述する。

- **ネガティブな見出しを避け、各時代の特徴を公平に記述する**: 各時代・フェーズにはメリットとデメリットの両面があった。徹底的に調査して比較した上で、それぞれの特徴を正確に反映した見出しと説明を書く。

- **図・表・箇条書きを最適な形式で選択する**: テキストで書くと読みづらいフロー説明や手順比較は、箇条書きや表を活用する。Mermaid図を使う場合は、何を伝えたいのかを明確にしてから作成する。

## 事実の正確な記述

- **Owner発言はすべてチャット履歴を確認してファクトチェックする**: Ownerの発言として記事に書く内容は、必ず実際のチャット履歴に存在することを確認する。覚えていない発言や確認できていない成果を記述してはならない。

- **定量データは厳密にカウントし出典を明記する**: 「80〜90件以上」「約92回」など曖昧なカウントや概算は不正確。カウント済みのデータがある場合は正確な数値を使い、その出典（ログファイル・スクリプト結果等）を確認する。曖昧なまま記述しない。

- **不完全な統計分析に基づく曖昧な記述は削除する**: 体系的な分析を行っていない状態で「分析したところ無相関だった」と記述することは、不完全な分析に基づく誤った印象を与える。分析が不完全であれば、その記述ごと削除する。

- **比較対象を明確にする**: 「9割以上正常に稼動」のような記述には適切な比較対象が必要。「サイクル数」と「メモ数」のように緩い相関しかない指標を比較対象として使わない。同種の指標同士で比較する。

- **事実・推測・経験則の3種類を明確に書き分ける**:
  - **事実**: 一次ソースで確認できた内容。そのまま断定的に書く。
  - **推測**: 一次ソースで確認できていない内容。「〜と考えられる」「〜の可能性がある」と明示するか、調査して事実に変換してから書く。
  - **経験則**: 実運用を通じて得た知見。「〜という経験則を得た」「〜ということが分かった」と出所を明示する。

- **技術的な原因分析は正確な因果関係で記述する**: 「中間ステートの欠如が原因」のように表面的な現象を原因として書かない。「トリアージするタイミングと手順が明確化されていなかった」のように、本質的な原因を一次ソースで確認してから記述する。

- **トレードオフを正直に記述する**: 何かを廃止・変更した場合、得られたものだけでなく失ったものも記述する。「管理コストが下がった」と書く場合は、「検索・追跡コストは上がった」という側面も同時に記述する。

- **段階的な経緯を持つ事実は、最終状態だけでなく変遷を正確に記述する**: 「役割が変化した」と書く場合は、実際に変化があったかを一次ソースで確認する。変化していない場合は「変化した」と書かない。
