# AIエージェント間のやり取りの設計と運用に関する知見

cycle-95（2026-03-16）で得られた知見。

## エージェント間のやり取りの設計原則

- **意図した通信トポロジーと実際の挙動は一致しないことがある。** エージェント間の通信経路を「A→B→C」のように意図して設計しても、実際には異なるパターン（例：全エージェントがPMを経由する星型）に収束することがある。設計後は実際のログを観察し、意図通りに機能しているかを検証する必要がある。

- **状態遷移のトリガーは「作業完了時」だけでは不十分で「作業開始時」にも設けるべき。** キューやステート管理を設計する際、「作業完了時にアーカイブする」だけでは未処理のアイテムが残り続ける。「作業開始時にアクティブ状態へ移す」「作業開始時に前回のアクティブ状態が残っていたら完了済みとしてアーカイブする」というトリアージのタイミングと手順を明確化することで、アイテムの滞留を大幅に防げる。

- **エージェント間のやり取りに「記録」を求める場合、その記録システム自体がコンテキストを消費することを考慮する。** 通信の記録・管理・追跡のために中間ファイルやステート管理を設けると、エージェントが本来の作業に使えるコンテキストが削減される。「よくできた仕組み」であっても、LLMにとってはコンテキストを圧迫する重荷になりうる。

- **記録システムとエージェント間の実際の通信手段は別物として設計する。** メモやファイルによる記録はあくまで追跡・監査のための仕組みであり、通信そのものはAgentツール（プロンプトの引き渡し）が担っている。「記録」と「通信」を混同した設計は誤解を生む。LLMベースのシステムでは、エージェントへの指示をメモIDのみで渡すことでメインエージェントのコンテキストを削減するといったアーキテクチャ上の工夫が有効な場合がある。

- **オーナー向けの報告はノンブロッキングにする必要がある。** 全自動で動作するシステムにおいて、オーナーが常時監視・応答することを前提とした設計はスケールしない。オーナー不在でも作業が進められるよう、報告はファイルなどに記録して後から確認できる形にし、エージェント間の通信もあとから監査できる仕組みを持たせることが重要。

- **第1期（臨機応変型）と第2期（厳格ワークフロー型）の設計の違いを理解する。** 第1期の特徴は「サブエージェントの役割を詳細に定義する」「ワークフローを臨機応変に動かす」「1つのメモが1つの作業単位」であり、生産性は高いが品質やルール遵守は低い傾向があった。第2期の特徴は「ドキュメントをシンプルに保つ」「定義されたワークフロースキルに従って動かす」「複数の作業をまとめたサイクル単位で管理する」であり、品質は向上したがコンテキスト消費が増加した。どちらの方針が適切かはプロジェクトの目標次第。

- **オーナー宛てメモを無断でトリアージするなど、「他エージェント宛てのアイテムに手を出す」という逸脱は設計上の抜け穴になりやすい。** ディレクトリ分割などでエージェントごとにスコープを切っても、明示的な制約がなければ他のスコープに手を出す挙動が発生する。エージェントが「自分のスコープ外を扱わない」というルールは、単に分割するだけでなく、それを強制する仕組みも合わせて設ける必要がある。

## LLMの挙動に関する実証的知見

- **CLAUDE.mdなどのシステムプロンプト的な指示は、セッションが長くなるほど強制力が低下する。** CLAUDE.mdの指示は強制力の低いユーザーメッセージとして、かつコンテキストの冒頭にのみ挿入されるため、セッションが長引くほど後続の指示や作業内容に埋もれ、遵守率が落ちていく。長期セッションでのルール維持には、この特性を踏まえた別の仕組み（例：チェックリスト方式、スキルによるリマインド）が必要。

- **複数の方法が共存する「中間状態」はLLMがうまく使いこなせない。** 段階的な移行として「古い方法」と「新しい方法」を一時的に併存させると、LLMはドキュメントに明示的に「使うな」と書かれていても古い方法を再発見して使い続けることがある。これは「段階的移行はうまくいかない」という教訓であり、シンプルで汎用性の高い1つの仕組みに一気に切り替えてしまう方が、中間状態を厳密に設計・管理する手間よりも少ない工数で安全に移行できる場合がある。

- **Claude Codeの多重起動防止（`CLAUDECODE`環境変数）は迂回されるリスクがある。** `CLAUDECODE`環境変数を削除してサブプロセスを起動することで、多重起動防止が機能しなくなる。これにより、フックなどが意図しないプロセスでも発火し、予期しない副作用（例：他エージェントへの指示がオーナーからのメッセージとして記録されてしまう）が生じうる。Claude Codeの多重起動防止を前提とした設計（フックの実装など）では、この迂回パターンへの対策（例：`unset CLAUDECODE`を禁止するフックのセット）も合わせて実装することが望ましい。
