# character-personality の判定の解析（cycle-284・全数検査）

サイト全 PV の 31% を占める `/play/character-personality` の**判定そのもの**を、
4択 × 12問 = **4^12 = 16,777,216 通りの全数列挙**で解析した記録。推定でも標本でもない。

**計器一式（重み表・判定規則・∀版の理想回答者テスト・全数検査）は `character-personality-scoring-cycle284/` に恒久化した**（同ディレクトリの README を先に読むこと）。

**この文書は cycle-284 の成果を恒久化したものである**（`tmp/` は保証の無い場所であり、
`.claude/rules/tmp-directory.md` により `docs/` から参照してはならないため）。
B-575 の次の着手時は、まずこの文書から始めれば 4^12 検証と 48 重み設計をやり直さずに済む。

## 2. 判定の構造（現物で確定）

24 タイプ ＝ **6 アーキタイプ**（`commander` / `professor` / `dreamer` / `trickster` / `guardian` / `artist`）の**対**。
コード註が 24/24 で明示（例: `// #1: blazing-strategist — commander × professor`）。

- 内訳: **15 組**（C(6,2)）＋ **同型 6**（X × X）＋ **逆順 3**（#22 guardian×professor / #23 artist×trickster / #24 guardian×commander は #8/#14/#4 の順序違い）。
- **36 の順序対すべてが実在タイプへ写像できる**（無ければ逆順で引く）＝取りこぼしゼロ。
- **同型強化タイプの本文は 6/6 が「あなたの◯◯が極限まで強まると…」**＝1 つのアーキタイプが突出した人を指す。判定規則はここから復元できる（発明ではない）。
- **設問の配点は既にアーキタイプ信号を encode している**（各選択肢の点を構成アーキタイプへ配ると 48 中 30 で単一アーキタイプが過半）。**設計意図（主軸×副軸）はデータの中に実在し、実行時に捨てられているだけ**（`scoring.ts` 冒頭が自ら `Scoring: method B (direct point allocation)` と書いている）。

## 3. 判定規則の案（cycle-284 で設計・受け入れ基準を通過）

1. 12 問の回答から **6 アーキタイプの得点**を合計する。
2. 1 位＝主軸、2 位＝副軸。
3. **1 位と 2 位の差が閾値以上なら「主軸 × 主軸」**（同型強化タイプ）。
4. そうでなければ「主軸 × 副軸」（順序対が無ければ逆順で引く）。

**閾値はデータから導出される**（恣意的な数値ではない）: 異型の本人 gap の最大 **9** < TH ≤ 同型の本人 gap の最小 **11** → **成立窓は [10, 11]**。

### 検証結果（TH=10・全数検査 16,777,216 通り）

- **理想回答者テスト: 24/24**（§4.1 の ∀ 版。現行は 2/24 で失敗＝通ることが自明でないゲート）
- **到達不能タイプ: 0 件**
- **主軸/副軸が決まらない回答**——**2つの値があり、定義差である（測定誤差ではない）**:
  - **20.50%**: **逆順フォールバックで一意に落ちる同点を「決着した」と数える**場合。実装上こちらが正しい——順序対 (P,S) が存在しなければ (S,P) を引くため、**主軸と副軸が入れ替わっても同じタイプが返る同点は、実際には決着している**。
  - **29.63%**: 主軸または副軸に同点があれば一律「未決着」と数える場合（保守的な上限）。
  - **重要な設計上の事実**: **逆順フォールバックが同点の約 9 ポイント分を無償で一意化している。** これは残る同点の決着規則を設計するときの直接の材料である（§3 の判定規則4「順序対が無ければ逆順で引く」の副次効果）。
  - いずれにせよ**現行の配列順決着 36.20% より改善**する。
- **一様測度での偏り**——同じく**定義差で2つの値がある**:
  - **68倍**（最大 7.31% / 最小 0.107%）: 未決着を除外して数えた場合。
  - **93.0倍**（最大 9.94% `blazing-strategist` / 最小 0.11% `ultimate-artist`）: 未決着を逆順フォールバックで一意化して全件数えた場合。
  - **異型18タイプのみなら 4.8〜6.6 倍**（現行 20 倍より改善）。
  - **同型 6 タイプ合計は 2.46% に潰れる**（`ultimate-artist` は TH=11 で 0.05%）。
- **正直な評価: これは明白な改善ではない。** 到達不能 0・理想回答者 24/24 は**測度非依存の真の勝ち**だが、一様測度での偏りは **20倍 → 68〜93倍に悪化**し、同型6タイプがほぼ誰にも届かなくなる。**「2タイプが本人に届かない」を「数タイプがほぼ誰にも届かない」と交換している。** 一様測度は来訪者の実像ではないが、**現行の 20 倍を同じ測度で断罪した以上、置換案でも同じ測度で正直に見る**（留保の選択的適用をしない）。

### 未解決（出荷前に決着が要る）

- **残る同点 20〜30% の決着規則。** ここを配列順に落とせば、36.20% を 20〜30% に減らしただけで病は残る。
- **同型 6 タイプの潰れ**（一様測度で合計 2.46%）。一貫して答える人ほど gap は開くはずだが、**測定していない**。
- **48 重みの人の目での読了**（設計はサブエージェント・PM 未読了）。
- **設問セット自体の非対称**: 各アーキタイプが主(3点)になれる設問数は、文面の素直な読みでは **professor 9・artist 7・他 8 が限界**で、完全な均等は現在の 12 問では**不可能**。＝配点だけでは均等化できず、**設問文にも手を入れる必要がある可能性**。

## 4. 受け入れ基準（次の着手時も維持する）

### 4.1 理想回答者テスト — **定義がすべてである（∃版は空虚なゲートになる）**

**「そのタイプの点を最大化する回答」という §1 の定義は、新設計には転用できない。** X×Y に単一の最大化元は存在しないため（単一アーキタイプの最大化は、意味論上その人が X×X＝同型強化である）。**必ず以下の分割モデルを使うこと。**

**分割モデル（∀版・これが正典）**:

1. 各設問を、X を最も強く押す選択肢の集合と Y を最も強く押す集合で分類する（`Q_X` / `Q_Y` / 両方が同一選択肢の `Q_both` / どちらでもない `Q_none`）。
2. 「X 寄りに答えた設問数 k」を、**X が主軸になりうる最小の k（kmin）**に固定する。
3. **その k における全分割について、例外なく X×Y が返ること**を合格条件とする（**∀**＝全称。「そういう分割が一つでも存在する」＝∃ ではない）。
4. 同点の選択肢は**副軸寄り**を選ぶ（配列順で選んではならない。下記の罠を参照）。
5. X×X の本人は「全問で X を最も強く押す選択肢を選ぶ人」。

**∃版（「そういう分割が存在すれば合格」）を使ってはならない。** 実測（cycle-284 のレビュアーが検定）: **∃版では TH=3〜12 のすべてが 24/24 を通り、重みをランダム化したゴミ設計すら 5 回中 1 回が通過する**。検出力がない＝**空虚なゲート**である。∀版は同じゴミ設計で最良 21/24 に留まり、th=10/11 でのみピークが立つ＝**検出力は本物**。

**閾値の導出も分割モデルに依存する**: 異型の本人 gap の**最大 9** < TH ≤ 同型の本人 gap の**最小 11** → **成立窓 [10, 11]**。この 9 と 11 は上記 ∀ 版でのみ再現する。

### 4.2 基準

1. **理想回答者テスト（§4.1 の ∀ 版）が 24/24 通ること**（測度非依存・現行 2/24 で失敗＝自明でない）。
2. 全数検査を再実行し、同点率・偏り・到達不能タイプ数を報告すること。**数値は定義を併記すること**（§3 の注記を参照）。
3. **残る同点の決着を配列順にしないこと。**

### 4.3 再現手順

- **全数検査**: 12問×4択の meet-in-the-middle。前半6問の全 4^6=4,096 通りの部分和（24次元 or 6次元）と、後半6問の 4,096 通りを総当たりで加算すれば 4^12=16,777,216 通りを数十秒で網羅できる（4,096 回のループ × 4,096行のベクトル演算）。
- **判定の実装**は `src/play/quiz/scoring.ts`（現行は `direct point allocation` ＋ strict `>` による配列順 tie-break）。**tie-break の再現は、実際の `determineResult` に数万通りを通して自作モデルと突き合わせて確認すること**（cycle-284 のレビュアーは 30,000/30,000 一致を確認している）。
- **配点表の抽出**は正規表現に頼らず、**実モジュールを import してダンプし突き合わせる**のが安全（cycle-284 のレビュアーは 192/192 エントリの byte 一致を確認）。

### 4.4 計器を書くときの罠（cycle-284 で PM が実際に踏んだ）

- **同点の選択肢を「配列の若い順」で選ぶコードを書かないこと。** cycle-284 の PM は検証スクリプトの `best_for` でこれを踏み、**このサイクルの主題そのもののバグを計器に仕込んだまま、合格している設計を棄却した**。
- **「全通り試した」と書く前に、探索した件数と全体の件数を数字で突き合わせること。** 同 PM は 4,096 分割中 11 本（0.27%）の連続プレフィックスを「全通り」と書いた。設問の並び順に意味は無いのに「最初の nx 問だけ X 寄り」という恣意的な形しか探索していなかった。
- **自分に不利な結果と有利な結果で、検証の強度を変えないこと**（AP-P02）。同 PM は他人の（作業が増える）結果には最大限の検証を掛け、自分の（作業を終わらせられる）結果には一切掛けなかった。
- **緩い計器で「通った」ことにしないこと**（§4.1 の ∃ 版の罠）。**計器そのものを、既知のゴミ設計に対して検定すること**——通ってしまうなら、その計器は無効である。
