# cycle-210 事前調査: ターゲットユーザー定義と text-replace タイル UI 設計の対応関係

作成日: 2026-05-26  
対象ツール: text-replace（テキスト置換）  
調査目的: タイル化計画立案のための判断材料整理

---

## 1. 各ターゲットユーザーの likes / dislikes と text-replace 関連ユースケース

### T1: 特定の作業に使えるツールをさっと探している人（初回来訪者）

**text-replace に関係する likes（yaml 引用）:**

- 「ページを開いた瞬間に入力欄が見えて、すぐ使い始められること」
- 「コピペで結果を受け取って、すぐ元の作業画面に戻れること」
- 「余計な説明や装飾がなく、用事だけ静かに片付けられる画面」
- 「入力した内容が外部に送られないなど、不安なく使えること」

**text-replace に関係する dislikes（yaml 引用）:**

- 「開いた瞬間に広告やポップアップで目的の入力欄が埋もれていること」
- 「使い始めるのに会員登録やアプリのインストールを求められること」
- 「仕様や前提が曖昧で、出てきた結果を信用していいのか判断できないこと」
- 「ツール冒頭に長い解説記事が挟まっていて、すぐ使えないこと」

**典型ユースケース（yaml の search_intents から推定）:**
T1 の search_intents には「文字数カウント」「全角半角変換」「改行削除」などが含まれる。text-replace に対応する検索語句として想定されるのは「テキスト置換 オンライン」「文字列置換 一括」「文章 置き換え ツール」などで、これは meta.ts の keywords と一致する。

**T1 の利用シナリオ（推定）:**

1. 議事録を社外送付前に「様」→「さん」に統一したい
2. 設定ファイルをコピペして「localhost」→「本番ドメイン」に書き換えたい
3. PDFからコピペしたテキストの「・」→「-」に統一したい
4. メールのテンプレートで「○○様」→「△△様」に変換したい

T1 は「1 回の置換で終わる」シナリオが大半。検索文字列と置換文字列を入れてコピーする 3 アクションで完結することを期待している。

---

### T2: 気に入った道具を繰り返し使っている人（リピーター）

**text-replace に関係する likes（yaml 引用）:**

- 「ブックマークしたURLを開けばすぐ目的のツールが表示されること」
- 「前回入力した値や設定が残っていて、作業がさらに短縮されること」
- 「同じ入力に対して前回と同じ結果が返ってくること」
- 「仕様変更があっても、これまでの使い方が壊されないこと」
- 「サイト内のすべてのツールやゲームの操作性やトーン&マナーが一貫していること」

**text-replace に関係する dislikes（yaml 引用）:**

- 「動作が遅いツール」
- 「URLが変更されたコンテンツにリダイレクトが設定されておらずたどり着けなくなること」
- 「以前と同じ入力なのに結果や挙動が前回と変わっていること」

**T2 の利用シナリオ（推定）:**

1. 毎週の議事録フォーマット統一を繰り返す（「、」→「，」など）
2. コードリリース前の設定値書き換えを毎回同じ手順で実行する
3. 定期的に送るメールテンプレートの宛名差し替え
4. タイルページをダッシュボード代わりにして、複数ツールを渡り歩く

T2 は「覚えた手順がそのまま使えること」を最重視する。タイル UI でブックマークして、開いたらすぐ同じ UI が同じ場所にあることが重要。

### S3: Webサイト製作を学びたいエンジニア

**text-replace への接点は低い。** このターゲットは「手元ですぐ試せるコード例」「設計判断の背景にある『なぜそうしたか』の説明」を求めており、テキスト置換ツール自体を使うニーズは薄い。強いて言えば、設定ファイルや正規表現チートシートを探す文脈で間接的に辿り着く可能性はあるが、主ターゲットではない。

**タイルへの貢献度: 低い（ニーズなし）**

---

### S1: AIエージェントやオーケストレーションに興味があるエンジニア

**text-replace への接点は低い。** このターゲットは「AIエージェントのワークフロー」「オーケストレーション設計」に興味があり、テキスト置換ツールは彼らの課題領域とほぼ交差しない。yaml には「実践的な話が少ない一般論」を嫌うとあり、ツール系コンテンツへの能動的な来訪は期待しにくい。

**タイルへの貢献度: 低い（ニーズなし）**

---

### M2: AIの日記を読み物として楽しむ人

**text-replace への接点はほぼない。** このターゲットはブログ「AIの日記」の読者であり、ツールへの直接的なニーズはない。サイト傍流ポジションであり、ツールページへの回遊は想定するが、text-replace タイル設計においては考慮不要。

**タイルへの貢献度: なし**

## 2. タイル UI で完結すべき最小機能（T1 / T2 視点）

### T1 が「タイル UI を初めて見て必要としているもの」

T1 の最大の要件は「開いた瞬間に使い始められること」（likes 引用）と「余計な説明なく用事だけ片付けられる画面」（likes 引用）。

タイル初見で T1 が期待するのは以下の 3 要素のみ：

1. **入力テキスト欄** — 貼り付けてすぐ使えること
2. **検索文字列 / 置換文字列の入力欄** — ラベルが明確であること
3. **結果表示 + コピーボタン** — 1 クリックでコピーできること

T1 は「正規表現」を知らない・使わない可能性が高い（yamlの `doesnt_know` に「AI 運営」が含まれる層であり、技術的な深さへの期待は低い）。「正規表現」「大文字小文字を区別」「先頭のみ置換」などのオプションを初見で見せることは T1 の dislikes「仕様や前提が曖昧」に反する認知負荷を与えかねない。

**T1 のタイルに必要な最小機能:**

- 入力テキスト（rows=2 程度、コンパクト）
- 検索文字列入力
- 置換文字列入力
- 置換結果の表示
- コピーボタン
- 「詳細ページで開く」リンク

**T1 のタイルに不要な機能（詳細ページに退避）:**

- 正規表現トグル
- 大文字小文字トグル
- 先頭のみ置換トグル

### T2 が「タイルに繰り返し戻って効率化したいこと」

T2 の最大の要件は「前回入力した値や設定が残っていて、作業がさらに短縮されること」（likes 引用）。

text-replace の場合、T2 が繰り返す典型パターンは「毎回同じ置換パターン（検索文字列・置換文字列）を使う」こと。ただしタイルには state 永続化（localStorage）が現時点で実装されていないため、これはタイルではなく詳細ページの責務として扱うのが現実的。

タイルとして T2 に提供できる価値は：

- 操作の一貫性（「前回と同じ UI が同じ場所にある」= likes 引用）
- すぐ操作できる UI（ブックマーク → 開く → 入力 → コピーの流れが途切れない）

**T2 のタイルに有効な機能:**

- T1 と同じ最小機能（入力 + 検索・置換欄 + 結果 + コピー）
- 置換件数の表示（「N件置換しました」= 現在の詳細ページにある機能。信頼性確認になる）

### 「これがあれば 90% のユースケースで完結する」機能セット（推定）

典型的なユースケース（全角半角統一、議事録整形、テンプレート差し替え、設定ファイル書き換え）の大半は：

- 固定文字列の全件置換（globalReplace: true, useRegex: false, caseSensitive: true がデフォルト）

つまり現在の logic.ts のデフォルト設定（useRegex: false, caseSensitive: true, globalReplace: true）がそのまま最大公約数になる。

**タイルは「デフォルトオプション固定での全件置換」に特化する。**
オプション操作を一切省略し、3 欄（入力・検索・置換）+ 結果 + コピーで完結させる。

## 3. 詳細ページにあるべき機能（タイルに載せない機能の振り分け）

| 機能                     | タイル   | 詳細ページ     | 理由                               |
| ------------------------ | -------- | -------------- | ---------------------------------- |
| 入力テキスト欄（rows=2） | 掲載     | 掲載（rows=8） | タイルは短文 → 詳細は長文          |
| 検索文字列入力           | 掲載     | 掲載           | 必須                               |
| 置換文字列入力           | 掲載     | 掲載           | 必須                               |
| コピーボタン             | 掲載     | 掲載           | 必須                               |
| 置換件数表示（N件）      | 掲載検討 | 掲載           | 信頼性に寄与、追加コスト小         |
| 正規表現トグル           | 不掲載   | 掲載           | T1 認知負荷、T2 用途は詳細ページへ |
| 大文字小文字トグル       | 不掲載   | 掲載           | 同上                               |
| 先頭のみ置換トグル       | 不掲載   | 掲載           | 同上                               |
| 詳細ページリンク         | 掲載     | 不要           | タイルから誘導する                 |

**詳細ページに来やすいターゲット:**

- T2 の中で「正規表現を知っているユーザー」（開発者・データ処理者）
- T1 の中で「1 回使ってみて機能不足を感じたユーザー」が詳細ページリンクから遷移する

## 4. 構造的差別化のポイント（T1 / T2 視点）

yolos.net の差別化軸は「登録不要・広告控えめ・シンプル UI」（CLAUDE.md の方向性）。text-replace において「他サイトに勝てる体験」として考えられる点：

**T1 への差別化:**

- **入力した瞬間にリアルタイム置換** — ボタン押下不要、即座に結果が見える（現在の実装は useMemo による即時反映で対応済み）
- **置換件数の明示** — 「本当に置き換わった？」という不安を件数表示で解消（T1 likes: 「結果の根拠や前提が必要最小限だけ添えられており、信頼して使えること」引用）
- **オプション 0 で使える** — 競合サイトの多くは正規表現・フラグなどのオプションが初期表示されており、T1 を混乱させる。yolos.net タイルは 3 欄（入力・検索・置換）+ コピーだけで完結

**T2 への差別化:**

- **タイルページで即座に利用できる** — 毎回 text-replace の詳細ページを探す必要なく、ツール一覧（タイルページ）から直接使える
- **一貫した UI** — T2 likes「サイト内のすべてのツールやゲームの操作性やトーン&マナーが一貫していること」に応える。既存 10 タイルと同型の UI

## 5. 見落としやすいユースケース

### 単純 1 対 1 置換以外で典型的に発生するパターン

1. **複数箇所の一括置換（これは既にデフォルトで対応）**
   全件置換（globalReplace: true）がデフォルトなので問題なし。

2. **空文字への置換（削除）**
   「不要な文字を削除したい」= 置換文字列を空にする。これは現在の実装で動作する。タイルでも置換欄を空にすれば自然に対応できる。

3. **全角スペース ↔ 半角スペースの置換**
   これは fullwidth-converter と棲み分け上は「fullwidth-converter の仕事」だが、「特定の文字だけ」変換したい場合（全体でなく、特定文脈の全角スペースのみ）は text-replace が適している。タイルでも対応可能。

4. **改行コードを別の文字に置換**
   改行 `\n` → スペースや記号への置換は text-replace で対応できるが、**UIとして直感的に入力が難しい**（改行をテキスト欄に貼り付けた結果、どう表現されるか分かりにくい）。この用途は line-break-remover に誘導する方が適切。

5. **引用符の統一**（`"` → `「`, `"` → `」` など）
   これは text-replace が最も適している。複数の置換を連続して行う場合、タイルでは 1 回ずつ実行する必要があり、詳細ページへの誘導が自然。

6. **タブ文字 ↔ スペースへの変換**
   タブ文字のテキスト欄入力は難しい（Tab キーがフォーカス移動になる場合あり）。これはタイルで完結させにくいため詳細ページで扱う想定。

## 6. 既存ツールとの棲み分け

| ツール                  | 担当するユースケース                                      | text-replace との境界                                   |
| ----------------------- | --------------------------------------------------------- | ------------------------------------------------------- |
| **fullwidth-converter** | 英数字・カタカナ・記号の全角 ↔ 半角を文字種単位で一括変換 | 文字種全体の変換。特定文字のみなら text-replace         |
| **kana-converter**      | ひらがな ↔ カタカナ、半角カナ ↔ 全角カナの変換            | 仮名変換に特化。汎用の文字列置換は text-replace         |
| **line-break-remover**  | 改行の削除・スペース置換・PDFスマートモード               | 改行コード操作に特化。改行以外の文字置換は text-replace |
| **url-encode**          | URL エンコード / デコード                                 | エンコード処理に特化。汎用置換は text-replace           |

**「どちらのツールに来るべきか迷わない設計」の指針:**

- **文字種の全体変換（入力テキストの英数字を全部半角にする）** → fullwidth-converter
- **仮名変換（ひらがなをカタカナに変換する）** → kana-converter
- **改行の除去（PDFコピペのガタガタを直す）** → line-break-remover
- **任意の文字列を任意の文字列に置き換える** → text-replace

text-replace は「汎用」であり、上記ツールが特化していない置換はすべて text-replace の守備範囲。来訪者が迷うケースとして「全角スペースだけを半角スペースに変えたい」があるが、これは fullwidth-converter だと文字種を「スペース」に限定する操作が必要で煩雑なため、text-replace が現実的な回答になる。

meta.ts の relatedSlugs に両方向のリンクがあるため、詳細ページ間の相互誘導は既に設定されている。

## 7. タイル UI 設計への総合推奨（planner 向け判断材料）

### タイルに載せるべき機能（確定）

1. **入力テキスト欄**（rows=2 固定、flexShrink: 0）
2. **検索文字列入力欄**（ラベル: 「検索」または「検索文字列」）
3. **置換文字列入力欄**（ラベル: 「置換」または「置換文字列」）
4. **置換結果表示エリア**（role="status" aria-live="polite"、flex: 1 + overflowY: auto）
5. **コピーボタン**（結果あり時のみ表示）
6. **詳細ページリンク**（「詳細ページで開く」）

### タイルに載せるべき機能（検討）

- **置換件数表示**（「N件置換しました」）— T1 の信頼性ニーズに応える。追加コストは小さく、既存詳細ページにもある機能。**掲載推奨。**

### タイルに載せないもの（詳細ページの責務）

- 正規表現トグル（useRegex）
- 大文字小文字トグル（caseSensitive）
- 先頭のみ置換トグル（globalReplace の OFF）

### オプション固定値

- useRegex: false（固定）
- caseSensitive: true（固定）
- globalReplace: true（固定） = 全件置換がデフォルト最大公約数

### UI レイアウトの検討

既存タイルの標準サイズ（cols=3 rows=2 = 400×264px）に対し、text-replace は他のタイルより入力欄が多い（入力テキスト + 検索欄 + 置換欄 + 結果欄の 4 欄）。以下の 2 案が候補：

**案 A: cols=3 rows=2（標準サイズ）**

- 入力欄・結果欄を rows=1 まで圧縮する必要があり、長文には不向き
- T1 の「すぐ使えること」に合致（タイル一覧での占有面積が少ない）

**案 B: cols=3 rows=3（やや大きめ）**

- QR コードタイルと同様の扱い（tile-declarations.ts 参照）
- 入力欄・結果欄をそれぞれ rows=2 程度確保できる
- text-replace は「貼り付けてすぐ確認する」ユースケースが主なので、ある程度の高さが必要

**推奨: 案 B（cols=3 rows=3）**
理由: text-replace の主ユースケースは「複数行テキストを貼り付けて一括置換する」であり、rows=2 では入力テキストが 1〜2 行しか見えず、T1 が「本当に置換されたか」を視認しにくい。T1 likes「結果の根拠や前提が必要最小限だけ添えられており、信頼して使えること」に反する可能性がある。

ただし、Playwright での実物確認（w375 / w1200）で最終判断すること。
