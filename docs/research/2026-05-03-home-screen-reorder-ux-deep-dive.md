# ホーム画面アプリ並び替え操作 UX 個別深掘り調査

## 前提：既存調査（2026-05-01）との差分

既存レポート `2026-05-01-dashboard-toolbox-phase1-best-practices.md` では以下が概観レベルで記述されていた。

- iOS：「長押しでジグルモードに入る。2モード分離」
- Android：「長押し→ドラッグ。編集モード不要」
- グリッド密度・削除安全策・編集モードの心理的位置付け・iPadOS の設計思想：**未記述**

本調査はこれらを個別操作レベルで深掘りする。

---

## 1. iOS / iPadOS のホーム画面 UX

### 1-1. ジグルモード（編集モード）の発動条件

**発動方法（iOS 13 以降、iOS 26 時点）：**

- ホーム画面の空白エリアをタップ＆ホールド（長押し）
- アプリアイコンを長押しし、コンテキストメニューから「ホーム画面を編集」を選択
- iOS 26 では「Edit」ボタン（左上）→「Customize」パネルで視覚スタイルも変更可能

発動後は全アイコンが一斉に揺れる「ジグル（wiggle）アニメーション」が始まる。Apple 自身の特許（US 8,423,911 B2、2013年取得）では「各アイコンが平均位置を軸に振動する」と表現されており、「固定レイアウトが流動的になった」ことを示す視覚フィードバックとして設計されている。

**ジグルモードの起源：** iPhone OS 1.1.3（2008年）が起源。当初はホーム画面の並び替えを on-device で行える最初の機能として導入された。

### 1-2. 解除条件

以下のいずれかで終了する。

| 解除方法                           | 備考                                    |
| ---------------------------------- | --------------------------------------- |
| 右上「完了（Done）」ボタンをタップ | iOS 13 以降で追加。最も明示的な終了操作 |
| ホーム画面の空白エリアをタップ     | ドラッグ中でなければ解除される          |
| サイドボタン（電源ボタン）を押す   | 画面ロックと同時に解除                  |
| ホームボタンを押す（旧機種）       | Face ID 機種では使用不可                |

変更は**自動保存**される。「保存」ボタンは存在しない。

### 1-3. ジグルモード中の視覚表現

- **揺れアニメーション**：全アイコンが連続して振動。「編集可能状態」の強力な状態表示
- **削除バッジ**：アイコン左上（あるいは右上）に「−（マイナス）」または「×」バッジが出現。アプリ削除の入口
- **ドックも編集対象**：ドック内アイコンも同様に揺れ、並び替え・追加・削除が可能
- **フォルダ展開**：フォルダも揺れ、内部のアイコンも編集可能状態になる

iOS 26 では Liquid Glass デザイン言語が導入され、アイコンの透明度やガラス感が増したが、ジグルアニメーション自体の仕組みは変更されていない。

### 1-4. グリッド密度（実機標準）

**iPhone（iOS 18 / iOS 26 時点）：**

| 状態                     | グリッド                                            |
| ------------------------ | --------------------------------------------------- |
| ウィジェットなし（標準） | 4 列 × 6 行 = 24 アイコン / ページ                  |
| ウィジェットあり         | 4 列 × 5 行 = 20 アイコン（ウィジェット占有分除く） |

iOS 18 から「任意の位置に配置」が可能になり、グリッドの隙間に空白を作れる。ただし**アイコンはグリッドにスナップする**（任意ピクセル配置ではない）。iPhone mini から iPhone Pro Max まで全モデルで同一グリッド。

**iPadOS（12.9インチ iPad Pro 等の大型モデル）：**

| 向き           | グリッド                  |
| -------------- | ------------------------- |
| 縦 (Portrait)  | 4 列 × 6 行 = 24 アイコン |
| 横 (Landscape) | 6 列 × 4 行 = 24 アイコン |

注目すべき点は、**iPad Pro（12.9インチ）でも iPhone と同じ 24 アイコン/ページ**という事実である（後述 §3 で詳説）。

### 1-5. ページ切替（横スワイプ）

ジグルモード中でも、画面端までアイコンをドラッグすると次のページへ自動遷移する。このページ間移動は「フィードバックが乏しく、意図しない画面遷移が起きやすい」という批判が多い（9to5Mac の分析）。iOS では複数アイコンの一括移動も可能（1本指でアイコンを掴みながら別の指で追加タップ）が、「操作が難しい」「実用的でない」という評価が多い。

### 1-6. フォルダ機能

- **作成**：ジグルモード中に 1 つのアイコンを別のアイコンにドロップすると自動作成
- **名前変更**：フォルダ展開後にジグルモードで名前欄をタップ
- **並び替え**：フォルダ内のアイコンも個別にドラッグで並び替え可能
- **ページ切替**：フォルダが 9 アイコン（3×3）を超えると内部でもページが生成される

### 1-7. ウィジェット並び替え（Smart Stack 含む）

- ウィジェットは Small（2×2 グリッド占有）/ Medium（4×2）/ Large（4×4）の 3 サイズ
- **Smart Stack**：同サイズのウィジェットを重ねたもの。ジグルモードで「スタックを編集」メニューが出現し、スタック内の並び順をドラッグ変更できる
- Smart Stack 内でのスワイプは「上下スワイプで次/前のウィジェットに切替」（通常利用時）と「ジグルモード中のドラッグで並び替え」で操作が完全分離されている
- Smart Rotate 機能：Siri インテリジェンスが利用パターンに基づいてウィジェットを自動切替

### 1-8. 削除操作の安全策

iOS のアプリ削除には**2ステップの確認プロセス**がある。

1. ジグルモード中にアイコンの「−」バッジをタップ
2. 「App を削除」ダイアログが出現（「App を削除」または「ホーム画面から削除」を選択）
3. 「App を削除」を選んだ場合、さらに「削除」の確認タップが必要

**Undo 機能は存在しない**。ただし App Store から再インストールは可能（データは iCloud バックアップがあれば復元可能）。削除の 2ステップ確認は「ホーム画面からの削除（Dock に移動等）」と「アプリ完全削除（アンインストール）」を明確に分離する安全設計になっている。

Screen Time の「制限」設定から App 削除を完全に無効化することも可能（子供の端末管理等に使用）。

### 1-9. ジグルモードの UX 批評（一次情報）

9to5Mac（2023年10月、Ben Lovejoy 著）の分析が最も詳細で、以下の批判が列挙されている。

1. **誤操作の連鎖**：「意図した画面を越えてドラッグしてしまいやすい」「意図しない位置に落としやすい」「意図しないフォルダを作りやすい」
2. **儀式的摩擦**：「並び替えが大変なので後回しにしがち」。整理作業を「やりたいこと」ではなく「覚悟のいる作業」にしてしまっている
3. **一括操作の限界**：マルチセレクト（1本指保持しながら別指でタップ追加）は「理論上可能だが実際は難しい」
4. **フル画面をオーバーフローしたときの挙動**：ページが満杯のとき新アプリを追加すると既存アプリが意図せず次ページへ押し出される

一方で、ジグルモードの設計意図は明確で評価できる点もある。

- 「固定状態」と「編集状態」を強力な視覚差異で分離し、**モードスリップ（意図せずモードを間違える）を防ぐ**
- 揺れアニメーションは「このアイコンは今つかんで動かせる」というアフォーダンスを視覚で伝える
- Apple はジグルモードを特許として保護するほど、この 2モード設計を設計哲学のコアとして位置付けている

## 2. Android（Pixel Launcher / Samsung One UI）のホーム画面 UX

### 2-1. 基本的な差異：Android は「ジグルモードなし」

Android ホーム画面の最大の特徴は、**専用「編集モード」を持たない**ことである。

| 操作                 | iOS（ジグルモード）              | Android（Pixel / One UI）              |
| -------------------- | -------------------------------- | -------------------------------------- |
| 並び替え開始         | 長押し→全体がジグル              | 長押し→**そのまま**ドラッグ開始        |
| 並び替え中の状態表示 | 全アイコンが揺れる（モード表示） | 掴んでいるアイコンのみが浮き上がる     |
| 並び替え終了         | 「完了」ボタンまたは空白タップ   | ドロップで即完了（モード終了なし）     |
| 削除                 | バッジタップ→2段階確認           | アイコンを上部「削除」ゾーンへドラッグ |

### 2-2. Pixel Launcher の詳細動作

**発動：** アイコンを長押しすると振動フィードバック（ハプティクス）とともにコンテキストメニューが出現。そのままドラッグを続けるか、メニュー外にドラッグすることで並び替えモードへ。

**コンテキストメニューの内容：** 「アプリ情報」「削除」「ウィジェット」等の選択肢が表示される。並び替えのためには「メニューをスキップしてドラッグ継続」が必要。

**削除の仕組み：** ドラッグ中に画面上部に「削除」ゾーン（赤いゴミ箱アイコン）が出現。そこへアイコンをドロップすると削除確認ダイアログが出る。

**Pixel Launcher の重大な欠陥（Android Police 調査）：**

1. **レイアウトロック機能なし**：一度整理したホーム画面を誤ってドラッグで崩してしまうリスクが常に存在。Samsung One UI や Nova Launcher はこれを「ホーム画面レイアウトをロック」機能で防止しているが、Pixel Launcher には 2026年時点でも実装されていない。
2. **ページ並び替え不可**：複数ページのホーム画面でページ順を変更する機能が存在しない。一覧表示（サムネイル）で管理することもできない（Android Police 調査）。
3. **一括移動不可**：アイコンを一度に 1 つしか動かせない。iOS のマルチセレクトドラッグに相当する機能なし。

**グリッド密度（Pixel Launcher）：**

| オプション        | アイコン数/ページ |
| ----------------- | ----------------- |
| 5×5（デフォルト） | 25                |
| 4×5               | 20                |
| 4×4               | 16                |
| 3×3               | 9                 |
| 2×2               | 4                 |

Android 12 で 4×5 オプションが追加。Pixel タブレット向けには 6×5（30 アイコン）オプションも存在する（Android 13 Beta 3 で追加）。

### 2-3. Samsung One UI の詳細動作

**アクセス方法：** ホーム画面の空白を長押しまたはピンチインジェスチャーでホーム画面設定パネルへ。

**One UI の優位性：**

1. **「ホーム画面レイアウトをロック」機能**：Settings > Home Screen > Lock Home Screen Layout。有効にするとアイコン・ウィジェット・ショートカットが固定され、誤ってドラッグで動かすことを防止。子供に貸すときや誤操作防止に有効。
2. **ページ一覧（サムネイル）表示**：ピンチインで全ページのサムネイル一覧を表示でき、ページ単位での削除・並び替えが可能。Pixel Launcher にはない機能。
3. **グリッドサイズの豊富な選択肢**：4×5、4×6、5×5、5×6 から選択。One UI 7 以降はデフォルトで 4×6 と 5×6 のみ提供（Good Lock の Home Up プラグインで 4×5 等も復活可能）。

**One UI のフォルダ拡大機能（One UI 7）：** フォルダを長押し→「拡大」でフォルダが大きくなり、内部のアイコンがより見やすくなる。iOS にはないフォルダ表現の拡張。

**One UI 7 の新機能（2025年）：** ウィジェットにラベル（名前）が付くようになり、アイコンとウィジェットが統一されたスタイルで整列。「パズル形式の自動配置」機能が検討されていることも報告されている（Sammy Fans、2025年2月）。

### 2-4. Pixel Launcher vs Samsung One UI の比較表

| 機能                       | Pixel Launcher                     | Samsung One UI                       |
| -------------------------- | ---------------------------------- | ------------------------------------ |
| 編集モード専用 UI          | なし（直接ドラッグ）               | なし（直接ドラッグ）だが設定パネル有 |
| レイアウトロック           | **なし**                           | **あり**                             |
| ページ一覧・並び替え       | **なし**                           | **あり**                             |
| 複数アイコン一括移動       | なし                               | なし                                 |
| グリッドカスタマイズ       | 4〜5種                             | 4種（One UI 7 デフォルト）           |
| ウィジェット追加・リサイズ | あり                               | あり（リサイズハンドル付き）         |
| 削除 Undo                  | なし                               | なし                                 |
| 設計哲学                   | シンプル優先（機能不足が批判対象） | 機能豊富（カスタマイズ重視）         |

## 3. iPadOS のホーム画面 UX：大画面設計の特殊性

### 3-1. グリッド密度がなぜ増えないのか

iPad Pro 12.9インチは iPhone の 3〜4 倍の画面面積を持つが、ホーム画面のアイコン数は iPhone と同じ 24 アイコン / ページである。これは Apple の意図的な設計判断であり、以下の複数の理由がある。

**理由 1：ウィジェットとの整合性**

iPadOS 15 以降、ウィジェットがホーム画面上に配置されるようになったが、アイコンとウィジェットの両方が同一グリッドに整列するよう設計されている。グリッドを高密度にするとウィジェットサイズとの整合が取れなくなる。

**理由 2：Apple の「ネガティブスペース」デザイン哲学**

Apple の HIG（Human Interface Guidelines）は、明確な視覚的階層・余白・読みやすさを優先する。高密度なグリッドはこれに反する。9to5Mac（Ben Lovejoy、2019年）はこの制約を「missed opportunity」と批判しつつも、「シンプルな UI はパソコンに苦手意識のあるユーザーに特に好評」と設計意図を認めている。

**理由 3：iPad の使用文脈**

iPad はタブレットとして手に持って使う場面も多く、指によるタップの誤操作を防ぐためにタップターゲットを大きく保つ設計が有利。高密度グリッドはこれに反する。

**ユーザーの不満：** 「12.9インチ iPad Pro で 24 アイコンしか表示できないのは空間の無駄」という批判が Apple Community でも多数見られる。サードパーティのジェイルブレイクツール（PadGrid 等）はこの高密度化ニーズに応えていた。

### 3-2. iPadOS の実際のグリッド数

**iPadOS 18（2024年）時点：**

| モデル / 設定              | Portrait | Landscape |
| -------------------------- | -------- | --------- |
| 標準 iPad（9〜11インチ）   | 4×6 = 24 | 6×4 = 24  |
| iPad Pro（12.9〜13インチ） | 4×6 = 24 | 6×4 = 24  |
| Small アイコン設定         | 5×6 = 30 | 6×5 = 30  |

iPadOS 18 以前（iPadOS 15〜17）：

| モデル                 | 標準グリッド                                       |
| ---------------------- | -------------------------------------------------- |
| iPad Pro / Air（大型） | 6×5 = 30（横）/ 4×6 = 24（縦）等、機種によって差異 |

iPadOS 18 から、iPhone 同様に「任意の位置への配置（空白を作る）」が可能になった。

### 3-3. Stage Manager との関係

Stage Manager（iPadOS 16 以降）は**ホーム画面の icon 並び替えとは完全に別の機能**である。

- Stage Manager 有効時：ホーム画面アイコンはそのまま存在するが、アプリを開くとホーム画面が隠れてアプリウィンドウが中央に表示される
- 左サイドには「最近使ったアプリのサムネイル一覧」が表示され、タップで切替
- Stage Manager は「マルチウィンドウ環境でのアプリ切り替え」を担当し、ホーム画面のレイアウト設計とは関係しない
- ホーム画面のアイコン並び替えは Stage Manager の有効/無効に関係なく、ジグルモードで行う

**大画面における並び替え UX の設計選択（Stage Manager 観点）：**
Stage Manager 環境では「アプリを探す」際にホーム画面に戻るよりも Dock や左サイドのサムネイルからアクセスする方が多くなる。このため、Stage Manager 利用者にとってホーム画面のアイコン並び替えの重要度は低下する。Apple は大画面での「作業効率」をホーム画面密度ではなく Stage Manager で解決するという設計思想を取っていると解釈できる。

## 4. 個別操作 UX の評価（yolos.net 道具箱への示唆）

### 4-1. 編集モード（ジグル等）は「儀式的」か「自然なフィードバック」か

**結論：低頻度ユーザーには儀式的になりうるが、設計次第でフィードバックとして機能する。**

判断根拠：

1. **NN/g の「モード設計ガイドライン」** によると、モードは「大量の機能を同一入力で扱うとき」に有効。ただし「少なくとも 2 つの視覚的指標を同時に使う」ことが必須。iOS のジグルモードは「揺れアニメーション」「削除バッジ」「完了ボタン」と3つの視覚差異を持ち、モード設計として優れている。

2. **低頻度ユーザー（M1b）の視点**：並び替えが「月に1回以下」のユーザーにとって、モードへの「入り方」を覚えておくことへの負担が生じる。9to5Mac の批判にあるように「やり方を知っていてもやりたくない」という心理的回避が起きる可能性がある。

3. **編集モードの利点（M1b 視点）**：一方で、「普通に使っているときに誤って並び替わることがない」という保護は M1b ユーザーが最も重視する「同じ道具が同じ場所にある安定性」を支える。Samsung One UI の「レイアウトロック」機能は、直接ドラッグ方式でも同様の安全性を確保しようとする試みである。

4. **「自然なフィードバック」として機能する条件**：
   - 編集モードの入り方が直感的（1ステップ以内）
   - 視覚的な状態差異が明確（背景変化、ドラッグハンドル表示等）
   - 終了が容易（「完了」ボタンが明確な位置に固定）
   - **誤って変わっても元に戻せる（Undo）**—これが iOS / Android ともに欠如しており最大の欠点

**yolos.net 道具箱への推奨（M1b 基準）：**
編集モードを採用する場合は、「誤ってレイアウトが変わらない」保護よりも「誤った場合に戻せる Undo」を優先すべきである。iOS / Android ともに Undo がないため、これがホーム画面 UX の最大の未解決問題となっている。yolos.net がこれを解決できれば競合優位になる。

### 4-2. モバイル w360 でのファーストビュー表示タイル数

**計算の前提：**

- 画面幅：360px（Android 標準的な mid-range 端末）
- 利用可能高さ：640〜780px（典型的デバイス、ブラウザナビゲーションバー・セーフエリアを除く）
- タイルサイズ：48dp（Material Design 最小タッチターゲット）〜88dp（一般的なアイコンサイズ）
- マージン：16px（左右）/ ガター：8px

**タイルサイズ別ファーストビュー表示数（w360 / h640）：**

| タイルサイズ       | 列数                                          | 表示行数 | ファーストビュー合計 |
| ------------------ | --------------------------------------------- | -------- | -------------------- |
| 80px（コンパクト） | 4列（マージン16+タイル4×80+ガター3×8=368≒OK） | 7行      | **28 タイル**        |
| 88px（標準的）     | 4列（16+4×88+3×8=388→やや超過。3列で検討）    | 6行      | **18 タイル（3列）** |
| 96px（大きめ）     | 3列（16+3×96+2×8=320）                        | 5〜6行   | **15〜18 タイル**    |

**実機プラットフォームとの比較：**

- Pixel Launcher デフォルト（5×5）= 25 タイル表示（ただし OS ネイティブグリッド、マージン不要）
- iOS 標準（4×6）= 24 タイル（OS ネイティブ）
- Web アプリ（ブラウザ）では OS ネイティブと異なり、**アドレスバー・ナビゲーションバーが利用高さを削る**

**実際の利用可能高さの推計（360×780 端末の場合）：**

- 総高さ：780px
- ブラウザアドレスバー：約56px（Chrome Android スクロール前）
- 固定ヘッダー（想定）：48〜56px
- 利用可能高さ：約668〜676px

タイル高 88px + ガター 8px = 96px / 行 → 約6.9行 = 実質 **6行表示**

4列 × 6行 = **24 タイルがファーストビューに収まる可能性がある**（ただし 88px では4列が厳しいため、3列×7行=21タイルが現実的）。

**設計推奨：** w360 での 4列配置を実現するには、タイルサイズを 72〜80px 程度に抑える必要がある。ただし最小タッチターゲット（iOS 44pt / Android 48dp）を確保するため、視覚サイズ 64px + パディング 8px = 実質タッチ 80px という構成が合理的。

### 4-3. 削除操作の安全策比較

| プラットフォーム    | 削除方法                                    | 確認                                  | Undo               |
| ------------------- | ------------------------------------------- | ------------------------------------- | ------------------ |
| iOS（ジグルモード） | バッジ「−」タップ → ダイアログ 2ステップ    | 「App を削除」→「削除」の 2タップ確認 | **なし**           |
| Android（Pixel）    | アイコンを上部「削除」ゾーンへドラッグ      | ドロップ後にダイアログ確認            | **なし**           |
| Android（One UI）   | 長押しコンテキストメニューから「削除」      | 確認ダイアログあり                    | **なし**           |
| **yolos.net 推奨**  | タイルの「×」ボタン（編集モード中のみ表示） | 確認ダイアログ + **Undo（30秒）**     | **実装を強く推奨** |

iOS も Android も Undo を持たない。これはホーム画面 UX の業界全体の未解決問題であり、yolos.net がウェブアプリとして Undo を実装することで差別化できる。

### 4-4. 大画面 / 小画面でのグリッド密度の違いと設計思想

| スクリーン                          | iOS / Apple                      | Android / Samsung        |
| ----------------------------------- | -------------------------------- | ------------------------ |
| スマートフォン（〜6.7インチ）       | 4×6 = 24（固定）                 | 4×4〜5×5、カスタマイズ可 |
| タブレット（iPad 11インチ）         | 4×6 = 24（スマートフォンと同じ） | 5×6 〜 6×5、デバイス依存 |
| 大型タブレット（iPad Pro 13インチ） | 4×6 = 24（**増えない**）         | Pixel Tablet：6×5 = 30   |

**Apple の設計思想：**
「大きい画面でも密度は増やさない」という一貫した方針。余白・タップターゲット・一貫性を優先する。Stage Manager や Split View で大画面の生産性を解決する戦略。

**Google / Samsung の設計思想：**
「大きい画面にはより多くのコンテンツを」というアプローチ。ユーザーがグリッドサイズをカスタマイズできる柔軟性を提供。

**Web アプリとしての推奨方針：**
yolos.net の道具箱はレスポンシブ設計が前提のため、デバイス幅に応じたグリッド変化は自然である。以下が推奨されるブレークポイント設計。

| ブレークポイント      | 推奨列数 | 根拠                          |
| --------------------- | -------- | ----------------------------- |
| w360（モバイル標準）  | 3〜4列   | タッチターゲット 48dp+ を確保 |
| w768（タブレット）    | 4〜5列   | iPad 標準相当                 |
| w1280（デスクトップ） | 5〜6列   | OS ホーム画面の密度相当       |

## 5. 並び替え方式の選択判断材料

### 5-1. iOS ジグルモード型（明示的編集モード）の採用判断

**採用すべき理由：**

- M1b（道具箱を毎日同じ使い方で開く）ユーザーが「誤って並び替わる」ことを防げる
- NN/g ガイドライン：「モード中の操作と通常操作を混在させると最大のリスク（モードスリップ）が発生する」
- 並び替え頻度が低い（セットアップ後はほぼ変更しない）ユーザー向けには、「儀式感」よりも「誤操作ゼロ」が優先される

**採用時の必須要件：**

- 編集モードへの入り口は 1 クリック以内
- モードの視覚差異は最低 2 つ以上（背景変化 + ドラッグハンドル出現等）
- Undo 機能（iOS/Android が提供していない唯一の差別化要素）
- 削除は 2ステップ確認必須（iOS 同様）

### 5-2. Android 直接ドラッグ型の問題点

- 「誤って動かしても戻せない」問題はそのまま引き継ぐ
- M1b の「慣れた操作手順が突然変わること」への懸念とも相性が悪い（誤操作で突然変わる）
- 採用するなら Samsung One UI の「レイアウトロック」相当機能が必須

### 5-3. NN/g「モード設計ガイドライン」の適用

NN/g の分析では、編集モードが有効に機能する条件として以下を挙げている。

1. モードの視覚差異を 2 つ以上提供する（ジグルモードは揺れ＋バッジ＋完了ボタンの 3 つで模範的）
2. データ損失や重大な誤操作が伴う操作は「別コントロール」を用意する（削除ボタンを編集モード中のみ表示）
3. モードの存在を「モードスリップ（利用者が意図せずモードを混同）」のリスクと常に天秤にかける

yolos.net の道具箱では「ツールを使う（= 通常モード）」と「ツールを並び替える（= 編集モード）」の 2 ユースケースが明確に分離されているため、**明示的編集モードとの相性が高い**。

## 6. まとめと設計判断サポート表

| 判断項目                 | iOS ジグルモード         | Android Pixel          | Android One UI         | yolos.net 推奨            |
| ------------------------ | ------------------------ | ---------------------- | ---------------------- | ------------------------- |
| 編集モードの有無         | あり（明示的）           | なし（直接ドラッグ）   | なし（設定パネルは別） | **あり（明示的）を推奨**  |
| 誤操作保護               | 高（モード分離）         | 低（ロックなし）       | 高（ロック機能あり）   | 編集モード + Undo         |
| 削除確認                 | 2ステップ                | 1ダイアログ            | 1ダイアログ            | 2ステップ + Undo          |
| Undo                     | **なし**                 | **なし**               | **なし**               | **実装して差別化**        |
| グリッド密度（モバイル） | 4×6 = 24                 | 5×5 = 25（デフォルト） | 4×6 または 5×6         | 3〜4列×5〜6行（設計依存） |
| 大画面で密度増           | **しない（哲学的決定）** | する（カスタマイズ可） | する                   | レスポンシブで自然増      |
| 儀式感                   | あり（低頻度ユーザー）   | なし（常時リスクあり） | なし（ロックで補完）   | 入り口1クリックで最小化   |

**最終的な設計推奨：**

1. **編集モードを採用する**（iOS ジグルモード型。ただし「揺れ」は使わず、ドラッグハンドル表示と背景変化で代替）
2. **Undo を実装する**（iOS/Android いずれも未実装。道具箱の差別化点になる）
3. **削除は編集モード中のみ表示 + 2ステップ確認**（誤操作防止）
4. **w360 では 3〜4列**（タッチターゲット 48dp 確保を最優先）
5. **大画面ではレスポンシブで列数増**（w768 で 4〜5列、w1280 で 5〜6列）
6. **「レイアウトロック」オプションは将来フェーズ**（Samsung One UI が持つ機能。M1b に有効だが初期リリースには不要）
