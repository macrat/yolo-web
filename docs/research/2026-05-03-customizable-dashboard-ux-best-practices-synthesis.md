# カスタマイズ可能なダッシュボード/ホーム画面 UX ベストプラクティス総合調査

## 本レポートの位置づけ

本レポートは cycle-177 planner 向けに、道具箱（タイル型カスタマイズダッシュボード）の UI/UX 設計判断材料を整理したものである。過去の调査レポート 5 本（2026-05-01〜2026-05-03）を基盤とし、アクセシビリティ・視覚フィードバック手法・ドラッグ操作パターンの体系的整理という不足していた観点を補完する。

**本レポートの基本姿勢**: 特定のスタイルを擁護・否定せず、選択肢の特性と trade-off を列挙する。yolos.net の特殊事情（個人サイト・落ち着いたトーン・道具コンセプト）を考慮した検討材料は提示するが、「こうすべき」という断定は行わない。

### 1-2. 評価軸ごとの比較

以下の評価軸で上記パターンを比較する。「評価」は文脈依存で変わるため高低でなく特徴を記述する。

**発見性（Discoverability）**: 編集モードの存在や入り方に気づきやすいか

- 常時編集可能型（Chrome、Firefox）: 高い。どこからでも操作できる。
- セミシームレス型（Start.me）: 中程度。ドラッグを試みれば分かるが、試みなければ分からない。
- 明示的ボタン型（Notion、Home Assistant）: 低い。ボタンを探さないと分からない。ツールチップや初回ガイドが必要。
- 長押し型（iOS、Android）: 発見コストは中程度。長押しは試行しないと分からないが、OS全体で統一されているため学習コストは低い。

**学習コスト**: 操作を習得するのにどれくらい時間がかかるか

- 常時編集可能型: ドラッグという操作は直感的だが、「スクロールとドラッグの混同」の罠がある。
- 長押し型（iOS）: 長押しという1つの入口を覚えれば十分。OS全体で一貫性がある。
- 明示的ボタン型: ボタンの位置を覚える必要があるが、操作は単純。初回後は素直に使える。
- セミシームレス型: 自動遷移のため「今どちらのモードにいるか」が分かりにくいことがある。

**操作精度**: 意図した操作を正確に実行できるか（誤操作率に逆相関）

- 常時編集可能型: 低い。スクロールとドラッグを誤ることが多い（iGoogle、Netvibes の報告）。
- 明示的ボタン型: 高い。View モードでは操作できないため誤操作が発生しない。
- 長押し型（iOS）: 中程度。長押し開始後のドラッグはやや難しい（特に意図しない位置へのドロップ）。
- Android 直接ドラッグ: 低い。誤ってドラッグが始まることがある。レイアウトロックがないとリスクが高い。

**誤操作率**: 意図しないレイアウト変更が発生しやすいか

| パターン                     | 誤操作リスク | 根拠                                                      |
| ---------------------------- | ------------ | --------------------------------------------------------- |
| 常時編集可能（シームレス）   | 高           | スクロール/ドラッグ混同。iGoogle・Netvibes で繰り返し報告 |
| セミシームレス（Start.me型） | 中           | 意図せずドラッグで編集モード入りする報告あり              |
| 長押し起動型（iOS型）        | 低〜中       | 長押し要件で誤発動は少ないが、発動後のドラッグミスはある  |
| 明示的ボタン型（Notion型）   | 低           | View モードで変更不可なため誤操作は構造的に発生しない     |

**アクセシビリティ**: キーボード操作・スクリーンリーダー対応・運動機能障害への配慮

- 常時編集可能型: キーボードのみで並び替える手段がないと問題。
- 明示的ボタン型: ボタン自体はキーボードでフォーカス可能。並び替えはキーボード代替が必要。
- 長押し型: タッチ前提のため代替操作の提供が必要（WCAG 2.5.7）。
- どのパターンでも: ドラッグの代替手段（ボタン/メニューによる移動）の提供が WCAG 2.5.7 AA 要件。

**モバイル操作性**: タッチ操作での快適さ

- 長押し型: モバイル向けに最適化されている。ただしページ間ドラッグが難しい。
- 常時編集可能型: スクロールとの競合が問題。
- 明示的ボタン型: ボタンがタッチターゲットを満たせば問題なし。ドラッグはモバイルで難しい場合あり。

## 2. 編集モードの視覚フィードバック手法

### 2-1. 手法カタログ

視覚フィードバック手法を体系的に列挙し、利点・欠点・適切な利用シーンを整理する。

#### A. 揺れ（ジグル/ウィグル）アニメーション

**説明**: 編集モード入り時に全タイル/アイコンが一定周期で揺れ続けるアニメーション。iOS ホーム画面が代表例。

**利点**:

- 「このアイコンは今つかんで動かせる」というアフォーダンスを強力に視覚伝達できる（アニメーションが静的より注目を集める）。
- 全アイコンが同時に揺れることで「全体が編集可能状態になった」というモード変化を一目で伝える。
- iOS ユーザーには学習済みのパターンであり、OS 外でも「揺れている = 動かせる」という連想が生まれやすい。

**欠点**:

- 継続的なモーション（揺れが続く）は前庭系感覚過敏（vestibular disorders）のユーザーに不快感や目眩を引き起こす可能性がある。`prefers-reduced-motion: reduce` への対応が必須。
- 低頻度ユーザーには「大袈裟」「賑やか」と感じる場合がある（9to5Mac の批評記事で言及）。
- Web 実装では CSS アニメーションの負荷が、特にタイル数が多い場合にパフォーマンスへ影響する可能性がある（最適化は可能だが考慮が必要）。
- 「落ち着いた・静かなデザイン」のコンセプトとは方向性の整合を要する。

**適切な利用シーン**:

- 派手な編集体験を価値として提供したい場合（ゲーム的・エンタメ的なサービス）。
- iOS ユーザーが多いサービスでモバイル操作のメタファを利用したい場合。
- タイル数が少なく（〜10枚程度）アニメーション負荷が小さい場合。

**prefers-reduced-motion 対応例**: 揺れアニメを完全に止め、代わりにタイルに細いボーダーやシャドウ変化のみを適用する。

---

#### B. ドラッグハンドルの出現

**説明**: 編集モード入り時にのみ各タイルの特定位置（通常は隅または中央）にドラッグハンドルアイコン（⠿ 6点ドット等）が出現する。ドラッグはハンドルを掴んで行う。Notion のブロック並び替えが代表例。

**利点**:

- ハンドルを明示することで「どこを掴めばドラッグできるか」が明確になる。
- 削除ボタン等と組み合わせて「編集中にのみ操作が見える」設計にしやすい。
- `prefers-reduced-motion` が有効でも機能に影響がない（ハンドル表示はアニメーションではない）。
- アクセシビリティ上も、ハンドルを `<button>` として実装し、キーボード操作（スペース/Enter で掴む → 矢印キーで移動）を提供できる。

**欠点**:

- ハンドルが小さすぎると（24px × 24px 未満）WCAG 2.5.8 Target Size 違反になる。
- デスクトップではハンドルのホバーで気づくが、モバイルではハンドルの存在自体に気づかないユーザーがいる。
- ハンドルが常に見えていると（非編集モードでも）レイアウトが煩雑になる。
- ハンドルのみの変化では「今が編集モードであること」の認識に他の手がかりも必要。

**適切な利用シーン**:

- デスクトップ中心のインターフェース。
- 「落ち着いた・シンプル」なデザインで揺れアニメを避けたい場合。
- コンテンツ密度が高くアニメーションが干渉しやすい場合。

---

#### C. 背景/オーバーレイ変化

**説明**: 編集モード入り時にページ全体または各タイルの背景色・不透明度が変化する。薄いオーバーレイを全体にかけることで「今は特別なモードにいる」を伝える。

**利点**:

- アニメーションを使わずにモード状態を明確に伝えられる。
- 揺れアニメと組み合わせることも、単独でも使える。
- 「通常モードとの違い」を色で伝えるため、低頻度ユーザーでも「あ、変わった」と気づきやすい。
- `prefers-reduced-motion` の影響を受けない。

**欠点**:

- コントラスト比要件（WCAG 1.4.3: 4.5:1 または 1.4.11: 3:1）を満たした上でのモード表現が必要。
- 暗いオーバーレイはタイル内のコンテンツを見づらくする可能性がある。
- モバイルで画面全体を暗くすると「消えた」と誤解されることがある。

**適切な利用シーン**:

- 「落ち着いた・静かな」コンセプトのサービス。
- 揺れアニメを使いたくない場合のモード表現の主役として。
- ハンドル表示や削除ボタン出現と組み合わせると効果的。

---

#### D. タイルへのボーダー/アウトライン追加

**説明**: 編集モード中にタイルにダッシュ枠線・実線ボーダー・グロー効果等を適用し、「編集可能状態」を示す。

**利点**:

- 非常に軽量でパフォーマンス負荷が低い。
- 「触れる場所」の境界を明示する点でアクセシビリティに貢献する。
- 揺れアニメ・ドラッグハンドルとの組み合わせで多層的な状態表現が可能。

**欠点**:

- ボーダーのみでは「ドラッグできる」というアフォーダンスが弱い。
- ダッシュ（破線）枠線は「このタイルはドラッグで移動できる空きスロットに配置される」という「ドロップ先」としての視覚表現と混同される場合がある。
- タイル数が多いと視覚的に騒がしくなる。

**適切な利用シーン**:

- ドラッグハンドルと組み合わせて使う補助的な手がかりとして。
- ドロップ先候補の視覚表現として（実線ではなくダッシュで）。

---

#### E. 削除ボタンの出現（バッジ/オーバーレイ）

**説明**: 編集モード中にのみ各タイルの隅に「×」または「−」ボタンが出現する。iOS の削除バッジが代表例。

**利点**:

- 「削除操作は今だけできる」ことが視覚的に明確。誤削除リスクを構造的に抑える。
- 出現すること自体が「今は編集モードだ」という状態の手がかりにもなる。

**欠点**:

- タッチターゲットが小さすぎると誤タップリスクがある（24px × 24px 以上の確保が必要、理想は 44px × 44px）。
- ボタンが各タイルに出現するためレイアウトが複雑に見える場合がある。

**適切な利用シーン**:

- 明示的編集モードと組み合わせる場合（常時表示だと削除への心理的障壁が下がりすぎる）。
- ドラッグハンドルと組み合わせて「編集モードのUIセット」として一体的に表示。

---

#### F. ハプティックフィードバック（バイブレーション）

**説明**: 編集モード入り時・ドラッグ開始時・ドロップ時に端末がバイブレーションを発生させる。モバイルネイティブアプリで一般的だが、Web では `navigator.vibrate()` API を通じて利用可能（ブラウザサポートに差異あり）。

**利点**:

- 視覚的なフィードバックを補完し、「何かが起きた」という確認を触覚で得られる。
- 視覚障害のあるユーザーへの補助的な情報提供になる。

**欠点**:

- Web の `navigator.vibrate()` は iOS Safari で未サポート（2026年時点）。
- 触覚過敏のあるユーザーには不快。OS レベルの「バイブレーションオフ」設定を尊重する設計が必要。
- 感度が端末によって大きく異なり、意図した効果が出ないケースがある。

**適切な利用シーン**:

- ドラッグ操作の開始確認（「掴めた」感覚）として補助的に使う。
- iOS の場合は Native アプリのみで実用的。Web アプリとしては視覚フィードバックの補助にとどめる。

---

#### G. サウンドフィードバック

**説明**: ドラッグ開始・ドロップ完了時にサウンドを再生する。

**利点**:

- 操作完了の確認を聴覚でも得られる。
- ゲーム的・エンタメ的なサービスでは没入感向上に寄与する。

**欠点**:

- 「日常の傍にある道具」のような落ち着いたコンセプトとは通常合わない。
- 公共の場・会議中・夜中の使用でユーザーが驚く。「音をオフにできる」設定が必須。
- スクリーンリーダーユーザーへの干渉リスクがある。

**適切な利用シーン**:

- 子供向けインターフェース・ゲーム要素のある UI に限定される。

### 2-2. 複数フィードバックの組み合わせ例

代表的な組み合わせパターンを整理する。いずれも組み合わせの「良し悪し」ではなく「特性」として整理する。

**パターン1（iOS型）**: 揺れアニメ ＋ 削除バッジ ＋ 「完了」ボタン

- 特性: 3つの異なる視覚手がかりが同時に機能し、モード状態の認識が高い。一方でアニメーションが継続するため動的。

**パターン2（ドラッグハンドル型）**: ドラッグハンドル出現 ＋ 削除ボタン出現 ＋ 背景変化（薄い）＋ 「完了」ボタン

- 特性: アニメーションを使わず静的な視覚変化のみ。落ち着いた印象。揺れアニメの代替として機能する。prefers-reduced-motion への追加対応が不要（元からアニメなし）。

**パターン3（シンプル型）**: タイルにボーダー追加 ＋ 「完了」ボタン

- 特性: 最小限の変化。シンプルだが「なぜボーダーがある？」とユーザーが気づかない可能性がある。補助的な説明テキストが必要。

**パターン4（Notion型）**: ドラッグハンドル出現のみ（モード遷移の背景変化なし）

- 特性: View モードと Edit モードの視覚的差異が最小限。Notion はデスクトップが主体で慣れたユーザー向けのため成立するが、一般向けダッシュボードでは「モード認識の失敗（モードスリップ）」リスクがある。

**NN/g の知見（公式記事 modes/）**: モードを使う場合は「少なくとも2つの冗長な視覚的指標」を同時に使うことが必須。1つだけでは見落とすリスクが高い。

## 3. ドラッグ&ドロップ並び替えの実装パターン

### 3-1. グラブ（掴み方）パターン

| パターン                     | 説明                                                           | 利点                                                         | 欠点                                                             | 主な採用事例                        |
| ---------------------------- | -------------------------------------------------------------- | ------------------------------------------------------------ | ---------------------------------------------------------------- | ----------------------------------- |
| **長押しグラブ**             | 要素を一定時間（400〜600ms）押し続けることでドラッグ開始       | モバイルでタップとドラッグを明確に区別できる                 | 開始まで待ち時間がある。フラストレーションを感じるユーザーがいる | iOS ジグルモード内ドラッグ、Android |
| **即時グラブ（Click Drag）** | マウスボタン押下 or タッチ開始で即ドラッグ                     | 素早い操作が可能                                             | スクロールとの競合が発生しやすい（特にモバイル）                 | Trello（デスクトップ）、iGoogle     |
| **ハンドル付きグラブ**       | 専用ドラッグハンドル要素のみからドラッグ開始可能               | ドラッグ可能な場所が明確。タイル内のクリック操作と競合しない | ハンドルが小さいと操作が難しい（モバイル特に注意）               | Notion、Home Assistant Sections     |
| **モーダル前のキャプチャー** | 「グラブして動かす」前にモードを明示的に切り替えてからドラッグ | 誤操作ゼロ。View モードではドラッグ自体を無効化              | 操作ステップが1段多い                                            | Notion（View/Edit分離）             |

### 3-2. dnd-kit での一般的なパターン

`@dnd-kit/core` + `@dnd-kit/sortable` の組み合わせで一般的に使われるパターン。

**Sensor 設定（デフォルト）**: Mouse Sensor（デスクトップ）+ Touch Sensor（モバイル）+ Keyboard Sensor の3つが標準で有効。

**ハンドル実装パターン**:

```
// useSortable の listeners（ドラッグイベント）を <button> ハンドルに割り当てる
// ref（ソータブル対象）はタイル全体に割り当てる
const { setNodeRef, attributes, listeners, transform } = useSortable({ id });
// <li ref={setNodeRef} ...>
//   <button {...listeners}>⠿</button>  ← ハンドルのみからドラッグ開始
//   {children}  ← クリックで通常操作
// </li>
```

**キーボード対応（sortableKeyboardCoordinates）**:
`@dnd-kit/sortable` 提供の `sortableKeyboardCoordinates` を Keyboard Sensor の `coordinateGetter` に設定すると、矢印キーで最近傍のソータブル要素に移動する動作が実現できる。デフォルトでは 25px 移動だが、sortable では「次の要素」に確実に移動するように振る舞いが上書きされる。

**スクリーンリーダー対応**:

- `DndContext` は各ソータブル要素に `aria-roledescription="draggable"` を付与する。
- オフスクリーンの `<div aria-live="assertive">` が「アイテム1を掴みました。矢印キーで移動し、スペースまたはエンターでドロップします」という手順を読み上げる。
- ドラッグ中は「アイテムが位置3へ移動しました」のようにライブリージョンで状態を通知する。
- すべてのテキストは `announcements` プロップとスクリーンリーダー指示文字列で日本語化できる。

### 3-3. WCAG 2.5.7 Dragging Movements との整合

**要件（Level AA）**: すべてのドラッグ操作に対し、ドラッグを使わない単一ポインター代替手段を提供しなければならない（例外: ドラッグ自体が本質的な操作の場合）。

**ソータブルリストへの適用**: 並び替えにドラッグのみしか手段がない場合は違反。以下の代替が必要：

- 各タイルに「上へ移動」「下へ移動」ボタン（矢印アイコン等）を編集モード中に表示する。
- または、タイルをタップ/クリックすると「移動先を選ぶ」ポップアップが出る。

**dnd-kit のキーボード Sensor との関係**: キーボード操作はドラッグ代替として機能するが、「ポインターのみ」のユーザー（マウスは使えるがドラッグはできない）には対応しない。WCAG 2.5.7 は「ポインター（マウス/タッチ）の代替手段」を要求しているため、キーボードとは別に、クリック操作でも並び替えができる手段が必要。

実装的に最も簡単な対応: 編集モード中にタイルの右側に「↑」「↓」ボタンを表示し、タップ1回で前後の要素と入れ替える。

### 3-4. スクリーンリーダー対応の追加注意事項

- `role="listbox"` + `aria-selected` より、`role="list"` + `aria-grabbed` のパターンが drag & drop に適している。
- ライブリージョンの `aria-live` は `"polite"` を基本とし、ドラッグ完了通知には `"assertive"` を使うとスクリーンリーダーが即座に読み上げる。ただし `"assertive"` の多用は他の読み上げを中断するため注意。
- 状態変化（編集モード ON/OFF）は `aria-expanded`、`aria-pressed`、`aria-current` のいずれかを状況に応じて使う。「編集モードボタン」であれば `aria-pressed="true"/"false"` が適切。

| **2.1.2 キーボードトラップなし**      | A      | 編集モード中の UI            | フォーカスが罠にはまらないこと                        |

### 4-2. prefers-reduced-motion の扱い方

`prefers-reduced-motion: reduce` がONのとき、モーションに依存したフィードバックを代替手段に置き換える。

**揺れアニメーション（ジグル）の代替策**:

- アニメーションを完全に停止し、タイルにボーダーやシャドウ変化のみを適用する。
- または、揺れを「瞬間的なスケール変化（一度だけ 1.0 → 1.05 → 1.0）」に置き換えて、繰り返しのモーションをなくす。

CSS 実装の基本パターン:

```css
@keyframes wiggle {
  0%,
  100% {
    transform: rotate(-1deg);
  }
  50% {
    transform: rotate(1deg);
  }
}
.edit-mode .tile {
  animation: wiggle 0.3s ease-in-out infinite;
}
@media (prefers-reduced-motion: reduce) {
  .edit-mode .tile {
    animation: none;
    outline: 2px dashed currentColor; /* 代替: アウトラインで状態を伝える */
  }
}
```

**ドラッグ中のアニメーション**: ドラッグ中の「他要素がシフトする」アニメーション（dnd-kit のデフォルト）も、`prefers-reduced-motion` 有効時は瞬間的な位置変化に切り替える。dnd-kit の `transition` プロップで制御可能。

### 4-3. Edit モードと Done モードの状態通知

スクリーンリーダーに対して現在のモード状態を正確に伝えるパターン。

**推奨パターン**:

```html
<!-- 編集モードボタン -->
<button aria-pressed="false" aria-label="道具箱を編集する">編集</button>

<!-- 編集モード入り後 -->
<button aria-pressed="true" aria-label="道具箱の編集を終了する">完了</button>

<!-- モード変化の通知（aria-live で即座に読み上げ） -->
<div aria-live="polite" class="sr-only">
  編集モードになりました。矢印キーで並び替えができます。
</div>
```

**aria-label の言語**: 日本語ユーザー向けには日本語で記述する。「Draggable item」ではなく「移動可能なツール」のように説明すると自然。

### 4-4. タッチターゲットの確保

- WCAG 2.5.8（Level AA）: 24×24 CSS px 以上
- Apple HIG 推奨: 44×44 pt（論理ピクセル）以上
- Material Design 推奨: 48×48 dp 以上

実装上、視覚サイズを小さく保ちつつタッチターゲットを大きくするには、`padding` または `::after` 擬似要素でタップ領域を拡張する方法が一般的。

削除ボタン（×）は特に誤タップリスクがあるため、44px 以上のタッチ領域を推奨する。

## 5. 「日常の傍にある道具」コンセプト向けの選択肢整理

### 5-1. 前提条件の整理

yolos.net の道具箱に固有の制約・コンテキスト:

- **反復使用 UI**: 毎日開くホームページ的使い方（M1b ターゲット）
- **落ち着いたトーン**: サイトコンセプトが「日常の傍にある道具」（派手・賑やかは合いにくい）
- **小規模ツール群**: 電卓・タイマー等の小さな道具が主体（ダッシュボードとして見た時に情報量が少ない）
- **モバイルも重要**: w360 での快適な操作が必要
- **localStorage 永続化**: クラウド同期なし（誤操作で消えたデータは復元困難）
- **個人サイト**: チーム・企業向けの複雑な編集 UI は不要

### 5-2. 案 A: 明示的2モード分離型（シンプルハンドル方式）

**概要**:

- 「編集」ボタン（右上固定）をタップ → Edit モード入り
- Edit モード中: 各タイルにドラッグハンドル（⠿）と削除ボタン（×）が出現、背景が薄く変化
- 揺れアニメーションなし
- 並び替え: ハンドルを掴んでドラッグ ＋ 「↑」「↓」ボタン（WCAG 2.5.7 対応）
- 削除: 5秒間 Undo バナー方式（確認ダイアログなし）
- 「完了」ボタンで View モードに戻る

**利点**:

- 落ち着いたトーンと整合しやすい（揺れアニメなし）
- prefers-reduced-motion への追加対応不要（元からアニメなし）
- 誤操作保護が高い（View モードでは変更不可）
- Undo バナーにより削除ミスからのリカバリーが容易
- WCAG 2.5.7 対応が「↑↓ボタン」で実現できる

**欠点**:

- 「編集ボタンの存在に気づかない」発見性の問題（初回オンボーディングが重要）
- View モードと Edit モードの視覚差異を揺れ以外で明確に作る必要がある
- 実装に「ハンドル実装」「モード状態管理」「Undo バナー」「WCAG 用の↑↓ボタン」が必要でやや複雑

**適合評価**: yolos.net の「落ち着いた・誤操作を恐れずに使える・モバイルでも快適」の3要件すべてに整合しやすい。

---

### 案 B: 明示的2モード分離型（揺れアニメ方式）

**概要**:

- 「編集」ボタン → Edit モード入り
- Edit モード中: タイルが揺れ（ジグル）+ 削除ボタン出現
- prefers-reduced-motion: reduce 時は揺れを止め、ハンドル表示に切り替え
- 削除・Undo・「完了」ボタン等は案 A と同様

**利点**:

- 「編集モードに入った」ことの認識が高い（揺れが注意を引く）
- iOS ユーザーには既知のメタファ（学習コストが低い層に対して）
- 案 A のハンドル実装より初期実装がシンプルな場合がある

**欠点**:

- 継続するモーションが前庭系感覚過敏のユーザーに影響する可能性
- 「日常の傍にある道具」の落ち着いたトーンと揺れアニメの動的な印象の整合を設計段階で要確認
- prefers-reduced-motion 対応で「揺れなし版」の別実装が必要になる

**適合評価**: 機能面では案 A と同等。デザイン上のトーン整合と prefers-reduced-motion 対応コストが追加される。「揺れが悪い」ではなく、コンセプトとの整合を設計判断として検討が必要。

---

### 案 C: セミシームレス型（長押し自動入り）

**概要**:

- 通常モード（View）: タイルをタップで起動。ドラッグは無効。
- 長押し（400ms）でそのタイルのドラッグが始まる（iOS 的な感覚）
- ドラッグ開始と同時に他タイルにも削除ボタンが出現（編集状態の視覚フィードバック）
- ドロップで配置確定。削除ボタンは一定時間後または「×」以外タップで消える
- 専用の「完了」ボタンはない

**利点**:

- 「編集ボタンを探す」ステップがない。発見性が高い。
- モバイルのタッチ操作と自然に合う（長押しはモバイルで一般的なジェスチャー）
- iOS ユーザーには「ホーム画面と同じ感覚」

**欠点**:

- 「意図せず長押しで編集モード入り」が発生する可能性（誤操作リスク）
- 「タイルをタップ」（起動）と「タイルを長押し」（編集）の区別を学習する必要がある
- 編集状態の終了条件が「自動タイムアウト」になると不明確になる
- Start.me のセミシームレス型では「意図せずレイアウトが変わった」報告が存在する
- Undo なしの場合、誤操作からのリカバリー手段がない

**適合評価**: モバイルファーストで発見性を重視する場合の選択肢。ただし「誤操作を恐れずに使える」の要件と一部競合する。Undo の実装で緩和できる。

---

### 案 D: 常時編集可能型（シームレス）

**概要**:

- 専用の編集モードなし。常時ドラッグ可能。
- タイルのホバー/タップで操作メニューが表示される（削除等）

**利点**:

- 実装がシンプル
- 編集モードという概念がない分、発見性の問題がない

**欠点**:

- スクロールとドラッグの競合リスク（モバイル）
- 「毎日開く場所で誤ってレイアウトが変わってしまう」リスクが M1b の安心感を損なう
- iGoogle・Netvibes・Protopage で繰り返し「意図せず動かしてしまった」報告が確認されている
- NN/g: 「反復使用 UI では要素の位置安定性が最重要」（空間記憶研究）

**適合評価**: 「誤操作を恐れずに使える・毎日同じ場所に同じ道具がある安定性」を重視する M1b ターゲットとは整合が難しい。

---

### 5-4. 組み合わせパターン

実際の実装では上記案の要素を組み合わせることも有効。代表的な組み合わせ案：

**組み合わせ X（案 A ベース ＋ 案 C の要素）**:

- 編集ボタンで明示的に Edit モード入り（案 A）
- Edit モード中に長押し開始で編集（発見性は低いが、ボタンを主経路にする）
- ハンドル＋削除ボタン＋背景薄変化で状態を伝える
- 初回ツールチップで「右上ボタンで並び替えができます」を案内

**組み合わせ Y（案 B ＋ prefers-reduced-motion 分岐）**:

- Edit モードボタン → 揺れアニメ ＋ 削除ボタン
- prefers-reduced-motion: reduce の場合はハンドル表示に自動切り替え
- 揺れの好みが分かれる場合でも両方に対応できる

## 7. planner 向け設計判断チェックリスト

以下は本レポートの内容から、planner が設計判断時に確認すべき事項の一覧である。

**編集モードの選択**:

- [ ] 明示的2モード分離型とシームレス型のどちらを採用するか
- [ ] 採用したモードで「誤操作リスク」と「発見性」のバランスが取れているか
- [ ] 初回オンボーディング（編集モードの存在を伝える手段）を設計しているか

**視覚フィードバックの選択**:

- [ ] 揺れアニメを使う場合、`prefers-reduced-motion` 時の代替実装が計画されているか
- [ ] 揺れアニメを使わない場合、NN/g が要求する「2つ以上の冗長な視覚的指標」を満たしているか
- [ ] 選択した視覚フィードバックが「日常の傍にある道具」の落ち着いたトーンと整合するか

**アクセシビリティ**:

- [ ] WCAG 2.5.7: ドラッグ代替手段（↑↓ボタン等）を提供しているか
- [ ] WCAG 2.5.8: 削除ボタン・ハンドルが 24×24px 以上（理想 44×44px）か
- [ ] スクリーンリーダー向け ARIA: `aria-live` でモード変化と操作結果を通知しているか
- [ ] `aria-pressed` または `aria-expanded` でEdit/Done の状態が伝わるか

**削除の安全策**:

- [ ] Undo バナー方式（5秒間）と確認ダイアログ方式のどちらを採用するか
- [ ] localStorage への書き込みタイミング（Undo を考慮した場合は遅延書き込みが必要）

**モバイル対応**:

- [ ] w360 でのタッチターゲット（44×44px 以上）が確保されているか
- [ ] ドラッグハンドルがモバイルで視認・操作しやすいサイズか
