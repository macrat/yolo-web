## 2. 各競合サイトの詳細

### A. regex101（https://regex101.com/）

**UI レイアウト**
ページを横 2 分割（左 60% / 右 40%）。左カラムに「Regular Expression」入力欄（1 行）とデリミタ `/…/gm` をインラインで表示し、その下に「Test String」テキストエリア（全幅・高さ可変）。右カラム上部に「Match Information」（マッチ詳細・グループ展開表示）、中段に「Explanation」（パターンの各トークンを自然言語で説明）、下段に「Quick Reference」。左サイドパネル（ハンバーガーで折り畳み）に Regex Debugger / Code Generator / Benchmark / Format Regex などの高度機能へのリンク。

**フラグ切替 UI**
パターン入力欄の右側にある `/ gm` ボタンをクリックするとフラグ選択ダイアログがポップアップ。デフォルト gm ON。対応フラグは g / m / i / s / u / y / d（7 種類）。ラベルは英語のみ（日本語訳なし）。

**マッチ結果ハイライト**
テキストエリア内のマッチ箇所を背景色（薄い黄色 / オレンジ）で強調。複数のキャプチャグループを異なる色相で塗り分け。色のみの識別（記号/番号の直接表示はインラインにはない）。右カラムの Match Information でグループ番号・マッチ文字列・インデックス範囲をリスト形式で表示。

**機能一覧**

- リアルタイムマッチング（デバウンスあり）
- マッチ詳細（マッチ数・位置・キャプチャグループ）をリスト表示
- Explanation パネル（各トークンを自然言語で説明）
- 置換（Substitution タブ）
- コードジェネレーター（JavaScript/Python/PHP 等のコードスニペット生成）
- Regex Debugger（マッチングの進行を可視化）
- Benchmark（パフォーマンス計測）
- フラグ切替（g/i/m/s/u/y/d）
- パターン保存・シェア URL 生成
- フレーバー選択（ECMAScript / PCRE2 / Python / Java / Go など 10 種以上）

**a11y**
カラーのみ判別（記号は右パネルにのみある）。テキストエリアと入力欄に aria-label は確認できず。カラーコントラストは暗色テーマ基準では概ね良好だが、ライトテーマでのハイライト色は淡い黄色であり WCAG AA は不明。

**リアルタイム vs ボタン**
リアルタイム処理（入力のたびに自動更新）。

**モバイル対応**
最低限のレスポンシブ。2カラムレイアウトはスマートフォンでは縦並びになるが、使用感はデスクトップ向けに最適化されている。

**広告・登録**
広告なし。アカウント登録（無料）でパターン保存が可能。登録なしでも全機能利用可。

### B. WEB ARCH LABO（https://weblabo.oscasierra.net/tools/regex/）

**UI レイアウト**
縦 1 カラム。ページ上部に正規表現メタ文字の早見表（テーブル）を 2 つ掲載し、その下にツール本体（パターン入力 1 行 / テキストエリア / 結果欄）を縦並びで配置。フラグ切替 UI はなし。

**フラグ切替 UI**
なし。フラグ選択不可。

**マッチ結果ハイライト**
不明（結果欄は空のテキストエリアで、入力してもハイライトは表示されない可能性が高い。スナップショット取得時は未入力状態）。

**機能一覧**

- パターン入力 / テストテキスト入力 / 結果表示のみ
- 置換なし
- フラグ切替なし

**a11y**
入力欄に placeholder 記載あり。aria-label の確認はできず。

**リアルタイム vs ボタン**
スナップショットではボタンが確認できず、リアルタイムの可能性がある。

**モバイル対応**
HTML が古典的で、基本的なレスポンシブ。

**広告**
Google AdSense 広告あり（Adsense の差し込みが確認されたが、ポップアップ型ではない）。

### C. アサリツールズ（https://tools.studioasari.co.jp/web-dev/regextester）

**UI レイアウト**
縦 1 カラム。STEP 1（パターン入力） → STEP 2（テストテキスト入力） → STEP 3（置換文字列、任意） → STEP 4（実行ボタン）の 4 ステップ構成。各 STEP が `<h2>` で見出し化されている。フラグはチェックボックス（縦並び）でパターン入力欄の下に配置。

**フラグ切替 UI**
チェックボックス 3 個：

- `g（global - 全体マッチ）`
- `i（ignore case - 大文字小文字無視）`
- `m（multiline - 複数行モード）`
  デフォルト g ON（推測。スナップショット時は確認できず）。u/s/y/d フラグには非対応。

**マッチ結果ハイライト**
マッチ実行ボタン押下後に結果表示。ハイライト方式は「視覚的強調表示」と自説明に記載があるが、具体的な CSS は不明（背景色のみと推測）。キャプチャグループ表示はリスト形式。

**機能一覧**

- ボタン押下式マッチング
- 置換プレビュー（$1, $2 参照）
- マッチ数・位置・キャプチャグループ表示
- エラー表示
- ブラウザ内完結（サーバー送信なしと明記）

**a11y**
STEP 構成で操作の流れが明確。各入力欄に label 要素あり。aria-label の確認はできず。

**リアルタイム vs ボタン**
ボタン押下式。

**モバイル対応**
レスポンシブ対応あり（自説明に記載）。

**広告**
Google AdSense 広告あり（サイドバー・コンテンツ間に複数）。登録不要。

### D. instant tools JavaScript正規表現テスター（https://tools.m-bsys.com/dev_tools/regexp.php）

**UI レイアウト**
縦 1 カラム。対象文字列テキストエリアが先頭にあり、その下にフラグ + パターン入力 + match ボタン + 置換入力 + replace ボタンをテーブルで横に並べた独自配置。サンプルをドロップダウンで選択できる機能が特徴的。

**フラグ切替 UI**
チェックボックス + テキスト：

- `i`（デフォルト OFF）
- `g`（デフォルト ON）
- `m`（デフォルト OFF）
- 「リアルタイムに処理」（チェックで自動更新）

**マッチ結果ハイライト**
「検索・置換結果」エリアにハイライト表示。具体的な表示方式はスナップショット未入力時には不明（赤色背景などの可能性）。

**機能一覧**

- リアルタイム / ボタン押下の切替
- 置換（replace）
- サンプルドロップダウン（「年月日」「メールアドレス」「半角記号を探す」「JS文字列リテラル取得（後方参照）」「HTMLタグ検索（最短一致）」「数値をカンマ区切りにする」など 20 種類）
- フラグ 3 種（g/i/m）

**a11y**
テーブルレイアウトで ARIA は不十分。ID のない checkbox。

**リアルタイム vs ボタン**
チェックボックスで切替可能。デフォルトはリアルタイム。

**モバイル対応**
テーブルレイアウトのため、スマートフォンでは横スクロールが発生する可能性が高い。

**広告**
Google AdSense 広告あり（複数、コンテンツ内に差し込み）。登録不要。

### E. Site24x7 正規表現テスター（https://www.site24x7.com/ja/tools/regex-parser.html）

**UI レイアウト**
2 カラム（左：パターン入力・フラグ・テキスト入力・実行ボタン / 右：マッチした情報・マッチした文字列）。言語切替タブ（JavaScript / PHP / Python）がパネル上部に配置。全体的にエンタープライズ向けツールサイトの一部として位置付けられており、大量のフッターリンクが存在する。

**フラグ切替 UI**
チェックボックス 3 個（ラベルは英日混在）：

- `Global(g)`
- `Case Insensitive(i)`
- `multiline(m)`
  デフォルト g ON。

**マッチ結果ハイライト**
テキストハイライトなし。マッチ結果を「FOUND "xxx" MATCHED AT 0-1」形式のテキストで右パネルに列挙。色は使用していない（テキストのみ）。

**機能一覧**

- ボタン押下式マッチング（「マッチ」ボタン）
- JavaScript / PHP / Python 切替
- マッチ位置情報のテキスト表示
- キャプチャグループ表示（MATCH 1 GROUP 1: "xxx"形式）
- 置換なし

**a11y**
テキストのみ（色なし）のマッチ結果表示は、色覚多様性ユーザーにも読みやすい。ただし aria-label はチェックボックスに「Enter your Input field」という汎用テキストが使われており不十分。UI 翻訳は機械翻訳品質。

**リアルタイム vs ボタン**
ボタン押下式。

**モバイル対応**
レスポンシブ。

**広告**
Zoho/ManageEngine 製品へのサインアップ促進バナーあり（広告というよりサービス誘導）。サインアップ不要。

**マッチ結果ハイライト**

### G. お便利ツール.com 正規表現チェッカー（https://www.oh-benri-tools.com/tools/programming/regexp）

**UI レイアウト**
縦 1 カラム。パターン入力欄（テキストボックス）とフラグチェックボックス（i/g/m）が上段。その下に「メタ文字インサートボタン群」（`\d`・`\w`・`\s` 等のボタンをクリックするとパターン欄に挿入）。チェック文字列エリアは中段（テキストエリアではなく、動的に追加できる入力行の集合体で、各行に「Edit/Remove」リンク）。最下段に正規表現リテラル表示欄（自動生成されたコピー用）。

**フラグ切替 UI**
チェックボックス 3 個（ラベルは日本語付き）：

- `i: 大文字小文字を区別せずにマッチ`
- `g: すべての一致を検出`（デフォルト ON）
- `m: 複数行モード`

**マッチ結果ハイライト**
「マッチしている文字列がある場合は、入力した中段のテキストエリアの文字が赤く表示されます（即時反映）」と自説明。赤色背景/赤色テキストのみ（色のみ判別）。

**機能一覧**

- リアルタイムマッチング
- メタ文字インサートボタン群（ひらがな `[ぁ-ん]`・カタカナ `[ァ-ヴ]` も含む）
- フラグ 3 種（g/i/m）
- 正規表現リテラル自動生成 + コピー
- ブラウザ内完結（明記あり）

**a11y**
label 要素の for/id 対応が不十分（aria-label なし確認）。赤色のみ判別は WCAG 1.4.1 不適合。

**リアルタイム vs ボタン**
リアルタイム処理。

**モバイル対応**
レスポンシブ（viewport 設定あり）。

**広告**
Google AdSense 広告あり（複数）。登録不要。

## 7. 正規表現の説明（Explanation）機能

regex101 のみが本格的な「Explanation」パネルを実装しており、パターンの各トークンを自然言語で説明する（例: `[A-Z]` → "a single character in the range between A (index 65) and Z (index 90)"）。他のすべての競合にはこの機能がない。

yolos.net がこの機能を実装すれば、日本語圏での強力な差別化軸になる。ただし実装コストが高く、next.js のクライアントサイドで正規表現パーサーを実装する必要がある（既存の OSS: `regexp-tree`、`regexpu-core` 等が利用可能）。cycle-215 のスコープとは別に中長期での検討項目とする。

## 8. ユーザーが実際に求める典型的需要（M1b プログラマ・M1a 文書編集者向け）

「正規表現 よくある」「regex 用途」「validation 正規表現」等の検索から得られた上位ユースケースを 10 例挙げる。これらは placeholder や detail ページのサンプル事例として活用できる。

| # | ユースケース | サンプルパターン（JavaScript） | 出典 |
| --- | ------------------------------------ | ---------------------------------- | ----------------------------------------- | ------ | -------- | ---------- |
| 1 | メールアドレス（簡易）バリデーション | `^[\w.+-]+@[\w-]+\.[\w.-]+$` | 複数ソース |
| 2 | 日本の固定・携帯電話番号 | `^0\d{1,4}[-‒]?\d{1,4}[-‒]?\d{4}$` | 複数ソース |
| 3 | URL 抽出 | `https?://[\w/:%#$&?()\~.=+\-]+` | MDN / instant tools サンプル |
| 4 | 郵便番号（ハイフンあり/なし） | `^\d{3}-?\d{4}$` | 複数ソース |
| 5 | 日付（YYYY-MM-DD） | `^\d{4}-(0[1-9]                    | 1[0-2])-(0[1-9]                           | [12]\d | 3[01])$` | 複数ソース |
| 6 | 全角数字の検出 | `[０-９]+` | instant tools サンプル / Qiita |
| 7 | 半角カタカナの検出 | `[･-ﾟ]+` | Qiita |
| 8 | HTML タグ除去 | `<[^>]+>` | instant tools サンプル |
| 9 | 連続する空白を1つにまとめる | `\s{2,}` | 複数ソース |

これらのサンプルをタイル UI の placeholder や「よく使うサンプル」ドロップダウンに追加することで、空の状態からでも即座に試せる UX が実現できる。

## 9. 技術的制約と来訪者価値の整合性

### 9.1 ブラウザ `RegExp` の対応機能（2026年5月時点）

Chrome / Edge / Firefox / Safari 最新版で **全対応** している機能:

- フラグ: g / i / m / s / u / y / d（hasIndices）
- 名前付きキャプチャグループ: `(?<name>...)`
- lookbehind: `(?<=...)` / `(?<!...)`
- Unicode プロパティエスケープ: `\p{Letter}` / `\P{...}`（u フラグ必須）
- Unicode Sets: `v` フラグ（Chrome 112+, Firefox 116+, Safari 17+）

**v フラグ（unicodeSets）** は Safari 17+（2023年9月以降）から使用可能。2026 年 5 月時点では主要ブラウザの現行バージョンですべて対応済みだが、古いブラウザでは未対応のためエラー処理が必要。

yolos.net の現行実装は g/i/m/s のみ対応。u/y/d の追加は技術的に容易だが、UI 説明文の整備が必要。

### 9.2 危険な正規表現（ReDoS）への対応事例

ReDoS（Regular Expression Denial of Service）はカタストロフィックバックトラッキングにより、悪意あるパターン（例: `(a+)+` に長い入力）でブラウザのメインスレッドを長時間ブロックする攻撃。

**競合の対応状況**: 調査した 8 サイトで明示的なタイムアウト実装を公表しているサイトはない。Site24x7 等のサーバー側処理はサーバーリソースを消費するが、クライアントをブロックしない。

**yolos.net の現行実装（優位点）**:

- Web Worker でメインスレッドから分離して正規表現を実行
- 500ms タイムアウトで Worker を強制終了
- 入力文字列の最大長 10,000 文字に制限
- デバウンス 300ms で Worker 起動頻度を制御

この実装は日本語圏の競合サイトと比較して最も堅牢なReDoS対策であり、差別化として積極的に訴求すべき。

### 9.3 実行タイムアウトの実装方針

Web Worker + setTimeout による強制終了は現在の最善実装。参考文献（Snyk ReDoS ガイド）では 100〜500ms が推奨値。yolos.net の 500ms は実用的に妥当。ReDoS が疑われる場合のエラーメッセージとして「パターンが複雑すぎる可能性があります。パターンを見直してください。」が表示されるのも適切。

## 10. 差別化軸サマリーと cycle-215 実装への示唆

### 現時点での yolos.net の差別化可能な軸

**強み（すでに実装済み）**:

1. **ReDoS対策**: Web Worker + 500ms タイムアウト + 入力長制限。競合サイトに明示的な対策なし。
2. **a11y（色のみ回避）**: マッチ結果をテキストリスト（#番号・位置・グループ）で表示。お便利ツール等の「赤色のみ」表示に優る。
3. **ローカル完結**: Worker でブラウザ内完結。プライバシー面で明示訴求可能。
4. **g/i/m/s の 4 フラグ対応**: 競合の多くが 3 フラグのみ。

**gap（競合優位点で yolos.net が未対応）**:

1. **テキストエリア内ハイライト**: 全競合が実装しており、yolos.net が唯一未実装（現状はリストのみ）。視覚的フィードバックが弱い。→ cycle-215 タイル化と同時に検討推奨
2. **サンプルドロップダウン**: instant tools の 20 サンプルが優秀。placeholder に例文を設定するだけでも改善効果大。→ cycle-215 に組み込み可能
3. **Explanation 機能**: regex101 のみの独自優位点。中長期で実装を検討。

### 訴求すべき差別化メッセージ（タイル説明文候補）

- 「ブラウザだけで完結・サーバー送信なし」
- 「無限ループしない（ReDoS対策済み）」
- 「g / i / m / s フラグ対応」
- 「置換プレビューあり」
