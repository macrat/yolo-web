# GA4データ活用の仕組み設計 — データに基づく運営判断と来訪者向け情報公開の提案

## 1. エグゼクティブサマリー

### 概要

本提案は、Google Analytics 4のデータをAIチームが自然言語で活用できるMCP環境を構築し、以下の2つの目的を実現するための仕組み設計である。

**目的1（内部利用）**: Phase 4-1「低価値コンテンツの削除」の前提条件として、GA4データに基づく削除判断フレームワークを確立する。AI例文導入（B-223/B-224）の効果検証も含め、データでコンテンツの価値を定量評価する。

**目的2（外部公開）**: GA4データをサイト上で来訪者向けに活用する。「今週の人気診断TOP5」のような動的ランキングは社会的証明として初回訪問者の回遊を促進し、PV向上に直接貢献する。

### 期待効果

- Phase 4-1の削除判断を直感ではなくデータで実施でき、価値あるコンテンツを誤って削除するリスクを防ぐ
- AIエージェントが任意のタイミングで自然言語でGA4データを照会できる分析環境の確立
- 人気ランキング表示により、初回訪問者のコンテンツ選択を支援しサイト内回遊を促進
- constitution.md Rule 3（AI運営の透明性）に沿ったデータ公開による信頼性向上

### コスト

GA4 Data APIは**完全無料**（クォータの範囲内）。MCP GA4サーバーは既存OSS（`surendranb/google-analytics-mcp`）の採用を推奨し、実装コストは最小。外部公開機能の実装コストはデータ取得方式により異なる（セクション5参照）。

## 2. GA4 Data API技術調査のまとめ

### 2.1 基本仕様

| 項目              | 内容                                      |
| ----------------- | ----------------------------------------- |
| 料金              | 完全無料                                  |
| Node.js公式SDK    | `@google-analytics/data`（npm）           |
| 認証方式          | サービスアカウント認証（推奨）、OAuth 2.0 |
| 対応GA4バージョン | GA4（Universal Analytics非対応）          |

### 2.2 APIクォータ（2026年3月時点）

出典: [GA4 Data API Quotas](https://developers.google.com/analytics/devguides/reporting/data/v1/quotas)

| クォータ種別           | 上限    |
| ---------------------- | ------- |
| 日次コアトークン       | 200,000 |
| 時間あたりコアトークン | 40,000  |
| 同時リクエスト         | 10      |

本サイトの想定利用量（MCP経由でのアドホック分析: 1回あたり数〜数十トークン、1日数十回程度）では、クォータの1%にも満たない。現行サイト規模では実質的な制約にならない。

### 2.3 取得可能な指標（Phase 4-1の判断に必要なデータ）

| ディメンション/メトリクス | GA4 API名                | 用途                     |
| ------------------------- | ------------------------ | ------------------------ |
| ページパス                | `pagePath`               | ページ別PV集計           |
| ページビュー数            | `screenPageViews`        | PV数                     |
| セッション数              | `sessions`               | 訪問数                   |
| エンゲージドセッション数  | `engagedSessions`        | 実際に読まれた訪問数     |
| エンゲージメント率        | `engagementRate`         | 質の高い訪問の割合       |
| 平均セッション時間        | `averageSessionDuration` | 滞在時間                 |
| 直帰率                    | `bounceRate`             | 離脱率                   |
| イベント数                | `eventCount`             | カスタムイベント発生回数 |
| イベント名                | `eventName`              | カスタムイベント種別     |

200以上のディメンション・メトリクスが利用可能であり、Phase 4-1の判断に必要なデータはすべてAPIで取得できる。

### 2.4 カスタムイベントの取得可否

現在のGA4実装（`src/lib/analytics.ts`）で送信しているカスタムイベントは、`eventName`ディメンションでフィルタリングすることで全件取得・分析が可能である。

| カスタムイベント     | 取得可否 | 活用用途                                                |
| -------------------- | -------- | ------------------------------------------------------- |
| `level_start`        | 可       | ゲーム/診断の開始率                                     |
| `level_end`          | 可       | ゲーム/診断の完了率（完了率 = level_end / level_start） |
| `unlock_achievement` | 可       | 実績解除によるエンゲージメント測定                      |
| `search`             | 可       | サイト内検索の利用状況                                  |
| `share`              | 可       | SNSシェア率（コンテンツの口コミ拡散力）                 |
| `content_rating`     | 可       | 来訪者満足度                                            |

特に `level_end / level_start` の完了率は、ゲーム・診断コンテンツのエンゲージメントを定量化する上で重要な指標となる。

### 2.5 Search Consoleデータの取得方法

GA4 Data APIとSearch Console APIの関係を以下に整理する。

| データ種別                                                           | 取得方法                     | 備考                                                    |
| -------------------------------------------------------------------- | ---------------------------- | ------------------------------------------------------- |
| GA4とSC連携設定がある場合の集計指標（`organicGoogleSearchClicks`等） | GA4 Data API経由で取得可     | ページ別クリック数の集計値のみ                          |
| **個別の検索クエリ（キーワード別データ）**                           | **GA4 Data APIでは取得不可** | Search Console API（`searchanalytics.query`）が別途必要 |
| ページ別の検索クエリ内訳                                             | Search Console API           | 1レスポンスあたり25,000行まで                           |

#### Search Console API概要（参考）

| 項目           | 内容                                                              |
| -------------- | ----------------------------------------------------------------- |
| 料金           | 完全無料                                                          |
| クォータ       | 1,200 QPM/サイト                                                  |
| 認証方式       | サービスアカウント認証可（GA4と同一のサービスアカウントを使用可） |
| 取得可能データ | クエリ別クリック数・表示回数・CTR・掲載順位                       |

**現時点での方針**: 本提案ではGA4 Data APIに絞って環境を構築する。検索クエリの詳細分析が必要になった際に、同一のサービスアカウントでSearch Console APIも追加接続できる設計とする。

### 2.6 認証方式（サービスアカウント）

ownerが以下を実施することでAIエージェントがAPI利用可能になる。

1. Google Cloud Consoleでプロジェクトを作成（または既存プロジェクトを使用）
2. Google Analytics Data APIを有効化
3. サービスアカウントを作成し、JSONキーファイルをダウンロード
4. GA4管理画面でサービスアカウントのメールアドレスに「閲覧者」ロールを付与
5. JSONキーファイルをローカルの安全な場所に保存（例: `~/.config/ga4-service-account.json`）

### 3.2 推奨案

**推奨: `surendranb/google-analytics-mcp`（ローカル実行）**

採用理由:

1. ローカル実行のためGA4データが外部に流れない（プライバシー保護）
2. 7つのMCPツールを提供し、ディメンション・メトリクスのスキーマ探索が可能（LLMが適切な名前を自己発見でき、初回設定の試行錯誤を軽減）
3. コンテキスト安全機能（大量データ警告・トランケート）がAIエージェント利用に必須
4. サービスアカウントJSONキー直接指定（gcloud CLI不要）でセットアップが容易
5. 積極的にメンテナンスされており、PyPI公開済みで導入が容易
6. サービスアカウント認証に対応しており、B-113提案書のMCPパターンと整合する

**将来的な移行先として: `googleanalytics/google-analytics-mcp`（公式版）**

公式版はリアルタイムレポート、アノテーション、マルチプロパティ等のメリットがある。ただし現時点ではGemini特化でClaude Code向けドキュメントがなく、gcloud CLI設定が必要なため、本提案では推奨しない。将来的にClaude Code対応が充実した段階で移行を検討する。

**次善: Stape GA4 MCP Server**（セットアップが最も簡単だが外部サーバー経由のためプライバシーリスクあり）

**除外: harshfolio/mcp-server-ga4**（アーカイブ済みのため採用不可）

### 3.3 セットアップ手順（推奨案採用時）

#### ownerが実施する設定作業

1. サービスアカウントの作成とJSONキーファイルの取得（セクション2.6参照）
2. JSONキーファイルを `~/.config/ga4-service-account.json` 等に保存
3. シェル環境変数の設定:
   ```bash
   # ~/.zshrc または ~/.bashrc に追加
   export GOOGLE_APPLICATION_CREDENTIALS="$HOME/.config/ga4-service-account.json"
   export GA4_PROPERTY_ID="プロパティID（数字のみ）"
   ```

#### `.mcp.json` への追記（B-113パターンと同方式）

```json
{
  "mcpServers": {
    "playwright": {
      "...": "..."
    },
    "google-analytics": {
      "type": "stdio",
      "command": "uvx",
      "args": ["google-analytics-mcp"],
      "env": {
        "GOOGLE_APPLICATION_CREDENTIALS": "${GOOGLE_APPLICATION_CREDENTIALS}",
        "GA4_PROPERTY_ID": "${GA4_PROPERTY_ID}"
      }
    }
  }
}
```

`${GOOGLE_APPLICATION_CREDENTIALS}` はClaude Codeがサーバープロセス起動時にシェル環境変数から展開してサーバーに注入する（B-113提案書のMCPパターンと同一の方式）。

#### B-113提案書のMCPアーキテクチャパターンとの整合

B-113（画像生成）と同一のstdio方式を採用する。

```
Claude Code  ---(MCP protocol/stdio)--->  MCPサーバープロセス  ---(HTTPS)--->  GA4 Data API
                                          [JSONキーパスはここにのみ存在]
```

APIキーに相当するJSONキーファイルのパスはMCPサーバープロセス内にのみ存在し、Claude Codeのコンテキストには流れない。多層防御（MCPプロトコル隔離 + ルールによる禁止）はB-113提案書と同様のパターンを適用する。

## 4. Phase 4-1削除判断フレームワークの提案

### 4.1 評価対象コンテンツ

`docs/site-concept.md`のセクション5に記載された削除候補コンテンツが対象である。ただしB-111の検討結果（四字熟語辞典へのAI例文全件導入、漢字辞典の小学1-3年生への限定導入を推奨。色辞典は見送り）を踏まえ、AI例文が実装予定の辞典については「再評価パス」を設ける（セクション4.5参照）。

既存トラフィックデータ（2025-12-17〜2026-03-16、90日間）における辞典カテゴリとcolorsカテゴリは別カテゴリとして集計されていることに注意。

| カテゴリ                      | 対象コンテンツ      | 既存PV（90日） | 検索クリック（90日） | B-111との関係                           |
| ----------------------------- | ------------------- | -------------- | -------------------- | --------------------------------------- |
| 辞典（/dictionary/）          | kanji, yoji の2種   | 100            | 2                    | kanji/yojiはAI例文導入予定 → 再評価パス |
| colors（/dictionary/colors/） | colors              | 24             | 1                    | B-111でAI例文見送り → 通常閾値を適用    |
| チートシート                  | 全7種               | 16             | 0                    | なし                                    |
| ツール                        | 全33種              | 86             | 0                    | なし                                    |
| ブログ                        | 5-6件（低品質記事） | 一部           | 一部                 | なし                                    |

注: colorsは実際のサイト構造では `/dictionary/colors/` 配下に存在しますが、トラフィック分析レポート（`docs/research/2026-03-17-traffic-analysis-for-url-restructuring.md`）では独立カテゴリ「colors」として集計されています。本表では辞典カテゴリと分離して記載しています。

### 4.2 判断に使用する指標

削除判断はPV数のみに依存せず、以下の多面的な指標で評価する。

| 優先度 | 指標                                      | GA4 API取得方法                            | 判断上の意味               |
| ------ | ----------------------------------------- | ------------------------------------------ | -------------------------- |
| 高     | ページビュー数（90日）                    | `screenPageViews`                          | 絶対的な利用量             |
| 高     | サイト全体に占めるPV割合                  | 上記÷全体合計                              | 相対的な重要度             |
| 高     | エンゲージメント率                        | `engagementRate`                           | 来訪者が価値を感じているか |
| 中     | 検索クリック数                            | Search Console連携データ or 既存分析データ | 外部からの流入価値         |
| 中     | カスタムイベント（share, content_rating） | `eventCount` + `eventName`フィルタ         | 来訪者アクション           |
| 中     | 完了率（ゲーム/診断のみ）                 | `level_end` / `level_start`                | コンテンツの完結性         |
| 低     | 直帰率                                    | `bounceRate`                               | ページの初期印象           |

### 4.3 判断基準（閾値）

現時点のサイト規模（月間PV約500（2025-12-17〜2026-03-16の90日間で1,543PV、月換算約514PV））を前提とした相対的な基準として提案する。サイト成長に伴い見直しが必要である。

| 判断           | 条件（すべてを満たす）                                                                     | 対応方針                                                            |
| -------------- | ------------------------------------------------------------------------------------------ | ------------------------------------------------------------------- |
| **削除**       | PV（90日）が10未満 **かつ** 検索クリックがゼロ **かつ** エンゲージメント率が30%未満        | 削除を実施。HTTPステータス410 Goneを使用し、sitemap.xmlから除外する |
| **存続を検討** | PV（90日）が10以上50未満 **または** 検索クリックが1件以上                                  | 改善施策を実施後、次サイクルで再評価                                |
| **存続**       | PV（90日）が50以上 **または** 検索クリックが5件以上 **または** エンゲージメント率が60%以上 | 存続確定。改善機会があれば別途検討                                  |

**注意点**: AI例文実装が予定されている辞典（四字熟語辞典・漢字辞典小学1-3年生）については、B-223/B-224完了まで削除判断を保留する（セクション4.5参照）。

### 4.4 新コンテンツ（占い・診断パーク）との比較評価

site-value-improvement-plan.mdの前提条件「低価値コンテンツの削除は、新コンテンツがサイトのトラフィックを十分に支えているかをGAデータで検証してから実施」に対応するための判断方法。

**「新コンテンツがサイトを十分に支えている」の定義（提案）**:

| 条件                 | 内容                                              | 根拠                                   |
| -------------------- | ------------------------------------------------- | -------------------------------------- |
| PV条件               | 占い・診断パークのPVがサイト全体の30%以上を占める | 単一コンテンツへの依存が高すぎない水準 |
| 成長条件             | 直近30日のPVが前月比で増加傾向にある              | 一時的なピークではなく持続的な成長     |
| エンゲージメント条件 | エンゲージメント率が50%以上                       | 来訪者が価値を感じていることの確認     |

上記3条件をすべて満たした場合に、削除フェーズを開始できると判断する。

### 4.5 B-111検討結果を踏まえた再評価パス

B-111でAI例文追加が推奨されている辞典（四字熟語辞典・漢字辞典小学1-3年生）については、以下の判断フローを適用する。

```
辞典の評価開始
  ↓
AI例文実装予定（B-223/B-224対象）か？
  ↓ No（AI例文実装予定なし: 色辞典等）→ 通常の削除判断閾値を適用（セクション4.3）
  ↓ Yes（四字熟語辞典・漢字辞典）→ B-223/B-224完了まで削除判断を保留
              ↓ B-223/B-224完了後 → AI例文追加後のデータで再評価
              ↓
            PV・エンゲージメント率が向上しているか？
              ↓ Yes → 「独自性あり」として存続確定
              ↓ No  → 通常の削除判断閾値を適用
```

色辞典（colors）はB-111で「AI例文見送り」と判断されたため、通常の削除判断閾値を適用する。

### 4.6 既存トラフィックデータとの整合

`docs/research/2026-03-17-traffic-analysis-for-url-restructuring.md`（2025-12-17〜2026-03-16、90日間）のデータが既に存在する。GA4 MCP環境構築後は最新データと比較することで、コンテンツのトレンドを把握できる。

| データ活用場面      | 活用方法                                                              |
| ------------------- | --------------------------------------------------------------------- |
| 削除判断の初期評価  | 既存の90日データを参照し、MCP環境構築前の暫定評価として使用可         |
| MCP環境構築後の比較 | 同期間のデータをGA4 APIで取得し、整合性を確認した上で最新データに移行 |
| トレンド把握        | 月別推移をGA4 APIで取得し、右肩上がりか横ばいかで判断を調整           |

## 5. 外部公開機能の検討

### 5.1 人気コンテンツランキング

#### 来訪者にとっての価値

「今週の人気診断TOP5」「みんなが遊んでいるゲーム」のような動的ランキングは、社会的証明として機能する。

- **初回訪問者の意思決定を支援**: 「何から始めるか」という迷いを解消し、最初のコンテンツ選択を促進
- **回遊促進**: 自分が楽しんだコンテンツ以外の人気コンテンツを知ることで、追加訪問を促進
- **信頼感の醸成**: 「他の人も遊んでいる」という証明がコンテンツへの信頼感を高める

**判断**: 人気ランキングは来訪者にとって高い実用価値があり、PV向上に直結する可能性がある。

#### 技術的実現方法の比較

| 方式                           | 概要                                                                                                                                                      | メリット                                                                       | デメリット                                                                               |
| ------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------- |
| **A: ISR方式**                 | Next.jsのISR（revalidate指定）でビルド時+定期的にGA4 APIを呼び出してランキングを更新                                                                      | データの鮮度が高い（数時間〜1日単位で更新可能）                                | サーバーサイドでの外部API呼び出しが発生。coding-rules.mdルール2との関係をowner判断が必要 |
| **B: GitHub Actions cron方式** | 日次cronでGA4 APIを呼び出してJSON（`public/ranking.json`）を生成し、Vercelへのデプロイをトリガー。GA4サービスアカウントJSONキーはGitHub Secretsで管理する | ビルド時静的生成に近い設計。coding-rules.mdルール1（静的最優先）との整合が高い | データ更新は日次（最大24時間の遅延）。GitHub Actions設定が必要                           |
| **C: 見送り**                  | ランキング機能を実装しない                                                                                                                                | 実装コストゼロ                                                                 | 来訪者への価値提供の機会を逃す                                                           |

**技術的な推奨**: B（GitHub Actions cron方式）を推奨する。理由は、サーバーサイドでのランタイムAPI呼び出しを避け、coding-rules.mdルール1（静的最優先）との整合が高いため。ただし、**方式A・Bのどちらを採用するか（またはCとするか）はcoding-rules.mdルール2との関係においてowner判断が必要**（セクション7参照）。

### 5.2 アクセス統計公開ページ

#### 来訪者にとっての価値

| 観点                                   | 評価                                                                       |
| -------------------------------------- | -------------------------------------------------------------------------- |
| メインターゲット（占い・診断ユーザー） | 統計ページへの関心は低い。**価値は限定的**                                 |
| サブターゲット（AIエンジニア）         | 「AIが運営するサイトの裏側」として知的好奇心を満たせる。**一定の価値あり** |

**判断**: 実装コストに対して来訪者への価値は人気ランキングより低い。人気ランキング（セクション5.1）を優先実装した後に、余力があれば検討する。

## 6. セキュリティ・プライバシーの整理

### 6.1 サービスアカウントキーの管理

| 項目                 | 方針                                                                                                                                                    |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 保管場所             | ローカル: `~/.config/ga4-service-account.json`（ホームディレクトリ配下、リポジトリ外）                                                                  |
| 環境変数             | `GOOGLE_APPLICATION_CREDENTIALS`（シェル環境変数でJSONキーファイルパスを指定）。`NEXT_PUBLIC_` プレフィックス禁止                                       |
| Vercel環境変数       | 外部公開機能を実装する場合はVercelの環境変数として設定                                                                                                  |
| GitHub Actions利用時 | サービスアカウントJSONキーはGitHub Secretsに登録して管理する。GitHub Actionsのワークフロー内で `${{ secrets.GA4_SERVICE_ACCOUNT_JSON }}` として参照する |
| バージョン管理       | JSONキーファイルは一切バージョン管理に含めない                                                                                                          |

### 6.2 MCP経由のアクセスにおけるセキュリティ

B-113提案書の多層防御パターンをGA4 MCPサーバーにも適用する。

| 防御層                   | 内容                                                                               |
| ------------------------ | ---------------------------------------------------------------------------------- |
| 第1層: MCPプロトコル隔離 | JSONキーのパスはMCPサーバープロセス内にのみ存在。MCPレスポンスにキー情報を含めない |
| 第2層: ルールによる禁止  | `.claude/rules/`にJSONキーファイルの直接読み取り禁止ルールを追加                   |
| 第3層: 読み取り専用権限  | サービスアカウントにはGA4「閲覧者」ロールのみを付与。データ変更は不可              |

GA4の読み取り専用APIであるため、Imagen 4のAPIキーと比較してリスクは低い（漏洩しても料金被害は発生しない）。ただし、GAデータは運営上の機密情報であるため、漏洩防止の基本的な多層防御は適用する。

### 6.3 公開データの範囲とプライバシー

外部公開機能で公開するデータは**集計データのみ**とする。以下のデータは一切公開しない。

- 個人を特定できる情報（IPアドレス、ユーザーID等）
- GA4の「ユーザー探索」で得られるような個人レベルのデータ
- セッションIDやデバイスIDに関連する情報

**日本法上の確認**: 集計・匿名化された統計データの公開は、個人情報保護法の「個人情報」に該当しない。GA4が提供する集計APIのレスポンスは既に匿名化されており、個人を特定するデータは含まれない。

### 6.4 プライバシーポリシーへの追記（外部公開機能を実装する場合）

外部公開機能を実装する場合は、プライバシーポリシーに以下を追記する。

- GA4で収集したアクセスデータを集計・匿名化した上でサイト上で公開することを明示
- 個人を特定できるデータは公開しないことを明記
- 表示内容（PV数、人気ランキング等）の更新頻度と提供元（Google Analytics）を記載

### ルール2との関係についての補足

coding-rules.mdのルール2は「外部APIの呼び出し、データベース、認証・ユーザー管理は実装しない」と包括的に定めており、`paths: src/**/*` の範囲に適用される。

GA4 Data APIの呼び出しをどこで行うかによって、ルール2との関係が異なる。

- **MCP経由の内部利用**（セクション3）: APIの呼び出しはMCPサーバープロセス（`src/`外）で行われるため、ルール2の適用範囲外。問題なく実施できる。
- **外部公開機能のAPI呼び出し**: GitHub Actions cron方式（B）の場合、API呼び出しはGitHub Actions内で行われ、`src/`の実装はJSON読み込みのみとなる。この場合のルール2との整合性についてはowner判断が必要。ISR方式（A）の場合も、Next.jsのサーバーサイド処理でのAPI呼び出しについてルール2の例外に該当するかはowner判断が必要。

B-113提案書では `mcp-image-server/` という `src/` 外のディレクトリに配置することでルール2の適用範囲外とした。GA4の外部公開機能についても、データ取得スクリプトを `scripts/` 等の `src/` 外のディレクトリに配置することで同様の整合を図ることができる。
