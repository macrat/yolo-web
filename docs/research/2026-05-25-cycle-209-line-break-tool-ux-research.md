# cycle-209 改行削除ツール UX 調査

## 1. visitor needs の実態：「PDF から改行を除去する」ニーズ

### 1.1 ユーザーの主要な痛点と使用文脈

linebreakremover.com のブログ記事（技術解説付き）によると、PDF コピペで改行が乱入する問題はレイアウト上の構造に起因する。PDF は視覚的な行ごとに改行コードを保存するため、コピーすると意図しない「ハードラインブレーク」が発生する。

**主要ユースケース（重要度順）:**

1. **コンテンツ制作・出版ワークフロー**: ライター・編集者が論文・報告書から引用する際に毎回 5〜10 分の整形作業が発生。年間換算で 40〜80 時間の非生産的作業
2. **学術・研究用途**: 論文引用・データ抽出（特に英文 PDF）でスペース欠落問題が顕在化。日本語は文字が連続しても意味が通じるが、英文は単語間スペースが失われると致命的
3. **業務文書処理**: 特許庁通知書類（Japio ツールが特化）など業務固有フォーマットへの対応
4. **翻訳ワークフロー**: DeepL / Google 翻訳に貼り付ける前処理として改行を除去するニーズが日本語圏で特に旺盛

**差別化訴求点（ユーザーが本当に求めるもの）:**

- 「貼り付けてすぐ結果が出る」即時性
- 「段落は壊したくない」構造保持への要求（= smart-pdf モードの差別化価値）
- プライバシー（ブラウザ完結・サーバー非送信）の明示

### 1.2 日本語 vs 英文での機能差別化の実態

| 用途                         | 推奨モード                    | 理由                                 |
| ---------------------------- | ----------------------------- | ------------------------------------ |
| 日本語テキスト（一般）       | 改行削除（remove）            | 和文は文字が直接連続しても読める     |
| 英文テキスト（一般）         | スペース置換（replace-space） | 単語間スペースが必須                 |
| PDF からのコピー（段落保持） | smart-pdf                     | 段落構造を保持しつつ行内改行のみ処理 |
| PDF からのコピー（英文論文） | smart-pdf + スペース置換      | 段落保持 + 英文単語スペース確保      |

yolos.net の line-break-remover は「smart-pdf モード」で段落区切りの保持と行内改行の処理を自動判別する機能を持つ。この点が競合と最大の差別化要素となる。

## 3. 2 階層オプション（モード選択 + サブオプション）の UI パターン

### 3.1 実例パターン分類

競合ツール・デザインシステムから確認した「モード選択後にサブオプションが現れる」実装パターンは以下の 4 種類：

#### パターン A: 常に表示（条件付き disabled/enabled）

**実例**: linebreakremover.com の「Custom」モード選択時の入力欄

- サブオプションは常に DOM に存在し、非対象モード選択時は `disabled` 属性
- **メリット**: ユーザーが何が変わるかを事前に把握できる。レイアウトシフトなし
- **デメリット**: disabled 状態の操作不能感がユーザーに「なぜ操作できないか」の認知負荷を与える
- **Smashing Magazine 推奨**: "Disable if you want the user to know a feature exists but is unavailable"

#### パターン B: 条件成立時のみ表示（レイアウトシフトあり）

**実例**: アサリツールズの「全てのスペース」チェック時のサブオプション、yolos.net 現行実装（`{mode === "smart-pdf" && ...}`）

- 非対象時は DOM から削除または `display: none`
- **メリット**: 認知負荷が低い。関連するモードを選んだ時だけ選択肢が現れる
- **デメリット**: レイアウトシフトが発生しうる。初回ユーザーがサブオプションの存在を知らない
- **NN/G 推奨**: "Hide if the value shown is currently irrelevant and can't be used"

#### パターン C: アコーディオン（高さアニメーション付き展開）

- CSS の `height: 0 → auto` トランジション（grid-template-rows を使う手法で実現可能）または `@starting-style` / `interpolate-size` を使用
- **メリット**: 展開する感覚でサブオプションの存在を事前通知しつつ、選択後に滑らかに表示
- **デメリット**: 実装複雑度がやや高い
- **適合場面**: タイル UI など高さ制約があるが、オプションの存在を示唆したいケース

#### パターン D: インライン埋め込み（プリセット選択型）

**実例**: ラッコ手帳の「Excel 用おまかせ」「Word 用おまかせ」ボタン

- 主要ユースケースをプリセットとしてワンタップ選択し、詳細設定は非表示のまま
- **メリット**: 上級ユーザー向けの詳細設定と初心者向けのプリセットを分離できる
- **デメリット**: smart-pdf モードのような「モードごとの異なるサブオプション」とは相性が異なる

### 3.2 WAI-ARIA / WCAG 観点での推奨事項

#### ラジオボタンが条件付き表示をトリガーする場合の問題点

GOV.UK アクセシビリティチームの調査（2021）によると：

- ラジオボタン（`role="radio"`）に `aria-expanded` を付与することは ARIA 仕様で未定義（サポートされていない）
- ラジオ選択で新しいコンテンツが現れる際、スクリーンリーダーユーザーに「変化が起きた」ことを通知する標準的な方法が存在しない
- ただし、**単一の入力フィールドが現れるケース**（= yolos.net の joinStyle ラジオ 2 択）は実際のユーザーテストで問題が少ないことが確認された
- 複数フィールドや複雑なコンテンツの条件付き表示は混乱を招く

**推奨する実装方針（yolos.net のケース）:**

1. `mode === "smart-pdf"` 時に表示されるサブオプション領域（`<fieldset>` + `<legend>`）に `aria-live="polite"` は不要（ラジオ操作が能動的なため）
2. サブオプション領域をスクリーンリーダーが認識できるよう、`<fieldset>` / `<legend>` でグループ化する（現行実装は `<div className={styles.radioGroup}>` のみでセマンティクスが弱い）
3. DOM からの削除（`display: none`）は `aria-hidden="true"` と同等の効果があるため、パターン B（条件付き表示）はアクセシビリティ観点で正当

#### WAI-ARIA Disclosure パターンとの関係

WAI-ARIA APG の Disclosure パターン（`role="button"` + `aria-expanded` + `aria-controls`）はボタンで制御されるパネルの展開・収縮に特化している。モード選択（ラジオ）からのサブオプション表示には直接適用できないが、設計思想は参考になる：

- **表示/非表示の二値状態**を明確に持つ
- 制御要素と被制御要素の**関連付け**（`aria-controls`）を持つ
- **視覚的な方向性**（矢印アイコン）でユーザーに展開可能性を示唆する

### 3.3 タイル UI（約 400×260〜380px）における「2 階層オプション」設計の推奨

#### 制約の整理

- タイル高さ: 260〜380px（拡大展開なし前提）
- コンテンツ: モード選択ボタン 3 つ + 条件付きサブオプション 2 択 + 入力エリア + 出力エリア + コピーボタン
- モード選択が最上位の判断（T1 ユーザーの最初の操作）

#### 推奨パターン: 「条件付き表示（アニメーション付き縮小展開）」

タイル UI の限られた高さを踏まえると、**パターン B（条件付き表示）をアニメーション付きで実装する**のが最も visitor 価値が高い。

理由：

1. smart-pdf モードを選んだことがない初回ユーザー（T1）はサブオプションの存在を知らなくてよい。むしろ選択肢が常に見えていると認知負荷が増す
2. smart-pdf モードを繰り返し使う T2 ユーザーは操作を記憶しているため、条件付き表示に驚かない
3. タイル高さが固定ならサブオプション表示でコンテンツが圧迫される問題がある → タイル高さを動的に伸ばすか、入力エリアを縮小する設計が必要

#### 高さ変動への対応

CSS `grid-template-rows: 0fr → 1fr` トランジションを用いた高さアニメーション、または CSS の `interpolate-size: allow-keywords` + `height: auto` トランジション（Chrome 129+ / Firefox 130+）でスムーズな展開が可能。

タイル自体の高さが可変（expandable tile）ならサブオプション領域を `overflow: hidden` + 高さトランジションで処理すると、レイアウトシフトの問題が軽減される。

## 4. yolos.net の line-break-remover の差別化機会

### 4.1 競合が持っていない機能（差別化できる要素）

| 機能                               | yolos.net | アサリ | Smart | 整形くん | ラッコ | Japio | LineBreaker.com | TextFixer     |
| ---------------------------------- | --------- | ------ | ----- | -------- | ------ | ----- | --------------- | ------------- |
| 段落保持（smart-pdf）              | ✓         | -      | -     | -        | -      | -     | -               | ✓（基本のみ） |
| 行内改行の処理選択（remove/space） | ✓         | -      | -     | -        | -      | -     | -               | -             |
| 連続改行の統合オプション           | ✓         | -      | -     | -        | ✓      | -     | -               | -             |
| リアルタイム反映                   | ✓         | -      | ✓     | ✓        | ✓      | -     | -               | -             |
| 処理件数のフィードバック           | ✓         | -      | -     | -        | ✓      | -     | -               | -             |
| 日本語/英文の使い分け説明          | ✓（FAQ）  | ✓      | -     | ✓        | -      | -     | -               | -             |
| 完全無料・登録不要                 | ✓         | ✓      | ✓     | ✓        | ✓      | ✓     | ✓               | ✓             |

**最大の差別化要素**: `smart-pdf` モード + `joinStyle` サブオプション（削除 / スペース置換）の組み合わせは、調査した競合 7 サイト中で唯一 yolos.net のみが提供。英語圏の TextFixer が段落保持モードを持つが、行内改行の処理スタイル選択は持たない。

### 4.2 UI 観点での改善機会

**現行の課題（タイル化に向けて）:**

1. **モード説明の不足**: 「PDFスマートモード」というラベルだけでは、初回ユーザーが何を意味するか把握しにくい。短い説明文（1 行）か tooltip の追加が有効
2. **サブオプションのセマンティクス**: 現行は `<div className={styles.radioGroup}>` でラップされているが、`<fieldset>` + `<legend>` によるアクセシビリティ向上が望ましい
3. **リアルタイム反映の説明**: 競合との差別化になるリアルタイム反映を、ユーザーが最初に気づきやすい形で示す（「入力するだけで自動変換」等の hint text）
4. **コピー後の視覚フィードバック**: 現行の「コピー済み」表示は短時間（2 秒）。T2 ユーザーの繰り返し利用を考えると適切な長さ

**競合との UX 差でユーザーが離脱しやすいポイント:**

- 「実行ボタンが必要」（アサリ・Japio・LineBreaker.com など）: yolos.net はリアルタイム反映で差別化できている
- 「段落が壊れる」（smart-pdf 非搭載競合全般）: これを事前に説明できれば来訪者の意思決定を助けられる

## 5. タイル化設計への示唆

### 5.1 タイル内 UI の推奨構造

限られたスペース（400×260〜380px）に収める場合の優先順位：

```
[モード選択: セグメントコントロール/ボタン群 3択]
[サブオプション: smart-pdf 選択時のみ条件付き表示（アニメーション付き）]
[入力 textarea: 3〜4行程度]
[出力エリア: 単行または折りたたみ形式]
[コピーボタン]
```

### 5.2 セグメントコントロールの採用について

3 モード（remove / replace-space / smart-pdf）の選択には **セグメントコントロール（button group）** が適している。

- 選択肢が 3 つ、相互排他的、常時 1 つが有効 → セグメントコントロールの典型的ユースケース
- GitHub Primer の実装では `aria-current="true"` を使用（`aria-checked` は radiogroup 向け）
- 現行実装（`role="radiogroup"` + `role="radio"` + `aria-checked`）は ARIA 的に正当だが、デザイン的にはボタン group のほうがコンパクトに見える

### 5.3 「常に表示（disabled）」vs「条件付き表示」の最終判断

**タイル UI での推奨: 条件付き表示（パターン B）を採用、アニメーション付き**

根拠：

- Smashing Magazine（2024）の "hidden vs disabled" ガイドラインでは、「現在は無関係で操作不能なもの」は隠すべきとされる。joinStyle は smart-pdf 非選択時に完全に無関係
- NN/G の進歩的開示の原則「只显示最重要的选项」とも整合する
- タイル高さが限られている場合、disabled のサブオプションが常に見えていると視覚的ノイズになる
- 初回ユーザー（T1）への認知負荷軽減の観点でも「現れる」体験のほうが「なぜ灰色なんだろう」という疑問を生まない

ただし、**アニメーション（高さトランジション）を付けることで「条件が変わると何かが起きる」ことをユーザーに通知**する。これが条件付き表示の唯一の欠点（発見性の低さ）を補う。

### 5.4 セマンティクス改善の提案（アクセシビリティ）

現行 Component.tsx の改善ポイント：

```
現行:
  <div className={styles.radioGroup}>
    <span className={styles.radioGroupLabel}>行内改行の処理:</span>
    ...

推奨:
  <fieldset className={styles.radioGroup}>
    <legend className={styles.radioGroupLabel}>行内改行の処理</legend>
    ...
  </fieldset>
```

これにより、スクリーンリーダーが「行内改行の処理、削除する（ラジオボタン）」「行内改行の処理、スペースに置換（ラジオボタン）」と正しくグループコンテキストを読み上げられるようになる。

## 6. 調査まとめ

### 競合の共通的な弱点

1. PDF コピペ時の「段落保持 + 行内改行処理スタイル選択」が存在しない（最大の差別化機会）
2. 多くが実行ボタン必須型でリアルタイム反映を提供しない
3. 「なぜ改行が入るか」の教育コンテンツが少なく、ツールのコンテキスト理解が促進されない

### 訪問者ニーズの実態

- 日本語圏では「PDF からコピーした文章を DeepL / Google 翻訳に貼る前処理」「論文・報告書からの引用整形」が主要ユースケース
- 英語圏では「コンテンツ制作・ライティングワークフロー」での年間 40〜80 時間削減が主訴
- 両者に共通するのは「段落は壊さずに行内改行だけ除去したい」需要 → smart-pdf の価値を前面に出すべき

### 2 階層オプションの設計原則

- タイル UI では「条件付き表示（アニメーション付き）」が認知負荷・スペース効率の両面で優位
- スクリーンリーダー対応には `<fieldset>` + `<legend>` によるグループ化が有効
- 2 レベルまでの開示階層が NN/G の推奨上限（3 レベル以上は禁忌）
- GOV.UK の研究から、シンプルなサブ入力フィールド 1〜2 個の条件付き表示はアクセシビリティ上許容範囲
