# 敬語早見表タイル UI 設計のための UX 調査

## 1. 既存の日本語「敬語」系 Web ツールの UX パターン

### 1.1 代表的なツールの概要と動線

**yolos.net 敬語早見表（https://yolos.net/tools/keigo-reference）**
現行ページをブラウザで確認。ページ上部に「動詞を検索...」プレースホルダ付きテキストボックス、その横に「すべて」「基本動詞」「ビジネス」「接客・サービス」の 4 カテゴリフィルタボタン。下部に「60件の動詞」件数表示と、「普通語 / 尊敬語 / 謙譲語 / 丁寧語」4 列テーブル。行の展開で例文が表示される。「よくある間違い」タブも存在。これは参照型（静的データ + インクリメンタルフィルタ）の実装であり、今回検討中のタイルへの変換対象。

**3秒敬語（https://3keigo.com）**
入力テキストエリアに「カジュアルな日本語文章」を貼り付け、「敬語にする」ボタンを押すと AI が敬語メールを生成する変換型ツール。チュートリアルでは「山田部長、突然すみません！資料の件でちょっと相談したいんで、来週どこかで少しだけ時間もらえませんか？」のようなサンプルを示す。入力→待機→出力の 3 ステップ構造。1 日 3 回超は要ログイン。緑一色のブランドカラーで視覚的に特徴的。

**Toolpods AI 敬語（https://toolpods.io/ai-keigo）**
「敬語レベル」コンボボックス（標準敬語ビジネス一般など）と自由テキストエリアの 2 入力構成。GPT-4.1 nano、Gemini 2.5 Flash、Gemini 2.5 Flash Lite、DeepSeek Chat の 4 モデル比較結果が並列表示される。「変換履歴」タブでログも確認可能。AIモデル比較が主目的で、参照辞書ではなく変換 + 比較型。

**so-zou.jp 文章校正ツール（https://so-zou.jp/web-app/text/proofreading/）**
テキストエリアに文章を貼り付けて「検査」ボタンを押し、Yahoo! JAPAN の校正 API でエラー箇所を色付け表示する校正型ツール。ら抜き言葉・二重敬語・助詞の抜けなどを指摘。黒背景・左サイドバーにナビゲーションリンクが並ぶ機能重視のレイアウト。最大 10,000 文字対応。登録不要・無料。

**タウンワークマガジン 敬語変換表（https://townwork.net/magazine/knowhow/manners/baito_manners/13309/）**
記事コンテンツとして静的 HTML テーブルを掲載。「行く→いらっしゃる（尊敬語）／参る・うかがう（謙譲語）／行きます（丁寧語）」のような変換表を縦並びに列挙。検索・フィルタ機能なし。読み物として消費するコンテンツ型で、参照に戻ってくることを想定していない。

**ビジネス敬語の達人 敬語変換表（https://bizkeigo.koakishiki.com/henkan/）**
カテゴリ別（行動動詞、思考・言語動詞など）に静的テーブルを並べる構成。検索なし。サイドバーに内部検索ウィジェット（キーワード検索→ページ移動）あり。

### 1.2 入力→出力動線の類型化

日本語敬語系ツールの動線は大きく 3 類型に分類できる。

**A. 変換型（AI 生成）**
入力フォームに文章全体を入力 → ボタン押下 → API/AI で結果生成 → テキスト出力 → コピー。代表: 3秒敬語、Toolpods AI敬語、LeapMe。待機時間が発生する。入力→変換→コピーの 3 ステップ。

**B. 参照・検索型（静的データ）**
検索ボックスにキーワード入力 → 即時フィルタ → テーブル行で尊敬語/謙譲語/丁寧語を確認 → コピー。代表: yolos.net（現行）。待機ゼロ。入力→確認→コピーの 3 ステップだが体感は 2 ステップ。

**C. 読み物・静的早見型**
記事内の固定テーブルをブラウザ内検索（Ctrl+F）で探す。代表: タウンワーク、ビジネス敬語の達人。ツールとしての意図はなく、参照ツールとしては不便。

ターゲット利用者（「この表現の敬語は？」と調べてすぐコピーして戻る）に適合するのは B 型のみ。A 型は文章生成用途であり今回は対象外。

## 2. 参照・検索型ウィジェット UI のベストプラクティス

### 2.1 コマンドパレット/クイック検索パターンとの類似性

400×400px タイルでの「検索 → 即時フィルタ → 結果をコピー」という動線は、VS Code のコマンドパレット、Figma のリソース検索、絵文字ピッカー（emoji-mart など）と構造的に同じ。

- 上部: 固定検索ボックス（常にフォーカス可能）
- 下部: スクロール可能な結果リスト
- 各行: ワンタッチで主要アクション（コピー）

このパターンの定石として確認されているのは以下の通り。

**検索ボックス**

- プレースホルダで何が検索できるかを示す（例: 「動詞を検索…」「行く、食べる など」）。
- 検索ボックスは常に画面最上部に固定。スクロールで隠れてはならない。
- オートフォーカス: ページ/ウィジェットを開いた瞬間にカーソルが入る設計が理想（400px タイルでは特に有効）。
- クリアボタン（×）を入力後に表示し、ワンクリックでリセットできるようにする。

**インクリメンタルサーチの実装上の注意**
日本語 IME 使用時、変換確定前の文字（未確定文字列）でも onChange イベントが発火し、誤ったタイミングでフィルタが走る問題がある。`compositionstart` / `compositionend` イベントを監視し、未確定文字列では検索を発火させないことが必須。React では `onCompositionStart` / `onCompositionEnd` を使って `isComposing` フラグを管理する。

**結果リスト**

- リストは固定高さのコンテナに収め、overflow-y: auto でスクロールさせる。
- 縦スクロールバーは OS 標準のもの（またはそれに近いもの）を使用する（NN/g 推奨）。
- 行の高さは 40〜48px が読みやすさと件数のバランスの目安。
- 絵文字ピッカー（368px固定高）などのリファレンス実装では「上部: 検索ボックス / 下部: 固定高さのスクロール可能ビューポート」の 2 ゾーン構成が標準。

**コピーアクション**

- 各行にコピーボタンを配置し、ホバー時に表示するか（クリーンな外観）、常時表示するか（発見しやすさ）を選択する。タスク完了速度最優先ならば常時表示が推奨される。
- コピー完了後に「コピーしました」ツールチップまたはボタンラベルの瞬間的変化（コピー → ✓ → 1秒後にリセット）でフィードバックを返す。アニメーション時間は 150〜300ms が適切。
- 何をコピーするかを明示する（「尊敬語をコピー」「謙譲語をコピー」のように対象を示す）か、行全体（普通語＋尊敬語＋謙譲語＋丁寧語）をセット形式でコピーするかを設計段階で決める。

**カテゴリフィルタ**

- 検索ボックスとカテゴリフィルタを組み合わせて使えるようにする（AND 条件）。
- アクティブフィルタを視覚的に区別する（背景色変化）。
- フィルタ変更時に件数をリアルタイム更新する（例: 「12件」）。

### 2.2 狭いスペースでのフィルタ UI の工夫

400px 幅のタイルではカテゴリフィルタボタンを横並びに 4 個以上置くと文字が折り返す可能性がある。対策として:

- ボタンラベルを短縮する（「基本」「ビジネス」「接客」など）。
- または垂直方向のドロップダウンに統合し、検索ボックスのみを常時表示する（1 行節約）。

## 3. 狭い高さ（400px）での検索結果リスト設計

### 3.1 レイアウト面積の試算

| ゾーン                              | 高さの目安                      |
| ----------------------------------- | ------------------------------- |
| タイルヘッダ（ツール名 + アイコン） | 36〜44px                        |
| 検索ボックス                        | 40px                            |
| カテゴリフィルタボタン行            | 32〜36px                        |
| 件数表示テキスト行                  | 20px                            |
| テーブルヘッダ行                    | 32px                            |
| **結果リスト可用領域**              | **約 128〜160px**               |
| コピーボタン等フッタ                | 0〜32px（行内に収める場合は 0） |

160px の可用高で 40px/行なら 4 行、48px/行なら 3 行しか表示できない。スクロールは必然だが、**最初の 3〜4 行で典型的なクエリの答えが見えることが重要**。

### 3.2 件数表示

- 検索後に「N件の動詞」形式でリスト上部に件数を表示する（現行 yolos.net でも実装済み）。
- 件数はフィルタ変更と同時に更新する。
- 「60件中 12件」のような全体件数との比較表示は、絞り込みの実感を与えるが、狭い UI ではノイズになる可能性あり。「12件」のみで十分。

### 3.3 スクロール

- overflow-y: auto を使い、コンテナ高さを明示的に指定（例: height: calc(400px - 180px)）する。
- 長すぎるリストのインジケータとして、コンテナ下端に軽いシャドウグラデーション（fade-out）を置くと「もっとある」を示唆できる。
- 無限スクロールや仮想スクロールは、60件程度のデータセットでは不要。シンプルな overflow スクロールで十分。

### 3.4 ハイライト

- マッチした文字列を `<mark>` タグまたは `<strong>` + カラー強調で示す（Algolia のベストプラクティス）。
- ハイライトは「なぜこの行がヒットしたか」をユーザーに伝えるための重要な手がかりであり、省略しない。
- 検索ボックスが空のときはハイライトなし（全件表示時はハイライト不要）。
- 「普通語」列のマッチ → 普通語をハイライト、「尊敬語」列のマッチ → 尊敬語をハイライト、という列横断ハイライトが理想（ユーザーが「敬語側のキーワード」で逆引きする場合への対応）。

### 3.5 テーブルレイアウト上の注意

尊敬語・謙譲語が長いフレーズになる語（例:「来る→いらっしゃる・おいでになる・お見えになる」）は一行に収まらず折り返す。400px 幅に 4 列テーブルを押し込むと各列が 80〜100px 程度になり読みにくくなる。対策:

**案 A: 2 列表示（普通語 + 選択した種別）**
タブ切り替えや切り替えボタンで「尊敬語」「謙譲語」「丁寧語」を選び、「普通語 | 選択した種別」の 2 列で表示する。コピーは 2 タップになるが、各セルの可読性が上がる。

**案 B: カード形式（1 件 1 カード）**
各動詞を縦積みカードにし、カード内に「尊敬語: …」「謙譲語: …」「丁寧語: …」を縦並びで表示。スクロールは増えるが情報密度が高く、コピーボタンをカード単位で配置しやすい。

**案 C: 4 列テーブルをスクロール可能に**
横スクロール（overflow-x: auto）を許容し、4 列すべてを維持する。PC では列幅が確保できるが、モバイルでは操作が煩雑になる。現行デスクトップページの移植として最も低コストだが、400px 幅タイルでは可読性が下がる。

**推奨: 案 A（2 列 + タブ切り替え）**。理由は、ターゲット利用者は「この動詞の尊敬語だけ知りたい」ケースが多く、全種別同時表示より「必要な種別を選んで確認してコピー」の動線の方がタスク完了が速い。

### 3.6 避けるべきアンチパターン

| アンチパターン                                     | 問題                               |
| -------------------------------------------------- | ---------------------------------- |
| 検索中にスクロール位置がトップに戻らない           | フィルタ後の先頭行が見えなくなる   |
| フィルタ変更時にアニメーションが長い（>300ms）     | 操作と結果の間にラグを感じる       |
| 行内テキストの折り返しで行高が不規則               | スクロール時に視線が追えなくなる   |
| コピーボタンがホバーでのみ表示                     | タッチデバイスで機能を発見できない |
| 件数表示なし                                       | どれくらい絞り込まれたか分からない |
| 結果 0 件時に何も表示しない                        | ユーザーがバグと思って離脱する     |
| テーブルヘッダを固定しない（スクロール時に消える） | 列の意味が分からなくなる           |

## 4. 結果ゼロ時（エンプティステート）の設計

- 「見つかりませんでした」のみの表示は「コンバージョンブロッカー」（Eleken/UXWorld より）。
- 短いメッセージ例（2 行以内）: 「「○○」に一致する動詞はありません。別のキーワードを試してください。」
- 「フィルタをリセット」リンクを添える（カテゴリフィルタが原因の場合）。
- 400px タイルでは、大きなイラストや絵文字は使わない。シンプルなテキストメッセージ + アクションリンクで十分。
- ウィジェット内の高さを変えない（エンプティ状態でも通常状態と同じ高さを保つ: Carbon Design System の原則）。

## 5. 敬語ツール特有の信頼性配慮

### 5.1 誤用リスクの質的重要性

敬語の誤用は業務・社会的信頼に直結する。yolos.net の憲法「人を悲しませない」に照らすと、誤った敬語を提示することは直接的に利用者を困らせる行為にあたる。したがって:

- データの正確性を最優先とし、AI 生成データではなく人間が確認・出典明示したデータを使う。
- 「参照型」早見表の場合、データソースを明示する義務がある。

### 5.2 文化庁の公式分類を基準にする

2007年の文化庁文化審議会「敬語の指針」（答申）が現行の公式基準。3分類（尊敬語・謙譲語・丁寧語）から 5分類（尊敬語・謙譲語 I・謙譲語 II（丁重語）・丁寧語・美化語）に改定されている。

- ただし 5 分類はビジネス実務でほぼ使われていないため、早見表としては「尊敬語・謙譲語・丁寧語」の 3 分類を UI に表示し、脚注に「文化庁指針（2007年）の5分類を簡略表示」と明示するアプローチが実用的。

### 5.3 注意喚起の最小限化

以下の注意をツールの UI に含めることを推奨するが、いずれも「最小限かつ常設しない」形で実装する。

**二重敬語の注意**

- 「おっしゃられる」（尊敬語の重複）や「ご利用させていただく」（謙譲語の重複）は典型的な二重敬語。
- 早見表の各行に「よくある誤用」コラムを展開表示できるようにする（行クリックで展開）。常に表示するとノイズになる。
- 既に現行の yolos.net に「よくある間違い」タブがあるため、その機能をタイル UI にも引き継ぐ。

**「ら抜き言葉」の注意**

- 参照型早見表では「ら抜き」は主要な問題にならないが（提示する敬語表現にら抜きを含まなければよい）、将来的に入力テキストのチェック機能を追加した場合の考慮として記録しておく。

**謙譲語を使う主語の注意**

- 謙譲語は「自分側の行為をへりくだる」表現。「部長が参ります」は誤用（目上に謙譲語を使うのは誤り）。
- 早見表の謙譲語列の脚注に「自分・自社側の行為に使います」を 1 行添える。

**「させていただく」の過剰使用**

- 文化庁が「適切な場面に限定すること」と指針で言及。early warning として脚注 1 行程度が適切。

### 5.4 出典表示の実装例（最小限）

```
（例）このデータは文化庁「敬語の指針」（2007年）および
標準的なビジネス日本語文献を参照し、編集しています。
個々の使用場面に応じて確認をお勧めします。
```

タイル UI では文字が読めない最小フォント（10〜11px）でフッタに 1 行添えるか、「?」アイコンのツールチップに収める。本文 UI の面積を出典説明に消費しない。

### 5.5 AI 生成データを使わないことの明示

so-zou.jp や Toolpods のような AI 変換ツールと差別化するため、「AI が変換したのではなく、人間が確認した固定データです」という信頼性の源泉を一言添える。ただし UI 上は簡潔に（例: 「編集済みデータ」バッジまたは説明 1 行のみ）。

## 6. 設計への直接適用推奨事項

### やること（Must）

1. **検索ボックスを最上部に固定し、ウィジェット展開時にオートフォーカス**する。プレースホルダに具体例「行く、食べる など」を入れる。
2. **IME 未確定中はインクリメンタルサーチを発火させない**（compositionstart/compositionend 対応）。
3. **結果リストに固定高さを与え、overflow-y: auto でスクロール**させる。テーブルヘッダはスクロール中も固定表示（sticky）。
4. **各行のマッチ文字列をハイライト**する（`<mark>` タグ推奨）。
5. **コピーボタンを常時表示**し、クリック後 1 秒間「コピー済み」フィードバックを返す。
6. **件数を動的に表示**する（「N 件」）。
7. **結果 0 件時に具体的メッセージ**と「絞り込みを解除」リンクを表示する。
8. **謙譲語列の脚注**に「自分・自社側の行為に使います」を 1 行添える。
9. **出典を最小スペースで表示**する（フッタ 10px またはツールチップ）。
10. **カテゴリフィルタのラベルを短縮**し、400px 幅で折り返しを防ぐ。

### やること（Recommended）

11. 2 列表示（普通語 + タブ切り替えで尊敬語/謙譲語/丁寧語）を採用し、各列の可読性を確保する。
12. 行クリックで「よくある誤用」の展開表示を提供する（常設しない）。
13. リスト下端にフェードアウト（box-shadow または gradient オーバーレイ）でスクロール可能を示唆する。
14. 検索クリアボタン（×）を入力後に表示する。

### 避けること（Must Not）

- ホバー限定のコピーボタン（タッチ非対応）。
- テーブルヘッダなし（スクロール後に列の意味が不明になる）。
- 結果 0 件の無言（空白のみ）。
- 300ms を超えるフィルタ更新遅延。
- 4 列テーブルを 400px 幅に詰め込む（各列 80〜100px は読みにくい）。
- AI 生成データの無確認掲載（誤った敬語を提示するリスク）。
- UI 面積の多くを出典・注意書きに割く（ツールの価値が見えなくなる）。

## 付記: 調査で確認できなかった事項

- 日本語の敬語参照に特化した検索機能付き小型ウィジェットの事例が、国内でほぼ存在しない（早見表は静的コンテンツか AI 変換ツールの 2 択）。yolos.net の現行実装（インクリメンタルサーチ + カテゴリフィルタ + テーブル）はこのカテゴリでほぼ唯一のアプローチであり、タイル UI への移植により差別化優位性を持てる可能性が高い。
- 「1 行クリックで謙譲語のみをコピー」対「行全体をフォーマット付きでコピー」のどちらが有効かは、利用者テストなしに確定できない。最初は「尊敬語をコピー / 謙譲語をコピー / 丁寧語をコピー」の個別ボタンを提供し、GA イベントで利用頻度を計測して最適化することを推奨する。
