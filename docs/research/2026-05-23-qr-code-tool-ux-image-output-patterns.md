# QR コード生成ツール UI/UX 調査

## 1. QR コード生成ツールの UI パターン（先行サイト調査）

### 1.1 調査サイト一覧と UI 構造比較

#### QRCG（qr-code-generator.com） — Bitly 傘下

**観察事実**（2026-05-23 Playwright 実機確認）:

- ヘッダー直下に「WEBSITE / DIGITAL BUSINESS CARD / TEXT / WIFI / PDF / APP STORE」の種別タブが横並びフラットタブとして 6 種表示（デスクトップ幅）
- 入力エリアは左 45%、QR プレビューは右 55% の左右 2 カラム分割。プレビューは常時表示で、入力が空のときはグレーアウトした QR 画像のプレースホルダーを表示する
- 「Your QR Code will be generated automatically」という文言がフォーム下部に表示され、リアルタイム生成であることをユーザーに事前通知している
- ダウンロードは「DOWNLOAD JPG」ボタン（▼矢印付きドロップダウン）で PNG / SVG / EPS は有料サインアップ要求
- DESIGN タブに「Frame / Logo / Color & Shape」の 3 サブタブ。フレーム選択 UI は小サムネイルのグリッド表示
- エラー訂正レベルの UI は非表示（PRO 機能として高エラー訂正レベルのみ有料で提供）
- 「Scan tracking」のトグルスイッチが入力エリア直下に表示される（オン/オフ切替）
- サインアップ後の動的 QR コード管理に導線が集中しており、無料ユーザーへの摩擦が多い

**設計指針**: 種別タブ + 左右 2 カラム分割は QR 系で最もポピュラーなレイアウトパターン。ただし無料ユーザーへの強い有料化誘導は「日常の傍にある道具」コンセプトと相反する。yolos.net は登録不要・全機能無料を差別化軸にすべき。

**アンチパターン**: 無料でダウンロードできる形式を JPG のみに制限し PNG/SVG に課金壁を設ける。QR コードに JPG は非推奨（圧縮ノイズで読み取り率が低下する）にもかかわらず無料提供形式として採用している点も問題。

#### QRCode Monkey（qrcode-monkey.com）

**観察事実**（2026-05-23 Playwright 実機確認）:

- ヘッダー直下に「URL / TEXT / EMAIL / PHONE / SMS / VCARD / MECARD / LOCATION / FACEBOOK / TWITTER / YOUTUBE / ...」の水平スクロールタブ（10 種以上）
- 左側に入力フォーム（幅 45%）、右側に QR プレビュー（幅 55%）の左右 2 カラム。プレビューは page load 時から表示済みでリアルタイム更新
- 入力フォーム下に「SET COLORS（折りたたみ）」「ADD LOGO IMAGE（折りたたみ）」「CUSTOMIZE DESIGN（折りたたみ）」の 3 アコーディオン
- 品質スライダー（Low Quality ←→ High Quality）がプレビュー直下に表示。「1000 x 1000 Px」の現在値をスライダー中央に表示
- ダウンロードは「Create QR Code」ボタン（緑）→「Download PNG」（青）の 2 段階操作。さらに下に「.SVG」「.PDF*」「.EPS*」のアウトラインボタンが並ぶ（\* = ログイン必要）
- 誤り訂正レベルは非表示（「CUSTOMIZE DESIGN」内に隠れていると思われるが Surface には出ない）
- 右カラム下部に「QR Code Templates」ボタン（フッター固定）

**設計指針**: 「Create QR Code」→「Download PNG」の 2 段階ボタンは「意図的に確認フローを踏ませる」設計。リアルタイムプレビューしながら設定し、準備ができたら「作成する」を押すという UX は来訪者に「完成品を作った」感を与える点で優れている。品質スライダーは直感的。

**アンチパターン**: タブが 10 種以上あり左端（URL）しか実質的に使われないと推測される場合、情報過負荷になる。水平スクロールタブはモバイルで選択肢の存在に気づかれにくい。

#### QRのススメ（qr.quel.jp）

**観察事実**（2026-05-23 Playwright 実機確認）:

- トップページは「URLをQRコードにします」という単一の input + 「作成する」ボタンのみの超シンプル構成
- 用途別ページへのカード型リンク（URLをQRコードに / メール作成用QR / おまとめQR）が下部に並ぶ
- 個別用途ページ（/url.php）では、左側に「データ入力」カード（URL input + 800文字制限注記）、右側に「使いみち」の説明カードが横並び
- 「作成する」ボタン押下後に別エリアに QR 画像が表示される（リアルタイム生成なし）
- 20 年以上の運営実績を強調。「登録不要で商用利用 OK」を明示
- エラー訂正レベル: 非表示（デフォルト固定）
- ダウンロード: ボタン生成後に画像として表示され、右クリック保存を案内（明示的なダウンロードボタンなし）

**設計指針**: 用途ページへの分離は SEO 最適化に有効。「URLだけ」「メールだけ」の単機能ページが検索クエリにマッチしやすい。来訪者が何をしたいかを明確にしてから入力させる設計は初心者に優しい。

**アンチパターン**: 明示的なダウンロードボタンがなく右クリック保存を案内するのは、モバイルユーザーにとって著しく不便。結果表示後に何もアクションがない UX は離脱要因になりうる。

#### CMAN（cman.jp/QRcode/）

**観察事実**（2026-05-23 Playwright 実機確認）:

- タブ構成: 「自由入力 / URL用QR / メールQR / 地図QR / 作成済一覧」（5 種）
- 入力エリアは縦並びで全幅 textarea、オプションが下部に並ぶ（横 2 カラムはなし）
- オプション UI が充実: セルサイズ（セレクトボックス「3倍 推奨」等）/ セル数（自動判別 推奨）/ **誤り訂正（セレクトボックス「レベルH 30%復元能力 推奨」等）**/ セル色（カラーコード直打ち）/ 囲み文字（ドロップダウン）/ 画像形式（PNG/GIF/JPEG）
- 誤り訂正レベルをセレクトボックスで全 4 段階 + 説明付きで表示。各選択肢に「推奨」ラベルをつけて判断を補助
- 生成は「作成する」ボタン押下（リアルタイム非対応）
- デザイン的には機能的な HTML フォーム感が強く、視覚的洗練度は低い
- 「58,577,000 回以上」の作成実績カウンターをヘッダーに表示（信頼醸成）

**設計指針**: 誤り訂正レベルをセレクトボックス + 説明文で表示する CMAN のアプローチは、上級ユーザーへの親切さと初心者向け「推奨」ラベルを両立する良い例。yolos.net 詳細ページでは参考にできる。

**アンチパターン**: 縦並びレイアウトのため、生成ボタンを押した後に下にスクロールしないと結果が見えない可能性がある。QR プレビューの常時表示がないため、設定変更の即時フィードバックがない。

#### goqr.me

**観察事実**（2026-05-23 Playwright 実機確認）:

- 3 カラムレイアウト: 左カラムに「1. Type」（アイコングリッドで種別選択）/ 中央カラムに「2. Contents」（テキスト入力 + 文字数カウンター）/ 右カラムに「3. Live preview」（QR プレビュー常時表示）
- 種別選択はアイコン 9 種（Web URL / Plain Text / Contact data / Phone number / Location / Facebook / Twitter / PDF / 鍵アイコン）をクリックして選ぶ。アイコングリッドで Type カラムに収まっており視覚的にわかりやすい
- リアルタイム生成（入力と同時に右カラムのプレビューが更新される）
- ダウンロードボタンは Live preview カラムには見当たらず、スクロール後に別エリアに表示（2026-05-23 確認時）
- 「0 characters」のリアルタイム文字数カウンターを入力エリア直下に表示
- 「Your QR code data is encrypted during transmission (TLS/SSL) and not stored.」というプライバシー注記をフォーム下に表示（信頼醸成）
- ページ下部に「Your QR code on:」としてマグカップ、Tシャツ、名刺、バッグへの印刷 mockup 画像を Zazzle 広告として表示（広告過剰の要注意ゾーン）

**設計指針**: 3 カラムで Type / Contents / Preview を役割分担した構成は来訪者の思考ステップと一致する。「プライバシーは SSL 通信・サーバー非保存」の宣言は来訪者の不安を先回りして解消する好例。yolos.net もブラウザのみで完結することを FAQ で明示しているが、ツールページでもインライン表示すべきか検討余地あり。

**アンチパターン**: ダウンロードボタンがファーストビューになく、スクロールが必要。グッズ広告がコンテンツより目立つ構造は来訪者体験を損なう。

#### Kaywa（qrcode.kaywa.com）

**観察事実**（2026-05-23 Playwright 実機確認）:

- ラジオボタン種別選択: 「URL / Contact / Coupon / More ▼」（3 種 + ドロップダウン）
- 入力エリアは中央幅広の単一テキストボックス、右側に QR プレビュー（薄いグレー背景カードに大きく表示）

### 1.2 UI パターン 総括

**入力フィールドの形状**:

- URL 専用 input（最多）: QRCG / QRのススメ / Kaywa / qrcode-maker.jp
- 種別タブ + 種別別フォーム: QRCode Monkey / goqr.me / QRCG / qrcode-maker.jp（タブ + 専用フォーム）
- 単一 textarea（自由入力型）: CMAN（「自由入力」タブ時）

**結果プレビューの配置**:

- 左右 2 カラム（入力左・プレビュー右）: QRCG / QRCode Monkey / Kaywa
- 3 カラム（Type / Contents / Preview）: goqr.me
- 縦並び（入力上・プレビュー下）: CMAN / qrcode-maker.jp / QRのススメ（作成後表示）
- リアルタイム常時表示: QRCG / QRCode Monkey / goqr.me / Kaywa（プレビューのみ、入力なしでもプレースホルダー表示）

**ダウンロード形式と配置**:

- PNG のみ無料 + SVG/EPS 有料: QRCG
- PNG デフォルト + SVG/PDF/EPS（無料 or 要ログイン）を別ボタンで提供: QRCode Monkey
- PNG・SVG・PDF 全部無料: qrcode-maker.jp
- 明示的 DL ボタンなし（右クリック保存案内）: QRのススメ
- SVG が download の最優先形式（ベストプラクティス）として推奨されている

**誤り訂正レベルの UI**:

- セレクトボックス + 説明付き（推奨ラベル）: CMAN（最も親切）
- 非表示（固定または隠蔽）: QRCG（PRO 機能）/ QRCode Monkey / Kaywa / QRのススメ / goqr.me
- yolos.net 既存実装: セレクトボックス（L/M/Q/H + 説明付き）→ CMAN と同等で良い設計

**リアルタイム vs ボタン押下**:

- リアルタイム生成が主流（QRCG / QRCode Monkey / goqr.me）
- ボタン押下型（QRのススメ / CMAN / qrcode-maker.jp）
- 2 段階（リアルタイムプレビュー → 「Create QR Code」ボタンで確定）: QRCode Monkey

**装飾機能**:

- ロゴ埋め込み: QRCode Monkey（アコーディオン内）/ QRCG（DESIGN タブ）/ qrcode-maker.jp
- 色変更: ほぼ全サイト
- 角丸・フレーム: QRCG / QRCode Monkey / qrcode-maker.jp
- yolos.net の差別化ポイント: 装飾なし・シンプル・高速 を「日常の傍にある道具」として軸にする

## 2. 画像/SVG 出力型ツールの a11y / UX ガイドライン

### 2.1 QR コード画像の alt 属性 / aria-label

**観察事実（WCAG / DAISY 出版アクセシビリティ KB 確認）**:

DAISY の出版アクセシビリティ KB（kb.daisy.org）では、QR コード画像のアクセシビリティについて以下のパターンを推奨している:

```html
<a href="https://publisher.example.com/promo">
  <img src="img/qr-promo.jpg" aria-details="qr-desc" />
</a>
<details id="qr-desc">
  <summary>詳細</summary>
  <p>
    QR コードが読み取れない場合は以下の URL を入力: publisher.example.com/promo
  </p>
</details>
```

- QR コード画像は「QR コードであること」と「リンク先または目的」を alt に含めるべき
- 例: `alt="yolos.net へのリンク QR コード"` または `alt="入力テキストの QR コード: [テキスト内容]"`
- QRTRAC ガイドライン: 「QR code linking to the pricing page at example.com/pricing」のように、リンク先を明示するスタイルが推奨
- WCAG 1.1.1（非テキストコンテンツ）: 画像が機能を持つ場合（スキャン可能）は alt が必須
- WCAG 1.4.11（非テキストコントラスト）: QR コードのモジュール（黒）と背景（白）のコントラスト比は 3:1 以上が必要

**yolos.net への適用指針**:

既存実装では `role="img"` と `aria-label="Generated QR code"` のみ（英語固定）。以下に改善する:

- `aria-label` を動的化: `aria-label={input ? `「${input}」の QR コード` : "QR コード（未生成）"}`
- 入力内容が長い場合は適切に truncate または「QR コード（URL: ...）」形式に
- 生成前のプレースホルダーエリアには `aria-label="QR コードプレビューエリア（テキスト入力後に生成されます）"` を付与

### 2.2 ダウンロードボタンのラベル

**観察事実（WCAG / UX アクセシビリティガイドライン確認）**:

- 「ダウンロード」のみは不十分。ファイル形式を含めるべき
- 推奨: 「PNG をダウンロード」「SVG をダウンロード」のように形式名を明示
- 複数形式を提供する場合は別々のボタン or ドロップダウンで形式を選択させる
- ドロップダウンにする場合は aria-haspopup="listbox" または aria-expanded などの適切な ARIA ロールが必要
- 既存実装: 「PNG形式でダウンロード」→ これは良い実装。形式名が明示されている

**yolos.net への適用指針**:

- 既存の「PNG形式でダウンロード」は WCAG 準拠として適切
- cycle-207 で SVG ダウンロードを追加する場合は「SVGをダウンロード」を別ボタンで提供
- ボタン内に `type="button"` を明示（既存実装で対応済み）

### 2.3 リアルタイム生成中のローディング表示の閾値

**観察事実（Nielsen Norman Group 応答時間ガイドライン確認）**:

Jakob Nielsen の 3 つの応答時間閾値（1993 年、現在も有効）:

- **0.1 秒（100ms）**: 操作が即時と感じられる。ローディング表示不要
- **1 秒**: 思考の流れが途切れるがユーザーは意識できる。通常のフィードバックは不要だが、何らかのシステム応答があることを示すと良い
- **10 秒**: ユーザーの注意が散漫になる限界。プログレスバーや「処理中...」の明示が必要

QR コード生成処理のパフォーマンス:

- ブラウザ側の qrcode-generator ライブラリによる SVG 生成: 通常 < 10ms（256 バイト以下の URL）
- 既存実装のハッシュ生成（crypto.subtle.digest）は < 1ms / < 1KB で「loading state なし」が明示されている

**yolos.net への適用指針**:

- qr-code ツールのリアルタイム生成（onChange トリガー）では loading state は不要
- SVG 生成は同期処理で即時完了（< 10ms）のため、Nielsen の「0.1秒以内」カテゴリに該当
- エラー発生時（入力が長すぎる等）は `role="alert"` で即時通知（既存実装で対応済み）
- ARIA の観点: 生成完了を `aria-live="polite"` の status ロールで通知すると良い
  - 既存実装: `dangerouslySetInnerHTML` で SVG を表示しているだけであり、生成完了の支援技術への通知なし
  - 改善案: 結果エリアに `role="status"` と `aria-live="polite"` を追加

### 2.4 プレビューエリアの最小サイズ

**観察事実（QR コードサイズガイドライン確認）**:

物理サイズの推奨値:

- 最小: 2cm × 2cm（ほとんどのデバイスで読み取り可能）
- 推奨最小: 2.5cm × 2.5cm（高い信頼性）
- ルール: 走査距離の 1/10 がサイズの最小値（20cm 先から読む場合 2cm）

デジタル表示の画素数:

- 絶対最小: 76px × 76px（最新デバイスのみ）
- 推奨最小: 150px × 150px（旧来デバイスも含む信頼できる最小値）
- 高品質表示推奨: 200px × 200px 以上

既存実装:

- `qr.createSvgTag(4, 4)` = セルサイズ 4px、マージン 4px のモジュールで SVG 生成
- SVG は `max-width: 100%` で表示。詳細ページ幅（最大 1200px）では十分な表示サイズになる
- `qr.createDataURL(8, 4)` = PNG 用は セル 8px で生成 → 最低品質 40 モジュール QR で 320px 以上

**タイル UI での最小サイズ検討**:

タイルサイズ cols=3 rows=2 = 400px × 264px。このサイズ内に QR プレビューを収める場合:

- 入力エリア（textarea）と QR プレビューを縦並びにすると、プレビューに確保できる高さは约 120-140px 程度
- 150px 推奨最小を縦に確保するとタイルの大半が QR 画像で占有される
- 解決策 A: プレビューをタイル内に配置し cols=3 rows=3 に拡大（400px × 400px）
- 解決策 B: タイル内では QR を小サムネイル（64-100px 程度）として表示し、詳細ページでの印刷・ダウンロードに誘導する
- 解決策 C: タイル内は入力のみ（textarea + 生成ボタン）で、クリックで詳細ページに遷移させる

## 3. タイル UI（ダッシュボードカード）における画像/SVG 出力の扱い

### 3.1 cols=3 rows=2（400px × 264px）での制約分析

**既存タイルとの根本的な差分**:

cycle-200〜206 の 7 タイルはすべて「textarea（固定高） → 結果テキスト（膨張またはゼロ）」の構造であり、AP-P21 では「入力側 = flexShrink:0 固定 / 出力側 = flex:1 可変」で対応できた。

qr-code は「textarea → SVG 画像」であり、画像は以下の点で従来と異なる:

1. **最小読み取りサイズ制約（150px 以上推奨）**: テキスト出力にはサイズ制約がなかったが、QR 画像には物理的な最小サイズがある
2. **アスペクト比固定**: QR コードは正方形（1:1）。テキスト出力は縦方向に自由に伸縮できるが、QR 画像は幅=高さの制約がある
3. **白余白（Quiet Zone）必須**: QR コードの周囲に最低 4 モジュール分の白余白が必要。余白込みで正方形サイズを確保する必要がある

**実際のサイズ計算**:

cols=3 rows=2 = 400px × 264px のタイル内での配分:

- タイル内部パディング: 8px × 2 = 16px（上下）→ 有効高さ 248px
- 種別タブ（URL のみで省略可）: 0px
- textarea（rows=2, 約 52px）
- gap: 8px
- 生成ボタン（約 36px）
- gap: 8px
- QR プレビュー: 248 - 52 - 8 - 36 - 8 = **144px**
- 「詳細ページで開く」リンク（約 18px）

→ QR プレビューに 144px 確保可能。150px の推奨最小をわずかに下回るが、表示はブラウザ上での参照用途（スマホでスキャンするにはこれ以上のサイズが必要）であるため、タイル内では「プレビューとして表示し、スキャンは詳細ページで」という設計割り切りができる。

**注意**: ボタン「QRコード生成」を省略しリアルタイム生成に切り替えることで、36px + 8px = 44px を節約できる → 188px のプレビューエリアが確保可能。

### 3.2 リアルタイム生成 vs ボタン押下（タイル文脈）

**観察事実**:

- リアルタイム生成が現代の主流（QRCG / goqr.me / QRCode Monkey プレビュー部分）
- qrcode-generator ライブラリの SVG 生成は同期処理 < 10ms で、入力 onChange ごとに実行しても体感できる遅延なし
- 既存実装（legacy）はボタン押下型。詳細ページではどちらでも許容できる
- タイル文脈では「入力してすぐ結果が見えること」がタイルの価値そのもの

**設計指針**:

- **タイル UI はリアルタイム生成が必須**: ボタン押下型では「入力 → ボタンクリック → 結果確認」という 3 ステップが必要で、タイルの即応性・コンパクト性の良さが失われる
- リアルタイム生成に切り替えると QRコード生成ボタンが不要になり、タイル内のスペースが節約できる
- ただし `onChange` 毎に生成するため debounce（例: 300ms）を検討すること

**詳細ページ**:

- 既存のボタン押下型をリアルタイムに変更する方が来訪者体験が向上する
- onChange + debounce でリアルタイム化 → 「生成ボタン」をなくす or「コピー」「ダウンロード」に特化させる

### 3.3 ダウンロードボタンのタイル内配置

**観察事実**:

- 既存タイル（hash-generator / fullwidth-converter 等）にはコピーボタンが詳細ページ Component に存在するが、タイル内にはない。「詳細ページで開く」リンクのみ
- QR コードの場合、ダウンロードはテキストのコピーよりも高い頻度で使われる最終アクション（来訪者が QR コードを作る目的は必ずダウンロードまたはスクリーンショット）
- しかしタイルサイズ 400px × 264px では、ダウンロードボタンを追加する余裕は限られる

**設計指針（3 案比較、AP-P17 対応）**:

**案 A**: タイル内にダウンロードボタン（PNG）を配置する

- メリット: タイルで完結する最高の UX。詳細ページ遷移なしにダウンロードできる
- デメリット: プレビューエリアが圧縮される。タイル内が窮屈になる
- 実現性: QR プレビュー 144px（リアルタイム時 188px）の下にダウンロードボタン（36px）を追加 → 残り 108px（リアルタイム時 152px）。スペース的にギリギリ

**案 B**: タイル内はプレビューのみ、ダウンロードは詳細ページに任せる

- メリット: タイルがシンプルで QR プレビューに集中できる。詳細ページでは SVG も含む複数形式を提供可能
- デメリット: タイルで完結しない。「QR 作れたけどどこでダウンロードするの？」という混乱が起きやすい
- 適切な誘導: QR プレビューをクリックで詳細ページに遷移 or 「詳細ページで DL」リンクをわかりやすく表示

**案 C**: QR プレビューをクリックするとダウンロード（クリック = PNG DL）

- メリット: ボタン追加不要でタイルがシンプル。UX 的には「QR を見たらそれをクリックして保存」という自然な操作
- デメリット: クリックでダウンロードが起動するという挙動が不明瞭でアフォーダンスが弱い。アクセシビリティ上、ダウンロードのきっかけが画像クリックになるため aria-label での説明が必須

**推奨**: 案 A（プレビュー下にダウンロードボタン配置）を cols=3 rows=2 または必要なら cols=3 rows=3 で実装。理由: タイルで完結することが来訪者にとって最も価値が高い。qr-code は「使って保存して終わり」という性質が強く、詳細ページへの遷移を強いることは最終アクションへの摩擦になる。

### 3.4 誤り訂正レベルのタイル内扱い

**観察事実**:

- 先行サイトの大半は誤り訂正を「全 ON 固定」「非表示」で対処している
- QRCG は「PRO 機能」として高エラー訂正のみ提供（無料は M 固定相当）
- CMAN は明示して選べるが「推奨」ラベルで誘導している

**設計指針**:

タイル内では誤り訂正レベルは省略し **M 固定**とする。理由:

- タイルは「素早く確認・使う場」であり、詳細設定はタイルの責務外
- M レベル（15%）は実用上の最もバランスが良い選択（業界でも M が標準）
- 省略することでタイル内スペースを節約できる
- 詳細ページでは L/M/Q/H を引き続き提供する

## 4. 「日常の傍にある道具」コンセプトに最適な QR ユースケース

### 4.1 日本国内の QR コード用途分布

**観察事実（複数情報源を統合）**:

**用途 1: URL の QR 化（最大ボリューム）**

国内外全サイトでトップ種別として扱われている。Google Chrome の標準機能にも「このページの QR コードを作成」が組み込まれており、最も一般化した用途。「ホームページ URL をチラシに印刷する」「イベント参加 URL を配布する」「名刺に自社サイト URL を入れる」等の実需が広い。

**用途 2: Wi-Fi 接続情報の QR 化（急速に普及中）**

複数の日本語サイト・TP-Link 等のルーターメーカーが「Wi-Fi QR コード」の作り方記事を 2024〜2025 年に大量に公開している。来客時や民泊・ゲストハウスでのパスワード共有用途が明確で、「複雑なパスワードを入力なしで接続」という価値が高く理解されやすい。SSID + パスワード + 暗号化方式（WPA/WPA2/WEP）の 3 フィールド入力が必要。

**用途 3: LINE 友だち追加 URL の QR 化**

LINE のビジネスアカウント / 個人アカウントの友達追加 URL を QR コード化する需要。日本固有の高需要ユースケース（LINE が国内 SNS シェア首位のため）。技術的には URL の QR 化と同じだが、来訪者の検索クエリ「LINE QR コード 作成」「LINE 友だち追加 QR」として独立した需要として存在する。

**用途 4: vCard（連絡先）の QR 化**

名刺交換をスマートフォンで完結させる需要。vCard 形式 or MeCard 形式（日本では NTT DoCoMo が普及させた MeCard が主流の時期もあったが、現在は vCard に収束しつつある）。入力フィールドは多い（名前・電話・メール・会社名・URL 等）ため、UI の複雑さが最も高い用途。

**用途 5: テキスト・メッセージ QR 化**

短いテキストを QR コードに埋め込み、スキャンして読み取れるようにする用途。「手書きを避けたい補足メモ」「展示物の説明文」等。頻度は URL より低いが確実な需要がある。

**量的推定（検索ボリューム近似）**:

検索クエリ「QRコード 作り方」のうち:

- URL の QR 化: 約 50-60%（最大多数）
- Wi-Fi QR: 約 15-20%（近年急増）
- LINE QR: 約 10-15%（日本固有）
- vCard QR: 約 5-10%
- テキスト・その他: 約 5-10%

### 4.2 タイル UI で主役に据えるべき用途

**観察事実**:

qr.quel.jp のトップは「URLをQRコードにします」という単一 input が最初に表示されている。最大多数の来訪者ニーズに最初にヒットする設計。

**設計指針**:

- タイル主役: **URL 入力（デフォルト）**。種別切り替えタブをタイル内に入れると複雑化するため、URL 専用に特化してシンプルに
- 詳細ページ: URL / テキスト / Wi-Fi / vCard / LINE URL（= URL と同じ扱い）の 5 種に対応
- Wi-Fi はフィールドが複数必要（SSID / パスワード / 暗号化方式）のため詳細ページ専用とする

## 5. 既存実装の評価と cycle-207 課題整理

### 5.1 既存実装（legacy）の評価

`src/tools/qr-code/Component.tsx` の現状:

| 項目           | 現状                                            | 評価                                                |
| -------------- | ----------------------------------------------- | --------------------------------------------------- |
| 入力フィールド | textarea（単一、URL/テキスト兼用）              | 良: シンプル。要検討: URL 専用 input に変更すべきか |
| 生成方式       | ボタン押下型（「QRコード生成」ボタン）          | 要改善: 詳細ページでもリアルタイム化の価値あり      |
| プレビュー     | ボタン押下後に表示（常時表示なし）              | 要改善: 初期から空プレースホルダーを表示すべき      |
| エラー訂正     | セレクトボックス（L/M/Q/H + 説明付き）          | 良: CMAN と同等の良い UX                            |
| ダウンロード   | 「PNG形式でダウンロード」ボタン                 | 良: ラベルが明確。SVG ダウンロード追加余地あり      |
| a11y           | `role="img"` + `aria-label="Generated QR code"` | 要改善: 英語固定 / 動的化と日本語化が必要           |
| 種別対応       | URL/テキスト統合（textarea 単一）               | 検討余地: Wi-Fi / vCard は詳細ページのみで対応      |

### 5.2 AP-P21 適用の判断

qr-code タイルは「textarea → SVG 画像」であり、従来の「textarea → テキスト」とは SVG 画像の最小サイズ制約という点で差分がある。

AP-P21 の役割分担パターン:

- **入力側（textarea）**: flexShrink: 0（入力欄を潰さない）
- **出力側（SVG 画像エリア）**: flex: 1（残りスペースを占有）

ただし SVG は正方形 (1:1 アスペクト比) なので、flex: 1 で高さが確保されても、width が height を超えると QR が正方形を維持しながら縮小される。結果として 150px 推奨最小を下回る可能性がある。

**解決策の検討（AP-P17 準拠 3 案）**:

1. **案 A**: cols=3 rows=3 に拡大（400px × 400px）。QR プレビューに十分なサイズを確保する。タイルが縦に長くなるが、QR コードという性質上、視覚的な存在感が必要
2. **案 B**: cols=3 rows=2 のまま、QR プレビューを 140px 固定（flexShrink: 0 / 正方形）にし、タイル内でスキャン用途を想定しない（= プレビュー専用と割り切る）。詳細ページへのリンクでダウンロードに誘導
3. **案 C**: cols=3 rows=2 のまま、リアルタイム生成でボタンを省略し、確保できるプレビューサイズを最大化（約 188px）。ダウンロードボタンを追加するとさらに圧縮されるため詳細ページに委ねる

## cycle-207 計画に直接効く 5 つの結論

1. **タイル UI はリアルタイム生成を採用し、「QRコード生成」ボタンを削除する**
   既存の詳細ページ（ボタン押下型）とは別に、タイル専用コンポーネントではリアルタイム生成（onChange + debounce 300ms）を採用する。これによりボタン分（約 44px）をプレビューエリアに転用でき、cols=3 rows=2（400px×264px）内で約 180-190px のプレビューを確保できる見込み。詳細ページも同時にリアルタイム化することで来訪者体験が向上する。

2. **AP-P21 の役割分担パターンを適用するが「SVG の最小サイズ制約」を計画段階で言語化する**
   従来の「textarea → テキスト」では出力の最小サイズ制約がなかった。「textarea → SVG 画像（QR コード）」では 150px 推奨最小がある。cols=3 rows=2 のタイルで確保できるプレビュー高さが推奨値を下回る可能性をゼロベースで計算し、cols=3 rows=2 据え置き案 vs rows=3 拡大案を AP-P17 に従って 3 案比較してから決定すること（参考: rows=3 では 400px×400px = 十分なプレビューサイズが確保可能）。

3. **ダウンロードボタンはタイル内に配置し「PNG をダウンロード」ラベルを使用する**
   来訪者が QR コードを作る最終目的は「保存して使う」であり、ダウンロードをタイル外に追い出すと致命的な UX 欠損になる。スペースが許す限りダウンロードボタンをタイル内に配置する。詳細ページでは PNG に加えて SVG も提供し「SVGをダウンロード」を別ボタンで追加する（印刷用途で SVG が最適形式）。

4. **詳細ページで種別タブを追加する: 最低 URL / テキスト / Wi-Fi の 3 種**
   URL 入力が最大多数（約 50-60%）だが、Wi-Fi QR は急速に需要が拡大しており来訪者価値が高い。Wi-Fi QR は SSID / パスワード / 暗号化方式の 3 フィールドが必要で既存の textarea 単一では対応不可。詳細ページで「URL / テキスト / Wi-Fi」の 3 種タブを実装することを計画段階で検討する（vCard は LINE QR と共に後回し候補）。タイル内は URL 専用にシンプル化する。

5. **a11y: aria-label を動的化し、生成完了を role="status" で通知する**
   既存実装の `aria-label="Generated QR code"` は英語固定かつ入力内容を反映していない。`aria-label={input ? \`「${input.slice(0, 30)}」の QR コード\` : "QR コードプレビューエリア"}`のように動的化する。また SVG 生成完了を支援技術に通知するため、プレビューコンテナに`role="status"`と`aria-live="polite"` を追加する。これは hash-generator タイル（cycle-205）で実装済みのパターンと同じアプローチ。
