# character-personality 結果体験の一次接地（cycle-274 / B-545）

日付: 2026-06-30
担当: PM（cycle-274）
方法: 実サイト yolos.net を来訪者として接地。(1) 単独結果ページ（第三者向けシェアランディング）を take-screenshot で撮影、(2) Playwright で実際に12問を遊び、本人向けインライン結果（ResultCard）に到達して全ページ撮影。スクリーンショットは `tmp/screenshots/`（非コミット）。

## 接地のレンズ（確立知見・実体確認済み）

- cycle-137 line 33: シェアの動機は「自己開示の報酬性」。「自分のことを言い当てられた感覚」＝「これ自分だ!」体験がシェアを促す（Harvard PNAS 2012）。
- cycle-136 line 81 / cycle-142 line 58,92: キャラ口調の「あるある（behaviors）」は共感とシェア動機を生む最重要コンテンツ。
- cycle-142 line 36: 来訪者の多数派は友人のシェアリンクから来る。
- ターゲット `自分に似たキャラや自分の性格を知りたい人.yaml`: likes=「当たってる」「新しい一面」の手応え・読み物として面白い独自性・自己紹介ラベルになり SNS に貼れる。

要するに、結果体験が「これ自分だ!」を強く生むほど、満足・シェア・再訪につながる。本接地は、その自己認識の山場がどこで生まれ/冷めているかを指差すために行った。

## 2つの面（コード上の事実）

`src/app/(new)/play/[slug]/result/[resultId]/` 冒頭コメントが明示する通り、結果体験には2面ある:

- **本人向けインライン結果**（`src/play/quiz/_components/ResultCard.tsx`）: 診断を遊んだ本人が `/play/<slug>` 上で見る。description は**全文表示**（切り詰めなし）。「これ自分だ!→見せたい」が最も効く面。
- **第三者向け単独結果ページ**（`/play/<slug>/result/<id>`）: 友人のシェアリンク等から来た、まだ診断していない第三者が見る。description は `DescriptionExpander` で4行クランプ＋「続きを読む」。

## 指差せる発見

### Finding B（本サイクルで是正）: 「このキャラの成り立ち」に内部の英語元型名が露出している

本人向けインライン結果で、description（あなたへの一人称メッセージ）の直後に出る「このキャラの成り立ち」カード（`CharacterPersonalityContent.tsx:218` が `archetypeBreakdown` を直接描画）の本文が、来訪者に見せる日本語コピーの中で内部分類の英語名をそのまま使っている。

実例（炎の詩人 blazing-poet の成り立ち本文・実画面で確認）:

> 「**commander**の爆発的推進力に**dreamer**の柔らかい感性が融合した形。…」

- 英語元型名（commander / professor / dreamer / trickster / guardian / artist）は `character-personality.ts:20` のコードコメントにのみ定義される**内部分類**で、FAQ 等を含めユーザーには一切説明されない。
- 全24結果の archetypeBreakdown に体系的に及ぶ。是正前の値を git diff で全件確認した結果、**24/24 すべての archetypeBreakdown 本文が英語元型名を含んでいた**（易しい一例の一般化ではなく全件の構造的欠陥）。なお grep の素の行ヒット数は batch1/2/3 で 46/43/33 件あるが、これはコメント・id スラッグ・color/scoring キー等を含む全行数で、archetypeBreakdown プロースの件数（24件）とは別である。
- 来訪者は自分について読んでいる流れの真ん中で、意味の分からない英語ジャーゴンに当たる。「これ自分だ!」の没入が冷める。constitution Rule 4（あらゆる面で最高の品質）に対する明確な品質欠陥であり、読者の誰が見ても「平易な自己認識の日本語の方が良い」と同意できる自明な改善対象。
- 制約: batch テストが archetypeBreakdown を 80〜150 字に制約（`character-personality-results-batch1.test.ts:71-76`）。書き直しもこの範囲を守る。

### Finding A（本サイクル非対象・既存 B-544 / B-526 へ）: 結果到達のリビール（山場）が無い

12問を答え終えても、結果は診断トップの静的な説明文の下に静かにインライン描画されるだけで、到達感・お祝い・視覚的な山場が無い。2分投資した来訪者の感情のピーク（リビールの瞬間）が平板。

これは emotional value としては Finding B より大きいが、(1) 視覚的クライマックス化は設計方向への賭けで cycle-268〜271 が4度失敗した領域、(2) 既に B-544（§7 フルクライマックス化）と B-526（インライン A/B `quiz_result_visual_v1` の月次読み）で、A/B の枠組みを経て進める前提で管理されている。本サイクルの「読者として自明な改善のみ出荷」方針の外にあるため本サイクルでは触れず、その前提を再確認した事実として記録する。

### Finding C（本サイクル非対象・carryover）: 第三者向け単独ページが description を切り詰める

友人のシェアリンクから来た第三者（来訪者の多数派ルート）が着地する単独結果ページで、description が4行で切り詰められ「続きを読む」になる。第三者は自分の話を読んでいるわけではないため「これ自分だ!」への直接の影響は本人向けより小さく、また「ティーザー＋診断 CTA」という変換設計としての一定の妥当性もある。価値優先（来訪者に結果を最後まで楽しんでもらう）の観点では再考余地があるが、本サイクルの自明改善の枠では Finding B より優先度が低い。次サイクル以降で判断する。

## 本サイクルの結論

Finding B を是正する。全24結果の archetypeBreakdown を、英語元型名を除き、来訪者自身について平易な日本語で語る自己認識の文に書き直す（「二つの面が同時に働く」という洞察は保持し、各結果のトーンに合わせ、80〜150字を守る）。Finding A・C は本ドキュメントに記録し、A は既存 B-544/B-526 の前提として、C は次サイクル判断として申し送る。

# 【追記 cycle-275 / B-545 本丸の急所への接地】

日付: 2026-07-01
担当: PM（cycle-275）
方法: (1) GA/BigQuery 生データ接地（cycle-274 が欠いた接地）、(2) 実画面を来訪者として再接地（回答完了→結果表示の遷移を Playwright で観察）、(3) 競合の結果リビール手法の Web 調査。cycle-274 の Finding A（結果リビール欠如）を、データ・実画面・競合知見の3面で検証し、本丸 B-545 の「具体的な急所」として据え直す。

## 1. GA 生データ接地（analyze-bigquery、2026-06-01〜30）

- サイト規模: 30日で PV 518 / セッション 392（小規模。以下は方向性の把握用で統計的頑健性は無い）。
- 診断が価値の中心（確定）: /play が全PVの58%（298PV）、character-personality 単独で全PVの34%（176PV）、エンゲージ 142秒/セッションで全セクション最長。道具箱 /tools は 43PV。
- character-personality の漏斗: 到達148→開始114（77%）→完了108（**開始者の95%が完走**）。離脱の主因は「到達したが開始しない」段階（約23%）。完走者はインライン結果を 138秒/セッション読み込む（サイト最高水準）。
- **シェア枯死**: 30日でシェアはサイト全体2件、character-personality **1件**。108人が完走し138秒読んでも「見せたい」がほぼ起きていない。
- **前提の方向修正（重要）**: 「多数派は友人のシェアリンク（単独result URL）から来る」は**このデータでは不成立**。単独result URL は30日で2PV、social流入0、着地はほぼ100%がオーガニック検索（google/yahoo）→インライン結果。友人シェアのループは現在“死んでいる”。

## 2. 実画面接地（ローカル dev・実プレイ12問）

- **回答完了→結果表示に演出は無し（即時・静的）**。最終回答クリック直後のキャプチャと結果ファーストビューは**ピクセル完全同一**。ため/ローディング/アニメ/お祝いは皆無。
- **完走しても自動スクロールが起きない**。クリック直後 `window.scrollY`=0。来訪者の視界は**遊ぶ前と同じ**（サイトヘッダ→パンくず→ページ h1「あなたに似たキャラ診断」→説明文）のままで、結果本体はその下（結果ラベル top≈327px）に静かに現れる。**来訪者は自分の結果を見せられてすらいない**。
- **勲章 first-view が無い**: 結果ラベルは小さいグレー、タイプ名はほぼ黒テキスト（--fg）、結果カードは背景色なし、固有色も象徴も主役に立たない。皮肉なことに象徴（絵文字）は「他のキャラも見てみよう」の**他タイプにだけ**付き、来訪者自身の結果には無い。cycle-274 Finding A を実画面で再確認。
- スクリーンショット: `tmp/screenshots/cycle-275/`（非コミット）。

## 3. 競合の結果リビール手法（Web 調査・出典付き）

- **最重要の方向修正**: 多秒の「分析中…」ローディングは確立した診断サービス（16Personalities・Truity・Crystal）の標準では**ない**（即時結果が主流）。「ため」を推奨するのはマーケ系クイズファネルのガイド。投資すべき「ピーク」は偽ローディングではなく**結果ページ自体の演出**（タイプ名・固有色・象徴・タグラインがアイデンティティとして立ち上がる短い動き）。
- 勝ちパターンの first-view 階層: 「記号 → 固有名/アーキタイプ名 → 色・象徴 → スコア/軸」。**アーキタイプ名（覚えやすい固有名）が「これ自分だ！」の核**（16P「建築家」・Crystal「The Initiator」・mgram「#タグ性格」）。
- 勲章感: 命名＋色分け＋象徴。**常にポジティブ**に。
- 「人に見せたくなる一枚」: 1枚で自己完結＋OGP最適化＋整形済みワンタップ共有（ShindanMaker 実証：タイプ名＋固有ハッシュタグ＋URL、字数超過でも本文をカットしタグ/URLは死守）。
- 理論的裏付け: Peak-End Rule（体験はピークと終わりで記憶される＝リビールは最重要）、自己開示の共有動機（68%が「自分が何者かを伝えるため」共有）、Barnum効果（肯定的・「あなた専用」感で的中感）。「深い喜び vs 表層の喜び」＝演出は正確で価値ある結果の上に載せてこそ効く。
- アンチパターン（当サイト価値観と一致）: 偽ローディング/偽プログレス、メール・シェア強制ゲート、500ms超の重いアニメ・大きな zoom/parallax、色だけで情報伝達、過剰 confetti・けばけばしさ。a11y: `prefers-reduced-motion` では動きを削除でなく opacity フェード等の穏当な代替へ、色は必ずテキスト/形と併記（WCAG 1.4.1）。

## 4. 本丸の急所マップ（事実。設計提案ではない）

道具箱前提の抑制設計（austere＝道具・辞典・道具箱では純粋に保つべき基調）が、**診断のクライマックス（結果面）にもそのまま被さっている**結果、価値の中心コンテンツの一番おいしい瞬間で来訪者価値が損なわれている。指差せる毀損点:

- **毀損点1（注意の誘導欠如・機能欠陥）**: 完走しても視界が結果へ動かない。108人の完走者の一部は「自分の結果が出た」ことに視覚的に気づけない。抑制以前の機能欠陥。
- **毀損点2（リビールの瞬間欠如）**: 設問→結果が即時・静的差し替え。Peak-End で最重要の「終わり＝ピーク」が平板。2分の努力に対する報酬の実感が弱い。
- **毀損点3（勲章の欠如）**: 結果が結果自身の固有色・象徴を封じた無彩カード。「自分はこれだ！」のアイデンティティが視覚的に立たない。結果データは固有色・象徴を既に持つのに、道具用の抑制（絵文字なし・派手色なし・--accent統一）がそれを封じている。§7 は結果クライマックスで固有色・象徴を主役に立てることを明示的に許可しているが、実装は基調の抑制に留まっている。

これは B-545 本丸（道具箱前提の根幹に診断が §7 例外として継ぎ足された構造）が、来訪者価値として噴き出している一症状である。根幹そのもの（アーキテクチャ・デザインシステムを診断中心へ据え直すか）は本追記では結論しない（開いた問いとして継続）。本追記は「根幹の抑制が来訪者価値を実際に損ねている点」を事実で指差すマップである。
