### 1.2 AP-P21 役割分担（4 ケース計測の内容）

## 2. cycle-206 の textarea×2 双方向構造パターン（fullwidth-converter）

### 2.1 タイル UI でのオプション収納方法

**採択: 案 B（固定 All-ON で省略）**（cycle-206.md L167-174）

fullwidth-converter は「英数字 / カタカナ / 記号」の 3 種チェックボックスを持つが、タイルでは全 ON 固定で省略。理由:

- T1 likes「すぐ使い始められる」最大化
- 最大公約数ユースケース（全変換）が大半の利用者の用途に一致（詳細ページの実装デフォルトも全 ON）
- AP-P21 リスク低（4 要素縦積みでベースラインに合わせやすい）
- 詳細ページとの責務分離が明確（タイル=即時性 / 詳細=細部制御）

3 案比較（cycle-206.md L155-165）:

- 案 A: タイル内に 3 チェックボックス全配置 → AP-P21 リスク高、詳細ページ差別化消滅
- 案 B: 固定 All-ON 省略（採択）
- 案 C: 事前設定セグメント → 中間的。詳細ページ差別化を部分侵食

### 2.2 AP-P17 の判断

AP-P17 は「対応方針が複数考えられる課題に対し、計画段階で 3 案以上をゼロベースで列挙し比較表を作ること」（planning.md L53）。

cycle-206 での適用:

- オプション収納方法（3 案比較 = 最重要論点として計画書に明示）
- 変換トリガー方式（リアルタイム vs 手動 vs 混在）
- 詳細ページ Component.tsx の取り扱い（touch する vs しない）

### 2.3 AP-P21 計測ケース

fullwidth-converter での 4 ケース（cycle-206.md L142-147）:

- (a) 空入力
- (b) 短い ASCII 入力（ABC123 6 文字→全角変換で同長 6 文字）

### 3.2 cycle-206 の「オプション 3 種チェックボックスのタイル省略」

出典: `docs/cycles/cycle-206.md` L153-179

前述 2.1 参照。「複数オプションをタイルで省略し全 ON 固定」は line-break-remover の `mergeConsecutive` / `smartPdfJoinStyle` の扱いに直接参照可能。

- fullwidth-converter の省略は「全 ON = 最大公約数」だったが、line-break-remover の `smart-pdf` モードは詳細ページでのみ選択可能とする可能性もある

kana-converter は 4 モードすべてを提供したが、2x2 グリッドに落ち着いた経緯は前述 1.6 参照。

line-break-remover の 3 モードは kana-converter の 4 モードよりラベルが短く、横一列 3 ボタンとして収まる可能性がある。ただし smart-pdf サブオプションの「条件付き表示」がタイル UI に収まるかは別途検討が必要。

## 4. AP-P21 / AP-P17 / AP-WF16 / AP-P19 の最新定義

### 4.1 AP-P21（planning.md L65-67）

固定枠 UI（タイル / カード / 限定領域）で flex container を使う計画を立てるとき、**枠内の子要素のうち「サイズが動的に膨張する側」と「ユーザー操作上の最低高さを保つべき側」が同居していないか** を検討したか？

cycle-202 で初導入、cycle-202〜208 で 7 連続適用済。line-break-remover も膨張ゼロ型（remove / replace-space）+ 縮小型（smart-pdf）のため役割分担パターン継続が必須。

対応方針が複数考えられる課題に対し、計画段階で **3 案以上をゼロベースで列挙し比較表を作ったか**。

### 4.4 AP-P19（planning.md L59-61）

過去サイクルで確定済みの判断根拠の中に **外部仕様（SEO 機能・ブラウザ API・Schema.org 仕様・サードパーティプラットフォーム機能の存続など）** への依存が含まれている場合、本サイクルで実装着手する前にその外部仕様の現在状態を **一次資料で WebFetch / WebSearch 確認** したか？

**cycle-207 での AP-P19 同型教訓**（planning.md と cycle-207.md の連関）:
cycle-207 では qrcode-generator ライブラリの `renderTo2dContext()` API シグネチャを直接 `node_modules` で実測確認することが必須だった。

- `node_modules/qrcode-generator/dist/qrcode.d.ts:50` の `renderTo2dContext(context: CanvasRenderingContext2D, cellSize?: number): void` = margin パラメータが API シグネチャに存在しないことを確認（cycle-207.md L151 / L492）
- 「createImgTag() が内部で createDataURL()（GIF 出力）を呼ぶ」という事実は dist/qrcode.js L577 の実測で確認（cycle-207.md L492）
- この「外部ライブラリ API シグネチャ確認」は AP-P19 と同型の教訓（外部仕様への依拠を一次資料で確認する必要性）

**cycle-209 への示唆**: line-break-remover は外部ライブラリを使用しない純粋な自製ロジック（logic.ts 参照）のため、AP-P19 の外部 API シグネチャ確認は不要。ただし Next.js 固有 API（permanentRedirect 等）を使う場合は AP-WF12 / AP-P19 の適用を検討すること。

## 5. tile-grid.ts の現状と運用ルール

出典: `/mnt/data/yolo-web/src/tools/_constants/tile-grid.ts`

### 5.1 定数定義

```
TILE_CELL_PX = 128    // 基本セルの幅・高さ (px)
TILE_GAP_PX = 8       // セル間マージン (px)

calcTilePixels(cols, rows):
  width  = 128 * cols + 8 * (cols - 1) = 136cols - 8
  height = 128 * rows + 8 * (rows - 1) = 136rows - 8
```

| サイズ        | 実ピクセル    |
| ------------- | ------------- |
| cols=3 rows=2 | 400px × 264px |
| cols=3 rows=3 | 400px × 400px |
| cols=2 rows=2 | 264px × 264px |

### 5.2 現状 TILE_DECLARATIONS の rows 運用

現在 9 エントリ（tilesCount=9）:

- cols=3 rows=2: 8 件（char-count / byte-counter / url-encode / base64 / html-entity / hash-generator / fullwidth-converter / kana-converter）
- cols=3 rows=3: 1 件（qr-code）

rows=2 が標準形、rows=3 は「画像/SVG 出力型ツールで大きいプレビューエリアが必要な場合の例外」という運用。

### 5.3 recommendedSize の運用ルール（cycle-207/208 から抽出）

- `rows=3` を採用する正当化根拠の例: 「SVG 272px 確保（QR 物理 2cm 相当 = 絶対最小読み取り条件クリア）という明確な来訪者価値起点」（cycle-207.md L239-240）
- 「Phase 10.1 ダッシュボード設計時の rows=3 サンプル数を増やすため」だけでは Decision Making Principle に反する（cycle-208.md L232）
- line-break-remover はテキスト出力型 → rows=2 が第一推奨

### 5.4 w360 viewport 対応の現状

B-434 で Phase 10.1 必須検討事項として温存中。Phase 8.1 内では w375/w1200/w1900 + タイルプレビューは w1200/w375 を採用（w360 は暫定許容）。
