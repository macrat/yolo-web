# text-diff 競合・先行サービス UI/UX 調査

## 調査概要

cycle-214 の text-diff リニューアルに向け、日本語圏 5 件・英語圏 3 件のオンライン diff ツールを Playwright で実機訪問し、入力レイアウト・差分表示形式・操作フロー等を一次資料として記録した。調査日時: 2026-05-28。

## D. ベストプラクティス（5 件）

### BP-1: モバイルは上下縦積み、デスクトップは左右並列

調査した 8 サービスのうち、ラッコ・ツールタロウ・スペースカウボーイの 3 件がモバイルで自動的に縦積みに切り替えていた。これらは実際に使いやすい。一方 difff・DiffSuite・Diffchecker・Mergely はモバイルでも左右並列を維持しており、375px では各エディタが 160〜170px 幅に圧縮されて非常に読みにくかった。**ブレイクポイントを設けてモバイルは縦積み（上: 変更前 / 下: 変更後）に切り替えることが必須。**

### BP-2: 差分粒度の切替は一定の需要があるが、デフォルトは「行+インライン」が最も使いやすい

最も高機能な Diffchecker が Smart/Word/Char の 3 択を提供し、他の多くは文字単位固定または行単位固定。実際の差分を観察すると、行単位でハイライトしつつ変更箇所をインラインで強調する「Smart」相当の表示が最も視認性が高い。単純に「文字単位固定」では行全体の見通しが悪く、「行単位固定」では細かい差分が不明確になる。

### BP-3: 「比較」ボタンの視認性を高くする + ボタン位置は 2 つの入力欄の間（または下）に配置

全サービスのボタン配置を比較すると、ラッコ（2 つの入力欄の真下中央）と Diffchecker（入力欄の下）が最も自然だった。difff とツールタロウは左寄りで目立たない。スペースカウボーイはボタンを disabled にして「テキストが空です」を視覚的に伝えているが、これは誤操作防止として優れた UX。

### BP-4: 差分には必ず追加=緑・削除=赤の 2 色を使い、変更箇所を明確に分離する

difff のように「変更箇所のみシアン 1 色」では追加か削除かが一目でわからない。ラッコ・Diffchecker・DiffSuite は追加=緑・削除=赤（またはピンク）を徹底しており、初見ユーザーが迷わない。この 2 色分離はほぼ業界標準となっている。

### BP-5: 結果は入力欄の下に表示し、入力欄はスクロールで残す

Text Compare! の「結果が上 / 入力が下」という逆転レイアウトは直感に反する（スクロールすると意図が逆転する）。ラッコ・Diffchecker のように入力欄が上部に残り、スクロールすると結果が現れる構造が最も自然。または Diffchecker のように入力と結果を別ページ（別状態）に分ける設計も有効。

## E. 避けるべきパターン（3 件）

### AP-1: モバイルで左右並列を維持する

difff・DiffSuite・Mergely はモバイルでも 2 カラムを維持するため、375px 幅では各エディタが 160px 程度になる。長い行は自動折り返しで読みにくく、スクロールも左右・上下の両方が必要になる。**モバイルでの左右並列維持は「使えない」レベルの UX 劣化を引き起こす。**

### AP-2: 差分を色だけで示し、追加/削除の区別ができない

difff の「変更箇所のみシアン 1 色」パターンが典型例。色覚多様性（例: 赤緑色盲）の場合、緑と赤を区別できないことがある。最低限、追加=+ 記号・削除=- 記号などのテキスト記号を補助として使用すべき。または WCAG AA を満たす色コントラスト比を確保した上で 2 色以上を使う。

### AP-3: 大量テキスト入力時にリアルタイム計算で UI フリーズする

DiffSuite・Mergely はリアルタイム自動計算だが、数千行のテキストを貼り付けると計算が追いつかず入力がもたつく可能性がある。ユーザーが「コピペしたらすぐ結果を見たい」場合には、ボタン押下式で適切なタイミングで計算するほうが安全。リアルタイム計算を採用する場合は debounce（300〜500ms 程度）が必須。

## F. yolos.net text-diff の論点 1〜5 への PM 判断材料

### 論点 1: 入力欄は「左右並列」か「上下並列」か

**結論**: デスクトップは左右並列、モバイル（< 768px 程度）は上下縦積みに変更する。

根拠:

- 調査した 8 サービス中、モバイルで縦積みにしているのはラッコ・ツールタロウ・スペースカウボーイの 3 件。いずれも実際に使いやすかった。
- 左右並列を維持しているサービス（difff・Mergely 等）は Playwright で実際に確認した結果、375px では入力欄が 160px 幅に圧縮され実用不可と判断。
- 現在の yolos.net 実装（Component.tsx）はすでに `.panels` で左右並列だが、CSS で `@media (max-width: ...)` による縦積み切替が必要かどうか確認が必要。

### 論点 2: diff 粒度のデフォルトと切替 UI

**結論**: デフォルトは「行単位（line）」のまま維持。切替 UI（セレクト or ボタン切替）は残す。

根拠:

- 現在の実装はすでに行/単語/文字の 3 択セレクト切替を持っており、Diffchecker の Smart/Word/Char 切替に相当する機能を持っている。これは競合に対するアドバンテージ。
- デフォルトを「行単位」にするのは、大半のユーザーが「どの行が変わったか」を確認したいケースが多いため。
- 文字単位は細かい校正作業向けで、セカンダリユーザー向けとして切替で提供する価値がある。
- ただし、セレクトボックスより「行 / 単語 / 文字」のボタン切替 UI（Diffchecker 方式）のほうが視認性が高い。リニューアル時はボタン切替を検討。

### 論点 3: 自動リアルタイム計算 vs 「比較」ボタン押下式

**結論**: 「比較」ボタン押下式を維持する。ただし debounce 付きのリアルタイム計算を補助として検討可（オプション扱い）。

根拠:

- 調査した 8 サービスのうちリアルタイム自動計算は DiffSuite と Mergely の 2 件のみ。残り 6 件はボタン押下式。
- yolos.net のターゲットユーザーは「コピペして結果を見たい」ケースが多い。貼り付けて「比較」ボタンを押すフローは自然で、誤動作を防ぐ。
- リアルタイム計算は大量テキスト時にパフォーマンスリスクがある（AP-3 参照）。
- ただし Diffchecker の「Real-time editor」チェックボックス（オプションとしてリアルタイム化可能）は UX 上の差別化になりうる。yolos.net では当面ボタン押下式に集中する。

### 論点 4: タイル UI に「2 つの textarea + 差分表示」を収めるか、「軽量プレビュー + 詳細リンク」か

**結論**: タイルには「軽量プレビュー（短い説明 + 詳細ページリンク）」を採用する。タイルに 2 つの textarea を収めるのは不適切。

根拠:

- 調査したどのサービスもトップページ的な「タイル」に 2 つの textarea を収めている例はなかった。各サービスは 1 ページ丸ごとをツールに使っている。
- 2 つの textarea をタイルに収めると、最低でも各 textarea を 200px × 100px 程度にしか取れず入力が極めて難しくなる。
- yolos.net のタイル UI 設計（現行: 他のツールも「ショート説明 + 詳細リンク」形式）と整合させる必要がある。
- タイルサイズは現行の他のタイルと同じ cols × rows（既存タイル定義を確認して合わせる）で、差分の説明テキスト + 「使ってみる →」リンクを表示する。

### 論点 5: 差分カラースキームの a11y

**結論**: 追加=緑・削除=赤の 2 色に加え、テキスト記号（+ / -）または太字・下線などの視覚的補助を併用する。

根拠:

- 調査した 8 サービス中、記号（+/-）を明示的に併用しているサービスは 0 件。全サービスが色のみに依存している。
- 色覚多様性（赤緑色盲）の場合、赤と緑の区別が困難。WCAG 2.1 SC 1.4.1「Color」は「色だけを唯一の視覚的手段にしてはならない」と規定。
- 競合との差別化として a11y 対応を明示的に採用することで、スクリーンリーダー利用者や色覚多様性のあるユーザーをカバーできる。
- 具体的には: 追加部分の先頭に「＋」記号、削除部分の先頭に「−」記号をスクリーンリーダー向けに `aria-label` で付与するか、視覚的にも記号を添えるデザインにする。色選択は WCAG AA 準拠の色コントラスト比（4.5:1 以上）を確保すること。

## G. 付録: 現在の yolos.net text-diff 実装メモ

`/src/tools/text-diff/Component.tsx` の現状:

- 入力欄: 2 つ、`.panels` div で左右並列（CSS 未確認のため実際のレスポンシブ挙動は要確認）
- 比較モード: `select` 要素で「行単位 / 単語単位 / 文字単位」の 3 択
- 実行方式: 「比較する」ボタン押下式（`setShowResult(true)`）
- 結果表示: `pre` タグで追加=`.added`（緑）/ 削除=`.removed`（赤）のインライン diff

競合と比較したうえで、以下の点が改善候補として挙げられる:

1. モバイルでの縦積みレイアウト対応
2. モードセレクトを直感的なボタン切替に変更
3. 差分表示に記号（+/-）を追加して a11y 強化
4. ボタン位置を 2 つの入力欄の真下中央に移動
5. 入力が空のときボタン disabled にして視覚フィードバックを提供
