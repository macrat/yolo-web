### 3-2. `page_referrer` 代替指標の機能評価

判定観点:「search 直後の page_view の pageReferrer が search 発火直前ページ以外（=検索結果クリックによる遷移）になっているセッション割合」

- 本データは「search イベント発火時の pageReferrer」であり、これは「search 発火ページへの遷移元 URL」を示す。
- search イベントはモーダル UI で発火するため、**search が発火したページは検索モーダルを開いた元のページ**。
- 以下を確認:
  - Row 1: Bing からの直接流入で /tools/keigo-reference にランディング後に search 発火 → **referrer が外部サイト（Bing）のため、検索後の page_view との区別ができない**
  - Row 4 & 5: kanji 詳細ページ → 別 kanji 詳細ページの遷移が search event で記録 → **referrer はサイト内前ページ（kanji/stroke/7 → kanji/杉 → kanji/核）** → これは search モーダルを使って kanji 内を横断しているパターン

### 3-3. 判定

この `pageReferrer + pagePath` データは「**search イベント自体が発火したページ**への遷移元」であり、「search 後に結果をクリックして着地したページ」ではない。

- 「search 直後の page_view」を分離するには event-sequence レベルのデータ（BigQuery ストリーミング or GA4 Explorer のパス分析）が必要。
- GA4 Data API の run_report では、イベントの発火順序（「search の直後に page_view」）を取得できない。
- **実質的に `page_referrer` が「結果クリック前後の区別」として機能しているかを判定できるデータを GA4 Data API では取得不可能**。

**`page_referrer` 代替指標: 機能しない（判定不可）**

| /memos      | 23                               | 63  | 0%     |
| /play       | 16                               | 26  | 0%     |

sessions 列は GA4 Data API の標準 `sessions` 指標（ページを landingPage としてカウントする landingPageSessions ではなく、当該ページが含まれるセッション総数）。直帰率 0% の解釈上の留意点：GA4 の「直帰率」は「エンゲージドセッション率の逆数」（= 10 秒以上滞在・コンバージョン・2 ページ閲覧のいずれかを満たさなかった割合）。カテゴリ一覧ページの直帰率は現状観測値として 0% であり、多くのセッションが GA4 のエンゲージドセッション定義（10 秒以上滞在・コンバージョン・2 ページ閲覧のいずれか）を満たしたためと推測される。ただし PV > sessions であることは全セッションが 2 ページ以上閲覧したことを意味するわけではない（例: 半数が 1 ページ離脱・半数が 3 ページ閲覧でも PV > sessions となりうる）。

**フォールバック動線の評価**: 現状成立している。ただし検索を利用した 5 セッションが一覧経由で同等の体験を得られるかは不明（クエリが複数カテゴリ横断のため）。

## 6. 各候補のリスク・上限値比較

### 6-1. (c) 検索撤去で失われる体験の上限値

- 影響セッション数: 最大 **5 / 727 = 0.69%** のセッション
- フォールバック動線（カテゴリ別一覧）は現状機能しているが、横断クエリ（kanji+tool など）を持つユーザーには代替にならない可能性がある
- 上限値の評価: 影響は極小だが、失われる体験の質的評価ができない状態での撤去はリスクがある

### 6-2. (a) 検索維持の運用コスト

- `search-index.json` 生成（ビルド時）のメンテナンス継続
- Fuse.js ライブラリの依存関係維持
- 新デザイン導入時の検索 UI 対応が必要（view 実装の追加コスト）
- 計測強化（search_result_click 等の追加実装）が必要で、単純維持でも中程度のコストが発生

### 6-3. (b) カテゴリ別検索分割のコストと懸念

- 一覧 UI をカテゴリ別に改修するコスト（中規模）
- 横断クエリ（kanji+tool 混在）を持つユーザーが複数カテゴリを個別に検索しなければならなくなる懸念
- search_term 分析（§2）では横断性を弱く示唆しており、(b) 分割は逆効果の可能性

### 6-4. 来訪者価値の観点からの評価

影響上限 0.69% というサンプル数では、3 候補の体験差を来訪者価値として確定できない。ただし、以下の点から **「サンプル待ち」ではなく「計測を強化した上での判断」** が正しいアプローチと判断する:

- 現在の `search` イベントのみでは候補選択に必要な情報（結果クリック率、検索→離脱率）が構造的に取得不可能
- 計測強化なしに待機しても、月次 1.7 セッションではいつまでも判断できない
- 計測強化（search_result_click 等）を先に実施することで、より少ないサンプルでも判断が可能になる

## 7. 総合判定

### 判定: 計測データ不足 → Phase 1.1 を **保留**

**根拠（精度向上版）:**

過去 90 日（および期間を 180 日に拡張しても同値）で `search` イベントが発火したセッションは 5 件（全体 727 セッションの 0.69%）。実質的な完成クエリは 5〜8 件のみで、これは検索利用セッション 5 件と整合する標準的な実装（150ms デバウンス付き）の結果であり、実装上の問題ではない。

search_term の分析から kanji・ツール・yoji 系が混在しており、横断検索の傾向が弱く示唆されるが、n=5 では統計的に確定できない。検索撤去 (c) の影響上限は最大 0.69% だが、フォールバック動線（カテゴリ別一覧）が横断クエリを持つユーザーに有効かどうかが不明なため、影響の質的評価ができない。

現状計測（`search` イベントのみ）では search_result_click・検索後離脱などのデータが GA4 Data API で構造的に取得不可能であり、(a)/(b)/(c) を根拠をもって選択できない。期間を 180 日に拡張しても n が変わらないことから、計測強化なしの待機は解決策にならない。以上から、**計測強化を先行させた上で再判断する保留** が適切。

### 保留時のフロー（計画書 L140-150 に準拠）

- (a) 判断に必要なデータ: 検索モーダル開閉イベント、検索結果クリックイベント（遷移先 URL 付き）、検索→離脱（結果クリックなし）の計測。これらを実装して 30 日以上収集した後に再判断。
- (b) 再判断推奨時期: 必要イベント追加実装後 30〜60 日（最短でサイクル 180 前後）。
- (c) 後続 backlog 状態遷移:
  - B-332（本タスク）: Blocked に遷移、再判断条件を Notes に記載
  - B-331（Phase 5: 検索実装の要否）: Blocked に遷移
  - B-334（Phase 4: 一覧設計）: Blocked に遷移
  - Phase 1.1 再判断 backlog を新規起票して Deferred 登録（必須）

### 再判断条件

**再判断条件**（次サイクルで検索方針を再判断するための条件）:

1. `search_result_click` イベントを実装する（コード変更を伴うため別サイクル起票）
2. 計測強化後 30 日以上のデータを蓄積する
3. 蓄積データで (a)/(b)/(c) を中立に再判断する。判断材料が依然として不足している場合は、当時のデータで最善の中立判断を下す（特定候補の選好バイアスは持たない）。

**質的閾値（参考）**: 完成クエリ（既知ツール/カテゴリ名と一致、または 3 文字以上の確定形）に対して、同一セッション内で完成クエリの後に `search_result_click` が記録された事例が 1 件以上あるか。これは判断材料の質を担保するための参考指標であり、未到達でも 30 日経過時点で再判断は実施する。

注: 計測強化（search_result_click 等の追加実装）は search モーダルの利用ユーザー行動には干渉しないため、search 発火セッション数（現状 1.7/月）を計測強化だけで増やすことはできない。閾値はサンプル数ではなく「判断に必要な情報の取得可否」に置く。

## 付記: trackSearch の実装挙動（訂正版）

`useSearch.ts` (L41, L131, L142-144) の実装:

- `DEBOUNCE_MS = 150` で `setTimeout` を使用（L41, L142-144）
- `performSearch(q)` の内部で `trackSearch(q)` を呼ぶ（L131）
- `src/lib/analytics.ts` (L60-67) の `trackSearch` は `sendGaEvent("search", { search_term: trimmed })` のみ

**正確な挙動**: keydown ごとに発火するのではなく、**入力が 150ms 止まるたびに 1 回発火する標準的なデバウンス実装**。「も」→「もさ」→「もさく」→「模索」のように、ユーザーがゆっくり入力すると停止のたびに 1 回ずつ発火する。これが 42 件中 30 件以上が入力途中文字列になっている原因。

この実装は問題ではなく標準的な UX パターンであり、「クエリ確定数 ≒ 実質的な検索行動数 5〜8 件」という整理はむしろ保留判定を強く支持する（イベント数の過大計上問題ではなく、サンプル数の絶対的不足が本質的課題）。
