## サマリ

400px 角のタイルで最短 UX を実現するには「判定バッジを最大・常時可視」「エラー理由を入力欄直下に 1 行ずつ即時展開」「タイポ提案はワンタップ採用リンク付き文言で最優先表示」という三点が核心。情報優先順位は ①判定（valid/invalid）→ ②エラー/警告理由（即時展開）→ ③タイポ修正提案（あれば最上段扱い）→ ④パーツ内訳（折りたたみ可）。デフォルト初期値はタイポ例アドレスが来訪者価値を最も短時間で伝える。コピー機能は「修正提案アドレスのワンタップコピー」に限定すれば価値がある。

## 1. 競合の結果提示 UI 実態調査（Playwright 実機確認）

### 1-1. VerifyForge / email-format-validator（クライアント完結・形式検証専用）

調査日時点で最も yolos.net に近い形式検証専用ツール。

**入力前の状態**

- 全幅のテキストボックス＋「Check Format」ボタン（右側に配置）の 1 行レイアウト。
- ボタンは入力前は disabled（グレーアウト）。
- 入力欄直下に「This tool checks email syntax only (RFC 5321 compliance). It doesn't verify if the email exists or can receive mail.」という注記バナーが常時表示される。
- デフォルトのプレースホルダーは `Enter email address to check format`（例アドレスなし）。

**有効アドレス入力後（test@gmial.com）の表示**

- 入力欄の下に区切り線なしで続けて「Email Structure」カード表示: Local Part / Domain / TLD の 3 列グリッドで `test` / `gmial.com` / `com` をボールド表示。
- さらに下に「Valid Format」と書かれた**緑背景カード**（左端に緑の丸チェックアイコン）＋「14/14 rules passed」の補足テキスト。
- 続けて「Passed Checks (14)」という折りたたみなしのフルリスト。各チェック項目が「Not Empty RFC 5321 §...」のように RFC 番号タグ付きで緑アイコン＋項目名＋説明テキストの 3 要素で列挙。合計で 14 行分のリストが続き、スクロールが必要。
- タイポ（gmial.com）は検出されない（形式上は有効なため）。

**無効アドレス入力後（invalid..email@）の表示**

- 「Invalid Format」と書かれた**赤背景カード**（左端に赤の×アイコン）＋「5/14 rules passed」。
- 続けて「Failed Checks (9)」リスト：各項目が赤×アイコン＋「Domain Part RFC 5321 §4.5.3.1.2」＋「Domain part (after @) must be 1-255 characters」の 3 要素。
- 「Passed Checks (14)」の展開は失敗リストの後に続けて出る。

**評価**

- 情報量が多すぎる（14 件のルールチェックをフル展開）。スクロールが必要で、タイル枠内に収まらない。
- RFC 番号タグは開発者には有用だが一般来訪者には冗長。
- パーツ内訳（Local/Domain/TLD）を結果の前に表示している点は参考になる。
- 「何が悪いか」は明確だが「どう直すか」の提案がない。

### 1-2. site24x7 email-validator（CAPTCHA＋サーバー検証型）

- CAPTCHA が必須のため、クライアント完結ツールとしての参考価値は低い。
- レイアウト: テキストボックス＋CAPTCHAイメージ＋「Validate」ボタンの縦積み構成。
- 結果 UI は今回確認不可（CAPTCHA ブロック）。

### 1-3. verifalia.com（Cookie 同意後フォーム表示）

- Cookie 同意後にフォームが表示されるが、実際のバリデーション UI は iFrame 内で動作しており Playwright のアクセシビリティツリーからは取得できなかった。
- サーバー検証主体の商用ツールであり、形式検証の参考としては限定的。

### 1-4. email-checker.net（MX/SMTP 検証型）

- 入力欄＋「Check」ボタンのシンプルな構成。
- サーバー側 MX チェックのため応答に数秒かかり、タイポ入力後でもフォームがリセットされる挙動が見られた。純粋な形式検証 UX の参考としては不向き。

### 1-5. yolos.net/tools/email-validator（現状確認）

実機確認による現在の実装状態：

**タイポアドレス（test@gmial.com）入力時**

- 入力欄下に薄いグレー背景カード内に以下を縦積み：
  - 緑バッジ「有効」（左寄せ、丸角タグ）
  - 「ローカルパート: test」「ドメイン: gmial.com」（等幅フォント）
  - 青枠カード「候補」見出し + 箇条書き「もしかして: test@gmail.com」（クリックで入力欄に反映するリンク形式）

**無効アドレス（invalid..email@nodot）入力時**

- 赤バッジ「無効」
- 「ローカルパート: invalid..email」「ドメイン: nodot」（等幅フォント）
- 赤枠カード「エラー」見出し + 箇条書き：
  - 「ローカルパートに連続したドットは使用できません」
  - 「ドメインに TLD（トップレベルドメイン）がありません」

**現状の強み**

- リアルタイム（文字入力ごと）に結果が更新される。
- 「もしかして」がクリックで採用できる。
- エラー文が日本語で平易。

**現状の改善余地**

- パーツ内訳（ローカルパート/ドメイン）がバッジと同列で表示され、判定バッジの視覚的優位性が薄い。
- 「候補」セクションが「有効」バッジの下に位置しており、タイポがある場合に警告として目立ちにくい。
- 400px 角のタイルに収めるための情報整理がまだ行われていない。

## 2. タイポ修正提案（did-you-mean）の UX パターン

### mailcheck.js の実績パターン

mailcheck.js（Kicksend 発のオープンソースライブラリ）が確立した標準パターン：

- **トリガー**: blur イベント（入力フィールドからフォーカスが外れた時点）で検証。入力中は表示しない。
- **表示文言**: 「Yikes! Did you mean [address]@[domain]?」の形式。ドメイン部分をクリッカブルリンクにする。
- **採用方法**: ドメイン部分のリンクをクリックすると入力欄が修正済みアドレスで上書きされ、提案が自動で消える。
- **却下方法**: 明示的な「却下」ボタンは設けず、ユーザーが続けて他の操作をすれば自然に消える。
- **効果実績**: Kicksend では確認メールのバウンス率が 50% 削減。

### yolos.net における適用指針

メール検証ツールはフォーム登録フローではなく「確認・分析」が目的のため、blur 後ではなく入力後即時（debounce 300ms 程度）に表示する方が自然。ただし以下の点を守ること：

- 提案アドレスはリンク形式で「ワンタップで入力欄を上書き採用」できる。
- 「もしかして」という表現（現状）は適切。ただし視覚的に警告色（アンバー/オレンジ）背景か强調ボーダーで、エラーとは色分けする。
- 警告（warnings）扱い（形式上は有効だがタイポ可能性あり）であり、エラー（形式無効）と混在させない。

## 3. 狭い枠（400px 角程度）での情報優先度

### 原則：判定結果を主役にし、詳細は従として折りたたむ

NN/G・Baymard・Smashing Magazine のガイドラインを統合した場合、コンパクトなバリデーション表示での情報優先度は以下の順序が最適：

**第一層（常時可視・最大面積）**

- 入力欄（テキストボックス）
- 判定バッジ: `有効` / `無効` / `警告あり` の三状態。色（緑/赤/アンバー）＋アイコン（✓/×/!）の二重符号化で色覚多様性に対応。文字サイズを大きくし、カード全体に背景色をつけることで一目で把握できる。

**第二層（エラー/警告がある場合に展開・スクロールなしで視野内）**

- エラー理由リスト: 1 行 = 1 エラー。具体的かつ平易な日本語で「何が悪いか」を列挙。
- タイポ修正提案: エラーより上（または同カード内の最上段）に配置。「もしかして: [corrected]」＋ワンタップ採用リンク。警告色背景。

**第三層（有効時・オプション展示）**

- パーツ内訳（ローカルパート / ドメイン）: 有効時のみ表示。コンパクトな 2 列表示。
- warnings リスト（一般的でない特殊文字、数字のみ TLD 等）: 有効だが注意が必要な場合のみ。

**第四層（折りたたみ・デフォルト非表示）**

- 詳細ルールチェックリスト（VerifyForge 方式の 14 件列挙）は 400px 枠では不要。関心のある来訪者向けに「詳細を見る」として折りたたみ可にする。

### 具体的なレイアウト案（400px 正方形）

```
┌─────────────────────────────────┐ ← 400px
│  [入力欄  test@gmial.com     ]   │
│ ───────────────────────────────  │
│ ⚠ 警告あり (有効な形式)           │  ← アンバー背景
│   もしかして: test@gmail.com  採用│  ← ワンタップリンク
│ ───────────────────────────────  │
│ ローカル: test  ドメイン: gmial.com│  ← グレー小文字
└─────────────────────────────────┘
```

```
┌─────────────────────────────────┐
│  [入力欄  invalid..email@nodot  ]│
│ ───────────────────────────────  │
│ ✕ 無効                           │  ← 赤背景
│  • ローカルパートに連続ドット不可  │
│  • TLDがありません                │
└─────────────────────────────────┘
```

```
┌─────────────────────────────────┐
│  [入力欄  user@example.com      ]│
│ ───────────────────────────────  │
│ ✓ 有効                           │  ← 緑背景
│   ローカル: user  ドメイン: example│
│   TLD: com                       │
└─────────────────────────────────┘
```

## 4. メール検証ツールでのコピー需要

### 「変換して持ち帰る」型との違い

パスワード生成やハッシュ計算などは出力値（生成物）をコピーして使うため、コピーボタンの来訪者価値が高い。メール検証ツールは「入力したものが正しいかを確かめる」という確認型で、来訪者はすでに手元にアドレスを持っている。

### コピー機能が有効な場面

唯一コピー需要があるのは**タイポ修正提案アドレスのコピー**。シナリオ例：

- メーリングリストに誤ったアドレスがあり、ツールで正しいアドレスを確認したあと、別のメールクライアントやスプレッドシートに貼り付けたい。
- 自分が入力しようとしているアドレスをツールで確認し、修正案を得てそのまま登録フォームに貼り付けたい。

### 実装指針

- 判定ステータスバッジや入力値そのもののコピーボタンは不要（来訪者は自分で入力したものを手元に持っている）。
- 「もしかして: [corrected]」の提案アドレスの横に小さなコピーアイコンを置く価値はある。ワンタップ採用（入力欄上書き）と並列で提供し、どちらかを選べるようにする。
- コピー成功後は「コピーしました」のトースト通知（1-2 秒で消える）を表示する。

## 5. 来訪者が「メールアドレスが正しいか確かめたい」と思う場面

調査・推論により以下のシナリオを特定。それぞれで最重要の出力が異なる。

### シナリオ A: フォーム入力前の事前確認

- 状況: 重要なサービスに登録しようとしている。自分が入力しようとしているアドレスに自信がない（特に会社のアドレスや珍しいドメイン）。
- 最重要の出力: 「有効」か「無効」かの即時判定 ＋ エラーがある場合の具体的な修正箇所。
- 次に重要: タイポ提案。
- 不要: 詳細なルールチェックリスト。

### シナリオ B: 他人のアドレスのタイポ疑惑

- 状況: 名刺・書類・聞き取りで得たアドレスが怪しい（`gmail.co.jp` や `yahoo.ne.jp` など）。
- 最重要の出力: タイポ提案（もしかして: ○○）。
- 次に重要: ドメイン部の表示（目視確認）。
- 不要: 詳細エラーリスト（形式的に有効なため表示されない場合も多い）。

### シナリオ C: メーリングリストや CSV の精査

- 状況: 大量のアドレスの中から形式不正なものを除去したい。
- 最重要の出力: valid/invalid の判定 ＋ invalid の理由（どのルール違反か）。
- 注: このユースケースは 1 件ずつの確認には向かない（バルク処理が本来の解）が、代表的なエラーパターンを学ぶ用途で使われる可能性がある。

### シナリオ D: エラーメッセージを理解したい（学習目的）

- 状況: フォームで「メールアドレスが無効です」と言われた。何が悪いか分からない。
- 最重要の出力: エラー理由の平易な日本語説明 ＋ 修正提案。
- 次に重要: パーツ内訳表示（どの部分が問題かを視覚化）。

## 6. 初期表示の設計（空欄 vs 有効例 vs タイポ例）

### 調査背景

NN/G は「プレースホルダーは消えるためラベルの代替にしてはいけない」と指摘しているが、ツール用途では「何ができるかを示す」目的でプレースホルダーや初期値を使うことは有効。

### 選択肢の評価

| 初期状態                            | 来訪者への価値伝達速度 | 操作コスト                 | 備考                                           |
| ----------------------------------- | ---------------------- | -------------------------- | ---------------------------------------------- |
| 空欄（プレースホルダーのみ）        | 低                     | 低                         | 何ができるか分かりにくい                       |
| 有効例（user@example.com が既入力） | 中                     | 中（消して打ち直し）       | 緑の「有効」が即見えるが驚きがない             |
| タイポ例（test@gmial.com が既入力） | 高                     | 中（採用ボタンを触るだけ） | 提案機能が即見える、ツールの価値が最短で伝わる |
| 無効例（invalid..email が既入力）   | 中                     | 低                         | エラー機能は見えるが、自分ごと化しにくい       |

**推奨: タイポ例（`test@gmial.com`）を初期値として設定する**

理由：

1. タイルを開いた瞬間に「もしかして: test@gmail.com」の提案が表示され、ツールの最大の差別化機能（タイポ提案）を即示できる。
2. 来訪者は「採用」を 1 タップするだけで使い方を体験でき、学習コストがゼロ。
3. 有効扱い（形式は正しい）なので「有効・警告あり」という状態を示せ、警告の概念も伝わる。
4. 有効例の場合は緑バッジだけ見えて「で、これ何に使うの？」となりがちだが、タイポ例なら即座に「これは便利」と理解される。

## 7. yolos.net 差別化軸との整合確認

| 差別化軸         | タイルウィジェット設計への影響                                                                                        |
| ---------------- | --------------------------------------------------------------------------------------------------------------------- |
| クライアント完結 | 「サーバーに送信しない」は小注記で示す（現状通り）。結果表示は 0 遅延を活かし打鍵ごとにリアルタイム更新する。         |
| 登録不要         | コピー・採用機能はアカウントなしで全機能使える。ログインへの誘導バナーなし。                                          |
| シンプル・高速   | 詳細チェックリストはデフォルト非表示（折りたたみ）。ページロード後即使用可能。                                        |
| MX/SMTP 検証なし | 「形式検証のみ」の注記を小さく 1 行表示し、期待値調整する。結果の断定表現（「このアドレスは届きます」等）は使わない。 |

## 8. まとめ：設計判断の推奨

1. **判定バッジを最大・最優先**: 有効/無効/警告の三状態を全幅カードの背景色＋大文字＋アイコンで表示。入力欄直下に配置。

2. **タイポ提案を筆頭に**: warnings セクション内でタイポ提案を最上段に置き、ワンタップ採用リンク＋小コピーアイコンをセット提供。

3. **エラー理由はバッジの直下に展開**: リアルタイムで更新。1 エラー 1 行、平易な日本語。RFC 番号などの技術情報はデフォルト非表示。

4. **パーツ内訳は有効時のみ・コンパクト表示**: ローカルパート/ドメイン/TLD を 3 列グリッドで小さく。無効時はエラーが優先なので非表示またはグレーアウト。

5. **詳細チェックリストは折りたたみ**: 「詳細を見る（14 項目）」リンクで任意展開。デフォルト非表示。

6. **初期値はタイポ例アドレス**: `test@gmial.com` など実際のタイポパターンを初期値に設定し、ページロード即座に提案機能を体験させる。

7. **コピーボタンは提案アドレスのみ**: 修正提案アドレスの横に小さなコピーアイコン。入力値や判定ラベルのコピーは不要。
