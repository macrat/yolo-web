# 道具箱 UI 設計判断：自動学習型 vs 手動配置型の UX 比較調査

## 調査概要

本調査は cycle-175 で事故認定された「UI/UX 調査の不足（自動学習型の調査欠落）」の是正措置として実施した。yolos.net が実装予定の「道具箱」—好みのツールをタイル状に並べてブラウザのホームページのように毎日開く機能—について、自動学習型（使用頻度・履歴ベースの自動並び替え）と手動配置型（ユーザーが能動的に組み立てる）のどちらが M1b ターゲット（気に入った道具を繰り返し使う人）に適切かを判断するための材料を収集した。

---

## 1. 自動学習型の動作事例と UX 評価

### 1.1 Chrome「よく使うサイト」（Most Visited）

**動作の仕組み**

Chrome の新タブページに表示される Most Visited（よく使うサイト）は、訪問頻度・滞在時間・最終訪問日時・ユーザーインタラクション量の複合スコアで自動ランキングが決まる。ユーザーは明示的に操作しなくてもタイルが入れ替わる。ピン留め機能はあるが、知らない間に並び順が変わることへのユーザー投稿が多数存在する。

**ユーザー体験上の問題**

Google Chrome コミュニティの投稿（support.google.com/chrome/thread/205554191）では、Android の Chrome で「頻繁に訪問するサイト」欄が突然並び替わり、Recent Searches と混在するようになったという複数の苦情が確認される。ユーザーが報告した具体的な問題は、「自分が意図しない順番になっている」「どのロジックで並んでいるかわからない」という2点に集約される。

Chrome は 2020 年代に入り「My Shortcuts（手動）」と「Most Visited（自動）」を明示的に切り替えられるようにした。これ自体が「自動のみでは不満を持つユーザーが多い」という Google 自身の判断を裏付ける。

**ピン留めの位置づけ**

Chrome の Most Visited では、ピン留めしたサイトは常に先頭グループに固定表示される。ピン留めしていないサイトは自動ランキングによって押し出される。つまり「固定枠 + 自動補充」という実質的なハイブリッドが Chrome のデフォルトになっている。

### 1.2 Microsoft Edge「クイックリンク」

**自動追加の問題**

Edge の新タブページは、ユーザーが操作しなくてもよく訪問するサイトを自動的にクイックリンクに追加する。Microsoft Q&A（learn.microsoft.com/answers/questions/2374432）には「自動追加を止めたい」「消しても戻ってくる」という苦情が複数掲載されている。

Edge の公式ドキュメントによると、クイックリンクを無効化する機能はあるが「特定サイトのみ自動追加を止める」という細かい制御はできない。ユーザーは全無効か自動追加を受け入れるかの二択を迫られる。

あるユーザーは「これが続くなら Edge をやめる」と脅迫的な表現で投稿しており、自動追加への反感の強さを示している。

**教訓**

ユーザーが自分でピン留めしたものと、システムが自動で追加したものが同じ領域に混在すると、「自分の意図した配置」と「システムの介入」の境界が曖昧になり、管理感が失われる。

### 1.3 Firefox「トップサイト」「ハイライト」

**2層構造の設計**

Firefox の新タブページは「Top Sites（よく使うサイト）」と「Highlights（最近訪問・ブックマーク）」の2系統を分離している。どちらも自動表示だが、Top Sites はピン留め可能で、ピン留めしたものは自動で変わらない。

Firefox コミュニティ（support.mozilla.org）には「新タブページが自分の望まないサイトを表示している」「自分がブックマークに追加したサイトだけを表示したい」という投稿がある。

**Sync によるピン解除問題**

Firefox サポートフォーラムには「Firefox Sync がピン留めを解除してしまう」という報告（support.mozilla.org/questions/1264361）もあった。これは自動処理がユーザーの意図した設定を上書きするという、より深刻な問題を示す。

### 1.4 Safari「よく使うサイト」（Frequently Visited）

**動作と問題**

Safari の Frequently Visited は訪問頻度に基づいて自動選択される。設定変更の方法を尋ねる Apple Community の投稿が多数存在し（discussions.apple.com/thread/253521778 等）、多くのユーザーが「非表示にしたい」「特定サイトだけ固定したい」という要望を持っている。

特に「非表示にしたはずが戻ってくる」というパターンは、自動機能への制御感喪失を示す典型例である。

### 1.5 iOS Siri 提案ショートカット / Android アプリ予測

**概要**

iOS の Siri Suggestions は時刻・場所・利用パターンをオンデバイスの機械学習で分析し、ロック画面・検索・ウィジェットにアプリ/アクション提案を表示する。Android のアプリ提案（Google Pixel Launcher のドック上部）は同様の機能。

**ユーザー評価**

Six Colors（sixcolors.com）の記事では Siri Suggestions ウィジェットを「often spot on（よく当たる）」と好意的に評価するレビューがある一方、Apple Community（discussions.apple.com）では「提案が消えた」「一度だけ便利な提案が出たが二度と出ない」という一貫性のなさへの不満が多い。

Android では Google Pixel の「アプリ自動配置」機能に対して「もう一度ホーム画面を整理させてほしい」「自動的にアイコンが動く」という苦情が複数確認される（support.google.com/pixelphone/thread/293288761）。Motorola のサポートページにもアプリ提案を「オフにする方法」を案内するページが存在し、無効化ニーズが高いことを示している。

**特徴的なパターン**

AI予測ショートカットに共通する問題は「一貫性のなさ」である。提案が出たり出なかったりすること自体が、M1b の「以前と同じ入力なのに結果や挙動が前回と変わっている」という不満と正確に一致する。

### 1.6 ブラウザ新タブ拡張（Toby / Momentum / Tabliss / Infinity New Tab）

**Toby**

Toby はコレクション（グループ）にタブを手動で保存する。公式サイトが「Toby does not save tabs automatically」と明記しており、意図的に自動保存をしない設計である。タブの保存はユーザーの明示的な操作のみで発生する。G2 のレビューでは「直感的なドラッグ & ドロップ」「自分の流儀でタブを整理できる」という点が高く評価されている（総合評価 4.6/5）。

ただし 2023 年以降、コア機能（60 タブ超の保存）を有料プランに移行したことへの不満が Firefox Add-ons レビューに集中しており、機能制限への反発が見られる。

**Momentum**

Momentum は美しい背景と時計・To-Do・フォーカスモードが主体のシンプルな新タブ拡張。リンク管理は手動ピン留めのみ。自動ランキング機能はなく、ユーザーが追加したリンクがそのままの位置で表示され続ける。レビュー総合評価は 9/10（性能・デザイン）と高い。

「リンクがサイドパネルに隠れている」という UX 上の懸念はあるが、「位置が変わる」という問題は報告されていない。

**Tabliss / Infinity New Tab**

Tabliss（github.com/joelshepherd/tabliss）は完全手動カスタマイズ型で自動ランキング機能は持たない。ユーザーがウィジェットを手動で追加・配置する。Infinity New Tab（Pro）は Chrome ブックマークとよく訪問するサイトの「両方」を表示できるが、ユーザーは表示するものを選べる。

**共通の傾向**

拡張機能レビューを見ると、自動ランキング型（Most Visited 相当）を持つ拡張は「意図しない変化」への苦情が、手動型（Toby / Tabliss）は「位置が変わらない安心感」が評価されている。

### 1.7 Pocket / Pinboard

**Pocket の並び順管理**

Pocket は基本的に「保存日時の新しい順」で表示される。後に「おすすめ（Recommendations）」タブが追加されたが、これはメインのリストとは別セクションに隔離されていた。ユーザーが「ドラッグして優先順位を変えたい」という要望は複数投稿で確認されるが、未実装のまま Pocket は 2025 年 7 月にサービス終了した。

なお、Firefox バグトラッカーに「Recommended by Pocket が無効化したはずなのに再び有効になる」という不具合報告（bugzilla.mozilla.org/show_bug.cgi?id=1774940）があり、自動機能の再有効化がユーザー設定を侵食する問題の事例として注目できる。

**Pinboard**

Pinboard（pinboard.in）は完全手動タグ付けのシンプルブックマークサービス。自動ソートや自動タグ提案はオプションのみ（Twitter お気に入りの自動取込み等）。コアは「ユーザーが能動的に整理する」という設計で、熱狂的なファンを持つ。

## 2. UX 研究からみた自動並び替えの問題

### 2.1 空間記憶（Spatial Memory）と安定した配置の重要性

Nielsen Norman Group の記事「Spatial Memory: Why It Matters for UX Design」（nngroup.com/articles/spatial-memory/）は、ダッシュボードやタイル UI の設計に直接的な示唆を与える研究である。

**核心的な知見**

- 空間記憶は「安定した UI」があって初めて機能する。要素が動くと空間記憶が崩壊し、ユーザーは再度「どこにある？」という視覚的探索を強いられる。
- カード UI / マソンリーレイアウトの「リフロー（要素の並び替え）」は空間記憶に対する最大の破壊要因のひとつと明記されている。
- 繰り返しアクセスするユーザーほど、安定した配置から恩恵を受ける。反対に、一度しか訪れない画面では空間記憶は関係ない。

**道具箱への適用**

道具箱は「毎日開く」前提の反復使用 UI である。NN/G が「繰り返しアクセスするインターフェースでは、要素の位置安定性が最重要」と結論づけていることは、自動並び替えが道具箱というコンセプトと根本的に相性が悪いことを示す。

### 2.2 筋肉記憶とインターフェース変更のコスト

Medium の記事「The Cost of Disrupting Muscle Memory」（@shafihireholi）は Figma の UI 刷新を題材に、インターフェース変更のコストを分析している。

**主な知見**

- ユーザーは繰り返しの操作で「筋肉記憶（Muscle Memory）」を形成する。要素の位置が変わると、この自動化された動作が誤作動を引き起こす。
- 変更は「遅いリアクションタイム」「高いストレス」「増えたエラー」という形でパフォーマンスを悪化させる。
- 美的改善のみで機能的改善を伴わないインターフェース変更は、純粋にネット負の体験をもたらす。

**道具箱への適用**

自動学習による並び替えは、ユーザーが意図しないタイミングで「昨日まであそこにあったツール」を別の場所に移動させる。これは M1b の dislikes「慣れた操作手順が突然変わること」と完全に一致する。

### 2.3 適応インターフェース（Adaptive Interface）のリスク

LogRocket の記事「Can adaptive interfaces evolve autonomously based on user behavior?」（blog.logrocket.com/ux-design/adaptive-interfaces-risks/）は適応インターフェースの具体的なリスクを整理している。

**識別された主要リスク**

1. **再帰的な狭小化ループ**: システムが「ユーザーがよく使うもの」のみを表示し続けると、ユーザーはさらにそれだけを使い、システムはそれを「好み」と解釈してさらに選択肢を絞る。表面上は「使われている」が、ユーザーは実質的に選択肢を失っている。

2. **エンゲージメントと好みの混同**: クリック率が高いことは「好み」ではなく「他に選択肢がない」という結果かもしれない。指標が健全に見えてもユーザー体験は悪化している可能性がある。

3. **発見と探索の喪失**: 最適化が繰り返しを強化し、新しい発見の余地を失わせる。

4. **透明性と制御の喪失**: 変化の理由が不明なままだと、ユーザーは「監視されている」「管理できない」と感じる。

5. **変化への対応不能**: システムはユーザーの行動を固定したアイデンティティとして扱う。実際の人間の行動は文脈・気分・ライフステージで変化するが、適応システムはこれを捉えられない。

**推奨されるハイブリッドアプローチ**

同記事は、適応機能を設計に取り込む場合は「安定した基盤の上に適応機能を乗せる」ことを推奨している。具体的には、コアナビゲーションと主要操作は常に安定させ、その上でレコメンデーションや補助情報のみを適応させる設計が望ましいとしている。

### 2.4 カスタマイゼーション vs パーソナライゼーション（NN/G）

Nielsen Norman Group の記事「Customization vs. Personalization in the User Experience」（nngroup.com/articles/customization-personalization/）は両者を明確に区別している。

**定義**

- カスタマイゼーション: ユーザーが制御する。ユーザーが明示的に設定を変更する。
- パーソナライゼーション: システムが制御する。システムがユーザーデータを分析して自動的に調整する。

**各アプローチの欠点**

- カスタマイゼーションの欠点: ユーザーの多くは「自分が本当に何を必要としているか知らない」か「設定を変更する動機がない」。セットアップの摩擦が高い。
- パーソナライゼーションの欠点: システムの推測が外れたとき、ユーザーは制御感を失う。プライバシー懸念がある。

**重要な結論**

NN/G はカスタマイゼーションの利点として「ユーザーが正確に望むものを得られる（なぜなら自分でコントロールしているから）」を挙げている。これは「気に入った道具を繰り返し使う M1b」にとって、パーソナライゼーション（自動学習）より手動カスタマイゼーションが基本的に適合することを示す。

また NN/G の「6 Tips for Successful Personalization」（nngroup.com/articles/personalization/）では、パーソナライゼーションを導入する際は必ず「逃げ道（opt-out）」を提供せよと規定している。自動学習型が仮に採用されるとしても、ユーザーが設定を上書きできる仕組みが必須である。

### 2.5 パワーユーザーと手動制御の親和性

UMD（メリーランド大学）の研究「The Importance of Agency, Privacy, and Power Usage」（terpconnect.umd.edu/~nan/738readings/Sundar%20Marathe%202010%20personalization.pdf）では、パワーユーザー（頻繁・深く使うユーザー）はシステム主導のパーソナライゼーションよりも、ユーザー主導のカスタマイゼーションに対してより肯定的な態度を示すことが確認されている。

道具箱を「毎日開いて繰り返し使う」M1b は、定義上パワーユーザーに分類される。この層が自動学習より手動制御を好む傾向は、学術研究レベルでも裏付けられている。

### 2.6 インターフェース習慣の破壊コスト（認知科学）

ScienceDirect に掲載された論文「Quantifying the impact of making and breaking interface habits」では、インターフェース習慣の形成と破壊を定量的に分析している。

**主な知見**

- ユーザーは繰り返しの操作で速度・正確性が向上するインターフェース習慣を形成する。
- 既存の習慣が破壊されると、パフォーマンスの向上が全て失われる。
- デザイナーはインターフェース変更（特に段階的なもの）には、ユーザー習慣への潜在的な破壊を十分に考慮すべきとしている。

自動並び替えは毎回の小さな「インターフェース変更」の連続である。習慣形成の妨害という観点では、定期的な自動変更は累積的に大きなコストを生む。

## 3. M1b との適合性分析

### 3.1 M1b の dislikes と自動学習型の矛盾

M1b の dislikes として定義されている2項目を自動学習型の実際の動作と照合する。

**Dislike 1「慣れた操作手順が、別デザインへの刷新や機能追加で突然変わること」**

自動学習型では、訪問頻度が変わるたびにタイルの並び順が変化する可能性がある。ユーザーが「道具箱の左上に電卓がある」という空間記憶を形成しても、その週に電卓をあまり使わなかったために位置が変わる。これは M1b が「最も嫌う」パターンと完全に一致する。

**Dislike 2「以前と同じ入力なのに結果や挙動が前回と変わっていること」**

自動学習型は「以前と同じ操作（道具箱を開く）」に対して「前回と違う配置」という結果を返す。これは M1b dislike 2 の定義そのものである。

**結論**

自動学習型は M1b の2つの dislikes と直接矛盾する。自動学習を導入することは、ターゲットユーザーが最も嫌う体験を中核機能として提供することになる。

### 3.2 手動配置型と M1b の適合性

**「能動的に組み立てる」というメタファとの整合**

手動配置は M1b の「気に入った道具を繰り返し使う」という行動パターンと整合する。M1b は自分で選んだツールを繰り返し使うことに価値を感じており、その選択プロセス自体（どのツールを道具箱に入れるか）がユーザーにとっての意味を持つ。

**安定性の確保**

手動配置型では、ユーザーが移動操作をしない限り配置が変わらない。空間記憶が安定して機能し、筋肉記憶が形成される。毎日同じ場所に同じツールがある、というのは M1b の期待と完全に一致する。

**カスタマイゼーションの摩擦**

NN/G が指摘するカスタマイゼーションの欠点（設定の摩擦）は、道具箱の文脈では部分的に緩和される。道具箱の初期設定は「追加してみたいツールを選ぶ」という積極的な動機から始まるため、受動的なユーザーには向かないが、M1b のような「ツールを能動的に選ぶことに意欲がある」ユーザーには障壁ではなく、むしろ「自分の道具箱を作る」というコアバリューそのものになる。

**初期セットアップの課題**

手動型の唯一の懸念は「空の初期状態」である。新規ユーザーが道具箱を開いたとき、全て空だと何もできない。これは既存の調査（2026-05-01 フェーズ1 ベストプラクティス）でも確認されており、Firefox の実装（デフォルトの人気サイト3件 + 空きスロット3件）のように、初期デフォルト値として「代表的なツール」を提案することで対処可能。

## 4. ハイブリッドアプローチの実例と評価

### 4.1 Chrome のハイブリッド実装

Chrome の新タブページは「My Shortcuts（手動）」と「Most Visited（自動）」の明示的な切り替えを提供している。ピン留めしたサイトは常に先頭に固定表示され、非ピン留めのスロットが自動ランキングで埋まる。

**評価**

Chrome のアプローチは「固定枠（ユーザー意図）+ 自動補充（ユーザー発見）」という論理的なハイブリッドだが、「自動補充」部分へのユーザーの不満も多い。特に「勝手に追加されたサイトを消したのにまた出てくる」という問題が複数報告されている。

### 4.2 Edge のハイブリッド実装

Edge は「ユーザーがピン留めしたリンク（固定・優先表示）+ 自動Top Sites（非ピン留め）」の構造だが、「自動追加を止める方法がない」という問題が深刻な不満要因となっている。制御の透明性がなければハイブリッドは機能しない。

### 4.3 Firefox のハイブリッド実装

Firefox の新タブページでは「Top Sites（自動表示、ただしピン留め可）+ Highlights（別セクション）」を使い、ピン留めした Sites とそうでないものを視覚的に区別している。これは比較的評判が良いが、「Highlights を消したい」という要望も多く、自動表示セクションを全体として無効化したいというニーズが存在する。

### 4.4 Start.me のハイブリッド実装

Start.me（start.me）は完全手動配置型のダッシュボードサービスとして設計されているが、「自動補充」機能はない。G2 等のレビューでは「自分のペースで整理できる」という点が高評価で、Softwareadvice や Capterra の評価も高い。Start.me のようなサービスが自動機能なしで成立していることは、手動型の独立した価値を示している。

### 4.5 ハイブリッドの必要条件

上記の実例から、ハイブリッドアプローチが機能するための条件を以下のように整理できる。

1. **手動部分が明確に優先される**: ピン留め（手動）が自動補充より常に優先表示される。
2. **自動部分の領域が明示される**: 「ここから先は自動提案」という境界が視覚的に明確。
3. **自動部分の無効化が簡単**: 設定に埋もれておらず、すぐにオフにできる。
4. **自動部分が手動部分を上書きしない**: 自動処理がユーザーのピン留めや配置を破壊しない。

Edge はこの条件 3 を満たさないため深刻な不満を生んでいる。Firefox は条件 2・3 を比較的満たしているため受け入れられている。

## 5. 判断材料の総括

### 5.1 自動学習型は M1b に適合するか

**適合しない。** 以下の3つの軸で矛盾する。

1. **M1b dislike との直接矛盾**: 自動並び替えは「慣れた手順が突然変わる」「同じ操作なのに結果が変わる」という M1b の2つの dislikes を毎回の利用で体験させる。

2. **空間記憶・筋肉記憶の阻害**: NN/G の空間記憶研究、筋肉記憶コストの分析、インターフェース習慣の定量研究がそれぞれ独立に「要素の移動はリピートユーザーに対してネガティブな影響を与える」と結論している。

3. **実装事例の評価**: Chrome / Edge / Safari / iOS / Android のいずれにおいても、自動並び替え・自動追加・自動提案への不満が多数確認されている。特にホームページ・スタートページという「毎日開くもの」においては、変化への忌避感が強い。

### 5.2 手動配置型は M1b に適合するか

**適合する。** ただし初期体験への工夫が必要。

M1b の「気に入った道具を繰り返し使う」「変化を嫌う」という特性に対して、手動配置型は最も直接的に応答する。Toby / Momentum / Start.me / Tabliss のような手動型サービスがいずれも高い評価を得ていることも傍証となる。

初期設定の摩擦は「デフォルトのおすすめツール（提案のみ）」を用意することで対処できる。ただしこの提案は「ユーザーがいつでも削除・変更できる」ことが前提であり、強制的な初期配置はすべきでない。

### 5.3 ハイブリッドは現実的選択肢か

**条件付きで現実的。** ただし「自動部分の位置」が決定的に重要。

実現可能なハイブリッドパターンは以下のとおり。

**推奨パターン: 固定エリア + 提案エリアの分離**

- 道具箱の主領域（上部・中央など目に入りやすい位置）: 手動配置のみ。ユーザーが追加・削除・並び替えを完全に制御する。自動処理はここを一切変更しない。
- 別セクション（下部など目立たない位置）: 「まだ道具箱にないツールの提案」を表示。ユーザーが無視しても主領域に影響がない。

このパターンは「手動配置の安定性を完全に保ちながら、新しい発見の機会を提供する」という両立を可能にする。

**避けるべきパターン: 混在型**

手動ピン留めと自動提案を同一グリッド内で混在させると、境界が曖昧になり「なぜここに増えた？」「自分が置いたのか自動で来たのか？」という混乱が生じる。Edge のケースがこの失敗パターンの典型である。

### 5.4 「能動的に組み立てる道具箱」メタファと M1b の整合

「能動的に組み立てる道具箱」というメタファは M1b の dislikes と矛盾しない。むしろ M1b の「気に入った道具を繰り返し使う」「自分のペースで行動する」という特性に対して正確に応答するメタファである。

自分で選んで置いた道具は動かない。これが M1b にとっての「安心感の源泉」となる。自動学習型は逆に、この安心感を毎回の利用で破壊する。

「ユーザーが組み立てる」という行為は、M1b の関与を前提とするが、M1b はツール選択に積極的であることが前提のセグメントであるため、このコスト（組み立て操作）はバリュー（自分の道具箱という所有感）に変換される。

## 6. 設計判断への示唆

以下は本調査の結果から導ける、道具箱 UI 設計への具体的示唆である。

**優先事項（Must）**

1. タイルの配置は手動操作（ドラッグ or 明示的なボタン）でのみ変化する。自動並び替えは実装しない。
2. 一度配置したタイルは、ユーザーが操作するまで位置が変わらない。localStorage や DB への永続化が必須。
3. 初期状態（空の道具箱）には、削除・移動自由なデフォルトのおすすめツールを数件表示し、セットアップへの案内を明示する。

**検討事項（Should）**

4. 「まだ道具箱にないおすすめツール」を主領域とは分離した別セクションで提案することは、発見性向上のために有効。ただし主領域に自動挿入しないこと。
5. ツール追加の導線は道具箱内に常時見えていること（空き感・伸ばしやすさの確保）。

**禁止事項（Must Not）**

6. 利用頻度・最終アクセス日時に基づくタイルの自動並び替え。
7. ユーザーの同意なしに自動で追加されるタイル。
8. 手動配置を上書きするような同期・自動処理。
