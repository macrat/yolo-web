# 道具箱 UI 設計判断のための個別操作 UX 深掘り調査

> **位置づけ**: `2026-05-01-dashboard-toolbox-phase1-best-practices.md` の「ダッシュボード型 Web サービスの編集 UI パターン」は概観止まりであった。本調査はそれを超える「個別操作 UX」「実機 UI 詳細」レベルの深掘りを提供する。

---

## 1. サービス別 個別操作 UX 詳細

### 1-1. Notion ダッシュボードビュー（2026年3月リリース）

**一次情報源**: Notion 公式ヘルプセンター `https://www.notion.com/help/dashboards`（Playwright で実機確認済み）

**編集モードの有無と切替**

- **明示的2モード分離**。ダッシュボードは常に「View モード」で表示される。編集したいときにデータベースヘッダーの `Edit` ボタンを押してモード切替する。
- 編集が完了したら `Done` ボタンで View モードに戻る。View モードでは誰でも閲覧・フィルター操作ができるが、「レイアウトは変更できない」点が明確に担保される。
- Notion がこの分離を選んだ理由は公式ドキュメントに明記されている：「A stable, app-like surface that's harder to accidentally break」「Clear separation between editing the layout and using the dashboard day to day」。

**並び替え操作**

- Edit モード内でウィジェットをドラッグして移動。行内の並び替えと行間移動の両方が可能。
- ドラッグが使えない場合は右クリック（またはタイトルクリック）でコンテキストメニューを開き move/reorder オプションを使用。
- 横幅はウィジェット間の境界をホバーしてハンドルを左右にドラッグして調整。行の高さはは行間の境界をホバーして上下ドラッグで調整。

**タイル追加 UX**

- Edit モード内で、行の `+` アイコンまたはテーブル下部の `+` をクリック。
- モーダルやサイドバーではなく、既存レイアウトにインラインで挿入される。
- Notion Agent（AI）を呼び出して「このデータベースのダッシュボードを作って」と指示すれば自動生成されたドラフトが展開する。

**タイル削除 UX**

- Edit モード内で、ウィジェットを右クリックしてメニューから `Delete` を選ぶ。
- 確認ダイアログの有無はドキュメントに明記されていない（UI 観察では即削除）。Undo 機能の記載もなし。
- ただし「View モードでは削除操作が不可」という編集モード分離自体が一定の安全策になっている。

**グリッドサイズ / タイル形状**

- 行ベースのグリッド。1行に最大4ウィジェット、ダッシュボード全体で最大12ウィジェット。
- タイルサイズ（横幅）はドラッグで自由に可変。高さは行単位で調整。固定サイズではない。
- 企業・チーム向け設計のため、タイル粒度はデータベースビュー（テーブル・チャート・カレンダー等）単位。個人の小ツールには過剰。

**モバイル密度**

- 公式ドキュメントに「Dashboard views are primarily a desktop-focused feature」の示唆がある。
- 2026年現在、モバイルでのダッシュボード Edit モードはサポートが限定的。

**ターゲット層**

- Business / Enterprise プランのみ（無料不可）。チームでのステータス管理・インシデント管理・経営報告用途が主。個人ユーザー向けではない。

### 1-2. Trello（カード並び替え・ボード）

**一次情報源**: Atlassian 公式ドキュメント（WebFetch 精読済み）

**編集モードの有無**

- **シームレス型**。ボード表示 = 常時編集可能。専用の「編集モード」は存在しない。
- ドラッグ操作はいつでもできる。ただしカード上のクリックはカード詳細を開く（ドラッグとタップ/クリックは物理的に分離されている）。

**並び替え操作**

- ドラッグ&ドロップのみ。キーボードショートカットや並び替えボタンはなし（移動先ボードへの「Move」は別操作）。
- ドラッグ中にカードが傾く（tilting）演出と、元の位置に半透明の ghost が残る視覚フィードバック。

**タイル削除 UX（重要）**

- **2ステップ削除 + 確認ダイアログ**の組み合わせ。
  1. カードをアーカイブ（一時非表示、復元可能）
  2. アーカイブ後に「Delete」オプションが出現 → クリックで確認ダイアログ → 永久削除
- この設計思想：「アーカイブ = クローゼットに入れる。削除 = ゴミ箱に入れて完全消去」と明確に区分。
- キーボードショートカットでアーカイブした際は「Undo」バナーが一時的に画面下部に表示（数秒間）。これはアーカイブのみが対象で永久削除には適用されない。

**グリッドサイズ**

- カンバン形式（リスト × カード）。タイルは縦方向にリスト内で積み重なる。横幅固定（リスト幅は可変）。

**モバイル密度**

- モバイルでは1カラム表示（各リストが縦に並ぶ）。ファーストビューに1リスト分のカードが収まる設計。

### 1-3. Raindrop.io（ブックマークコレクション）

**一次情報源**: 公式ヘルプサイト精読 + Playwright 実機確認

**編集モードの有無**

- **シームレス型**（ただし並び替えには「ソートを Manual に切替える」前処理が必要）。
- 「Sort: Manual」に設定した状態でのみドラッグが有効。デフォルトは日付順。

**並び替え操作**

- Manual ソート選択後にドラッグ&ドロップで並び替え。アニメーションは流体的で高速。
- iOS モバイルでは長押し後に編集モード（カードが少し持ち上がった状態）に遷移してドラッグ。ただし masonry ビュー・card ビューでは iOS でドラッグ非動作のバグが 2025年時点でも存在（list ビューのみ動作確認済み）。

**タイル削除 UX**

- 削除したアイテムは「Trash コレクション」に移動（即削除ではない）。
- ドキュメント記述：「Nothing is removed permanently until you choose to.」
- ゴミ箱からドラッグで元のコレクションに戻せる。または Trash 内で永久削除を実行。
- Undo ボタンは存在しないが、Trash が実質的な Undo 機構として機能する。

**グリッドサイズ / タイル形状**

- Grid・Headlines・Masonry・List の4表示モードを切替可能。
- モバイルアプリでの Grid ビューはデフォルト2カラム（ユーザー変更不可、要望は多数 pending）。

**モバイル w360 での密度**

- Grid ビューで2カラム。各カードは正方形サムネイル。w360 では1カラム当たり約 180px。
- List ビューでは1カラム縦並び。ファーストビューに7〜9件表示可能。

**ターゲット層**

- リサーチャー・クリエイター・デザイナー。「整理されたブックマークから情報を引き出す」がコアユース。ホームページ代わりの使用は想定外。

### 1-4. Start.me（カスタムスタートページ）

**一次情報源**: 公式ブログ記事調査・サポートドキュメント確認

**編集モードの有無**

- **セミシームレス型**。左クリック長押し（ドラッグ開始）でページが自動的に編集モードに入る。ヘッダーに「organize your widgets by dragging them」のメッセージが表示。
- `Done` ボタンで編集モードを終了。
- または、ウィジェット右上の三点メニュー（More Icon）から明示的に設定・移動ができる。

**並び替え操作**

- ドラッグ&ドロップが主。モバイルでは「三点メニュー → 移動」で代替（モバイルでのドラッグは一部制限あり）。
- 2024年アップデートで「別ページへのウィジェットドラッグ」が可能になった。

**タイル追加 UX**

- ヘッダーの `+ Add widgets` ボタン、またはヘッダー内の `+` ボタンをクリック。
- ウィジェットギャラリーが開き、カテゴリ別に選択。RSS・ブックマーク・付箋・カレンダー等を選択して追加。

**タイル削除 UX**

- ウィジェット右上の三点メニューから削除。確認ダイアログあり（要確認クリック）。
- Undo なし。削除は即実行。

**グリッドサイズ / タイル形状**

- 列数を 1〜7 で設定可能。列幅を small〜extra large で調整。
- ウィジェットは列単位でスタック。フリー配置ではない。
- モバイルでは自動的に1カラム表示に縮退。「5カラムに設定した内容を iPad で見ると3カラムになり、はみ出した2カラム分のアイテムは残りの列に統合される」。

**モバイル w360 での密度**

- スマートフォンでは1カラム表示が基本。ウィジェット全体がファーストビューには収まらず縦スクロールで閲覧。
- カラムがあっても縦に積み重なるため、w360 ではブックマークリスト型の縦スクロールUIになる。

**ターゲット層**

- 「毎日ブラウザのホームページとして使う人」が明確な想定ユーザー。学校・ビジネス・個人の500万人超が利用。yolos.net 道具箱と最もコンセプトが近い。

### 1-5. Toby for Chrome（タブ管理ダッシュボード）

**一次情報源**: 公式サイト・Chrome Web Store・複数レビュー

**編集モードの有無**

- **シームレス型**。新規タブを開くと Toby ダッシュボードが表示され、常時ドラッグ可能。
- Collections（タブグループ）はサイドバー左側に一覧。右側のメインエリアにカード形式でタブが表示。

**並び替え操作**

- タブ間・コレクション間でドラッグ&ドロップ。
- コレクション自体の順序もドラッグで変更可能（サイドバー内でコレクションを上下にドラッグ）。

**タイル追加 UX**

- 現在開いているタブを1クリックでコレクションに保存。
- または URL を入力して手動追加。

**タイル削除 UX**

- 右クリックメニューから削除、またはホバー時の「×」ボタン。
- Undo なし。確認ダイアログの有無は 2024年更新後は不明（ユーザーレビューで「タブが突然消えた」報告が多数）。
- 2024年以降の有料化に伴い、ユーザーがデータ消失を報告しているケースが目立つ（設計の問題ではなくアカウント同期の問題の可能性が高い）。

**グリッドサイズ / タイル形状**

- カードグリッドレイアウト。カードは均一サイズの長方形（タイトル・サイト名・favicon 表示）。
- ダッシュボード全体はデスクトップ向けで、w360 での最適化は事実上なし。

**モバイル密度**

- Toby の主用途はブラウザ拡張（デスクトップ）。iOS/Android アプリもあるが、モバイルでの利用は二次的。モバイル w360 での最適化は公式に謳われていない。

**ターゲット層**

- 多くのタブを開いて仕事をするナレッジワーカー・デベロッパー。「毎日ブラウザのホームページとして使う」ユースケースよりも「特定のプロジェクト作業時にセッション管理する」ユースケースが主。

**2024年以降の懸念**

- 60件制限・ログイン強制・有料化転換によりユーザー離脱が相次ぐ。サービス持続性に不安。

### 1-6. Bento.me（リンクインバイオ系ダッシュボード）

**一次情報源**: TechCrunch 取材記事（2023年）・サービス終了確認（2026年2月）

**サービス状況**

- **2026年2月13日にサービス終了**。Linktree に移管。現在は利用不可。

**設計思想（終了前の UI）**

- ベントーグリッド形式。各ブロックを large/small/horizontal/vertical/square などの形状で自由に組み合わせるレイアウト。
- ドラッグ&ドロップでブロックを自由に並び替え。モバイルでも「リンクの追加・ウィジェットの移動が容易」を売りにしていた（TechCrunch 取材より）。
- タイルサイズが可変（1セル・2セル横・2セル縦・4セル等）で、ピンタレスト的な非均一グリッドを実現。

**削除 UX**

- 詳細なドキュメントは残存しないが、インライン編集（専用編集モードの表示が不明確）でホバー時の操作メニューから削除できた模様。

**yolos.net への参考価値**

- 非均一タイルサイズ + 自由配置という設計思想は yolos.net 道具箱のフェーズ2以降の参考になる。
- ただしサービス終了済みのため直接の実機検証は不可。類似サービスとして Linktree や Taplink が後継として機能中。

### 1-7. iGoogle（廃止・歴史的参考）

**一次情報源**: Wikipedia・デジタルアーカイブ・blog記事

**設計思想**

- 2005年〜2013年運営。Google が提供したカスタムスタートページ。
- ガジェット（iframe ベースのミニアプリ）をタブに配置。ドラッグ&ドロップで並び替え。
- **常時編集可能型（シームレス）**。専用の編集モードなし。ドラッグをいつでも開始可能。
- ガジェットの削除は「×」ボタンクリックのみ。確認ダイアログは廃止に向けて削除されたが、廃止初期は「Delete」リンクをクリックしても反応しないバグが発生（シームレス型の複雑さを示す事例）。
- 廃止理由の一つに「スマートフォン・SNS の台頭でパーソナルホームページの必要性が低下した」がある。

**現代への示唆**

- 「常時編集可能型は誤操作が多発する」という問題を先取りしていた。後継の Start.me や Protopage が「より明示的な編集モード」に移行したのはその反省から。

### 1-8. Netvibes / igHome / Protopage（カスタムスタートページ系）

**一次情報源**: 廃止報告記事・Protopage 公式ヘルプ（Playwright 実機確認）・比較レビュー

**Netvibes**

- 2025年6月2日にサービス終了。Dassault Systèmes が 2012年に買収後、個人向けサービスは実質放置状態だった。
- 最盛期：タブ×ウィジェットの2段階構造。ドラッグ&ドロップで並び替え。**常時編集可能型**（シームレス）だったが、「別カラムへのドラッグが外れやすい」という操作精度の問題があった（対照: Protopage は同操作をより正確に実装）。

**igHome**

- iGoogle の設計を模倣したサービス。2025年時点でウィジェットの多くが動作不良で「参照非推奨」評価。事実上の廃止状態。

**Protopage**（2025年現在も稼働中）

Playwright で実機確認（`https://www.protopage.com/web/help/overview`）した内容：

- **タブ × カラム構造**。タブ内で 2〜5 カラムを設定可能。または「freeform positioning（自由配置）」モードでどこにでもウィジェットを置ける。
- **シームレス型**。常時ドラッグ可能。ウィジェットのタイトルバーをドラッグして移動。コーナーをドラッグしてリサイズ。
- ウィジェットを複数カラムにまたがるサイズに拡張可能（react-grid-layout 的なスパン設定）。
- 削除は「×」ボタンのみ（確認なし・Undo なし）。
- モバイル: Protopage Mobile（2008年〜）は存在するが、デスクトップ用レイアウトとは別設計でシンプルなリスト表示に近い。
- Netvibes と比較してドラッグ精度が高く、カラムへの落下が安定している。
- 外観は「2015年頃のデザイン」という批評が多く、現代的とは言えないが機能は充実。

## 2. 観点別 横断分析

### 2-1. 編集モード「分離」vs「シームレス」の設計思想の違い

| 観点               | 分離型（Notion・iOS など）                              | シームレス型（Trello・Protopage・iGoogle） |
| ------------------ | ------------------------------------------------------- | ------------------------------------------ |
| 誤操作リスク       | 低い（View モードでは変更できない）                     | 高い（スクロールとドラッグの混同が多発）   |
| 操作の発見容易性   | 低い（編集モードの存在に気づかないユーザーがいる）      | 高い（どこでもドラッグできる）             |
| 適する並び替え頻度 | 低頻度（セットアップ→安定使用サイクル）                 | 高頻度（毎日変えるユーザー）               |
| 適する規模         | 大規模・複雑（Notion の12ウィジェット上限でも十分複雑） | 小規模・シンプル                           |
| NN/g の推奨        | モードが必要な場合はビジュアル差異を大きくせよ          | モードスリップのリスクを回避せよ           |

**NN/g の知見**（公式記事 `nngroup.com/articles/modes/` より）：

- モードが正当化されるのは「利用可能な入力手段が少なく、提供したい操作が多すぎる場合」。
- モードを使う場合は「冗長な視覚的指標（色の変化・カーソル変化・ラベル）」を組み合わせること。
- Spring-loaded mode（ボタンを押している間だけモードが有効）を代替として検討する価値がある。

**yolos.net への示唆**：

- M1b ターゲット（リピーター・安定使用者）には「分離型」が有利。同じ場所に同じ道具がある安心感を壊さない。
- ただし「編集モードに入れることを知らない」オンボーディング失敗リスクがある。最初のアクセス時のツールチップ等が必須。

---

### 2-2. 並び替え頻度が低いユーザーにとって編集モードは儀式か / 自然か

**観察事実：**

- iOS ジグルモードは「長押し」という物理的な動作が必要で、誤ってモードに入ることが少ない。Apple は iOS 27 で Undo/Redo ボタンをジグルモード内に追加予定と報告されており、長期ユーザーの「誤操作への不安」を認識していることが伺える。
- Notion ダッシュボードは明示的な `Edit` ボタンを置き、公式ドキュメントで「このモードでないとレイアウト変更できない」と宣言することで、View モードでの誤操作を完全に防ぐ。

**「儀式的」という評価は当たるか：**

- 並び替え頻度が「月1回以下」のユーザーにとって、編集モードへの切替は「ちょっと面倒」と感じる可能性がある。しかし逆に、普段は「触ってしまわないか」という不安がない状態で使えることの安心感の方が価値が高い。
- Start.me のセミシームレス型（ドラッグ開始で自動的に編集モードに入る）は折衷案だが、「意図せずドラッグしてレイアウトが変わってしまった」報告がユーザーコミュニティに存在する。

**結論**：並び替え頻度が低い M1b ユーザーには「編集モードは儀式」ではなく「安心の鍵」。明示的な Edit/Done 分離が適切。

---

### 2-3. 削除の Undo / 確認の採用率とトレードオフ

**調査対象8サービスの削除 UX まとめ：**

| サービス              | 削除方法                     | 確認ダイアログ         | Undo / 復元                  |
| --------------------- | ---------------------------- | ---------------------- | ---------------------------- |
| Notion                | 右クリック → Delete          | 不明（即削除の可能性） | なし                         |
| Trello                | 2ステップ（アーカイブ→削除） | 永久削除前に確認あり   | アーカイブは Undo バナーあり |
| Raindrop.io           | 削除 → Trash 移動            | なし                   | Trash からドラッグで復元     |
| Start.me              | 三点メニュー → 削除          | 確認あり               | なし                         |
| Toby                  | ホバー × / 右クリック        | 不明                   | なし（データ消失報告あり）   |
| Protopage             | × ボタン                     | なし                   | なし                         |
| iGoogle               | メニュー削除                 | 廃止前後でバグあり     | なし                         |
| Home Assistant (参考) | 削除メニュー                 | Section 削除は確認あり | なし                         |

**採用パターンの傾向：**

- 「Trash / Archive に移動 → そこから永久削除」パターン（Trello・Raindrop.io）: 最も安全。削除の取り消しが自然にできる。
- 「確認ダイアログのみ」パターン（Start.me）: 中程度の安全策。ただしダイアログに慣れると無意識にクリックするリスク（NN/g の警告）。
- 「即削除・何もなし」パターン（Protopage）: 最も危険。プロトタイプ的な小規模サービスで見られる。

**NN/g の確認ダイアログ研究の知見（精読）：**

- 確認ダイアログは「真に不可逆な操作」に限定して使うべき。頻繁に出ると「狼少年」効果で無視される。
- Undo の方が優れている。ユーザーが「ああ、消してしまった」と気づいた瞬間に復元できる。
- Gmail の「Undo Send」（数秒間表示）はこのパターンの優れた実装例。

**yolos.net「localStorage で操作の容易さを優先」とのトレードオフ：**

- localStorage で保存するレイアウト情報は「ブラウザ上のローカルデータ」であり、クラウドバックアップはない。
- 削除したタイルを「Trash」に保持し続けることは localStorage への追加負荷となるが、タイル情報は小さくフェーズ1では問題にならない。
- 推奨：**5秒間 Undo バナー方式**。Gmail 型の軽量 Undo を実装し、「あなたのツールボックスから [ツール名] を削除しました。[元に戻す]」を画面下部に数秒間表示。localStorage への書き込みを Undo タイムアウト後に実行。確認ダイアログ不要でフローが自然。

### 2-4. モバイル密度の業界相場

**各サービスの w360 実態：**

| サービス             | w360 レイアウト                    | ファーストビュー内タイル数     |
| -------------------- | ---------------------------------- | ------------------------------ |
| Raindrop.io（Grid）  | 2カラム（固定）                    | 約6枚（3行×2カラム）           |
| Raindrop.io（List）  | 1カラム                            | 7〜9件                         |
| Start.me             | 1カラム（自動縮退）                | ウィジェット1〜2個（縦に長い） |
| Toby                 | デスクトップ最適化、モバイル非対応 | -                              |
| Notion Dashboard     | デスクトップ主体、モバイル制限     | -                              |
| Protopage            | 別途モバイル版（シンプルリスト）   | 5〜7件                         |
| iOS ホーム画面       | 4〜6カラム（アイコン小）           | 20〜24個（4列×5行）            |
| Android ウィジェット | グリッドスナップ                   | サイズ依存                     |

**業界相場の整理：**

- 「スタートページ系」サービス（Start.me・Protopage）はモバイルで1カラムに縮退。タイルの情報密度は低く、大きな操作領域を確保。
- 「ブックマーク管理系」（Raindrop.io）はモバイルで2カラム。情報密度高め。
- iOS/Android のシステム UI はアイコン小・密度高（1アイコンが 80×80px 程度）。

**yolos.net 道具箱への示唆：**

- 「ページを開いた瞬間に入力欄が見えてすぐ使える」（M1a の要求）を実現するには、ファーストビューに少なくとも2〜4タイルが収まる密度が必要。
- w360 でタイル1枚が画面幅全体を占めると、スクロールしないと2つ目のツールが見えない問題が生じる。
- 推奨：**w360 では2カラム（各 160px 程度）をベースとし、ウィジェット本体（計算機・時計等）は1カラム幅のタイルで全画面展開する設計**。Raindrop.io の Grid ビューが参考になる。
- ツールタイル（クリックで起動するもの）は 2カラムで十分。ウィジェット（インタラクティブなもの）はタップ後に全画面展開または大きなモーダルで表示するパターンが適切。

## 3. yolos.net 道具箱 設計判断まとめ

### 3-1. 並び替え UI の推奨

**推奨：明示的2モード分離（Edit/Done）+ ドラッグ&ドロップ**

根拠：

1. M1b（リピーター）が最重視する「慣れた操作手順が突然変わらない安定性」には、View モードでの誤ったレイアウト変更を防ぐ分離型が必須。
2. 並び替え頻度が低い（月1回以下と想定）ユーザーには、編集モードの「儀式感」よりも「安心感」の方が価値が高い。
3. Start.me（最も親和性の高いサービス）も Edit/Done 分離型に近いセミシームレス型を採用している。
4. NN/g の研究：「誤操作でレイアウトが崩れる」モードスリップのリスクを避けるため、分離型を推奨。

**実装上の注意：**

- 編集モードへの入口は「ページ右上のハンマー/鉛筆アイコン」1クリック以内に固定。
- 編集モード中はタイルにドラッグハンドルが出現するビジュアル変化を明確に。
- 「Done」ボタンで View モードに戻る際のアニメーションで「今から安定使用モードになりました」を伝える。

### 3-2. 編集モードの存在意義

**存在意義：「毎日使う場所」だからこそ「変えてしまう事故を防ぐ」ための分離が必要**

- iGoogle・Netvibes・Protopage のシームレス型を実際に使ったユーザーの報告：「スクロールしようとしてウィジェットを動かしてしまった」が繰り返し出現。
- 毎日開く場所で「いつものツールがどこかに行った」という体験は M1b ユーザーの最大の dislikes 要因。
- 「モードが面倒」は「知った後の一時的な感情」。「いつものが消えた」は「長期的な信頼喪失」。設計コストとしては前者を選ぶべき。

### 3-3. 削除安全策の推奨

**推奨：5秒間 Undo バナー方式（確認ダイアログ不要）**

根拠：

1. NN/g の研究：確認ダイアログは「本当に不可逆な操作」のみに限定。頻繁に使うと無視される。
2. 道具箱タイルの削除は「致命的な不可逆操作」ではない（同じツールを後で再追加できる）。
3. Trello の「アーカイブ→削除の2ステップ」は過剰。道具箱タイルのスコープでは不要。
4. Raindrop.io の「Trash」方式も localStorage のみの実装では管理が煩雑になる。
5. Gmail 型「Undo バナー（5秒間）」が最もフリクション低く・安全策として十分。
6. localStorage への書き込みタイミングをバナー消滅後（5秒後）にすることで、実装としても整合する。

**実装案：**

```
[×] あなたの道具箱から「電卓」を取り除きました。  [元に戻す]
```

画面下部に 5 秒間表示。バナー消滅後に localStorage を更新。

### 3-4. モバイル密度の推奨

**推奨：w360 では 2カラム（1タイル = 約 160px 幅）**

根拠：

1. Raindrop.io の Grid ビューが w360 で 2カラムを採用しており、業界相場として成立。
2. M1a（作業ツールをすぐ使いたい）には「ファーストビューで複数ツールが見える」密度が必要。1カラムだとスクロールが必須になる。
3. Start.me の1カラム縮退は「ウィジェット内の情報が多い」ため必要だが、道具箱タイルは「アイコン＋ツール名」の薄い情報量なので2カラムでも十分な可読性を保てる。
4. 最低タッチターゲット：44×44px（Apple HIG）または 48×48px（Material Design）を各タイル内のインタラクティブ要素に適用することで、2カラム（160px 幅）でも安全なタッチターゲットを確保できる。

**参考数値（業界相場）：**

- w360 × 2カラム = 各カラム約 160px（マージン・ガター除く）
- 各タイル高さ: 80〜100px が適切（アイコン 40px + テキスト 20px + 余白）
- ファーストビュー（w360 × h700 想定）に収まるタイル数: 2カラム × 6〜7行 = 12〜14タイル

### 3-5. タイル追加 UX の推奨

**推奨：Edit モード内にのみ「+ 追加」ボタンを表示**

根拠：

1. View モードで「+ 追加」ボタンが常に見えていると、M1a ユーザーが「ツールを探す」ことと「ツールを追加する」を混同する。
2. Edit モード内の「+」ボタンは「今は変更モードにいる」という文脈の中で自然に機能する。
3. Notion のインラインインサートパターン（行の + アイコン）は学習コストが低い。
